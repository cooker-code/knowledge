# 个人知识整理规范

## 目标

协助整理个人知识点，在阅读文章时判断：对象是什么、在技术体系里的位置、横向对标、适用边界与代价、是否值得长期积累。

## 基础文件

| 文件 | 职责 | 不写什么 |
|---|---|---|
| `AGENTS.md`（本文件） | LLM 整理规则，引用其余三个文件 | 不重复用户画像、分类表、模板字段 |
| `用户画像.md` | 用户认知水平、内容取舍、提取分级 | 不写具体技术分类和文章笔记 |
| `目录划分.md` | 一级/二级类目、归类优先级、跨域边界 | 不写文章完整整理结果 |
| `技术定位卡.md` | 整理文章的输出模板和字段 | 不维护完整分类表 |

归类查 `目录划分.md`，认知水平查 `用户画像.md`，输出结构查 `技术定位卡.md`。

## 整理流程

整理任何文章前必须完成：

1. 读 `用户画像.md` → 判断用户对该主题的认知层级
2. 读 `目录划分.md` → 确定归入哪个二级类目
3. 检查目标二级目录的 `AGENTS.md` 排重准则和已有三级节点 → 进入已有节点 / 建议新增 / 创建节点
4. 按 `技术定位卡.md` 输出技术 `AGENTS.md` 或文章核心知识点

每篇文章必须先回答：
- 对象是什么（技术/产品/框架/实践案例）
- 解决的核心问题是什么
- 在全局架构中属于哪一层
- 对标哪些技术方案
- 对用户的价值（现在用 / 以后关注 / 仅了解 / 可忽略）
- 进入已有节点还是触发新增三级节点提醒

先输出"认知校准点"（与用户已有判断的冲突/偏差/补充），再输出"待吸收点"。

## 使用到的 Skill

| Skill | 触发场景 | 职责 | 收口要求 |
|---|---|---|---|
| `knowledge-distiller` | 用户要求"蒸馏"某个一级、二级或三级目录，批量处理 `文章/` | 做目录级全量蒸馏：补核心知识点、更新 `AGENTS.md` / 知识地图、保留有贡献文章并改 `done-`、删除无贡献文章 | 必须先确认目标根目录；一级目录任务必须覆盖该一级下所有二级和三级节点，最终按二级输出 `done`、裸文章、断链、未引用 `done` |
| `article-knowledge-extractor` | 阅读、整理、归类、吸收单篇或多篇文章 | 做文章级判断：读 `用户画像.md`、`目录划分.md`、`技术定位卡.md`，先提认知校准点，再提待吸收点 | 必须说明认知增量、归类依据、阅读投入和是否进入长期知识；不能只说"总结文章" |

使用顺序：
- 目录级批量任务：先用 `knowledge-distiller` 确认范围和收口标准，再用 `article-knowledge-extractor` 的文章判断流程做分类、排重和认知增量判断。
- 单篇文章任务：直接按 `article-knowledge-extractor` 流程处理。
- 当用户说"一级目录下所有"时，即使链接指向某个二级目录，也必须把目标上提到对应一级目录并覆盖全部二级/三级节点；不能只完成被链接的子目录后声明完成。

## 内容理解四步法

> **标题只是入口，正文主问题才是分类依据。** 任何分类、排重、新建/合并判断都以四步法结论为准。

```
① 看结构 → ② 找主问题 → ③ 评论据质量 → ④ 判断认知增量
```

**① 看结构**：扫小标题、加粗词、代码块标题。聚焦单一问题继续深读；大杂烩或多问题拼接信息密度低，倾向略读。

**② 找主问题**：看正文第一屏 + 结论段，**不看标题**。问"这篇文章要解决读者的哪一个具体困惑？"。标题与主问题不一致是常见情况，分类只看主问题。

> ⚠️ **标题陷阱**：标题中出现的框架名、产品名、技术词是"入口"不是"主题"。常见错误模式：
> - 标题含框架名（DeerFlow、LangChain）→ 直接归入该框架目录。正确做法：看文章是研究该框架的具体机制，还是以该框架为论据论证另一个问题。
> - 标题含场景词（多 Agent 协作、AI 时代）→ 归入最近似的 Agent 目录。正确做法：看主问题是场景本身，还是借场景讲 Git/工具/规范等具体操作。
> - 标题含通用范式词（子代理、ReAct）→ 归入范式目录。正确做法：看文章讲的是通用机制，还是某个产品的具体能力边界（如 Claude 的硬约束）。

**③ 评论据质量**：

| 论据类型 | 价值 | 处理 |
|---|---|---|
| 机制说明（原理 / 流程图 / 源码链路） | 最高 | 蒸馏判断准则 |
| 实测数据（有版本 / 基线 / 环境） | 高 | 蒸馏时标注版本 |
| 生产案例（有排障路径和结论） | 中 | 只提取可复用准则 |
| 配置清单（只列参数，无机制） | 低 | 不单独新建知识点 |
| 营销 / 故事叙述 | 无 | 跳过 |

**④ 判断认知增量**（对比目标节点已有知识点）：

- 提供已有知识点没有的失败模式 / 边界条件 / 版本差异 / 可操作判断 → 精读候选
- 只是重述已有内容 → 略读或跳过

## 各级 AGENTS.md 和知识地图的职责

**根 `AGENTS.md`（本文件）**：只维护整理流程、内容理解方法、目录规则和 Skill 使用规则。它是模型执行知识整理任务的总规范，不写具体技术分类细节，不写文章笔记。

**一级 `AGENTS.md`**：维护该一级目录下的二级目录状态、一级内路由边界、全量蒸馏收口记录。它用于防止“只处理某个二级目录却声明一级完成”。

**二级 `AGENTS.md`**：维护领域入口、三级节点索引、排重指纹、用户认知重点和待补缺口。它是新文章进入该领域时的排重入口，不是完整技术说明；可以简洁，但不能缺少已覆盖节点和排重准则。

**三级 `AGENTS.md`**：维护技术定位、官方锚点、上下游、横向对标、文章处理 SOP、排重判断和后续追查。它是模型处理该技术节点的规则入口，不等同于知识地图。

**知识地图 `编号_知识地图.md`**：给人读的流程地图和知识点索引，回答“这个技术/领域有哪些模块、已沉淀哪些核心知识点、还缺什么”。三级节点的新建/精修应优先把知识地图放在三级目录根部；历史目录若仍放在 `编号_核心知识点/` 内，精修时迁回根部并更新链接。

**核心知识点 `编号_核心知识点/*.md`**：承接长期可复用的判断准则、边界、反例、失败模式、版本差异和实践验收。不要把单篇文章摘要直接堆进核心知识点。

**版本记录 `编号_版本记录.md`**：只记录依赖官方文档、官方 Changelog 或 GitHub Releases 的版本、兼容性、配置/API 变化；不能凭文章或记忆猜版本。


目录规则：
- 一级目录已定型，不新增、不改名
- 二级目录是认知领域，不是文章分组，新增须谨慎
- 三级节点是可长期积累的技术对象，至少有一篇精读文章支撑
- 二级目录已有三级节点时，新文章只能进三级节点的 `文章/`，不能再放二级 `文章/`
- 二级目录没有合适三级节点时，可以用 `二级编号_核心知识点/` 承接二级直承主题；后续文章足以形成稳定技术对象时，再拆出三级节点
- `文章/` 只保留来源锚点；保留文章必须加 `done-` 前缀，并被核心知识点、知识地图或 `AGENTS.md` 引用
- 不保留长期“路由表”“临时目录”“文章池”；批量路由结论必须沉淀到 `AGENTS.md`、知识地图或核心知识点
- 编号规则：一级 2 位（如 `03`），二级 4 位（如 `0302`），三级 6 位（如 `030202`）


## 核心规则

- **批判姿态**：判断原文是事实、经验还是营销包装；识别缺失的前提条件、代价、替代方案。
- **排重**：问题指纹 = 技术本体 + 所属模块 + 核心机制 + 解决问题 + 适用边界。指纹相同且无新增量时追加原文链接，不新建笔记。
- **原文**：只放链接和追查锚点，不复制全文；技术图保留或 Mermaid 重建，封面/二维码/营销图删除。
- **联网**：初始化阶段只用本地资料；联网补证只在后续精修节点时进行，并标明这是补证阶段。
- **Skill**：按上方"使用到的 Skill"执行；目录级蒸馏必须使用 `knowledge-distiller`，文章级判断必须使用 `article-knowledge-extractor`。
