# 个人知识整理规范

## 目标

协助整理用户的个人知识点，让用户在阅读某一篇文章时，能更全面、更有侧重地判断：

- 这篇文章讲的对象是什么。
- 它在完整技术体系里处在哪个位置。
- 它和哪些技术、方案、概念横向对标。
- 它的适用边界、代价和风险是什么。
- 它是否值得进入用户的长期知识积累。
## 基础文件与文件职责

本知识库的基础文件固定为 4 个：

| 名称 | 文件 | 职责 |
|---|---|---|
| 根 AGENTS | `AGENTS.md` | 维护知识库整理总规则、目录层级职责、处理流程和禁止事项 |
| 用户画像 | `用户画像.md` | 维护用户认知水平、内容取舍、认知校准优先级 |
| 目录划分 | `目录划分.md` | 维护一级/二级类目、归类优先级、跨域边界和新增目录准入 |
| 技术定位卡 | `技术定位卡.md` | 维护二级 `AGENTS.md`、三级 `AGENTS.md`、流程总览和文章核心知识点的输出模板 |

后续提到“基础文件”时，只指这 4 个文件。后续提到“根 AGENTS”时，只指 `/Users/blank/note/knowledge/AGENTS.md`。

- `AGENTS.md`：LLM 整理知识时必须遵守的规则，也就是“根 AGENTS”。
- `CLAUDE.md`：只引用 `AGENTS.md`，避免多份规则漂移。
- `用户画像.md`：唯一的用户画像与认知水平来源，维护用户当前能力假设、提取重点和内容分级。
- `目录划分.md`：唯一的分类准则来源，维护技术域、二级类目、边界判断、跨域示例。
- `技术定位卡.md`：唯一的文章整理模板，维护输出字段和卡片结构。
- 一级类目 `AGENTS.md`：维护该一级技术域的范围、二级认知领域索引、跨二级流程和治理规则。
- 二级类目 `AGENTS.md`：维护该问题域的认知假设、排重准则、已覆盖技术和待补边界。
- 三级技术/主题节点 `AGENTS.md`：放具体技术或稳定专题的纵向知识地图。
- `编号_核心知识点/`：按层级承担不同职责，不使用 `README.md` 作为主要内容文件。

## 基础文件分工

一个准则只放在一个地方，避免多份规则漂移。

| 文件 | 职责 | 不应该放什么 |
|---|---|---|
| 根 `AGENTS.md` | 说明 LLM 的工作目标、整理流程、引用哪些基础文件 | 不重复维护用户画像、分类表、边界案例、模板字段 |
| `用户画像.md` | 维护用户画像、认知水平、内容提取分级、画像更新规则 | 不写具体技术分类表，不存具体文章整理结果 |
| `目录划分.md` | 维护一级类目、二级类目、归类优先级、跨域工具边界示例 | 不写具体文章的完整整理结果 |
| `技术定位卡.md` | 维护整理一篇文章时的固定输出模板 | 不维护完整分类表，只引用 `目录划分.md` |

判断用户认知水平、决定哪些内容要了解/理解/记住/实践，以 [用户画像.md](用户画像.md) 为准。

分类、边界、跨域判断以 [目录划分.md](目录划分.md) 为准。

整理文章时的输出结构以 [技术定位卡.md](技术定位卡.md) 为准。

## Skill 执行要求

整理文章时应使用 `.claude/skills/article-knowledge-extractor/SKILL.md` 中的流程。

如果运行环境不会自动触发该 skill，也必须手工执行同样流程，并在报告中说明：

- 已读取 `用户画像.md`、`目录划分.md`、`技术定位卡.md`。
- 已按“认知校准点优先于待吸收点”的顺序处理。
- 已做原目录纠偏、图片筛选、排重判断。

不能只说“总结文章”，必须明确文章对用户的认知增量和是否值得投入阅读。

## 原文处理原则

- 初始化阶段只使用本地资料：`本地文章目录`、已有 `knowledge`、本地 `wiki` 和脚本输出。不要联网查询官网、GitHub、官方文档或外部文章。
- 初始化阶段遇到官网/GitHub/版本状态缺口时，先标为“后续补证”，不要为了补全锚点中断全量初始化。
- 联网补证只在后续精修三级技术/主题节点 `AGENTS.md` 或正式核心知识点时进行，并且必须明确这是补证阶段，不属于全量初始阶段。
- `knowledge` 只放结构化理解、技术定位、横向对比、文章索引和后续追查锚点。
- 一篇原文可以被多个专题引用，但只能通过链接引用。
- 引用原文时优先保留可追查锚点，例如本地路径、原始链接、标题、关键段落、关键图。
- 不复制原文全文，只提炼核心知识点、判断、对比和可复用结论。

## 流程优先的知识沉淀法

任何目录都不能先从文章清单开始整理。必须先写清该模块的宏观应用流程，再把文章内容吸收到具体流程节点里。

固定顺序：

1. 先定义本模块的应用流程：从什么输入开始，经过哪些设计、实现、验证、治理、运维或复盘节点，最终产出什么。
2. 再定义每个流程节点应该回答的问题：这个节点要做什么判断、产出什么结果、常见风险是什么。
3. 再读取文章：判断文章补的是哪个流程节点，而不是按文章标题新建孤立总结。
4. 再对比已有沉淀：判断文章内容是补充、冲突、不同层次、更优做法，还是低价值重复。
5. 最后写入 `编号_核心知识点/`：必须把文章观点转化为节点知识、判断准则、边界和可执行动作。

目录最低交付标准：

| 对象 | 必须具备 | 不合格形态 |
|---|---|---|
| 一级/二级/三级目录 `AGENTS.md` | 有本模块宏观流程，并能指向核心知识点 | 只有文章数量、文章目录或初始化统计 |
| `编号_核心知识点/知识地图.md` | 有流程节点、节点知识点、文章吸收位置、待补缺口 | 只列文章链接，不解释文章补了哪个环节 |
| 单篇或主题知识点 | 写清文章对某个流程节点的补充、冲突或降权结论 | 复述文章内容、做摘要、只贴原文链接 |

新文章进入时的判断表：

| 判断项 | 必答问题 |
|---|---|
| 流程节点 | 文章优化的是流程中的哪个环节 |
| 关系 | 对当前沉淀是补充、冲突、不同层次、更优做法还是低价值重复 |
| 吸收结论 | 哪条判断准则、实现方式、风险边界或验证动作要写入知识点 |
| 不吸收理由 | 如果不进入核心知识点，原因是已知、标题党、证据不足还是低置信分类 |

## 图片处理原则

文章中的图片要区分“技术图”和“配图”。

必须保留或重建的图片：

- 架构图：展示系统组件、上下游、模块关系。
- 流程图：展示执行链路、数据流、状态流转。
- 说明图：展示存储结构、索引结构、调度关系、计算过程。
- 对比图：展示技术差异、演进路线、性能关系，且有明确技术含义。

应删除或不进入知识点的图片：

- 封面图、二维码、公众号引导图。
- 人物照片、会议海报、营销配图。
- 表情包、装饰图、无技术信息的截图。
- 只为了排版美观但不帮助理解技术机制的图片。

处理要求：

- 原文已有可用技术图时，在“原文锚点 / 关键图”里保留图片路径或原文位置。
- 如果原文图片在 Markdown 中缺失，但正文明确引用“架构图/流程图/下图”，要标记“原图缺失，需要回原文查看”。
- 如果原图不可用，但文章描述足够明确，可以用 Mermaid 重建简化架构图，并注明是“基于原文描述重建”。

## 知识目录结构

具体知识内容按“一级类目 -> 二级类目 -> 具体技术/主题节点 -> AGENTS.md”的方式组织。

```text
knowledge/
  AGENTS.md
  CLAUDE.md
  用户画像.md
  目录划分.md
  技术定位卡.md
  03_数据工程与数仓/
    0302_离线数仓/
      AGENTS.md
      030202_Hive/
        AGENTS.md
        030202_核心知识点/
          文章标题或主题.md
      030203_Kyuubi/
        AGENTS.md
      030205_Spark/
        AGENTS.md
```

规则：

- 一级目录已经定型，只能使用当前文件系统中已存在的一级目录。不要主动新增、改名、拆分或合并一级目录；如果文章明显超出现有一级目录，只能在路由表中标为“一级目录候选 / 需用户确认”，并说明为什么现有一级目录无法承载。
- 二级目录是用户的技术认知领域，不是文章主题的临时分组。新增二级目录必须谨慎，默认不主动创建；只有当现有二级目录无法表达一个稳定认知领域时，才作为“二级认知领域候选”提醒用户，并同时给出应补的二级 `AGENTS.md`、知识地图和边界规则。
- 三级技术/主题节点是整理者需要主动发现和细分的层级。处理文章时，应在当前二级认知领域下识别可长期积累的具体技术、稳定专题、关键机制或工程方法；现有三级节点不够细时，可以提出或创建新的三级节点，但必须给出判断依据。
- 二级类目目录统一使用 4 位前缀：前两位继承一级目录编号，后两位是二级编号，例如 `0701_后端架构`、`0902_文档与知识工具`。
- 三级技术/主题节点目录统一使用 6 位前缀：前四位继承二级类目编号，后两位是该二级类目下的节点编号，例如 `030202_Hive`、`020104_Pi`、`070103_Python`。
- `编号_核心知识点/知识地图.md` 是固定知识地图入口；目录名按父目录编号生成，文件名固定为 `知识地图.md`。一级目录使用 `一级编号 + 00`，例如 `03_数据工程与数仓/0300_核心知识点/知识地图.md`；二级目录使用 `二级编号 + 00`，例如 `0303_实时计算/030300_核心知识点/知识地图.md`；三级目录使用自身 6 位编号，例如 `030301_Flink/030301_核心知识点/知识地图.md`。
- `文章/`、`编号_核心知识点/` 是固定内容容器，不属于分类节点；其中 `编号_核心知识点/` 通过父目录编号保证排序和定位，不新增分类层级。
- 二级目录如果已经有三级技术/主题节点，二级目录下不能再保留 `文章/`；原文必须进入对应三级节点的 `文章/`。没有三级节点的二级目录，可以继续保留二级 `文章/`。
- 不再创建或保留 `99`、`xx99_待人工判断`、`xx99_待归类` 这类临时目录。低置信文章只在路由表或处理日志中记录判断，不能作为长期知识目录保留。
- 二级类目目录只承载该类问题域，不直接塞一篇大杂烩文章。
- 每个二级类目建议有 `AGENTS.md`，作为该问题域的认知与排重入口。
- 二级类目 `AGENTS.md` 应包含：用户在该问题域的认知假设、排重准则、已覆盖技术、待补技术、典型误归类。
- 每个三级技术/主题节点必须有 `AGENTS.md`，作为该技术或专题的纵向知识地图。
- `AGENTS.md` 应包含定位、官网/GitHub、架构图、核心模块、上下游、横向对标、阅读路线。
- 有价值文章提取出的内容放在该三级目录下的 `三级编号_核心知识点/`，一篇文章或一个主题一个文件。
- `编号_核心知识点/` 文件只保留文章中可复用的机制、判断、边界和追查锚点，不做全文搬运。

### 层级内容职责

| 层级 | 目录职责 | `AGENTS.md` 职责 | `编号_核心知识点/` 职责 | `文章/` 职责 |
|---|---|---|---|---|
| 根目录 | 只放基础文件和一级目录 | 根 AGENTS，维护总规则 | 不设置根级 `编号_核心知识点/` | 不放文章 |
| 一级目录 | 固定大技术域，不新增、不改名 | 说明一级域范围、二级认知领域索引、跨二级主流程 | 长期只保留 `知识地图.md`；历史路由表可暂存为处理记录，但不算核心知识点 | 不新增一级 `文章/`；历史遗留文章应进入批量路由 |
| 二级目录 | 用户的技术认知领域 | 维护问题域画像、流程、排重准则、已覆盖/待补三级节点 | 保留二级知识地图；没有三级节点时可暂存该问题域核心知识点，有三级节点后文章级知识应下沉 | 没有三级节点时可保留原文；已有三级节点时不能再新增二级 `文章/` |
| 三级目录 | 可长期积累的技术对象或稳定专题 | 维护技术纵向知识地图、上下游、对标、阅读路线 | 放该技术/专题的 `知识地图.md` 和文章提炼出的核心知识点 | 放该技术/专题的原文锚点 |

一级目录下的 `xx00_核心知识点/知识地图.md` 是一级 `AGENTS.md` 的辅助文件：一级 `AGENTS.md` 负责定义这个技术域“是什么、管哪些二级认知领域、边界是什么”，`知识地图.md` 负责把该一级域的宏观应用流程、二级/三级节点吸收位置、待补缺口和跨节点关系落成可检查的知识地图。它不是文章笔记容器，也不替代一级 `AGENTS.md`。

### 三级节点自主发现规则

新增三级节点不是为了让目录更细，而是为了让某个技术对象或稳定专题可以被长期积累、排重和复用。

可以新增三级节点的情况：

- 文章讨论对象已经不是二级目录的泛化问题，而是一个明确技术、产品、协议、运行时、架构模式或工程方法。
- 该对象能沉淀出独立的定位、核心模块、上下游、横向对标、适用边界和阅读路线。
- 它与现有三级节点的问题指纹不同，不能自然合并到已有节点。
- 它能补齐当前二级认知领域流程中的一个稳定环节，而不是一次性资讯、热点名词或营销包装。
- 至少有一篇精读/实践价值文章，或多篇略读文章共同指向同一稳定主题。

不能新增三级节点的情况：

- 只是低置信分类、标题党、工具清单、版本资讯或一次性观点。
- 只是已有三级节点的同义词、子功能、部署方式或案例，不具备独立知识地图。
- 只是原目录里出现过，正文主问题并不支持它成为长期节点。
- 需要新增二级认知领域才能成立，但用户尚未确认二级目录。

新增或建议新增三级节点时，必须在本次报告或路由表中单独提醒：

| 字段 | 要求 |
|---|---|
| 新增状态 | `已创建` / `建议新增` / `暂不创建` |
| 目标路径 | 使用下一个可用 6 位编号，不重排已有目录 |
| 所属二级认知领域 | 说明为什么仍属于当前二级目录 |
| 触发文章或证据 | 列出原文锚点、关键标题或批量路由依据 |
| 新增理由 | 说明它补的是哪个流程节点或知识空白 |
| 与已有节点差异 | 说明为什么不是合并到现有三级节点 |
| 风险与后续动作 | 标记低置信、待补证、待用户确认或需要补 `AGENTS.md` |

如果实际创建了三级目录，最低动作是同步创建该目录 `AGENTS.md`；若文章已进入正式沉淀，还要创建或更新 `三级编号_核心知识点/知识地图.md`。如果只是候选，不创建目录，只写入路由表或处理日志。

## 批量路由优先

面对大量待处理文章时，不要直接对每篇文章生成完整笔记。先做批量路由表，再决定是否正式沉淀。

批量路由表至少包含：

| 字段 | 作用 |
|---|---|
| 原始文件 | 保留追查锚点 |
| 原始目录 | 判断是否有目录误导 |
| 最终归类 | 按 `目录划分.md` 重新归类 |
| 技术对象 | 归到已有三级节点、建议新增三级节点，或暂不创建三级节点 |
| 三级节点状态 | 已有节点 / 新增已创建 / 建议新增 / 暂不创建 |
| 阅读投入建议 | 跳过 / 略读 / 精读 / 实践 |
| 认知校准点摘要 | 判断是否会改变用户已有准则 |
| 冲突点 | 记录分类、标题、图片、证据、时效、排重等问题 |
| 置信度 | 高 / 中 / 低，低置信不直接写正式笔记 |
| 图片状态 | 无图 / 有技术图 / 原图缺失 / 可重建 |
| 是否正式沉淀 | 只有精读/实践且有认知增量的文章进入正式笔记 |

默认规则：

- `跳过`：不写正式笔记，只保留跳过理由。
- `略读`：只进入路由表，除非补足一个明显技术空白。
- `精读`：写入对应 `编号_核心知识点/`，但先查重。
- `实践`：写入对应 `编号_核心知识点/`，并在后续追查中留下实验或验证动作。

### 阅读投入门槛

阅读投入建议不是按标题关键词决定。标题中出现“实战、架构、源码、原理、最佳实践”只能提高关注度，不能直接判定为 `实践`。

| 建议 | 判定标准 | 输出动作 |
|---|---|---|
| 跳过 | 标题党、清单、面试题、营销文、重复常识、无证据观点 | 只保留跳过理由 |
| 略读 | 资讯、版本发布、工具盘点、背景材料，只有少量生态信息 | 路由表保留，不写核心知识点 |
| 精读 | 能补全架构位置、模块机制、横向对标、适用边界或认知校准 | 进入候选池，先查重再决定是否沉淀 |
| 实践 | 有可运行、可验证、可排障、可迁移的动作 | 写入核心知识点，并留下实验或验证动作 |

`实践` 必须同时回答：

| 门槛 | 必须说明 |
|---|---|
| 可运行 | 是否能复现一个最小例子、命令、配置或 SQL |
| 可验证 | 是否有输入、输出、指标、对照组或验收标准 |
| 可排障 | 是否能沉淀失败模式、日志信号、定位路径 |
| 可迁移 | 是否能迁移到用户已有的数据工程、Agent 工程、OLAP 或分析场景 |

不满足这些门槛的文章，即使标题叫“实战”，也只能判为 `精读`。

### 冲突点提示

批量路由和单篇整理都必须显式记录冲突点。冲突点优先级高于普通摘要。

| 冲突类型 | 判断方式 | 处理 |
|---|---|---|
| 原目录冲突 | 原目录与按技术本体判断的最终归类不一致 | 按 `目录划分.md` 重路由 |
| 关键词误导 | 正文提到相邻技术，但文章主问题不是该技术 | 分类以标题、路径、主问题为主，正文只辅助 |
| 标题/观点降权 | 标题包含“神器、100%、取代、暴增、终极、硬刚”等强宣传词 | 只保留可证据化部分 |
| 图片缺失 | 正文提到架构图、流程图、Figure、下图，但 Markdown 无图 | 标记“原图缺失”，必要时 Mermaid 重建 |
| 实践/资讯混杂 | 发布、月报、推荐类文章带实践或框架名 | 降为略读或精读候选 |
| 证据不足 | 性能、准确率、收益没有基线、数据量、环境、版本 | 标记待验证，不写成结论 |
| 排重冲突 | 问题指纹与已有核心知识点相同 | 合并或追加原文，不新建 |
| 低置信分类 | 当前规则无法稳定归类 | 只写入路由表或处理日志，不创建 `99` 临时目录，也不硬塞到相近目录 |

### 全量处理准入

不要把“全量”混同为“全量写笔记”。全量分两层：

| 阶段 | 是否可以进入 | 判断 |
|---|---|---|
| 全量轻量路由 | 可以 | 10% 灰测已经暴露出主要系统问题，继续随机灰测收益不高 |
| 全量正式沉淀 | 不可以 | 仍存在图片缺失、低置信分类、实践判定偏宽、目录覆盖不足 |

进入全量轻量路由时，必须保留：

- 路由明细。
- 三级节点状态。
- 冲突点。
- 置信度。
- 图片状态。
- 阅读投入建议。
- 是否正式沉淀。

全量正式沉淀只能处理全量路由之后筛出的高价值候选，优先满足：

- 认知校准点强。
- 技术图可恢复或可重建。
- 与用户当前技术栈 level 提升相关。
- 不与已有问题指纹重复。
- 可以补二级类目 `AGENTS.md` 或三级技术/主题节点 `AGENTS.md` 的长期结构。

### 可吸收的 LLM Wiki 机制

当前知识结构不采用纯 `wiki` 的“一个概念一个文件 + 强制 wikilink 网络”，但吸收其中的治理机制：

- 原文锚点必须可追查。
- 技术图要本地化、标缺失或重建。
- 知识条目使用 `draft / verified / proven` 成熟度。
- 批量处理要保留路由表和处理日志。
- 定期检查断链、重复问题指纹、矛盾、图缺失、过时和低置信条目。

## 基本姿态

整理时要有批判角色，不能把原文说法直接当结论。

必须主动判断：

- 原文是在讲完整技术，还是只讲一个局部模块。
- 原文是在讲事实、经验、营销包装，还是个人观点。
- 原文有没有遗漏前提条件、代价、替代方案和失败场景。
- 原始目录是否误导分类，例如把 Agent 工具文章放到大模型目录，把 OLAP 实践放到数仓目录。

## 整理输出要求

整理任何文章前，必须先做四步：

1. 读取 [用户画像.md](用户画像.md)，判断用户对该主题大概处在哪个认知层级。
2. 读取 [目录划分.md](目录划分.md)，判断文章应归入哪个一级/二级类目。
3. 检查目标二级目录下已有三级节点和二级 `AGENTS.md` 排重准则，判断是进入已有节点、建议新增三级节点，还是正式创建三级节点。
4. 按 [技术定位卡.md](技术定位卡.md) 输出技术 `AGENTS.md` 或文章核心知识点。

每篇或每组文章整理时，必须先回答这 6 个问题：

| 问题 | 目的 |
|---|---|
| 这篇文章讲的是技术、产品、框架、架构模式，还是实践案例？ | 判断文章类型 |
| 它解决的核心问题是什么？ | 防止被标题带偏 |
| 它在全局架构中属于哪一层、哪个模块？ | 做纵向定位 |
| 它对标哪些已有技术或方案？ | 做横向比较 |
| 对用户的价值是什么：现在用、以后关注、仅了解、可忽略？ | 给出使用判断 |
| 它应该进入已有三级节点，还是触发新增三级节点提醒？ | 促使整理者主动细分二级认知领域 |

归类时不要在 `AGENTS.md` 内重新推导分类表，必须查 [目录划分.md](目录划分.md)。

判断内容是否需要用户了解、理解、记住、实践时，不要凭通用标准，必须查 [用户画像.md](用户画像.md)。

整理文章时，先输出“认知校准点”，再输出“待吸收点”。认知校准点用于标出文章观点与用户已有判断框架、价值观、分类准则之间的冲突、偏差、补充或待验证之处；它比普通知识点摘要更重要。

## 认知水平与排重落点

用户认知水平分三层落地：

| 层级 | 文件 | 作用 |
|---|---|---|
| 总画像 | `用户画像.md` | 维护用户跨领域的认知水平、价值观、内容取舍规则 |
| 问题域画像 | 二级类目 `AGENTS.md` | 维护用户在该二级类目下的认知假设、排重准则、已覆盖技术 |
| 单篇判断 | `编号_核心知识点/*.md` | 记录这篇文章相对用户的认知校准点、待吸收点、已知可跳过 |

不同文章是否重复，不能只看标题或技术名。排重时必须生成“问题指纹”：

```text
技术本体 + 所属模块 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

判定规则：

- 如果问题指纹相同，且没有新的认知校准点、边界、反例、实践证据，则不新建文章笔记，只追加到已有笔记的“相关原文”。
- 如果同一主题提供了新的反例、版本差异、工程代价、适用边界或可实践验证，则可以新建或扩展已有笔记。
- `技术 AGENTS.md` 中的“已沉淀核心知识点 / 解决什么问题”是排重入口，不是唯一依据。
- 二级类目 `AGENTS.md` 中的“排重准则”是该问题域的主要排重依据。

## 技术定位卡必须更清楚

技术定位卡不要只复述文章内容，必须区分三件事：

| 层次 | 必须写清楚 |
|---|---|
| 文章内容 | 这篇文章实际讲了什么，讲到哪个模块，没有讲什么 |
| 技术位置 | 这个技术在完整系统架构中的位置，上游、下游、依赖和产出是什么 |
| 使用准则 | 什么场景该用，什么场景不该用，和谁对标，选型时看哪些指标 |

实际输出字段按 [技术定位卡.md](技术定位卡.md) 执行。

## 纵向分析要求

纵向是回答“它在完整技术体系中的位置”。

必须包含：

- 全局架构：这个技术整体由哪些模块组成。
- 本文位置：当前文章只涉及哪个模块，不能把局部说成整体。
- 上下游：输入是什么，输出是什么，依赖谁，被谁消费。
- 核心机制：真正有价值的是算法、存储结构、执行模型、协议、编排方式，还是工程封装。
- 运行链路：接入、配置、运行、排障、监控的大致路径。
- 边界条件：数据量、延迟、并发、团队能力、基础设施要求。

## 横向对比要求

横向是回答“它和谁解决类似问题，差异在哪里”。

对比时必须包含：

| 维度 | 要求 |
|---|---|
| 对标对象 | 同类技术、传统方案、上一代方案、替代方案都要考虑 |
| 解决问题 | 先确认是否真的是同一个问题，不要因为标题相似就比较 |
| 实现方式 | 比较架构路线、数据模型、执行方式、状态管理、存储方式 |
| 优势 | 写清楚优势来自哪里，不写空泛结论 |
| 劣势 | 写清楚代价、限制和不适用场景 |
| 使用判断 | 具体到“什么情况下选 A，什么情况下选 B” |

## 文章命名规则

- 用户可读内容用明确题名，例如 `数仓与Hive.md`、`Flink实时计算.md`。
- 不再把主要内容写成 `README.md`。
- LLM 规则统一写入 `AGENTS.md`，Claude 入口写入 `CLAUDE.md` 并引用 `AGENTS.md`。

修改后应检查本地 Markdown 链接是否仍能落到真实文件。
