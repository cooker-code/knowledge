## 核心问题

模型服务不是“模型能跑起来”这么简单，而是统一接口、路由、限流、密钥、缓存、显存、量化、监控、故障恢复和成本治理的组合。文章若只提供一键部署或项目清单，只能作为入口锚点，不能沉淀为生产可用准则。

## 判断准则

| 模块 | 可复用判断 |
|---|---|
| 推理网关 | 重点看 OpenAI 兼容接口、provider 路由、重试、限流、密钥隔离、账单统计和观测指标。 |
| 本地部署 | 重点看模型格式、量化精度、GPU 架构、KV Cache、显存余量、并发和推理框架版本。 |
| Docker/DMR | 价值是标准化模型分发、运行和 OpenAI API 暴露；边界是性能、模型生态和生产运维能力。 |
| vLLM | 重点看连续批处理、PagedAttention、量化、多 GPU 和服务化接口，不只看启动命令。 |
| 一键 AI 助手 | 如果主问题是桌面助手或工作站体验，不能归为模型服务生产准则。 |

## 认知校准点

| 常见错误认知 | 正确理解 |
|---|---|
| 网关性能对比越高越好 | 没有并发模型、硬件、请求长度、缓存和失败率，倍数指标只作追查线索。 |
| Docker 能运行模型就等于生产部署 | 生产还需要监控、鉴权、灰度、限流、回滚和资源隔离。 |
| Ollama 简单就一定适合生产 | 本地易用和服务化治理是两条评价轴。 |
| 双卡跑通 Qwen 就代表可稳定服务 | 还要看吞吐、长上下文 KV、量化精度、冷启动、OOM 和失败恢复。 |
| AI 助手项目属于模型服务 | 只有涉及模型服务运行形态时才归 0105，否则应路由到工具或 Agent 应用。 |

```mermaid
  E --> F["业务调用"]
```

## 待验证缺口

- Bifrost 与 LiteLLM 需要同请求长度、同模型、同并发下复测，不能直接采信标题倍数。
- Docker Model Runner 与 Ollama 的边界要补官方文档、模型格式支持和生产运维案例。
- Qwen 本地部署要补 vLLM 版本、量化格式、显存峰值、吞吐和长上下文稳定性。
