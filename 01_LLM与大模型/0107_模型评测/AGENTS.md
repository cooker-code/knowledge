## 三级节点入口

暂无三级节点；当前以二级 `0107_核心知识点/` 承接模型评测通用准则。

**建立三级节点的触发条件：**

| 条件 | 说明 |
|---|---|
| 某类评测方法积累 ≥3 篇精读文章 | 如业务 benchmark 构建、自动校验、持续监控各自 3 篇 |
| 能形成可复用的评测流程和验收标准 | 不是只有榜单截图 |
| 与业务场景的选型决策强相关 | 否则保持原文锚点层 |

候选三级节点：`业务评测集`、`自动校验（LLM-as-Judge）`、`持续监控`

## 用户认知重点

| 项 | 内容 |
|---|---|
| 已知基础 | 知道公开 benchmark（MMLU、HumanEval 等），了解单次实测局限性 |
| 待补边界 | 业务场景评测集如何构建；LLM-as-Judge 的适用边界和偏见控制 |
| 易偏差点 | 榜单排名当作业务能力；单次 prompt 当评测结论 |
| 优先抽取 | 评测集设计、Checker 实现、模型版本追踪、持续回归方案 |

## 排重指纹

```text
评测对象 + 评测方法（自动/人工/混合）+ 核心机制 + 解决问题 + 任务域/样本/版本边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是公开 benchmark | 仅留影响选型的能力维度，不重复记榜单数字 |
| 都是 LLM-as-Judge | 区分单模型自判、多模型交叉、Position Bias 控制、一致性验证 |
| 都是评测平台/框架 | 按评测集管理、Checker 扩展、并发运行和成本控制拆分 |
| 没有任务域/基线/版本 | 降权，不新建知识点 |

## 待补缺口

| 优先级 | 项 | 为什么补 |
|---|---|---|
| 高 | 业务评测集构建方法 | 公开 benchmark 无法替代业务场景验证 |
| 高 | LLM-as-Judge 偏见控制 | 高频使用 Claude/GPT 做自动评测，偏见和一致性是关键风险 |
| 中 | 持续监控与版本回归 | 模型版本更新时的性能对比和自动报警 |
