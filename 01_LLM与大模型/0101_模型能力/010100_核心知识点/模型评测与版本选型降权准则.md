## 一句话结论

模型选型不能按“最新、最强、榜单第一”决策，而要按业务任务、版本时效、部署提供方、上下文长度、工具/格式稳定性、成本延迟和可自动验证的业务回放综合判断；公开榜单和单次实测只能做线索，不能替代自己的 Checker。

## 认知校准点

| 校准点 | 文章观点/信息 | 与用户认知或价值观的关系 | 处理建议 |
|---|---|---|---|
| 新版本不必然更好 | GPT-5、Claude 4.6、DeepSeek/Qwen 成本优化版本都可能牺牲部分能力 | 纠偏版本崇拜 | 每次模型升级都按新模型重跑业务评测 |
| 部署方会改变效果 | 官方 API 和三方部署同模型可能表现不同 | 补充选型边界 | 评测记录必须包含 provider、模型字符串、时间、参数 |
| 榜单只做线索 | STUPIDMETER、KingBench、ARC 等榜单任务域有限或未核验 | 符合反标题党价值观 | 不把榜单写成选型结论 |
| 单次实测有价值但不充分 | Opus 4.5 实测能揭示真实任务边界，但样本小 | 补充证据分级 | 把单次实测归为 qualitative case |
| Checker 是长期壁垒 | 格式输出、工具调用、循环调用、端到端任务都可构造 Checker | 强相关用户工程习惯 | 后续模型评测优先沉淀 Checker，而非总结跑分 |

## 冲突点

| 冲突类型 | 具体表现 | 影响 | 处理 |

| 排重冲突 | 多篇 Claude/国产模型实测都在讲“按任务分流” | 重复问题指纹 | 合并为一篇选型准则，不逐篇扩写 |

## 待吸收点

| 分级 | 内容 | 为什么值得吸收 | 后续动作 |
|---|---|---|---|
| 记住 | 模型评测必须包含业务任务、成功标准、自动化校验、成本、延迟、provider、版本 | 直接影响升级决策 | 建一个本地模型评测记录模板 |
| 理解 | 开源评测集适合初筛，但业务回放才能验证工程适配 | 评测不能停在榜单 | 从真实日志/任务中抽样构造评测集 |
| 理解 | 端到端评测要看规划、错误恢复、上下文管理、目标对齐 | 单点能力不代表 Agent 能力 | 对 Agent 任务做过程指标 |
| 记住 | 结构化输出和工具调用是模型差异的关键，不是小功能 | 决定下游系统稳定性 | JSON/XML 解析和工具参数 Checker 必备 |
| 理解 | Claude 适合复杂推理/主力编程，国产模型可按大文档、批量内容、后台任务做分流 | 对标不是二选一 | 按任务建立模型路由策略 |
| 了解 | Qwen 本地部署的关键约束是量化格式、GPU 架构、KV Cache 和推理框架版本 | 影响自部署成本 | 后续归入模型服务或本地部署专题 |

## 纵向理解

| 维度 | 判断 |
|---|---|
| 全局架构 | 候选模型 -> 公开 benchmark 初筛 -> 业务评测集 -> 自动 Checker -> 成本/延迟/provider 对比 -> 灰度 -> 线上监控 |
| 本文位置 | 当前组文章覆盖评测框架和横向任务分流，不覆盖完整平台实现 |
| 核心机制 | 有答案的自动评测、端到端回放、在线 Checker、版本/provider 记录、失败样本闭环 |
| 使用链路 | 每次升级先冻结评测集和基线，再跑候选模型，最后用失败样本更新 Checker |
| 前置条件 | 有代表性业务样本、明确成功标准、日志、成本统计、可复跑环境 |
| 边界 | 创意任务和开放任务需要人工或 LLM judge，但 LLM judge 本身要抽样校验 |
