# LLM 表格理解能力边界

## 来源
- [LLM遇上表格：4类表示、5大任务、3大机会](../文章/done-LLM遇上表格：4类表示、5大任务、3大机会.md)

## 核心问题
LLM 处理表格的真实能力边界在哪里，以及为什么换个序列化格式就能让模型表现大幅下降。

## 判断准则

### 表格四种输入表示法及其取舍

| 表示方式 | 优点 | 缺点 | 典型场景 |
|---|---|---|---|
| Serialization（序列化） | 直接用文本，最简单 | 结构信息易丢失 | WTQ, TabFact |
| Schema only（仅表头） | 省 token | 细节全丢，只适合 Text-to-SQL 规划 | Spider, SEDE |
| Image（截图） | 保留完整视觉结构 | 受分辨率和 OCR 限制 | 视觉模型 VISTABNet |
| Table Encoder（专用编码器） | 结构感知最强 | 需额外预训练 | TableGPT2, TAPAS |

**关键数据**：同一任务把 Markdown 换成 LaTeX，EM 分数最多差 **20%**，说明序列化格式是隐性变量，必须在评测时固定。

### 模型能力断层：复杂表输入时人机差距极大
- 层级多维表（HiTab）：人类 83% → SOTA 模型 <50%
- 多层级表 MULTIHIERTT：同样模型翻车
- 长表、多表联查、跨文档表：是目前模型真正的性能陷阱

### 五大任务分类

| 任务 | 输入 | 输出 | 典型难点 |
|---|---|---|---|
| Table QA | 表格+问题 | 单元格/数字/自由文本 | 层级表和多表推理 |
| Table-to-Text | 表格+高亮区域 | 描述摘要 | 逻辑推断而非简单复述 |
| Fact Verification | 表格+声明 | Supported/Refuted/NEI | 数值比对和逻辑推理 |
| Text-to-SQL | 问题+数据库 schema | SQL 查询 | 意图理解、多表 JOIN、复杂聚合 |
| Leaderboard Construction | 论文表格 | (任务,数据集,指标,分数) 四元组 | 跨论文归一化对齐 |

### Text-to-SQL 的关键局限
- 简单 SQL 效果可接受，联表等复杂查询效果仍有明显差距
- 工程规避方案：减少联表查询，创建宽表（wide table）
- Spider2 引入意图级问题（"给我一份关键销售报告"），模型需推断需要查哪些字段，难度大幅上升

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 换格式无所谓，意思一样 | 序列化格式是隐性变量，Markdown ↔ LaTeX ↔ JSON 间最高差 20 点 EM |
| 模型能处理复杂表 | 现有 SOTA 在多层级表上比人类低 30+ 点，不是小差距 |
| Text-to-SQL 已经够用 | 联表、意图级查询是当前边界；生产环境优先降低联表复杂度 |
| 90% 的 benchmark 覆盖了真实任务 | 大多数 benchmark 只考"检索+简单数学"，需要推理的任务几乎空白 |

## 待验证缺口
- 不同序列化格式对推理型任务（诊断、预测、洞察）的影响尚无系统评测
- 科学论文中消融表+长文本是未经充分测试的难例
- 格式互译任务（让模型在 JSON/Markdown/LaTeX 间转换且不损失语义）作为能力提升路径待探索
