## 一句话结论

长上下文、结构化输出和低 token 成本都不是模型“自动变强”的证据，而是调用链工程问题：上下文越长越需要筛选和隔离，结构化输出越关键越要 schema/解码约束，token 越贵越要评估压缩格式和模型精度/显存/延迟的取舍。

| 校准点 | 文章观点/信息 | 与用户认知或价值观的关系 | 处理建议 |

## 横向对标

| 对标技术 | 实现方式 | 优势 | 劣势 | 适合场景 |
|---|---|---|---|---|
| 全量塞上下文 | 把文档/工具/历史全部放入 prompt | 简单，省检索工程 | 污染、干扰、成本高 | 小规模低风险任务 |
| RAG | 检索相关片段 | 控制上下文大小 | 召回错误会污染答案 | 知识库问答 |
| 修剪/摘要 | 删除或压缩上下文 | 降成本、降干扰 | 可能删掉关键证据 | 长文档、多轮任务 |
| 上下文隔离 | 多 Agent/多线程分治 | 降路径依赖 | 汇总成本和一致性问题 | 可并行研究/搜索 |
| 约束解码 | token 级限制输出合法性 | 格式稳定 | 开销和实现复杂 | API 工具调用、生产接口 |
| TOON/自定义格式 | 压缩重复结构字段 | token 省 | 生态弱、解析风险 | 模型内部输入层 |
