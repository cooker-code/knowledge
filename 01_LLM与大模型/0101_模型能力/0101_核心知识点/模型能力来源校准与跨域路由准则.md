## 核心问题

01 目录只承接模型能力本体、训练、服务、多模态和评测。很多文章虽然标题里有“大模型”“Claude”“Kimi”，正文主问题却是 Agent 编排、AI 编程流程、开源工具套件、时序预测或资料清单。处理这类文章时，不能继续把 01 当泛 AI 收纳池。

## 判断准则

| 文章主问题 | 01 内处理 | 更稳定的长期落点 |
|---|---|---|
| 大小模型分工、强模型按需介入、能力/成本路由 | 只吸收“模型分级路由”准则 | `02_Agent与AI工程` 的 Agent 编排或工具调用 |
| 开源 LLM 套件、私有化问答、产品功能清单 | 只保留“产品清单证据需降权” | RAG、模型服务或具体工具节点 |
| 单一 AI 产品使用教程 | 只保留能力边界字段，不沉淀教程 | 电脑工具或 AI 编程工具 |
| 时序大模型、行业垂直预测模型 | 只保留“非通用 LLM”提醒 | 机器学习/推荐与预测或数据工程场景 |
| 公司流程、工程管理、AI 写代码流程 | 不作为模型能力结论 | `02_Agent与AI工程/0206_AI编码方式` 或个人效率 |
| PDF/资源合集 | 只做来源索引，不生成知识点 | 后续按具体正文重新拆分 |

## 认知校准点

| 校准点 | 处理方式 |
|---|---|
| “模型名”不是分类依据 | 先看正文要解决的主问题；Claude 文章也可能是工程管理、工具调用或 Agent 运行方式。 |
| “开源/商业级/最强”默认降权 | 没有版本、基准、部署规模和失败场景时，不写成模型能力结论。 |
| “产品会用”不等于“模型能力可复用” | Kimi、h2oGPT 等产品文章只在涉及长上下文、私有部署、检索能力边界时进入 01。 |
| 行业模型要看任务本体 | Timer-XL 是时序预测基础模型，不是通用大模型能力；如果后续精读，应转入机器学习预测。 |
| 管理流程文章不进入模型能力 | Anthropic 工程管理文章只说明 AI 改造流程，不说明 Claude 模型本身能力。 |

## 后续处理规则

- 后续 01 文章进入时，先用“模型能力字段”抽取：能力维度、模型版本、上下文、成本、provider、评测样本、失败边界。
- 抽不出上述字段，只能作为来源校准或跨域路由证据，不能新建模型能力知识点。
- 如果文章主要讲 Agent 调用、AI 编程流程、工具运行底盘，直接按 `02_Agent与AI工程` 路由，不在 01 扩写。

## 待验证缺口

- Anthropic Advisor Tool 需要后续补官方 API 文档和真实任务评测，才能判断它是模型服务路由、Agent 编排，还是工具调用能力。
- h2oGPT/Kimi 类产品需要拆出真实架构或评测证据，否则只保留为低权重来源。
- Timer/Timer-XL 后续若继续积累，应从 01 迁到机器学习时序预测节点。
