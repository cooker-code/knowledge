# 你的提示词技巧可能过时了！最新版Claude 提示词工程最佳实践发布！

**上下文工程** (Context engineering) 是 Agent 的核心，而**提示工程** (prompt engineering) 是其基础。

Claude 在最近的博客中发布了最新版的提示词工程实践，博客中不仅有常见的，诸如具体明确清晰的指令、必要的示例信息这些内容，更重要的提供了现代模型与之前模型的差异以及一些很重要的经验（错误处理、测试、整合等），这对于我们当下使用**Claude 新模型**具有重要的指导意义。

本文整理如下。

---

### 给予 Claude 表达不确定性的许可

当无法确定正确答案时，AI 可能会推测并返回错误内容。此时，如果明确**允许 AI 表达不确定性**而非强行猜测，就能有效减少“幻觉”，提升输出的可靠性。

**示例**："请分析这些财务数据并识别其中的趋势。如果数据不足以得出明确结论，请直接说明，不要进行推测。"

### 思维链（CoT）提示

虽然 Claude 目前已经提供了扩展思维功能，可以自动结构化推理，但在解决复杂专业的任务时，手动 CoT 依然很有用，以下是各类思维链的案例：

**1. 基本思维链**：`在写邮件之前逐步思考。`

**2. 引导思维链**：

```
在写邮件之前思考。首先，考虑到他们的捐赠历史，思考什么信息可能吸引这个捐赠者。然后，考虑"关爱儿童"项目的哪些方面会引起他们共鸣。最后，使用你的分析撰写个性化的捐赠者邮件。
```

**3. 结构化思维链**：使用标签将推理与最终答案分开。

```
在 <thinking> 标签中思考邮件。首先，分析什么信息会吸引这个捐赠者。然后，识别相关的项目方面。最后，在 <email> 标签中使用你的分析撰写个性化的捐赠者邮件。
```

### 控制 AI 输出格式

对于现代 AI 模型，有几种有效的方法来控制响应格式：

**1. 告诉 AI 做什么而不是不做什么**

不要说："不要在你的响应中使用 markdown"  
尝试："你的响应应该由流畅的散文段落组成"

**2. 匹配你的提示风格与期望的输出**

提示中使用的格式风格可能影响 AI 的输出风格。如果你想要最少的 markdown，减少提示中的 markdown。

**3. 明确格式偏好**

对于格式的详细控制：

```
在写报告或分析时，使用完整的段落以清晰流畅的散文写作。使用标准的段落换行进行组织。主要将 markdown 用于内联代码、代码块和简单标题。  
  
除非你呈现真正离散的项目，其中列表格式是最佳选择，或者用户明确要求列表，否则不要使用有序列表或无序列表。  
  
不要用项目符号列出项目，而是将它们自然地融入句子中。你的目标是可读的、流畅的文本，自然地引导读者理解思想。
```

### 提示链

提示链是将复杂的任务拆成多个小步骤实现，每个提示处理一个阶段，下一个提示使用上一次阶段的输出。

这种方式类似 Agent 的 LOOP 循环，通常使用编程的方式实现，不过你也可以手动执行。这在**复杂任务、多轮迭代、分步骤分析、中间验证**等场景非常有用。

**示例：研究摘要**

1. 1. **第一个提示**："总结这篇医学论文，涵盖方法论、发现和临床意义。"
2. 2. **第二个提示**："审查上述摘要的准确性、清晰度和完整性。提供分级反馈。"
3. 3. **第三个提示**："基于此反馈改进摘要：[来自步骤 2 的反馈]"

### XML 标签

除特定场景：区分内容边界、复杂的提示词，现代模型已经不需要 XML 标签就能更好地理解结构，同时 Token 还更小。

### 角色

你一定定义过“你是一名 xx 专家”这种提示词，但现代模型足够复杂，生硬的角色提示通常不是必要的。

而且不要过度限制角色。"你是一个有用的助手"通常比"你是一个只使用技术术语且从不犯错的世界知名专家"更好。过于具体的角色可能会限制 AI 的有用性。

**现代替代方案**：明确说明你想要什么视角更有效："分析这个投资组合，专注于风险承受能力和长期增长潜力"而不是分配角色。

## 实践经验

### 整合各种提示词技术

将各类提示词技术根据合适的场景进行组合才能发挥出模型最大的作用。

**组合多种技术的示例**：

```
从这个季度报告中提取关键财务指标并以 JSON 格式呈现。  
  
我需要这些数据进行自动处理，所以你的响应只包含有效的 JSON，不要前言或解释，这一点至关重要。  
  
使用这个结构：  
{  
  "revenue": "带单位的值",  
  "profit_margin": "百分比",  
  "growth_rate": "百分比"  
}  
  
如果任何指标在报告中没有明确说明，使用 null 而不是猜测。  
  
用开括号开始你的响应：{
```

这个提示结合了：

* • 明确的指令（确切要提取什么）
* • 上下文（为什么格式重要）
* • 示例结构（展示格式）
* • 表达不确定性的许可（如果不确定使用 null）
* • 格式控制（以开括号开始）

### 如何选择正确的提示词技术

博客中提供了一个决策思路：

1. 1. 你的请求清晰明确吗？如果不是，首先处理清晰度
2. 2. 任务简单吗？只使用核心技术（具体、清晰、提供上下文）
3. 3. 任务需要特定格式吗？使用示例或预填充
4. 4. 任务复杂吗？考虑分解（链）
5. 5. 需要推理吗？使用扩展思维（如果可用）或思维链

### 从简单的提示词开始

从简单开始，只在必要时才使用更加复杂的提示词，并且每次增加都要进行测试，以确定是否复杂性带来了真正的改善。

没有所谓的提示词模板，一切都要在尝试和测试之后定论。

### 避免常见错误

* • **不要过度设计**：更长、更复杂的提示并不总是更好。
* • **不要忽略基础**：如果你的核心提示不清晰或模糊，高级技术不会有帮助。
* • **不要假设 AI 会读心**：具体说明你想要什么。留出模糊给 AI 误解的空间。
* • **不要一次使用每个技术**：选择解决你特定挑战的技术。
* • **不要忘记迭代**：第一个提示很少完美工作。测试和改进。
* • **不要依赖过时的技术**：XML 标签和生硬的角色提示对于现代模型来说不太必要。从明确、清晰的指令开始。

### 处理复杂任务

对于复杂任务，上下文的管理本身充满挑战。虽然现代模型已经显著的改进了上下文感知能力，但任务拆分仍然非常重要。

在处理长上下文时，在**开头或结尾**构建关键细节，同时拆分任务，这能让模型更好的聚焦任务和专注的解决问题。

---

最后，和人与人的交流一样，提示词最重要的就是**沟通**。

最好的提示不是最长或最复杂的，而是使用最小必要结构可靠的实现你目标的那个。
