## 核心问题

语音多模态文章要区分三层：模型能力、交互系统和普通 TTS 工具。只有能改变语音模型选型、实时交互、长音频处理或私有部署判断的内容，才进入 0106；简单 Python 播报教程只作为低权重工具锚点。

## 判断准则

| 判断项 | 可复用准则 |
|---|---|
| ASR/长音频 | 看最长输入、说话人分离、时间戳、热词、流式处理和多语言边界。 |
| TTS | 看音色一致性、情感控制、时长控制、延迟、版权和滥用防护。 |
| 全双工语音交互 | 看打断、端点检测、对话状态、RTC 延迟、模块化部署和隐私边界。 |
| 私有化部署 | 看是否依赖外部 API、硬件要求、模型权重、实时性和数据合规。 |
| 普通 TTS 库 | 只说明离线播报能力，不代表大模型语音能力。 |

## 认知校准点

| 常见错误认知 | 正确理解 |
|---|---|
| 会说话就是语音大模型 | 普通 TTS、情感 TTS、全双工语音 Agent 是不同层级。 |
| 开源就能私有化生产 | 还要确认权重、延迟、硬件、并发、协议、日志和安全策略。 |
| 情感自然就是可信 | 越自然越需要欺诈防范、水印、授权和人工披露机制。 |
| 长音频能力只看分钟数 | 还要看说话人跟踪、错字率、上下文保持和流式稳定性。 |

## 语音交互链路

```mermaid
flowchart LR
  A["麦克风/音频流"] --> B["VAD/EoT"]
  B --> C["ASR/说话人跟踪"]
  C --> D["对话管理"]
  D --> E["LLM/工具调用"]
  E --> F["TTS/情感控制"]
  F --> G["RTC 播放与打断"]
```

## 待验证缺口

- VibeVoice 需要补官方技术报告、模型许可、TTS 代码移除原因和真实延迟数据。
- FireRedChat 需要补架构图、LiveKit 延迟、EoT/VAD 误判样本和私有化部署门槛。
- IndexTTS2.0 需要补时长控制的可复验样本和情感控制失败模式。
