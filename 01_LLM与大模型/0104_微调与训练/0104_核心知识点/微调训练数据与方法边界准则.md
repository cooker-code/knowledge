## 核心问题

微调不是“把资料丢给模型学会”。它解决的是行为分布、输出风格、领域术语和任务格式的稳定性问题；如果问题本质是事实更新、私有知识问答或少量上下文补充，优先考虑 RAG、长上下文或提示词，而不是微调。

## 判断准则

| 判断项 | 可复用准则 |
|---|---|
| 什么时候微调 | 当任务需要稳定输出格式、领域风格、专有步骤或偏好行为，且样本可以批量构造和验证时。 |
| 什么时候不微调 | 私有事实频繁变化、答案必须可追溯、数据量很少、没有评测集时，优先 RAG/Prompt。 |
| 数据质量 | 数据要有任务指令、输入、期望输出、反例和拒答边界；只做文档切片生成问答会放大噪声。 |
| 方法选择 | 低资源先看 LoRA/QLoRA；全参数微调只有在算力、数据、评测和部署链路都稳定时才考虑。 |
| 工具框架 | LLaMA Factory、EasyDataset 只能降低上手门槛，不能替代数据定义、样本审查和评测闭环。 |

## 认知校准点

| 常见错误认知 | 正确理解 |
|---|---|
| 本地知识库等于训练模型 | 知识库是检索供证；微调是改变模型行为和输出分布。 |
| 微调教程能直接迁移到企业内部数据 | 企业数据需要脱敏、分层、版权/权限检查、样本审查和失败样本回流。 |
| 工具支持某模型就代表效果好 | 工具链支持只说明能跑，效果取决于数据、超参、基线和评测。 |
| 垂直领域模型越大越好 | 领域模型要按任务覆盖、推理成本、稳定性和可维护性权衡。 |

## 微调流程骨架

```mermaid

  A["明确任务边界"] --> B["构造训练样本"]
  B --> C["样本审查与脱敏"]
  C --> D["选择微调方法"]
  D --> E["训练与验证"]
  E --> F["业务评测集回放"]
  F --> G["部署与失败样本回流"]
```

## 待验证缺口

- LoRA/QLoRA 的 rank、target modules、量化格式和推理延迟需要后续用真实小样本实验补证。
- EasyDataset 自动生成样本的质量需要抽样审查，不能把生成样本直接当训练真值。
- DeepSeek-R1 领域微调文章偏教程，后续要补可复现环境、评测集和失败案例。
