# 用户画像与认知水平

本文件是 4 个基础文件之一，只维护用户认知假设、内容取舍和认知校准规则。基础文件包括：根 `AGENTS.md`、`用户画像.md`、`目录划分.md`、`技术定位卡.md`。

## 作用

本文件用于帮助 LLM 在整理文章时判断：

- 用户当前大概理解到哪一层。
- 哪些内容只需略过，哪些内容需要补全。
- 哪些知识点应该让用户“了解、理解、记住、实践”。
- 哪些文章内容对用户是新增认知，哪些只是重复信息。
- 文章观点与用户已有认知、判断准则或价值偏好是否冲突。

注意：画像是工作假设，不是固定标签。所有判断必须基于用户表达、已有知识库内容和用户反馈持续修正。

本文件不负责维护目录结构、分类表或输出模板：

- 目录层级、一级/二级/三级新增准入，以根 `AGENTS.md` 和 `目录划分.md` 为准。
- 文章、流程总览、二级 `AGENTS.md`、三级 `AGENTS.md` 的输出字段，以 `技术定位卡.md` 为准。
- 本文件只回答“对用户来说是否新增、是否值得读、应该了解/理解/记住/实践到什么程度”。

## 当前画像假设

| 维度 | 判断 | 置信度 | 依据 |
|---|---|---:|---|
| 技术方向 | 数据工程、数仓、OLAP、Agent 工程、LLM 工程化 | 中 | 用户持续围绕 Hive、数仓、OLAP、Agent、Claude、Airflow 等分类边界讨论 |
| 阅读目标 | 不只是收藏文章，而是形成可复用知识结构 | 高 | 用户要求纵向、横向、技术定位卡、目录划分和核心知识点 |
| 认知偏好 | 关注准则、边界、对标、架构位置、可复用判断 | 高 | 用户多次纠正“按目录/工具名归类”的问题 |
| 当前水平 | 不是入门型读者，更接近有实践经验但需要体系化抽象的工程读者 | 中 | 能指出 Airflow 归类问题、要求区分数据工程与 OLAP、Agent 与 LLM |
| 主要痛点 | 读文章时抓不住特点，不知道是否需要，不知道横向对标和纵向位置 | 高 | 用户明确描述过痛点 |
| 输出偏好 | 表格、架构图、边界判断、核心结论、追查锚点 | 高 | 用户多次要求表格化、纵横对比、锚点 |

## 认知价值观

这些不是情绪偏好，而是用户判断文章价值的标准。整理文章时，必须用这些标准审视原文观点。

| 价值观 | 含义 | 整理时的判断 |
|---|---|---|
| 反标题党 | 不接受只讲“很强、很快、很香”的文章 | 遇到宣传话术要降权，要求证据和适用边界 |
| 重系统位置 | 技术必须放进完整架构里理解 | 只讲局部功能但不讲上下游时，要标为缺口 |
| 重横向对标 | 技术必须知道替代方案和边界 | 没有对标对象的“最佳实践”不直接采信 |
| 重工程落地 | 关注部署、运维、成本、稳定性、权限、治理 | 只讲概念不讲代价时，要标出风险 |
| 重长期沉淀 | 文章价值在于能否形成判断准则 | 只提供资讯但不能沉淀规则的内容只需了解 |
| 反泛泛总结 | 不要把已知常识重新包装成知识点 | 用户已知内容要标出并压缩，不重复消耗注意力 |

## 认知校准优先级

整理每篇文章时，优先级是：

```text
认知校准点 > 待吸收点 > 已知可跳过 > 背景信息
```

| 术语 | 含义 | 例子 |
|---|---|---|
| 认知校准点 | 文章可能纠正、挑战、补充或污染用户已有判断框架的地方 | 文章把 Airflow 说成机器学习核心技术，这与“技术本体优先”的准则冲突 |
| 待吸收点 | 用户尚未稳定掌握，但值得进入长期知识库的机制、边界、对标或方法 | Hive Metastore 的 ObjectStore 与 Direct SQL 差异 |
| 已知可跳过 | 用户大概率已知道，重复讲会浪费注意力的内容 | Hive 是离线数仓工具、Airflow 是调度工具 |
| 背景信息 | 仅帮助理解上下文，但不值得沉淀为核心知识的内容 | 发布会背景、公司案例铺垫 |

认知校准点优先于待吸收点，因为用户真正需要的是判断框架变得更准，而不是收集更多碎片知识。

## 认知水平分层

整理文章时，不要只判断“懂/不懂”，要判断用户在该技术上的层级。认知 level 针对具体技术独立评估，不代表用户整体能力。

| 层级 | 判断标准 | 输出策略 |
|---|---|---|
| L1 名词识别 | 知道名词和大概场景，但说不清解决什么核心问题 | 先解释技术本体、输入输出和典型场景，不展开复杂实现 |
| L2 能使用 | 能照教程使用、配置或读懂案例，但模块边界和失败场景不稳定 | 补全架构位置、核心模块、上下游、常见坑 |
| L3 能判断 | 能说清适用/不适用场景，能做基础横向对标 | 强化对标、选型边界、优劣势、替代方案和反例 |
| L4 能设计与排障 | 能放进真实系统设计，考虑容量、成本、稳定性、治理、运维和排障路径 | 提供架构权衡、风险清单、验证指标、故障定位链路 |
| L5 能沉淀与迁移 | 能抽象成准则，迁移到相邻技术，并指导后续排重或实践 | 提炼原则、反例、复用条件，减少基础解释，要求沉淀成规则 |

### 认知提升判定

认知 level 不能因为“读过一篇文章”就提升，必须有可验证证据。

| 提升 | 判定证据 | 不足以提升的情况 |
|---|---|---|
| L1 -> L2 | 能独立跑通最小例子，或说清输入、输出、核心配置和常见使用场景 | 只知道名词、只收藏教程 |
| L2 -> L3 | 能给出横向对标、适用边界和至少一个不适用场景 | 只会照着案例改参数 |
| L3 -> L4 | 能为真实场景设计方案，并列出失败模式、排障路径和验证指标 | 只会说“看数据量/看场景”但没有具体判断维度 |
| L4 -> L5 | 能沉淀成可复用准则，并在 2 个以上不同文章、项目或问题中复用 | 单次经验、单篇文章结论、没有反例 |

### 认知成熟度

level 说明“用户掌握到哪一层”，成熟度说明“这个判断有多可信”。

| 成熟度 | 含义 | 处理 |
|---|---|---|
| draft | 新提取或单一来源判断 | 可以用于整理文章，但必须标记为工作假设 |
| verified | 被至少 1 篇高质量文章、一次实践或一次用户反馈验证 | 可以作为后续文章排重依据 |
| proven | 在 2 个以上不同场景复用，且没有被用户纠正 | 可以沉淀为稳定准则 |

### 认知确认闭环

认知画像的目标不是给用户贴标签，而是让每篇文章都能帮助确认、纠偏或提升用户的判断框架。

```text
初始 level 假设
  -> 阅读文章时提取认知校准点
  -> 判断待吸收点是否补足 level 缺口
  -> 形成可验证动作或判断题
  -> 用户反馈 / 实践验证 / 多篇文章复用
  -> 更新 level 或成熟度
```

每次整理文章时，必须判断它对当前技术 level 的作用：

| 作用 | 判断方式 | 输出 |
|---|---|---|
| 确认 | 文章支持现有判断，且有更好的证据或案例 | 提升成熟度，不一定提升 level |
| 纠偏 | 文章指出现有分类、边界、对标、实践方式有误 | 写入认知校准点，必要时下调 level 或修改目录规则 |
| 补缺 | 文章补上用户不稳定的模块、机制、边界或排障路径 | 写入待吸收点，可能推动 level 提升 |
| 扩展 | 文章引入相邻技术、替代方案或迁移场景 | 写入横向对标或后续追查 |
| 干扰 | 文章标题党、证据弱、分类混乱或重复已知 | 降权、跳过或只追加原文 |

### level 更新规则

更新认知 level 时必须保留证据，不能只写“感觉更懂了”。

| 字段 | 要求 |
|---|---|
| 技术栈 | 精确到具体技术或问题域，例如 `Flink Join`、`Paimon 合并引擎` |
| 原 level | 更新前的判断 |
| 新 level | 更新后的判断 |
| 成熟度 | draft / verified / proven |
| 证据 | 文章、实践、用户反馈、排障记录或可复用准则 |
| 缺口 | 还缺什么才能进入下一层 |
| 下次验证 | 下一篇文章或下一次实践要验证什么 |

如果证据不足，只能更新成熟度或补充缺口，不能提升 level。

### level 提升检查题

整理文章后，可以用这些问题判断是否真的提升：

| 目标 level | 必须能回答 |
|---|---|
| L2 | 这个技术的输入、输出、核心配置、最小运行链路是什么 |
| L3 | 它和 2 个相邻方案相比，什么场景该选它，什么场景不该选 |
| L4 | 真实生产场景中它会在哪些地方失败，怎么监控、排障、验证 |
| L5 | 能否沉淀成一条可复用准则，并指导下一篇相似文章的排重或判断 |

### 文章到认知提升的映射

| 文章类型 | 通常能推动的 level | 注意事项 |
|---|---|---|
| 入门教程 | L1 -> L2 | 对用户已熟领域多半是已知可跳过 |
| 架构/原理 | L2 -> L3 | 必须有模块边界和横向对标，否则只是背景知识 |
| 生产实践/排障 | L3 -> L4 | 必须有环境、问题、证据、定位链路和结果 |
| 对比/选型 | L2 -> L3 或 L4 | 必须确认比较的是同一个问题 |
| 复盘/方法论 | L4 -> L5 | 必须能抽象成准则，并被多个场景复用 |
| 产品发布/资讯 | 通常不提升 level | 只作为了解，除非改变技术边界或生态判断 |

## 文章内容提取分级

LLM 整理文章时必须按下面分级判断内容，不要把所有信息平铺。

| 分级 | 含义 | 应如何处理 |
|---|---|---|
| 忽略 | 标题党、营销话术、重复常识、无证据结论 | 不进入核心知识点 |
| 了解 | 新名词、新产品、新生态动态，但暂时不影响判断 | 放入“了解即可” |
| 理解 | 能补全技术位置、模块、机制、边界 | 放入核心知识点 |
| 记住 | 会反复影响选型、排障、建模、架构判断 | 用明确短句沉淀 |
| 实践 | 值得后续实验、验证、脚本化或项目落地 | 放入后续追查或待实验 |

## 文章观点偏差判断

文章观点和用户认知价值观可能发生冲突。整理时要单独标出，不要混在普通摘要里。

| 偏差类型 | 判断方式 | 输出方式 |
|---|---|---|
| 夸大收益 | 只给性能数字、效果提升，但没有上下文、基线、代价 | 标为“证据不足” |
| 混淆分类 | 把工具按使用场景误归类，而不是按技术本体和文章主问题归类 | 标为“分类偏差” |
| 只讲优点 | 不讲失败场景、运维成本、迁移成本、权限风险 | 标为“边界缺失” |
| 用案例代替准则 | 只讲某公司实践，不说明适用条件 | 标为“不可直接迁移” |
| 新瓶装旧酒 | 把用户已知常识包装成新概念 | 标为“已知可跳过” |
| 和用户目标不匹配 | 文章热闹但不能帮助判断是否需要、怎么对标 | 标为“低沉淀价值” |

输出顺序固定为：

1. 认知校准点：文章哪些观点需要纠偏、补边界、降权或重新理解。
2. 待吸收点：哪些机制、对标、模块、方法值得用户吸收。
3. 已知可跳过：哪些内容用户大概率知道。
4. 阅读投入建议：跳过、略读、精读、实践。

## 针对用户的提取重点

优先提取：

- 技术本体：它到底解决什么问题。
- 架构位置：它在完整系统中属于哪一层。
- 模块划分：它由哪些核心模块组成。
- 横向对标：它和谁解决相似问题，差异在哪里。
- 适用边界：什么场景该用，什么场景不该用。
- 工程风险：性能、稳定性、运维、权限、治理、迁移成本。
- 可追查锚点：本地文件、原文链接、关键段落、关键图。

降低权重：

- 纯宣传口号。
- 没有上下文的性能数字。
- 没有对标对象的“最佳实践”。
- 缺少失败场景的产品介绍。
- 和用户当前知识结构无关的泛泛教程。

## 输出决策规则

| 用户认知状态 | 输出应侧重 |
|---|---|
| 用户可能已经知道 | 不重复讲基础定义，直接讲差异、边界、反例 |
| 用户可能知道名词但不熟系统位置 | 补架构图、上下游、核心模块 |
| 用户可能缺横向对比 | 必须给对标表 |
| 用户可能要未来复用 | 提炼成短规则和判断准则 |
| 用户可能要动手实践 | 给实验方向、验证指标、风险点 |

## 画像落点

用户认知水平不是只写在一个地方，而是按粒度分层使用。

| 落点 | 文件 | 用途 |
|---|---|---|
| 总体认知画像 | `用户画像.md` | 判断用户跨领域的能力假设和价值观 |
| 二级类目认知画像 | `knowledge/<一级类目>/<二级类目>/AGENTS.md` | 判断用户在某个问题域下已经知道什么、缺什么、容易被什么误导 |
| 三级技术/主题知识地图 | `knowledge/<一级类目>/<二级类目>/<三级技术或主题节点>/AGENTS.md` | 维护该技术或稳定专题的纵向结构、对标对象和已沉淀问题 |
| 单篇文章判断 | `编号_核心知识点/` 下的主题 `.md` 文件；不使用 `编号_核心知识点.md` 单文件 | 记录这篇文章相对用户的认知校准点、待吸收点和已知可跳过 |

## 排重与认知增量

不同文章是否值得单独沉淀，不按标题、作者、发布时间判断，而按“问题指纹”和“认知增量”判断。

问题指纹：

```text
技术本体 + 所属模块 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

| 情况 | 处理 |
|---|---|
| 同一技术、同一模块、同一机制，只是换了说法 | 不新建，标为已知可跳过或追加原文 |
| 同一主题但补充了边界、失败场景、版本差异 | 合并到已有主题，增加认知校准点 |
| 同一主题但提供了可实践验证路径 | 可以新建实践型核心知识点，或扩展已有主题 |
| 标题不同但问题指纹相同 | 视为重复，不被标题带偏 |
| 标题相同但解决问题不同 | 可以分开沉淀 |

## 画像更新规则

当出现以下情况时，需要更新本文件：

- 用户明确纠正分类或判断。
- 用户反复追问某类边界问题。
- 用户表现出某领域比当前假设更熟或更不熟。
- 用户要求“沉淀画像”“记住我的偏好”“调整认知水平”。

更新时必须保留：

- 新判断。
- 证据。
- 置信度。
- 对后续文章整理策略的影响。
