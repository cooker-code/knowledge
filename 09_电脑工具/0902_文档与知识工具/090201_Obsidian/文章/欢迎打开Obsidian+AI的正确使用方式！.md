我之前写过一篇 [关于 Obsidian + Claude Code 的分享](https://mp.weixin.qq.com/s?__biz=Mzk2NDU0NDczMg==&mid=2247492488&idx=1&sn=3dcc5caf6dec1dc4a1ca276577d3920d&scene=21#wechat_redirect)，里面其实分享了我的Ob+AI演化路径：最开始我是把 AI 当成笔记增强工具，只围绕单篇内容做生成和交流；后来随着支持整库检索和对话的方案出现，AI 开始能够围绕整个 Obsidian 仓库工作；再后来随着 Claude Code 这样的代理式工具出现，AI 不再只是 `能回答`，而是开始 `能操作`，可以直接参与到笔记整理、批量编辑、结构调整、元数据维护和工作流执行中。

所以当我读 Karpathy 的帖子时，最大的感受不是 `这方法真新`，而是 `原来这件事可以被总结成一个这么清晰的模式`。过去我更多是在实践层面不断试、不断调；而他做的，是把这条路抽象成一个统一框架：原始资料进入系统，LLM 在规则约束下持续维护 wiki，而 Obsidian 作为最自然的知识工作台来承载这一切。

## 02 Karpathy 到底提出了什么

Karpathy 在 X 上那条帖子的核心想法，其实可以概括成一句话：**不要只把 LLM 当成一个临时问答工具，而要把它变成一个持续维护个人知识库的协作者。**

> 另外，对于 AI 与我们的关系，我一直认为 AI 是我们的得力助手，是伙伴关系，是协作关系，是互为导师与学生的关系，Ta 是在辅助我们高效解决各种问题/需求。AI 不应该是我们的🐮🐴。

我们现在最常见的 LLM 文档使用方式，其实大家都很熟悉：传一批文件进去，需要的时候再问，模型再去检索相关内容，最后组织成一个回答，这其实就是RAG（例如NotebookLM、ima知识库等）。

这种方式当然能解决很多问题，我自己也一直在用，未来肯定还会用🤣。但用久了你会慢慢发现，它更像是「每次都重新来一遍」。你今天问一个问题，模型会去找材料、拼答案；你明天换个角度再问，它大概率还是要重新找、重新拼，而且前面的理解并不会很自然地沉淀为系统的一部分。

Karpathy 提出的这个想法，其实正是这件事的下一步：LLM 不只是一个提问时出现的回答器，它会围绕原始资料，增量构建并持续维护一套持久存在的 wiki。新资料进来之后，模型会去读它、提炼重点、更新已有页面、补交叉链接、修订旧总结，必要时还会把新旧信息之间的冲突标出来。

也就是说，知识不是每次查询时重新推导一次，而是被编译进 wiki 里，然后持续维护下去。

关键：**wiki 是一个持久存在、会不断复利的产物。**

因为一旦这件事成立，很多东西就变了：

* 交叉引用不需要每次重新发现，前后矛盾可以提前被标出来；
* 一些跨多篇资料才能得出的综合判断，也不用每次提问都重新拼装；
* 你读过的内容、问过的问题、做过的比较、形成过的分析，都可以继续沉淀回这套系统里，让它越来越丰富。

## 03 三层架构

我觉得还是很有必要说说 LLM Wiki 的三层结构：Raw sources、The wiki、The schema。

第一层是Raw sources：原始资料层，文章、论文、图片、网页剪藏、会议记录、项目材料，都放在这里。它们是事实来源，LLM 只读不改。

第二层是The wiki：这里是一组由 LLM 生成并维护的 Markdown 页面，包括摘要页、主题页、实体页、对比页、综合页。你主要负责阅读、判断和提问，LLM 主要负责编写、更新和保持一致性。

第三层是 The schema：Karpathy 在文档里把它定义成一个约束 LLM 行为的说明文件，比如 `CLAUDE.md` 或 `AGENTS.md`，用来规定 wiki 的结构、命名方式、工作流、更新方式和维护规则。

对于第三层，其实我最近就在做这件事。前段时间，我还在 ob中文论坛上写了一个帖子[3]，但是由于各种原因，这个帖子目前只写了个开头🤣，争取抽空将它补完。

这里面就是在介绍我改进后的 PRKA 和正在实践的 Ob+Claude code 方案。你如果看我现在实际在用的 `CLAUDE.md`，就会发现它已经不是一段普通提示词，而是一份真正意义上的系统宪法：它规定了 Claude Code 在我这个 Obsidian vault 里工作的身份、语言、语气、环境假设、目录结构、模板映射、命名规则、链接逻辑、整理流程和维护命令。

> 这里还是一个不断探索的方案，完整的 CLAUDE.md 这里就先不分享了

我还规定了完整的目录结构：`@Inbox` 负责临时入口，`100-Projects` 负责活跃项目，`300-Knowledge` 负责长期知识，`304-AI_Lab` 负责 AI 资产，`305-Toolbox` 负责工具技能等。

此外，我还定义了 `/extract`、`/organize`、`/draft`、`/weekly`、`/polish` 等这类命令，用来做知识提取、收件箱整理、文章起草、周回顾和文稿润色。

这其实和 LLM Wiki 里说的 schema，本质上是同一件事。

## 04 让系统运行起来

看了前面的介绍，感觉讲得头头是道的，问题来了：

该如何让这套AI wiki运行起来呢？

这里Karpathy分享了3个操作和2个文件，下面我来一一介绍一下，希望未来能够深入实践一下：

### 三个操作

**第一个是Ingest（摄入）**

新资料进入后，LLM 会读它、写摘要、更新相关页面、更新索引，并把这次动作记录到日志里。单个来源可能涉及 10-15 个维基页面。Karpathy喜欢一次摄取一个来源并**保持参与——阅读摘要，检查更新，并指导 LLM 强调哪些内容**（这个参与还是很有必要的，如果连这个参与都省了，那你的wiki将很难控制）。但你也可以批量摄取多个来源，减少监督。你可以根据自己的风格开发工作流程，并在架构中记录下来，以供将来使用

**第二个是Query（提问）**

LLM 会先基于现有 wiki 回答问题，而那些真正有价值的回答，比如对比表、分析页、某次新的连接发现，还可以继续写回 wiki。

你针对维基提出疑问。LLM 会搜索相关页面，阅读它们，并综合出一个带有引用的答案。答案的形式可以根据问题的不同而有所变化——可能是一份 Markdown 页面、一张对比表格、一个幻灯片演示（Marp）、一张图表（matplotlib）或一个画布。

重要的洞察是：好的答案可以作为新页面存档回wiki。你提出的比较、分析、发现的关联——这些都有价值，不应消失在聊天历史中。这样，你的探索就能像摄入的资料来源一样，在知识库中不断积累

**第三个是Lint（巡检）**

定期请 LLM 对维基进行健康检查。

关注点包括：页面间的矛盾、已被新资料取代的过时主张、没有入站链接的孤立页面、被提及但缺乏独立页面的重要概念、缺失的交叉引用，以及可以通过网络搜索填补的数据空白。

LLM 善于提出可以研究的新问题和可以寻找的新资料来源。这有助于在维基不断增长的过程中保持其健康状态。

除此之外，Karpathy 还专门提到了两个关键文件：**index.md** 和 **log.md**。

* index.md 负责内容导航。它是整个知识库的目录——每个页面都包含链接、一行摘要，以及可选的元数据（如日期或来源数量）。按类别（实体、概念、来源等）组织。LLM 会在每次摄取内容时更新它。当回答查询时，LLM 会先读取索引以找到相关页面，然后深入查看。这种方法在中等规模（约 100 个来源、数百个页面）下效果出奇地好，并且避免了基于嵌入的 RAG 基础设施需求。
* log.md 是按时间顺序记录的。负责记录 ingest、query 和 lint 这些动作，让整套知识库有一条清晰的演化时间线。一个实用技巧：如果每个条目都以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title` ），那么日志就可以用简单的 Unix 工具进行解析—— `grep "^## \[" log.md | tail -5` 可以获取最后 5 个条目。日志提供了知识库演变的时间线，并帮助 LLM 了解最近已完成的工作。

## 05 一些工具与技巧

### Cli工具

在某些时候，你可能希望构建一些小型工具，以帮助 LLM 更高效地操作维基。  
一个针对维基页面的搜索引擎是最显而易见的选择——在规模较小时，索引文件就足够了，但随着维基的增长，你会想要一个合适的搜索功能。qmd 是一个不错的选择：它是一个针对 Markdown 文件的本地搜索引擎，采用混合 BM25/向量搜索和 LLM 重新排序，全部在设备上完成。它既提供了 CLI（这样 LLM 可以通过 Shell 调用它），也提供了 MCP 服务器（这样 LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的东西——当需要时，LLM 可以帮助你即兴编写一个简单的搜索脚本。

> 根据自己的需要选择工具

### 文章剪藏

主要采用Obsidian Web Clipper ，这也就是obsidian官方开发的一个浏览器扩展，能将网页文章转换为 Markdown。对于快速将来源资料导入你的原始格式收藏库非常有用。

> 我自己也一直在用 [这次齐活了，Obsidian官方剪藏利器来了！](https://mp.weixin.qq.com/s?__biz=Mzk2NDU0NDczMg==&mid=2247492345&idx=1&sn=01fbb4466e5ad92b76cbd0beeda72390&scene=21#wechat_redirect)

### 将图片下载到本地。

可以在 Obsidian 设置 → 文件与链接中，将"附件文件夹路径"设置为一个固定目录（例如 `raw/assets/` ）。

然后在设置 → 热键中，搜索"Download"以找到"为当前文件下载附件"并为其绑定一个热键（例如 Ctrl+Shift+D）。

剪辑文章后，按下热键，所有图片就会下载到本地硬盘。这是可选的，但很有用——它让 LLM 能够直接查看和引用图片，而不是依赖可能失效的 URL。

需要注意的是，LLM 无法原生地一次性读取包含内嵌图片的 Markdown 文件——变通方法是先让 LLM 读取文本，然后再单独查看部分或全部被引用的图片以获得额外上下文。这有点笨拙，但效果足够好。

### Obsidian图谱

> 终于能将图谱用起来了？！  我其实一直在使用局部图谱

Obsidian 的图谱视图是查看维基结构的最佳方式——可以看到哪些内容相互关联、哪些页面是枢纽、哪些页面是孤岛。

### Marp

Marp是一种基于 Markdown 的幻灯片格式。Obsidian 有一个对应的插件。对于直接从维基内容生成演示文稿非常有用。

> 这个插件我分享过：[这次你真的可以用Obsidian做PPT了，配置极简，效果惊艳！](https://mp.weixin.qq.com/s?__biz=Mzk2NDU0NDczMg==&mid=2247492164&idx=1&sn=4c29b06b3560fbbcc96da6bbba92c89e&scene=21#wechat_redirect)

### Dataview

> 突然觉得在AI时代Dataview又可以重度使用了，需要什么样的查询代码，直接让AI撰写即可

Dataview 是 Obsidian 的一个插件，用于对页面前置元数据进行查询。如果你的 LLM 为维基页面添加了 YAML 前置元数据（如标签、日期、来源数量等），Dataview 可以生成动态表格和列表。

## 06 写在最后

其实，对我以及很多小伙伴来说，选择Obsidian的原因之一应该是它的双链，我们可以通过双链将笔记链接起来，让整个笔记库不只是资料堆放处，而是一个会逐渐长出结构的知识网络。

那么现在有了AI，我们正在进入下一个阶段：

也就是在这套已经存在的结构之上，让 AI 真正参与进来，去做那些最重复、最琐碎、最容易漏掉的维护工作，比如补链接、更新主题页、整理新旧笔记之间的关系、生成索引页、记录关键操作、做周期性巡检。

如果这条路继续往前走，那么以后我们维护的，可能就不再只是一个 Obsidian 笔记库，而是一套由人和 AI 协作维护、持续生长的个人知识系统。

简而言之，人继续负责判断方向、挑选来源、决定哪些关系真的重要；AI 则去做整理、补链、更新、巡检和沉淀。
