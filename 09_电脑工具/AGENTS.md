# 电脑工具

## 模块定位

本模块放个人电脑、开发环境和知识生产环境中的工具型对象。重点不是“软件新闻”，而是工具如何进入工作流、如何配置、如何自动化、如何验证输出，以及什么时候值得长期保留。

| 二级目录 | 职责 | 具体工具入口 |
|---|---|---|
| [0901_开发工具与CLI](0901_开发工具与CLI/AGENTS.md) | 命令行工具、开发环境工具、构建发布工具、浏览器自动化工具、终端工作流 | [uv](0901_开发工具与CLI/090101_uv/AGENTS.md)、[OpenCLI](0901_开发工具与CLI/090102_OpenCLI/AGENTS.md)、[npm](0901_开发工具与CLI/090103_npm/AGENTS.md)、[Git](0901_开发工具与CLI/090104_Git/AGENTS.md)、后续可抽 Maven、Playwright CLI |
| [0902_文档与知识工具](0902_文档与知识工具/AGENTS.md) | Markdown、知识库、文档生成、图表绘制、知识入库和内容排版工具 | [Obsidian](0902_文档与知识工具/090201_Obsidian/AGENTS.md)、Mermaid、Quarkdown、Doc2Wiki 等候选 |

## 与其他一级类目的边界

| 文章主问题 | 应归类 |
|---|---|
| 工具如何安装、配置、调用、自动化、接入本机工作流 | 电脑工具 |
| 工具背后的模型调用、Agent 工作流、上下文工程、评估机制 | Agent 与 AI 工程 |
| 工具背后的系统架构、接口契约、并发、测试、发布、可观测性 | 工程与架构 |
| 工具服务的是数据开发、OLAP 查询、指标分析，且主问题是数据链路 | 数据工程、OLAP 或数据分析目录 |
| 只是软件清单、版本资讯、营销推荐 | 略读或低置信记录，不直接沉淀，也不创建临时目录 |

## 具体工具抽离规则

满足任一条件时，可以从二级目录中抽离为具体工具目录：

| 条件 | 说明 |
|---|---|
| 文章数量足够 | 同一工具已有 3 篇以上原文锚点 |
| 工作流稳定 | 能形成安装、配置、使用、自动化、排障、迁移的闭环 |
| 用户长期使用 | 与用户本机知识库、开发环境或日常工作流强相关 |
| 认知增量明确 | 能形成选型边界、失败模式、替代方案或可验证动作 |

当前已抽离：

- `uv`：从开发工具与 CLI 文章中独立出来，承载 Python 包管理、项目管理、解释器管理、脚本运行、构建发布和迁移排障工作流。
- `OpenCLI`：从开发工具与 CLI 文章中独立出来，承载网站、浏览器登录态、桌面应用和本地工具的 CLI 化接口层。
- `npm`：按用户指定规划为开发工具与 CLI 的具体工具入口，先承载 Node 包管理、全局 CLI 安装、脚本执行、registry 和发布边界；当前尚未迁入专门原文。
- `Git`：从开发工具与 CLI 文章中独立出来，承载本地版本控制、worktree、gh、提交规范、GitButler 和回退恢复工作流。
- `Obsidian`：从开发工具与文档工具文章中独立出来，承载 Obsidian CLI、Web Clipper、插件、Mermaid、自动入库等知识系统工作流。

## 宏观流程

| 节点 | 要回答的问题 |
|---|---|
| T00 使用场景 | 工具解决的是开发、自动化、文档、知识入库、绘图、发布还是运维问题 |
| T01 安装配置 | 依赖、权限、路径、认证、环境变量和本机约束是什么 |
| T02 操作接口 | CLI、GUI、插件、API、文件协议、快捷键如何触发动作 |
| T03 自动化链路 | 能否和脚本、Agent、浏览器、剪贴板、知识库、定时任务串起来 |
| T04 输出验证 | 输出是否可检查、可回放、可恢复、可迁移 |
| T05 维护边界 | 版本变化、数据锁定、平台依赖、同步冲突和替代方案是什么 |

## 使用规则

- 新工具文章先做路由，不直接精读。
- 工具清单默认略读；只有能补 T01-T05 的边界、动作或失败模式，才进入核心知识点。
- 具体工具目录优先沉淀工作流和排障准则，不复制完整教程。
