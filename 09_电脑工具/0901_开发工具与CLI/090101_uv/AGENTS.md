## 技术定位

| 项 | 内容 |
|---|---|
| 工具名 | uv |
| 一级类目 | 电脑工具 |
| 二级类目 | 开发工具与 CLI |
| 技术本体 | Python 包管理、项目管理、解释器管理、虚拟环境、脚本运行和构建发布工具 |
| 全局位置 | 位于 Python 代码仓库、本机解释器、`.venv`、`pyproject.toml`、`uv.lock`、PyPI/镜像源和 CI/发布流程之间 |
| 主要使用者 | Python 开发者、数据脚本维护者、Agent/CLI 工具开发者、需要频繁切换项目环境的工程师 |
| 主要产出 | `.venv`、`uv.lock`、`pyproject.toml` 变更、脚本运行环境、构建产物、发布动作 |
| 不等同于 | Python 语言机制、打包生态全史、Ruff/Ty 类型或代码质量工具本身 |

## 宏观流程

| 节点 | 要回答的问题 |
|---|---|
| U00 场景判断 | uv 是替代 `pip`、`venv`、`pyenv`、`pipx`、Poetry/PDM 的哪一部分，还是只用于单次脚本运行 |
| U01 安装与解释器 | uv 自身如何安装，Python 版本如何安装、切换、固定和验证 |
| U02 项目初始化 | `uv init`、`pyproject.toml`、项目布局和 `.venv` 如何形成稳定项目骨架 |
| U03 依赖与锁文件 | `uv add`、`uv sync`、`uv lock`、版本约束、升级策略和镜像源如何治理 |
| U04 脚本与工具调用 | `uv run`、单文件脚本依赖声明、远程脚本和 CLI 工具调用的权限与可回放性 |
| U05 构建与发布 | 包构建、PyPI/TestPyPI 发布、令牌、版本号和发布后验证如何闭环 |
| U06 迁移与排障 | 从 `pip`、`requirements.txt`、Poetry、PDM、pyenv 迁移时，如何定位依赖冲突、路径错乱和锁文件差异 |

## 横向对标

| 对标对象 | 对比重点 | 使用判断 |
|---|---|---|
| `pip` + `venv` | uv 把依赖解析、虚拟环境和命令运行合到统一 CLI | 新项目或本机脚本优先考虑 uv；极简老项目可继续保持原状 |
| `pyenv` | uv 可覆盖部分 Python 版本安装和选择动作 | 只为项目解释器管理时可收敛到 uv；系统级多语言运行时管理仍需单独判断 |
| Poetry / PDM | 都围绕 `pyproject.toml`、依赖声明、锁文件和发布流程 | 团队已有稳定 Poetry/PDM 流程时不要只因“更快”迁移，先看锁文件、发布和 CI 成本 |
| `pipx` | 都能运行或安装命令行工具 | 临时运行脚本和项目内命令可用 uv；长期全局 CLI 隔离需要确认维护方式 |
| Ruff / Ty | 同属 Astral/Rust 化 Python 工具链，但解决代码质量和类型检查问题 | 不把 Ruff/Ty 的收益混写成 uv 的能力，只在工具链组合层记录 |

## 排重准则

问题指纹：

```text
uv + 工作流节点 + 命令接口 + 输入输出文件 + 版本/环境/发布边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是命令清单 | 没有验证动作、失败模式、迁移边界时，只追加到原文锚点 |
| 都是“更快”宣传 | 没有数据量、网络、缓存、平台和基线时，只作为待补证，不写成稳定结论 |
| 都是项目初始化 | 按普通应用、库项目、脚本项目、CLI 工具项目拆分 |
| 都是依赖管理 | 区分新增依赖、锁定环境、升级依赖、镜像源、冲突排查 |
| 都是发布 PyPI | 必须单独记录构建验证、令牌安全、版本号和发布后验收 |
| 都是远程脚本运行 | 必须记录来源可信度、版本固定、网络依赖和本机权限风险 |

## 认知校准优先级

| 优先级 | 要校准的问题 |
|---|---|
| P0 | uv 是否真的能成为用户新 Python 项目的默认入口，而不是只在单篇文章里显得“更快” |
| P0 | `uv add` 生成的版本约束、依赖升级和过期检查体验，在当前版本里是否仍存在原文所说的问题 |
| P1 | 远程脚本运行是否适合进入自动化工作流，还是只能作为一次性可信脚本执行 |
| P1 | PyPI 发布流程中，uv 覆盖了哪些动作，哪些仍需要独立的凭证、安全和回滚规则 |
| P2 | uv 与 Ruff、Ty 的组合应沉淀为 Python 工具链准则，还是继续按单工具拆分 |
