# opencli-rs：把网站、桌面应用和本地工具，都变成一行命令

这几年，大家一边在讲 AI Agent，一边也在遇到一个很现实的问题：

**模型很聪明，但它摸不到真实世界。**

它知道怎么分析网页、怎么整理信息、怎么写脚本，但如果不能稳定地拿到网站数据、不能调用本地工具、不能操作桌面应用，那很多“自动化”最后都只是停留在嘴上。

`opencli-rs` 想解决的，就是这个问题。

一句话理解：

**它把网站、桌面应用、甚至你本地已有的 CLI，统一封装成“可被命令行和 AI 调用的接口”。**

也就是说，原来你可能要手动打开浏览器、复制链接、翻接口、写脚本、处理登录状态；现在它想做的，是把这些步骤压缩成一条命令。

比如：

```
opencli-rs hackernews top --limit 10
opencli-rs twitter search "rust lang" --limit 10
opencli-rs codex read
```

从这个角度看，`opencli-rs` 不是一个普通爬虫，也不是一个简单的命令合集，而更像一个面向 AI 时代的“统一执行层”。

## 它到底是什么

`opencli-rs` 是对原版 `OpenCLI` 的 **Rust 重写版**。

项目 README 里给出的定位很直接：

* 一行命令获取网站信息
* 支持 55+ 站点、333 个命令
* 支持浏览器复用登录态
* 支持把 Electron 桌面应用 CLI 化
* 支持集成本地 CLI，比如 `gh`、`docker`、`kubectl`

它最值得注意的一点，不是“支持的网站多”，而是它把不同来源的能力统一成了同一种使用方式。

在 `opencli-rs` 里，大致有三类能力：

## 第一类：Public 模式

这类命令直接走公开 API，不需要浏览器，也不依赖登录。

例如：

* Hacker News
* arXiv
* Wikipedia
* Apple Podcasts

这种方式最快、最稳，也最适合批量化调用。

## 第二类：Browser 模式

这类命令需要复用浏览器会话，尤其适合需要登录的网站。

例如：

* Twitter/X
* Bilibili
* 小红书
* Reddit
* 知乎

它不是让你手工导 Cookie，而是通过 Chrome 扩展和后台 daemon 复用你已经登录的浏览器状态。这一点很重要，因为现实世界里，很多站点根本不是“给你一个 API key 就完事”的。

## 第三类：Desktop 模式

这部分是我觉得最有意思的地方。

`opencli-rs` 不只是抓网页，它还试图把一些桌面应用也变成命令行接口，比如：

* Cursor
* Codex
* ChatGPT
* Notion
* Discord

这意味着什么？

意味着 AI 不只是“查网页”，还可能进一步“读对话”“发消息”“导出内容”“读取当前窗口信息”。这已经不是传统 CLI 的思路了，而是在把图形界面软件重新组织成可组合的工具。

## 它为什么不只是一个“爬虫工具”

很多人第一眼看到这种项目，会自然把它理解成“采集工具”或者“命令行爬虫”。

但我觉得这会低估它。

因为它真正有意思的地方，在于它的抽象层次比爬虫更高。

它的核心不是“抓网页”，而是：

**把外部世界包装成 AI 能稳定调用的命令。**

这个思路和传统自动化脚本有几个明显区别。

### 第一，它不是一次性脚本，而是统一接口

大多数自动化脚本的问题在于：

* 每个网站单独写一套
* 参数格式不统一
* 输出格式不统一
* 登录方式不统一
* 复用成本很高

而 `opencli-rs` 试图把这些差异压到内部，外部暴露成统一命令。

对人来说，这意味着更省事。

对 AI 来说，这意味着**可发现、可组合、可推理**。

### 第二，它的适配器是声明式的

这个项目一个很巧妙的设计，是大量站点命令并不是手写 Rust 逻辑，而是来自 `adapters/` 目录里的 YAML 文件。

比如一个公开 API 站点，可以写成这种形式：

* 先 `fetch` 一个接口
* 再 `map` 字段
* 再 `filter`
* 再 `limit`

而像 Twitter/X 这种复杂网站，则会走：

* `navigate`
* `intercept`
* `evaluate`
* `scroll`
* `collect`

也就是说，它把“如何从网站拿到数据”拆成了一条条流水线步骤。

这件事的意义非常大：

**新增一个站点命令，不一定要改 Rust 代码，只要补一个 YAML 适配器就行。**

这让扩展成本明显降低，也让 AI 自动生成适配器成为可能。

### 第三，它天然适合给 Agent 用

从代码结构上看，`opencli-rs` 的 CLI 命令是动态注册出来的。

启动时它会：

* 发现内置适配器
* 发现用户自定义适配器
* 加载外部 CLI 定义
* 再把这些内容统一挂到命令树里

这意味着工具集合不是写死的，而是可以不断生长的。

如果你在 `AGENT.md` 或规则文件里让 AI 先跑一次 `opencli-rs --help` 或相关列举命令，它就能知道当前环境有哪些真实可用工具。这比让模型“猜”你机器上有什么能力，要可靠得多。

## opencli-rs 的价值，不在“能跑命令”，而在“能统一世界”

如果只看表面，`opencli-rs` 做的是命令封装。

但如果再往深一点看，它实际上在统一三种原本分裂的世界：

* 网站数据
* 桌面应用
* 本地命令行工具

这三类东西过去是分开的：

* 网站靠爬虫或 API
* 桌面应用靠人工操作
* 本地工具靠 shell 命令

而 `opencli-rs` 在尝试把它们都拉进同一个接口层里。

这对 AI Agent 来说尤其关键。

因为 Agent 最怕的不是不会推理，而是：

**每接一个新系统，都要重新适配一遍。**

今天接 Twitter，明天接 Notion，后天接 GitHub CLI，再后天接本地 Electron 应用。如果每个系统都用不同方法接入，Agent 很难变成真正可复用的基础设施。

`opencli-rs` 的思路，是把这些接入方式尽量标准化。

这就是它最有想象力的地方。

## 为什么 Rust 版值得关注

这个项目不是从零发明概念，而是把已有的 OpenCLI 用 Rust 重写了一遍。

表面看，这是“语言迁移”；但实际上，它解决的是一个很现实的问题：

**如果一个工具想成为 Agent 的基础设施，它必须轻、快、稳。**

根据项目 README 给出的数据，Rust 版在几个指标上都明显更适合作为底层工具：

* 二进制体积更小
* 无需 Node.js 运行时
* 内存占用更低
* 一些命令启动和执行更快

这类差异，对于手动用命令的人可能只是“感觉更顺”；但对于会频繁调用工具的 Agent 来说，这种差异会被不断放大。

一个工具每次少几十 MB 内存、少几秒延迟，累计下来就是完全不同的体验。

所以我会觉得，`opencli-rs` 的 Rust 重写不是“技术炫技”，而是它从实验项目走向基础设施的一步。

## 它最适合什么人

如果你只是偶尔查个网页，可能感受不到它的全部价值。

但如果你属于下面几类人，这个项目会很有吸引力：

### 1. 在搭 AI Agent 的人

你需要给模型一个稳定、统一、可发现的外部执行层。`opencli-rs` 正好就是为这个场景设计的。

### 2. 经常写自动化脚本的人

你可能已经厌倦了每个网站都单独维护脚本、Cookie、接口和输出格式。这个项目提供了一种更标准化的方式。

### 3. 想把桌面应用纳入自动化的人

很多工作流卡在 GUI 上，不能真正串起来。而 opencli-rs 的 Desktop 模式，正是在尝试打通这一层。

## 我对这个项目的一个判断

如果只把 `opencli-rs` 看成“多站点命令行工具”，那它当然已经很强。

但我觉得更重要的是，它代表了一种趋势：

**未来的软件，不只是有 GUI，也不只是有 API，还应该天然有一层 Agent 可调用的 CLI 接口。**

网站、应用、工具，最后都应该能被统一成“可组合的能力模块”。

而 `opencli-rs` 做的，就是朝这个方向迈出了一步。

它不一定已经完美，但它的产品方向是非常清晰的：

**不是给人多一个命令，而是给 AI 多一只手。**

## 结语

今天大家谈 AI Agent，常常讨论模型能力、提示词、记忆、规划。这些都重要。

但真正决定 Agent 能不能落地的，往往不是“会不会想”，而是“能不能做”。

`opencli-rs` 这类项目的价值，就在这里。

它把原本分散的网站接口、浏览器会话、桌面应用、本地工具，尽量整理成统一、可调用、可扩展的一层。这层能力一旦成立，AI 才真的有机会从“会回答”，走向“会执行”。

如果你关心 AI Agent 的下一步，我觉得这个项目很值得看。

因为它回答的不是“模型还能更聪明吗”，而是另一个更现实的问题：

**当模型已经足够聪明之后，我们该怎么把世界交到它手里。**
