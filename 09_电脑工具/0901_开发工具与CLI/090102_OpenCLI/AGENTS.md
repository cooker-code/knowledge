## 技术定位

| 项 | 内容 |
|---|---|
| 工具名 | OpenCLI / opencli-rs |
| 一级类目 | 电脑工具 |
| 二级类目 | 开发工具与 CLI |
| 技术本体 | 将网站、浏览器登录态、桌面应用和本地工具封装成可被命令行与 Agent 稳定调用的统一接口层 |
| 全局位置 | 位于本机终端、浏览器会话、桌面应用、站点适配器、结构化输出和 Agent 自动化流程之间 |
| 主要使用者 | 需要用命令行抓取站点信息、复用登录态、驱动桌面应用或给 Agent 暴露外部执行能力的用户 |
| 主要产出 | CLI 命令、JSON/文本输出、adapter 定义、浏览器连接状态、可组合的自动化步骤 |
| 不等同于 | 普通爬虫脚本、浏览器自动化框架本体、MCP 协议本体、Agent 框架本体 |

## 宏观流程

| 节点 | 要回答的问题 |
|---|---|
| OCLI00 场景判断 | 当前任务是公开站点查询、登录态站点查询、桌面应用操作、本地 CLI 统一封装，还是只是一篇工具推荐 |

| OCLI03 适配器模型 | public、browser、desktop、本地 CLI 适配分别如何取数、复用状态、声明字段和处理分页/滚动/拦截 |

| OCLI06 安全与隐私边界 | 如何限制浏览器会话、Cookie、私信/桌面应用读写、敏感数据输出和高权限命令调用 |

## 横向对标

| 对标对象 | 对比重点 | 使用判断 |
|---|---|---|
| Browser Use / Playwright 自动化 | 浏览器自动化直接操作 DOM，OpenCLI 追求把常用动作压成稳定命令和结构化输出 | 高频、结构化、可重复查询优先 CLI；页面探索、调试和未知交互仍需浏览器自动化 |
| MCP Server | MCP 强调工具协议与模型集成，OpenCLI 强调本机命令、管道和站点/桌面适配 | 需要标准工具协议时用 MCP；需要低成本终端自动化和脚本组合时用 OpenCLI |
| bb-browser | 两者都把网站 CLI 化，但运行时、adapter 生态和覆盖方式不同 | 文章中对覆盖数、稳定性和安装体验的结论必须补证后再沉淀 |
| 自写爬虫脚本 | 自写脚本灵活但复用和维护成本高，OpenCLI 试图提供统一命令树和适配器模型 | 单次低频任务可脚本；长期 Agent 工作流优先考虑可发现、可组合、可验证的工具接口 |
| 官方 API / SDK | API 稳定、合规，但很多站点没有合适 API 或登录态成本高 | 有官方 API 优先 API；只有浏览器可见数据时再评估 OpenCLI 的登录态复用风险 |

## 排重准则

问题指纹：

```text
OpenCLI + 接入对象 + 运行模式 + 命令契约 + 输出验证/排障边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是站点 CLI 化 | 按公开 API、浏览器登录态、桌面应用、本地 CLI 封装拆分 |
| 都是工具清单 | 没有命令契约、验证动作或失败模式时，只保留原文锚点 |
| 都是 Agent 调用工具 | 如果重点是 Agent 规划、权限、评估或上下文管理，转到 `02_Agent与AI工程` |
| 都是浏览器自动化 | 如果重点是 Playwright 测试或页面操作机制，转到 Playwright CLI 或工程测试目录 |
| 都是 Rust 重写 | 只有在影响安装、运行时、启动速度、二进制分发或长期维护时才进入 OpenCLI 核心知识点 |

## 认知校准优先级

| 优先级 | 要校准的问题 |
|---|---|
| P0 | OpenCLI 的长期价值不是“又一个爬虫”，而是让外部系统形成 Agent 可发现、可组合、可验证的命令接口 |
| P0 | 结构化 JSON 输出比 DOM 浏览更适合 Agent，但前提是字段稳定、错误可识别、登录态可控 |
| P1 | Rust 重写、覆盖站点数量和性能数据属于待补证事实，不能直接从文章宣传写成结论 |
| P1 | 复用浏览器登录态会放大隐私和权限风险，必须把敏感站点、桌面应用和高权限本地命令分级 |
| P2 | OpenCLI、bb-browser、MCP、Playwright 应按任务接口层区分，不按“都是给 Agent 用”混写 |
