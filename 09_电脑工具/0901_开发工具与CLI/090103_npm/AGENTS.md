## 技术定位

| 项 | 内容 |
|---|---|
| 工具名 | npm |
| 一级类目 | 电脑工具 |
| 二级类目 | 开发工具与 CLI |
| 技术本体 | Node.js 生态的包管理、脚本执行、全局 CLI 安装、包发布和 registry 交互工具 |
| 全局位置 | 位于 Node.js 运行时、`package.json`、`package-lock.json`、`node_modules`、npm registry、全局 prefix、CI 和发布流程之间 |
| 主要使用者 | JavaScript/TypeScript 开发者、前端工具链使用者、需要安装全局 CLI 或发布 npm 包的用户 |
| 主要产出 | 依赖树、锁文件、脚本执行结果、全局命令、registry 登录状态、发布包版本 |
| 不等同于 | Node.js 运行时机制、前端框架、构建工具本体、pnpm/yarn/bun 的全部生态 |

## 宏观流程

| 节点 | 要回答的问题 |
|---|---|
| N00 场景判断 | 当前是在做项目依赖管理、一次性执行、全局 CLI 安装、脚本运行、workspace 管理还是包发布 |
| N01 安装与版本 | Node.js、npm、Corepack、全局 prefix、PATH 和本机权限如何确认 |
| N02 项目依赖与锁文件 | `package.json`、`package-lock.json`、`node_modules`、语义化版本和依赖解析如何治理 |
| N03 脚本与临时执行 | `npm run`、`npm exec`、`npx` 如何触发命令，版本和当前工作目录如何固定 |
| N04 registry 与认证 | registry、镜像源、`.npmrc`、token、私有源和代理如何配置与验证 |
| N05 全局 CLI 生命周期 | 全局安装、升级、卸载、命令路径、权限和多 Node 版本冲突如何处理 |
| N06 发布与供应链安全 | 包名、版本、构建产物、登录态、2FA、token、发布后验证和撤回限制如何闭环 |
| N07 CI 与迁移排障 | CI 缓存、锁文件漂移、平台差异、npm/pnpm/yarn 迁移和安装失败如何定位 |

## 横向对标

| 对标对象 | 对比重点 | 使用判断 |
|---|---|---|
| pnpm | 磁盘复用、workspace、依赖隔离、安装速度和命令习惯 | 多包仓库和前端工程优先评估 pnpm；全局 CLI 或保守项目可继续 npm |
| yarn | workspace、锁文件、生态兼容和团队历史 | 已有 Yarn 项目不因单篇文章迁移，先看锁文件和 CI 成本 |
| bun | 运行时、包管理、脚本执行和工具链一体化 | 只有当文章主问题是包管理或脚本执行时才与 npm 对比；运行时机制归 Node/Bun 技术目录 |
| npx / npm exec | 一次性执行和项目内依赖执行的版本控制差异 | 长期自动化优先固定项目依赖；临时命令必须记录版本和来源 |
| uv / pipx | 都涉及 CLI 工具安装和脚本运行，但生态不同 | Python 工具链归 uv；Node 工具链归 npm/pnpm/yarn |

## 排重准则

问题指纹：

```text
npm + 工作流节点 + 输入输出文件 + 命令接口 + 版本/registry/权限边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是安装命令 | 只有 `npm install` 清单不沉淀，必须有路径、版本、失败模式或验证动作 |
| 都是全局 CLI | 按安装、PATH、升级、卸载、权限、多个 Node 版本冲突拆分 |
| 都是依赖管理 | 区分新增依赖、锁定环境、升级依赖、审计漏洞、镜像源和冲突排查 |
| 都是构建脚本 | 如果主问题是前端构建机制，转到 `07_工程与架构/0702_前端工程` |
| 都是 Node 生态 | 如果主问题是运行时、事件循环、服务架构，转到 `07_工程与架构/0701_后端架构/070102_Node` |

## 认知校准优先级

| 优先级 | 要校准的问题 |
|---|---|
| P0 | npm 目录只沉淀包管理和 CLI 工作流，不承载所有 Node.js 或前端工程文章 |
| P0 | 全局 CLI 安装必须记录真实二进制路径、PATH、生效验证和卸载方式 |
| P1 | `npx` 和 `npm exec` 适合临时动作，但长期自动化要固定版本，避免时间漂移 |
| P1 | registry、镜像源和 token 是发布/安装排障核心，不应只记命令 |
| P2 | npm 与 pnpm/yarn/bun 的比较要落到锁文件、workspace、CI 和迁移成本，不写泛泛优劣 |
