# 090103_npm知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 09_电脑工具 / 0901_开发工具与CLI / npm |
| 解决的问题 | npm 如何进入 Node.js 项目依赖、全局 CLI、脚本执行、registry 认证和包发布工作流 |

## 宏观流程

| 节点 | 节点问题 | 当前原文锚点 | 吸收结论 | 待补缺口 |
|---|---|---|---|---|
| N00 场景判断 | 是项目依赖、临时执行、全局 CLI、workspace 还是发布 | 暂无 | 先判断 npm 是否是主问题，避免把所有 Node 文章塞进来 | 需要从文章池做 npm/pnpm/yarn 路由 |
| N01 安装与版本 | Node.js、npm、Corepack、prefix、PATH 如何确认 | 暂无 | 本机工具类知识必须记录真实路径和验证命令 | 需要补本机 `node -v`、`npm -v`、`npm prefix -g` 等证据 |
| N02 项目依赖与锁文件 | 依赖声明、锁文件、安装策略和版本漂移如何治理 | 暂无 | npm 知识点不能只写命令，要写 `package-lock.json` 和 CI 验收 | 需要补 lockfile、`npm ci`、`npm install` 差异 |
| N03 脚本与临时执行 | `npm run`、`npm exec`、`npx` 如何稳定执行 | 暂无 | 长期自动化要固定版本；临时执行要记录来源和时间漂移风险 | 需要补 `npm exec` 与 `npx` 的当前语义 |
| N04 registry 与认证 | registry、镜像源、`.npmrc`、token 和代理如何配置 | 暂无 | 安装失败和发布失败常在 registry/认证层，不应只看包名 | 需要补私有源、镜像、token 权限和代理排障 |
| N05 全局 CLI 生命周期 | 全局安装、升级、卸载和命令路径如何维护 | 暂无 | OpenCLI、Playwright CLI、Vercel CLI 等文章里的 `npm install -g` 可作为候选锚点 | 需要补全局命令路径冲突、多 Node 版本和权限问题 |
| N06 发布与供应链安全 | 包如何构建、登录、发布、验证和撤回 | 暂无 | 发布必须单独处理 token、2FA、版本号和发布后验证 | 需要补最小包发布和回滚限制 |
| N07 CI 与迁移排障 | CI 缓存、lockfile、npm/pnpm/yarn 迁移如何定位问题 | 暂无 | 与 pnpm/yarn/bun 的比较必须落到仓库和 CI 成本 | 需要补迁移案例和失败样例 |

## 文章进入时的判断表

| 判断项 | 必答问题 |
|---|---|
| 流程节点 | 文章补的是 N00-N07 的哪个环节 |
| 主问题 | npm 是主角，还是只是某个工具的安装方式 |
| 可运行 | 是否能复现安装、脚本、全局命令、registry 配置或发布动作 |
| 可验证 | 是否能检查锁文件、命令路径、版本、registry、发布包或 CI 结果 |
| 可排障 | 是否提供网络、权限、缓存、镜像、token、lockfile 或平台差异的定位方式 |
| 可迁移 | 是否能迁移到用户本机 CLI、前端项目、Agent 工具安装或发布流程 |

## 认知校准点

| 校准点 | 当前处理 |
|---|---|
| `npm install -g` 是很多工具的入口，但不是每篇工具文章都应归入 npm | 只有安装路径、版本、权限、升级、卸载或 registry 问题成为主问题时才迁入 |
| npm 目录不是 Node.js 目录 | 运行时、后端架构和前端框架文章继续放对应技术目录 |
| npx/临时执行便利但可能漂移 | 长期自动化必须固定版本和来源 |
| npm/pnpm/yarn/bun 比较要落到工程约束 | 不按“更快/更现代”做泛化结论 |
