## 1  问题背景

面试中被问到：在一个 Agent 系统中，模型会经历多步推理和工具调用，形成一条完整的执行链路。如何评估这条链路中的推理质量？

我当时的回答是：想办法把推理过程量化，用中间 function call 的函数选择准确率来衡量推理是否正确。面试官追问"有没有其他办法"，说明这个回答虽然切中了一个维度，但远不够完整。

## 2  问题本质：为什么 Agent 推理评估很难？

传统 NLP 评估通常只看最终输出（一个答案、一段文本），但 Agent 的特殊性在于：

* **链路很长：** 一个任务可能涉及 5-20 步推理 + 工具调用，最终结果正确不代表中间每一步都合理
* **过程不透明：** 模型的 CoT（Chain-of-Thought）和工具选择背后的"思考"难以直接度量
* **正确路径不唯一：** 同一个任务可能有多条合理的执行路径，不能简单比对"标准答案"
* **错误会级联：** 早期一步推理失误可能导致后续整条链路偏离，但最终结果也可能碰巧正确

所以评估需要从多个层次、多个角度切入。

## 3  评估方法全景

### 3.1  最终结果评估（Outcome-based Evaluation）

**思路：** 不管中间过程如何，只看最终任务是否完成。

**常见指标：**

* 任务成功率（Success Rate）：是否达成了用户目标
* 答案准确率：最终输出与标准答案的匹配度
* 完成度（Completion Rate）：对于多目标任务，完成了几个子目标

**优点：** 简单直接，和用户体验最对齐。

**局限：** 无法区分"歪打正着"和"正确推理"，也无法定位链路中具体哪一步出了问题。

### 3.2  中间步骤的工具调用评估（这是我面试时的回答）

**思路：** 将每一步的 function call 选择与预定义的标准路径对比。

**常见指标：**

* 函数选择准确率（Tool Selection Accuracy）：每一步调用的函数是否正确
* 参数准确率（Argument Accuracy）：传入的参数是否正确
* 调用顺序一致性：工具调用的先后顺序是否合理
* F1 Score：将标准路径中的工具调用集合与实际调用集合做 precision / recall

**优点：** 可量化、可定位问题步骤。

**局限：** 需要预先标注"标准执行路径"，而合理路径可能不止一条；过于严格的对比会误判合理的替代路径。

### 3.3  过程奖励模型（Process Reward Model, PRM）

**思路：** 训练一个专门的模型，对链路中每一步推理进行打分，而不只是看最终结果。

**做法：**

* 收集人工标注的逐步推理质量数据（每一步标记"正确/中性/错误"）
* 训练 PRM 学习给中间步骤打分
* 在评估时，让 PRM 对 Agent 链路的每一步输出一个置信度分数

**代表工作：** OpenAI 的 "Let's Verify Step by Step" 论文，证明了 PRM 在数学推理中显著优于只看结果的 ORM。

**优点：** 能精确定位推理链路中的薄弱环节。

**局限：** 需要大量逐步标注数据，标注成本很高。

### 3.4  LLM-as-Judge 评估推理过程

**思路：** 用另一个强大的 LLM 审查 Agent 的完整推理轨迹（trajectory），对每一步或整体进行评判。

**实现方式：**

* 将 Agent 的完整轨迹（包括 CoT、tool call、tool response）输入 Judge 模型
* 让 Judge 从多个维度打分：推理逻辑是否连贯、工具选择是否合理、是否存在冗余步骤、是否遗漏关键信息
* 可以设计结构化的评分 rubric，让 Judge 按维度逐项评估

**优点：** 不需要预先标注标准路径，适用于开放式任务。

**局限：** Judge 本身可能有偏差，且对超长轨迹的评估能力会下降。

### 3.5  轨迹对比与偏好排序

**思路：** 给同一个任务生成多条轨迹（不同采样或不同策略），通过对比来评估质量。

**做法：**

* 针对同一 query，让 Agent 跑多次，产出多条轨迹
* 用人工或 LLM-as-Judge 对轨迹进行排序（哪条推理过程更好）
* 也可以计算轨迹之间的一致性（多次执行是否趋同于同一条路径）

**优点：** 天然适合 GRPO 等基于 group 的训练范式。

**局限：** 多次采样成本高，排序标准可能主观。

### 3.6  基于状态的评估（State-based Evaluation）

**思路：** 不看 Agent 选了什么工具，而是看每一步执行后"世界状态"是否向目标靠近。

**做法：**

* 定义任务的关键状态节点（key milestones / checkpoints）
* 检查 Agent 在执行过程中是否依次达到了这些中间状态
* 例如：订机票任务 → 是否获取了出发地、目的地、日期 → 是否搜索了航班 → 是否选择了符合条件的航班

**优点：** 路径无关，只关注"进展"，容忍多种合理路径。

**局限：** 需要针对每类任务定义状态空间，泛化成本高。

### 3.7  效率与成本指标

**思路：** 即使推理正确，也要评估效率。

**常见指标：**

* 步数（Step Count）：完成任务用了多少步，是否有冗余
* Token 消耗：推理过程的总 token 数
* 工具调用次数：是否有不必要的重复调用
* 延迟（Latency）：端到端完成时间
* 纠错能力（Recovery Rate）：遇到工具调用失败后，能否自我修正并继续

## 4  对比总结

| 评估方法 | 评估对象 | 是否需要标注 | 适用场景 | 关键局限 |
| --- | --- | --- | --- | --- |
| 最终结果 | 输出 | 需要标准答案 | 通用 | 无法定位问题 |
| 工具调用准确率 | 每步调用 | 需要标准路径 | 路径明确的任务 | 不容忍替代路径 |
| PRM | 每步推理 | 需要逐步标注 | 长链路推理 | 标注成本高 |
| LLM-as-Judge | 完整轨迹 | 不需要 | 开放式任务 | Judge 有偏差 |
| 轨迹对比 | 多条轨迹 | 可选 | 配合 RL 训练 | 采样成本高 |
| 状态检查 | 中间状态 | 需定义状态 | 流程型任务 | 泛化成本高 |
| 效率指标 | 资源消耗 | 不需要 | 通用 | 不反映推理质量 |

## 5  复盘与反思

回顾面试中的回答，"用 function call 的函数选择准确率"是一个合理的切入点，但存在几个问题：

1. **只覆盖了一个维度：** 工具选择准确率只是"中间步骤评估"中的一个指标，面试官期望的是一个更系统的框架。
2. **忽略了"不需要标注标准路径"的方法：** LLM-as-Judge 和状态检查等方法不依赖预定义路径，适用范围更广，这是很重要的补充。
3. **没有提到过程奖励（PRM）：** 这是当前学术界和工业界的热门方向，提到它可以展示对前沿研究的了解。

**更理想的回答框架：**

> *"Agent 链路的推理评估可以从几个层次来看：最基础的是看最终任务是否成功；往细了看，可以评估每一步的工具选择和参数是否正确，也就是和标准轨迹做对比；但标准路径不一定唯一，所以更灵活的做法是用 LLM-as-Judge 审查完整轨迹，或者定义中间状态检查点来看推理进展；学术上还有 Process Reward Model 的方向，训练模型对每一步推理打分。实际系统中一般是最终结果 + 中间步骤 + 效率指标的组合。"*

## 6  延伸思考

面试官可能继续追问的方向：

* **如何构建 Agent 评估 Benchmark？** 目前有 AgentBench、WebArena、SWE-bench 等，它们的设计思路和评估维度有何不同？
* **评估与训练的闭环：** 评估结果如何反馈回训练？PRM 分数能否直接作为 GRPO 的 reward 信号？（注意：这和上一道面试题形成了关联）
* **多 Agent 协作场景的评估：** 如果多个 Agent 协作完成任务，如何归因哪个 Agent 的推理出了问题？
* **在线评估 vs 离线评估：** 生产环境中如何实时监控 Agent 推理质量？可以用哪些 proxy 指标？
* **Reward Hacking 在 Agent 场景的表现：** Agent 是否会学到"走捷径"完成任务但实际推理质量很差的策略？
