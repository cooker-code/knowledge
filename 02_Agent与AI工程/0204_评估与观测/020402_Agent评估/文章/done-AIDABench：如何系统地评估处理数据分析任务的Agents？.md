这是一篇由 **商汤科技（SenseTime Research）** 和 **上海人工智能实验室** 联合发表于 2026 年的论文，提出了 **AIDABench**，这是一个针对 **AI 数据智能分析（AI Data Analytics）** 的综合基准测试。

该基准旨在填补现有评估在“端到端真实工作流”方面的空白，特别关注企业级场景中常见的复杂数据处理任务。以下是详细解读：

### 📖 一、 研究背景与核心动机

随着 LLM Agent 从简单的文本处理向自主处理复杂多步数据工作流演进，现有的评估基准存在以下局限：

1. **孤立能力评估：大多只测试单一维度（如仅测试代码生成或仅测试图表绘制），无法反映端到端的任务有效性。**
2. **场景过于简化：缺乏对真实职场中异构数据（Excel, CSV, PDF, DB）混合处理的模拟。**
3. **缺乏深度洞察：现有模型往往只能做描述性统计，缺乏解释性和评论性洞察。**

**AIDABench 的目标**：提供一个科学、 holistic（整体性）的评估标准，涵盖从数据读取、清洗、聚合到可视化和文件生成的完整链路，服务于企业采购、工具选型和模型优化。

### 🧩 二、 AIDABench 基准设计

#### 1. 任务构成

基准包含 **600+** 多样化的文档分析任务，分为三大核心能力维度：

| 维度 | 占比 | 核心能力要求 | 示例任务 |
| --- | --- | --- | --- |
| **问答 (QA)** | 37.5% | 数值推理、趋势分析、极值提取、分散度度量 | “计算Q3平均销售额并对比Q2增长率” |
| **文件生成 (File Gen)** | 43.3% | 数据清洗、格式规范化、跨表联动、去重、缺失值填充 | “合并两个Excel sheet，按产品分组计算新用户流失率，输出Summary.xlsx” |
| **数据可视化 (Vis)** | 19.2% | 图表类型选择、样式定制、多图表组合 | “绘制2024年黄金价格的30日均线与实际收盘价对比折线图” |

#### 2. 难度分布

* **Easy (≤6 steps): 29.5%**
* **Medium (7-12 steps): 49.4%**
* **Hard (≥13 steps): 21.1%**
* **特点：超过 70% 的任务属于中高难度，且 27.4%的任务涉及 跨文件推理（最多涉及1个任务处理14个文件）。**

#### 3. 数据来源

* 源自真实职场场景：财务报表、销售记录、考勤日志、项目追踪表等。
* 经过专家标注和隐私脱敏处理。
* 每个任务的地面真值（Ground Truth）由至少两名专家交叉验证，标注耗时约 1-2 小时/题。

### 🛠️ 三、 评估协议与方法论

#### 1. 推理协议 (Inference Protocol)

* **Tool-Augmented: 模型通过 execute\_code 工具调用 Python 代码进行数据检查、转换和输出。**
* **Sandbox: 无状态容器化沙箱，预装 pandas, matplotlib, openpyxl等库。**
* **Plan-Execute-Verify Loop: 模型迭代编写代码，验证中间结果，最多允许 20 轮交互。**

#### 2. 专用评估器 (Specialized Evaluators)

为了克服传统指标（如 BLEU）在结构化任务中的不足，设计了三种专用评估器：

* **QA Evaluator:**

+ 使用 Binary Judge (LLM-as-a-Judge)。
+ 定量问题严格匹配数值（允许舍入误差）；定性问题检查是否覆盖核心论点。

* **Visualization Evaluator:**

+ **Correctness Scorer: 检查图表数据是否正确（如轴标签、数据点）。**
+ **Readability Scorer: 检查可读性（如字体、配色、布局、图例）。**
+ 最终得分 Sviz=0.7×C(χ)+0.3×R(χ)Sviz=0.7×C(χ)+0.3×R(χ)。

* **Spreadsheet File Evaluator:**

+ **Stage 1 (结构检查): 检查 Sheet 名称、行列数、表头。**
+ **Stage 2 (内容采样): 对小文件全量比对，大文件采样比对（归一化数值、日期、字符串）。**
+ **Stage 3 (约束验证): 检查特定业务逻辑（如去重策略、排序规则、Top-K）。**
+ **早期停止机制: 一旦发现致命错误立即判定失败，节省 Token。**

### 📊 四、 主要实验结果

评估了 **11 个 SOTA 模型**（包括 Claude Sonnet 4.5, Gemini 3 Pro, Qwen3-Max, DeepSeek-V3.2 等）。

#### 1. 整体表现

* **最佳性能: Claude Sonnet 4.5 取得最高分，Pass@1 为 59.43%，Pass@3 为 70.78%。**
* **挑战巨大: 即使是最强模型，在 Hard 任务上的成功率也显著下降，表明复杂数据分析仍是 LLM 的短板。**
* **模型差距: 小参数模型（如 Qwen3-30B）在文件生成任务上表现明显落后于超大模型。**

#### 2. 分维度表现

* **数据可视化: 得分最高（多数顶尖模型 Pass@3 > 70%），Gemini 3 Pro 在此项表现最佳 (78.97%)。**
* **文件生成: 得分最低（最佳 Pass@1 仅 ~49%），是主要的瓶颈所在，因为涉及复杂的结构化操作和严格的格式要求。**
* **问答 (QA): 居中，Claude 在此项表现最好。**

#### 3. 辅助信息的影响 (Ablation Study)

* 提供 **Auxiliary Spreadsheet Summary**（电子表格摘要，如元数据、稀疏单元格映射）能显著提升大多数模型的性能，尤其是中小模型。
* 移除摘要后，8/11 的模型性能下降，其中 Qwen3-30B 下降超过 10 分。

#### 4. 错误分析

* **QA 主要错误: 数值计算错误 (Claude: 67.1%)。**
* **Vis 主要错误: 非英文文本渲染乱码 (Claude: 83.9%)，这是当前多模态生成的通病。**
* **File Gen 主要错误: 语义理解偏差 (Semantic Misunderstanding)。例如，模型未能正确理解“保留首次出现的重复项”或错误的透视操作，导致输出结构完全错误。DeepSeek 和 Qwen 还常因超出轮次限制而失败。**

### 💡 五、 核心贡献与启示

1. **首个端到端数据智能基准: 超越了单纯的 CodeGen 或 TableQA，涵盖了真实办公场景中的“脏活累活”（清洗、对齐、多文件关联）。**
2. **结构化评估范式: 提出的三阶段文件评估器和可视化双维度评分，为后续研究提供了更可靠的评估工具。**
3. **揭示了当前模型的局限性:**

* **长程依赖与状态管理: 在处理多步数据转换时，模型容易丢失上下文或产生累积错误。**

* **结构化输出的严谨性: LLM 擅长生成自然语言，但在生成严格符合业务逻辑的 Excel/PDF 结构时仍显笨拙。**

* **效率问题: 文件生成任务消耗的 Token 和交互轮次远高于 QA 和 Vis，成本高昂。**

### 📝 总结

**AIDABench** 是 AI Agent 进入企业级数据应用领域的重要里程碑。它明确指出：**当前的 LLM 虽然能写出漂亮的代码，但在处理真实世界中混乱、多维、需严格合规的数据分析任务时，仍有很长的路要走。** 对于开发者而言，提升 Agent 在 **文件生成** 和 **长程状态跟踪** 方面的能力是接下来的重点。
