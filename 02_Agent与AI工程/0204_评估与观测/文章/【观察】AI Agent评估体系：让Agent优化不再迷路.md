现象

现在大大小小的团队，基本都能搭一套业务 Agent：                          支持 RAG、Workflow、MCP、Skill、Memory……开源框架拿来改一改就能跑

但到了真实细分场景，常见问题马上出现：

* 换模型（换更智能的的模型）
* 改Prompt（更准确的说，是不断堆叠prompt）

做完这些操作后，很多团队其实回答不了一个核心问题：改完之后，整体表现到底变好了多少？优化到什么程度算及格？什么程度算优秀？

 

定义

Agent评估调优是指通过建立量化指标与可观测性（Trace/Span）来拆解定位问题，并据此对Agent进行闭环迭代改进的系统化方法

 

拆解

先抽象出2类场景agent，一个是问答，一个是执行类；

问答类，包含如：智能客户、知识库、陪伴类、话术类等

执行类，包含如：数据查询、调用外部MCP工具等

 

/\*评估指标

|  |  |  |  |
| --- | --- | --- | --- |
| 类 | 问答类 | 经验值 | 备注 |
| 技术指标 | 正确率（事实） | 及格：80%-89%  优秀：90%-95%  天花板：96%-98% | 和数据集配合可测量 |
| 溯源率 | 及格：60%-75%  优秀：80%-90%  天花板：90%-95% | 内置（langfuse内置） |
| 拒答率 | 及格：5%-12%  优秀：3%-5%  天花板：1%-3% | 和数据集配合可测量 |
| 业务指标 | 复制/点赞/引用率 | 及格：15%-25%  优秀：30%-40%  天花板：1%-3% | 自定义 |
| 点踩率 | 及格：5%-10%  优秀：2%-4%  天花板：<1% | 自定义 |

 

 

|  |  |  |  |
| --- | --- | --- | --- |
| 类 | 执行类 | 经验值 | 备注 |
| 技术指标 | 成功率 | 及格：70%-84%  优秀：85%-94%  天花板：95%-98% | coze罗盘有内置 |
| 工具调用准确率 | 及格：75%-85%  优秀：88%-94%  天花板：95%-97% | coze罗盘有内置 |
| 单任务推理步数 | 及格：5-7步  优秀：3-4  天花板：2-3 | 自定义 |
| 业务指标 | 单任务运行成本 | 及格：人工成本50%  优秀：人工成本20%  天花板：人工成本10% | langfuse内置 |
| 平均响应延迟 | 及格：5-10s  优秀：2-4s  天花板：<1.5s | langfuse内置 |

其中单任务运行成本，比如简单任务大于0.05$，复杂任务大于0.5$，则不及格！

如上的参考值是给各位一个平均的参考概念，我自己两类都参与做过项目，实际测下来工具的调用准确率很高，而平均响应延迟就和是否开启thinking关系很大的，所以具体的场景要具体对待！

 

/\*可观测工具和理论概念

 

以langsmith为例来了解

Trace(追踪):Trace 代表一次完整的用户请求或操作，是一个独立、可追踪的执行单元!

Observations(观测): 每个 Trace 内部的组成单元，根据性质不同分为三种

1）Span(跨度)：代表一个有时长的连续操作块，例如数据检索、API 调用、一段代码逻辑等

2）Generation(生成):一个专用于记录 LLM 调用的 Span，会额外捕获模型名称、参数、Token 使用量和成本等信息

3）Event(事件):代表一个没有持续时长的瞬时事件，用于记录运行过程中的关键点

 

Session (会话)：将一系列归属于同一次用户交互的 Traces 组合在一起【注意这里的session是表征一次连续的对话或交互过程，比如可能30分钟后，你再点击就是另外一个sessionid了】

Score (评分)：可以是数值、类别或布尔值！

3）离线筛选BadCase

 

设置筛选规则：通过Langfuse的API或CLI，定时查询生产数据，筛选出评分较低（例如relevance < 0.3）或命中特定坏标签的Trace。

添加到标注队列：将这些筛选出的Trace自动添加到Langfuse的Annotation Queue中。，供人工审核

 

4）沉淀为Dataset

Annotation Queue中的样本经过人工确认和修正后，就可以作为标准答案，加入 Dataset中。

如名为production-badcases-v1。

构建测试项：为数据集添加Item。一个Item包含三部分信息：

```
Input：用户的原始问题 Expected Output（可选）：经过人工确认的、正确的回答或预期结果 Metadata：可以标记这个用例的来源（例如“来自生产环境的Bad Case”）或问题类型（例如“意图分类错误”）。
```

5）：在 CI/CD 中自动验证

拥有了一个代表真实用户痛点的DataSet后，就可以融入开发流程，在每次上线前自动运行。

 

执行数据集运行：在CI/CD流程（如GitHub Actions）中加入一个任务，使用Langfuse CLI或SDK来运行一个 Dataset Run 。

 

运行评估：在Dataset Run中，应用代码会逐条处理数据集里的每个Item。同时，你的评估函数会对其输出进行打分（score）。

 

生成对比报告：运行完成后，Langfuse会自动生成一份详细的报告，并与之前（例如上周或上一个版本）的Dataset Run结果进行比对，清晰展示质量是提升还是下降。

 

设置上线标准：根据报告结果来决定CI/CD流程的成败。例如，如果核心指标的总体平均分比基线下降了超过5%，则CI/CD任务失败，阻止低质量版本上线，

 

 

 

 

为什么需要人工筛选？简单来说，自动化评分负责高效，人工筛选则负责精准和深度。这里面需要不断去维护的就是DataSet. Dataset中的Bad Case

```
推荐起点：30% ~ 50%30% Bad Case + 70% 正常/简单 Case：适合大多数应用初期的数据集。既能保证对主要流程的覆盖，又能重点回归历史问题
```

 

Todo

1、如果你开始在做agent的搭建，需要认证去考量你的agent的表现的评估（包含技术指标、业务指标）！这里让我们类比传统机器学习，每次算法迭代后，效果是变好了还是变差了，技术指标会有类似AUC等离线验证，离线通过了再上线！

2、然后，花时间去构建Dataset，将agent的评估集测试run放到CI/CD中流程中！

3、建立线上评估，比如使用langfuse是线上的每次交互都会打分，coze罗盘是采样（自动化任务）做打分！
