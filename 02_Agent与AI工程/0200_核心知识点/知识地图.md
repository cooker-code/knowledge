| 项 | 内容 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| A00 任务边界 | Agent 要自动化什么任务，人工接管点在哪里 | Agent 必须有任务、状态和退出条件 | Agent 框架、工作流文章进入 |
| A01 Context Engineering | 如何发现、选择、切分、压缩、缓存和注入上下文 | 上下文质量决定执行上限，但长窗口不是充分解 | Context Engineering、Chunking、Prompt Cache 文章进入 |
| A02 工具权限 | 工具如何声明、调用、鉴权、审批和审计 | 工具调用必须最小权限和可追责 | MCP、Tool、Skill、权限文章进入 |
| A03 Harness Engineering | 单 Agent、多 Agent、长任务如何被运行时约束 | 编排核心是状态、环境、质量门禁和回滚 | Harness、状态机、质量门禁文章进入 |
| A04 Prompt Engineering | 系统提示词、用户提示词、示例和反例如何成为工程接口 | Prompt 是接口契约，不是玄学话术 | Agent Prompt、AI Coding Prompt、Prompt 治理文章进入 |
| A04b 推理范式 | 复杂任务要线性拆解、多路径搜索，还是借助外部反馈闭环 | CoT、ToT、ReAct 是推理路径控制方式，不是正确性证据 | CoT、ToT、ReAct、Reflection、LATS 文章进入 |
| A05 Memory Management | Agent 什么时候记、记什么、怎么检索和遗忘 | 记忆必须有写入门槛、污染治理和评估 | Memory、长期记忆、记忆压缩文章进入 |
| A06 评估观测 | 如何记录 Trace、失败类型、评测和回归 | 没有评估观测就不能生产化 | 评估、观测、坏例治理文章进入 |
| A07 交付闭环 | AI 输出如何接入代码、文档、数据或业务动作 | 产物必须可运行、可验证、可回滚 | AI Coding、交付、审查文章进入 |

## 文章进入时的对比规则

| 判断项 | 处理方式 |
|---|---|
| 补充 | 文章提供新的模块边界、失败模式、实现步骤、验证指标或工程权衡时，写入对应流程节点 |
| 冲突 | 文章观点和当前节点准则不一致时，先记录冲突原因，不直接覆盖已有沉淀 |
| 不同层次 | 入门教程、源码机制、生产实践、产品发布分属不同层次，必须标注层次后再吸收 |
| 降权 | 只有标题党、工具清单、无证据收益、泛泛最佳实践时，不进入核心知识点 |

## 使用方式

- 先读本文件的宏观流程，再读具体主题知识点。
- 新文章不要单独写摘要，先归入流程节点，再判断补充、冲突、不同层次或降权。
- 如果某个节点已有多篇文章，后续应拆成更细的 `0200_核心知识点/<主题>.md`，而不是继续扩展文章列表。
