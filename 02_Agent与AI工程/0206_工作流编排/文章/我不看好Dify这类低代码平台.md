dify 刚开始是惊艳的。

23年 LLM 刚火起来的时候，相比起 langchain 等 AI workflow 调用框架，dify 独树一帜。

大量的人希望拿 LLM 做出千奇百怪的 AI agent 软件应用出来。dify 完成了一件事情，那就是**大幅度降低了搭建 AI 智能对话web页面和后端业务流的难度，并建立了自己的搭建体系**。

以往搭建软件，总得来个程序员；而有了 dify 就甚至连程序员都不需要了，稍微具备一些软件方面的知识，直接在网页上面做连连看——搭业务流程线即可。

dify 中设计并提供了各种各样的功能，方便搭建 AI agent 的用户：设置模型参数，设计流程，查看每一步骤的耗时，查看告错信息，编写处理流程。可以说是一应俱全：

dify 这个低代码平台，提供了就目前看来，我认为的AI 智能体低代码领域，最好的使用体验。

## 搭建 AI 智能体，不用程序员了？

但是，这一切的幻象都在我近期真正使用了 dify 之后，开始逐个破灭。

是的，我原本想使用 dify 来搭建智能体业务流。

当我在 dify 平台上做连连看的时候，它弹出了红色的错误日志，说实话，如果你不会写代码，很难看懂下图中红色的文字，像天书一样。

所谓的低代码平台，乍一听以为可以脱离编程语言了，打造了像连连看一样的小游戏，在平台上构建一个类似于业务流程图一样的东西，就算大功告成。而事实上，你不会写 python 代码，还真就干不了用 dify 搭建智能体的事。

## 低代码，真的方便吗？

一个编程语言的代码文件，里面全是英文符号，乍看起来确实是挺恶心的，每个字符之间什么关系，总归得有点智商和学历才能看懂。

dify 至少能够使程序员少看这些繁杂的英文符号，大大减少了开发量，也算是功德一件。

不过我在使用 dify 的时候，仍然发现了一些问题。dify 平台提供了组件**问题分类器**：

这个组件里会调用 LLM 实现对用户问句的分类，之后可针对不同的类别，流转入不同  workflow 流程中。但这个组件**不能直接流式输出 LLM 的分析过程**，至少我在平台上找来找去，没找见如何流式输出，也就是打字机式地输出 AI 模型的响应结果。这一点对于我所搭建的产品的设计要求比较重要，dify 满足不了。或者说，可能 dify 提供了，但是我自己太笨了，没找到，这个寻找与学习的成本，比较高。

还有，这个问题分类器，我不知道为什么里面会预设了这么多内置的 prompt？（下图中是dify 的 问题分类器里预设的英文内容，里面很长，还给了example，不是我人工填写的）

我希望对于 prompt，能够 100% 自主掌控，不然调用次数多了的话，dify 内置的 prompt 很长，浪费我的 token，这笔钱和支出，我是舍不得的。

再者，我希望在搭建业务流，也就是做连连看的时候，能够充分灵活，可为我所控制。我想在下图中，红色箭头处连上一条线，但是 dify 平台不允许这么做，不符合 dify 连线的规则规范。

以上两点，又暴露出 dify 平台的另外一个问题：**看似简单，但不方便，不灵活。** 这也是各种低代码平台的通病。

dify 已经在尽力实现平台的灵活易用性，提供无数的组件、功能，甚至允许用户自定义插件，但是，学习成本太高了。

我学习 Python，关键字和语法大概也就二三十条，一个智商正常的大学生，学习一周，就可以逐渐上手，以后不论编写什么程序，都是同样的语法，同样的使用方式，这些编程语法知识，我可以用到老。而低代码平台呢？我学会了 dify，换一个平台，逻辑又变了。而 dify 也只能编写智能体这一种功能而已，它不能编写别的形形色色的软件。

## 沉重的部署

如果你用过 dify，不妨后台看一下，启动 dify 平台需要多少资源？它需要启动好几个docker容器。而直接用 python 语言调用类似 langchain 的框架，只需要启动一个服务而已。

这是我吐槽 dify 的第三个点。当然，我听说 dify 也已经在逐渐优化改善平台，减轻部署重量了。

## dify 的死穴

上面都是我遇到的问题。我想你可能会说，是我对 dify 不熟练，还没学会使用，就急着写文章来喷 dify。

然而并不是这样，恰恰是 dify 平台做的太认真了，太仔细了，所以我才写这篇文章出来。

相比起我用过的大量付费商用的软件，它们丑陋、垃圾、难用，反而 dify 平台界面美观，页面点击流畅，搭建智能体所需的按钮、参数配置一应俱全，我不能骂 dify 工程师们不用心，相反，他们把平台做成这样，很用心，这一点我要夸。

**dify 就像一把瑞士军刀。**

什么意思呢？瑞士军刀做工精良，功能齐全，把一大堆功能巧妙地安排在一个很小的空间里，方便携带，不得不说，瑞士军刀的设计者、工程师和工人们都非常认真。也非常出名。

**然而现实生活中，几乎没什么人会去使用瑞士军刀。**

是的，瑞士军刀，除了在极特殊的场景，没人会真的拿瑞士军刀去开酒瓶子，剪纸剪线，挖耳屎……

原因无他，**瑞士军刀偏离了日常生活使用的最低成本值。** 人们就是觉得开瓶器更方便，更易用，所以不可能会去用瑞士军刀。也恰恰是使用普通的剪刀非常方便，用完往旁边一扔就行了，根本不用收纳。

dify 同样犯了这个死穴。 **dify 偏离了搭建软件的最低成本值。**

说回搭建程序软件，它永远都**在便捷性和灵活性之间寻找平衡点**：

* 最早人们使用汇编语言，发现实在太恶心，太不方便了，于是人们发明了 C 语言；
* 然后人们还是觉得 C 语言里指针十分恶心人，又发明了 Python语言。
* 现在人们依然觉得 Python 还是满篇文本文件里，class、def、int 英文字母一箩筐那一套，于是，**低代码编程被提了出来**。

但问题是，所有的各种低代码平台，都往**便捷性**上使劲偏离，导致欠缺**灵活性**。

这个问题与 Python 何其相似？！Python 就是加强了编程的**便捷性**，牺牲了像 C 一样的**灵活性**，灵活地操作计算机内存。因此，程序运行性能，内存效率都比不上C。

但是，Python 发展到现在，真没有哪个编程语言能够继续优化替代它。它已经可以说是发展到编程语言**便捷性** 的尽头了，在保证**灵活性** 的基础上。

是的，真正的软件编程的**最低成本值**，就卡在了 Python 和 C 之间，与什么低代码平台没有关系。

举个例子，我们正常人吃西瓜，都是拿菜刀来切。

但是，神经病会拿下面这种鬼东西切西瓜：

不论商家宣传**切西瓜神器**有多么新颖、方便，只要是个智商正常的人，都知道这玩意就是智商税。因为它就像瑞士军刀一样，已经偏离了人们日常生活使用的最低成本值——切西瓜神器刀片多，很容易划伤手；也只有切西瓜这一种功能，不能干别的；清洗收纳都不方便、占空间……等等。

我们拿 Python 等语言编写 AI 智能体流程，就好比正常人拿菜刀切西瓜。

拿 dify 来搭建 智能体流程，就好比拿着切西瓜神器去切西瓜。

也不是不能干，但就是学习成本高，不灵活，笨重。就像我上面举的例子一样，dify 里面时不时会弹出来代码告警日志，还需要插入编程语言函数来处理数据，**既然我都要在 dify 里写代码了，为啥不直接用 python 写代码呢？？**

## agentic AI 超级智能体

现在 AI 编写程序的能力，我不敢说非常强吧，但各家公司都在往这个方向发力。

AI 都能写代码了，让它帮我搭一个AI agent 还是个复杂的事吗？

我现在写代码，很大一部分工作都是 AI 代劳，我负责复制粘贴，检验写得对不对。那么，**低代码平台所谓省去了人力编写的劳动量，还有价值吗？**

更何况，AI 现在在朝着多步任务推进，也就是，只设置了 AI 模型要完成任务的终点，至于过程如何推理分析，全都不管，走的就是强化学习的路子。

我很看好这条路子，虽然过程很坎坷（数据的稀疏，算力的欠缺等问题）。

说白了。 AI 持续的发展，给 dify 这个软件平台，当然也包括所有其它的低代码平台，盖上了最后一撬黄土。

## dify 的出路？

我个人觉得，dify 应当想办法尽量往零代码平台方向发展，也就是完全抛弃在里面嵌入代码模块，返回报错日志给用户这些功能。

因为，你既然要打造低代码平台，不如一步走到黑，说白了，做成**零代码平台**嘛！

产品的发展方向，也应该是像 Photoshop、Pycharm、VSCode 这种方向发展。不过也不乐观~~~因为 AI 冲击这几款产品都在疯狂找出路，往 AI 上靠。

但即便dify 下大力气实现了零代码，它也只能完成智能体这一个功能，dify 的前景肯定比 Pycharm、VScode 这几款产品小得多。

甚至，我觉得 dify 的前景都不如一些具体的 AI 智能体，因为那些可以实打实得看到钱，拿得到利润。而 dify 的处境如果不调整的话，只会往下坡路上走。

不好意思，我就是一个不吐实话不快的人。
