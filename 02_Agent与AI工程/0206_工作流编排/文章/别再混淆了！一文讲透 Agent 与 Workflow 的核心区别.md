# 前言

> 大模型火了，智能体（Agent）和工作流（Workflow）也成了圈里圈外的高频词。
>
> 面试常问，架构常争，到底该怎么选？今天一次性说清楚。

最近和朋友聊大模型落地，发现一个很有意思的现象：

很多人把Agent挂在嘴边，仿佛不用 Agent 就落伍了；

另一拨人则坚持Workflow 才是稳定的王道，认为 Agent 只是个噱头。

其实，两者没有高下之分，只有合不合适。 

今天这篇文章，我们就来彻底搞懂：Agent 和 Workflow 到底有什么区别？什么时候用 Workflow？什么时候该上 Agent？

# 先看两个直观的例子

### 1. Workflow 版 —— 客服机器人处理退货

你输入：“我要退货”

机器人：“请提供订单号”

你输入：“123456”

机器人查询订单 → 符合退货条件 → 生成退货单。

每一步都是写好的。

它不会突然问你“要不要换成优惠券”，也不会跟你聊“你吃了没”。

因为它只是按剧本演戏。

### 2. Agent 版 —— 帮你规划旅游的智能体

你输入：“3000 元预算，北京出发去杭州，三天两晚。”

Agent 会这样工作：

* 思考：先查交通费用。
* 行动：调用搜索工具，查到高铁往返约 1200 元。
* 观察：还剩 1800 元。
* 思考：需要查酒店和景点门票。
* 行动：搜索经济型酒店、热门景点票价。
* 观察：发现可能超预算。
* 思考：调整方案，找免费景点或更便宜的住宿。
* 再次行动……直到给出完整计划。

它是即兴发挥，自己想办法达成目标。

# 核心区别：控制权归谁

| 维度 | Workflow（工作流） | Agent（智能体） |
| --- | --- | --- |
| 决策者 | 开发者 / 设计者 | 模型（LLM）自己 |
| 路径 | 确定、固定 | 动态、不可预测 |
| 适应性 | 差，遇到异常常失败 | 强，能调整策略 |
| 工具调用 | 硬编码，谁调用谁 | 自主规划，按需调用 |
| 稳定性 | 高，易调试 | 较低，可能“跑飞” |
| 成本与延迟 | 低，快 | 高，慢 |
| 适用场景 | 重复性高、流程明确的任务 | 开放性、多步推理的复杂任务 |

一句话总结：

* Workflow 是确定性的：输入 A，必然走到输出 B。
* Agent 是自主性的：给定目标，自己想办法，路径不唯一。

# 为什么大模型时代这个问题更重要？

以前，Workflow 靠规则引擎、脚本语言实现。

现在，大模型（LLM）可以充当大脑。

* 在 Workflow 中：LLM 只作为一个「聪明节点」使用。比如用 LLM 做情感判断、内容总结，但流程骨架仍是代码控制。

+ 优点：省钱、速度快、可观测。

* 在 Agent 中：LLM 是「CEO」——自己定计划、选工具、执行、反思、重试。

+ 优点：极其灵活，能解决开发者都没预料到的复杂问题。
+ 缺点：费钱、慢、可能陷入死循环。

# 企业实战中怎么选？给三个黄金原则

✅ 优先 Workflow 的情况

* 业务流程非常明确，几乎没有例外。例如：数据清洗 ETL、定时报表生成、标准退换货流程。
* 对成本、延迟敏感。例如：每调用一次 Agent 要消耗上千 token 和几秒钟，Workflow 可能只需要几十毫秒。
* 需要强审计、确定性。例如：金融交易对账，不允许“随机应变”。

✅ 考虑 Agent 的情况

* 目标明确，但达成目标的路径不固定。例如：“帮我分析这份财报，找出风险点”。
* 需要多步推理 + 外部工具。例如：先搜索、再计算、再对比、最后生成报告。
* 环境动态变化，需要实时调整策略。例如：自动比价购物助手，不同网站价格随时变动。

✅ 最佳实践：Workflow + Agent 混合架构

用 Workflow 搭建稳定的脚手架，在关键决策节点用 Agent 填充。

举个例子：智能客服工单处理系统

**Workflow 部分**

* 接收工单 → 提取用户ID、订单号 → 查询用户基本信息 → 判断问题类型（根据关键词）

**Agent 节点**

* 目标：解决「用户想改收货地址」这类问题
* Agent 自主行为：

1. 思考：需要先查订单是否已发货
2. 行动：调用查询物流状态工具 → 获取结果
3. 观察：若未发货 → 调用修改地址工具 → 通知用户
4. 观察：若已发货 → 思考：无法改地址，需生成安抚话术 + 建议联系快递员
5. 反思：是否遗漏了用户备注？再调用读取用户备注工具确认

* 最终输出：完成改地址或给出替代方案

# 面试满分回答模版

我认为核心区别在于控制权的归属。

Workflow 是确定性的，路径预先定义，适合重复性高、成本敏感的任务。Agent 是自主性的，模型自己做计划、调工具、反思重试，适合开放、不确定的问题。

实际工程中，两者不是二选一，而是互补。我通常会优先设计 Workflow 作为稳定骨架，在需要动态决策的节点引入 Agent 能力，避免过度设计。
