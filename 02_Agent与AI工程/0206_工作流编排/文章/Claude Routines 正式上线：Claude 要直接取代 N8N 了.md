# 

## 开篇：一场迟到的"正面对决"

Anthropic 刚刚发布了一个名为 **Routines**（例行任务）的新功能。简单来说，它让 Claude 可以通过**定时调度（schedule）**、**触发器（trigger）**、甚至**webhook** 的方式自动启动任务。

这意味着什么？**这一次，Claude 正式闭环了——它摇身一变，成为了一个专门的自动化平台，直接对标 N8N 这类"拖拽式"的无代码自动化工具。**

在此之前，作者（视频作者）曾多次表示"N8N 要完了，因为某某产品上线了，而且做得更好"。但每次都会发现，某些特定功能在 N8N 这类无代码平台上依然无法被完美替代，始终达不到一对一的覆盖。

**而 Routines，是 Claude 给出的真正的"一对一替代方案"。** 它具备调度能力、具备工作流编排能力，完整复刻了 N8N 的核心功能，是 agentic（代理式）执行知识任务的下一阶段。

本文将：

* • 展示几个真实的 Demo
* • 一步一步带你在 Claude 桌面端和 API 两种方式下搭建自己的 Routines
* • 深入拆解这个功能的 UX 和工作原理
* • 演示如何把现有 N8N 工作流一键迁移到 Routines

## 一、Demo 1：每日邮件汇总 + 回复草稿

作者展示的第一个最"原型化"的例子，是一个**每日邮箱汇总 + 起草回复**的 Routine。

操作流程非常直接：

1. 1. 点击 **"Run now"**（立即运行）——这是一个 demo/test 功能，可以立刻执行并实时查看输入输出
2. 2. 点击 **"Runs"** 按钮，你会看到其实他做的事情就是"喂了一个 prompt"
3. 3. 这**和 Claude Code 本质上是完全一样的东西**，只不过运行在一个**标准化的云端容器**上，而不是在作者自己的电脑上

> ⚠️ 这里要注意一个关键区别：因为运行在云端，所以你**不会在旁边"盯着它、手动引导它"**。因此：
>
> * • 指令一定要**更清晰**
> * • 要确保它拥有**所有需要的信息**

除了在界面里测试以外，你还可以：

* • **按计划定时运行**
* • 通过 **webhook** 触发
* • 通过 **API 请求** 触发

也就是说，你既可以基于"**输入数据**"触发，也可以基于"**输出数据**"触发。非常强大。

> 📌 如果你看过作者之前关于 **agentic workflows（代理式工作流）** 的视频，Routines 基本上就是那种工作流的**标准化、企业级版本**。

### 这个 Demo 实际在做什么？

这个 Routine 正在做的事情：

* • 搜索 Gmail 里的邮件
* • 基本上把作者平时手动做的事情**全都做了一遍**，和在本地电脑上运行没什么区别
* • 唯一的差异是它在云端运行
* • 经过一系列工具调用，直到达成"完成的定义"——在这个案例里就是：**任务结束后，用 Slack connector 给作者发一条更新消息**

### 关于官方文档的"吐槽"

作者特别提到一点：Claude Code 官方文档里列出的大多数用例都**过于技术化**，比如：

* • backlog maintenance（待办积压维护）
* • alert triage（告警分诊）
* • bespoke code review（定制化代码审查）

说白了，**大多数人根本不知道这些词在说什么**。

但作者希望大家明白：**Routines 本质上就是"自动化"的直接替代品——你可以用自然语言自动化任何东西。**

为此，作者还开发了一个 Claude Skill（技能），你可以把它导入自己的工作区，然后把任何**现有的工作流**（无论是自然语言的描述、SOP 文档，还是 N8N / make.com 上的无代码工作流）交给这个 Skill，它就能帮你转换成 Routine。

### 结果展示

回到 Skill 页面，可以看到它找到了 2 封未读邮件。但作者关心的**不是**这个线程里的对话本身——

作者想看的是**发到 Slack 私信里的那条更新**，因为如果把它当成传统自动化来用，这才是通知应该到达的地方。

打开 Slack，果然在 12:01 收到了通知：

* • 它把未读邮件拉过来了
* • 给出了高层次的摘要
* • 为一封邮件起草了一个**礼貌拒绝**的回复
* • 为另一封起草了一个**接受**的回复

作者还打开 Gmail 里的草稿箱确认——所有草稿都已经写好了。比如一封回复："嘿 Corey，谢谢联系，告诉另一个 Nick 我拥有那个东西，很乐意上节目，把你那边可行的几个时间段发过来。" 连收件人地址都已经自动填好了。

## 二、如何设置定时调度？

上面这个 Demo 作者是用 "test 功能" 实时跑的，但实际使用中你肯定希望它**自动运行**。设置方法非常简单：

1. 1. 回到刚才创建的 Routine
2. 2. 点击那个小按钮，选择不同的触发器
3. 3. 当前是 "Call via API"，切换为 **Schedule**
4. 4. 出现一个可视化界面，可以选择：**每小时 / 每天 / 等等**
5. 5. 因为这是邮件分诊流程，作者希望在起床前就跑完。作者最近大约 5:20 起床，所以把时间设为 **5:10**
6. 6. 重要提示：**你可以同时添加多个触发器**
7. 7. 保存后，回到 Routines 主页，会看到一个小日历视图
8. 8. 可以看到"每日邮箱汇总+起草"已经安排在 5:10 运行

**这意味着：作者什么都不用做，同样的流程就会自动执行**：

* • agent 通过 Gmail connector 检查邮箱
* • 按照给定的 SOP / 逻辑运行——在本例中就是"查看之前是否有过邮件往来"
* • 起草回复
* • 通过 Slack 发送给作者

## 三、Demo 2：通过 API 触发的"会议记录 → 商业提案"Routine

作者展示的第二个 Routine，接收一段由 **Fireflies** 生成的会议记录（Fireflies 是一个会议转录服务，它会加入你的通话、收听对话，然后把所有内容存成文本）。

为了演示，作者用 **API 请求** 的方式触发（同时也说明了 webhook 的工作方式——自动触发）。

### 演示过程

1. 1. Routine 叫 **"Transcript to Proposal"**
2. 2. 直接点 "Run now" 没用，因为**还没有实际的 transcript**——指令写的是"通过 API 调用给我一段记录"
3. 3. 作者打开一个 Claude Code 实例，让它用这段 transcript 发送一个 API 请求
4. 4. 按下回车（API key 是为了 demo 硬编码进去的，作者半开玩笑说"反正我刚才已经泄露过 API key 了"）
5. 5. 它会先把 curl 请求作为文本载荷发出去
6. 6. 然后完全自主地生成一份提案

API 请求触发后，Routine 随即在云端运行，拿到了完整的 transcript 和对话里的交易条款。

### 展开看具体在做什么

打开这个 run 来看：**"我通过 API 调用给你一段会议记录。我希望你用我的另一个 AI agent 在一个 managed session 中创建完整的提案。"**

这里出现了一个新概念——**Managed Sessions（托管会话）**：

* • 它是另一种设置不同端点的方式
* • 让 AI 能够创建一个**互联的托管 agent 网络**
* • 每个 agent 都有自己**独立的容器**（出于安全和隔离考虑）

验证 Slack 所需资源就绪后，它开始生成高质量的提案。

### 这到底解决了什么问题？

作者在这里花了一段时间系统性地讲清楚——**Routines 到底解决了什么**。

**旧的自动化设计方式**：

1. 1. 一个**事件/外部触发器**（比如每天早上 5 点的定时）
2. 2. 把这个事件喂进 **N8N** 这样的平台
3. 3. 平台负责按照你创建的一连串逻辑处理——就是那些**拖拽式的节点**
4. 4. 最后把输出推送到 Slack、CRM、数据库等外部系统

作者特别强调：中间那一段逻辑——也就是**搭建自动化的核心工作量**——非常耗时：

* • 拖拽所有节点
* • 设置所有凭证
* • 做所有认证
* • 获取数据、映射变量到正确字段

这些才是一个自动化从业者\*\*"真正的重头工作"\*\*。

**新的方式（Routines）**：

1. 1. 事件依然是事件——API 调用、webhook，或者定时任务
2. 2. 但不再需要把事件送进 N8N、也不再需要自己从零搭建一堆拖拽式节点
3. 3. 你只需要**用自然语言写一段高层级的指令**——这显然比拖拽要容易得多
4. 4. 然后它就可以像 N8N 之前做的那样，把输出推送到 Slack、CRM 等

**核心观点**：**Routines 完美解决了"中间"这一段的问题。** 这是 Claude 真正的、一对一的 N8N 替代品。

### 提案生成的结果

等待流程结束，作者打开生成好的提案：

* • 已经按照 managed agent 的模板生成
* • 所有数据都被拉取进去了
* • 内容示例："我们是一家 AI 内容写作市场，连接企业客户与经过审核的自由撰稿人……"
* • 这就是他们**日常实际发给客户的提案**

> 💡 作者感慨：**两分钟**就能把一个 Routine（基于 API 事件的系统）集成到你的基础设施里。而以前，他手动设计一份这样的提案要花 **2.5 到 3 个小时**。

## 四、Routines 界面深度拆解

### 主页长什么样？

打开方式：`claude.ai/code/routines`

主页你会看到：

* • 所有 Routine 以**网格视图**展示
* • 每个卡片显示：标题、运行时间、下次计划运行时间
* • 还有一个**分类列表**（系统自动分类的）
* • 还有一个**日历视图**——能直观看到什么时候会执行什么

作者自己创建的几个 demo 包括："daily Nixer I've mentioned scan"、"morning inbox drafts"、"news video ideas"。今天这几个 Routine 分别会在 6:51、7:43、8:17 运行。

### 从零创建一个新的 Routine

点击右上角的 **"New Routine"**，会依次问你几件事：

#### 1. 名字

作者示范输入：**mailbox drafter**

#### 2. 描述 Claude 在每次会话中应该做什么

这里就是给它一个 prompt——**这个 prompt 本质上等同于一个 Skill**。

> 就像 Skill 一样，你需要给模型一份**标准化的步骤清单**，让它执行一些对你有经济价值的工作。  
> Routine 的描述也应该按照 **SOP 或步骤清单**的形式来写。

作者的**重要建议**：

**比写 Skill 的时候更加精准一些。**

原因是：

* • 在 Skill 里，你可以**边运行边调整**、临时改变任务方向
* • 但 Routine 是**完全无人值守**的——它基本上需要**每次都近乎完美地工作**
* • 所以要尽可能**清楚、精准**地描述，**减少它可能出错的空间**

作者展示的实际 prompt 示例：

> "用提供的 Gmail connector 拉取我所有的未读邮件（稍后详谈）。对每一封未读邮件，检查是否有与该联系人的过往对话。如果有，拉取过往对话作为上下文，然后基于你对我和任务上下文的理解起草回复。完成后，用 Slack connector（稍后详谈）给我发一条更新。"

关于长度：作者没发现有**长度限制**（他反复粘贴长文本测试过）。**建议倾向于写多一些上下文，而不是写少。**

#### 3. 选择代码仓库

作者选了一个叫 "this business one" 的仓库。

#### 4. 选择模型

作者选了 **Opus 4.6 1 mil**。

#### 5. 选择云环境

你可以创建一个云环境，带上一堆：

* • 环境变量
* • 密钥
* • API 凭证
* • 等等

作者选择默认环境。

#### 6. 选择触发器

三种选择：

* • **Schedule（定时）**
* • **GitHub event（GitHub 事件）**
* • **API 调用**

这个用例其实适合定时运行，但为了演示，作者选了 API。好处是可以很快在另一个 Claude Code 实例里调用并测试。添加触发器后，会得到一段 **curl 请求片段**——这段代码可以给任何模型使用。

#### 7. 添加 Connector（连接器）

这一步非常关键。

* • 点击 "Add Connector"，连接 Gmail
* • 如果你还没有 connector，去 **Claude Code Settings → Connectors**，点击 Connect
* • 它会要求你把 Claude 连接到 Gmail 账户，完成 OAuth 登录
* • 本例还需要 Slack connector——在搜索框里搜、点加号，完成新一轮 OAuth 授权即可
* • 回到 Routine，手动添加这些 connector，会拿到一个 **token**
* • 把 token **复制并安全保存**

至此，你的 Routine 已经准备就绪。

### 运行测试

点击右上角 "Run now" 启动工作流：

* • 用 GUI 方式只是为了演示
* • 新的 run 出现，它就会发送那条消息
* • 完成后，它用 Slack connector 发送更新
* • 可以看到它已经开始调用 tool search 了

> 🎯 关键点：**作者自己在这里看着它运行，但正常使用下，你根本不需要知道它在运行。** 它在你设定的时间点触发，你早上醒来就会在 Slack 里看到所有邮件和起草好的回复。

最终结果：

* • 两封邮件都被处理
* • 一封是"有人邀请你上播客"
* • 另一封是"有人向你咨询几个问题"

## 五、如何把 N8N 工作流一键迁移到 Routines？

最后一段，作者展示了**如何把第三方工具（比如 N8N）上已经搭好的工作流转换成 Routine**。

### N8N 的便利特性

N8N 的一个很酷的功能：

* • 鼠标悬停
* • 按住 Shift
* • 按 **Cmd+C** 复制（或者右键 → Copy）

然后你就拿到了一堆 **JSON** ——这就是这些节点在文本形式下的语法表示。

### 转换步骤

1. 1. 回到 **Antigravity**（作者使用的环境，里面有 Claude Code 窗口）
2. 2. 输入刚刚复制的 JSON
3. 3. 在最上方加一句话：**"Use the routine generator to turn this N8N workflow into a routine"**
4. 4. 完成

这里用到的就是**作者为大家开源的那个 Skill**——它把 N8N 的 JSON 转换成可以用自然语言调用的 Routine。

### 关于"要不要把所有工作流都迁移过来"？

作者给出了非常务实的建议：

> 我不会鼓励你把所有工作流、或者从 N8N / 其他无代码工具迁移到 Claude 的后端。

原因是：

* • 使用 **token**（Routines）会比使用**纯算力**（N8N）更贵一些
* • 这并不是一对一的经济账

真正的要点是：

> 如果你今天要**新搭建**一个工作流（以前要花好几个小时搞 N8N 的那种），那么**直接用 Routine 一次性搞定**可能更合理。

### 迁移演示结果

Claude 执行了任务：

* • 加载 Routines
* • 因为作者没提供具体调度上下文，它就默认安排了一个 schedule
* • 生成的 Routine：**从 Hacker News 的 Algolia API 拉取文章，提取 hits，格式化成 markdown 报告并提交**——这正是原 N8N 流程在做的事
* • 原 N8N 流程：点 Execute → 运行 scraper → 抓取 Hacker News 上的一堆 hits（比如"开源 AI 才是前进之路"这样的评论）
* • 新 Routine：完全一样的效果，只是"Routine created and fired. Hacker News AI stories fetch."

### 修改起来更简单

作者还演示了 Routine 相比 N8N 的**修改便利性**：

* • 现在这个 Routine 只是抓数据，但数据就"卡在那里"
* • 如果在 N8N 里，修改起来**非常麻烦**
* • 在 Routines 里：直接点 Connectors → 加 Slack → Save
* • 可以设定在 7:33 MDT 运行、或通过 API 请求触发
* • 还可以回到对话里说："**很好，更新一下让它在抓取完后给我发一条 Slack 消息。**"
* • **3 秒钟**，它就能发一个 HTTP 请求修改那个 Routine

不需要拖拽任何节点。**快得多，也容易得多。**

## 六、作者的实际应用场景

作者提到他的 agency（服务公司）正在/即将用 Routines 做的事情：

### 1. 替换所有提案生成器

把所有内置的提案生成流程换成 Routines。

### 2. 销售电话的后续自动化

* • 销售电话结束后，收到一个 webhook，附带 transcript
* • 把 transcript 喂给一个 Routine
* • 自动生成**电话结束后的跟进邮件**
* • 同时生成一个**工作流图示草稿**，作为邮件附件——"为了给客户留下更用心、质量更高的印象"

### 3. 合同签约后的自动化

* • 监控客户是否签了提案
* • 一旦签约，webhook 触发另一个 Routine
* • 发送邮件，包含：

+ • 入职欢迎
+ • 日历邀请
+ • 感谢他们加入的消息

### 总结观点

> **你现在基本上可以把业务中所有"非人类面对面"的环节全都自动化。**
>
> 这其实以前也能做到，只是当时**非常费力**、需要**相当多的专业知识**。
>
> 现在，只要你理解 Routine 的规范，以及本视频展示的这些内容，**你就已经完全够用了**。

作者最后总结：
