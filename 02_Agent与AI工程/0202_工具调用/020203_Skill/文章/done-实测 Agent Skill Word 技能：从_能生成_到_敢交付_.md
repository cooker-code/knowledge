## **1  评论区的问题**

上篇文章发出去之后，评论区问得最多的不是"怎么装"，而是：

**敢不敢用到最后一步？**

* 图片会不会丢？
* 排版会不会乱？
* 能不能按学校/公司的模板来？

问得好。Word 文档不是玩具——它是要交给领导、导师、甲方看的。

所以这篇我专门测了一下，看看 Word 技能的真实边界在哪。

## **2  实测：两个真实场景**

### **2.1  开题报告**

这是最常见的学术场景。我给了一个比较严格的格式要求：

```
bash

创建一个开题报告，要求：
- 一级标题：黑体，三号，加粗
- 二级标题：黑体，四号
- 正文：宋体，小四，1.5倍行距
- 包含：研究背景、研究现状、研究方法、预期成果、参考文献
```

生成结果：

▲ 开题报告word截图

**检查结果：**

* ✅ 一级标题：黑体、三号、加粗——正确
* ✅ 正文：宋体、小四、1.5 倍行距——正确
* ⚠️ 二级标题：样式定义了，但文档里没实际用到
* ❌ 参考文献：只是普通文本，没有悬挂缩进，不符合 GB/T 7714 标准

结论：**基本格式能做到，但细节有瑕疵。**

### **2.2  项目方案**

这是工作场景。要求更复杂：封面、目录、表格、页眉页脚。

```
bash

创建一个项目方案文档，要求：
- 封面页（标题、编制单位、日期）
- 自动目录
- 正文包含表格
- 页眉显示文档标题
- 页脚显示页码
```

生成结果：

▲ 项目方案word截图

**检查结果：**

* ✅ 封面页：有，居中排版正常
* ⚠️ 目录：有（需要右键更新域才能显示）
* ✅ 表格：有，边框和表头居中都正常
* ⚠️ 页眉：有，但标题少了"智能"二字（与封面不一致）
* ✅ 页脚：有，显示"第 X 页"

结论：**复杂结构能做，但细节会出错。**

## **3  真实的能力边界**

两个场景测下来，发现一个规律：**大框架能做对，小细节容易错。**

### **3.1  能做的**

| 功能 | 说明 |
| --- | --- |
| 基本格式 | 字体、字号、行距、对齐 |
| 标题样式 | 能定义，但要检查是否实际应用 |
| 表格 | 带边框、表头、对齐 |
| 页眉页脚 | 文字、页码 |
| 目录 | 自动生成（需手动更新） |
| 封面页 | 居中排版、空白间距 |
| 分页 | 手动分页符 |

### **3.2  容易出错的**

| 功能 | 实测问题 |
| --- | --- |
| 样式应用 | 定义了但可能没用上（如二级标题） |
| 内容一致性 | 页眉和封面标题可能不一致 |
| 参考文献 | 只是普通文本，不符合学术规范 |
| 目录 | 生成后必须手动更新 |

### **3.3  做不了的**

| 功能 | 说明 |
| --- | --- |
| 复杂模板 | 不能读取现有模板并填充 |
| 精确排版 | 毫米级的位置调整做不到 |
| 复杂表格 | 合并单元格、斜线表头等 |
| 图片定位 | 能插入，但位置控制不精确 |

## **4  我的用法**

我现在不会用它来生成"最终交付版"。

我的用法是：**用它生成初稿，然后检查和微调。**

这里有个技巧：**用不同的模型做不同的事。**

这篇文章的测试结果，其实是用 Codex（GPT5.2）检查出来的。它发现了页眉标题不一致、参考文献格式不对这些细节问题。

不同模型有不同的特点：

* Claude 擅长生成内容和结构
* GPT 系列幻觉低，擅长检查细节

所以我的工作流是：

1. 用 Claude Code 生成 Word 初稿
2. 用 Codex 检查格式和细节问题
3. 发现问题后，让 Codex 修复（它能发现的问题，通常也能解决）
4. 最后用打开确认

用几次就知道哪个模型适合干什么了。

## **5  总结**

回到开头的问题：敢不敢用到最后一步？

我的答案是：**看情况。**

* 如果是内部文档、初稿、模板——直接用，没问题
* 如果是正式交付、格式严格——用它生成初稿，使用 codex 在检查一下调一下，不行只能自己上手了。

Word 技能的真实边界：

* 能做：基本格式、表格、目录、页眉页脚、封面页
* 容易错：样式应用、内容一致性、参考文献格式
* 做不了：精确排版、复杂模板、复杂表格

知道边界在哪，才能用得放心。

当然，这和模型能力有关系，也可能今天模型降智了。以后模型更强大，这些都能解决。
