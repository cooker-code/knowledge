我们每天都会接触到大量信息，本质上这是一个从 **“知识输入”** 到 **“技能输出”** 的过程。通过不断学习和转化知识，我们掌握新的技能，也提升了解决问题的能力。

**Claude Skills** 是 Anthropic 公司为 Claude 大模型开发的一套“能力插件系统”。它的目标，是让 AI 变得更“懂事”——也就是更准确地理解我们的意图；同时也更“持家有道”——帮我们更高效、更省资源地完成任务。

## 亲身体验 CodeBuddy 的 Skills 能力

腾讯在这方面跟得挺紧，CodeBuddy 是目前国内首个在 AI IDE 中实现 Skills 协议的产品。

今天，我们就来实际体验一下 CodeBuddy 的 Skills 功能，真正感受 **“在 AI IDE 里使用 Skills”** 是什么样的。相比之下，Claude Code 目前还只是一个 CLI 工具，使用体验上还是差了一些。

### 准备技能库

Anthropic 官方开源了一些 Skills，我们可以先把它们克隆到本地。

找到 CodeBuddy 的安装目录（一般是 `~/.codebuddy`），把想用的 Skills 复制进去。

### 查看可用 Skills

在 CodeBuddy 的输入框中输入“列出能用 Skills”，它就会自动识别并列出当前可用的技能。CodeBuddy 支持在个人或项目级别设置技能，目前都是通过识别文件夹中的 `SKILLS.md` 文件来实现的。

## 试试官方提供的技能

接下来，我们看看官方技能具体能做什么。

### 场景一：用 docx 技能生成文档

**输入指令：**

> 请熟悉当前工程，生成一份 Word 文档，介绍这个工程的主要功能清单。

大约 **两分钟** 后，CodeBuddy 输出了一份 `功能清单.docx`，内容清晰，格式也没问题。

**它的工作原理大概是这样的：**

把 Word 文档看作是一个由 XML 文件组成的压缩包，然后通过 JavaScript 或 Python 的文档处理库直接操作里面的 XML 结构，实现文档的创建或编辑。整个过程不需要打开 Microsoft Word，完全自动完成，既保证了格式标准，又节省了时间。

本质上，它是通过脚本调用各种系统工具来完成处理。试想一下，如果这个过程完全交给大模型从头生成，那得消耗多少 Token 啊？

## 一些目前存在的不足

### 生态还不够成熟

目前官方提供的 Skills 数量还比较少，和 MCP 这类成熟协议的生态相比还有差距，难以满足多样化的需求。另外，也还没有一个像 NPM 那样统一的 Skills 资源中心。

### 自定义有门槛

虽然 Skill 是用 Markdown 文件来定义的，但一旦涉及调用系统资源或复杂操作，就需要写脚本。如果想深度定制，对用户的技术能力有一定要求。目前来看，基于产品文档或开发文档来生成技能，是一个不错的实践方向。生态中出现的 skill seek 这类工具，也值得关注。

### 其他 AI 厂商跟进较慢

目前看来，除了腾讯的 CodeBuddy，国内其他 AI 厂商对 Skills 协议的跟进还比较少（可能是还没正式发布）。

## 总结

从技术发展的角度来看，笔者对 Claude Skills 的未来还是比较乐观的。

尽管目前它还处于 v1 阶段，在**稳定性、工具链成熟度和开发者体验**方面都还有挑战，但它的设计理念是清晰且具有前瞻性的。它精准地切中了当前 AI 发展的一个核心问题：如何安全、可靠且高效地将大语言模型的理解能力，与结构化的实际操作系统结合起来。

笔者所看好的，并不是它现阶段的表现，而是它所选择的这条技术路径未来可能带来的可能性。一旦背后的执行引擎和开发生态跟上，它很可能会极大释放 AI 的生产力。
