我对Agent Skill的理解是错误的，以及我是如何重构它们的

### 章节 1: 发生了什么

Agent Skill于 10 月 16 日发布。

作者立即开始构建它们。

在两周内，作者完成了多个大型Skill（例如：`cloudflare` - 1,131 行，`shadcn-ui` - 850 行，`nextjs` - 900 行，`chrome-devtools` - 超过 1,200 行）。

作者的仓库因此获得了 **400+星标**。

但是，当 Claude Code 激活多个相关Skill时，上下文窗口会急剧增长（加载 5-7 个Skill意味着增加 5000-7000 行）。

作者起初认为这是必须接受的，认为将所有内容放在一个巨大的 `SKILL.md` 文件中意味着代理能获取所有信息，从而获得更好结果（更多信息=更好的结果）。

**作者认识到这是错误的。**

### 章节 2: 残酷的真相

解决方案一直就在作者眼前，但作者却错过了。

作者一直在把Agent Skill当作文档堆，而不是它们本来的样子：**上下文工程问题**。

作者感到沮丧，因为自己甚至在 `skill-creator` Skill中记录了“渐进式披露”原则，却没能理解其在实践中的含义。

作者感到恼火，因为花了两个星期调试“上下文增长”问题和缓慢的激活时间，而问题完全是自找的。

每个巨大的 `SKILL.md` 文件在 90% 的时间里都在加载无关信息。

### 章节 3: 重构之前：灾难

Skill文件结构示例（`.claude/skills/` 目录下包含多个大型文件）。

总计：36 个Skill共约 15,000 行（大约 120K 到 300K 个 token）。

**问题**：激活 `devops` 上下文意味着立即加载 2,500+ 行代码，其中大部分从未被使用。

### 章节 4: 重构之后：渐进式披露架构

作者使用三层加载系统进行了重构：

**第一层：元数据（始终加载）**

* • 仅 YAML 前导部分。
* • 约 100 个字。
* • 足够让 Claude 判断该Skill是否相关。

**第二级：`SKILL.md` Skill入口（Skill激活时加载）**

* • ~200 行以内。
* • 包含概述、快速入门、导航图。
* • 仅引用但不包含其内容。

**第三级：引用文件和脚本（按需加载）**

* • 每段 200-300 行。
* • Claude 仅在需要时读取详细文档。
* • 模块化，专注于单一主题。

### 章节 5: 技术细节 - 数字

**claude-code Skill重构：**

* • 之前：一个文件包含 870 行代码。
* • 之后：181 行代码 + 13 个参考文件。
* • **减少：79%（token 效率提升 4.8 倍）**。

**完成阶段 1 和 2 的重组：**

* • 之前：36 个独立Skill，共 15,000 行代码。
* • 之后：整合为 20 个专注的Skill组（初始加载 2,200 行代码 + 45 个参考文件），包括 `devops`, `web-frameworks`, `ui-styling`, `databases`, `ai-multimodal`, `媒体处理`, `chrome-devtools` 等。
* • **初始激活减少：85%**。

**实际影响：**

* • 激活时间：~500 毫秒 → <100 毫秒。
* • 上下文溢出：快 → 慢。
* • 相关信息比例：10% → 90%。

### 章节 6: 根本原因分析

**根本性错误：作者把“可用信息”和“已加载信息”混淆了。**

**更深层次的理解错误：Agent Skill不是文档。**

它们是**针对开发工作流程的特定能力和知识**。每个Skill代表一种**能力**（例如：`devops` 是部署无服务器函数的能力，`ui-styling` 是设计一致界面的能力，`sequential-thinking` 是一种问题解决方法论）。

作者之前将每个工具视为需要单独的文档输出（36 个独立Skill），这是错误的。Skill应该按**工作流程能力**来组织，而不是按工具。

整合有效的原因：从“36 工具特定Skill”变为“20 工作流能力组”，思维模式从“文档”转变为“开发工作流”。

**200 行的限制**是基于 LLM 能高效扫描上下文来决定下一步加载什么。保持入口点在 ~200 行以内，Claude 可以快速理解Skill内容、决定读取哪个参考文件、仅加载该文件（总计 400-700 行高度相关上下文）。

**这是上下文工程学的基础知识，而作者却错过了。**

### 章节 7: 经验教训

1. 1. **200 行规则很重要** - 它是快速导航和上下文泥潭之间的区别。
2. 2. **渐进式披露并非可选项** - 每个超过 200 行的Skill都应该被重构。无一例外。如果无法在 200 行内容纳核心指令，说明入口点塞入了太多内容。
3. 3. **引用是第一等公民** - 作者曾将 `references/` 视为“可选的额外文档”。错误。引用才是真正的工作发生的地方。`SKILL.md` 只是地图。
4. 4. **测试冷启动** - 清空上下文，激活Skill，并进行测量。如果首次激活加载超过 500 行，说明做得不对。
5. 5. **指标不会说谎** - 4.8 倍的 token 效率提升是“偶尔有效”和“稳定可靠”之间的区别。

### 章节 8: 总结

**Skill ≠ 文档**

Skill是在特定工作流程时刻激活的**能力**（例如：编写测试 → 激活 `code-review`，调试生产环境 → 激活 `sequential-thinking`，部署基础设施 → 激活 `devops`，构建 UI → 激活 `ui-styling` + `web-frameworks`）。

每项Skill都教会 Claude 如何**执行**特定的开发任务，而不是工具的作用是什么。

把它们当作文档来处理会失败，因为文档是被动参考材料，Skill是主动工作流程知识。

渐进式披露有效，因为它符合实际开发方式：扫描元数据判断相关性 → 读取入口点了解工作流模式 → 按需加载特定引用获取实现细节。

每一步都小而精，目标明确，这才是构建有帮助而非令人不知所措的能力的方法。

最痛苦的教训是作者自己记录了解决方案却没理解它。

**最终的教训：上下文工程不是加载更多信息，而是在正确的时间加载正确的信息。**
