论文首页

如果说前几个月不少 Agent 论文还在讨论“怎么让智能体拥有技能”，那这篇 **Trace2Skill** 更进一步问了一个非常关键的问题：

**Agent 在执行任务时积累下来的那些局部经验，到底能不能被真正整理成稳定、可迁移、可复用的技能？**

这不是一个小问题。

因为今天很多系统确实已经会“从轨迹中总结经验”了，但这些经验经常有三个毛病：

* • 很碎，只对应某一条具体轨迹里的某个坑
* • 很脆，换模型、换任务、换分布后就失效
* • 很散，最后变成一堆 memory / patches / notes，很难真正作为技能长期维护

Trace2Skill 想解决的，正是这里的断层：

> **不是把轨迹经验存下来，而是把它蒸馏成一份像人类专家手写 SOP 那样、但又比人工技能更贴近真实执行证据的 skill。**

## 一句话先看懂

如果用一句话概括 Trace2Skill，我会这样总结：

> **它不是让 Agent 顺着每条新轨迹一点点修技能，而是先并行分析大量成功与失败轨迹，再把局部 lesson 统一蒸馏成一份冲突更少、迁移性更强的技能文档。**

这个“先广泛观察，再统一归纳”的思路，其实很像人类专家写操作指南时的方式。

人不会因为看了一次失败案例就立刻改整本手册；人往往会先看很多案例，再总结哪些规律是稳定的、哪些只是偶然现象。
Trace2Skill 的核心价值，就在于它把这个“人类写技能的经验路径”做成了一个自动化框架。

## 一、这篇论文到底想解决什么问题？

今天越来越多的 LLM Agent 都在依赖 `skills`。

这里的 skill，不只是几句提示词，而是一类结构化、可复用的任务指导文档，里面通常会包含：

* • 什么时候该用某种方法
* • 步骤应该怎么走
* • 哪些坑最容易踩
* • 哪些脚本、参考资料、辅助文件值得配套带上

问题在于，高质量技能长期都靠人工写，扩展速度很慢。

而自动生成 skill 的已有做法，作者认为又有两个典型问题：

* • 一类过于依赖模型参数知识，写出来的 skill 缺少真实环境下的细节和坑点
* • 另一类会根据新来的轨迹顺序式更新 skill bank，但容易把很多“局部 lesson”直接糊到技能上，最后得到的是碎片化、局部过拟合的结果

论文因此把矛头指向了一个更深层的问题：

**轨迹中的经验是局部的，但技能必须是全局的。**

如果把局部经验一条条硬塞进技能库，结果通常会出现：

* • 规则彼此冲突
* • 技能数量不断膨胀
* • 查询和维护越来越困难
* • 一些只对单个样本成立的“经验”误被写成普适规则

所以 Trace2Skill 的核心任务并不是“从轨迹中提一点建议”，而是：

**如何把很多轨迹中的局部 lesson，蒸馏成一份单一、全面、可迁移的 skill。**

## 二、它和已有在线 skill evolution 工作有什么本质区别？

论文一开始就给了一个非常直观的对比图。

Figure 1

*原论文 Figure 1：左边是已有工作常见的顺序式更新方式，技能库随着一条条新轨迹不断被修改；右边是 Trace2Skill 的做法，把一池轨迹并行分析后再统一蒸馏成技能。*

这张图背后其实对应了两个非常重要的差别。

### 1. 碎片化积累 vs. 单技能蒸馏

很多在线方法会不断新增 patch、memory 或小 skill。
短期看，这样更新很灵活；长期看，就会出现技能越来越碎的问题。

而 Trace2Skill 更接近“把多条证据汇总后，写成一版新的标准作业程序”。

### 2. 顺序式反应 vs. 整体式归纳

顺序式更新最大的问题，是每一轮都在对“最近刚看到的轨迹”做反应。
这容易让 skill 受到近期样本的干扰，发生所谓的 **sequential drift**。

Trace2Skill 则选择先看一批轨迹，再统一合并 patch，从全体经验里找更稳定的规律。
这使它更像“归纳”，而不是“追着案例打补丁”。

如果把这个差别说得再白一点：

> **已有很多方法在做 trajectory-to-memory，Trace2Skill 真正在做的是 trajectories-to-skill。**

## 三、Trace2Skill 的整体框架是什么？

论文的方法图非常清楚，几乎可以当成整篇文章的骨架。

Figure 2

*原论文 Figure 2：Trace2Skill 的三阶段流程。Stage 1 生成轨迹，Stage 2 由成功/失败分析子代理分别提出 skill patch，Stage 3 再通过层次化合并得到最终 evolved skill。*

整套方法分成三步。

### Step 1：先 rollout，得到成功与失败轨迹池

给定一个初始技能 `S0`，Agent 会在一组演化任务上运行，生成一批执行轨迹。

这里的 `S0` 有两种来源：

* • **Deepening 模式**：从人类专家已经写好的 skill 出发
* • **Creation 模式**：从模型凭参数知识草拟的一版 skill 出发

然后作者把这些轨迹按结果拆成两类：

* • `T+`：成功轨迹
* • `T-`：失败轨迹

这一步不是简单收集数据，而是在为后面的“经验蒸馏”准备原始证据。

### Step 2：并行子代理分析轨迹，提出 patch

每条轨迹会交给一个专门的 analyst 子代理处理，输出一份 skill patch。

这里论文做了一个很聪明的非对称设计。

**成功轨迹分析器 A+**
成功案例比较好处理，所以采用单次分析流程：清洗轨迹、识别可泛化的有效行为模式、输出 patch。

**失败轨迹分析器 A-**
失败分析更难，所以作者没有用“一次性 LLM 总结”，而是做成了一个 ReAct 风格的 agentic loop。
这个分析器可以反复检查执行痕迹、看输入输出文件、核对 ground truth，直到找到真正的失败原因，或者用尽预算。

这一步很关键，因为论文其实在强调一个判断：

**失败里最有价值的不是“错了”，而是“为什么错”。**

如果失败原因没找准，后面的 patch 就很容易把偶发现象写成错误规则。

### Step 3：把所有 patch 层次化合并，蒸馏成最终 skill

前面每个 patch 都只来自单条轨迹，天然很局部。
所以 Trace2Skill 最核心的一步，其实发生在合并阶段。

作者把所有 patch 放进一个层次化 merge 过程里：

* • 去重
* • 冲突检测
* • 文件格式校验
* • 保留高频、稳定、能反复出现的规律
* • 丢弃低支持度、疑似偶然的 patch

论文把这个 merge 过程直接解释成一种 **inductive reasoning**。

因为系统并不是简单拼 patch，而是在问：

* • 哪些观察在不同轨迹中反复出现？
* • 哪些修改更像是普适规律？
* • 哪些建议只服务于极少数案例，不该进入主 skill？

这也是 Trace2Skill 和“经验记忆库”最大的不同：

> **它不是把经验存起来等待检索，而是把经验进一步压缩、归纳、提炼成技能本体。**

## 四、这个 skill 在论文里到底长什么样？

论文把 skill 明确写成一个结构化目录：

* • 一个根文档 `SKILL.md`
* • 一组辅助资源 `R`

这些辅助资源通常包括：

* • `scripts`
* • `references`
* • `assets`

这个定义很重要，因为它说明 Trace2Skill 不是在生成单段 prompt，而是在演化一个真正可使用的知识目录。

从工程角度看，这有三个直接好处：

* • 它天然适合 Agent 框架接入
* • 它比“把所有经验都塞进上下文”更便于维护
* • 它让技能变成了可以长期版本化的外部资产

论文也特别强调，最终 skill 在推理时是**直接预置使用**的，不需要额外的检索模块。
这点在后面的实验里也会成为它优于 retrieval-memory baseline 的原因之一。

## 五、为什么作者认为这种方法更像“人类专家写技能”？

这篇论文有个很打动我的地方，是它没有把“并行多 Agent + merge”包装成纯工程 trick，而是把整个方法论落回到一个很自然的人类行为上：

**真正高质量的技能，通常不是对单个案例的立即反应，而是对一批案例的系统归纳。**

作者觉得已有在线方法和人类写技能相比，主要差在两点：

* • 人类会先积累较广的领域理解，再写 SOP
* • 人类更倾向于写一份完整技能，而不是无限膨胀的小 patch 集合

Trace2Skill 就是在模拟这种流程：

1. 1. 先看很多真实执行
2. 2. 再抽出局部 lesson
3. 3. 最后进行整体蒸馏

从这点看，它更像一个“经验编辑部”，而不是一个“经验备忘录”。

## 六、主实验到底证明了什么？

论文主实验集中在 spreadsheet domain，因为表格任务很适合检验 skill 的真实价值：

* • 文件系统交互复杂
* • xlsx 内容肉眼不好直接读
* • 工具调用顺序和细节很关键
* • 很多错误来自环境约束而不是纯推理错误

作者使用的主要 benchmark 包括：

* • `SpreadsheetBench-Verified`
* • `SpreadsheetBench` 的 Soft / Hard 评分
* • OOD 测试集 `WikiTableQuestions`

先看最核心的主结果表。

Table 1

*原论文 Table 1：主结果表。作者同时比较了 skill 作者模型、skill 使用模型、deepening / creation 两种起点，以及 error / success / combined 三种轨迹来源。*

这张表信息很多，但真正值得记住的有四点。

### 1. 人类写的 skill 很强，但不一定可迁移

论文里 human-written baseline 对 122B Agent 很有帮助，比如：

* • SpreadsheetBench-Vrf：`48.33`
* • WikiTQ：`74.68`

但它迁移到 35B Agent 上时却明显失效，甚至不如 `No Skill`。

这说明一个非常现实的问题：

**“好 skill”并不一定是跨模型稳健的 skill。**

### 2. 只靠参数知识起草 skill，效果接近没有

作者把由模型仅凭参数知识起草出来的 skill 叫做 `Parametric`。
它整体表现接近 `No Skill`，这很关键，因为它说明：

**真正有价值的技能内容，很多并不在参数里，而在真实轨迹暴露出的执行细节里。**

### 3. Trace2Skill 的 deepening 能稳定强化人类技能

从 human-written skill 出发，经过 Trace2Skill deepening 后，性能普遍上升。

很典型的一组结果是：

* • 122B authored `Deepening +Combined`：相对 human-written 在 `SprBench-Vrf` 上提升 `+21.50`
* • 同组在 `Hard` 上提升 `+12.50`
* • 同组在 OOD `WikiTQ` 上也有 `+4.56`

这说明它不是只在训练域里做微调，而是真的把技能进一步打磨得更稳了。

### 4. 从零创建 skill 也能有效，甚至部分场景超过人工技能

更有意思的是 `Creation` 模式。

比如由 35B 模型作者生成的 `Creation +Error`，给 122B 用户模型使用时，在 `WikiTQ` 上相对 parametric baseline 直接涨了：

* • `+57.65`

这是全表最抓眼球的数字之一。

它说明：

> **只要轨迹证据足够扎实，即便从一个几乎无效的 parametric 草稿出发，也能蒸馏出真正可用的技能。**

为了证明这不是“只会修 Excel 技能”的方法，论文还补了两个跨域实验。

### 1. 数学推理

### 2. 视觉问答 DocVQA

也就是说，“task execution capability”和“skill authoring capability”并不是一回事。

## 八、分析实验里最重要的三个结论

主结果说明它有效，但真正让我觉得这篇论文扎实的，是后面的分析实验。
因为作者不只是告诉你“涨了”，还在回答“为什么是这样涨的”。

### 1. 并行蒸馏优于顺序编辑，而且快得多

Table 4

*原论文 Table 4：并行 consolidation 与顺序式编辑对比。除了性能，时间差也非常明显。*

这张表很有代表性。

在 122B 用户模型上：

* • `Seq-B=4`：Vrf `59.00`
* • `Seq-B=1`：Vrf `61.83`
* • `Parallel`：Vrf `65.83`

而耗时上：

* • `Seq-B=1` 约 `60 min`
* • `Seq-B=4` 约 `15 min`
* • `Parallel` 约 `3 min`

也就是说，Trace2Skill 不是只在“更贵的配置下换来一点涨分”，而是同时赢了：

* • 质量
* • 稳定性
* • 计算效率

这进一步证明它的方法优势不是偶然，而是来自“先冻结初始 skill、再统一归纳”的结构设计。

### 2. 一份蒸馏技能，优于检索式经验银行

Table 5

*原论文 Table 5：Trace2Skill 与 ReasoningBank 风格 retrieval-memory baseline 对比。两者使用的是同一池成功/失败轨迹，但一个蒸馏成 skill，一个保留为检索记忆。*

这里作者拿它和 `ReasoningBank` 风格方法比较，很公平，因为底层都用了同一批轨迹。

结果是：

* • 对 122B 用户模型，Trace2Skill 相比 ReasoningBank 在 `Vrf / Soft / Hard` 上分别多出 `+13.8 / +7.1 / +8.2`
* • 对 35B 用户模型也全面领先

作者给出的解释我认为非常合理：

* • 检索依赖 query 和 memory 的表面相似性
* • 分布一变，召回就可能失真
* • 检索片段还要和任务上下文抢注意力
* • skill 预先注入系统提示，相当于把经验更早地整合进推理过程

换句话说：

**memory 更像临时调用的备忘录，skill 更像事先内化的工作规范。**

### 3. Agentic 失败分析，明显强于单次 LLM 总结

Table 6

*原论文 Table 6：对比 agentic error analysis 与单次 LLM error analysis。后者不能查看文件、不能反复定位根因，因此更容易把局部现象误判成原因。*

这是我个人最喜欢的一张消融表。

作者把自己的 `+Error` 分析器和一个简化版 baseline 对比：

* • baseline 只给错误轨迹做一次性 LLM 总结
* • Trace2Skill 的 `A-` 则允许多轮检查、看文件、核对 ground truth

结果是，`Agentic +Error` 在四组 Author-Mode 组合下的 `Avg` 全部更高。

尤其明显的有：

* • 122B Deepening：`40.75` vs. `28.58`
* • 35B Creation：`39.06` vs. `25.76`

论文还做了定性分析，发现单次 LLM 特别容易把一些表面报错信息过度解释成根因，甚至会在输出本来已经正确的情况下虚构失败原因。

这其实很能说明问题：

> **高质量 skill 不是“从错误里随便总结一句话”，而是要建立在被验证过的错误机制之上。**

## 九、这篇论文最值得记住的技术洞察是什么？

如果我把整篇论文压成几条 takeaway，我会保留下面这几条。

### 1. 轨迹经验不是不能用，而是不能直接用

轨迹里的 lesson 很宝贵，但它天然是局部的。
如果不做统一蒸馏，最后得到的往往不是 skill，而是碎片化 patch 集合。

### 2. skill 的价值，在于“归纳后前置”

memory 是任务来时再去找。
skill 是在任务开始前，就把规律作为工作规范写进系统。

这两者在推理时机上差别非常大。

### 3. 失败分析的质量，直接决定技能质量

如果失败原因没找对，skill 就会被错误经验污染。
Trace2Skill 在错误分析阶段投入较多 agentic 机制，这一步并不是额外装饰，而是整个方法可信的基础。

### 4. 小模型也可以写出有迁移性的 skill

论文一个很亮眼的点是：

**开源 35B 模型已经足以参与 skill evolution。**

这降低了整个方向对闭源超大模型的依赖，也让“技能蒸馏”更有机会变成可以规模化落地的工程流程。

---

## 十、我对这篇论文的几点思考：它还有哪些不足？

这篇论文整体很强，但如果站在“下一步还能怎么做”的角度，我觉得至少还有五个明显空间。

### 1. 目前仍然偏单技能视角，离大规模 skill ecology 还有距离

Trace2Skill 当前更像是在回答：

**“如何把一个领域内的一份 skill 写得更好？”**

但真实 Agent 系统往往不是只有一份 skill，而是：

* • 多 skill 共存
* • skill 之间存在边界重叠
* • 不同 skill 有不同粒度和调用时机

所以未来一个非常值得做的方向是：

**从 single-skill evolution 走向 multi-skill ecosystem evolution。**

也就是不仅要会蒸馏一份技能，还要会：

* • 自动拆 skill 边界
* • 自动发现冲突技能
* • 自动决定哪些规则应该进入主 skill，哪些应被拆成独立子 skill

### 2. 它对“高频规律更可信”的假设是合理的，但可能压制长尾关键经验

Trace2Skill 在 merge 阶段会更偏向保留高支持度 patch。
这对获得稳健 SOP 很有效，但也带来一个潜在问题：

**某些低频但致命的 edge case，可能被当作偶然现象弱化掉。**

尤其在高风险场景里，少见错误未必不重要。

未来可以考虑：

* • 给 patch 增加“风险权重”而不只看频次
* • 让低频但高危经验进入专门的 exception / safety 层
* • 建立主 skill 与 long-tail reference 的分层蒸馏机制

### 3. 现在的 merge 还是偏“文本层归纳”，未来可以更结构化

虽然论文已经在做冲突检测和格式校验，但 skill 本体依然主要是文本主导。

未来一个很自然的升级方向是：

* • 把 SOP 结构显式建模成流程图或状态机
* • 把工具使用约束抽成 schema-aware rules
* • 把 patch 合并升级成“程序化语义合并”

这样做的好处是：

* • 更容易做一致性检查
* • 更容易验证某条新规则是否和旧规则矛盾
* • 更容易跨模型、跨 Agent 框架迁移

### 4. 当前实验已经覆盖 spreadsheet、math、VQA，但仍缺少更重交互环境

这篇论文的跨域验证已经不错了，但如果未来想把结论推得更远，还需要在更复杂环境里继续检验，比如：

* • GUI agent
* • web agent
* • 软件工程多文件修复
* • 长时程科研 workflow

因为这些场景里，skill 的表现不只取决于规则写得好不好，还取决于：

* • 环境反馈是否稀疏
* • 中间状态是否容易观测
* • 执行链是否足够长

如果 Trace2Skill 在这些环境里依然稳定，那它的说服力会再上一个台阶。

### 5. 它证明了“技能可迁移”，但还没完全回答“技能何时失效”

论文最亮眼的贡献之一是跨模型迁移，但我更想继续知道的是：

* • skill 迁移的上限在哪里？
* • 什么时候会发生上下文错配？
* • 哪类规则是模型无关的，哪类规则是 harness 相关的？

未来可以考虑建立一套更细的 skill 诊断维度，比如：

* • 模型无关规则
* • 工具链依赖规则
* • 环境版本依赖规则
* • prompt 风格依赖规则

把“技能有效”进一步拆成“为什么有效、在哪有效、在哪开始失效”，这个方向会非常有价值。

## 十一、如果沿着这篇论文继续做，还能有哪些创新？

如果把 Trace2Skill 当成一个起点，我觉得后面至少有几条很值得追的路线。

### 1. 做成持续在线但分阶段蒸馏的 hybrid 框架

不是每来一条轨迹就立刻改 skill，
也不是永远离线攒一大批再统一蒸馏。

更理想的方式可能是：

* • 在线收集 patch 候选
* • 周期性批处理蒸馏
* • 在关键性能下降时触发 targeted re-distillation

这会让 Trace2Skill 更接近真实生产环境。

### 2. 引入 skill-level verifier

现在系统主要验证 patch 和技能格式，以及看最终任务是否涨分。
未来可以加入专门的 `skill verifier`，自动检查：

* • 新规则是否和旧规则冲突
* • 是否存在重复表述
* • 是否违反工具接口事实
* • 是否引入了无法执行的步骤

这会让 skill 演化从“经验归纳”进一步迈向“可验证知识工程”。

### 3. 做跨任务族的 meta-skill distillation

目前论文更像在一个领域内写一份强 skill。
但未来完全可以继续往上走一层，蒸馏出更抽象的 meta-skill，比如：

* • 如何系统验证文件写回
* • 如何在工具链中做显式 read-back
* • 如何处理多步编辑中的状态漂移

这些不属于某个单一 benchmark，却可能在很多 Agent 任务里复用。

### 4. 让 skill 不只写给 Agent，也写给 skill author agent

Trace2Skill 已经说明“会做任务”和“会写技能”是两种能力。
下一步完全可以把“如何分析轨迹、如何提出 patch、如何识别高价值 lesson”本身也做成上层元技能。

也就是说：

**不仅演化任务技能，还演化‘演化技能的技能’。**

这会是一个非常有意思的方向。

---

## 结语

我觉得 Trace2Skill 最有价值的地方，不只是它把分数做上去了，而是它把一个长期被模糊处理的问题讲清楚了：

**Agent 的经验，只有在被正确归纳、正确压缩、正确组织之后，才有资格叫 skill。**

很多系统已经会“记住经历”，但距离“写出一份可迁移、可维护、可复用的技能”其实还差很远。
Trace2Skill 做的，就是把这一步补上。

从趋势上看，这篇论文也很像一个信号：

* • 未来 Agent 能力竞争，未必只在模型参数
* • 也未必只在工具数量
* • 很可能越来越体现在“外部化经验资产”的质量上

而 skill，可能就是其中最重要的一种形态。
