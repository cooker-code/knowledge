Agent Skills 万千应用 · 第05篇

▌ 01｜场景痛点：周报最难的不是写，而是写得有价值

周五下午，项目群里还在刷消息，会议纪要散在文档里，Excel 里有一堆进度数据，脑子里却只剩一句话：这周又要写周报了。

很多人的周报都像流水账：做了什么、开了什么会、修了几个问题。老板看完还是不知道项目是否健康、哪里需要支持、下周能不能交付。

所以周报/月报真正难的不是“凑字数”，而是把零散事实整理成一份能辅助决策的工作总结。

这就是第 05 篇要讲的 Skill：周报/月报 Skill。它让 Agent 不只是帮你写文字，而是按固定逻辑把事实、进展、风险和计划整理成汇报。

▌ 02｜Skill 实操效果：同样信息，输出完全不同

案例一：研发项目周报。你只给 Agent 一段散乱记录：本周做了支付接口联调，修了几个 Bug，参加两次会议，测试环境还有问题。

普通提示词通常会输出：“本周完成接口联调，修复 Bug，参加会议，下周继续开发。”这句话没错，但老板看不出重点。

使用周报/月报 Skill 后，输出会变成：本周目标是支付接口联调；当前完成 80%；阻断 Bug 从 7 个降到 2 个；风险是测试环境不稳定；需要协调测试同事排查；下周计划是完成回归并提交验收版本。

案例二：销售月报。普通写法是“本月拜访 12 家客户，新增 36 条线索，成交 3 单”。Skill 输出会进一步补齐：线索来源、转化率、成交金额、流失原因、下月重点客户清单，以及需要老板拍板的价格策略。

它的价值不是把话写长，而是把结果写清楚：进展在哪、风险在哪、下一步要谁支持。

▌ 03｜Skill 简介：它是什么，包含哪些文件

周报/月报 Skill 是一个面向中文办公场景的工作总结能力包，适合研发、销售、运营、项目管理、个人复盘等场景。

目前网上有通用写作类、文档类 Skill，但没有一个完全贴合“中文职场周报/月报”的标准包，所以本篇我直接生成了一份中文增强版：work-summary-skill-zh-v1.zip。

它的目录结构很简单：SKILL.md 负责定义触发条件、输入要求和输出规则；references/ 存放周报模板、月报模板、老板视角检查清单；scripts/ 提供把任务记录整理成初稿的辅助脚本；assets/ 存放可复用模板。

它解决的真实问题是：把聊天记录、任务清单、会议纪要、Excel 数据，整理成一份结构清晰、重点明确、老板愿意看的工作总结。

▌ 04｜核心机制：SKILL.md 怎么设计

这个 Skill 的关键不是“帮我写周报”，而是先要求 Agent 做判断。

第一层是渐进式加载：启动时只读取 Skill 名称和说明；当用户要求写周报、月报、项目总结、复盘时，再加载完整 SKILL.md；如果是销售月报，再读取销售模板；如果是研发周报，再读取项目模板。

第二层是固定工作流：收集事实、归类去重、提炼价值、暴露风险、形成计划、语气检查。

第三层是输出约束：少写空话，多写数据；少写“已完成”，多写“完成后产生什么影响”；风险不能藏起来，需要明确是否需要资源支持。

这让 Agent 从“文案生成器”变成“工作汇报整理员”。

▌ 05｜使用方式：ChatGPT / Claude / Hermes / OpenClaw 都能用

在 ChatGPT 中，可以把 Skill 包作为项目资料上传，再上传任务记录、会议纪要或数据表，让它按周报/月报 Skill 输出。

在 Claude 中，可以把它作为 Skills 上传，或在 Claude Code 环境里放入对应 Skills 目录。

在 Hermes 和 OpenClaw 中，可以把文件夹解压到本地 skills 目录或项目上下文中，再用任务命令触发：请用 work-summary-skill 生成本周项目周报。

使用时最好提供三类输入：本周做了什么、关键数据是什么、有哪些风险或需要支持。输入越真实，输出越像可交付总结。

▌ 06｜避坑指南：周报最容易写坏的 5 件事

第一，不要让 Agent 编事实。没有数据就写“未提供”，不要为了好看而补数字。

第二，不要写成流水账。老板不关心你开了几次会，更关心这些动作带来了什么结果。

第三，风险不要模糊。比如“进度略有延迟”不如写清楚：接口联调延期 2 天，原因是测试环境不稳定。

第四，月报不要只写本月做了什么，还要写下月重点和需要决策的事项。

第五，注意脱敏。客户名称、金额、内部问题，如果要发到公开场景，必须先做脱敏处理。
