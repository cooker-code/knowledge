# Skills 的调试：你的 Skill 为什么"时灵时不灵"

> "它昨天还好好的，今天怎么就不动了？"

## 从一个真实的抓狂时刻开始

你写了一个总结会议纪要的 Skill，测试了两次，很好用，心满意足地关掉了编辑器。

第二天开会，会议结束，你把录音转录文本粘进去，输入"帮我整理一下"——

OpenClaw 给了你一段普通的总结。没有你在 Skill 里指定的"决策项 / 待办 / 遗留问题"三段式结构。就是一段话，跟没装 Skill 一样。

你再试一次，换了个说法："会议纪要整理"——这次触发了，格式对了。

再试一次，原来那句"帮我整理一下"——又没触发。

这不是 bug，这是你的 Skill 写了一半。

## 问题一：它根本没触发

**最常见的原因：你写触发条件时，用的是你"描述任务"的语言，不是你"真实开口"的语言。**

你在 description 里写的是：

> 当用户需要整理会议内容、提取关键信息时激活

但你实际用的时候，从来不会这么说话。你说的是"帮我整理一下"、"这个会讲了啥"、"刚才那个会你帮我记一下"。

这两种表达之间的语义距离，比你想象的要大。

**怎么修：** 打开你上周真实用 OpenClaw 的记录，把你贴进去的那第一句话原原本本复制出来，放进 description 的触发词里。不要润色，不要书面化，就是那句话。

如果你刚装好还没有记录，做一个测试：把这个任务发给一个不知道你 Skill 存在的同事，让他帮你处理，看他问你"你想要什么"时的第一句话——那句话，比你自己想的任何触发词都准。

## 问题二：触发了，但不是每次都触发

这个更烦。因为它"大部分时候好使"，你会开始怀疑是不是自己哪次表达有问题，而不是 Skill 有问题。

**真实场景：** 你有两个 Skill，一个是"总结文章"，一个是"提炼行动项"。两个 description 里都写了"帮我看看这篇"。结果就是——每次 OpenClaw 在抢占，今天心情好走这个，明天走那个。

检查方法很简单：把你所有 Skill 的 description 里的触发词全部列出来放在一起，肉眼扫一遍，找重叠的词。重叠的地方，就是打架的地方。

解法不是删掉重叠词，而是在其中一个 Skill 里加一个"排他信号"。

比如"提炼行动项"里加上：`用户说"有哪些要做的"、"下一步怎么办"、"谁负责什么"时激活——纯粹的内容理解需求不在此列`。

这句"不在此列"，就是在告诉 OpenClaw：你碰到的如果不是这种情况，别来找我。

## 问题三：触发了，但输出每次都不一样

这是最难察觉的问题，因为每次输出看起来都"差不多对"。

**一个具体的例子：**

你的简历分析 Skill 里写着"给出三条修改建议"。但跑了五次，你发现：

* 第一次，建议写得很具体："把'参与项目协作'改成'主导跨部门需求对齐'"
* 第二次，建议变成了："建议突出你的领导力和跨部门经验"
* 第三次，建议只有两条，第三条变成了一个问题："你是否有量化的项目结果可以补充？"

同一个 Skill，同一份简历，三次结果完全不同颗粒度。

**为什么会这样：** 你只说了"给三条建议"，没说建议长什么样。

**修法只需要一步：** 在格式要求下面加一个"示例输出"，哪怕是假的、简短的，也要有。比如：

```
示例：第2条工作经历，把"参与多个项目协作"改为"主导跨部门需求对齐，推动 xx 项目按期交付"——动词从"参与"换成"主导"，直接命中 JD 关键词。
```

有了这个例子，后续每次输出的颗粒度、结构、措辞风格，都会向这个样本对齐。一个例子，抵得上你写半页格式规范。

## 还有一种情况，你可能从没想到要测试

给你的 Skill 一个**不完整的输入**，看它怎么反应。

还是简历分析 Skill：如果用户只粘了 JD，没粘简历，会发生什么？

大概率是：它会自己脑补一份"通用简历"，然后给你一份看起来有模有样、其实完全没用的分析报告。不会报错，不会提示，就是静静地给你输出一堆废话。

你需要在 Skill 里加一行：

> 如果用户没有提供简历，不要进行匹配分析。只输出 JD 核心要求的拆解，并提示用户补充简历。

这一行，能省掉你很多次"这分析怎么这么奇怪"的困惑时刻。

## 调试的节奏

不要等到 Skill"坏掉"了再调试。

装好一个新 Skill 之后，用真实任务跑五次——不是测试用的假材料，是你这周实际会遇到的东西。五次跑完，你会发现 80% 的问题都在同一个地方。改一次，再跑三次确认，基本就稳了。

Skill 第一次写完，只是"能用"。调试之后，才是"你的"。
