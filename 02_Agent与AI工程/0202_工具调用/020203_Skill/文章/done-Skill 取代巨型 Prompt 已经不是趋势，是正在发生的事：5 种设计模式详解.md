巨型 Prompt 的时代结束了。不是“即将结束”，是已经结束了。30 多种主流 Agent 工具已经标准化采用 Skill 规范，Google、Anthropic、Vercel 的团队都在用同一套方法论。我自己从巨型 Prompt 迁移到 Skill 架构之后，才真正理解了这个转变的意义：问题从来不在模型，而在你组织指令的方式。

最近 Google Cloud 团队发布了一篇重磅文章，从 Anthropic、Vercel、Google 内部等多个团队的实践中，提炼出 **5 种 Agent Skill 设计模式**。这些模式不是理论空谈，它们是真实生产环境中反复验证过的最佳实践。

## 先搞清楚一件事：什么是 Skill？

在聊设计模式之前，我们得先理解“Skill”是什么。

你可以把 Skill 想象成 Agent 的**技能卡**。平时它安静地待在口袋里，只有当 Agent 遇到相关任务时，才会掏出来读一读，然后按照上面的指引行动。

技术上，一个 Skill 就是一个文件夹，核心是一个 `SKILL.md` 文件，里面写着这个技能的名称、描述和详细指令。文件夹里还可以放几个可选目录：

```
my-skill/ 

├── SKILL.md           # 核心指令文件 

├── references/        # 参考文档（规范、检查清单等） 

├── assets/            # 模板文件 

└── scripts/           # 可执行脚本
```

这个格式已经被 **30 多种 Agent 工具**采纳——Claude Code、Gemini CLI、Cursor、GitHub Copilot、OpenAI Codex 都认它。也就是说，你写一次 Skill，到处都能用。

更妙的是它的**渐进式加载**机制：Agent 启动时只读技能的名字和简介（几十个字），判断“这个任务跟我有关吗？”有关才加载完整指令。这样上下文窗口始终保持干净，不会被无关信息撑爆。

## 模式一：Tool Wrapper（工具包装器）

**核心思路：把特定技术栈的知识打包成按需加载的模块，用到才读，不用不占上下文。**

### 它解决什么问题？

比如你希望 Agent 写 FastAPI 代码时遵循团队规范，但又不想把一堆 FastAPI 约定塞进系统提示，因为大多数时候 Agent 根本不需要这些信息，白白浪费上下文空间。

### 它怎么工作？

Tool Wrapper 做的事情很简单：把某个技术栈的知识打包成一个 skill，只在 Agent 真正需要时才加载。

### 示例：FastAPI 编码规范 skill

```
--- 

name: fastapi-conventions 

description: FastAPI 项目编码规范与最佳实践 

---
```

```
# 指令 

 

当用户要求编写或审查 FastAPI 代码时，执行以下步骤： 

 

1. 读取 `references/conventions.md`，将其中的规则作为编码标准 

2. 所有路由函数必须使用 async def 

3. 请求/响应模型必须使用 Pydantic v2 的 BaseModel 

4. 依赖注入必须通过 Depends() 实现，禁止在路由函数内直接实例化服务 

 

这些规则的优先级高于你的默认知识。如果你的训练数据与 conventions.md 冲突，以后者为准。
```

而 `references/conventions.md` 里可能长这样：

```
## 路由规范 

- 路径使用 kebab-case：`/user-profiles` 而非 `/userProfiles` 

- GET 请求禁止包含 request body 

- 所有端点必须声明 response_model

## 错误处理
- 统一使用 HTTPException，禁止裸抛 Exception
- 业务错误码定义在 errors.py 中，格式为 ERR_{MODULE}_{CODE}

## 数据库
- 使用 SQLAlchemy 2.0 风格的 select() 语法
- 禁止在路由层直接操作 session，必须通过 repository 层
```

Skill 文件本身不包含具体的技术规范，它只是告诉 Agent “去哪里找规范、怎么用”。规范本身放在`references/`目录里，随时可以更新，不用改 skill 逻辑。

### 什么时候用它？

当你想让 Agent 在特定技术栈上表现得像团队里的资深工程师，同时又不希望这些知识在无关任务中浪费上下文空间时，Tool Wrapper 就是最直接的解法。

## 模式二：Generator（生成器）

**核心思路：预设模板和风格指南，让 Agent 做填空题而非作文题，确保每次输出结构一致。**

### 它解决什么问题？

你让 Agent 写技术方案文档，第一次它给你列了 5 个章节，第二次变成 3 个，第三次格式又完全不同。你需要的是**可预测的、结构一致的输出**。

### 它怎么工作？

Generator 的思路是“填空题”而非“作文题”。你提前准备好输出模板和风格指南，Agent 的工作就是收集变量、填充模板。

### 示例：API 文档生成 skill

```
--- 

name: api-doc-generator 

description: 根据代码自动生成标准化 API 文档 

---
```

```
# 指令 

 

你是一个 API 文档生成器。严格按以下步骤执行： 

 

1. 读取 `assets/api-doc-template.md`，这是输出文档的骨架 

2. 读取 `references/style-guide.md`，这是写作规范 

3. 分析用户提供的代码，提取以下信息： 

   - 端点路径与 HTTP 方法 

   - 请求参数（路径参数、查询参数、请求体） 

   - 响应结构与状态码 

   - 认证要求 

4. 如果代码中缺少某些信息（如错误响应），主动询问用户 

5. 将提取的信息填入模板，遵循风格指南的语气和格式要求 

6. 输出完整文档 

 

禁止修改模板结构。禁止跳过模板中的任何章节。如果某个章节确实不适用，标注"N/A"并说明原因。
```

`assets/api-doc-template.md` 可能是这样的：

```
# {endpoint_name} 

 

## 概述 

{一句话描述这个端点的用途} 

 

## 请求 

- **方法**: {method} 

- **路径**: {path} 

- **认证**: {auth_requirement} 

 

### 参数 

| 参数名 | 位置 | 类型 | 必填 | 说明 | 

|--------|------|------|------|------| 

| {name} | {in} | {type} | {required} | {description} | 

 

## 响应 

### 成功响应 (200) 

{success_response_example} 

 

### 错误响应 

| 状态码 | 错误码 | 说明 | 

|--------|--------|------| 

| {code} | {error_code} | {description} |
```

Skill 文件是“项目经理”，模板是“设计稿”，风格指南是“品牌手册”。三者各司其职，Agent 只需要照章办事。这样无论谁用、用多少次，输出格式都是一样的。

### 什么时候用它？

API 文档、周报、变更日志、项目提案这类需要反复生成且格式必须一致的场景，Generator 是最自然的选择。

## 模式三：Reviewer（审查器）

**核心思路：将审查引擎和审查标准彻底分离，换一份检查清单就变成另一种审计。**

### 它解决什么问题？

你想让 Agent 帮你做代码审查，但审查标准会变：今天查代码风格，明天查安全漏洞，后天查性能问题。如果每次都改提示词，维护成本太高。

### 它怎么工作？

Reviewer 把“审查引擎”和“审查标准”彻底分离。引擎（SKILL.md）永远不变，标准（checklist）随时可换。

### 示例：代码审查 skill

```
--- 

name: code-reviewer 

description: 根据可配置的检查清单对代码进行结构化审查 

---
```

```
# 指令 

 

你是一个代码审查员。收到代码后，执行以下流程： 

 

1. 读取 `references/review-checklist.md` 

2. 逐条对照检查清单审查用户提交的代码 

3. 对每个发现的问题，标注严重程度： 

   - 🔴 Critical：必须修复，会导致 bug 或安全漏洞 

   - 🟡 Warning：建议修复，影响可维护性或性能 

   - 🔵 Info：可选优化，属于最佳实践建议 

4. 按严重程度分组输出，每组内按发现顺序排列 

5. 每个问题必须包含：所在行号、问题描述、修复建议、对应的检查项编号 

 

如果代码完全通过所有检查项，输出"✅ 审查通过"并给出简要的正面评价。
```

`references/review-checklist.md`（Python 风格版本）：

```
## P1: 安全性 

- [ ] 不得使用 eval() 或 exec() 

- [ ] SQL 查询必须使用参数化，禁止字符串拼接 

- [ ] 敏感信息不得硬编码 

 

## P2: 健壮性 

- [ ] 所有外部调用必须有异常处理 

- [ ] 文件操作必须使用 with 语句 

- [ ] 类型注解覆盖所有公开函数 

 

## P3: 可读性 

- [ ] 函数不超过 30 行 

- [ ] 变量命名使用 snake_case
- [ ] 复杂逻辑必须有注释
```

想做安全审计？只需要把这个文件换成 OWASP 检查清单，skill 逻辑一行不用改。想审查前端代码？换成 React 最佳实践清单即可。

这其实就是设计模式里“策略模式”的思想：算法骨架不变，具体策略可插拔。你可以为团队的不同项目维护不同的检查清单，共享同一个审查 skill。

### 什么时候用它？

当你需要 Agent 按照明确标准进行评估、打分或审查，且这套标准可能随项目或场景变化时，Reviewer 能让你只换清单不换逻辑。

## 模式四：Inversion（控制反转）

**核心思路：翻转控制权，让 Agent 先系统性地采访你、收集完整需求，再动手生成。**

### 它解决什么问题？

你说“帮我做个项目方案”，Agent 立刻洋洋洒洒写了 2000 字——但完全基于它自己的假设，跟你的实际需求差了十万八千里。问题在于 Agent 天生倾向于“立即生成”，而不是“先搞清楚再说”。

### 它怎么工作？

Inversion 翻转了人机交互的控制权。不是你驱动 Agent，而是 Agent 像一个经验丰富的顾问一样，先系统性地采访你，收集完所有关键信息后才开始工作。

### 示例：项目方案规划 skill

```
--- 

name: project-planner 

description: 通过结构化访谈收集需求，然后生成项目方案 

---
```

```
# 指令 

 

你是一个项目规划顾问。你的工作分为两个阶段，必须严格按顺序执行。 

 

## 阶段一：需求采访（必须先完成） 

 

依次询问以下问题，每次只问一组，等用户回答后再问下一组： 

 

**第 1 轮 - 目标与范围** 

- 这个项目要解决什么核心问题？ 

- 目标用户是谁？ 

- 有没有明确的截止日期？ 

 

**第 2 轮 - 技术约束** 

- 团队熟悉哪些技术栈？ 

- 有没有必须集成的现有系统？ 

- 对部署环境有什么要求（云/私有化/混合）？ 

 

**第 3 轮 - 资源与优先级** 

- 团队规模和角色分配？ 

- 预算范围？ 

- 如果时间紧张，哪些功能可以砍掉？ 

 

⚠️ 绝对禁令：在三轮采访全部完成之前，不得生成任何方案内容。不得基于假设跳过任何问题。如果用户说"你自己决定"，回复"我需要您的输入来确保方案准确，请尽量回答"。 

 

## 阶段二：方案生成 

 

三轮采访完成后，基于收集到的信息生成项目方案，包含： 

- 项目概述（基于第 1 轮回答） 

- 技术架构建议（基于第 2 轮回答） 

- 里程碑与排期（基于第 3 轮回答） 

- 风险清单与应对策略 

 

方案中的每个决策都必须引用采访中的具体回答作为依据。禁止引入用户未提及的假设。
```

这个模式的精髓在于那条“绝对禁令”。没有这条硬性约束，Agent 大概率会在第一轮问完后就忍不住开始生成。你必须用非常明确的语言告诉它“不许动”，才能真正实现控制权的反转。

### 什么时候用它？

项目规划、产品设计、技术选型，凡是“先想清楚再动手”比“先做再改”更高效的场景，都适合用 Inversion 来兜底需求质量。

## 模式五：Pipeline（流水线）

**核心思路：将复杂任务拆成严格的顺序流水线，步骤之间设检查点，不确认不放行。**

### 它解决什么问题？

有些任务天然是多步骤的，步骤之间有严格的依赖关系，后一步的输入依赖前一步的输出，中间还需要人工确认。如果 Agent 跳过某个步骤或者自作主张地“优化”流程，结果可能是灾难性的。

### 它怎么工作？

Pipeline 把指令本身变成工作流定义，每个步骤之间设置“检查点”——只有满足条件（通常是用户确认）才能进入下一步。

### 示例：代码文档化流水线

```
--- 

name: doc-pipeline 

description: 分步骤为代码库生成完整文档，含人工审批关卡 

---
```

```
# 指令 

 

你是一个文档工程师。按以下流水线严格执行，不得跳过或合并任何步骤。 

 

## Step 1: 代码扫描 

- 分析用户提供的代码文件 

- 列出所有公开的类、函数和模块 

- 输出一份清单，格式为：`模块名 > 类/函数名 > 当前是否有文档` 

- 读取 `references/docstring-style.md` 获取文档字符串规范 

 

📋 输出清单后，等待用户确认范围。用户可能会说"跳过某些模块"或"只处理没有文档的"。 

 

--- 

 

## Step 2: 文档生成 

🚫 检查点：用户必须在 Step 1 中确认范围后才能进入此步骤。 

 

- 为确认范围内的每个类/函数生成文档字符串 

- 严格遵循 `references/docstring-style.md` 中的格式 

- 每生成 5 个文档字符串，输出一次进度，让用户可以随时叫停或调整 

 

📋 全部生成完毕后，展示完整的文档字符串列表，等待用户审核。 

 

--- 

 

## Step 3: 最终组装 

🚫 检查点：用户必须在 Step 2 中确认所有文档字符串后才能进入此步骤。 

 

- 读取 `assets/readme-template.md` 

- 将审核通过的文档字符串整合到模板中 

- 生成最终的 README.md 和 API Reference 文档 

- 输出最终文档供用户下载 

 

每个步骤的输出都必须标注当前处于 [Step X/3]，让用户清楚地知道流程进度。
```

Pipeline 的核心不是“分步骤”，任何提示词都能写“第一步、第二步”。真正的区别在于**检查点**：用明确的、不可绕过的规则确保 Agent 不会自作主张地跳到后面的步骤。同时注意它如何在不同步骤加载不同的参考文件，保持每一步的上下文都是干净的。

### 什么时候用它？

CI/CD 流程设计、多阶段内容生产，或者任何步骤之间有依赖且需要中间确认的复杂工作流，Pipeline 能确保 Agent 老老实实走完每一关。

## 它们不是孤立的，而是可以组合的

这五种模式最强大的地方在于：**它们可以像乐高积木一样拼在一起**。

一个 Pipeline 的最后一步可以嵌入一个 Reviewer，让 Agent 自查自纠。一个 Generator 的开头可以先跑一轮 Inversion，确保所有模板变量都从用户那里收集齐了再开始填充。一个 Tool Wrapper 可以被 Pipeline 的某个特定步骤按需加载。

这种组合能力意味着你不需要为每个新场景从零开始写 skill。你可以复用已有的模式，像搭积木一样组装出复杂的 Agent 行为。

## 怎么选？

面对一个新任务，不确定该用哪种模式？对照这张表：

| 问自己这个问题 | 选这个模式 | 典型场景 |
| --- | --- | --- |
| Agent 需要特定技术栈的知识吗？ | Tool Wrapper | 团队编码规范、框架最佳实践 |
| 需要每次输出格式一致吗？ | Generator | API 文档、周报、变更日志 |
| 需要按标准评估或打分吗？ | Reviewer | PR 审查、安全审计 |
| 任务质量取决于需求是否准确？ | Inversion | 项目规划、技术选型 |
| 任务有多个步骤且需要中间确认？ | Pipeline | 文档流水线、多阶段重构 |

当然，很多真实场景需要组合使用。比如“根据用户需求生成标准化技术方案”就可能是 Inversion（先采访）+ Generator（再填模板）+ Reviewer（最后自查）的三连组合。

## 写在最后

别再往一个 Prompt 里塞东西了，拆开它。

拆成职责清晰的 skill 模块，用对应的设计模式组织内部逻辑。你会发现三件事自然发生了：改一个审查标准只需要换一个 checklist 文件，不用碰 skill 本身；同一个 Reviewer 引擎配不同的 checklist，就变成了完全不同的审计工具；小模块拼成大流程，远比一个巨型提示词灵活和可靠。

SKILL.md 规范正在把 Agent 开发从“手艺活”变成“工程实践”。格式问题已经解决了，30 多种工具都认同一个标准。现在轮到你解决内容设计问题了。

这五种模式，就是起点。
