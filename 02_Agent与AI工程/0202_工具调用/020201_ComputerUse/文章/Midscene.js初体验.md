## 引言

在现代软件开发中，UI 自动化测试一直是个令人头疼的问题。传统的自动化测试需要编写大量的选择器代码，维护成本高，而且界面一旦变化就需要大量修改。更糟糕的是，跨平台测试往往需要为 Web、Android、iOS 分别编写不同的测试脚本。

如果能用自然语言描述测试场景，让 AI 自动理解界面并执行操作，会不会让测试变得更简单？Midscene.js 正是这样一个革命性的工具——它通过 AI 视觉能力理解用户界面，让你可以用自然语言编写自动化测试脚本。

## 什么是 Midscene.js

Midscene.js 是一个开源的 AI 驱动操作助手，专为 Web、移动端自动化和测试场景设计。它的核心创新在于：无需编写复杂的选择器代码，只需用自然语言描述你想要测试的功能，AI 就能自动识别界面元素并执行相应操作。

### 核心特性详解

#### 1. 自然语言驱动的自动化

这是 Midscene.js 最具突破性的特性。你不再需要写 `document.querySelector('#submit-button')` 这样的代码，而是可以直接说"点击提交按钮"。

**支持两种脚本格式：**

* **JavaScript SDK**：适合集成到现有测试框架中，提供完整的编程能力
* **YAML 格式**：更简洁直观，适合快速编写测试场景

**工作原理：**  
Midscene.js 会截取当前界面截图，通过视觉 AI 模型理解界面结构和元素含义，然后根据你的自然语言描述找到对应元素并执行操作。这意味着即使开发团队修改了元素的 class 名称或 id，只要视觉效果不变，测试脚本通常不需要修改。

#### 2. 全平台覆盖

**Web 自动化：**

* 集成 Puppeteer 和 Playwright，无缝对接现有工具链
* 支持桌面浏览器桥接模式，可以直接控制你正在使用的浏览器

**Android 自动化：**

* 通过 adb（Android Debug Bridge）连接本地设备
* 支持真机和模拟器

**iOS 自动化：**

* 基于 WebDriverAgent 协议控制设备
* 支持真机调试

这种跨平台能力意味着你可以用统一的方式编写测试脚本，大大降低了多端测试的维护成本。

#### 3. 丰富的开发者工具

**可视化测试报告：**  
生成详细的执行报告，包含每一步的截图和操作说明，方便问题定位和回溯。

**Playground 调试环境：**  
提供交互式调试界面，可以实时查看 AI 的识别结果，调整测试步骤。

**脚本缓存机制：**  
智能缓存重复场景的识别结果，提升执行效率，减少 API 调用成本。

**MCP 协议支持：**  
开放 Model Context Protocol，允许其他 MCP 客户端直接调用 Midscene 的能力（目前支持 Web 和 Android）。

#### 4. 三大 API 体系

**交互 API：**  
模拟用户操作，如点击、输入、滚动等。

**数据提取 API：**  
从界面或 DOM 结构中提取所需数据，支持结构化输出。

**实用 API：**

* `aiAssert()`：智能断言，用自然语言描述预期结果
* `aiLocate()`：定位元素，返回元素位置信息
* `aiWaitFor()`：智能等待，直到某个条件满足

## 模型选择：为什么需要视觉模型

### 视觉能力的重要性

传统的自动化测试依赖于 DOM 结构分析，但这种方式存在明显局限：

* 无法处理 Canvas 绘制的内容
* 难以理解复杂的 CSS 布局
* 对动态生成的内容识别困难

而支持视觉的 AI 模型能够像人类一样"看"界面，理解元素的视觉含义和位置关系。这使得 Midscene.js 能够处理各种复杂场景，包括图表、地图、自定义绘图等传统方法难以自动化的界面。

### 推荐模型

**主流选择：**

* **Gemini**：Google 的多模态模型，视觉理解能力强
* **Claude**：Anthropic 的模型，在复杂推理场景表现优秀
* **GPT-4 Vision**：OpenAI 的视觉模型，综合能力均衡

**本文推荐：Doubao-1.5-UI-TARS**

UI-TARS（User Interface - Task Automation and Reasoning System）是字节跳动开源的专门面向图形界面交互的 Agent 模型。它的优势在于：

* **原生 GUI 优化**：专门针对界面理解和操作场景训练
* **类人交互能力**：具备感知、推理和行动的完整能力链
* **本地化支持**：对中文界面和中文指令支持更好
* **高性价比**：在火山引擎平台调用成本相对较低

## 实战：使用 Midscene.js 生成自动化测试

本节将演示如何通过 MCP（Model Context Protocol）方式使用 Midscene.js。我们将使用 Zed 编辑器作为 MCP 客户端，连接 Midscene 服务，并自动生成 Playwright 测试用例。

### 前置准备

**环境要求：**

* Node.js 16+
* 支持 MCP 的客户端（本文使用 Zed 编辑器）
* 可访问的 AI 模型 API（本文使用火山引擎的豆包模型）

**依赖说明：**  
使用 `pnpx @midscene/mcp` 可以直接运行 Midscene MCP 服务，无需手动安装。pnpx 会自动下载并执行最新版本。

### 配置 MCP Server

在 Zed 编辑器的 MCP 配置中添加以下内容：

```
"mcp-midscene": { 

  "enabled": true, 

  "command": "cmd", 

  "args": ["/c", "pnpx", "@midscene/mcp"], 

  "env": { 

    "MIDSCENE_MODEL_NAME": "doubao-seed-1-6-vision-250815", 

    "OPENAI_API_KEY": "", 

    "MCP_SERVER_REQUEST_TIMEOUT": "800000", 

    "MIDSCENE_USE_DOUBAO_VISION": "1", 

    "OPENAI_BASE_URL": "https://ark.cn-beijing.volces.com/api/v3" 

  } 

}
```

**配置参数说明：**

* `command` 和 `args`：指定通过 cmd 执行 pnpx 命令启动 MCP 服务
* `MIDSCENE_MODEL_NAME`：指定使用的模型名称，这里使用豆包的 UI-TARS 模型
* `OPENAI_API_KEY`：填入你的 API 密钥（火山引擎控制台获取）
* `MCP_SERVER_REQUEST_TIMEOUT`：设置请求超时时间为 800 秒，因为视觉模型推理可能需要较长时间
* `MIDSCENE_USE_DOUBAO_VISION`：启用豆包视觉模型支持
* `OPENAI_BASE_URL`：火山引擎 API 端点地址

### 启用浏览器桥接模式

为了让 Midscene 能够控制你的浏览器，需要安装并启用浏览器扩展：

1. 安装 Midscene 浏览器扩展（Chrome Web Store 搜索 “Midscene”）
2. 打开扩展，启用"桥接模式"
3. 设置为"允许连接"状态

桥接模式允许 MCP 服务通过本地连接直接控制浏览器，无需启动额外的浏览器实例。

### 生成测试用例

配置完成后，在 Zed 编辑器中输入以下自然语言指令：

```
使用 midscene 打开 https://jasminides.com/，生成网站的 Playwright 自动化测试用例，测试以下功能： 

- 文章的打开和关闭 

- tag 页的功能 

- 搜索页的功能   

- rss 页的功能 

 

将生成的用例保存到 ut.ts 文件中
```

### 执行过程展示

**初始化阶段：**

Midscene 开始连接浏览器，加载目标网站。

**界面分析阶段：**

AI 模型正在分析网站结构，识别各个功能模块和可交互元素。

**测试执行阶段：**

根据指令依次测试各个功能点，AI 自动识别按钮位置并执行点击、导航等操作。

**代码生成完成：**

最终生成完整的 Playwright 测试用例代码，并保存到指定文件。整个过程无需人工干预，AI 自动完成了从页面理解到代码生成的全流程。

## 评价与展望

### 优势分析

**1. 极大降低测试门槛**  
不需要深入了解选择器语法和页面结构，会写自然语言就能编写测试用例。这让产品经理、测试人员也能参与自动化测试编写。

**2. 维护成本低**  
基于视觉识别而非 DOM 结构，界面重构后测试脚本通常不需要修改。传统自动化测试最大的痛点——频繁的脚本维护得到了有效缓解。

**3. 跨平台统一**  
一套思路适用于 Web、Android、iOS，减少了技术栈的学习成本。

**4. 智能化程度高**  
AI 能够理解上下文，处理一些动态场景和异常情况，比硬编码的脚本更具弹性。

### 当前局限

**1. 执行速度**  
每次操作都需要 AI 推理，速度比传统选择器方式慢。对于需要大量重复执行的回归测试，可能需要考虑性能成本。

**2. 准确性依赖模型**  
识别准确率受限于底层 AI 模型能力。复杂界面或相似元素可能出现误判。建议在关键步骤添加明确的断言验证。

**3. 成本考虑**  
视觉模型 API 调用有一定费用。虽然提供了缓存机制，但大规模使用仍需评估成本。

**4. 调试复杂度**  
出现问题时，相比传统测试更难定位是指令描述问题还是 AI 理解问题。好在提供了可视化报告辅助调试。

### 适用场景

**最适合：**

* 快速原型验证和探索性测试
* UI 经常变动的项目（减少维护工作）
* 跨平台端到端测试
* 非技术人员参与的测试编写

**需谨慎：**

* 性能要求极高的场景
* 预算有限的小团队（AI API 调用费用）
* 需要精确控制每个细节的底层测试

### 未来展望

Midscene.js 代表了 UI 自动化测试的一个重要方向——让 AI 像人类一样理解和操作界面。虽然目前还存在一些限制，但随着视觉模型能力的提升和工具链的完善，这种自然语言驱动的测试方式很可能成为主流。

特别是在以下方面有很大的发展空间：

* **模型优化**：专门针对测试场景微调的模型会带来更高的准确率和更低的成本
* **混合模式**：结合传统选择器和 AI 识别的优势，在速度和智能之间取得平衡
* **协作增强**：与录制回放、可视化编辑等工具结合，形成完整的低代码测试平台

对于愿意拥抱新技术的团队，现在就是一个很好的尝试时机。从小范围试点开始，逐步探索 AI 驱动测试的最佳实践，为未来做好准备。
