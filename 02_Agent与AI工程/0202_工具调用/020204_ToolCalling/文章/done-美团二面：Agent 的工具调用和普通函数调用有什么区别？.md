面试官：你简历上写了做过 Agent 项目，说说你对 Agent 工具调用的理解。

我：Agent 工具调用就是让大模型可以调用外部工具，比如调用搜索引擎、数据库查询、API 接口这些，扩展 LLM 的能力边界。

面试官：那它和我们平时写代码时的函数调用有什么区别？不都是调用一个函数吗？

我：呃，确实都是调用函数，但 Agent 的工具调用是由大模型来决定什么时候调用、调用哪个工具。

面试官：好，那我问你三个问题。第一，大模型怎么知道有哪些工具可以调用？第二，工具调用失败了怎么办？第三，工具的参数是怎么传进去的，大模型会不会传错？

我：这个，工具列表应该是提前注册好的，失败的话，可能需要重试，参数的话，大模型会根据上下文生成。

面试官：你这是背的教材吧？我问你，普通函数调用失败了会怎样？Agent 工具调用失败了又会怎样？这两者的错误处理机制完全不同，你能说清楚吗？

我：（沉默）

今天就把这道题彻底讲清楚。

## 简要回答

Agent 工具调用和普通函数调用的核心区别在于：普通函数调用是编译时确定的、同步的、确定性的；而 Agent 工具调用是运行时决策的、异步的、带有不确定性的。

具体来说，普通函数调用由程序员在代码中写死了调用逻辑，编译器在编译时就知道要调用哪个函数、传什么参数。而 Agent 工具调用是由大模型在运行时根据用户输入和上下文动态决定的，调用哪个工具、传什么参数、甚至要不要调用，都是大模型推理出来的。

更关键的是，普通函数调用失败会直接抛异常，程序崩溃或进入异常处理流程。但 Agent 工具调用失败后，大模型可以看到错误信息，然后自己决定是重试、换个工具、还是直接告诉用户"我搞不定"。这种自主错误恢复能力是普通函数调用不具备的。

面试官最爱追问的点：工具调用的参数是怎么生成的？大模型会不会传错参数导致工具调用失败？

Agent 工具调用整体架构

## 为什么需要工具调用

大模型本身只是一个文本生成器，它的知识来自训练数据，存在三个致命局限。

第一，知识截止日期。GPT-4 的训练数据截止到 2023 年 10 月，你问它"2024 年世界杯冠军是谁"，它只能说"我不知道"或者瞎编一个答案。

第二，无法执行计算。你问它"23857 乘以 49621 等于多少"，它大概率算错，因为大模型不是计算器，它只是在预测"这个位置最可能出现什么数字"。

第三，无法访问实时数据。你问它"我的订单 202604280001 现在到哪了"，它根本不知道你的订单系统在哪，更不可能去查数据库。

就像一个博学的教授，他知识渊博，但他不会用计算器、不会上网查资料、不会去仓库查物流。如果你给他配上这些工具，他就能解决更多实际问题。这就是工具调用存在的意义。

没有工具调用的 LLM，只能基于训练数据生成文本，遇到需要实时数据、精确计算、外部系统交互的问题就无能为力。有了工具调用，LLM 变成了一个可以调度各种工具的"协调者"，能力边界被极大扩展。

工具调用的价值

## 工具调用的核心机制

我们用一个具体例子走一遍完整流程。用户问："帮我查一下北京明天的天气，如果下雨就提醒我带伞。"

第一步，LLM 接收到用户输入，开始推理。它分析这个问题需要两个信息：明天的天气预报、是否下雨。这些信息它自己没有，需要调用工具。

第二步，LLM 查看工具注册表。系统启动时，开发者已经把可用的工具注册好了，每个工具有名称、描述、参数定义。比如：

```
{
   "name": "get_weather",
   "description": "获取指定城市指定日期的天气预报",
   "parameters": {
     "city": {"type": "string", "description": "城市名称"},
     "date": {"type": "string", "description": "日期，格式YYYY-MM-DD"}
   }
 }
```

第三步，LLM 决定调用 get\_weather 工具，并生成参数。它根据用户输入推理出 city 是"北京"，date 是"2026-04-29"（假设今天是 4 月 28 日）。注意，这个参数是 LLM 推理出来的，不是程序员写死的。

第四步，系统执行工具调用。调用 get\_weather("北京", "2026-04-29")，返回结果：{"weather": "小雨", "temperature": "15-22°C"}。

第五步，LLM 看到工具返回结果，继续推理。它发现天气是"小雨"，满足"下雨"的条件，于是生成最终回复："北京明天有小雨，气温 15-22°C，记得带伞哦。"

整个流程中，LLM 做了三次推理：第一次推理出需要调用工具，第二次推理出工具参数，第三次推理出最终回复。这三次推理都是动态的、基于上下文的，不是写死在代码里的。

工具调用完整执行流程

## 和普通函数调用的五大区别

### 区别一：调用时机的确定性

普通函数调用的时机是编译时确定的。你在代码里写了 `result = calculate(a, b)`，编译器在编译时就知道这行代码会调用 calculate 函数。

Agent 工具调用的时机是运行时决策的。同样的用户输入"帮我查天气"，LLM 可能调用 get\_weather 工具，也可能不调用（如果它判断用户只是随口一说）。调不调用、什么时候调用，完全取决于 LLM 的推理结果。

### 区别二：参数来源

普通函数调用的参数由程序员在代码中指定。你写 `calculate(10, 20)`，参数就是 10 和 20，不会变。

Agent 工具调用的参数由 LLM 从自然语言中提取。用户说"帮我查北京明天的天气"，LLM 要从这句话里推理出 city 是"北京"，date 是"明天对应的日期"。这个推理过程可能出错，比如用户说"帮我查那个城市的天气"，LLM 可能不知道"那个城市"指的是哪里。

### 区别三：错误处理机制

普通函数调用失败会抛异常，程序要么崩溃，要么进入 try-catch 块。异常处理逻辑是程序员写死的，比如"如果失败就重试 3 次，还失败就记录日志"。

Agent 工具调用失败后，错误信息会返回给 LLM，LLM 看到错误信息后自己决定怎么办。比如调用 get\_weather 返回"城市名称不存在"，LLM 可能会：

* 重新推理，换个城市名称格式再试（"北京" → "Beijing"）
* 调用另一个工具（换个天气 API）
* 直接告诉用户"抱歉，我查不到这个城市的天气"

这种自主错误恢复能力是普通函数调用不具备的。普通函数调用的错误处理是静态的、预设的，而 Agent 工具调用的错误处理是动态的、基于推理的。

### 区别四：调用链的可观测性

普通函数调用的调用链是确定的，你可以用调试器单步跟踪，看到每一步的执行顺序和变量值。

Agent 工具调用的调用链是不确定的。同样的用户输入，LLM 可能调用工具 A，也可能调用工具 B，甚至可能先调用 A 再调用 B。你无法提前预测调用链，只能在运行时观察 LLM 的推理过程。

这给调试带来了巨大挑战。普通程序出 bug，你可以打断点、看日志、复现问题。Agent 出 bug，你可能需要看 LLM 的推理日志，分析它为什么做出了错误的工具选择或参数生成。

### 区别五：性能开销

普通函数调用的开销是纳秒级的，CPU 直接跳转到函数地址执行。

Agent 工具调用的开销是秒级的。每次工具调用前，LLM 要做一次完整的推理（几百毫秒到几秒），生成工具名称和参数。工具执行完后，LLM 还要再做一次推理，决定下一步做什么。

假设一个 Agent 任务需要调用 5 个工具，每次 LLM 推理耗时 1 秒，工具执行耗时 0.5 秒，总耗时就是 5 × (1 + 0.5) = 7.5 秒。这比普通函数调用慢了几个数量级。

## 工具调用的三大失败模式

### 失败模式一：参数提取错误

用户说"帮我查后天的天气"，LLM 需要把"后天"转换成具体日期。如果今天是 4 月 28 日，后天就是 4 月 30 日。但 LLM 可能算错，传了 5 月 1 日，导致查询结果不对。

更常见的是参数缺失。用户说"帮我查天气"，没说城市，LLM 可能瞎猜一个城市（比如默认北京），也可能返回错误"缺少必需参数 city"。

解决思路：在工具定义中明确参数的格式和约束，比如 date 参数必须是 YYYY-MM-DD 格式。对于缺失的参数，让 LLM 主动追问用户，而不是瞎猜。

### 失败模式二：工具选择错误

用户说"帮我订一张明天去上海的机票"，系统里有两个工具：search\_flights（搜索航班）和 book\_flight（预订航班）。LLM 可能直接调用 book\_flight，但这个工具需要航班号参数，用户没提供，调用失败。

正确的流程应该是：先调用 search\_flights 搜索可用航班，展示给用户选择，用户选定后再调用 book\_flight 预订。但 LLM 可能跳过第一步，直接尝试预订。

解决思路：在工具描述中明确工具的前置条件和使用场景。比如 book\_flight 的描述里写"需要先调用 search\_flights 获取航班号"。也可以用 ReAct 框架，让 LLM 在每次工具调用前先输出推理过程（Thought），提高决策质量。

### 失败模式三：工具执行超时或返回异常

工具调用了外部 API，API 可能超时、返回 500 错误、返回空结果。普通程序遇到这种情况会抛异常，但 Agent 需要把错误信息返回给 LLM，让 LLM 决定怎么办。

比如调用 get\_weather 返回"API 超时"，LLM 可能会：

* 重试一次
* 换个天气 API
* 告诉用户"抱歉，天气服务暂时不可用，请稍后再试"

但如果错误信息不够清晰，LLM 可能做出错误决策。比如 API 返回"500 Internal Server Error"，LLM 可能不知道这是临时故障还是永久故障，盲目重试多次浪费时间。

解决思路：对工具执行结果做标准化处理，把各种错误码转换成 LLM 能理解的自然语言描述。比如"API 超时"转换成"天气服务响应超时，建议重试或使用备用服务"。

## 对比总结

| 对比维度 | 普通函数调用 | Agent 工具调用 |
| --- | --- | --- |
| 调用时机 | 编译时确定，代码写死 | 运行时决策，LLM 推理 |
| 参数来源 | 程序员指定，硬编码 | LLM 从自然语言提取 |
| 错误处理 | 抛异常，静态处理逻辑 | 返回错误给 LLM，自主恢复 |
| 调用链 | 确定，可预测 | 不确定，依赖推理 |
| 性能开销 | 纳秒级 | 秒级（含 LLM 推理） |
| 可调试性 | 高，可单步跟踪 | 低，需观察推理日志 |
| 适用场景 | 确定性任务，性能敏感 | 开放性任务，需要灵活决策 |

## 面试总结

第一步（15秒）：先给核心定义。"Agent 工具调用和普通函数调用的核心区别在于：普通函数调用是编译时确定的、同步的、确定性的；而 Agent 工具调用是运行时决策的、异步的、带有不确定性的。"

第二步（45秒）：展开核心机制，用具体场景走一遍流程。"比如用户问'帮我查北京明天的天气，下雨就提醒带伞'。LLM 会先推理出需要调用 get\_weather 工具，然后从自然语言中提取参数 city 是'北京'、date 是'明天对应的日期'，调用工具获取天气数据，最后根据结果判断是否下雨并生成回复。整个过程中，调用哪个工具、传什么参数、怎么处理结果，都是 LLM 动态推理出来的，不是程序员写死的。"

第三步（30秒）：主动说局限。"Agent 工具调用的主要局限有三个：一是参数提取可能出错，LLM 可能把'后天'算成错误的日期；二是工具选择可能出错，LLM 可能跳过必要的前置步骤；三是性能开销大，每次工具调用前后都要做 LLM 推理，耗时是秒级的。在性能敏感的场景，还是要用普通函数调用。"

如果面试官追问："工具调用失败了怎么办？"

回答要点："Agent 工具调用失败后，错误信息会返回给 LLM，LLM 可以自主决定恢复策略。比如参数错误就重新推理参数，工具选择错误就换个工具，API 超时就重试或换备用服务。这种自主错误恢复能力是普通函数调用不具备的。但前提是错误信息要足够清晰，让 LLM 能理解失败原因。"

如果面试官追问："怎么保证 LLM 不会传错参数？"

回答要点："完全保证不可能，但可以降低出错概率。一是在工具定义中明确参数格式和约束，比如日期必须是 YYYY-MM-DD 格式。二是对提取出的参数做校验，不符合格式就让 LLM 重新推理。三是对于关键参数，让 LLM 主动追问用户确认，而不是瞎猜。四是在 prompt 中给出参数提取的示例，提高 LLM 的提取准确率。"
