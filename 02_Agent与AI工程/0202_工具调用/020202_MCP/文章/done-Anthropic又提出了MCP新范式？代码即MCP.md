模型上下文协议（Model Context Protocol，简称MCP）是一个开放标准，用于连接AI代理与外部系统。

自2024年11月MCP发布以来，采用速度非常快：社区已经构建了数千个MCP服务器，所有主流编程语言都有可用的SDK，行业已将MCP视为连接代理与工具和数据的事实标准。

然而，随着连接工具数量的增长，预先加载所有工具定义并通过上下文窗口传递中间结果，会拖慢代理速度并增加成本。

最近，Anthropic提出了MCP新范式。

如何让代理更高效地与MCP服务器交互，如何管理更多工具的同时使用更少的token？

工具导致的过度token消耗让代理效率降低

随着MCP使用规模的扩大，有两种常见模式会增加代理成本和延迟：

• **工具定义过载上下文窗口**

• **中间工具结果消耗额外token**

大多数MCP客户端会预先将所有工具定义直接加载到上下文中，使用直接工具调用语法将它们暴露给模型。

这些工具定义可能包含详细的描述、参数说明和返回类型，每个工具定义都要占用大量token空间。

工具描述占用的上下文窗口空间越多，响应时间和成本就越高。当代理连接到数千个工具时，它们需要在读取请求之前处理数十万个token。

更严重的问题是，大多数MCP客户端允许模型直接调用MCP工具。

例如，你可能要求代理："从Google Drive下载我的会议记录，并将其附加到Salesforce的潜在客户记录中。"

模型会进行这样的调用：首先调用Google Drive获取文档，返回完整的会议记录文本（可能包含数万token），这些内容被加载到模型上下文中。然后，模型需要再次将这些完整文本写入Salesforce更新调用的参数中，这意味着同样的内容要流经模型两次。

对于2小时的销售会议记录，这可能意味着处理额外的5万个token。

更大的文档甚至可能超过上下文窗口限制，导致工作流中断。

在处理大型文档或复杂数据结构时，模型在工具调用之间复制数据时也更容易出错。

代码执行让MCP的上下文效率大幅提升

随着代码执行环境在代理中变得越来越普遍，一个解决方案是将MCP服务器作为代码API呈现，而不是直接工具调用。

代理可以编写代码来与MCP服务器交互。这种方法解决了两个挑战：代理可以只加载它们需要的工具，并在执行环境中处理数据，然后再将结果传回模型。

实现方式有多种。一种方法是从连接的MCP服务器生成所有可用工具的文件树。每个工具对应一个TypeScript文件，例如：

• **servers/google-drive/getDocument.ts** - 从Google Drive读取文档

• **servers/salesforce/updateRecord.ts** - 更新Salesforce记录

代理通过探索文件系统来发现工具：

* 列出./servers/目录以找到可用的服务器（如google-drive和salesforce）
* 读取它需要的特定工具文件（如getDocument.ts和updateRecord.ts）来理解每个工具的接口。

这让代理只加载当前任务所需的定义。这种方式可以将token使用量从15万个token减少到2000个token——时间和成本节省了98.7%。

Cloudflare也发布了类似的发现，将MCP的代码执行称为"代码模式"。

核心洞察是相同的：大语言模型擅长编写代码，开发者应该利用这一优势来构建能够更高效地与MCP服务器交互的代理。

MCP代码执行的优势

通过按需加载工具、在数据到达模型之前进行过滤，以及在单一步骤中执行复杂逻辑，MCP的代码执行使代理能够更高效地使用上下文。

使用这种方法还有安全和状态管理方面的好处。

**渐进式披露**：模型非常擅长导航文件系统。将工具作为文件系统中的代码呈现，允许模型按需读取工具定义，而不是预先全部读取。或者，可以在服务器中添加search\_tools工具来查找相关定义。

例如，在使用上述假设的Salesforce服务器时，代理搜索"salesforce"并只加载当前任务所需的工具。在search\_tools工具中包含详细级别参数，允许代理选择所需的详细程度（如仅名称、名称和描述，或包含模式的完整定义），也有助于代理节省上下文并高效查找工具。

**上下文高效的工具结果**：在处理大型数据集时，代理可以在返回结果之前在代码中过滤和转换结果。

例如，获取一个包含1万行的电子表格。没有代码执行时，所有行都会流经上下文。有了代码执行，代理可以在执行环境中进行过滤，只看到5行而不是1万行。类似的模式适用于聚合、跨多个数据源的连接，或提取特定字段——所有这些都不会使上下文窗口膨胀。

**更强大且上下文高效的控制流**：循环、条件语句和错误处理可以使用熟悉的代码模式完成，而不是链接单个工具调用。

例如，如果你需要在Slack中等待部署通知，代理可以编写一个while循环来轮询消息，直到找到"部署完成"的消息。这种方法比在代理循环中交替使用MCP工具调用和sleep命令更高效。

此外，能够编写出条件树并执行它，也节省了"首次token时间"延迟：不必等待模型评估if语句，代理可以让代码执行环境来完成这项工作。

**隐私保护操作**：当代理使用MCP进行代码执行时，中间结果默认保留在执行环境中。这样，代理只能看到你明确记录或返回的内容，这意味着你不希望与模型共享的数据可以在工作流中流动，而永远不会进入模型的上下文。

对于更敏感的工作负载，代理框架可以自动对敏感数据进行标记化。

例如，假设你需要从电子表格将客户联系信息导入Salesforce。代理编写的代码会处理所有行，但MCP客户端会在数据到达模型之前拦截数据并对PII（个人身份信息）进行标记化。真实的电子邮件地址、电话号码和姓名从Google Sheets流向Salesforce，但永远不会通过模型。这可以防止代理意外记录或处理敏感数据。你也可以使用它来定义确定性安全规则，选择数据可以流向和来自哪里。

**状态持久化和技能**：具有文件系统访问权限的代码执行允许代理在操作之间维护状态。代理可以将中间结果写入文件，使它们能够恢复工作并跟踪进度。代理还可以将自己的代码持久化为可重用函数。一旦代理为任务开发了工作代码，它可以保存该实现以供将来使用。

这紧密联系到"技能"（Skills）的概念——可重用指令、脚本和资源的文件夹，用于提高模型在专门任务上的性能。

在这些保存的函数中添加SKILL.md文件会创建一个结构化技能，模型可以参考和使用。随着时间的推移，这允许你的代理构建一个更高级能力的工具箱，演化出它最有效工作所需的脚手架。

需要注意的是，代码执行也引入了自己的复杂性。运行代理生成的代码需要一个安全的执行环境，具有适当的沙箱、资源限制和监控。这些基础设施要求增加了操作开销和安全考虑，而直接工具调用可以避免这些。

代码执行的好处——降低token成本、减少延迟和改进工具组合——应该与这些实现成本进行权衡。

总结

MCP为代理连接到许多工具和系统提供了基础协议。然而，一旦连接了太多服务器，工具定义和结果可能会消耗过多的token，降低代理效率。

虽然这里的许多问题感觉是新的——上下文管理、工具组合、状态持久化。

但它们有来自软件工程的已知解决方案。代码执行将这些已建立的模式应用于代理，让它们使用熟悉的编程构造来更高效地与MCP服务器交互。如果你实现了这种方法，我们鼓励你与MCP社区分享你的发现。
