- 本模块先看宏观流程，再看文章：[知识地图](020800_核心知识点/知识地图.md)。

## 类目定位

| 项 | 内容 |
|---|---|
| 一级类目 | Agent 与 AI 工程 |
| 二级类目 | Context Engineering |
| 核心问题 | 如何发现、选择、压缩、缓存、切分、注入和回收上下文，让 Agent 在有限窗口里稳定完成任务 |
| 不解决什么 | 不把 MCP 仅因全称含 `Context` 归入本目录；不承接模型长上下文本身能力；不承接通用 RAG 知识库建设 |
| 用户当前认知假设 | L2-L3：知道上下文影响 AI 编程质量，后续重点是上下文来源、装载时机、压缩边界、缓存收益和失败信号 |

## 用户认知重点

| 认知项 | 当前假设 | 后续整理策略 |
|---|---|---|
| 已知基础 | 用户已经知道“上下文越多不一定越好” | 不重复解释窗口长度，重点写选择和验证 |
| 待补边界 | 工具上下文、文件上下文、记忆上下文、RAG 上下文容易混用 | 每篇文章必须说明上下文来源、注入位置和退出条件 |
| 易偏差点 | 标题容易把 Context Engineering 包装成“提示词升级版” | 优先记录成本、缓存、污染、过期和召回失败 |

## 排重准则

问题指纹：

```text
上下文来源 + 装载机制 + 压缩/缓存/切分策略 + 解决问题 + 失败边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都讲上下文管理 | 继续比较是搜索、切分、压缩、缓存、按需加载还是会话治理 |
| 只讲长窗口 | 如果不讨论选择、压缩、缓存或验证，优先归到 LLM 与大模型 |
| 只讲 MCP | 如果主问题是工具协议、server 或 tool schema，归到工具调用 |
| 只讲 RAG 建库 | 如果主问题是索引、召回、重排，归到 RAG 与知识库 |
| 有上下文污染或缓存失效案例 | 优先沉淀为认知校准点 |
