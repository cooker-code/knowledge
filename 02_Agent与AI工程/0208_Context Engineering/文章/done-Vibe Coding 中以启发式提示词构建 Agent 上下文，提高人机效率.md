在 AI 协作编程中，Agent 的输出质量直接取决于其对上下文的理解深度。**启发式提示词**是一种通过精心设计的提问来引导 Agent 主动分析场景、发现线索的方法，从而产生高质量输出。高质量包括：

- 代码结构：模块化清晰、层次分明、符合预设风格

- 充分复用：最大化利用现有代码、历史上下文和调用关系

- 方案精准：基于证据驱动的决策，避免猜测和偏差

**方案原理**：让 Agent 模拟人类思考，主动构建上下文，而非被动执行指令。

### **提示词示例与应用**

###

#### **问题域一：开发复杂任务**

**提示词 1**："先完成组件的重构，调用位置使用注释写入占位代码或占位功能描述，编译成功后再实现细节"

* **为什么管用**：

+ 编译错误会让 Agent 逻辑陷入混乱、失去任务焦点。通过移除部分调用逻辑，使AI 优先解决组件内部问题，再解决外部使用问题。
+ 通过"占位+逐步开放"确保代码结构（多文件层次）优先被审查，避免陷入逻辑细节。

**提示词 2**："使用 ASCII 描述项目C4 架构图"。

* **为什么管用**：

+ 模型熟悉 C4 Model概念（环境、容器、组件、代码），使用多层结构描述项目，可以引导 Agent 基于层次关系启动代码探索。
+ 当 C4 Model 成为上下文的一部分，后续对话使用的实体概念更容易与代码语言产生关联。

**提示词 3**："以 ASCII 图描述与 X 功能有关的代码结构，画出组件、类、接口粒度的调用关系"。

* **为什么管用**：

+ 准备好任务关联功能的代码符号，在后续对话中提高代码生成复用率、减少关联代码遗漏修改的可能性。
+ 明确表达的范围、要求 Agent 输出代码语言，为后续代码探索提供线索。
+ 确认 Agent 对口语表达的"功能点"理解是正确的，避免改动到不相关的代码。

**提示词 4**：面对需求："有哪些方案？"

* **为什么管用**：

+ 开放式提问激发发散思考，先让 AI 提出不同方案、然后让 AI 对比每个方案的优缺点。

+ 对话中呈现出的代码调研和弊端分析，在生成过程会起到帮助。

**提示词 5**：分析当前代码的导入语句，列出潜在运行时错误的风险点，并建议验证方案。

* **为什么管用**：

+ 导入语句体现出代码文件结构方面的问题，找到导入文件、阅读文件内容，可以给 Agent 提供判断结构可用性的信息。

**提示词 6**："根据 git diff，以 ASCII 画出文件结构、方法调用关系，审查代码。"

* **为什么管用**：

+ 保持提交前审查代码的习惯。从 git diff 作为起点，降低审查工作量。
+ 通过“文件结构、调用关系”促使 Agent 扩展阅读关联代码，提高画图结果的真实性。
+ 参考：[轻提示：从结构向内，审查代码变更](https://mp.weixin.qq.com/s?__biz=MzA3ODM0NzkzNw==&mid=2648025417&idx=1&sn=31d178d2ba5ca94f9bc23cf80b88fd2f&scene=21#wechat_redirect)

提示词 7："以 Story List 格式输出需求：User Story + BDD 场景（Given-When-Then）"

* **为什么管用**：

+ 通过 Given-When-Then 激发场景化思考，构建验收标准上下文
+ 同时与我对齐实现范围，避免过度编码
+ 参考：[Story Driven 开发模式](https://mp.weixin.qq.com/s?__biz=MzA3ODM0NzkzNw==&mid=2648025386&idx=1&sn=c78b6bb9bdf4452d4f491530fb008bf4&scene=21#wechat_redirect)

#### **问题域二：分析异常根因**

**提示词 1**："调研证据、只分析不改代码。"

* **为什么管用**：

+ 引导 Agent 根据异常信息、堆栈，遍历出相关代码。再根据代码推理运行效果。
+ 推理过程 Agent 可能会发现推理结论与表现不符，触发重新思考。

**提示词 2**：审查这个迭代的代码变更，从调用关系中找出证据，证明是否需要重构。

* **为什么管用**：

+ 异常往往是新的变更导致的，通过"代码变更"明确重构范围与新迭代有关，而不是全局代码库范围。
+ "找出证据"激发 Agent 的证据驱动思维，避免 Agent 根据"直觉"改到不相关的代码。

**提示词 3**："import 代码行需要强制验证可以导入，避免这样的错误：[**ImportError** 错误示例]。"

* **为什么管用**：

+ 通过"导入语句"引导 Agent 主动扫描依赖关系，构建证据链条，避免根据直觉修改代码。

#### **问题域三：知识沉淀**

**提示词 1**："从对话/文档中提取记忆组件：事件（events）、事实（facts）、技能（skills），更新到 ./memory "

* **为什么管用**：

+ 引导 Agent 结构化地分析信息，区分时间绑定的事件、稳定的知识、可执行的方法，构建可检索、可重用的知识库上下文。
+ 对话过程中提取操作步骤；记录每个功能的使用流程，流程作为自动化测试的。
+ 事件（episodic）：带时间上下文的发生（who/when/where/what）
+ 事实（semantic）：上下文无关的规则、偏好、配置
+ 技能（procedural）：步骤化的工作流、故障排查方法
+ 参考：[智能体的长期记忆](https://mp.weixin.qq.com/s?__biz=MzA3ODM0NzkzNw==&mid=2648025380&idx=1&sn=dd643e762b444ef2e700527dc14128d0&scene=21#wechat_redirect)

**提示词 2**："分析代码变更（git diff），使用 Story AC 格式总结变更内容、解决的问题，写入 commit message"

* **为什么管用**：

+ 以用户价值为中心，将一次代码变更绑定到明确的用户故事与可验证的场景（BDD）。
+ 在提交层面沉淀"证据链"：变更动机（Problems）、预期行为（AC）、影响范围（Breaking/Docs/Tests）。
+ 让历史提交天然可审查、可回溯、可用于自动化生成变更日志与回归测试。

**提示词 3**："将功能描述写在类的 docstring 上面。"

* **为什么管用**：

+ docstring 避免写"代码实现了什么"，而是"为什么要做"、"实现的功能性需求背景是什么"。
+ Agent 未来通过语义、变量名检索到函数时，有更充分的理由确认函数实现，避免迭代时改坏旧的逻辑。

### **简单总结**

1. **分析场景与目标**

* 始终从"为什么需要这个上下文？"开始
* 避免泛化提问，聚焦具体问题域

2. **构造线索导向提问**

* 使用开放词："探索、分析、找出证据、列出、解释"
* 预判代码范围，主动提供功能名称、代码描述
* 主动触发架构审查、逻辑审查，避免劣化
