**1.****背景与痛点：通用 AI 的“水土不服”**

在日常的后端开发与运维中，我们经常遇到这样的场景：开发人员写了一个标准的 SQL 查询，测试环境中一切正常，。

然而，这段代码上线后却导致了数据库告警。原因往往不在于 SQL 语法本身，而在于**企业内部特定的数据上下文**：

* **表体量差异**：某张看似普通的日志表可能拥有 5 亿行数据，严禁全表扫描，测试环境可能只有几千条数据。
* **特殊字段限制**：某些表包含 CLOB/BLOB 大字段，内部 DBA 规范明令禁止 SELECT \*，必须按需取值。
* **环境限制**：某些老旧系统在白天严禁开启并行查询（Parallel），否则会挤占核心交易资源。

这些内容，通用大模型也许能揣测一二，但是它具备强大的 SQL 语法知识，缺失了企业的“私有知识库”，比如表结构元数据、DBA 开发规范、历史故障复盘等等。

为了解决这个问题，基于 **Spring AI**搭建了一套 **RAG（检索增强生成）** 系统，将企业的一份份开发文档变成了代码开发阶段的“智能防火墙”。

**2.****技术架构选型**

研发过程遵循“轻量级、低侵入、私有化”的原则进行架构设计：

* **应用框架**：**Spring Boot 3 + Spring AI**。利用 Spring 生态的强抽象能力，屏蔽底层模型差异。
* **大模型运行时**：**Ollama (本地部署 Qwen2.5-7B)**。确保敏感的表结构和 SQL 语句不出内网，符合数据安全合规要求。
* **向量数据库**：**PostgreSQL + pgvector**。利用关系型数据库处理结构化元数据，同时利用向量插件处理语义检索，避免引入额外的中间件（如 ES 或 Milvus），降低运维复杂度。

**简化版流程：** *Input SQL -> Embedding -> Vector Search (Local Rules) -> Prompt Engineering -> LLM -> Optimization Advice*

**3.****核心实现：如何注入“DBA 专家经验”？**

RAG 的核心不在于模型多大，而在于**知识库的质量**。在这个系统中，我没有单纯地灌入文档，而是将知识分为两类进行向量化存储：

**强制性规范（Hard Rules）**针对特定表的硬性限制，直接转化为自然语言规则存入向量库。

**关键词**：T\_ORDER\_MAIN, SELECT \* 

**规则内容**：【DBA 铁律】严禁在 T\_ORDER\_MAIN 表使用 SELECT \*。该表含有 SNAPSHOT\_XML 大字段，全量查询将导致严重的网络 IO 阻塞。建议仅查询必要字段。

**历史故障经验（Experience）**将过去生产环境发生的事故复盘转化为知识条目。

**关键词**：O\_BILL\_DETAIL, UPDATE, 死锁

**规则内容**：【历史复盘】对 O\_BILL\_DETAIL 表进行高并发更新时，严禁使用 Bitmap 索引，曾导致 ORA-00060 死锁。建议在高并发场景下移除该索引或严格按主键顺序更新。

**代码实现逻辑** 

利用 Spring AI 的 VectorStore 接口，我们将用户的 SQL 输入转化为向量，去数据库中检索 Top-K 最相关的“内部规则”，然后组装 Prompt。

**4.****效果对比：RAG 带来的质变**

为了验证效果，我们以查询核心订单表为例。

**场景**：

开发人员试图执行下面语句

```
SELECT * FROM T_ORDER_MAIN WHERE ID = :id。
```

未开启 RAG（纯通用模型）：

**AI****回复**：该 SQL 语句使用了主键查询，逻辑简单且效率较高，建议确保 ID 字段有索引即可。 *(点评：AI给出了通用的正确答案，但在我们系统中，这条 SQL 上线可能会导致 IO 故障。)*

**开启 RAG（加载内部知识库）：**

**系统回复**：根据内部规范，T\_ORDER\_MAIN 表包含大字段，严禁使用SELECT \*。请明确列出所需字段，例如 SELECT ID, ORDER\_NO, STATUS等。

**5.****进阶功能：专家反馈闭环 (HITL)**

系统不仅仅是静态的查询工具，为了实现系统的演进与进化，我们还设计了Human-in-the-Loop机制。

当 DBA 发现 AI 的建议不准确，或有新的慢 SQL 案例产生时，可以通过后台接口直接录入新的“专家规则”。系统会自动将其向量化并更新到知识库中。

这意味着，人类专家只需要登录系统，对前一日产生的自动化运维记录进行阅读和评价**，这些内容就会变成系统里立刻生效的专家意见**，真正实现了知识的沉淀与复用。

**6.****总结与展望**

通过 Spring AI + RAG 技术，我们以极低的成本解决了一个痛点：**如何让AI 既懂技术，又懂业务上下文。**

这套系统目前已在内部开发环境试运行，达成了下列设计目标：

1. **降低沟通成本**：开发人员不需要反复询问 DBA 表结构禁忌。
2. **前置风险规避**：将显而易见的性能隐患拦截在代码编写阶段，而非测试或生产阶段。
3. **技术栈自主可控**：基于 Java 生态和开源模型，符合企业对技术栈统一和数据安全的要求。

未来，计划将该能力集成至 CI/CD 流水线中，实现 SQL 变更的自动化前置审查。
