对于工程行业的兄弟们来说，RAGFlow 不应该只是一个“聊天机器人”，它应该是一个能瞬间查到《建筑设计防火规范》第几条第几款、能对比三份《施工组织设计》不同点的“超级资料员”。

之前的文章大家已经学会了基础部署。今天这篇文章，我汇总了GitHub 社区、以及近期微信公众号技术大V们的实战经验，专门针对工程文档（图文混排、大表格、长规范）的痛点，列出一套“精度提纯”的深水区操作指南。

如果不按这个流程走，你的知识库就是一个“人工智障”；按这个走，它就是你的“总工级”助手。

一

解析层

工程文档的“去噪”与“结构化”（最关键一步）

工程人最头疼的是什么？是 PDF 里的扫描件、双栏排版、还有该死的跨页表格。很多公众号爆款文章都指出：RAG 的上限在于 Parse（解析）。

1别再无脑选 General，工程文档有专属“手术刀”

在 RAGFlow 的 Knowledge Base 设置中，Parser 的选择直接决定生死：

▶ 针对《国标规范》（GB/GB50xxx）：切记使用 Manual 或 Laws 模式

痛点：规范通常有严格的层级（1.0.1, 1.0.2）。

操作：社区实测，对于带层级编号的文档，Laws 模式能更好地识别“条款”边界，不会把“3.1 混凝土”和“3.2 钢筋”混在一起。

▶ 针对《工程量清单》或《材料表》：必须用 Table 模式

公众号大V经验：RAGFlow 内置的 DeepDoc 对表格的还原度是目前开源界的第一梯队。如果你的 Excel 或 PDF 里全是参数表，选 Table 模式，它会将表格转化为 HTML/Markdown 结构储存在向量库里。

效果：当你问“C35混凝土的单价是多少”时，它能精准定位到那个单元格，而不是把整行数据读给你。

▶ 针对扫描版《旧图纸/旧档案》：开启 OCR 是底线

在 System Setting 里，务必确认 OCR 模型（如 PaddleOCR 或 RapidOCR）已加载。对于工程老旧档案，Layout Analysis（布局分析）能识别出什么是“图注”，什么是“正文”，防止把图纸边角的“设计院名称”当成正文检索出来。

2这里的坑：页眉页脚的“幽灵干扰”

工程标书每一页都有“XX项目施工组织设计”的页眉。如果不处理，每一块切片（Chunk）里都有这句话。

高阶技巧（Regex 过滤）：

在解析设置中，找到正则表达式过滤（Regex Filter）。填入：

^.\*施工组织设计.\*$ 或 Page \d+ of \d+

作用：强行在切片前清洗掉这些噪音，否则检索相关度会被严重稀释。

二

检索层

解决“查不准”的工程术语难题

为什么你搜“灌注桩断桩处理”，它给你回“灌注桩施工工艺”？因为向量认为它们很像。但在工程上，这是事故处理和正常施工的区别，失之毫厘，差之千里。

3混合检索（Hybrid Search）的黄金比例

工程领域存在大量特有名词（如：HSE、BIM、C30、HRB400）。

纯向量检索（Vector）的弱点： 它懂语义，但对数字和缩写不敏感。

纯关键词检索（Keyword）的弱点： 它不懂“水泥”和“砼”是一回事。

工程界最佳实践配置：开启 Hybrid Search。

权重调整：将 Keyword 权重拉高到 0.4，Vector 设为 0.6（默认通常是 0.3/0.7）。

理由：工程查询往往带有明确的关键词（如规范编号、材料型号），提高关键词权重能防止模型“自由发挥”。

4Embedding 模型的“国产化”替换

不要用默认的英文模型！国内工程文档全是中文专业术语。

推荐模型：BAAI/bge-m3 或 BAAI/bge-large-zh-v1.5。

为什么？bge-m3 支持长文本（8192 tokens），非常适合处理长篇大论的施工方案；它对中文工程术语的理解远超 OpenAI 的默认模型。

三

深度优化

公众号大V都在推的“重排”与“图谱”

这是从“好用”进阶到“商用”的分水岭。

5Rerank（重排序）：工程精度的“质检员”

很多公众号文章在评测 RAGFlow 时都强调：没有 Rerank 的 RAG 是没有灵魂的。

原理：先粗搜出来 100 条规范条文，然后用 Rerank 模型（它是专门用来判断“这两句话是不是逻辑相关”的）进行精选。

配置作业：

模型选择：bge-reranker-large。

Top-K 设置：粗排 50 条 -> 重排留 5 条。

实战场景：搜索“外架搭设规范”。

无 Rerank： 可能会搜出“内架”、“盘扣架”等沾边的内容。

有 Rerank： 模型会精准识别“外架”这个核心约束，把不相关的剔除。

6尝试 GraphRAG（知识图谱）：解决“多跳查询”

这是 RAGFlow 最近在社区大火的功能。

场景：你问“A项目的混凝土供应商是谁，他们的资质过期了吗？”这个问题需要跨越两个文档：一个《供货合同》（查供应商），一个《供方资质表》（查资质）。

开启 Graph 模式：如果你的机器资源允许，在知识库配置中开启 Knowledge Graph 提取。它能建立实体关系，回答这种复杂的逻辑推理问题。

四

提示词工程

把 AI 调教成“严谨的总工”

工程容不得瞎编。我们要用 Prompt 给 AI 戴上“紧箍咒”。

工程专用 Prompt 模板（直接复制）：

```
身份设定：你是一名具有20年经验的高级注册土木工程师，严谨、专业、负责。  任务：基于以下【参考资料】（通常是国标规范、施工方案或图纸），回答用户关于工程技术、安全或法规的问题。  【参考资料】： {knowledge}  约束条件： 1. 引用来源：必须明确指出你的回答依据是哪一本规范的第几条（例如：根据 GB50016-2014 第5.1.1条）。 2. 拒绝脑补：如果【参考资料】中没有提及该问题的答案，请直接回答“当前资料库中未找到依据”，严禁根据你的通用知识编造数据。 3. 安全第一：如果涉及安全参数（如承载力、防火等级），请原文摘录，不要进行概括或改写，以免产生歧义。 4. 格式要求：如果涉及步骤，请用 1. 2. 3. 列表形式输出。  用户问题：{input}
```

五

社区高频避坑（FAQ）

这里汇总了 GitHub Issues 里工程师们踩烂了的坑，帮你省下几十个小时的 Debug 时间：

1. Q:解析 Excel 乱码怎么办？

A:工程类 Excel 表头往往很复杂（合并单元格）。建议： 尽量在上传前将 Excel 另存为 CSV，或者手动把多级表头改为单级表头。虽然 RAGFlow 很强，但“喂”给它干净的数据，效果永远最好。

2. Q:为什么搜不到 CAD 图纸里的字？

A:RAGFlow 目前不直接支持 .dwg 文件。必须： 将 CAD 批量转为高清晰度 PDF，并确保转换时是可以选中文本的（矢量 PDF），如果是截图转的 PDF，则必须开启 OCR。

3. Q:Chunk Token 设多少合适？

A:对于工程规范，默认的 512 往往会把一条完整的“强条”切断。建议： 设为 800-1000，并保留 200 左右的 Overlap（重叠），确保上下文连贯。

总结

要想在工程领域用好 RAGFlow，核心心法只有十二个字：

“特定解析、混合检索、重排必开”

与其不断换大模型，不如花半小时把你的《施工组织设计》和《国标规范》按照上面的方法重新清洗、建立索引。你会发现，搜出来的结果，终于像个懂行的人说的了。
