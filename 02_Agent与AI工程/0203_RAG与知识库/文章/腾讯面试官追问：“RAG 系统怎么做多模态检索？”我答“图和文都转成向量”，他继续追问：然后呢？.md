大家好，我是帅地。

今天来拆一个面试里很容易被问深、但很多人其实只会停留在 **CLIP** 这个名词上的问题：

**RAG 系统如何支持多模态检索？图文检索到底怎么实现？**

这个问题表面上看像是在问“你知不知道多模态模型”，但面试官真正想听的，其实不是你会不会背 CLIP、BLIP 这些名词，而是你有没有真正想清楚：*图片怎么入库、文本怎么对齐、查询怎么走链路、检索回来以后又怎么给大模型用。*

---

**💡 精简回答**

我理解多模态 RAG 的核心，不是“知识库里多存几张图片”，而是把图片、文本、OCR 内容、图注这些不同模态的信息，**映射到一个可对齐的语义空间里**。这样用户输入一段文字，可以召回相关图片；上传一张图片，也可以反查对应的文字说明、页面上下文，甚至同主题的图文内容。

真正落地时，一般要分成四步。

1. 离线构建知识库，把文档里的正文、图片、OCR 结果、图注、页码、上下文关系一起解析出来。
2. 分别对文本和图片做向量化，用多模态编码模型建立跨模态对齐。
3. 在线查询时先判断用户输入的模态和意图，再发起跨模态召回。
4. 对召回结果做重排和证据组织，把真正相关的图文片段一起交给大模型生成答案。

所以它不是“图片也做个 embedding”这么简单，而是一整条链路都要围绕**跨模态对齐 + 证据组织**来重做。

---

**📝 详细解释**

## 本质：多模态检索不是“给图片配一句描述”这么简单

很多人第一次做图文检索，第一反应是：给每张图片生成一句 caption，然后还是按文本去搜。这个方法不是完全不能用，但它只能算最原始的弱多模态方案。因为图片里真正重要的信息，很多时候根本装不进一句描述里。

比如用户上传一张流程图截图，想找“这张图对应的是哪份方案文档”；又比如用户给你一张商品图，想找“类似风格但材质不同的产品”；再比如医疗场景里，医生上传一张影像图，希望系统返回类似病例和诊断说明。这些问题如果只靠 caption，信息会丢得很厉害，最后检索效果通常不稳。

所以多模态 RAG 真正要做的事情，是让“同一个语义”的图和文，在向量空间里能够彼此靠近。比如“一只橙色猫趴在沙发上”这句话，和对应的图片虽然原始形式完全不同，但经过多模态编码之后，它们应该在语义空间里距离很近。这样文本查图、图查文才真正成立。

## 一个能上线的系统，离线构建通常比在线检索更难

很多人把注意力都放在“查询怎么搜”上，但实际项目里，最麻烦的往往是离线阶段。因为企业知识并不是规规矩矩地放在 txt 里，它可能藏在 PDF、PPT、扫描件、网页截图、表格、图表甚至流程图里。

这时候你不能只把“图片文件本身”存进去，还要把它和周围语境一起存进去。比如一张架构图，它本身是一类证据；图的标题、图下说明、所在页码、所属章节、前后段落，又是另一类证据。如果这些关系在离线阶段没保住，最后知识库里就只剩一张孤立的图片向量。用户搜“重排模块在系统链路里处于哪一步”，系统也许能把那张图召回来，但不知道它来自哪一页，不知道旁边解释文字是什么，更不知道 OCR 能不能补出额外线索。

所以离线阶段通常至少要做这几件事：把正文文本抽出来，把文档中的图片区域切出来，对图片做 OCR，把图注、标题、相邻段落、页码、文档路径都保留下来，然后给每个对象打上 metadata，标明它是什么模态、来自哪个文档、位于哪一页、和哪段正文关联最强。到这里，多模态知识库才算真正有了地基。

## 图文检索为什么能成立，关键在“统一语义空间 + 分模态治理”

到索引这一步，常见思路其实就两类。

一种是把文本和图片都映射到统一向量空间里，放在一个向量库中，通过模态标签区分 image 和 text。这样做的好处是天然适合跨模态检索，文本查图、图片查文都比较直接。

另一种是双索引架构，文本有一套索引，图片有一套索引，查询阶段分别召回，再在融合层合并结果。这样工程上更灵活，方便对不同模态单独调参，也更容易做性能优化。

但不管你底层选哪种，最好都别把多模态理解成“所有内容丢进一个库就完事”。真正稳一点的做法，通常是**语义上统一，工程上分治**。图片条目除了原图向量，最好还保留 OCR 文本向量、caption 向量和周边上下文；文本条目除了正文向量，也要保留关联图片的 ID 和位置。这样用户提问时，系统既能靠图片语义召回，也能靠 OCR 和上下文补召回，不至于只命中半截信息。

可以把整条链路理解成这样：

```
用户输入（文本 / 图片）  
        ↓  
查询理解（模态识别、改写、过滤）  
        ↓  
多模态编码（text encoder / image encoder）  
        ↓  
混合召回（图片向量、文本向量、OCR、metadata）  
        ↓  
跨模态重排（Reranker / VLM）  
        ↓  
证据组织（文本片段 + 图片 + 图注 + 来源）  
        ↓  
LLM / VLM 生成答案
```

真正的多模态 RAG，不是“检索完就结束”，而是要把召回到的多种证据重新组织成模型能消费的上下文。

## 在线阶段最容易被忽略的一步，是先判断“用户到底在找什么”

同样是一张图进来，用户意图可能完全不同。

他说“帮我找这张图对应的原文解释”，这是图查文；他说“找和这张图风格相近的商品主图”，这是图查图；他说“这张图第三个模块是什么意思”，这其实已经不是纯检索了，而是检索加读图理解。

文本查询也一样。用户问“肺结节典型影像表现”，他不只是想看几张相似 CT 图，而是想看图像加诊断描述；用户问“这款鞋适合什么场景穿”，系统也不该只返回商品图片，还要把材质、款式、评论区反馈这些文字证据一起带回来。

所以一个稍微完整点的系统，在线阶段通常都会先做 query routing。先判断这是文本查图、图查文、图查图，还是直接该走视觉问答链路。很多系统效果不好，不是编码模型不行，而是所有 query 都走了一条统一链路，最后该查图的去查文，该读图的去做相似度搜索，结果自然不稳。

## 为什么很多系统“能召回相关图片”，却还是答不好

因为“相关”不等于“正确”。

图像向量很擅长找视觉上相似的东西，但用户真正要的，经常不是像素相似，而是语义相关。比如医疗场景，用户问的是某种病灶的典型表现，你不能只返回几张看起来都差不多的 CT 图，而是要把影像、诊断结论、病例文本一起带上。电商场景也一样，用户上传一张衣服图，系统不只是找外观近似商品，还要补上面料、版型、适用季节、风格标签，不然大模型后面还是容易胡说。

所以线上通常不会只靠第一阶段的向量召回。更稳的做法是先粗召回一批图文候选，再用跨模态 Reranker 或视觉语言模型做精排。第一阶段解决“别漏”，第二阶段解决“别乱”。如果对准确性要求更高，还会把图片和文字证据一起送进视觉语言模型做二次验证，确认这段文字到底是不是在解释这张图，而不是刚好和它出现在同一页里。

## 实战里真正难的，往往不是模型名字，而是这些脏活累活

第一个坑，是图片信息不完整。很多知识库里的图不是高清原图，而是扫描件、低清截图、复杂报表或者嵌在 PPT 里的小图。单纯做 image embedding，语义往往很弱，必须结合 OCR 和版面关系，不然召回很飘。

第二个坑，是模态不平衡。如果库里 90% 是文本、10% 是图片，文本结果很容易把图片结果淹没；反过来，如果图片很多但文本描述弱，文本 query 又容易找不到真正相关的图。所以实际系统里常常要做模态配额、模态权重或者重排阶段的模态平衡。

第三个坑，是上下文组织。大模型并不会天然理解“一张图 + 一段 OCR + 一段正文”其实是同一组证据。所以你不能把十张图、十段字乱塞进 Prompt。更好的做法是按证据单元来组织，比如“图像 + OCR + 附近解释 + 来源”一起送进去，让模型看到的是成组的信息，而不是素材堆。

第四个坑，是时延和成本。文本 embedding 很便宜，图片编码和视觉重排通常更贵。如果每个 query 都全量跑视觉模型，线上延迟很容易炸掉。所以工程上一般都是文本优先做粗筛，只有命中图像意图、或者图像候选已经进入前几名时，才启用更重的跨模态模型。

## 多模态 RAG 的价值不只是“搜到图”，而是“基于图文联合回答”

我觉得这才是这个方向真正有意思的地方。

很多人一谈多模态检索，脑子里还是传统搜索：给我返回几张图片或者几段文字。但在 RAG 里，检索只是中间过程，终点是生成。也就是说，更有价值的系统，不是简单告诉你“这几张图可能相关”，而是能基于召回到的图片、OCR、正文和上下文，直接生成一个有依据的解释。

比如用户问“这张商品图适合什么场景穿”，系统最理想的输出不是主图列表，而是综合图片里的版型风格、文字里的面料信息、评论里的体感反馈，最后生成一个更接近真实决策的回答。医疗、工业质检、设计素材库、企业文档问答，后面都会往这个方向走。

## 写在最后

很多人一提多模态 RAG，第一反应就是“CLIP + 向量库”。这当然不算错，但如果只答到这里，其实还停留在“知道名词”的阶段。

真正把这个问题讲透，至少要把这条链路说完整：**跨模态怎么对齐，离线怎么保住图文关系，在线怎么按意图路由，召回之后又怎么做重排和证据组织。**

你会发现，它和文本 RAG 的差别，不是“多了一个图片编码器”这么简单，而是整条链路都要围绕多模态重新设计。解析更复杂，索引更细，检索更讲究，生成也更依赖证据编排。这也是为什么多模态 RAG 现在很热，但真正做得稳的系统并不多。

如果面试官接着追问一句：“那你们线上到底怎么落地？”你就别再只说“用 CLIP”了。把离线解析、统一语义空间、混合召回、跨模态重排、证据组织这五步讲完整，基本就已经和大多数只会背概念的人拉开差距了。
