大家好，我是张未果～

今天刷推的时候，看到AI大佬Karpathy分享了他的知识库管理方式，蛮值得借鉴的，于是就在这篇文章里总结一下。

Karpathy把他的知识库管理分为下面的几个环节：

......

数据输入

Karpathy把原始文档的所有东西，文章、论文、代码库、数据集、图像等索引到一个 raw/目录中，然后使用 LLM 逐步地“编译”出一个 Wiki。

这个 Wiki 实际上就是一个存放在目录结构中的.md文件集合，之所以用.md格式，是因为它天然对AI就很友好。

Wiki 内容包括：

* 对 `raw/` 中所有数据的总结
* 反向链接
* 按概念对内容进行分类
* 为这些概念撰写文章并相互链接

为了将网页文章转换为.md文件，Karpathy喜欢使用Obsidian Web Clipper扩展，并配合快捷键将所有相关图像下载到本地，以便LLM能够轻松引用它们。

......

### IDE

### 

Karpathy使用Obsidian作为 IDE 的“前端”，在里面查看原始数据、编译后的 wiki，以及衍生的可视化内容。

他还特意强调，wiki的所有内容都是由LLM写入和维护的，他几乎不手动编辑。

......

### 问答

### 

当wiki足够大之后，你就可以向LLM agent提各种复杂问题了。

它会：

* 自动查阅相关内容
* 进行研究并生成答案

现在LLM的能力很强了，不需要用复杂的RAG，LLM通过自动维护索引文件和文档摘要，已经能很好地读取和利用相关数据。

......

### 输出

### 

Karpathy在这一环节的操作方式，对我的启发很大。

以前我跟LLM交流，都是在聊天框里，聊过了就算完了，没有进行保存和归档。

Karpathy说在跟LLM交流的时候，会让它把回答输出为：

* Markdown 文件
* 幻灯片
* matplotlib 图像

然后他再“归档”回 wiki 中，从而不断增强知识库，这种探索过程本身就会不断累积价值。

......

### 质量检查

Karpathy还会用 LLM 对整个 wiki 做“健康检查”，例如：

* 找出不一致的数据
* 补全缺失信息（通过联网搜索）
* 挖掘潜在关联，建议新的文章主题

这些操作可以逐步提升 wiki 的数据质量和完整性。

LLM 在提出进一步研究问题方面也非常有帮助。

......

### 额外工具

### 

Karpathy会开发一些辅助工具来处理数据，例如：

* 写了一个简单的 wiki 搜索引擎（带 web UI）
* 更常见的情况是把这个工具通过 CLI 交给 LLM 使用，作为复杂查询的工具

......

### 进一步探索

### 

随着仓库变大，自然会想到：

* 生成合成数据
* 微调模型

让 LLM 把这些知识“记进权重”，而不是仅依赖上下文窗口。

......

总结下流程：

1. 收集原始数据，各种类型的都收集

2. 用LLM编译为结构化的.md文档形式wiki

3. 与LLM进行问答交流，并把回答沉淀回wiki本身

4. 使用LLM定期给wiki做体检

......

这套用wiki管理知识的方式，我十几年前就用过。

这个还真不是在吹牛哈，我当时用的是开源的wiki软件。

但是手动整理各种资料，真的极其考验人的执行力和精力。

我坚持了一段时间就放弃了。

但现在有了AI，很多事情都不一样了。比如，存储原始资料、结构化提取内容、互动式问答，这些问题都已经得到了很好的解决。

所以，在当下的时间节点，执行这套知识库管理的摩擦力还是蛮小的。

我会按照Karpathy的方式执行一段时间，然后汇报使用经验。
