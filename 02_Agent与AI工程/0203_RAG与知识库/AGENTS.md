## 用户认知重点

| 项 | 内容 |
|---|---|
| 已知基础 | RAG 是外部知识增强，不是模型训练 |
| 待补边界 | 文档解析、chunk、embedding、召回、重排、引用、评估、知识生命周期 |
| 易偏差点 | 工具清单文章易把"支持格式多"当成质量高 |
| 优先抽取 | 可验证的解析质量、召回质量、引用正确率与维护成本 |

## 排重指纹

```text
知识系统对象 + 链路节点 + 核心机制 + 解决问题 + 质量/成本/维护边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是文档解析 | 按 PDF、HTML、表格、知识图谱、多模态拆分 |
| 都是 RAG 优化 | 区分解析、切分、召回、重排、生成、引用、评估 |
| 都是知识库产品 | 先判断是 RAG 系统、文档管理、AI 工作流还是个人知识管理 |
| 只有工具清单 | 没有选择准则和质量指标时只略读 |
| 与 LLM Wiki 对比 | 必须说清是检索系统、预编译知识库还是个人知识沉淀流程 |

## 待补缺口

| 优先级 | 项 | 为什么补 |
|---|---|---|
| 高 | RAG 检索工程 | 决定召回质量与可观测性 |
| 高 | Rerank / Hybrid Search | 已有机制边界，仍需本地评估和权重实验 |
| 高 | knowledge lint | 已有机制沉淀，仍需项目内规则或脚本落地 |
| 高 | RAGFlow 评估 | 已覆盖切块和召回，下一步要补评估闭环 |
| 高 | 文档解析工具 | 仍需 MinerU/Docling 等工具补证与实测 |
| 高 | GraphRAG/多模态图谱 | 关系、多跳、图表文档是向量 RAG 的重要边界 |
