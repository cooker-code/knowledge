我越来越觉得，Prompt 工程最容易被讲歪的地方，就是总有人把它讲成“怎么把模型哄高兴”。其实不是。你不是在碰运气，你是在控制输出质量。

---

## 先说结论

Prompt 不会让模型变得更聪明。

它真正做的，是在模型已经具备的海量能力里，决定什么被激活，什么被压住，什么以什么方式组合出来。

所以，Prompt 不是抽卡，也不是什么神秘配方。它更像一个质量阀门：一边收边界，一边放能力。

---

## 为什么这件事值得现在讲

现在聊 Prompt 工程，很容易滑到两个极端。

一种讲法很玄，仿佛多一句角色设定、换一个措辞、加一句“深呼吸”，结果就会突然跃迁。

另一种讲法则是“prompt 无用论”。模型越来越强，于是很多人顺手得出一个结论：Prompt 工程会越来越不重要。尤其在 AI Coding 场景里，既然模型已经能读代码库、理解上下文，那还写什么 CLAUDE.md。

我不认同这两种看法。

真正的问题不是“还要不要写 Prompt”，而是你到底把 Prompt 当成什么。如果你把它当成口诀，那当然会失望；如果你把它当成边界设计和质量控制，它的重要性反而会越来越高。

---

## 最值得展开的几个点

### 1. Prompt 的作用，不是增强能力，而是调度能力

这里最关键的一点是：模型的能力在训练完成时就已经确定了。你后面写进去的每一行 Prompt，都不是给模型新增能力，而是在已有能力里做选择。

这件事说穿了，其实不复杂。Prompt 同时在做两件方向相反、但又不能拆开的事：约束和释放。

只约束不释放，结果通常会很规矩，也很平。只释放不约束，结果可能很有才华，但容易跑偏。好的 Prompt，本质上是在这两者之间找平衡。

**这对读者的价值：** 如果你平时写 Prompt，总想着“再补一句模型会不会突然更懂”，那现在可以换个角度了：别再指望模型突然开窍，先想清楚你到底在限制什么、释放什么。

---

### 2. 不是所有任务，都适合把推理过程强行摊开

我觉得这部分特别重要。

Michael Polanyi 有一句很有名的话：我们知道的东西，往往多于我们能直接说出来的东西。放到模型任务里，这个判断非常实用。

像数学推理、多步骤逻辑、算法执行这类程序化任务，把步骤显性化，通常有帮助。

但像架构判断、代码审美、创作风格、设计优雅度这类任务，事情就没这么简单了。你越强行要求模型一步一步把思路摊开，越可能把原本依赖整体整合能力的输出给拆掉。

这也是为什么 Chain-of-Thought 不是放之四海而皆准的答案。有些任务适合显性化，有些任务不适合。

**这对读者的价值：** 这会直接改掉很多人的使用习惯。不是所有任务都该把模型逼成“展示解题过程”的机器。任务类型不同，Prompt 的组织方式也该不同。

---

### 3. 真正有效的 Prompt，更像在设边界条件，而不是列执行清单

这是我最认同的一层。

很多人写 Prompt，还是在按“指令列表”的思路来写，恨不得把每条规则都列满。但更有效的做法，往往不是把清单越写越长，而是把边界条件设对。

说白了，你不是在微操模型内部每一步怎么发生，而是在给一个会涌现结果的系统划空间。

所以，规则清单不一定最强，概念锚点反而可能更有效。你不是靠二十条细则拼出一个好判断，而是靠边界、倾向和上下文，让正确的判断自己长出来。

这件事放回工程实践，也很容易理解。教一个新工程师时，你不会先甩给他二十条机械规则，而是会让他读项目、看 code review、理解设计取向。对模型也是一样。

**这对读者的价值：** 如果你正在用 Claude Code、Codex 或其他 coding agent，这一点尤其有用。很多时候真正决定结果的，不是规则列得多细，而是你有没有把任务边界、设计取向和质量标准讲清楚。

---

### 4. Prompt 问题的底层，其实是认知立场问题

这也是我觉得这条长文最不该被忽略的地方。

一个系统不可能在完全没有立场、没有投入、没有约束的状态下，稳定地产生高质量判断。

所以，那些看起来特别“正确”的指令，比如“完全客观”“不要做任何假设”“保持绝对中立”，未必真的更高级。很多时候，它们只是把认知条件剥空了。

这不是更严谨，反而可能更空。

**这对读者的价值：** 写 Prompt 时，不是越空越高级，也不是越中性越有效。很多场景里，把目标、判断标准和要服从的现实约束说清楚，反而更重要。    

---

## 如果你也在做类似事情，可以怎么用

### 1. 先写边界，再写要求

如果你希望模型产出稳定结果，先把它该服从什么、不要偏到哪里、什么算好结果说清楚，再考虑补规则。

---

### 2. 按任务类型决定要不要强制展开思路

程序化任务可以更明确步骤；直觉性、审美性、架构性任务，反而要警惕过度微操。

---

### 3. 把给 Agent 的说明文档，当成质量控制层

无论是 CLAUDE.md、system prompt，还是项目内的工作说明，本质上都不只是“补充说明”，而是在给模型设边界条件。

---

## 最后一句判断

Prompt 工程真正要解决的，从来不是“怎么把模型哄得更聪明”，而是怎么让一个本来就很强的系统，更稳定地朝正确方向工作。
