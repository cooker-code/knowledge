### 写在前面

说实话，我刚开始用ChatGPT的时候，经常会得到一些**非常泛泛**的回答。比如问它"帮我写个API"，它就给你来一段教科书式的代码，看起来没问题，但实际场景根本用不上。

后来我才意识到，问题不在模型，而在于**我不会提问**。

这就像你去餐厅点菜，如果只说"给我来点吃的"，大厨也不知道该给你做什么对吧？但如果你说"来一份少油少盐的宫保鸡丁，不要花生"，那得到的结果肯定截然不同。

我们来看看这篇文章里提到的10个技巧。这些方法不是什么秘密，都来自于各大公司的研究论文，但**很少有人真正去用**。

### 技巧1：角色约束提示（Role-Based Constraint Prompting）

这个技巧的核心思路是：**别让AI瞎猜你想要什么**。

我们经常会直接问"帮我写段代码"，但这太模糊了。AI不知道你的技术栈、不知道你的限制条件、也不知道你要什么格式。

更好的做法是给它安排一个**具体的角色**，然后列出明确的约束：

```
你是一位有10年经验的Python工程师，专注于数据管道优化。  
  
任务：构建一个实时ETL管道，处理每小时1000万条记录  
  
约束条件：  
- 必须使用Apache Kafka  
- 内存占用不超过2GB  
- 延迟低于100ms  
- 零数据丢失  
  
输出格式：可直接用于生产环境的代码，带详细注释
```

你看，这样一来，AI就知道自己的"身份"和"限制"了。它不会给你推荐Redis（因为你要Kafka），也不会写那种内存占用5GB的方案。

**核心原则**：角色 + 约束 + 格式，三者缺一不可。

P.S. 我试过用这个方法让GPT-4帮我优化SQL查询，效果比之前提升了**至少10倍**。

### 技巧2：验证链（Chain-of-Verification）

这是Google研究团队用来**消除幻觉**的方法。

什么是幻觉？就是AI一本正经地胡说八道。比如你问它某个冷门库的API用法，它可能会编造一个看起来很合理、但根本不存在的方法。

验证链的思路很简单：让AI先给出答案，然后**自己质疑自己**，最后修正答案。

模板是这样的：

```
任务：解释Transformer如何处理长上下文窗口  
  
步骤1：给出你的初步答案  
步骤2：生成5个能暴露答案错误的验证问题  
步骤3：回答每个验证问题  
步骤4：基于验证结果，给出最终修正后的答案
```

为什么这么做有效？因为AI在生成内容时，第一反应往往是基于"模式匹配"（听起来对的），而不是"逻辑推理"（真的对）。但当你让它主动去挑刺，它就会进入一种**更严谨的推理模式**。

文章里提到，这个方法能把复杂技术问题的准确率从60%提升到92%。

没错，**几乎翻倍**。

### 技巧3：负例示范（Few-Shot with Negative Examples）

Anthropic发现了一个有趣的现象：告诉AI"**不要这样做**"和告诉它"应该这样做"一样重要。

我们平时用Few-Shot（给几个例子让AI模仿）的时候，通常只给正面例子。但这有个问题：AI可能理解不了你真正想避免什么。

更好的做法是：好例子 + 坏例子，并且说明为什么坏。

```
我需要你写冷启动邮件的主题。这里有一些例子：  
  
✅ 好例子："关于你们Q4工程路线图的快速问题"  
✅ 好例子："看到你关于分布式系统的文章——对这个有什么想法？"  
  
❌ 坏例子："紧急：限时优惠！！！"  
为什么不好：垃圾邮件触发词，虚假紧迫感  
  
❌ 坏例子："你不会相信我们做了什么..."  
为什么不好：标题党，没有上下文  
  
现在，为这个产品写5个主题：能减少40%云成本的SaaS工具
```

这个方法能**消除80%的通用AI回复**。

你会发现，加上负例之后，AI输出的内容会变得更有针对性，而不是那种"看起来什么都对，但其实什么都不对"的废话。

### 技巧4：结构化思考协议（Structured Thinking Protocol）

这是OpenAI的GPT-5团队用于复杂推理任务的方法。

核心思路是：**强制AI分层思考，而不是直接蹦答案**。

模板长这样：

```
在回答之前，完成以下步骤：  
  
[理解阶段]  
- 用你自己的话重述问题  
- 明确真正被问到的是什么  
  
[分析阶段]  
- 拆解成子问题  
- 列出所有假设和限制条件  
  
[策略阶段]  
- 列出2-3种可能的方案  
- 评估每种方案的权衡  
  
[执行阶段]  
- 给出最终答案  
- 解释你的推理过程  
  
问题：5人创业团队做B2B SaaS，预计第一年1000用户，应该用微服务还是单体架构？
```

为什么这么做？因为很多时候，AI给出的答案是**最常见的答案**，而不是**最适合你的答案**。

比如你问"应该用微服务吗"，它可能直接告诉你微服务的优点。但如果你强制它先理解场景（5人团队、1000用户），它就会意识到：**单体架构可能更合适**。

这个方法让AI从"背书"变成了"顾问"。

### 技巧5：置信度加权（Confidence-Weighted Prompting）

Google DeepMind用这个方法做**高风险决策**。

简单说就是：别只要答案，还要AI告诉你它有多确定。

```
回答这个问题：Rust会在2030年前取代C++成为系统编程的主流吗？  
  
请提供：  
1. 你的主要答案  
2. 置信度（0-100%）  
3. 你做了哪些关键假设  
4. 什么情况会改变你的答案  
5. 如果置信度<80%，给出替代答案
```

这个技巧**非常实用**。

我之前用AI做技术选型建议，它总是特别自信地告诉我"用XX技术最好"。但后来我加上置信度要求，它会老实说"我只有60%确定，因为我不知道你们团队的技术栈"。

这就防止了我们**基于AI的假自信做决策**。

当然，AI可能会高估自己的置信度，但至少它会让你意识到：这个答案可能不是铁板钉钉的。

### 技巧6：边界上下文注入（Context Injection with Boundaries）

Anthropic的工程师用这个方法处理**大量上下文**。

核心是：给AI海量信息，但明确告诉它**只能用这些信息**。

```
[上下文]  
[粘贴你的文档、代码、研究论文]  
  
[聚焦]  
只使用上下文中的信息回答。如果上下文中没有答案，请说"提供的上下文中信息不足"。  
  
[任务]  
[你的具体问题]  
  
[约束]  
- 引用上下文中的具体章节  
- 不要使用上下文之外的通用知识  
- 如果存在多种解释，列出所有可能
```

为什么要这样？因为AI有个**致命问题**：它会混合训练数据中的知识和你提供的上下文，然后给你一个"看起来对"但其实跟你的文档不符的答案。

这个方法能**消除专有系统上的幻觉**。

我用它来让GPT-4读我们公司的内部API文档，效果好得惊人。它不再瞎编API参数了。

### 技巧7：迭代精炼循环（Iterative Refinement Loop）

OpenAI研究团队用多轮提示来**逐步优化输出**。

模板是这样的：

```
[迭代1]  
创建一个[草稿/大纲/初始版本]：[任务]  
  
[迭代2]  
审查上面的输出，找出3个弱点或缺陷  
  
[迭代3]  
针对所有已识别的弱点重写输出  
  
[迭代4]  
最终审查：这能直接用于生产环境吗？如果不能，缺什么？
```

文章里说得很直白：**单次输出总是垃圾**。

这个我深有体会。AI第一次给你的答案往往是"能用"的级别，但离"好用"还差得远。通过强制它自己审查、修正，能把质量提升到90%。

好吧，这其实有点像我们写代码时的Code Review，只不过这次是AI自己给自己审查。

### 技巧8：约束优先（Constraint-First Prompting）

Google Brain的研究员会**先列约束，再说任务**。

```
硬约束（不可违反）：  
- 必须用Rust编写  
- 不能使用任何外部依赖  
- 必须能在stable Rust 1.75+上编译  
- 二进制文件最大5MB  
  
软偏好（尽量优化）：  
- 编译时间快  
- 内存分配最小化  
  
任务：写一个CLI工具，解析10GB的CSV文件并输出带schema验证的JSON  
  
请确认你理解所有约束再开始。
```

这个方法能防止AI给你"**技术上正确但实际上没用**"的答案。

比如你问它"怎么优化这段代码"，它可能建议你用某个库。但如果你提前说"不能引入新依赖"，它就会想别的办法。

约束越明确，答案越实用。

### 技巧9：多视角提示（Multi-Perspective Prompting）

Anthropic的宪法AI（Constitutional AI）用多个视角来**减少偏见、改善推理**。

```
从以下视角分析[话题/问题]：  
  
[视角1：技术可行性]  
[具体维度]  
  
[视角2：商业影响]  
[具体维度]  
  
[视角3：用户体验]  
[具体维度]  
  
[视角4：风险/安全]  
[具体维度]  
  
综合：  
整合所有视角给出最终建议，并明确说明权衡
```

这个方法能让你得到**战略性思考**，而不是表面建议。

举个例子，如果你问"我们应该从Postgres迁移到DynamoDB吗"，AI可能直接告诉你DynamoDB的优点。但如果你要求它从技术、商业、用户体验、风险四个角度分析，它会给你一个**更全面的评估**。

其实这就是强制AI做决策分析框架，避免单一视角的盲点。

### 技巧10：元提示（Meta-Prompting）

这是**核心武器**。

OpenAI的红队（专门找漏洞的团队）用这个方法来突破他们自己的模型。

思路很简单但很强大：**让AI给自己写最佳提示词**。

```
我需要完成：[高层目标]  
  
你的任务：  
1. 分析什么才是这个目标的完美提示词  
2. 考虑：具体性、上下文、约束、输出格式、需要的示例  
3. 写出那个完美的提示词  
4. 然后执行它并给出输出  
  
[目标]：[你的实际目标]
```

为什么这么牛？因为AI对"什么样的提示词能让自己表现最好"，比你更清楚。

这就像雇了一个实时的提示词工程师。

文章作者说他用这个方法已经6个月了，效果是：

* • 技术文档上的幻觉率**降到0**
* • 代码生成迭代速度**快3倍**
* • 复杂分析任务准确率**超过90%**

没错，这些数据听起来有点夸张，但我试过几次，效果确实**远超预期**。

### 总结

这篇文章最打动我的一句话是：

**初学者和专家的差距不在于用的模型，而在于怎么跟模型说话。**

我们花很多时间研究新框架、新语言、新工具，但很少有人认真学习怎么和AI交流。

这10个技巧其实都围绕一个核心：**把AI当成需要明确指令的助手，而不是能读懂你心思的魔法师**。

你给它的信息越具体、约束越明确、反馈越清晰，它的输出就越接近你想要的。

好吧，我承认，刚开始用这些方法会觉得有点麻烦。但当你习惯了之后，你会发现：**写一个好提示词，比改10次垃圾输出要高效得多**。

P.S. 这些技巧都来自公开的研究论文，不是什么秘密。但正如文章所说：没人去读那些论文。现在你知道了，试试看吧。
