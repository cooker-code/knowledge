想象一下：你的代码库已经 5 万行了。你得想办法让 Claude 在这个巨型项目里保持**专注**、**跑得更快**，并且**不搞砸**。

## Prompt 6：上下文窗口管理器——在大项目里保持专注

**常见问题：** 你的代码库有 50,000 行，分散在 800 个文件里。你让 Claude 重构认证中间件。它开始读取不相关的文件：支付处理器、管理仪表盘、邮件模板。令牌（Token）计数暴涨。Claude 试图在上下文里塞太多不相关代码，导致回复变得很泛泛。

性能开始下降。本来 20 分钟搞定的事，因为 Claude 在不相关的上下文里挣扎，拖成了两个小时。

**提示词：**

```
**对于此任务，你只准使用以下特定文件：**

[使用Tab键补全添加目标文件]
-@src/auth/middleware.ts
-@src/auth/session.ts
-@types/user.ts
-@config/security.ts

忽略代码库里的其他所有东西。
任务：[你的具体目标]

如果你需要其他上下文，请**停止并询问**我应该添加哪些文件。
不要自己擅自读取文件。
```

**支持的命令：**

```
# 切换焦点时，清除对话历史
/clear

# 压缩历史记录，保留关键上下文
/compact 保留 auth 重构的实现细节，
丢弃关于邮件模板的早期讨论

# 使用管道模式进行针对性查询
claude -p "解释会话验证逻辑" < src/auth/middleware.ts
```

**为什么管用：** 明确的文件范围可以**防止上下文被污染**。Claude 会把计算资源集中在相关代码上，而不是扫描数千行与当前任务无关的代码。“询问后再读取”的规则，可以阻止它自己去猜测和扩展上下文。

**大型代码库策略：**

* • **模块化会话：** 每个功能领域开一个独立的终端会话。别把认证工作和支付处理工作混在一起。
* • **Git 工作树：** 多个分支在不同的目录里 → 可以在没有合并冲突的情况下，同时开启多个 Claude 会话。
* • **分层 `CLAUDE.md`：** 项目级模式放在根目录的 `.claude/CLAUDE.md`，模块特定的指导放在子目录里。

**建议：** 用 `/memory` 命令查看 Claude 当前对你项目“了解”多少。编辑这个文件，可以删掉过时的上下文，或者添加新的架构决策。

**效果：** 正确的上下文范围是 Claude 到底是**“理解”**你的代码库，还是基于不足信息**“猜测”**的区别。在超过 50,000 行的项目中，管理得当的上下文能持续保证代码质量。

## Prompt 7：性能分析器——不靠分析工具也能找到瓶颈

**常见问题：** 你的仪表盘加载很慢。用户抱怨。你怀疑是数据库查询的问题，但搭分析工具很费时间，读懂火焰图需要又掉好几根头发。

同时，仪表盘因为需要5秒来加载数据，用户不断的失去耐心，而这些数据本应该不到1秒就能显示出来的。

**提示词：**

```
**性能分析请求**

分析此代码的性能瓶颈：@[文件或函数路径]

重点关注：
1.**时间复杂度**-标记 O(n²)或更差的运算。
2.**不必要的重新渲染**(React)或冗余计算。
3.**数据库查询效率**(N+1问题，缺少索引)。
4.**内存泄漏**或过度分配。
5.**应该异步**的阻塞操作。

对于每个问题：
-解释性能影响（尽可能量化）。
-显示有问题的代码模式。
-提供优化后的实现。
-估计改进幅度。

**按影响优先级排序**：找出重要的更改。
```

**为什么管用：** Claude 能瞬间发现算法效率低下的地方——这些模式，如果你手动去分析，需要仔细查看分析器数据。它可以捕获常见的反模式（嵌套循环、缺少记忆化、同步 I/O 阻塞热路径）——这些都是开发人员实现时容易忽略的问题。

**Claude 能识别的性能优势：**

* • **React 组件：** `useEffect` 缺少依赖导致渲染循环，昂贵的计算没有包裹在 `useMemo` 中。
* • **数据库查询：** N+1 问题，一个查询变成 50 个；外键缺少索引。
* • **算法：** 应该是 O(nlogn) 的排序，却跑成了 $O(n^2)$；不必要的数组拷贝造成垃圾。
* • **异步操作：** 同步文件 I/O 阻塞了 Node.js 事件循环。
* • **内存管理：** 从未清理的事件监听器，从未关闭的数据库连接。

**举个实际的例子（分析仪表盘组件性能）：**

```
发现 3个关键问题：
1.第67-89行：用户数据聚合使用了嵌套循环
当前：O(n²)-1000条记录耗时3.2秒
解决方案：使用Map进行 O(n)分组
影响：1000条记录耗时200毫秒（快了16倍）
2.第112行：useEffect 每次渲染都重新获取数据
当前：每分钟50多次 API 调用
解决方案：添加正确的依赖项数组[userId, dateRange]
影响：减少到每个会话2-3次 API 调用
3.数据库：user_orders 表缺少 user_id 上的索引
当前：用户仪表盘查询时间200毫秒
解决方案：CREATE INDEX idx_user_orders_user_id ON user_orders(user_id)
影响：查询时间5毫秒（快了40 倍）
```

**建议：** 结合**扩展思考**进行复杂分析：“深入思考此数据聚合管道中的性能瓶颈”。

**效果：** 不用分析工具，就能捕获 **60–80%** 的性能问题。剩下的 20–40% 才需要实际分析，但 Claude 已经大大缩小了你的搜索范围。

## 指令 8：文档生成器——让文档保持实时更新

**常见问题：** 你完成了认证重构。跑得完美。你对自己保证：“明天”就写文档。

六个月过去了。一个新开发人员加入了。他问认证是怎么回事。你花了两个小时解释会话管理、令牌刷新逻辑和安全注意事项，因为你**从来没写过文档**。

新开发人员实现了一个绕过你安全中间件的功能，因为他根本不知道它的存在。现在你的生产环境出了安全漏洞。

**提示词：**

```
**文档模式**

为以下内容生成全面的文档：
@[文件、模块或功能路径]

结构：
1.**目的**：这段代码是干嘛的（2-3句话）。
2.**架构**：它如何与系统集成。
3.**关键组件**：主要函数/类及其职责。
4.**使用示例**：常见用例和代码片段。
5.**边缘情况**：已知限制、陷阱、性能说明。
6.**测试**：如何测试这段代码，验证了什么。
7.**依赖项**：外部库及其要求。

格式：[README.md |JSDoc注释|内联文档]
同时更新 CLAUDE.md，包含这个实现的模式。
```

**为什么管用：** Claude 直接阅读**实现代码**——文档肯定和现实情况一致。它从实际代码中捕获**边缘情况**，而不是从你逐渐模糊的记忆中获取。结构化格式确保不同模块之间的文档保持一致性。

**不同上下文的文档类型：**

* • `README.md`：模块级文档，用于架构理解。
* • `JSDoc`/`TSDoc`：函数级 API 文档，用于实现细节。
* • `ARCHITECTURE.md`：系统设计和组件关系。
* • `CLAUDE.md`：项目模式和约定，用于 AI 辅助开发。

**举个实际的例子（刚刚完成的速率限制中间件的实现）**

```
“为未来开发人员记录速率限制系统：
-令牌桶算法的工作原理
-限制在哪里配置（环境变量）
-常见错误场景和客户端如何处理
-路由处理程序中的示例用法
-性能影响和Redis依赖项
-安全注意事项（分布式攻击）

格式为 README.md，并在@CLAUDE.md 的‘安全模式’部分记录”
```

**建议：** 在**构建时**就要求文档，而不是之后补：“实现密码重置功能，然后生成涵盖邮件流程、令牌生命周期和安全模型的文档。”

**效果：** 因为文档是从实际代码生成的，所以它**始终保持最新**。新团队成员上手更快，因为文档反映的是真实情况。代码审查也自动包含了文档更新。

## Prompt 9：代码审查助理——抓住 Linters 漏掉的问题

**常见问题：** 你跑了 `lint`。没报错。你跑了测试。全通过了。你提交了拉取请求（PR）。

你的技术负责人审查后，要求你改：

“这个变量名有歧义。”

“这条注释过时了。”

“这违反了我们的架构模式。”

“如果这里网络失败了会怎么样？”

这些反馈都**很有价值**。但没有一个能被自动化工具捕获。你又花了一个小时解决这些**主观质量问题**，而它们本可以在你请求审查之前就修复。

**提示词：**

```
**代码审查模式**

审查此 PR 的**主观质量问题**：
@[此 PR 中更改的文件]

关注：
1.**清晰度**：是否有误导性的命名、令人困惑的逻辑、缺少解释？
2.**可维护性**：未来的开发人员能理解吗？
3.**架构**：这是否符合我们的模式？有没有代码异味？
4.**边缘情况**：什么情况可能崩溃？缺少了什么验证？
5.**测试**：测试是否全面且有意义？
6.**文档**：代码是否自文档化，还是需要注释？

提供具体的**逐行反馈**：
-什么有问题，以及原因。
-对维护的影响。
-具体的改进建议。

要直接且严格。我需要诚实的反馈。
```

**为什么管用：** Claude 能识别自动化工具忽略的模式——那些将**可维护代码**和**技术债**区分开来的主观元素。它可以捕获注释里的错别字、过时的 TODO，以及虽然能工作，但损害可读性的混乱命名。

**审查重点领域：**

* • **命名精确性：** 像 `temp`、`data2` 或 `handleClick2` 这样没有传达目的的变量。
* • **注释准确性：** 和代码不符的过时注释；缺乏对不明显逻辑的解释。
* • **模式一致性：** 偏离了既定的代码库模式。
* • **架构对齐：** 耦合度高、功能大包大揽的“上帝对象”、违反单一职责原则。
* • **安全意识：** 潜在的注入漏洞、认证绕过、数据泄露。

**举个例子（审查支付处理实现）**

```
第 47行：变量'result'对于支付确认数据来说过于笼统。
→重命名为'paymentConfirmation'，明确内容和目的。

第89行：注释说“验证输入”，但函数还处理了支付。
→违反单一职责。拆分为 validatePaymentRequest()和 processPayment()。

第112行：缺少网络超时的错误处理。
→用try-catch包裹，专门处理超时场景。当前代码会让用户看到通用的“出错了”。

第156行：测试模拟了支付网关，没有验证集成。
→添加带有沙盒 API 密钥的集成测试，以捕获实际失败。
```

**建议：** 在提交 PR 之前，先自己跑一遍这个指令。修复 Claude 找出的明显问题。你就能向人工审查者提交更干净的代码，缩短审查周期。

**效果：** 在人工审查之前，捕获 **70–80%** 的“请求更改”反馈。代码质量提高了，因为主观问题在开发早期而非审查后期就被发现了。

## Prompt 10：持续学习循环——构建机构知识

**常见问题：** 每次 Claude 会话都从零开始。你解释架构，Claude 生成代码，会话结束。

第二天：新会话，同样的解释。你又描述了一遍状态管理方法，又解释了选择 Zustand 而非 Redux 的原因，又阐明了认证流程。

三个月后，你仍然在每个会话里解释相同的架构决策，因为 **Claude 在对话之间没有记忆**。

**解决方案——系统化更新 `CLAUDE.md`：**

**提示词：**

```
**完成任何重大变更后：**

用我们学到的内容更新 CLAUDE.md：
部分：[架构|模式|测试|注意事项]
记录：
1.这解决了什么问题？
2.为什么选择这种方法，而不是其他方案？
3.未来变更应遵循什么模式？
4.应该避免什么错误？

为未来的会话编写清晰、可操作的指南。
```

**`CLAUDE.md` 结构（它会随着项目发展而演变）：**

```
# 项目背景

## 架构决策
-状态：Zustand（非Redux）-减少75%样板代码，简化思维模型。
-数据获取：ReactQuery用于缓存和请求去重。
-认证：httpOnly cookie 中的 JWT -防止 XSS，自动处理刷新。
-数据库：Prisma ORM 配PostgreSQL-类型安全，迁移管理。

## 代码风格与模式
-只用 ES 模块（禁用CommonJS）。
-带Hook的函数组件（废弃类组件）。
-功能边界周围的错误边界（而不是每个组件都加）。
-共享逻辑用自定义Hook（前缀为'use'）。

## 测试要求
-所有业务逻辑的单元测试（ReactTestingLibrary）。
-所有 API 路由的集成测试（supertest）。
-不模拟数据库（使用带迁移的测试数据库）。
-禁用快照测试（太脆弱，倾向行为测试）。

## 常用命令
```bash
npm test          # 运行完整测试套件
npm run test:watch  # 开发模式的监视模式
npm run lint      # ESLint + Prettier 检查
npm run type-check  # TypeScript 验证
npm run dev       # 本地开发（端口 3000）
npm run build     # 生产构建

## 已知注意事项与边缘情况
- 会话 24 小时后过期 - 客户端必须处理 401 并重定向。
- 文件上传限制为 10MB - 上传前显示错误。
- 速率限制：每用户每分钟 100 个请求 - 实现指数退避。
- 必须验证 Webhook 签名 - 没有 HMAC 检查，绝不信任负载。
- 日期处理：所有时间存为 UTC，客户端转换为用户时区。

## 性能考虑
- 仪表盘查询成本高 - 始终分页，绝不获取所有用户。
- 图像上传必须进行客户端优化（压缩后最大 2MB）。
- 用户关系中的 N+1 查询风险 - 正确使用 Prisma 'include'。
- Redis 缓存 TTL：用户数据 5 分钟，静态内容 1 小时。
```

**为什么管用：** 每次会话都基于**先前知识**，而不是从零开始。架构决策可以持续数周。常见错误记录一次，就能永久预防。新团队成员可以**即时继承**这些机构知识。

**维护节奏：**

* • **每天：** 在开发过程中发现边缘情况时，及时添加注意事项。
* • **每周：** 审查过时信息，更新已演变的模式。
* • **重大变更后：** 记录新的架构决策及其理由。
* • **修复错误后：** 在“注意事项”部分添加预防策略。

**效果：** 维护 `CLAUDE.md` 一个月后，会话效率能提升 40-60%。Claude 无需解释，就能建议符合你架构的解决方案。由于常见错误被明确记录，错误预防能力也提升了。
