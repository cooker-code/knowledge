## 类目定位

| 项 | 内容 |
|---|---|
| 一级类目 | Agent 与 AI 工程 |
| 二级类目 | Prompt Engineering |
| 核心问题 | 如何把系统提示词、用户提示词、示例、反例、格式约束和安全边界设计成可复用的 Agent 指令接口 |
| 不解决什么 | 不承接纯模型提示词技巧、图像视频提示词、营销式 Prompt 清单；这些默认留在 LLM 与大模型或个人成长 |
| 用户当前认知假设 | L2-L3：知道 Prompt 有用但不稳定，当前重点是把 Prompt 当作工程接口而不是玄学话术 |

## 排重准则

问题指纹：

```text
提示词层级 + 约束对象 + 示例/反例机制 + 输出契约 + 失败边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是提示词技巧 | 没有任务类型、输入输出和失败边界时降权 |
| System Prompt vs User Prompt | 层级不同，可以拆分 |
| Prompt 与 Skill | 如果核心是能力封装和文件结构，归工具调用 / Skill |
| Prompt 与 Harness | 如果核心是状态机、门禁和运行时，归 Harness Engineering |
| Prompt 注入 | 作为 Prompt 安全边界记录，同时可与安全与权限互链 |

## 待补问题

| 问题 | 为什么要补 | 优先级 |
|---|---|---|
| Prompt 作为接口契约 | 防止只沉淀“怎么问” | 高 |
| CoT / ToT / ReAct 的工程边界 | 防止把推理范式当万能提示词，明确什么时候迁到工具调用或 Harness | 高 |
| Prompt 版本治理 | 团队使用时需要审查、回滚和变更记录 | 中 |
| Prompt 失效案例 | 只有反例才能判断提示词边界 | 高 |
