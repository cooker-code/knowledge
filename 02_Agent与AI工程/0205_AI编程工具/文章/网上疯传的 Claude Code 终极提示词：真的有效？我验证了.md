这个提示词到底在说什么

最近在中文技术社区流传一份 Claude Code / Codex 的「目标 goal 终极提示词」，声称可以实现「代理全天运行、零人工干预」，设定目标后退后一步，回来时收获已完成的工作。看起来像是每个开发者的梦中情词。

原文结构很清晰：先定义最终目标，再给背景信息、成功标准、操作规则、质量标准和最终交付物。最后加上一句「先输出计划，然后从头到尾执行，在完成或真正被阻塞前不要检查或停止」。

看着很猛，但问题是：它到底管不管用？

社区验证：这不是新鲜事，但确实有效

我在 Reddit r/ClaudeCode、Indie Hackers 和 Dev.to 上交叉搜索了类似的 prompt 模板和社区实际反馈。结论并不意外：

●Anthropic 官方最佳实践文档明确建议：在 Claude Code 中使用**明确的验证标准**和**定义什么是「Done」**。这份网传提示词的「成功标准」「自我验证」「始终朝目标前进」与官方推荐的方法论一致。

●Reddit 社区有用户实测：「给 Claude Code 设定大量规则并安排好后，它写了 99% 的代码且不需要修改。」关键点不在提示词本身，在于**约束质量**。

●Indie Hackers 上有人逆向了 Claude Code 的系统提示架构，指出：让 Agent 自主工作最有效的方式是**在 user message 中注入上下文**，而不是把所有规则塞进 system prompt。这点和提示词中的「先规划」思路吻合。

**小结：**这份提示词的方法论没有问题——它本质上是一个结构化的任务指令模板。核心思路与 Anthropic 官方推荐做法一致，社区实测也确实产出过高质量代码。但问题出在「终极」两个字上。

为什么别照抄：四个翻车点

**1. Token 燃烧问题**

网传文案写着「全天运行」，但 Claude Code 有 Token 限制——Max 计划是 5 小时窗口，Pro 计划更短。有用户实测 3 分钟内烧完 5 小时额度。一个不加约束的 prompt 让 AI「自主推进」，结果可能不是代码写完了，是额度先没了。

**2. 规则是软约束，AI 可以不听**

有人写了 300+ 行 TDD 规则放进 CLAUDE.md，结果 Claude Code 选择性忽略。因为写在规则文件里的东西只是「期望」，不是强制执行。真正管用的是 Hooks——代码层面的强制检查，不通过测试就不允许提交。提示词里写「自我验证」，但如果没有实际的测试脚本来验证，那就是空话。

**3. 模板是通用的，项目是具体的**

提示词里的占位符「【你正在构建什么】」「【技术栈】」「【当前状态】」看似简单，但如果填写质量差，输出就差。一个 React 前端项目和一个 C 编译器项目需要的成功标准、验证方法完全不同。套用同一份模板，等于什么都没定义。

**4. 「零人工干预」是误导**

Anthropic 自己做过实验：让 Claude 在几天内从零构建一个能编译 Linux 内核的 C 编译器——但这不是一个 prompt 搞定的，是**约 2000 次会话**累积的结果。所谓「退后一步回来收获已完成工作」，只适用于有完备测试、定义清晰的小任务。

警告照抄模板的三个信号

**Token 消耗暴增** — AI 在没有实际约束的上下文里反复推理

**输出质量下降** — 通用模板没有针对性信息，AI 开始编造

**偏离原始目标** — 模糊的成功标准导致 AI 做了别的事

正确的用法：让 AI 根据自己的项目改

这份提示词真正的价值在于**框架**，不在于**内容**。拿到它之后不要直接粘贴使用，而是当作一个改写的起点。

**第一步：填真实的约束**

不要写「构建一个 Web 应用」，要写「用 FastAPI + SQLite 实现用户登录，支持 JWT token，密码用 bcrypt 加密，已有数据库迁移脚本在 /migrations 目录」。越具体，AI 的自主输出越可靠。

**第二步：把成功标准写成可验证的命令**

原模板写「交付的最终成果无错误运行」——这是废话。改成：

```
成功标准：
1. pytest tests/ 全部通过（0 failures）
2. curl localhost:8000/health 返回 200
3. 代码覆盖率 >= 80%
```

**第三步：加预算限制**

模板没有提到 token 预算。加上：「如果你已经运行超过 30 轮或上下文超过 80% 还没完成，停下来报告进度，不要继续。」这条比「始终朝目标前进」更实用。

**第四步：加范围边界**

模板有「始终朝目标前进」，但没有说什么是不在范围的。明确写上：「不要改动已测试通过的文件」「不要新增依赖，除非在成功标准中明确列出」。

全量提示词原文（供参考）

以下是社区流转的完整模板原文。不建议直接复制使用，而是当作改写起点：

```
/Goal 【最终目标 - 用一句话描述
  "完成"是什么样子】

::: 背景信息
- 项目：【你正在构建什么】
- 技术栈：【语言、框架、基础设施】
- 当前状态：【目前已有的内容】
- 工作目录：【路径或仓库】
- 约束条件：【预算、时间等】
- 受众：【为谁而做】

::: 成功标准（全部必须为真）
1. 【具体可衡量的成果】
2. 【具体可衡量的成果】
3. 【具体可衡量的成果】
4. 交付的最终成果无错误运行
5. 提供证明（截图、测试输出、URL）

::: 操作规则（不可商议）
1. 先规划。编写任何代码前先输出
  编号任务清单。
2. 自主工作。除非确实被阻塞，
  否则不要询问澄清。
3. 自我验证。每个步骤后：运行测试、
  检查输出、确认有效。
4. 自行调试。失败时，诊断并修复。
  不要交还给我。
5. 善用工具。使用所有 MCP、终端、
  Web、代码执行。
6. 严禁占位。无 TODO、无存根。
7. 记录进度。跟踪已完成/进行中/
  决策/阻塞项。
8. 始终朝目标前进。
9. 若被阻塞。记录阻塞点，
  并行推进其他一切。
10. 停止前检查成功标准全部达成。

::: 质量标准
- 代码：干净、已类型化、遵循约定
- 设计：像已获得充分融资的初创产品
- 输出：经得起资深代码评审
- 文档：每个新模式/变量均有记录

::: 最终交付物
- 确认每项成功标准均已满足
- 每个创建/修改的文件
- 运行/测试/部署方法
- 做出的决策与已知限制

先输出你的计划。然后从头到尾执行，
在完成或真正被阻塞前不要检查或停止。

设定目标。退后一步。
回来时交付已完成的工作。
```

**建议**把空白占位符【】替换成你项目里真实的信息。成功标准写具体的测试命令，而不是模糊描述。这才是这个模板真正起作用的地方。

给小白的三条行动建议

▸**先试一个小功能**：别拿整个项目试。选一个已经定义清楚、有测试的功能点（比如「增加一个登录 API 端点」），填入模板后运行，看输出是否符合预期。这比直接让 AI 重构核心代码安全得多。

▸**成功标准必须能跑**：不要写「代码无错误」，写「执行 pytest 通过且覆盖率 80% 以上」。如果项目没有测试，先让 AI 写测试，再用 goal 提示词让它写功能。反过来你会浪费大量时间调试。

▸**每次只改一个变量**：第一次用原版模板，第二次加预算限制，第三次加范围边界。逐次验证哪个改动真正提升了输出质量。别一次改八个地方然后不知道哪个有用。
