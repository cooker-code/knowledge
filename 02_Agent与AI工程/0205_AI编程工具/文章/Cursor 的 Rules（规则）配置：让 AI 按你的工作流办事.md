### 开篇

你肯定也遇到过：同一个问题，不同时间问出来的风格完全不一样；昨天还说好的错误处理，今天又换了一套。真凶往往不是工具，而是“没有把话说在前面”。Cursor 里有个很好用但常被忽略的入口——Rules（规则）。把团队的编码风格、异常策略、注释语言、提交规范这些“大家心照不宣”的东西写清楚，AI 和同事都会更省力。设一次，全项目受益。

### 入口在哪

* 先别急着开干，先找到门在哪：
* 路径：Settings → Rules, Memories, Commands → Rules
* 支持新增/编辑多条规则，按优先级排列；可结合 Profiles 为不同场景启用不同规则集。

### 为什么一定要配 Rules

* 说白了，我们不是不会写，而是写着写着就跑偏；把轨道铺好，车自然不乱开：
* **统一输出**：让 AI 产出的代码风格、注释语言、错误处理方式保持一致。
* **减少反复说明**：把“老生常谈”的要求写成规则，避免每次都重新强调。
* **可迁移**：换仓库、换任务，直接切换 Profile 即可沿用同一套约束。

### 规则怎么写更有效

* 规则不是作文比赛，短平快更有执行力：
* **短句 + 祈使**：一句话一个动作，避免长段落。
* **先原则，后细节**：用“必须/禁止/优先”描述边界，再落到实现偏好。
* **可验证**：尽量写成能被检验的约束（如“返回值使用 Result 类型”）。
* **按优先级排序**：把关键的不变项放在前面，避免冲突时被覆盖。

### 一份可直接复用的规则清单（示例）

下面这份就当“起步模板”，拿去即用，按你的技术栈再微调：

```
# 输出与风格  
- 所有代码与注释使用简体中文注释，英文保留专有名词。  
- 代码必须可运行、可复制，包含必要的 import 与依赖提示。  
- 变量与函数命名使用完整单词，避免缩写；函数名用动词短语。  
  
# 结构与控制流  
- 优先使用早返回；避免超过 3 层的深度嵌套。  
- 仅在确有必要时使用 try/catch；禁止空 catch。  
- 函数保持单一职责，超过 80 行必须拆分。  
  
# 错误与日志  
- 统一错误信息格式：`[模块] 语义化错误 + 解决建议`。  
- 在关键分支添加结构化日志（等级、场景、关键参数）。  
  
# 文档与示例  
- 提供“最小可运行示例”与“边界条件案例”。  
- 仅写必要注释，解释设计动机与非显而易见的约束。  
  
# Git 提交  
- 使用约定式提交：`feat|fix|docs|refactor|test|chore: 概述`。  
- PR 描述包含动机、取舍、风险与回滚方案。
```

### 三个高频场景的专项规则

* 行业千差万别，但骨架是通用的，先把“骨感部分”立起来：
* **后端接口**：

+ 所有接口返回统一响应包装；错误码表参照既有枚举；分页参数固定为 `page`、`pageSize`。

* **前端组件**：

+ 组件必须无副作用；Props 明确类型；状态提升优先；样式使用模块化并避免全局污染。

* **数据脚本**：

+ 默认幂等；添加 `--dry-run` 开关；对受影响的行数进行阈值保护与确认。

### 一分钟上手

* 现在就动手，三步把规则落地：
* 打开 Rules 设置页，创建一个“项目通用规则”分组：

1. 粘贴上面的清单并按你的技术栈微调；
2. 把“输出与风格”“结构与控制流”放在最顶端；
3. 保存为一个 Profile（如“Dev-Strict”）。

* 在需要更轻量的探索场景，新建 Profile（如“Exploration”），只保留“输出与风格 + 文档与示例”。

### 进阶用法

* 玩得顺手以后，再把这些“加速器”装上：
* **与 Keyboard Shortcuts 联动**：给“插入规则摘要”或“打开规则面板”绑定快捷键，写到一半随时复查。
* **与 Default Layout 联动**：编码型 Profile 用 Editor 作为默认布局；方案型 Profile 用 Agent 作为默认布局，放大规则对对话的约束力。
* **与 Memories 区分**：Rules 定义“恒定准则”，Memories 存“项目具体事实”（依赖、端口、约束版本等）。

### 常见误区

* 这些坑我都踩过，帮你先标红：
* 误区 1：把需求文档原封不动丢进规则。规则要短、可执行。
* 误区 2：规则越多越好。超过 20 条就难以维护；精炼、合并、排序更重要。
* 误区 3：只写“应该”。请用“必须/禁止/优先”明确强度。

### 结尾

说到底，规则不是束缚，而是把“共识”写清楚。把经验写进 Rules，AI 才能真正变成“团队的一员”。今天花 10 分钟把骨架搭好，明天你就会发现：答复更稳了、代码更整齐了、沟通更省心了。
