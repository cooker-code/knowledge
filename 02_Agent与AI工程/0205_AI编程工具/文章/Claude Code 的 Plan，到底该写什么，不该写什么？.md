我现在越来越觉得，很多人不是不会用 Claude Code，而是把 Plan 这件事做得太满了。

真正有效的 Plan，不是提前把执行细节写完，而是先把边界定清楚，把后面的执行空间留出来。再往后，任务拆分、sub-agent 分工、文档沉淀和 loop 验证，才是把大任务跑顺的关键。

---

## 先说结论

Plan 最适合承载的，是接口、边界和数据流，不是具体文件路径、代码片段和一长串实现步骤。

任务一旦变重，继续把计划写长，通常解决不了上下文问题。更稳的办法是把 task 拆小，把执行交给 sub-agent，把过程落到文档里，再补上验证和重试机制。

---

## 为什么这件事值得现在讲

很多人一开 Claude Code，就会本能地想把计划讲得越细越好，觉得这样更稳。

但我现在的判断正好相反：计划太细，往往不是更稳，而是更重。上下文会被塞满，模型自己的探索空间会被压缩，最后执行效果反而容易变差。

真正该想清楚的，不是“要不要先 plan”，而是 Plan 应该停在哪一层，后面的工作又该怎么接出去。

---

## 最值得展开的几个点

### 1. Plan 里写什么、不写什么

Plan 该写的是接口定义和数据流向，不该写的是具体文件路径和代码片段。

原因很简单。探索代码库本来就是 agent 的工作。如果人提前把路径、实现方式、局部代码都规定死，不一定会更高效，反而容易制造上下文污染。计划一旦太细，也会直接压缩模型发挥空间。

**这对读者的价值：** 如果你现在写 Plan 的方式，已经越来越像把实现步骤逐条铺开，那大概率已经写过头了。更有效的做法，通常是把结构说清，把细节留给执行阶段去发现。

---

### 2. 用 sub-agent 解决上下文不够的问题

大任务的上下文不够，本质上不是提示词问题，而是任务拆分问题。

我的建议会更直接一点：不要把所有事情都压在一个 agent 身上。一个 agent 负责 plan，多个 sub-agent 负责并行执行；或者一个 agent 负责规划，另一个专门负责测试。任务拆小之后，单个上下文的压力自然会降下来。

放到 Claude Code 里，这件事通常可以更明确地做：先 create tasks，再决定哪些部分并行，哪些部分单独分给测试、review 或验证。相比把 Plan Mode 当成主方法，这条路更适合大任务。

**这对读者的价值：** 如果你遇到的核心问题是“这个任务太大，一次说不清、一次也跑不完”，那更该优化的往往不是 prompt，而是任务结构。先拆，再分工，再并行，通常比继续补计划更有效。

---

### 3. 用 Epic / Sprint 文档驱动，而不是只靠一轮对话

如果一个任务会跨几轮推进，我不太建议把所有信息都压在聊天上下文里。

更稳的做法，是把任务先当成 epic，写 overview，再拆成 sprint 文档；或者先形成 PRD / Spec，再继续拆成 Tasks 文档逐个推进。重点不在于文档变多，而在于文档开始承担流程组织的作用。

这样做有两个直接好处：执行路径会更清楚，后面回溯也更方便。很多关键判断如果只留在聊天记录里，过几轮之后就很难再捡回来。

**这对读者的价值：** 如果任务涉及多人协作、跨阶段推进，或者后面还要复盘，那么把 plan 落成 epic、sprint、tasks 这类文档，通常比把信息全压在一次对话上下文里更稳。

---

### 4. 自动化 loop 机制，决定流程能不能长期跑起来

再往下一层，我更看重的是 loop 机制是不是完整。

比较像样的流程，不该只是“接单 → 干活 → 交付”这么短。更合理的是：Issue 驱动进来，先执行，做完先质检，隔一段时间再做独立验证；如果失败，就带着上下文重试；连续失败 3 次，再挂起并叫人接手。

到这一步，讨论的已经不只是 Plan 该怎么写，而是 Agent 流程能不能形成闭环。Plan 定边界，sub-agent 拆执行，文档管过程，loop 负责把质量控制和异常回退补上。

**这对读者的价值：** 如果你想把 Claude Code 从一次性帮手，慢慢变成可持续运行的工作流，loop 机制几乎绕不过去。没有验证和重试，前面的 plan 和拆分只能算半套系统。

---

## 哪些细节容易被忽略

### 真正该优化的，往往不是 Plan Mode 本身

* 问题通常不在某个模式按钮怎么开，而在整个 Agent 工作流怎么组织。
* Plan 只是其中一层，后面还连着 task 拆分、sub-agent 分工、文档沉淀和验证闭环。
* 所以不要把注意力全放在“Plan Mode 怎么用”上，更该看的是整套协作结构是不是顺。

### 这几层最好连起来看

* Plan 写粗一点，是为了给执行留空间。
* sub-agent 拆分，是为了解决大任务上下文不够的问题。
* Epic / Sprint / Tasks 文档，是为了把执行过程沉淀下来，方便 review 和回溯。
* loop 机制，则是在前面这些基础上，把重试、验证和人工接管补齐。

---

## 如果你也在做类似事情，可以怎么用

### 1. 先约束 Plan 的颗粒度

优先写清楚接口、边界、依赖关系和数据流，不急着把文件路径和代码片段塞进去。

---

### 2. 把大任务改写成任务系统，而不是大段计划

先 create tasks，再考虑哪些部分可以并行，哪些部分需要单独 agent 负责测试、review 或验证。

---

### 3. 让文档和 loop 接上执行过程

如果任务会持续推进，就把 plan 落成 epic、sprint、tasks 这类文档；如果还要追求稳定性，就把质检、独立验证、失败重试和人工接管也补上。

---

## 最后一句判断

## 我现在对 Claude Code 的一个判断很明确：Plan 不该负责把路的每一步都走完，它更该负责把边界先定清楚，剩下的交给拆分、执行、验证和回环。
