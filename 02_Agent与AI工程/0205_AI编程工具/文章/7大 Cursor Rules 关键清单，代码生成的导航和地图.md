### 一、为什么要先设规则？

* 统一风格与约定，避免生成代码与既有规范冲突。
* 限制工具的自由度，防止引入不合规库或反模式。
* 提高可维护性：新代码应与团队长期维护成本一致。
* 把“模糊的期望”转化为“可执行的约束”，让 AI 输出更稳定。

### 二、规则的取舍与数量（5–10 条为宜）

高质量的规则不在于数量，而在于“`清晰 + 关键`”。

建议把规则控制在 5 到 10 条，覆盖最能影响质量与一致性的领域。

每条规则应是可执行、可验证的：能被人或自动化工具检查。

### 三、哪些主题必须包括（优先级指南）

* 语言与版本（如：TypeScript 4.9，Node 18）
* 架构风格（MVC、SPA、微服务、单体等）
* 命名与文件组织（文件夹约定、组件命名规则）
* 错误处理与日志策略（统一错误类型、埋点策略）
* 依赖与库禁用清单（允许的核心库、禁用或需审查的库）
* 测试覆盖与质量门槛（单测覆盖率、必测项）
* 性能/安全约束（响应时间、输入校验、审计要求）
* 提交与 PR 规范（commit message 格式、单 PR 大小上限）

### 四、如何写出一条“好”规则（模板）

* 明确要解决的问题（为什么）
* 具体的约束或例子（怎么做）
* 可验证的衡量方式（如何检查）

示例：

> “避免在前端直接使用 any 类型（为什么：影响类型安全）。约束：PR 不允许引入 any，除非在评论里注明临时原因并附任务号（怎么做）。CI 使用类型检查工具阻断（如何检查）。”

### 五、将规则落地到工具链

* 编辑器层面：把 .cursorrules 放在仓库根目录，团队成员通过版本控制共享。
* 自动化层面：在 CI 中加入规则校验（lint、类型检查、自定义脚本）。
* 代码生成层面：在对 AI 发起提示/指令时，显式加入“遵守 .cursorrules 中的规则”或把规则作为 prompt 的一部分。
* 评审层面：PR 模板中加入“检查规则项”核对清单。

### 六、利用 Cursor 的自动化能力（实战建议）

* 快速起步：Cursor 提供 `/generate rules` 命令，可以扫描现有代码生成初版规则（尤其对代码风格有帮助）。
* 手工补充：自动生成的规则适合基础风格，但架构、设计模式和核心业务约束需要`人为补齐`。
* 不要完全依赖一键生成：AI 生成的“模板化”规则常含泛泛而谈的表述，建议人工精炼、删冗留精。

一定要自己动手改一下，不让光是写`rules`，Cursor都能给你干翻天！

### 七、示例：典型的 .cursorrules（TypeScript 项目）

```
# .cursorrules（示例片段）  
# 语言与版本  
language: "TypeScript"  
ts_version: "4.9"  
  
# 代码规范  
rules:  
  - name: "strict-mode"  
    desc: "开启 TypeScript 严格模式"  
    enforce: true  
  - name: "no-any"  
    desc: "禁止使用 any，特殊情况下需注释并关联任务"  
    enforce: true  
  
# 组件与状态管理  
component:  
  - "函数式组件优先"  
state:  
  - "优先使用 React Context，小型应用可使用简单 local state"  
  
# 测试  
testing:  
  - "单元测试覆盖率 >= 80%"  
  - "使用 React Testing Library 进行组件测试"  
  
# 依赖管理  
dependencies:  
  allowed:  
    - "react"  
    - "react-dom"  
    - "lodash (需评估)"  
  forbidden:  
    - "moment" # 使用 date-fns 或 dayjs 替代
```

### 八、维护规则的实践建议

* 规则即代码：把规则文件纳入版本控制并写变更日志。
* 周期性复审：每个迭代（或每季度）评估规则是否匹配现状。
* 以度量驱动调整：通过缺陷率、PR 驳回原因、自动化检查的失败率来判定哪些规则需要松紧。
* 培训与 onboarding：把关键规则写进新成员入门文档，减少沟通成本。

### 九、常见误区

* 规则越多越好：反而会抑制创新与速度，应只保留高影响力项。
* 把“模糊目标”写成规则：如“写出干净的代码”不可检验，需拆成具体实践。
* 完全依赖自动生成：将自动生成的内容作为草稿，而非最终稿。

### 10分钟实践行动清单

1. 先在仓库根目录创建 .cursorrules，写出 5–10 条覆盖“语言/版本、命名、依赖、测试、异常处理”的核心规则。
2. 把规则接入 CI 和编辑器，确保“写代码时就被提醒、提交时就被校验”。
3. 使用 Cursor 的 /generate rules 快速生成样板，然后人工精炼成团队标准。
4. 将规则文件作为团队合同——短、小、可验证，并定期复审。

### 最后

AI不是万能的，更不要让AI自由发挥！

要把 AI 当作需要明确指令和监督的得力助手，而不是能读懂你心思的神奇黑盒。

规则就是给AI工作的“地图”。给 AI 明确边界，才能让它在正确的轨道上高效输出；把规则写清、做细、能检验，能让团队少走弯路，多做有价值的开发工作。
