Claude Code 在 `5.29`上了一个叫 `Dynamic Workflows`（动态工作流）的东西，现在还是研究预览，得升到 v2.1.154 以上才有。我用了一段时间，踩过几个坑，这里把心得记一下。

先说结论：它不是"更猛的自动模式"。你最好把它当成——Claude 替你写了一段编排脚本，丢到后台跑一大堆子代理，跑完给你一个结果。理解了这一点，剩下的就好说了。

## 它和子代理、技能有什么不一样

这三样都能干多步骤的活，容易混。区别就一句话：**谁拿着那张计划表**。

平时你用子代理或者技能，拿计划表的是 Claude 自己。它一回合一回合地想"下一步派谁去做什么"，每个中间结果都塞回它的上下文里。活儿一大，上下文很快就满了，思路也容易飘。

工作流是把这张计划表挪进了代码。Claude 给你的任务写一段 JavaScript 脚本，由一个独立的运行时在后台跑。循环、分支、中间结果全在脚本变量里待着，不进 Claude 的上下文——它的上下文最后只留一个汇总结论。

|  | 子代理 | 技能 | 工作流 |
| --- | --- | --- | --- |
| 是什么 | Claude 派出去的工人 | Claude 照着做的指令 | 运行时执行的脚本 |
| 谁定下一步 | Claude，逐回合 | Claude，按提示 | 脚本 |
| 中间结果在哪 | Claude 上下文 | Claude 上下文 | 脚本变量 |
| 能复用的是 | 工人的定义 | 那套指令 | 整个编排逻辑 |
| 规模 | 每回合派几个 | 同上 | 一次几十到几百个 |

挪进代码还顺带带来一件事，我觉得这才是它真正值钱的地方：它能跑一套**可重复的质量套路**。比如让几个代理各自查，再让另一批代理专门去推翻前面那批的结论；或者同一个方案让它从三个角度各写一版再互相比。这种东西你在一问一答里很难稳定做出来。

## 什么时候用，什么时候别用

得先泼盆冷水：这玩意儿很费 token。一次运行烧的量可能比你平时一整次会话还多。所以真正用得好的人，第一件事不是想着怎么花活，而是知道**大部分时候根本不该开它**。

值得开的，是那种"一个对话根本协调不过来"的活。我自己用下来主要这么几类：

整个仓库级别的扫——跨服务找 bug、做性能审计、做安全审计。Claude 并行地把仓库翻一遍，每条发现再单独验证一次，所以报告里出来的基本是真问题，不是它顺嘴瞎猜的。

大迁移——换框架、清理废弃 API、上千个文件的语言移植这种。一个最出名的例子是 Bun 的重写，有人用动态工作流把它从 Zig 移植到 Rust，七十五万行代码，原来的测试套件跑过了 99.8%，从第一次提交到合并花了十一天。

还有就是那种错了代价很大、你想被人复查两遍的活。工作流会让 Claude 独立做几次，再派一批"对手"代理专门去挑刺，挑完了结果才到你手上。

反过来，下面这些就别上工作流，纯属浪费：改一个文件、修一个你已经定位清楚的小 bug、或者那种你需要边做边插话来回讨论的探索性任务——因为它跑起来之后是**不让你中途插话的**。

记一句话就够了：**只有当"协调"本身成了瓶颈，才轮到它。**

## 三种开法

### 想先看看它长什么样，就跑 `/deep-research`

这是内置的，最稳的入门方式。给它一个问题，它会从好几个角度撒网搜，把搜到的来源抓回来互相对，给每条结论投票，最后吐一份带引用的报告，没通过交叉验证的结论直接扔掉。

```
/deep-research v20 到 v22 之间 Node.js 的权限模型改了什么？
```

跑的时候你的会话照样能用，最后拿到的是报告，不是一长串流水账。想感受"后台编排"是什么意思，从这儿开始最直观。

### 心里有数了，就在提示里塞个 `workflow`

如果你已经认定这活该用工作流，最干净的办法是在你的提示里随便哪儿写上 `workflow`这个词。Claude Code 会把它高亮出来，然后给这个任务写脚本，而不是一回合一回合地手动做。

```
Run a workflow to audit every API endpoint under src/routes/ for missing auth checks
```

好处是它只把这一个任务变成工作流，不动整个会话。万一手滑触发了，按 `alt+w`这次就不算。

### 懒得判断，就开 `ultracode`让它自己看着办

`ultracode`把 `xhigh`的推理强度和自动编排捆在一起。开了之后，会话里每个像样的任务它都会自己掂量要不要变成工作流。

```
/effort ultracode
```

一个请求可能连着触发好几个工作流——一个搞懂代码，一个动手改，一个验证。代价是每个请求都更慢更费钱，官方也建议配着自动模式一起用。

提醒一句，这是我自己吃过亏的：`ultracode`只管当前会话，开新会话会自动回去。所以干完大活记得 `/effort high`切回来，不然你回头写个小脚本它也给你调一支大军。

## 怎么把脚本写得让它做对

脚本是 Claude 替你写的，但写得好不好，看你怎么描述任务。几条经验：

**说清边界，别只说目标。**"审计认证逻辑"太虚；"审计 `src/routes/`下每个端点是不是漏了鉴权检查"就实在多了。边界清楚，它才能把活切成干净的并行小块。

**把验证标准也写进去。**工作流的强项就是交叉核查，你不写它就少做。与其说"找出所有 N+1 查询"，不如加一句"每个发现都给出能复现的代码片段和文件行号"。

**主动帮它省钱。**描述任务时直接告诉它哪些阶段用不着最强的模型，让它把那些环节甩给小模型。一句话有时候能省一大截。

## 跑起来之后，盯着它

工作流在后台跑，但后台不等于黑箱——随时 `/workflows`就能列出在跑的和跑完的，方向键选一个回车进去看进度。进度视图按阶段列出各自的代理数、token 总量、用了多久，你能一层层钻进去，看某个代理具体在干嘛、调了什么工具、出了什么结果。

几个键值得记：

```
↑ / ↓     选阶段或代理  
Enter / → 钻进去；Esc 退一层  
p         暂停 / 恢复  
x         停掉选中的代理；焦点在整个运行上时停掉整个工作流  
r         重启选中的运行中代理  
s         把这次的脚本存成命令
```

输入框下面那条任务面板也会滚一行进度，方向键聚焦、回车展开。

权限这里有个坑得说一下：你的权限模式只管启动时那个确认框；工作流派出去的子代理，不管你会话是什么模式，**一律按 `acceptEdits`跑**，文件改动自动批。但不在允许列表里的 shell 命令、网页抓取、MCP 工具，还是可能跑到一半弹出来问你。所以要跑长活之前，最好先把代理会用到的命令加进允许列表，免得它跑半截卡那儿等你点确认。

## 成本这块，得有点纪律

再啰嗦一遍，因为它真的费钱：工作流一口气派一堆代理，一次跑下来的 token 可能明显超过你在对话里做同一件事，而且照常算进你套餐的用量和限额。

几条我自己定的规矩：

第一次用，挑个小范围的活先跑，摸清在你这边大概是个什么量级再说。大跑之前 `/model`看一眼当前模型，尤其你平时爱切小模型干杂活的话。任务里主动让它把不吃重的阶段交给小模型。任何时候都能从 `/workflows`里把一次运行停掉，已经跑完的不会白跑。

还有，第一次触发工作流时，Claude Code 会先把要跑的内容摆给你看——CLI 里会列出规划好的各个阶段，给你"跑""跑且这个项目以后别问了""看看原始脚本""取消"几个选项，`Ctrl+G`能在编辑器里打开脚本，`Tab`让你开跑前再改改提示。开跑前瞄一眼计划，是最便宜的一道防线，别嫌麻烦。

## 跑通一次，就把它存下来

这是"凑合能用"和"用得顺"的分界。Claude 给某个你以后还要重复的活写了个工作流——比如每个分支都要走一遍的代码评审——别让它随会话没了。

`/workflows`里选中那次运行，按 `s`，存的时候 `Tab`切两个位置：`.claude/workflows/`放项目里，跟着仓库共享给所有人；`~/.claude/workflows/`放你主目录，所有项目都能用但只有你看得见。存完它就成了一个 `/<名字>`命令，以后会话里直接调，还会出现在 `/`的自动补全里。两个地方重名的话，项目里那个优先。

把"跑通过一次的编排"沉淀成一个团队能共用的命令，这是工作流比一次性子代理强的地方。一次投入，反复吃。

## 几个能直接抄的模板

全仓库安全加固：

```
Run a workflow to scan the entire codebase for missing input validation,  
unsafe auth checks, and unsanitized SQL, and for each finding include a  
reproducible snippet and the file:line location.
```

大规模迁移：

```
Run a workflow to migrate all components under src/ from the deprecated  
v1 API to v2, porting one file at a time, with a reviewer agent verifying  
each port stays behavior-identical before moving on.
```

方案压力测试：

```
Run a workflow to draft three independent migration plans for moving our  
session store off Redis, then have adversarial agents try to break each  
plan and report which assumptions don't hold.
```

这几个的共同点：边界清楚、自带验证或对抗、能拆成并行的小块。工作流最吃这种形状的活。

## 几条硬限制，别撞上

脚本是在一个跟对话隔开的环境里跑的，有几条死规矩心里得有数：

跑的时候不让你插话，只有代理的权限弹窗能让它停下来。所以要是某个阶段之间你必须签字确认，就把那个阶段拆成单独一个工作流。

脚本本身碰不了文件系统和 shell，读写文件、执行命令都是代理干的，脚本只负责调度它们。

最多 16 个代理并发（机器核少就更少），一次运行总共封顶 1000 个代理，防止它陷进死循环。

恢复只在同一个会话里管用——你停了再续，跑完的代理返回缓存结果，剩下的实时跑；但你一旦退出 Claude Code，下次会话这个工作流就得从头来。

要是团队里有人不该碰它，可以关：`/config`里关掉 Dynamic workflows，或者 `~/.claude/settings.json`里写 `"disableWorkflows": true`，或者设环境变量 `CLAUDE_CODE_DISABLE_WORKFLOWS=1`。组织层面就在受管设置里统一关。关了之后内置命令没了，`workflow`关键词不触发了，`ultracode`也从 `/effort`菜单里消失。

## 最后

用顺手之后我的体会是：这东西不是替你"想"的，是替你"协调"的。你的角色从一步步指挥，变成了定目标、划边界、把结果。任务真的大、协调成了瓶颈，再请它出场；平时盯紧 `/workflows`别让后台变黑箱；钱这块先小范围、先看计划、该用小模型用小模型；跑通了顺手 `s`一下存起来。就这些。
