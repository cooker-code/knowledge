## 工程落地心法

规格驱动开发的核心，是通过标准化的规格文件将模糊需求转化为`可执行、可跟踪、可验收的开发流程`。本质是把传统软件工程的严谨性融入 AI 编程，让开发从 “凭感觉” 的 “氛围编程”（Vibe Coding）变成 “可把控”（Viable） 的工程落地过程。

一位网友推荐了Github开源的一套更为专业的`规范驱动开发`的工具 Spec kit。别着急，立马来试试！

## Spec Kit 为何物？

Spec Kit是Github开源的一个开发工具包，基于一个简单而强大的理念：**先写规范，再写代码**。

不仅仅是方法论，它这是一套可落地的工具链，包含了：

* • Specify CLI工具
* • 自动化脚本
* • 模版文件

用一套固定的`流水线命令`，把`需求→计划→任务→实现→校验`串起来，尽量达到自动化、结构化地完成。

* • 从确立团队原则（/constitution）
* • 说清楚要做什么（/specify）
* • 把不清楚的先问清（/clarify）
* • 定技术方案（/plan）
* • 自动拆任务（/tasks）
* • 一致性分析（/analyze）
* • 开始落地实现（/implement）

最后，交给Agent们执行，诸如`Claude/Cursor/Copilot`等AI编程智能体来执行。这套工具，当前支持的AI Agents包括：

Spec Kit 的价值不在于“它能不能一键出代码”，而在于它把团队的“意图”变成有约束、有证据链、可再生的资产。当规格成为“唯一真相”，当测试把“真相”守住，代码就不再是“手工翻译”，而是可再生成的“表达”。如果你也在找一种高质高效、又不放弃工程纪律的 AI 开发方式，那么也许可以一起试试它。

## Trae + spec-kit 不支持，又能怎？

细心的同学，可能会发现 Trae 并没有在 Spec Kit 的官方支持范围之内，但并不是不能用。笔者尝试了在Trae使用，竟然走通了，而且效果还是很不错的。

由于不是详细的教程，这里只是把过程记录下来。`安装spec-kit的过程在此就忽略`了，可以移步官方学习。

### 1. 初始化工程

在安装了 Spec kit命令行的系统中，运行初始化项目的命令：

```
specify init deeptalk
```

选择合适的Agent，由于官方没有列出 Trae， 笔者选择`Cursor`。创建完成后再导入到 Trae 工具。这里笔者选用了`Claude-4-sonnet`模型作为`运行大脑`。

### 2. 翻译相关模版文件

由于生成的文件全是英文的版本，笔者尝试将其先翻译成中文，方便查看。

```
将 #.cursor 目录下的所有md文件的核心内容翻译成中文。
```

于是就得到了一份中文的`**.zh.md 文件`。

### 3. 制定项目规章（最重要）

首先，需要建立项目的核心治理原则和开发标准。在对话框中输入：

```
#speckit.constitution.zh.md  制定以代码质量、测试标准、用户体验一致性及性能要求为核心的原则
```

等待Agent执行完后，我们会看到生成了 `constitution.md` 文件，里面的内容是《规格管理和开发工具套件项目章程》，是整个过程的灵魂所在！

摘取了几个核心章节：

#### 🎯核心原则

1. 1. 代码质量至上 - 严格的代码质量标准和审查要求
2. 2. 全面测试标准 - TDD开发模式，90%以上测试覆盖率
3. 3. 用户体验一致性 - 统一的设计系统和交互标准
4. 4. 性能要求标准 - 明确的性能基准和监控要求
5. 5. 工具链标准化 - 统一的接口规范和命令格式

#### ⚙️ 技术标准

* • 开发环境要求（Node.js、Git、Markdown等）
* • 代码组织原则（模块化、目录结构、模板驱动）
* • 安全要求（输入验证、权限管理、安全审计）

#### 🔄 工作流程约定

有点像之前提到的6A工作流，包含6步开发流程：

1. 1. 需求分析 ( /speckit.clarify )
2. 2. 规格制定 ( /speckit.specify )
3. 3. 计划制定 ( /speckit.plan )
4. 4. 任务分解 ( /speckit.tasks )
5. 5. 质量检查 ( /speckit.analyze )
6. 6. 实施执行 ( /speckit.implement )

#### 🎯 主要用途

1. 1. 项目治理 - 为DeepTalk项目制定统一的开发标准和规范
2. 2. 质量保证 - 确保代码质量、测试覆盖率和性能标准
3. 3. 团队协作 - 规范团队的工作流程和协作方式
4. 4. 工具标准化 - 定义speckit命令行工具的接口规范
5. 5. 文档管理 - 建立统一的文档标准和格式要求

### 4. 生成需求规格说明书

接下来，描述要构建什么，专注于什么功能，而非技术实现。在对话框中输入：

```
#speckit.specify.zh.md  想实现一个类似 Kimi 的 AI 聊天应用，支持流式AI对话，支持查看历史聊天，用户可以建立新对话，支持通过github登录。
```

等待Agent执行完后，我们会看到：

生成了一个`001-ai-chat-app`，里面内容就是接下来AI生成的规则。包括：

1. 1. spec.md - 功能规范
2. 2. plan.md - 技术架构与执行计划
3. 3. tasks.md - 详细任务分解
4. 4. data-model.md - 数据模型设计
5. 5. contracts.md - API接口规范
6. 6. research.md - 技术调研
7. 7. quickstart.md - 快速开始指南
8. 8. checklists/requirements.md - 需求检查清单

### 5. 技术实现规划

接下来这一步，主要是指定`技术栈和架构设计`。在对话框中输入：

```
#speckit.plan.zh.md  使用vite+vue3+TDesign作为前端框架，使用node.js作为后端技术栈，采用一体化部署方案。调用 gIm4.6的api参考:https://docs.bigmodel.cn/api-reference/
```

等待Agent执行完后，我们会看到 plan.md 记录了详细的开发计划：

### 6. 自动分解任务

接下来这一步，将实现规划分解为详细的`可执行任务`。在对话框中输入：

```
#speckit.tasks.zh.md  
```

等待Agent执行完后，我们会看到 tasks.md 记录近 86个具体任务。

* • 86个具体任务 ，按6个阶段组织：

+ • 阶段1：项目初始化 (12个任务)
+ • 阶段2：基础设施 (21个任务)
+ • 阶段3-6：4个用户故事实现 (43个任务)
+ • 阶段7：完善优化 (10个任务)

* • 并行执行策略 ：65%任务可并行开发
* • MVP定义 ：44个任务，约2周开发时间

这里其实笔者有点差异，为啥估算出是约`2周开发时间`, 等后续开发《AI编程实践手册》的时候再深究其背后的细枝末节。

### 7. 实现策略

在生成完各种约定后，笔者问了下 "请问现在具备开始开发的条件了吗？"

Trae 给了几个选择：

* • 自动化实施
* • 手动按任务执行
* • MVP优先开发

至此，有没有感觉，它真的是个非常精通软件工程思想的智能体，每一步都在规范开发流程。

笔者选择了自动化实施。。。
