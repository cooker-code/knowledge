这一年里，AI 把做东西这部分卷得差不多了，尤其是 Vibe Coding 这种方式，基本上你会说话，就能生出一个能用的网页。

但问题马上就变了味儿：做得出来不等于有人用，更不等于能收钱。 真正难的部分变成了怎么让用户看到这个网页、怎么让它能付费、怎么少折腾一堆支付对接。

我最近在用的一条路是这样的：

网页用 Vibe Coding 开发 → 上架到一个叫 Mulerun（骡子快跑） 的平台 → 用它现成的流量入口和计费能力来变现。

这条路我不是纸上谈兵的。之前我在 Mulerun 上用 n8n 做过几款 AI Agent，平台会给积分，也能结算一点美金；在上个月花了三个小时做出来的 n8n 工作流变现了 50 美金。

最近它开始支持用 iframe 的方式嵌网页，我就把自己做的网页也丢上去了，这样我就不用自己去接支付、也不用自己拉一堆用户，平台那边有人看到就能直接用。

为了让你看得不迷糊，下面我先把这三个名字讲清楚，知道它们各干啥，后面再讲怎么从 0 到 1 搭、怎么对接、怎么上架。

### Mulerun 是什么？

先把它想象成一个别人家开的 AI 小产品集市。

你可以在上面发布自己的 AI Agent / 工作流 / 网页，让别的用户来用。它有两个关键价值：

1. 1.有入口：你不是孤零零一张网页在互联网上飘，它有一套自己的产品列表/发现页，相当于帮你做了最基础的曝光。
2. 2.有计费：它自己有积分/计费系统，你的东西挂上去之后，可以直接走它的支付，不用你再去对接 Stripe、微信、什么三方支付这一堆麻烦。

一开始它主要让大家发 n8n 工作流，最近开始支持嵌入网页，这就正好和 Vibe Coding 搭上了。

### n8n 是什么？

这是一个很火的可视化自动化工具。

你可以在一张画布上把“拿到用户输入 → 调大模型 → 存结果 → 发给用户”这条链子用拖拽拼出来，不一定要写代码。

它灵活、能接 AI、能做成一个功能，所以 Mulerun 一开始就支持它：大家把自己做的 n8n 工作流发上去，别人就能直接用，还能得到收益。

你之前如果用过 n8n 在 Mulerun 上发过产品，那现在这个发网页你会觉得特别顺，因为思路是一模一样的：做一个东西 → 发到平台 → 用平台的计费。

### iframe 又是什么？

别被名字吓到，它就是 HTML 里面的一个标签，作用只有一句话：在一个网页里再嵌一个网页。

Mulerun 现在支持你用 iframe 的方式把自己的网页塞进去，所以流程就变成了：

1. 1.你自己用 Vibe Coding 生了一个网页；
2. 2.把这个网页的地址告诉 Mulerun；
3. 3.Mulerun 在它自己的页面里嵌你这个网页；
4. 4.用户看到的是 Mulerun 的产品，但里面跑的是你的页面；
5. 5.收费还是走 Mulerun。

这就是它厉害的地方：你只负责把“能跑的网页”做出来，运营、入口、计费都让平台管。

### 这篇文章要干什么？

下面我会把这条路拆开讲成三段：

1. 1.用 AI / Vibe Coding 把网页生出来（这一段解决“我有个东西”）
2. 2.怎么用它的计费上架（这一段解决“别让用户白用”）
3. 3.怎么把这个网页对接到 Mulerun（这一段解决“平台能看到你的东西”）

# 整体结构

本次示例用到的工具：

* 开发环境 / Vibe Coding 编辑器：Qoder
* 前端部署：Cloudflare Pages
* 后端逻辑 / 鉴权 / 调用 Mulerun API：Cloudflare Workers
* 代码托管：Git + GitHub
* 大模型与计费平台：Mulerun

示例项目代码已开源：

https://github.com/Cing-self/mulerunproject.git

你可以直接 Fork / Clone 下来跟着操作。

# 网页开发

网页开发分为这么几步，找需求 -> 跑通 MVP -> 完善 -> 部署上线 -> 后续运维。

在开始之前，我们先把需要的环境搭建起来。

## 一、环境搭建

1. 1.注册 Cloudflare：https://www.Cloudflare.com/zh-cn/

也是可以使用 Github 或者 Google 账号进行登录。

1. 2.Qoder 下载地址：https://qoder.com/

下载完 Qoder 以后，用 GitHub 或者 google 账号进行登录，然后打开一个空白的文件夹。

1. 3.配置 MCP：

在 Qoder 页面中，点击右上角头像-> Qoder 设置->MCP 服务->。添加

将如下信息进行配置：

```
{  
  "mcpServers":{  
    "mulerun-docs":{  
      "url":"https://mulerun.com/docs/mcp"  
    },  
    "Cloudflare":{  
      "command":"npx",  
      "args":[  
        "@Cloudflare/mcp-server-Cloudflare@0.2.0",  
        "run"  
      ],  
      "env":{  
        "Cloudflare_API_TOKEN":"你的Cloudflare API令牌",  
        "Cloudflare_ACCOUNT_ID":"你的Cloudflare 账户ID"  
      }  
    }  
}  
}
```

MCP 配置中，涉及到 Cloudflare\_API\_TOKEN 和 CLOUDFLARE\_ACCOUNT\_ID

* Cloudflare\_ACCOUNT\_ID：Cloudflare 需要提供一个 accountId，如下图所示，刚才注册登录完 Cloudflare 后，会进入个人账号页面，从浏览器 url 上能直接获取到自己的 accountID。

* Cloudflare\_API\_TOKEN：在 Cloudflare 左侧菜单，选择管理账号->账户的 API 令牌，创建令牌，选择“编辑 Cloudflare Workers”模版。

最终需要有如下的权限：（Cloudflare Pages 的编辑权限需要手动添加）

点击下一步以后，最后就能拿到令牌。

1. 4.准备 mulerun 的 API Key 和 Agent Key

登陆 mulerun 平台（https://mulerun.com/），进入到 creator 后台（https://mulerun.com/creator，如果进不去，需要申请成为 creator）

在左侧菜单 API KEYS 中创建一个 KEY，即为 API KEY，该 KEY 用于调用 Mulerun 平台提供的各类大模型 API。

不同的 Agent 的，Agent key 也就不同，刚开始开发可以先新建一个 Agent，不提交。

1. 5.配置 Qoder rule

在 Qoder 页面，右上角头像-> Qoder 设置->规则 中，添加一个新的规则，（规则请从 GitHub 项目中去获取，在。qoder/rules/路径下）

类型改为始终生效：

在项目中放两个文件，文件名分别是 PROJECT\_INITIALIZATION\_GUIDE.md 和 DEVOPS\_GUIDE.md。（也都放在 GitHub 的 项目根目录下了）

这几个文件维护的是项目的初始化、开发规范以及一些通用的要求。有了它们，基本上 80%的开发工作都不用担心了。（目前还不是很完善，不能一次跑通，后续会继续改进）

## 二、跑通 MVP

点击 Qoder 左侧的 Quest 模式，在输入框中添加上下文文件 PROJECT\_INITIALIZATION\_GUIDE.md，以及 Rules。

输入自己的需求，比如，我想让他做一个根据提示词生成图盘的网页。

之后 Qoder 就会开始进行设计，如果对设计方案不满意，可以再让他修改，直到满意为止。这部分非常重要，这个关系到后续的开发质量，所以最好在这一环节就拉齐方案。

对方案没异议的话，就可以点击右上的开始任务按钮。

之后，他会根据我们刚才配置的 Qoder Rule，先进行项目的初始化，把该需要的工具、环境配置先安装好。

然后开始进行开发，AI 会先开发前端代码、然后再转向 worker 后端代码开发。开发完以后，进行本地开发环境测试，这过程中 AI 可能会询问你一些是否允许的命令请求操作，点击允许即可。这一步是 AI 会在你本地启动服务，这样你可以在你本地电脑环境进行测试。

等都执行完以后，AI 会做一个任务总结，本次都实现了什么。

在 worker 项目里，会有一个。dev.vars 的配置文件，我们将刚才从 mulerun 上获取到 AGENT\_KEY 和 API\_KEY，复制粘贴到这个文件里面进行替换，跟 Qoder 说你已经配置好 环境变量，帮我重启下服务，以及最底下有一个 CREATOR\_MULTIPLIER 配置，这个是用来定价的，比如说我打算定成本价的 5 倍，就配置 5.0。

这时候，我们只需要本地在浏览器打开 localhost:3000 这个地址，进行访问。那真实的场景中，一把过的几率比较少，大概率会出现报错。

比如我这次，就遇到了页面空白，我只需要右键谷歌浏览器页面，点击检查。底部就会弹出一个控制台，选择 console 页签，页面空白大概率是前端报错了。

果然，我们可以看到在 console 中有一个报红的记录，将报错的记录复制，然后发给 qoder，qoder 会一步一步帮你排查解决。

这过程中可能会遇到不同的错误，但给 AI 错误信息无外乎这么几个地方：

1. 1.页面直接给出错误信息。
2. 2.检查->console 里面有错误信息。
3. 3.检查-> network 请求报错了。

我大概修复了 4 次，整个页面终于能正常启动，效果大概是这样的：

一开始，我们点击生成图像时，正常来说是会报错的，比如我这提示的是签名验证失败。因为我们这个项目最终是要嵌入的 mulerun 上的，所以需要对所有的请求接口进行鉴权，正常在本地是用不了的，需要先发布到 mulerun 上。（如果需要在本地可以调试，跟 AI 说，我希望在本地测试的时候先不进行签名验证）

整个项目需要自己跑通，目前还做不到一次性过，或者直接根据 GitHub 提供的 demo 项目进行修改。

## 三、部署上线

现在，我们将项目进行线上部署，前端项目会部署到 Cloudflare 的 Pages 中，后端会部署到 Cloudflare 的 worker 中。

我们只需要跟 qoder 说部署到线上，它就会自动的帮你进行上线部署，这其中可能会打开 Cloudflare 让你进行账户的登录。

部署完以后，我们打开 Cloudflare，计算和 AI-> workers 和 Pages 页面下，能看到刚刚 启动的两个服务。我们点击 进去第二个，提供域名的。

在这个页面中，可以看到线上的网页地址，该域名可以直接打开访问。

我们进入线上环境，进行访问，测试生成图像功能，目前还是有报错 405。这个问题是因为后端域名配置错了，那只需要将这个报错发送给 Qoder，它会进行处理。直到出现在生成图像时，接口返回签名验证失败。

## 四、上传到 Mulerun

1. 1.创建一个新的 Agent，选择 IFRAME 的导入方式，将项目的 URL 地址分别填入 START URL 和 SHARE URL。

* START URL 是在 Mulerun 平台点击 Run 时，会嵌入的 URL 地址，也是用户体验的起始页面。
* SHARE URL 是会话结束后将用户跳转到的 URL，以及在其他人查看该会话回放时，他们看到的也是这个 URL。
* Max Age：会话存活的最长时间，最大是 120 分钟，可以根据自己的任务时长和期望用户停留的最长时间来设置。

1. 2.填入相关的信息，这里 showcase 图至少需要上传两种。

1. 3.个人密钥信息这里不需要配置 ，直接点击下一步。

1. 4.在 Submit and Monetize 环节，也不需要进行计费设置，在正式提交前，先点击 TEST FIRST，先测试一下。

比如我这生成完图像以后，结果没有显示出来，而且也没有下载按钮。

这时候我就让 AI 帮我处理一下，它也很快帮我定位到了问题。

修复完以后，目前能够正常出图，右上角也能够看到我们本次消耗的 credits。

## 五、后续运维

接入完 iframe 只是开始，真正决定你能不能长期赚到钱的是：

1. 1.你是否能及时发现错误

* 在 Worker 的 Observability 里打开日志；
* 出错时，用日志 + Qoder 帮你定位问题；

1. 2.你是否会跟进用户体验

* 有报错不仅要修，还要想办法复现、自己测一遍；
* 看看是不是需要加点提示文案、loading 状态、失败重试等。

为了方便持续集成，你可以在 Qoder 的服务集成里接上 GitHub：

* 让 Qoder 把项目纳入 git 管理；
* 每次改完，让它帮你推送到远程仓库；
* 再通过 Cloudflare 部署，形成一条完整的迭代链路。

# 心得分享

## iFrame 方式的避坑

1. 1.API\_KEY 和 Agent\_KEY 泄漏：如果你是从前端直接调用 Mulerun 平台的 API 接口，那你肯定需要把 API\_KEY 和 Agent\_KEY 放到请求中，这样你的请求中就暴露了你的密钥，会被抓取。这也是为什么我加了一层 worker 的重要原因。
2. 2.鉴权机制：前端只是一个展示的页面，所有关于模型调用、计费的逻辑都放到了后端 worker 上，那你需要想清楚什么样的请求能被正常处理。本篇文章中演示的是，只有 mulerun 嵌入的方式才能正常调用 worker 的 API 接口，否则会提示验证签名失败。
3. 3.计费说明：计费是最重要的一环，本篇文章是按照倍数进行定价，可在环境变量中自行配置。你也可以根据你自己的需求进行定价，这个只需要跟 AI 说明你想要的计费方式是怎样的

如果你已经有一个能跑的网页，那下一步就别再停在本地预览了：
