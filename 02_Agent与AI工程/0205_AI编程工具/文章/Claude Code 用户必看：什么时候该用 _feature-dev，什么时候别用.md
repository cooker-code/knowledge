如果你用 Claude Code 做过不止小修小改，大概已经吃过"上下文漂移"的苦：你描述想要什么，Claude 写代码，然后接下来一小时都在修补那些偏离实际需求的改动。

`/feature-dev` 是 Anthropic 官方做的 Claude Code 插件，直接治这个问题。它给你一个**结构化的 7 阶段功能开发工作流**——带着 Claude 理解你的代码库、澄清模糊需求、设计架构方案，并且在真正写代码前后都有质量把关。

下面说清楚它是什么、什么时候用、7 个阶段具体怎么跑，以及怎么融入真实开发节奏。

---

### 什么是 `/feature-dev`

`/feature-dev` 是 Anthropic 官方维护的插件，装在 Claude Code 插件市场里。装完之后，你的命令列表里多了一条 `/feature-dev`，背后还有三个专门的 Agent 待命。

调用方式很直接，带描述：

```
/feature-dev Add user authentication with OAuth
```

或者裸跑 `/feature-dev`，插件会交互式引导你。它不会直接跳去写代码，而是跑完一套 7 阶段流程：**发现需求、探索代码库、澄清问题、设计架构、实现、质量审查、最后总结**——中间多个节点会停下来等你确认。

---

### 跟普通提示的区别在哪？

Claude Code 的普通提示是开放式的。你说要什么，Claude 自己理解，输出质量全看你的措辞、上下文窗口里堆了什么、以及对话里攒了多少"噪音"。短平快的单文件任务没问题，但一旦涉及多文件联动、架构决策、分步实现，很快就崩。

`/feature-dev` 用结构化流程替代这种"想到哪写到哪"。它在特定阶段派出三个专职 Agent——`code-explorer`、`code-architect`、`code-reviewer`——各自有明确分工。在你看到方案并点头之前，一行代码都不会写。

区别不只是"更有条理"。结构本身改变了 Claude 能产出什么，因为每个阶段都建立在上一阶段**验证过的理解**之上，而不是 Claude 在真空里做假设。

---

### 为什么偏偏是"中等复杂度"任务

这个插件对任务大小有讲究。太简单，7 阶段的开销不值；太复杂，结构化工作流也装不下系统级变更所需的上下文。

舒适区是那些**需要规划、涉及多文件、但有明确边界**的任务：加 API 端点、写 UI 组件、接第三方服务、扩展现有功能——你能预先定义验收标准，而且不用审计整个系统就能验证完成度。

---

## Agent 开发流程的演进

早期工具补全单行代码。聊天助手扩展到多文件建议和设计方案。现在的结构化 Agent 模式更进一步：端到端规划并执行定义好的功能。

---

### 从 Copilot 到自主功能 Agent

GitHub Copilot 这类早期工具在**行或函数级别**干活。聊天助手进化到多文件建议，但输出质量还是取决于你的提示质量。现在到了**结构化 Agent 工作流**阶段，`/feature-dev` 这种插件在定义好的流程里编排多个专职 Agent，你的角色从"写提示的人"变成"审方案、拍板的人"。

这里有个反直觉的教训：**给 AI 更多自主性，如果没有结构约束，产出质量反而下降**——这正是 7 阶段设计要解决的问题。

---

### 为什么"限定范围的自主"反而更好

LLM 在明确框架里推理表现最好。范围清晰意味着决策空间小，模型的脑力用在刀刃上。开放式提示逼着模型隐式解决各种歧义，而这些决议往往跟你的真实意图对不上。

结构化工作流提升一致性，原因跟"定义良好的函数比万能函数更少 bug"一样：边界砍掉了不必要的自由度，让 Claude 专注在你需要的地方。

---

## 7 阶段详解：实际怎么跑

这是 `/feature-dev` 跟普通斜杠命令的本质区别。每个阶段有明确目标，多个节点会暂停等你输入。

**Phase 1. 发现（Discovery）**

Claude 先澄清功能请求：要解决什么问题、有什么约束、需求长什么样。如果你最初的描述模糊，歧义在这里就被掐掉，不会带到后面。

**Phase 2. 代码库探索（Codebase Exploration）**

插件并行启动 2-3 个 `code-explorer` Agent，各自负责代码库的不同侧面——类似现有功能在哪、架构模式是什么、相关抽象有哪些。Agent 返回文件引用和实现痕迹，Claude 读完识别出的文件，在提任何建议之前先建立对代码的实际理解。

**Phase 3. 澄清问题（Clarifying Questions）**

看完代码库后，Claude 标出还没明确的东西：边界情况、错误处理、集成点、向后兼容顾虑、性能需求。它把这些问题列成清单等你回答。官方文档强调：**设计开始之前，不该有任何歧义**。

**Phase 4. 架构设计（Architecture Design）**

插件并行启动 2-3 个 `code-architect` Agent，各自设计不同实现方案。通常产出三类：最小改动方案（最大复用现有代码）、干净架构方案（优先可维护性和分离度）、以及两者之间的务实平衡。Claude 审完所有方案，基于你的实际代码库形成建议，讲清楚权衡，然后等你选。

**Phase 5. 实现（Implementation）**

只有你明确批准了选的架构，实现才开始。Claude 读遍前面阶段识别的所有相关文件，按批准的方法写功能，遵循探索期间发现的代码约定，全程跟踪进度。

**Phase 6. 质量审查（Quality Review）**

三个 `code-reviewer` Agent 并行跑，各有侧重：一个查简洁性、DRY 原则、优雅度；一个查 bug 和功能正确性；一个查项目约定和抽象是否一致。关键是**置信度过滤**，只报置信度 80% 以上的问题，大幅减少误报。Claude 呈现发现，问你怎么办：现在修、记下来稍后修、还是就这样继续。

**Phase 7. 总结（Summary）**

Claude 记录：做了什么、工作流期间的关键决策、改了哪些文件、建议的后续步骤。

---

**人机检查点是核心**

嵌入流程的这些检查点“发现后、澄清后、架构选择后、质量审查”，是跟"让 Agent 完全自己跑"的本质区别。真正重要的事项上，你始终在决策环里。

---

## 什么时候用 /feature-dev

调用之前先问：任务有没有清晰边界、能不能验证完成？满足的话，大概率适合。

---

### 加新接口或 API 逻辑

加 REST 接口、GraphQL resolver、认证流程都是经典场景。范围明确：路由处理器、请求校验、业务逻辑、响应格式。Phase 3 会问错误处理、认证要求、集成点——正是那种不问清楚就会踩坑的问题。

```
/feature-dev Add OAuth authentication supporting Google and GitHub
```

发现和澄清阶段会消化歧义，你不需要在第一条命令里塞满所有细节。

---

### 写 UI 组件或功能模块

前端组件天然有边界：定义的 props、state、行为，验收标准可见可测。全栈模块如通知系统、文件上传流程、分页数据表也符合同样模式。

Phase 2 的 `code-explorer` 会识别代码库里现有的 UI 模式，Phase 4 的 `code-architect` 会设计跟这些模式对齐的方案。

---

### 快速原型新功能

早期实验和 MVP 脚手架阶段，`/feature-dev` 让你拿结构化输出，不用逐文件微管理。Phase 3 的澄清问题在这里特别有用，它们会浮出你还没完全想清的需求，正好在假设固化进代码之前解决。

---

### 理解并扩展现有代码库

这是**最被低估的用法**。Phase 2 的并行 `code-explorer` Agent 即使你打算自己实现，也能给你一份结构化地图：入口在哪、执行路径怎么走、数据流什么样、关键文件是哪些。新接项目或者重温几个月没碰的代码时，价值很高。

---

## 什么时候**别**用

不是所有开发任务都适合功能级 Agent 工作流，超出范围使用往往拖慢你。

---

### 琐碎变更或微小编辑

修错别字、改配置值、重命名变量，不值得走 7 阶段。为这些小事跑 `/feature-dev` 浪费上下文窗口和时间，毫无收益。直接提示解决。

---

### 大规模架构重构

重构整个数据层、迁移框架、重新设计核心系统边界需要不同方法。这些任务超出插件能装的上下文，涉及的架构判断需要你亲自来。拆成边界明确的功能任务，每个有界部分分别用 `/feature-dev`，别整体一次性处理。

## 结语

到这里，我们已经把 /feature-dev 的底层逻辑拆解清楚了：它本质上不是在教 Claude 写更好的代码，而是通过 7 阶段的流程约束和三个专职 Agent 的协作，把“盲目输出”变成了“按图施工”。选对战场，那些有清晰边界、涉及多文件的中等复杂度任务，是你不踩坑的前提。

但理论讲得再透，终究要上手敲击才算数。这个官方插件到底怎么装？装完怎么在真实的终端环境里第一次跑起来？7 个阶段的交互长什么样，又有哪些能让你少走弯路的配置技巧？
