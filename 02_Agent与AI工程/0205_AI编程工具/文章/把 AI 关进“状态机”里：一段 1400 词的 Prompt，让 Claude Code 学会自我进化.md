# 把 AI 关进“状态机”里：一段 1400 词的 Prompt，让 Claude Code 学会自我进化

在使用 CLI 级别的 AI 辅助编程工具（如 Claude Code 或 Cursor）时，大部分资深开发者最后都会感到一种深度的疲惫感：**AI 的能力虽然惊艳，但它没有“状态记忆（State Memory）”。**

你费尽心思教它一套你们公司内部的 React 组件规范，或者纠正了它在写某个 SQL 上的特定坏习惯，可一旦你关闭当前的 Session 会话，这台价值数千行代码算力的引擎，就会在下一次启动时立刻“格式化大脑”，变回那个什么都不知道的白板。

这就是当前 AI 生产流中最大的痛点：我们每天都在重复地训话、发包、纠错。为了填补这个黑洞，很多人开始疯狂地写超长篇幅的 `rules.md`，或者强行去给 AI 开发各种复杂的拦截脚本与 Hooks。但这往往适得其反，极度庞杂的硬编码规则很快就塞爆了上下文窗口。

今天，我们要拆解由加密圈极客 ChristopherA 在 GitHub Gist 上开源的一份顶级架构逻辑：**Self-Improving Claude Code**。它不是什么外挂长篇大论，而仅仅是一段长约 1400 个 tokens 的**种子系统提示词（Bootstrap Seed Prompt）**。把这段 Prompt 喂进当前目录，你的 AI 就会被强行植入一套“带防呆网关的文件状态机架构”，从而开启自我进化与永久记忆的链路。

## 1. 痛点降维：从“阅后即焚”到“文件态长效记忆” (State Engine Architecture)

我们先来直视一下传统 AI 辅助编程场景下的高频崩溃瞬间。

**【Before：引入种子 Prompt 前的灾难体感】**  
当你主导跨度长达数周的重型代码库重构时，AI 总是处于“迷失坐标”的状态。昨天你分发给它的任务是“搞定路由系统的第一阶段拦截”。今天你再度打开会话对它说“继续推进昨天晚上的工作”，它会立刻开始胡言乱语，甚至因为忘记了昨晚做出的某个重要接口设计，直接把昨天刚定版的代码体系给你推翻。  
为了不让它跑偏，你只能被迫化身“人肉备忘录”，每天花大量时间把当前的进度、遗留的 Bug 和下阶段的诉求手打在输入框里喂给它。

**【After：植入 Seed Prompt 后的生产力跃升】**  
ChristopherA 做出的第一个也是最震撼的系统架构级变更是：**通过 Prompt 极其强硬地勒令大模型把自己的“脑容量”强制挂载到本地文件上（File-based State）**。

在植入该指令后，Claude Code 会在项目的 `.claude/` 目录下长出一个自我管理的基建空间。Prompt 里下达了一道不容商量的铁律：

> *"Sessions end. Memory doesn't persist. Files do. (会话终将结束，记忆不会永存，但文件可以。)"*

从此以后，AI 在进行任何复杂推进之前，都必须强制走以下流水线：

2. **闭环落盘机制**：在你的终端会话即将自然销毁（或者遇到非平凡量级的工作节点结束）时，AI 会被触发执行一套反思动作（Reflect & Triage），它会将当前项目的 **状态（Status）、已完成项（Done）、下一步具体动作（Next）** 以及**遗留风险项（Open questions）**，老老实实地写入一个本地的状态文件里。
4. **读脑重建机制**：当你明天开启一个新的会话，不管你发什么指令，AI 做的第一件事，是静默拉取并读取那个状态文件（State Engine Synchronization）来重构昨日的上下文环境。

这就等于把原本“阅后即焚”的离散大模型计算调用，强行改造成了一台带有长时持久层（Persistence Layer）的状态自动机。

## 2. 极致克制网关：反盲目增生与收敛原则 (Anti-proliferation Pattern)

如果我们纵容 AI 拥有自己的本地存取权限去记录它的“心得体会”，一场可预见的灾难就会随之而来：它会因为极度亢奋而在目录底下疯狂拉屎——建立成百上千个毫无意义的 `learnings_test_1.md` 或者 `temp_rules.txt`，最终因为海量的垃圾文本导致后续上下文读取耗时极速升温。

高级架构师和民间爱好者的本质分水岭，在于前者懂得设计**防呆抑制器 (Fool-proofing Constraints)**。

在这个 Seed Prompt 中，存在几条极其刚性的代码纪律边界：

* **「Context Discipline (上下文纪律约束)」**：明确将窗口资源定义为极其珍贵的「Public Good」。规定核心控制配置不得超过 100 行代码，所有累积在内存里的通用行为指令必须被压缩至 200 行以内。
* **「Triage 强制分发网关」**：绝对禁止 AI 无脑输出简单的流水账日记或 List 列表。每一次系统被呼唤着去“执行经验采集（Learning loop）”时，它发现的每一个模式漏洞都必须过一道严苛的 Triage 筛子卡口：
* **Apply now（即刻生效实施）**：马上介入修改对应的文件逻辑。
* **Capture（挂载截获）**：打上时间戳并带入业务底座上下文，规整存入专属记忆库 `learnings.md`。
* **Dismiss（全盘驳回废弃）**：必须向用户做出驳回辩护说明理由，绝不支持无效信息遗留过夜。
* **「禁止擅自拉屎建文件（Anti-proliferation）」**：Prompt 从底层按下了默认刹车：“**任何新文件的诞生都必须有极其强悍的业务辩护理由（Needs justification）**”。系统的首选动作永远是复用和编辑现有基建实体。如果必须要进行结构性的裂变或者创建新的配置文件，模型必须调用 `AskUserQuestion` 强行阻断进程挂起，申请人类架构师（User）做确认放行操作！这就彻底杜绝了模型在黑盒后台放飞自我搞基建的惨案。

## 3. 结构涌现：抛弃臃肿预设，让系统被业务逼着进化 (Structural Emergence)

这份开源精华的第三层哲学，是对当前业界一股歪风的重构降维。目前网上流传着许多试图通过预建几百个文件夹、预埋几十个 Hook 脚手架去规范 LLM 行为的做法。但这对于中小团队纯属折腾。

ChristopherA 的核心主张是：**真正的企业级架构设计体系绝不会试图去猜想用户到底会在哪一个细分模块碰到天花板，它只会给出那颗能够承压生长的「底层循环种子内核（Recursive improvement kernel）」**。

当你刚刚把种子部署下去的时候，你的 `.claude/` 空间里可能是空空荡荡甚至仅有一点薄弱的几点 `learnings.md` 记录。但随着你天天按着大模型在业务战壕里摸爬滚打，架构体系会自己长出来：

* 当 `learnings.md` 里关于同一类 Bug 或代码洁癖的报错达到了两次以上（阈值触发），它会自动提取出高优先级的 `.claude/rules/` （硬派纪律引擎）。
* 等到了第十五二十天，这个 Rules 也快堆积到 50 行即将突破限制卡槽顶端时，它将引发**结构体系崩塌涌现操作（Structural Emergence）**：它会把这条大红线规矩再精准拆分剖离。

整个演进链路上不需要人类架构师天天写脚本维护它的行为体系，它仅仅是因为**业务底座压力阈值满了，自己不得不倒逼着向更解耦的方向分裂。**这种不预设立场、让繁复层级由真实疼痛生长而出的设计美感，才是这个 Prompt 最值钱的压箱底部分。

## 架构师定调 (Architect's Verdict)

所谓好的提示词工程（Prompt Engineering），早就应该脱离那些类似于“你是一个 20 年经验的全栈大佬深呼吸一口气请给我发一段贪吃蛇代码”这种中二且幼稚的阶段了。

**在这个大模型上下文无限膨胀的节点里，算力并不是无休止去乱读和猜测用户深层次诉求的廉价消耗品。**

如何将高可用后台架构里的**状态机存流（State Engine Synchronization）**、**高水位防爆卡口（Quality Release Gate）**以及**微服务裂变思维（Structural Emergence Evolution）**用自然语义重新写进大模型的潜意识？这份长达 1400 个字符的纯净文本《Self-Improving Claude Code》提供了一个足以载入史册的最佳实践框架。

如果你现在还在为昨天告诉了 AI 怎么写某个 API 接口，今天它依然在那儿犯同样的白痴错误而疯狂摔鼠标的话，花三分钟把这个种子克隆注入你的工作流里。别再用给婴儿喂饭的手段去对待你的 AI 本地部署助手了，**给它上一套带着防呆报警模块和长效记忆的锁链，让它学会这最重要的一手：永远不要在同一个地方挨第二次骂。**
