## 概况

Claude Code （下称CC）是 Anthropic 提供的强大命令行 AI Agent 工具，与传统的 Chatbot 不同，CC 在 Terminal（linux bash、CMD、Powershell）中运行，Agent可以探索整个代码库，用户可以授予 Agent 相关权限，以修改文件、代码，运行命令，自动进行调试与 debug。

> Tip
>
> 文中会使用形似该处的“Tip”栏目表示最佳实践，虽然原非“最佳”的实践，但也能避过一些雷点。

### 核心优势

* • **项目化上下文**：探索整个项目目录，而非仅当前打开的代码文件，获取全面的上下文信息
* • **系统级权限**：可以直接操作文件系统，进行文件管理、自动化编写代码并测试
* • **多 Agent 架构**：使用主 Agent 领导多个Subagent 分配任务，处理复杂项目更高效
* • **IDE 集成**：不需要更换 IDE，可与 VSCode 等Editor直接集成

### 同类工具

除了 CC，市面上还有其他的 AI 编程工具：

* • Google的Gemini CLI、OpenAI的Codex、阿里的Qwen Code、腾讯的CodeBuddy等命令行工具
* • Cursor、Trae IDE、CodeBuddy IDE、Augment等AI IDE
* • Github Copilot、Roo Code、Augment等VSCode插件

> Tip
>
> 为什么我们仍然建议使用CC？
>
> 1. 1. 命令行工具相对于IDE与插件具有环境兼容性与性能优势，可以跨Windows与Linux平台使用。
> 2. 2. 命令行工具的性能是最接近人类直接使用命令行的，人类所能利用命令行达成的操作工具均可更快更好地做到
> 3. 3. 命令行工具相对开放，CC可以通过npm下载，并且可以接入其他模型接口
> 4. 4. 命令行工具的扩展性与定制性极高，许多开发者开发了相关插件
> 5. 5. CC的工程架构具有先发优势，领先于同行的其他命令行工具，自动化水平与细节打磨也更加卓越。采取闭源策略，更新频繁，社区活跃。

## 安装与开始使用

安装CC之前，建议准备网络环境来让命令行使用代理，需要魔法软件能够连接美国节点，并能修改系统代理，最好具备虚拟网卡（TUN）模式。可以找到魔法软件的端口设置，然后修改系统代理的环境变量

```
http_proxy=127.0.0.1:xxxx  
https_proxy=127.0.0.1:xxxx

curl -vvk google.com

### Claude Code 安装与使用配置

#### 安装

在Windows终端与Linux bash中输入以下命令，即可安装

```
npm install -g @anthropic-ai/claude-code  
claude --version
```

然后直接输入`claude`使用

#### API 配置

此时的CC是无法使用的，需要登录，如有Claude会员，可以直接登录，然而我们建议使用其他模型替代。

Tip

我们出于以下几条标准以选择模型：

1. 1. 由于Anthropic的极端反华立场与严厉的区域政策，与中国有关联的账号有非常高的可能性被封禁，例如使用了不纯净IP；另外，Claude系列模型价格极高，订阅费用也较贵，所以难以把官方模型作为主力模型使用。
2. 2. 一切非官方途径，如中转站、会员拼车、免费公益站，存在经济风险与数据安全风险，并且无法保证提供模型的型号与质量，可能涉及灰产、自动路由降级，不建议作为主力模型来源，有特别需要时少量使用。
3. 3. 使用国外模型在不能够使用代理的环境（如服务器）无法使用，且存在数据安全风险。
4. 4. Agent的Token消耗量极大，约比普通问答至少高出2-3个数量级，即使使用Deepseek、智谱这样业内最低价的模型，一小时的开发也需要几元到十几元的成本，Kimi、Qwen的价格是他们的数倍。尽管带来了可观的收益，但仍旧不建议使用API按量计费，而是选择提供订阅服务的厂商。

在综合考虑性价比、智能水平（参考Benchmark结果、开发者社区风评）后，可采用以下方案：

1. 1. 智谱GLM4.6：唯一订阅制国模，物美价廉，选择合适套餐用量Nearly Unlimited，Benchmark结果超越Claude Sonnet 4，社区风评极佳，绝对主力模型！
2. 2. Deepseek V3.2 exp：极其低廉的API价格，高智能水平，作为备用模型。
3. 3. Claude Sonnet 4.5：碾压性的编程性能，当前最佳开发模型，可以通过官方非官方渠道少量使用，备用模型。

在一个模型反复无法解决问题时，不妨更换一个模型尝试。

> Caution
>
> 智谱的订阅在CC中需要使用Anthropic接口，在其他Agent中需要使用特殊接口，否则将会计费或者消耗资源包，具体请查阅智谱官方技术文档。

有两种方法可以设置模型，第一种是在系统环境变量中设置（以下为Windows）

```
setx ANTHROPIC_BASE_URL "https://open.bigmodel.cn/api/anthropic"  
setx ANTHROPIC_AUTH_TOKEN "your-api-key"  
setx ANTHROPIC_MODEL "glm-4.6"  
setx ANTHROPIC_DEFAULT_HAIKU_MODEL "glm-4.6"
```

也可以在用户文件夹的.claude/settings.json设置（注意最后一行没有逗号）

```
{  
    "ANTHROPIC_BASE_URL": "https://open.bigmodel.cn/api/anthropic",  
    "ANTHROPIC_AUTH_TOKEN": "your-api-key",  
    "ANTHROPIC_MODEL": "glm-4.6",  
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "glm-4.6"  
}
```

## CC的设计哲学与关键组件

### 上下文工程

上下文（Context）是AI Agent工作的基础与输入，与过去提示词（Prompt）概念不同的是，提示词的宗旨在于在每次交互中利用利用自然语言的约束与提示来尽可能提高AI的表现，而上下文工程的核心观念是为AI提供充足的信息，包括你的项目架构，内部或外部的技术文档，规定的工作流程与技术规格。上下文的长度要比提示词长得多，由许许多多不同的部分组成，并且也不是一次性使用，而是在一个项目里持续不断地为AI提供信息，提高AI输出内容的稳定性与对用户需求的吻合度。

所谓上下文工程（Context Engineering）与提示词工程（Prompt Engineering）的最终目的就是**提高大语言模型这个概率模型的输出的确定性**。因此，优秀的，充足的上下文能够尽可能多的提供AI Agent工作所需要的信息，要求AI按照既定的规则工作而不输出用户所不希望看到的结果，例如引入新的bug、过度开发、破坏现有架构等。网友曾总结「CC八荣八耻」：

> 以瞎猜接口为耻，以认真查询为荣。  
> 以模糊执行为耻，以寻求确认为荣。  
> 以臆想业务为耻，以人类确认为荣。  
> 以创造接口为耻，以复用现有为荣。  
> 以跳过验证为耻，以主动测试为荣。  
> 以破坏架构为耻，以遵循规范为荣。  
> 以假装理解为耻，以诚实无知为荣。  
> 以盲目修改为耻，以谨慎重构为荣。

而现有的AI工具上下文容量并不足够长：国产模型普遍只有128K-256K的上下文容量，即使是一些标称容量1M甚至2M的模型，在上下文过长（超过200-300K）时也会注意力分散，幻觉率大大提高，性能下降。因此，确保上下文的长度与有效性是控制AI行为稳定性最有效的手段。这并不意味着总是要节省上下文空间的使用，恰恰相反，应该在上下文容量范围内塞入更多的规则，来更好的控制AI Agent工作，余下的上下文则作为AI聊天与工作时的“缓存”。

CC的架构做出了许多努力，以消除多余的上下文的影响，确保模型运行时不受无效信息干扰，这种行为往往被称为“上下文隔离”。

#### 上下文压缩

CC在聊天上下文接近模型的上下文容量上限的时候会进行上下文压缩（阈值为92%），这将会把现有的聊天与工作成果汇总成Summary然后继续接下来的聊天，以此来实现连续不断地聊天与工作。同时，你也可以使用`/compact`命令手动进行上下文压缩，这可以成为一部分工作的结尾和另一部分工作的开始。另外，如果上下文已经无用，使用`/clear`来进行全新的聊天，也可以通过`/resume`命令回到这个项目文件夹所进行过的任意一个聊天。

> Tip
>
> 智谱的模型后端会自动将上下文压缩，所以可以将CC客户端的上下文压缩功能关闭，以防止运行时突然的卡顿与丢失上下文，这偶尔也会导致GLM模型显得有些“健忘”。

#### CLAUDE.md

CLAUDE.md是CC在项目文件夹工作时所默认读入的上下文文件，并且，在用户文件夹的`.claude/`目录中也存在一个“系统级”的提示词，你可以将你对CC的所有具有领导性的要求写在里面。而在项目内，根目录的CLAUDE.md提供本项目的信息、要求。

CC的高明之处在于，CLAUDE.md在项目目录中呈树状存在，子目录的CLAUDE.md在CC在上级目录工作时不会被读取，而当CC进入这个子目录时，会自动读取子目录下的CLAUDE.md来获得该目录的信息，确保AI专注于本目录的工作而不被项目的其他模块所干扰。使用`/context`来审阅目前的各类上下文占用了多少模型的上下文容量，使用`/memory`来管理这些“记忆”（也就是CLAUDE.md）。

> Tip
>
> CLAUDE.md中也可以通过“@”来通过绝对路径或相对路径引用其他文件，CC会自动在memory中读入他们，在斜杠命令中也可以这么做。当前最多支持5级嵌套，使用引用功能可以在不同地方复用规则，降低维护成本。

#### Subagent（子代理）

子代理实现了项目中不同业务的划分，由于上下文长度与模型注意力的有限，我们不可能将所有的要求都写入主代理的CLAUDE.md中，这只会导致主代理注意力混乱，指令遵循能力下降。因此，使用子代理功能可以将不同的专业的、定制化的需求与角色分配给不同的子代理，使用`/agents`命令来开始自定义自己的子代理，子代理的提示文件会保存在项目的`agents/`目录或是用户目录的`.claude/agents/`目录下。

当有需求时，主代理会自动调用子代理，然后将所需的上下文传送给子代理进行工作，你不能直接与子代理进行沟通，而由主代理作为桥梁。子代理之间的上下文完全隔离，确保每一个子代理的专业性和指令遵循，目前而言，子代理并不能直接调用子代理，这可能会导致循环调用的问题。

当然，有时上下文也不需要隔离，一个项目应该有一套各个子代理与开发者共同遵守的规定，系统级或项目提示词可以起到这个作用，并将自动继承给子代理，明确要求子代理遵守以达到这一目的。

> Tip
>
> 维护一整个子代理团队是个累人的事情，在没有特别的需求时，建议在系统级提示词中阐明子代理的作用与项目的技术规格，并让CC自动生成子代理。

> Tip
>
> 合理配置子代理可使用的工具以限制子代理的能力，在配置的工具过多时，若没有明确要求，Agent调用工具的能力会大大下降。

#### `.claudeignore`

偶尔，CC也会读入太多的无用信息，在`.claudeignore`中编写忽略清单，比如密码、数据库或是隐私信息，来防止CC读取，这也有助于控制上下文来提高CC性能表现。

### 用工具代替模型智能

同样，为了提高AI工作的准确性与稳定性，在进行任何操作时，使用具有确定结果的工具就要好于基于概率的语言模型，Anthropic为CC配备了基础的系统工具，同时，用户也可以定制自己的工具来完成工作。

#### 系统工具

CC可以使用读写工具，搜索工具，以及使用命令行的任何命令来完成文件系统的交互，这与一般人类无二。

#### MCP

MCP（Model Context Protocol）是我认为Anthropic最伟大的发明之一，这是一种工具调用协议，能够让其他程序、服务器用固定格式与模型交互，例如，可以调用搜索MCP、Web fetch工具为CC提供网络信息，用RAG（Retrieval-Augmented Generation）工具帮助CC在知识库中检索信息，用Word编辑MCP为CC提供与Word文件交互的能力。

MCP提供了一种可能，让大语言模型超出了原有的聊天、或是只在命令行内工作、编写代码、文本的功能，而能够通过一个通用接口和任何一个应用进行交互，传递信息。使用`/mcp`命令管理已有的MCP服务器，网上的MCP服务器通常提供了安装的指引，项目与系统的MCP设置会储存在`~/.claude.json`或项目的`.mcp.json`里。

> Tip
>
> 智谱提供了官方的网络搜索MCP与视觉理解MCP（GLM4.6已经支持原生多模态），在使用一些访问外网的MCP时，请保持代理环境。

#### 自定义工具

你也可以自己编写工具以供CC调用，只需要符合协议即可，这一点请于CC文档中查阅。

### 工作流程自动化

经过配置之后，使用CC等代理工具的目的，就是为了减少人类在工作中的参与程度，如果只需要人类在AI工作完成后提供审阅与检查，就可以完成相同的工作，那么一个优秀的开发者就可以独立完成过去一个团队才能完成的工作。为了实现流程的方便与完整，CC引入了以下组件。

#### 自定义斜杠命令

这一节或许应该写在上下文一节中，然而自定义斜杠命令与代理并不相同，代理提供了一个角色以供主代理调用，但自定义斜杠命令本质上只是起到了节约时间的作用——你可以将提示词记在剪贴板中，这某种意义上可以与自定义斜杠命令起到相同作用。

自定义斜杠命令的主要意义是记录一个指令，或者说固定的工作流，保存在项目或者用户文件夹的`commands/`目录下，你可以在聊天时方便的输入`/`来检索并调用，例如，可以规定一个`/review-code`的提示词，内含你需要的审阅代码的流程与要求。斜杠命令的提示词中可以使用变量占位符`$ARGUMENTS`，让斜杠命令有更强的功能性与兼容性。

> Tip
>
> 精密的流程控制是提高AI性能与自动化水平的有效手段，一个详细的，具有清晰步骤与规则的工作流说明可以让AI稳定地工作很长时间。同时，事无巨细地规定AI的工作范围，也可以让AI减少自我发挥的空间，AI总是过于自信、过于高估其水平而缺乏反思。

> Tip
>
> 网络上有许多的现成开发框架，可以提供非常完备的、先进的系统化AI协作开发策略，这在业内被称为“敏捷开发”，相关的项目有BMAD、Github Spec Kit等，内含若干提示词与命令。

#### 计划模式与思考模式

在AI开始干活前，你最好先计划一下，准确的说是规划一下行动的步骤，CC单击两次`shift+tab`进入计划模式（一次是auto accept edits），在使用原生模型的时候，计划模式会使用opus模型，但我们外接国模之后并不会使用更强的模型。计划模式会限制代理进行任何的工具使用，包括读写操作，不妨让代理先仔细研究你的需求再开始行动，代理会制定一个TODO list来跟踪每个任务。使用`/todos`来查看现有的TODO list。

使用tab键来开关思考模式，也可以通过显式输入“ultrathink”来激活最大程度的思考思考，并附带彩虹特效（显式输入思考是早期版本留下的彩蛋），开启思考模式理应只在混合思考模型生效，在不同模型表现不同，有时也可以搭配思考型MCP使用。

#### 权限许可

考虑到代理的威力过大，很有可能对现有架构进行一些破坏性修改或删除重要文件，CC的所有工具调用与命令在执行前都会请求用户许可，你可以单次许可，或者在这个项目里始终许可，这会被记录在项目目录的`.claude/settings.local.json`文件中。仔细评估流程自动化程度与安全性的平衡，为命令授予许可。使用`/permissions(allowed-tools)`命令管理许可。有时，可以始终许可一些高频的工具（例如fetch），防止代理的自动工作总是被请求许可打断。

#### Hooks（钩子）

钩子是一个相当高阶的工具，可以在CC在进行某一个操作之前或之后固定触发，具体来说，有

* • **PreToolUse**: 在工具调用之前运行（可以阻止它们）
* • **PostToolUse**: 在工具调用完成后运行
* • **UserPromptSubmit**:  当用户提交提示时运行，在Claude处理之前
* • **Notification**:  当Claude Code发送通知时运行
* • **Stop**: 当Claude Code完成响应时运行
* • **SubagentStop**:  当子代理任务完成时运行
* • **PreCompact**: 在Claude Code即将运行压缩操作之前运行
* • **SessionStart**:  当Claude Code启动新会话或恢复现有会话时运行
* • **SessionEnd**: 当Claude Code会话结束时运行

利用钩子可以在流程中添加固定的代码审查、自动格式化、日志记录等功能，或者让CC在需要操作时提醒你，这一切取决于你的想象力。使用`/hooks`命令来管理钩子。

### 版本控制

#### 利用Git

有了AI Agent以后，Git的版本控制操作变得容易许多，你可以通过自然语言要求Agent帮你暂存文件，提交版本，还可以自动生成.gitignore文件等。提交的命名与摘要也可以通过AI记录的日志自动完成，结合钩子功能可以将Git版本控制融入到每一次开发中，不丢失任何版本。

另外，CC可以使用MCP或Github CLI与Github进行交互，并内置了Github插件以自动化处理issues。

#### 检查点回滚

Claude Code 2.0.0中，CC终于推出了检查点（Check point）功能，你的每个prompt都会被记录成一个检查点，可以随时通过`/rewind`命令撤销一个或多个更改，这可以作为比Git更微观的版本控制，避免AI破坏代码原有架构并要求修改。

### 其他神奇妙妙工具

#### `/output-style`

CC内置了三种输出风格：

* • 默认模式
* • 解释模式（Explanatory）：会输出一个Insights框概括知识点
* • 学习模式（Learning）：AI会在工作时将关键组件留给人类完成，共同开发

同时，也可以通过`.output-style:new`自定义输出风格，这会替换CC的内置提示词。

#### `/statusline`

设置CC的状态栏，会显示在聊天框下面，使用命令会让CC按照你的要求配置，也可以直接获取其他开发者定制的插件。如Haleclipse/CCometixLine: Claude Code statusline tool written in Rust
