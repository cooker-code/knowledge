很多人第一次折腾本地 AI 工作流，最先想到的动作，都是一样的：

多开几个 profile。

写作一个，研究一个，代码一个，发布一个。

看起来像是分工了，但用不了多久，系统就会开始变乱：谁都能做一点，谁也没有真正的边界；任务每次都靠临时解释；skill 到处乱装；结果回交格式也不统一。最后你得到的，不是一支团队，而是一堆互相打架的窗口。

所以这篇文章想解决的，不是“怎么多建几个身份”，而是一个更关键的问题：

怎么把 AI 从单点工具，真正组织成一套可复用的本地团队？

这两件事的差别非常大。

前者解决的是“我能不能多开几个助手”。
后者解决的是“我能不能把任务稳定分出去、收回来、复用起来，而且越跑越顺”。

如果你也在搭自己的本地多智能体系统，那我建议你先别急着继续加角色。先把“团队”这件事想明白。因为真正决定上限的，从来不是 agent 数量，而是组织方式。

## 一、为什么真正有用的，不是更多 Profile，而是团队化设计

我后来越来越确定一件事：

多开 profile，只是表层动作；团队化，才是底层设计。

一个真正可用的本地 AI 团队，至少要先回答 5 个问题：

1. 1. 谁负责什么。
2. 2. 任务怎么分发。
3. 3. 结果怎么回交。
4. 4. 能力装在哪。
5. 5. 出问题以后，怎么验证和修复。

如果这 5 个问题没有提前设计，系统很快就会出现几个典型症状：

* • 默认主智能体什么都做，子智能体只是“名义分工”
* • skill 到处乱装，后面根本不知道该维护谁
* • 每次协作都靠临时说明，无法复用
* • 看起来是多智能体，实际上只是多窗口单兵作战

所以更稳的做法，不是继续加角色，而是先把 default 退回到“编排者”的位置，再让每个子角色只承担自己最该承担的职责。

比如一套最小可运行团队，可以这样拆：

* • default：总控与编排，只负责拆解、路由、验收、总结
* • ruoyu：金融与新闻分析
* • fourier：Obsidian、内容整理、公众号与多平台分发
* • alan：资料搜集、来源验证、研究工作
* • mira：叙事结构、文章改写、表达优化
* • turing：自动化、代码实现、调试与工程架构

这套设计最大的价值，不是“看起来更专业”。

而是每次新任务进来，你都能立刻判断它该走哪条路线：

写公众号，不该让研究型角色硬写；
做研究，不该让发布型角色乱查；
做自动化，也不该让默认主智能体临时兼任工程师。

系统一旦有边界，才会开始有稳定性。

图示说明：先把角色边界画清楚，再谈协作效率。边界不清，后面所有“多角色协作”都会慢慢塌回单体模式。

## 二、一套可复用的本地 AI 团队，核心就看这 4 层

如果把整套系统拆开看，我认为最关键的是 4 层：角色层、规则层、能力层、验证层。

### 1）角色层：先把“谁是谁”写清楚

角色层不是起名字，而是明确边界。

每个 profile 都应该有自己的 SOUL.md，用来固定它是谁、该做什么、不该做什么。这里最关键的一刀，是把 default 改成“只编排，不代做”。

这一步看起来简单，实际上非常重要。

很多多智能体系统最后失效，不是子智能体不够强，而是主智能体太爱自己下场。它一旦开始代做，整个团队结构就会迅速塌回单体模式。

### 2）规则层：让所有角色站在同一块地面上

角色分出来以后，还需要一个所有人都能读到的共享工作区。

我的做法是单独建立一个 hermes-ws 目录，把团队协作规则都放进去，再把所有 profile 的默认工作目录统一指向这里。这样每个角色启动时，看到的就是同一套地面规则，而不是各自为政。

这层里最值得提前准备的，一般包括：

* • AGENTS.md：角色分工与路由原则
* • TASK-CARDS.md：任务卡格式
* • SKILL-OWNERSHIP.md：skill 归属规则
* • QUICKSTART.md：使用入口与常见路线
* • SMOKE-TEST 文档：验证记录

### 3）能力层：skill 要跟角色走，而不是全堆到 default

这是很多人最容易偷懒，也最容易把系统做乱的一层。

短期看，把所有 skill 都堆到 default 身上，确实最方便；但长期看，维护成本会越来越高。因为你会越来越难判断：这个能力到底是公共能力，还是某个角色的专属能力？

更稳的做法，是让 skill 跟角色走。

金融类能力给 ruoyu；
笔记、公众号、内容发布能力给 fourier；
研究能力给 alan；
写作改写能力给 mira；
代码与自动化能力给 turing。

最好连凭证也按角色隔离。比如金融数据 token，就只放进 ruoyu 自己的 .env，而不是全员共享。

这样做的价值很现实：既能降低权限耦合，也能让你后面扩展新角色时，不必重新清理一堆历史遗留。

### 4）验证层：系统不是“搭完就算”，而是“跑通才算”

很多人搭系统，只搭不测。结果结构看起来很完整，一跑就散。

所以一定要做 smoke test，而且第一轮不要上太重的任务。最好的方式，是先跑一条轻量但完整的闭环：

让 default 先生成任务卡；
再把任务交给某个子角色产出结构化结果；
最后回到 default 做验收、总结和下一步建议。

只要这条闭环能跑通，你就已经验证了三件事：

* • 路由规则有效
* • 角色边界有效
* • 回交流程有效

这比一上来就拿复杂实时任务压测，价值更高。

图示说明：真正可复用的不是某一个强 agent，而是“角色 + 规则 + 能力 + 验证”四层一起成立。

## 三、如果从零开始搭，建议按这 7 步推进

如果你准备从头搭一套本地 AI 团队，我建议顺序不要乱，直接按下面这条线走。

### 第 1 步：先检查当前 Hermes 状态

先确认系统能不能正常运行、已有 profile 是什么、关键配置文件在哪、后台依赖有没有缺失。

别一上来就改文件，先把现场摸清楚。

### 第 2 步：先做备份

这是最容易被忽略，但最值得保留的动作。

至少把 config、SOUL、已有 profile 的关键配置先备一份。因为一旦开始改角色边界，后面大概率会反复调整。没有备份，回滚成本会非常高。

### 第 3 步：再创建新角色

如果你已经有一个相对稳定的基础 profile，优先用 clone 方式创建新 profile。这样模型、认证、基础参数都能直接继承，能省掉大量重复劳动。

### 第 4 步：建立共享工作区

把团队协作反复需要引用的规则文档，都放进同一个工作区里。然后把所有角色的默认工作目录统一过去。

这样你不是在维护六套环境，而是在让六个角色读取同一个团队操作面板。

### 第 5 步：重写每个角色的 SOUL.md

这一步不能偷懒。

真正让团队成型的，不是 profile 被创建出来，而是每个角色都被写清楚边界：

谁能接什么任务；
谁不能越界；
哪些任务必须先路由；
哪些结果必须按固定格式回交。

这些都应该写进角色定义，而不是靠你自己临场记忆。

### 第 6 步：补齐任务卡和 skill 归属规则

任务卡解决的是“任务怎么分发”；
skill owner 规则解决的是“能力该装到谁身上”。

这两件事一旦缺位，后面就会重新退回最原始的口头协作状态：每次都重新解释，每次都不一样，每次都不能复用。

### 第 7 步：做第一次真实闭环验证

第一轮最好别上太重的任务。

可以先选一个轻量场景，比如让金融角色产出一个固定主题的观察框架，或者让写作角色整理一版结构化提纲。先验证结构，再逐步压复杂度，这是更稳的节奏。

当 default 能生成任务卡，子角色能按职责回交，default 又能基于回交做总结与下一步建议，这条团队链路才算真正活了。

图示说明：别追求一开始就复杂，先跑通一条最小闭环，系统才会越来越稳。

## 四、搭完之后，最容易踩的两个坑

这次搭建里，我印象最深的坑有两个。

图示说明：排查系统时，不要只看“能不能调用”，而要看 route、owner、dependency、verify 这些具体信号有没有失真。

### 坑 1：把“工具能调用”误以为“工作流已经可用”

实际上，这两件事差得很远。

工具可用，只代表单点动作能做；
工作流可用，代表整条链路可以稳定重复调用。

比如第一次做在线验证时，浏览器工具报错，缺少 chrome-headless-shell。这个时候如果你不继续排查，就很容易误判成“角色设计有问题”。

但真正的问题，可能只是底层依赖没补齐。

后来把 Playwright 相关浏览器依赖补上以后，至少可以确认：问题已经从“浏览器没装”压缩成“网络层访问仍需继续排查”。

这就是修系统时非常关键的一点：

你要不断把问题缩小，而不是笼统地觉得“整个系统不行”。

### 坑 2：角色有名字，但没有专属能力

如果子角色只是被命名，却没有真正独有的 skill，它很快就会退化成一个“提示词上的职业扮演者”。

看起来分工了，实际上没有新增能力。

所以我后来非常重视一件事：尽快给某个角色注入一项真正属于它的专属能力。

比如给 ruoyu 配一套 Tushare 查询能力，把 token 放进它自己的 .env，把接口逻辑放进脚本，而不是只塞进 prompt。

这样它才不只是“更像一个金融分析师”，而是真的多了一项其他角色没有的执行能力。

这一步特别关键。

因为它意味着你的系统，开始从“角色分工”进入“角色 + 专属能力”的阶段。到了这里，多智能体团队才真正有了扩展基础。

## 五、最后给你一个最实用的判断标准

如果你今天也想搭一套自己的本地 AI 团队，我最建议先做的，不是继续加人，而是先把现有结构跑顺。

你可以先问自己 5 个问题：

* • 主智能体是不是已经退回编排者位置？
* • 每个角色有没有清晰边界？
* • 团队有没有共享规则文档？
* • skill 和凭证是不是按 owner 隔离？
* • 有没有一条已经验证通过的最小闭环？

这 5 个问题里，只要有 2 个答不上来，你现在拥有的大概率还不是团队，而是一组尚未真正组织起来的 profile。

一句话总结：

多开几个 profile，只是把 AI 分身；
搭一套本地团队，才是把 AI 组织起来。

接下来最值得做的，也很明确：

先选一个最高频场景，把它沉淀成一条固定路线。比如“研究 -> 写作 -> 发布”，或者“路由 -> 分析 -> 总结复盘”。

当你能把一条路线反复跑顺，再给其中某个角色注入专属 skill，系统就会开始进入真正的复利阶段。

到那时，你得到的就不再是几个能聊天的助手，而是一套可以持续工作的本地 AI 团队。
