| 项 | 内容 |

## 架构图

```mermaid
flowchart LR
  Req["需求 / 问题"] --> Clarify["澄清与边界确认"]
  Clarify --> Plan["计划 / 设计 / 验收标准"]
  Plan --> Explore["代码库探索 / 上下文建模"]
  Explore --> Act["实现 / 多 Agent 执行"]
  Act --> Verify["测试 / 静态检查 / 运行验证"]
  Verify --> Review["人工与 Agent Review"]
  Review --> Summary["复盘 / 规则 / 知识沉淀"]
  Summary --> Clarify
```

## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 需求澄清 | 把自然语言需求转成边界、约束和验收标准 | 模糊需求是否会直接进入编码 |
| 计划与设计 | 把需求拆成方案、任务和验证路径 | 计划是否具体到文件、接口、数据和风险 |
| 上下文建模 | 让 Agent 获得代码结构、业务术语和调用链 | 上下文是动态搜索、规则文件、代码图谱还是人工摘要 |
| 执行编排 | 让单 Agent 或多 Agent 按计划实现 | 规划者和执行者是否分离，能否隔离上下文 |
| 验证闭环 | 用测试、CI、静态检查和人工评审约束输出 | 是否有可复现的验收信号 |
| 复盘沉淀 | 把踩坑、模式和决策沉淀为规则或 Skill | 是否能减少下一次同类任务返工 |

## 上下游

| 方向 | 对象 | 关系 |
|---|---|---|
| 上游 | PRD、需求讨论、缺陷报告、代码仓库 | 提供目标、边界和工程事实 |
| 下游 | 代码变更、测试报告、评审结论、发布结果 | 验证 AI 执行是否可交付 |
| 依赖 | Claude Code、Cursor、OpenCode、Codex CLI、CI、测试框架 | 提供执行、搜索、验证和持久化能力 |

## 横向对标

| 对标技术 | 对标点 | 优势 | 劣势 | 使用判断 |
|---|---|---|---|---|
| Vibe Coding | 快速原型 | 启动快，适合探索 | 易跳过边界、验证和维护性 | 原型阶段可用，不适合作为交付主流程 |
| Spec-Driven Development | 规格先行 | 需求和验收更清楚 | 规范过重时影响节奏 | 需求稳定、团队协作、长期维护时优先 |
| Claude Code Feature Dev | 阶段化功能开发 | 规划、探索、实现、审查链路清楚 | 依赖项目规则和验证习惯 | 适合仓库级功能开发 |
| Cursor Plan | IDE 内计划执行 | 与编辑器工作流贴近 | 长任务和跨工具自动化受 IDE 形态限制 | 适合交互式增量开发 |
| OpenCode / oh-my-opencode | 终端多 Agent 编排 | 更强调并行、Skill 和命令式工作流 | 配置和治理成本更高 | 适合需要终端自动化和多模型协作的任务 |
