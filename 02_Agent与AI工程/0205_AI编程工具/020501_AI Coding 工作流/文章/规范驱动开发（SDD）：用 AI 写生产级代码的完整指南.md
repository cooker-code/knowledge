> 把"规范"变成事实来源，让 AI 按规范稳定生成代码，再用系统化校验守住质量。

---

用 Claude Code、Cursor、CodeX 写代码的时候，你有没有过这种感受？宣传都说"90% 代码由 AI 生成"，真用起来却是调试时间变长、漏洞防不胜防、功能能跑但业务需求没满足。

怎么把"聊天式"的编码摸索，升级成团队能依赖、能验证、能上线的生产流程？答案就是：**规范驱动开发（Spec-Driven Development）**。

> 以下内容译自《Spec-Driven Development in 2025: The Complete Guide to Using AI to Write Production Code》

## 什么是规范驱动开发，它和传统开发有什么不同？

**规范驱动开发**就是：用一份详细的规范当蓝图，让 AI 照着它生成代码。你写清楚要什么，AI 负责实现。

传统开发是"需求 → 设计 → 手写代码 → 测试"。规范驱动变成了"需求 → 详细规范 → AI 生成 → 验证"。

两个关键区别：

* • **先写规范，再写代码**：开发者把精力花在架构、需求和验证上
* • **质量靠流程把关**：每次发现错误就改进规范，下次生成的代码质量自然更高

TDD 用测试来约束行为，规范驱动把范围扩大到完整实现。两者都和敏捷不冲突——规范可以在迭代中逐步完善。

再说说"vibe coding（氛围编码）"——就是没有规范、靠对话试来试去的编码方式。做原型、做试验、写小工具挺好，但问题是质量不稳定、文档不全、技术债越堆越多。规范驱动讲的刚好相反：结构化的规范加上标准流程，适合生产系统、企业应用和复杂架构。

两者不是谁取代谁——摸索阶段用 vibe coding，正式交付用规范驱动。有个研究数据挺有意思：只要需求写得清楚，LLM 的实现质量就很好。输出质量跟规范的详细程度直接挂钩——输入模糊，输出也模糊；规范写得细，代码就能直接上线。

## 为什么在 AI 辅助开发中，规范正在成为事实来源？

**技术上**的原因很直接：现在上下文窗口够大（200K+ tokens），能塞下一整份规范；模型也能读懂 OpenAPI、JSON Schema 这些结构化格式。

**业务上**的好处更实在：

* • 规范可以在不同 AI 工具间复用，不怕被一家绑定
* • 文档自然就有了，架构决策也记录在案
* • 团队靠"规范评审"对齐，合规和审计也有据可查

**质量控制**也变得有章可循：先审规范再生成代码，测试需求提前写清楚，安全要求、性能约束都在规范里。什么算"可以上线"，在动手写代码之前就定好了。

**投入产出**：写规范花几个小时，但手写实现要好几天。类似的特性下次直接复用，调试时间减少——需求清楚嘛。验证标准摆在那，新人上手也快。

**两个实际案例**：Google 的 AI 工具包做迁移，80% 的代码由 AI 写，迁移时间减了一半。Airbnb 六周用 LLM 迁移了 3,500 个测试文件——本来预计要 1.5 年。

## 哪些工具和平台支持规范驱动工作流？

2024–2025 年出了 15 个以上的相关平台。分三类，没有哪个"最好"，看团队情况选。

### AI 原生 IDE

* • **AWS Kiro**：三阶段工作流（规范 → 计划 → 执行），深度整合 AWS，处理老旧代码库很拿手
* • **Windsurf by Codeium**：有 Cascade Agent、强上下文感知和项目长期记忆
* • **Cursor**：20 美元/月，迭代快，社区活跃

### 命令行工具（CLI）

* • **Claude Code**：上下文长、能自治编程、有 Git 集成
* • **Aider**：终端里结对编程，开源、容易自动化，适合 CI/CD
* • **Amazon Q Developer**：自动升级 Java 版本、修废弃 API、自修复编译错误

CLI 工具适合自动化场景，想融入 CI/CD 的话，CLI 是第一选择。

### 集成开发工具

* • **GitHub Copilot**：市场老大，建议接受率约 33%，企业版 19 美元/人/月，入门最省事
* • **GitHub Spec Kit**：开源参考实现，展示规范驱动的 4 阶段工作流

### 企业平台

* • **HumanLayer**：强调"人工介入"的可控自动化
* • **Tessl**：以规范为中心持续重新生成代码
* • **Lovable**：关注 UI 的可视化规范工具

**选型要点**：团队规模、场景（新项目还是存量代码）、预算、和现有 CI/CD 的集成难度、学习成本。用好标准规范格式（OpenAPI/JSON Schema/Markdown）可以减少被锁定的风险。

## 怎么写一份有效的 AI 代码生成规范？

写规范是个技术活：**要清晰**（别让人猜）、**要完整**（边界和约束写清楚）、**上下文要给够**（让 AI 理解业务背景）、**要具体**（给例子比抽象描述管用）、**要可测**（验证标准提前定好）。

### 一份好规范该写什么

目标与价值、上下文与约束（架构、依赖、环境、性能要求）、功能需求、非功能需求（安全、性能、可扩展性）、边界与错误处理、测试标准、输入输出示例。

### 写多长？

| 对象 | 大概字数 |
| --- | --- |
| 基础函数 | 100–200 字 |
| API 端点 | 300–500 字 |
| 组件或模块 | 500–800 字 |
| 系统架构 | 1000–2000 字 |

### 几个有用技巧

* • 先给例子再讲抽象要求
* • 用 JSON Schema 或 TypeScript 接口定死输出格式
* • 给出反面例子（"不要做 X"）
* • 参考项目里已有的代码风格
* • 说清楚怎么测试

### 常见坑

* • 话说一半（"做好看一点""保证安全"——然后呢？）
* • 不写边界情况
* • 上下文给太少
* • 安全要求、性能目标只字不提
* • 没有测试方案

### 对比一下

**模糊版本**："做一个用户认证系统"

**详细规范**："在 Node.js Express API 中实现基于 JWT 的认证。要求：bcrypt 加盐 12 轮；刷新令牌 7 天有效、访问令牌 15 分钟有效；每 IP 15 分钟最多 5 次登录；MongoDB 存用户 email 和密码；用 Joi 校验（邮箱格式、至少 8 位密码）；错误响应用正确的 HTTP 状态码；单元测试覆盖正常流程和所有错误场景。安全方面：日志里不能出现密码；令牌用 HTTP-only cookies；给前端域名配 CORS。"

最好建一套模板库：函数模板、API 模板（含 OpenAPI）、前端组件模板、数据库迁移模板。模板能大幅提升效率和一致性。

## 怎么确保 AI 生成的代码达到生产级标准？

有数据说 67% 的开发者在学习阶段调试时间变长了。常见安全问题包括硬编码密码、SQL 注入。没有系统化的验证，技术债就会越积越多。而出事了，责任还是团队扛。

需要一个**五支柱验证框架**：

### 安全验证

上 SAST 工具、扫依赖漏洞、检测硬编码密钥、检查输入校验、防 SQL 注入和 XSS。

### 测试要求

设单元测试覆盖率底线；做 API 集成测试；关键用户流程做端到端测试；边界场景要覆盖；每次改动跑回归测试。

### 代码质量

强制执行 lint/format；关注圈复杂度；设可维护性阈值；文档写完整；命名规范统一。

### 性能验证

定响应时间目标；设内存和 CPU 限制；优化数据库查询；上缓存策略；做负载测试。

### 上线就绪

用配置管理（别硬编码）；环境变量用对；日志和可观测性配好；出错要优雅降级；回滚方案准备好；上线前监控告警都配齐。

**持续验证**：每次提交自动安全扫描、测试套件当闸门、质量阈值强制执行。

## 现实局限和挑战

诚实点说，AI 生成的代码有它的毛病：

### 代码质量

每次生成完都要验证。经常出现"幻觉依赖"——引用不存在的库。边界条件容易漏。性能上可能出现 N+1 查询。安全问题也可能混进来。

### 规范成本

写一份好的规范要花时间——每个特性几小时。规范质量直接决定输出质量，偷不了懒。而且规范要跟着代码一起维护。

### 团队阻力

有些开发者会抗拒——"AI 会不会取代我？" 写规范对很多人来说是陌生的技能。67% 的团队在学习阶段调试时间会增加。

### 不适合的场景

探索性工作、研究型项目、需求一天变三次的、需要极致性能优化的——这些场景老老实实手写或者 vibe coding 更合适。

**怎么降低风险**：从小范围试点开始，验证框架设成强制要求，人审不能少，持续培训，混合工作流，时间预期别太乐观。

## 实施路线图

### 阶段一：试点（第 1–4 周）

1–2 个开发者，在非关键的新功能上试。工具选 GitHub Copilot 或 Cursor，主要目标是学习和积累反馈。

### 阶段二：团队扩展（第 5–12 周）

推广到全体开发者。建立团队规范模板和评审流程，搞规范写作工作坊。目标：50% 以上的新功能用规范驱动。

### 阶段三：全组织推广（第 13–24 周）

规范驱动变成默认工作流。定好政策——规范评审、质量闸门、安全标准。跟踪 ROI 和满意度。目标：80% 以上采用率，ROI 为正。

**关键是**：管理层要支持、要有"种子选手"带头、时间预期要实际（6–12 个月成熟）、持续培训和调整。

## 怎么测试和调试 AI 生成的代码？

### 系统化调试流程

1. 1. 稳定复现问题
2. 2. 检查规范是不是写清楚了
3. 3. 识别常见的 AI 错误模式
4. 4. 把出错场景写回规范
5. 5. 按改进后的规范重新生成
6. 6. 用扩展测试验证修复

### 常见的 AI 错误

* • **幻觉依赖**：引用不存在的包
* • **边界没处理**：空值、异常值没检查
* • **理解偏差**：需求理解错了
* • **安全漏洞**：SQL 注入、XSS、硬编码密钥
* • **性能问题**：N+1 查询、低效算法
* • **错误处理不一致**：异常没好好处理

### CI/CD 里的"重试环"

生成 → 测试 → 逮到错误 → 更新规范 → 重新生成。通常 2–3 轮就到生产质量了。

代码评审的标准不变——AI 写的和人类写的一视同仁：先看是不是符合规范，再检查边界，用安全清单筛查一遍，确认测试覆盖到位。

## 进阶用例

### 代码迁移

升级语言版本、单体拆微服务、数据库迁移、API 版本升级。Google 在自己的仓库里做到了 75% 以上的 AI 生成修改成功落地。

### 遗留系统现代化

AI 辅助增量重构，给没测试的旧代码生成测试，给没文档的系统生成文档。

### 混合工作流

核心逻辑手写，样板代码 AI 生成。走"AI 草稿 → 人类评审 → 手动优化"的循环。

### 架构级规范

给多组件系统写设计规范，设计微服务架构，用 OpenAPI 设计 API，生成基础设施即代码。

## 怎么融入现有工作流？

**CI/CD**：规范作为流水线输入，规范变了就触发重新生成，安全/测试/质量闸门层层把关。

**版本管理**：规范当主要工件入库，生成代码也入库方便追溯。

**敏捷整合**：把写规范放进用户故事和迭代计划，评审时重点看"是不是符合规范"。

**度量指标**：跟踪规范编写时间、生成成功率、缺陷密度。研究数据：开发者速度提升 55%，每周省 2–3 小时，高阶用户能省 6 小时以上。

## 总结

规范驱动开发的核心转变很简单：从"上来就写代码"变成"先写清楚规范"。形式化的规范让 AI 生成的代码更一致、更可维护、更能直接上线。

工具很多，选适合自己团队规模的就行。要保证生产质量，五支柱验证框架不能省。别想一口吃成胖子——分阶段推进，先试再扩。

**你的下一步**：看看团队里 AI 编码用得怎么样；选 2–3 个匹配的工具；挑个非关键的新功能做试点；建好验证框架；定出规范和模板；设好衡量指标；按阶段推进。
