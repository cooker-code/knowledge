### 为什么我把 Cursor 当成主力（而不是完全替代 IDE）

说实话，我主要写 Java，用 IDE 开发，但在做前端复杂交互页面（尤其是 Vue）时，我发现 Cursor 的自然语言到代码的链路极省心：

* 草图直接丢给它，描述交互和 API 入参/返回 JSON，点击 apply → 得到可运行的组件骨架和交互逻辑。
* 复杂交互越多，Cursor 的理解能力越显著，比纯补全工具（如 Copilot 的部分场景）更接近“同事讨论”的感觉。
* 缺点也很真实：有时会出现风格不统一的样式、主题色跑偏等小问题，但整体收益仍然大于代价。

要点提醒：把 Cursor 当成“会理解上下文的同事或队友”，而不是“完美的自动化出品机”。它把重复劳动和试错搬走，让你把精力放在高价值决策上。

### 实战流程：从草图到可跑的 Vue 页面（推荐步骤）

1. 准备：草图+接口契约

* 把页面草图、交互说明、API 的请求/响应 JSON 准备好。越具体越好（字段、错误码、边界条件）。

2. 首次请求：用自然语言+上下文一次性给出任务

* 示例 Prompt（可直接复制到 Cursor 聊天框）：

```
     我有一个 Vue 页面草图，包含：列表、筛选条、详情弹窗和批量操作。后端接口：
     GET /api/items -> { items: [...], total: number }
     POST /api/items/batch -> { success: boolean }
     请生成基于 Vue 3 + Composition API 的组件骨架，使用 axios 调用接口，包含基本样式和关键交互逻辑，考虑加载态和错误处理。
```

3. 迭代：对交互细节、样式风格、主题色进行精细指令

* 如果主题色跑偏，指明变量或 CSS 变量，让 Cursor 按主题色替换。

4. 本地验收：运行、测试、提交（务必提交）

* 当结果满意时立刻提交代码。重要原因：Cursor 后续修改会基于当前仓库状态，未提交的变更可能被覆盖或导致它“改出问题”。

### Cursor 的常见坑和解决方法

* 坑：风格/主题不统一

+ 解法：在 prompt 或项目规则里声明样式指南（主题色、字体、CSS 框架）。

* 坑：生成的 API 调用逻辑不完全符合你的异常处理策略

+ 解法：把错误处理策略写成规则或在 prompt 中明确，例如“所有接口失败需弹 Toast 并上报 Sentry”。

* 坑：频繁试验导致代码混乱

+ 解法：使用分支 + 小步提交，把实验限制在可回滚范围内。

### 什么是 CursorRules？为什么你越早用越省心

CursorRules 是官方提供的规则系统，用来定制化指导 Cursor 的行为：代码风格、架构、UI 设计、性能、安全策略等。

好处是`把“每次都重复写规则”的成本降到最低`。

规则分两种：

* 全局规则（Rules for AI）：通过设置界面定义，适合个人偏好（比如“始终使用 TailwindCSS”）。
* 项目规则（Project Rules）：位于项目根目录的 `.cursor/rules` 下，使用 `.mdc`（Markdown）文件定义，优先级高于全局规则，适合团队或项目级别的约束。

优先级：项目规则 > 全局规则。

示例：在项目里创建 `.cursor/rules/chrome-plugin.mdc`，内容示例：

```
# Chrome 插件项目规则 - CursorRules

## 代码风格
- 使用 TypeScript 严格模式
- 优先函数式写法，避免类
- 变量命名使用驼峰，状态变量以 is/has 开头

## 架构与实践
- 遵循 Manifest V3
- 所有外部请求通过 src/services/http.ts 封装
- 所有 UI 组件使用 TailwindCSS 类名，避免内联样式

## UI 与可访问性
- 使用 Material 风格按钮尺寸
- 支持键盘导航与 aria 标签
```

把这样的文件加入项目后，Cursor 在生成或修改代码时会优先遵循这些规则，团队一致性大幅提升。

### CursorRules 使用建议与规则集合获取方式

* 先从最关键的 3 条规则开始（样式、API 封装、异常处理），后续逐步补充；规则过多反而会导致 AI 行为僵化。
* 在团队里维护一套共享规则集合（例如 `cursor.directory/rules`），新项目直接拷贝、微调即可。
* 定期审查并演进规则：当团队技术栈或流程变化时，及时同步规则内容。

### 三条工作流程硬性建议（节省踩坑时间）

1. 做实验先用分支，确认后再合并主分支。
2. 满意的结果立即提交并打 tag（避免上下文丢失）。
3. 把常用 prompt 与规则放到 README 或 docs，团队成员可以复制粘贴复用。

以上仅是个人使用中的经验之谈，不一定完全正确。如果有什么不正确、不严谨的地方请及时指正，不胜感激。
