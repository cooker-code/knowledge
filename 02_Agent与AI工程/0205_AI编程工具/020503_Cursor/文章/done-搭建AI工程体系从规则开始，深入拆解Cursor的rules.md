先从我们的交易系统说起。

这个系统前期主要是在 Cursor 里推进开发的。2025 年底到 2026 年初，项目的 `.cursor/rules/` 目录里积累了 9 个规则文件，总共 2323 行，接近 100KB。它们分别覆盖主索引、行为约束、开发流程、交易理念、数据库规范、测试纪律、架构约束、事件驱动规范和技术债清理。

后来我们又做了一个治理系统。到那个阶段，执行重心已经开始逐步从早期的 Cursor IDE 交互，转向 Codex CLI 驱动的多工作树链路。仓库里的 `AGENTS.md` 直接把 `main` 标成 "Cursor / human — interactive dev, merge operations"，而 `review-work`、`codex-work`、`codex-gpt-work` 则明确交给 `Codex CLI` 负责审校和实现。也正因为前后两段分别落在 Cursor 和 Codex CLI 上，回头再看同一类问题时，才更能确定：规则的边界不是某一个产品的问题。

交易系统这套规则不是随手写的几条注意事项。主索引规则里有一张 22 行的阅读优先级表，P0 到 P2 三级，标注了每个文档在什么场景下必读。有一张 22 条的关键规则速查表。有一套 9 步 AI 工作流程，从"接收任务"到"更新交接文档"，每一步该读什么、该做什么。

这些规则不是空话。行为约束规则里写着：

> **所有交易决策必须由AI模型做出，禁止硬编码规则。**

后面跟着正例反例的代码对比，再后面跟着一行加粗的注释：**血泪教训 (2025-12-17)：硬编码算法导致真金白银损失。**

这条规则是用钱买来的。

开发流程规则里写着：

> **禁止"想一出是一出"的行为！必须遵循以下流程：**

>

> 1. 完整阅读相关代码和文档 → 2. 制定完整方案（写下来）→ 3. 用户确认方案 → 4. 批量执行，一次到位 → 5. 验证结果

后面跟着一个禁止行为表，20 多条，每条标注了后果。"最小修复/临时修复/快速修复——浪费时间，问题反复"。"边做边想，没有完整方案——反复修改，险些酿成大错"。"返回假数据占位——误导用户，数据污染"。

这些不是假设。每一条后果描述的背后都是一次真实的事故。

AI 确实守了规矩。规则说先读文档再开发，AI 会先读。规则说不要硬编码，AI 不硬编码。规则说完整实现不要跳过功能，AI 确实不跳。规则说 git add 和 git commit 分开执行，AI 分开执行。

但结果，还是经常不能用。

---

不对劲的地方不在"AI 违规了"，而在"AI 完全合规但做出了用不了的东西"。这个感觉最初是模糊的——好像规则越多，AI 反而越像一个焦虑的实习生：每一步都在小心翼翼地不犯错，但你问它"这一轮到底要交付什么"，它答不上来。

回头翻那套规则，慢慢能看清楚它解决了什么、没解决什么。

**规则解决了四类真问题。**

第一类是**导航**。AI 进入一个有几十个目录、几百个文件的仓库，如果没有导航，它不知道先读什么。主索引规则里的优先级表和工作流程，解决的就是这个问题，告诉 AI "进来先读这几个文件，按照这个顺序"。

第二类是**行为边界**。什么事不能做、做了会怎样。"禁止硬编码交易决策"是行为边界。"禁止删除他人代码"是行为边界。"前端API不要用相对路径"是行为边界。这些规则画了一个"不可以"的圈。

第三类是**流程纪律**。"先读代码再改代码"、"先制定方案再执行"、"改完要验证"。这是过程约束。

第四类是**验证纪律**。行为约束规则里有一整段关键操作校验规范，操作完成后必须立即验证数据完整性，不能只返回 `{"success": True}` 就完事。测试规则还要求用浏览器工具实际跑，不能只说"已验证"。

这四类问题确实被解决了。AI 不再乱闯仓库、不再硬编码、不再跳过方案确认直接改代码、不再假装验证了。

**但有三类问题，规则完全没碰到。**

第一个是**执行坐标**——"本轮你要做什么"。规则可以说"先读文档、再制定方案、用户确认后再执行"，但它没有说"你这一轮的交付物是什么、做到什么程度算完成、哪些东西明确不做"。这三个信息需要随任务变化，写不进静态规则。

第二个是**跨轮状态**——"上一轮做到哪了"。这个问题在交易系统的规则体系里有一个很有意思的痕迹：主索引规则的 P0 必读列表里，有一项是 `docs/AGENT_HANDOFF.md`，注释写着"Agent交接文档（承上启下）"。规则规定"每次Agent工作结束时，必须更新本文档"。

但当你打开 `AGENT_HANDOFF.md`，发现它已经膨胀到了 274KB，版本号到了 v1.6.68，里面记录着从 v1.1.2 到 v1.6.68 的每一次变更。一份 274KB 的交接文档，本身就是"规则解决不了跨轮状态"这个问题的症状。规则说"更新交接文档"，AI 确实更新了——它不断地往里追加，但没有人（包括规则自身）定义什么该留、什么该删、下一轮 agent 进来应该看哪一段。

第三个是**交付方向**——"在这些约束下，朝哪个方向走"。规则画了圈（不要做 X），但圈内的空间是巨大的。AI 可以在圈内做 A，也可以做 B，也可以做 C，都不违规。问题是 A、B、C 里只有一个是这一轮需要的。

这三个缺口，不是规则写得不好，而是规则这个形式装不下这些东西。

---

这个判断不是事后聪明。它在 `phase-pressure-map.md` 里有直接的文字记录——这是一份复盘工件，记录了每个工程阶段旧方法失效的真实压力点。

"跳 1：无规则 → Cursor Rules"那一节写着：

> 旧方法的失效点：AI agent 进入仓库后没有导航，随意修改代码；没有所有权概念，AI 可能覆盖关键业务逻辑；AI 倾向于硬编码快捷方案而不是使用模型决策。

"跳 2：Rules → Spec 结构标准"那一节写着：

> 旧方法的失效点：规则能约束"怎么写代码"，但不能定义"交付什么才算完成"；规则是被动的（"不要做 X"），不是主动的（"请完成 Y 并交付 Z"）；多个 AI 按同一套规则工作，但各自理解不同；没有验收标准，无法判断"做完了没有"。

这段话不是我写的解读。它是项目演化过程中留下的原始记录。规则的上限，在这里已经写得很清楚了。

`real-artifact-map.md` 的 Rules 阶段记录进一步印证了这个判断。R1 到 R6，六个代表性工件，每个都标注了"解决了什么"。R1 对应主索引规则，解决的是"AI 进入仓库后先读什么"的导航问题。R2 对应行为约束规则，解决的是"禁止 AI 硬编码业务逻辑，强制用模型决策"。R3 对应开发流程规则，解决的是"防止 AI 跳过方案确认直接改代码"。

注意：没有哪个工件声称自己解决了"定义交付物"或"传递跨轮状态"的问题。

---

到这里，如果只是说"规则管了行为但管不了方向"，还只是一个工程层面的观察。更有意思的问题是：**这背后有没有技术机制层面的解释？为什么规则在 agent 壳层里天然就只能做到这一步？**

有。而且不止一个机制在起作用。

### Context Assembly：所有信号在同一个窗口里抢注意力

在我们前期主要用 Cursor 做开发的那个交易系统里，用户看到的是"规则文件"和"对话窗口"分开的两个界面。但对于底层的语言模型来说，没有"分开"这回事。

更接近真实的机制是：在 Cursor、Codex CLI 以及其他同类 agent 壳层里，系统提示、规则文件、当前对话历史、局部代码上下文、工具调用结果，最终都会以某种装配方式进入同一个模型上下文，由同一个注意力机制统一处理——

* **系统提示**

  （agent 壳层自身的系统指令）
* **规则文件**

  （比如交易系统里 `.cursor/rules/` 下命中 glob 或标记 alwaysApply 的 `.mdc` 文件）
* **用户的即时指令**

  （当前这一轮用户说了什么）
* **对话历史**

  （之前几轮的交互记录）
* **局部代码上下文**

  （光标附近的代码、打开的文件）
* **工具调用结果**

  （文件搜索、代码执行、终端输出等）

这些内容全部被拼接成一个线性的文本序列，装进同一个上下文窗口，由同一个 attention 机制处理。

规则在这个序列里，就是众多段落之一。它不享有特殊的"执行权重加成"。它和用户的即时指令、工具调用返回的 500 行终端输出、对话历史里的上一轮代码修改结果，处于完全平等的注意力竞争关系中。

这不是某一个工具独有的设计缺陷，而是当前大多数 LLM agent 壳层都绕不开的共性：对模型来说，它看到的不是“规则界面”“聊天界面”“工具结果界面”这些前端分层，而是一段被装配好的上下文。底层模型并不知道哪些段落“在产品界面里看起来更重要”。

区别只在于装配策略和读取策略会有差异。有的壳层更倾向于按规则匹配自动带入，有的更依赖会话启动时的 instructions，有的更依赖执行中的文件检索和工具回读。但无论差异怎么存在，“所有信号在同一个窗口里竞争”这个基本事实是一样的。

所以"规则写了就一定被严格执行"这个假设，在技术机制上就不成立。更准确的表述是：**规则被写进了 prompt，然后和其他所有信号一起，被模型自行权衡。**

### Instruction Competition：规则越多，互相削弱越明显

交易系统那套以 Cursor 为主的规则包有 9 个文件、22 条关键规则。这些规则在大多数场景下互相兼容。但在某些边界场景下，它们之间会产生张力。

举一个从规则原文能直接看到的例子。行为约束规则里说：

> **所有交易决策必须由AI模型做出，禁止硬编码规则。**

开发流程规则里说：

> **严格禁止跳过功能实现或简化功能！**
>
> ✅ **完整实现所有功能，不允许跳过**

这两条规则在 99% 的场景下不冲突。但设想一个场景：AI 在实现一个新的交易执行路径时，发现调用 AI 模型服务的依赖暂时不可用。"完整实现所有功能"指向"先把功能跑通"，"禁止硬编码交易决策"指向"不能绕过模型"。两条规则在这个具体场景下产生了方向性张力。

模型不是"选择违反哪条"。从失败模式看，更像是这样：**模型在多条指令之间做了一个对当前轮次 salience 最高的信号更有利的权衡。**如果用户刚刚在上一轮说了"把这个功能跑通"，那么"完整实现"的权重会被当前对话信号放大，而"禁止硬编码"作为背景规则的权重相对被稀释。

这就是 instruction competition——指令竞争。规则的数量和竞争的激烈程度正相关。9 个文件、22 条规则，意味着 22 个信号在争夺模型的"遵从预算"。这不是说规则太多所以要删减，而是说：**规则越多，任何单条规则在特定场景下被"完整遵从"的概率越低。**因为模型的 attention 是有限资源，分配给每条规则的份额是此消彼长的。

到了治理系统阶段，这个问题并没有因为壳层逐步切到 Codex CLI 就消失，反而被看得更清楚了。治理系统的主规则里规定：

> 单次循环最多完成 3 个 task（或 1 个复杂 spec）；建议每轮 1～3 项，不必凑满。

> 每完成一个完整 spec（所有 task [x]）后，或累计执行超过 5 轮后，必须执行 /compact 压缩当前对话上下文。

用规则来管理 context 溢出、用规则来约束规则本身的衰减——这是一种"规则管理规则"的递归结构。它确实有效地缓解了问题，但也暴露了一个根本矛盾：**如果规则需要用更多的规则来保证自身被遵从，那么这个体系的维护成本是递增的，而可靠性不是。**

### Recency Bias：对话越长，规则越"远"

这个机制和 context assembly 竞争直接关联，但值得单独拆出来说，因为它解释了一个非常具体的现象：**为什么 AI 在对话前几轮表现得很守规矩，但到了第 8 轮、第 10 轮就开始"忘记"规则。**

更接近真实的机制是这样的：规则通常在 prompt 的靠前位置被加载（作为 system prompt 或对话的早期部分）。随着对话推进，每一轮的用户指令、AI 回复、工具调用结果都在 prompt 的末尾不断追加。到了第 10 轮，对话历史的长度可能已经是规则文本的 5 到 10 倍。

Transformer 的 attention 机制对位置是敏感的。虽然理论上长上下文窗口内的 token 都"看得到"，但在实际生成里，prompt 末尾的信号，也就是最新几轮的对话，通常会表现出更强的牵引力。这就是 recency bias。

规则在对话第 1 轮时是"最近的信号之一"。到了第 10 轮，它变成了"很早之前的一段背景文本"。**规则的文字没有变，但它在 prompt 里的相对位置变了，它获得的注意力权重就变了。**

这就是为什么治理系统的规则里会有"累计执行超过 5 轮后，必须执行 /compact"这条规定。/compact 的本质是什么？是重置对话上下文，让规则重新回到"靠近 prompt 末尾"的位置。这是一个工程上的缓解手段，但它本身也是一条规则，也就是说，它也会受到 recency bias 的影响。**如果到了第 6 轮时模型已经因为 recency bias 开始忽略背景规则，那么"第 5 轮必须 /compact"这条规则本身，也可能在第 6 轮被忽略。**

这个问题在 2026 年的长上下文模型上并没有消失。窗口更大，意味着能塞进去的信息更多了，但 recency bias 的方向没变。新的信号仍然更容易牵动当前输出，早期信号仍然更容易退成背景。窗口更大，竞争反而更激烈，因为参与竞争的信号更多了。

### 被动约束 vs 主动执行入口：规则天然的语气限制

回到交易系统那批规则原文，有一个句式上的共同点：

* "禁止硬编码交易决策"
* "禁止想一出是一出"
* "严格禁止跳过功能实现"
* "不要删除他人代码"
* "关键操作完成后必须立即验证"
* "必须先阅读以下文档"

这些全部是约束——告诉 AI "不要做什么"或"必须先做什么再做什么"。

agent 需要的另一类信息是执行入口——"本轮做什么、做到什么程度算完、不做什么"。区别在哪？

约束："禁止硬编码交易决策"。

执行入口："本轮为 `strategy_execution_monitor` 任务新增链路追踪节点生成逻辑，修改 `agent/runtime.py` 的 `execute_step` 方法，交付物是能在 Activity 页面展开查看的决策链路节点列表，最小验收标准是本地 pytest 通过且浏览器端到端验证截图。不改动数据库 schema。"

约束是静态的、通用的、背景性的。执行入口是动态的、任务特定的、前景性的。规则文件的设计范式，无论是 Cursor 的 `.mdc`，还是 Codex CLI 的 instructions，都天然更适合承载前者。

这不是说规则"应该"去承载执行入口。相反，**规则不应该承载执行入口，因为执行入口需要随任务变化，而规则是跨任务的。**这正是为什么光有规则不够：一个 agent 可以完美遵守所有约束（不硬编码、先读文档、不删他人代码），同时在圈内空间里做出任何方向的产出——因为没有人告诉它"这次应该朝哪个方向走"。

`phase-pressure-map.md` 里的"跳 2"记录，描述的就是这个转折点：

> 规则是被动的（"不要做 X"），不是主动的（"请完成 Y 并交付 Z"）；多个 AI 按同一套规则工作，但各自理解不同；没有验收标准，无法判断"做完了没有"。

这不是规则的失败。这是规则的天花板。

### 2026 年的变化：窗口更大了，竞争没消失

到 2026 年，几件事确实在变。

第一，长上下文和多步执行比前一年更强了。无论你看的是 IDE agent 还是命令行 agent，能塞进去的上下文更多了，能连续做的动作也更多了。最早那种“规则文本太长，根本带不进去”的问题，在今天已经不是主要矛盾。

第二，工具调用和执行链也更成熟了。agent 会搜仓库、读文件、跑命令、回头修错误，这些动作比过去更像真正可用的工作链，而不只是演示。

第三，规则被带进上下文的概率确实高了一些。可这并不等于规则就自动变成了执行入口。

但 instruction competition 和 recency bias 并没有因为窗口变大而消失。

窗口更大意味着能塞进去的信息更多。规则能塞进去了，但用户的即时指令也能塞更多，工具调用的返回结果也能塞更多，对话历史也能保留更多轮。参与竞争的信号总量增加了，规则的相对份额不一定提高了。

执行链条更长意味着对话轮次更多。agent 一轮做 3 步和一轮做 8 步，对 recency bias 的影响是不同的——步数越多，规则在 prompt 中的相对位置越远。

规则越来越多反而让竞争更激烈。交易系统的规则从无到 9 个文件 2323 行，到了治理系统阶段，又在此基础上叠加了 spec 循环、批次控制、context 压缩、worktree 隔离、审校流程。规则的总量还在增长，每一条新规则加入 prompt 后，就是多一个信号参与注意力竞争。

所以 2026 年的现实是：**规则仍然是底板——没有规则的 agent 仍然会乱闯仓库、硬编码、跳过验证。但指望规则一层解决所有问题，在 2026 年仍然不现实。**不是因为规则写得不够好，而是因为 context assembly 竞争、instruction competition、recency bias 这些机制，是 LLM agent 架构层面的结构性约束，不会因为窗口变大或规则写得更精致而消失。

---

### 从规则到规则之外

回头看从交易系统到治理系统这条规则演化线，有一条很清楚的脉络：

交易系统那批主要跑在 Cursor 上的规则，解决了导航、行为边界、流程纪律、验证纪律。然后碰到了天花板，管不了执行坐标、跨轮状态、交付方向。于是 `AGENT_HANDOFF.md` 出现了，作为"规则之外的信息容器"，被规则本身标记为 P0 必读。但它膨胀到 274KB 之后，自身也变成了一个问题。

治理系统到了逐步转向 Codex CLI 的阶段，又尝试了更进一步的方案，8 步 workflow loop、spec 循环、批次控制、强制 /compact，再加上多工作树的审校链路。它在用规则来管理规则的衰减。这在工程上确实有效，但它同时也证明了：**规则体系越成熟，就越需要规则之外的东西来补它的缺口。**

那些"规则之外的东西"是什么？从这两段演化看，至少包括：执行 spec（定义"做什么到什么程度算完"）、交接文档（传递"上一轮做到哪了、失败了什么"）、状态工件（持续可查询的"当前世界模型"）。这些不在本篇的范围内，但它们出现的原因，在本篇应该已经讲清楚了：规则是被动约束，不是执行入口。这件事有技术机制层的解释。

**规则没有坏。它在做它能做的事。**

坏的是"规则应该能解决所有问题"这个预期。

## 可直接落地的规则包模板

下面这套不是给新手看的 checklist，而是一份可以直接进项目的规则层骨架。它基于前期 Cursor 工程和后期逐步转向 Codex CLI 的治理系统这两段经验设计，你可以把它映射到 `.cursor/rules/`、instructions 文件或其他等价载体里。重点不是壳层名字，而是这套结构到底在补哪一类失效。

边界也先说清楚：这套模板解决的是导航、边界、流程和验证纪律，不解决执行坐标、跨轮连续性和全局状态。也就是说，它能把规则层立起来，但不能替代 spec、handoff 和 truth/state。

### 目录结构

```
.cursor/rules/                      # 或等效的 agent 规则目录
├── {project}_core.mdc              # 主索引（必填）
├── {project}_principles.mdc        # 行为约束（必填）
├── {project}_development.mdc       # 开发流程（必填）
├── {project}_testing.mdc           # 测试纪律（推荐）
├── {project}_architecture.mdc      # 架构约束（推荐）
└── {project}_domain_{name}.mdc     # 领域规则（按需）
```

**为什么这样拆**：主索引 + 子规则的分离结构不是为了美观，而是为了控制规则包进入上下文时的体积和竞争强度。主索引每次都要进，子规则尽量按需进。如果所有规则都强行每轮带入，规则越重，真正被执行时反而越容易互相稀释。

### 主索引 `{project}_core.mdc` 最小字段定义

```
---
description: "Global workspace rules index — [一句话说明项目]"
globs:
  - "*"
alwaysApply: true     # 主索引必须 alwaysApply
---
```

```
# {项目名} 工作指导总纲

> **版本**: v{X.Y.Z} ({日期})

## 一、AI 必读文档（阅读优先级表）
# ⚠️ 常见失效位：这张表最容易被 recency bias 覆盖。
# 对话超过 5 轮后，AI 大概率不会再主动回来看这张表。
# 缓解：在表底加一行"每 5 轮重新阅读本表"（但这本身也会被忘）。
# 根本缓解：不依赖规则驱动阅读行为，用 spec 或交接文档承载"本轮要读什么"。

| 优先级 | 文档路径 | 何时必读 | 内容摘要 |
|--------|---------|---------|---------|
| P0     | 本文档   | 每次    | 核心索引 |
| P0     | {交接文档路径} | 每次 | 当前状态与待办 |
| P1     | {子规则路径}  | {场景} | {一句话} |

## 二、关键规则速查表
# ⚠️ 常见伪完成位：AI 阅读了速查表但只记住了表头和最后一行。
# 中间的规则（第 5-15 条）是最容易在长对话中丢失的。
# 缓解：把最关键的 3 条规则放在表的最前面，而不是按分类排序。

| # | 规则名 | 一句话 | 详细文档 |
|---|--------|--------|---------|
| 1 | {最高优先级规则} | ... | ... |
| 2 | {次高优先级规则} | ... | ... |

## 三、AI 工作流程
# ⚠️ 常见失效位：流程的第 1-2 步（读文档、确认方案）最容易被跳过。
# 因为用户的即时指令通常是"做XX"，模型会直接跳到"做"那一步。
# 缓解：在流程第 1 步用否定式强调——"不执行本流程直接编码 = 违规"。
# 但注意：否定式表述本身也受 instruction competition 影响。

1. 接收任务
2. 阅读本文档
3. 阅读交接文档
4. 根据任务类型阅读相关子文档
5. 检查影响范围
6. 阅读相关代码
7. 制定方案并确认
8. 开始编码
9. 验证结果
10. 更新交接文档
```

### 行为约束 `{project}_principles.mdc` 最小字段定义

```
---
description: "Core principles — apply when editing production code"
globs:
  - "**/*.py"
  - "**/*.ts"
  - "**/*.tsx"
  # 只匹配需要约束的文件类型，不要 "*"
alwaysApply: false    # 行为约束不需要 alwaysApply，由 glob 触发
---
```

```
# 核心原则规范

> **版本**: v{X.Y.Z}

## 一、{最高优先级约束}  🔴🔴🔴
# 字段设计原因：每个约束必须包含：
# 1. 规则原文（"禁止/必须"）
# 2. 正例代码（✅）
# 3. 反例代码（❌）
# 4. 教训来源（可选但强烈建议——"2025-12-17 硬编码导致损失"这类真实日期+后果）
#
# 为什么需要正例/反例代码对：
# 纯文字规则被模型"理解"的方式是概率性的。
# 代码对比是更确定性的信号——模型在代码生成时会直接参考正例模式。
#
# ⚠️ 常见失效位：反例代码比正例代码更容易被模型"记住"并反向使用。
# 如果反例代码写得太完整（可直接运行），模型可能直接复制反例。
# 缓解：反例代码故意写不完整，或在关键位置加 # 禁止！注释。

**{规则原文}**

✅ 正确：
{正例代码}

❌ 禁止：
{反例代码}

**教训来源**：{日期}，{一句话后果描述}

## 二、禁止行为表
# ⚠️ 常见伪完成位：禁止行为表超过 15 条后，AI 对中间条目的遵从率显著下降。
# 这不是模型的缺陷，而是 instruction competition 的直接后果。
# 缓解：控制在 10-15 条以内。超过的部分拆到子规则文件，按 glob 按需加载。

| 禁止行为 | 后果 |
|----------|------|
| {行为描述} | {后果描述} |
```

### 开发流程 `{project}_development.mdc` 最小字段定义

```
# 开发规范

## 一、重构/修改前的强制流程 🔴🔴🔴
# ⚠️ 这是整个规则包中最容易失效的部分。
# 原因：用户的即时指令（"改一下这个bug"）是当前轮次 salience 最高的信号，
# 直接跳过"先读代码、制定方案"这两步的诱惑极大。
# 
# 最小验收口径：AI 在编码前是否输出了"方案确认"文本段？
# 如果 AI 的回复直接是代码修改，没有方案段，就是流程失效。

1. 完整阅读相关代码和文档
2. 制定完整方案（写下来——必须有文字输出，不能只在"内部思考"）
3. 用户确认方案
4. 批量执行，一次到位
5. 验证结果

## 二、完整实现原则 🔴🔴🔴
# ⚠️ 常见误判位：AI 认为"数据为空所以功能不显示"是合理的。
# 实际上应该显示"暂无数据"而不是隐藏整个功能块。
# 这条规则和"快速实现"的用户指令产生 instruction competition 的频率最高。

## 三、Git 规范
# 最小验收口径：git add 和 git commit 是否分开执行？
# staging 里是否只有本任务文件？
```

### 常见规则失效位总结

| 失效位 | 机制 | 典型表现 | 缓解方式 | 缓解的局限 |
| --- | --- | --- | --- | --- |
| 阅读优先级表被跳过 | Recency Bias | 对话第 6 轮后 AI 不再主动回查文档 | 在表底加"每 5 轮重读"；用 /compact 重置 | "每 5 轮重读"本身也受 recency bias 影响 |
| 中间规则被遗忘 | Instruction Competition | 速查表第 5-15 条遵从率下降 | 最关键的放最前面；拆文件按需加载 | 拆文件增加维护成本 |
| 强制流程被跳过 | Salience Bias | 用户说"改这个bug"，AI 直接改不做方案 | 否定式强调"不走流程=违规" | 否定式本身也参与指令竞争 |
| 反例代码被正向使用 | 模式匹配 | AI 复制了反例的代码结构 | 反例写不完整，关键位置加 # 禁止！ | 不完整的反例降低了可读性 |
| /compact 规则被忘记 | Recency Bias | 超过 5 轮后既没 compact 也没遵从 compact 规则 | 壳层自动提醒（如果壳层支持） | 依赖壳层实现，非规则层能控制 |
| 禁止行为表膨胀 | Instruction Competition | 表超过 15 条后中间条目失效 | 控制在 10-15 条，超出部分拆子文件 | 子文件依赖 glob 精确匹配 |
| 跨轮状态丢失 | 规则范式局限 | 第二轮 agent 不知道第一轮做到哪 | 规则指向交接文档 | 交接文档的维护本身无法被规则保障 |

### 规则包最小验收口径

一套规则包在部署前，至少验证以下几点：

1. **主索引是否 alwaysApply**

   ：不是的话，某些文件类型下 AI 完全看不到规则
2. **子规则的 glob 是否覆盖了目标文件类型**

   ：漏配 `.md` 很常见——结果是 AI 编辑文档时不加载任何规则
3. **最高优先级规则是否在速查表前 3 条**

   ：第 4 条开始的遵从率在长对话中显著下降
4. **正例/反例代码对是否完整**

   ：只有文字描述没有代码对比的规则，被"正确理解"的概率更低
5. **禁止行为表是否超过 15 条**

   ：超过则拆分
6. **是否有至少一条 context 管理规则**

   （如"N 轮后 /compact"）：没有的话，长对话中所有规则的衰减没有任何缓解
7. **总规则量是否已经重到需要分层**

   ：如果规则越来越像一整本手册，就别再继续往一个入口里堆，应该开始拆出 spec、handoff 和状态层
