#### 一、skill 是什么？

#### 

skill的实质是一套写在markdown文件中的测试标准作业程序，简称SOP。

* 可将测试专家的经验、流程、模板等固化下来，让AI按照标准规则干活，是一套可复用的规则
* 将AI从聊天chat 升级为专业领域的QA小能手

#### 二、skill的基本结构

#### 

##### 1、skill的结构

##### 

skill-name/

    |-SKILL.md       #必有的主文件，定义了skill的描述、步骤与规则、输出格式

    |-references/   # 非必须的参考资源如业务背景

    |-scripts/         # 非必须的脚本，主要是⼤模型可调⽤的脚本&⼯具 

    |-examples/     # 非必须的示例⽂件，一般写可供AI参考的同类的案例

* skill文件的主要内容

skill.md文件中使用已有的skill 主要是/docx 解析需求文档，和 /xlsx导出excel测试用例文件，文件内容如下：

```
---name: generate-testcasesdescription: Generate structured software test cases from DOCX requirement documents and export them into Excel (.xlsx) files. Use when the user provides a requirements document (especially .docx) and asks to create or update test case spreadsheets.---# 从 DOCX 需求文档生成测试用例并导出为 Excel## 使用时机在以下场景下使用本 Skill：- 用户提供 **需求文档（DOCX 格式）**，希望自动生成测试用例- 用户希望把测试用例整理到 **Excel（.xlsx）表格** 中- 用户提到 “测试用例设计”“从需求生成测试用例”“导出/生成测试用例 Excel/表格” 等关键词## 总体流程每次使用本 Skill 时，遵循下面的固定流程：1. **确认输入与期望输出**   - 确认用户已经上传或提供了 DOCX 需求文档路径   - 确认用户希望的输出形式：     - 仅要测试用例列表（文本/Markdown）     - 需要生成 Excel 文件（.xlsx）   - 如果用户没有特别说明 Excel 模板/字段，使用默认字段集2. **分析需求文档结构（DOCX）**   - 阅读 DOCX 文档，优先关注：     - 标题、章节结构（如：模块、功能点、用例场景等）     - 清晰标注的需求 ID（如：REQ-001、3.2.1 等）     - 列表项、业务流程描述、功能说明、边界/异常描述   - 将需求按 **模块 / 功能点 / 需求条目** 进行拆分，形成“候选需求列表”3. **设计测试用例字段结构**   - 默认推荐的 Excel 字段（列）为：     - `用例ID`     - `模块/子系统`     - `用例标题`     - `用例类型`     - `前置条件`     - `测试步骤`     - `预期结果`     - `优先级`   - 若用户已经有固定模板（字段名、顺序、是否必填）：     - 以用户模板为主，**将上面的信息映射/适配到用户字段**   - 如果用户没有说明模板，直接使用默认字段并在输出说明中简要列出这些列名4. **从需求推导测试用例**   - 对每个需求条目：     - 识别其 **正常场景**、**边界场景**、**异常/错误场景**、**组合场景** 等     - 优先覆盖：       - 功能是否被正确实现（正向用例）       - 输入边界（最小值/最大值/空值/非法格式等）       - 授权/角色/权限相关场景       - 异常路径（接口失败、网络问题、依赖服务异常等）       - 并发场景 如多人同时触发某个功能   - 为每个场景生成：     - 简短清晰的 **用例标题**     - 明确的 **前置条件**（如登录状态、已有数据、系统配置等）     - 步骤化的 **测试步骤**（按 1/2/… 描述）     - 和步骤一一对应的 **预期结果**     - 合理的 **优先级**（P0/P1/P2 或 高/中/低）     - 合适的 **用例类型**（功能/边界/异常/接口/性能/安全等）5. **与用户确认策略（如有必要）**   - 当以下情况发生时，先向用户简要确认再继续：     - 需求文档结构混乱、缺少明显模块/需求 ID     - 测试用例数量可能非常大，需要控制粒度或优先级     - 不清楚用户偏好的字段命名方式或优先级划分标准   - 确认方式要简洁，例如：     - 是否需要拆分到非常细的步骤级别？     - 是否只关注核心功能（高优先级）？     - 是否有现成的 Excel 模板需要对齐字段名？6. **整理为结构化数据**   - 在正式写入 Excel 之前，先在对话中以 **Markdown 表格** 或 **结构化 JSON 列表** 的形式列出部分/全部测试用例，便于用户预览和修改   - 推荐 Markdown 表格示例（可只展示部分字段）：     - 列：`用例ID`、`模块/子系统`、`用例标题`、`用例类型`、`前置条件`、`测试步骤`、`预期结果`、`优先级`7. **导出到 Excel（.xlsx）**   - 当用户明确需要 Excel 输出时：     - 使用/xlsx 的skill     - 第一行写入字段名（列头）     - 后续每一行写入一个测试用例   - 若用户没有指定文件名，默认命名规则：     - `testcases_from_<原docx文件名_去后缀>.xlsx`   - 生成完成后，将文件路径告知用户，并简要说明：     - 用了哪些列     - 约有多少条用例8. **复核与微调**   - 简要自查：     - 是否每个需求至少有 1 条正向用例     - 关键约束/边界条件是否有覆盖     - 重要异常路径是否有覆盖   - 如用户提出修改（比如列名调整、增加字段、合并/拆分用例），优先在内存中的结构化数据上调整，再重新导出 Excel## 具体操作指引（给 Agent）当用户触发此 Skill 时，按下面步骤执行：1. **获取输入信息**   - 确认 DOCX 文件路径或用户上传的文件名   - 询问（如果用户未明确）：     - 是否有已有的 Excel 模板（有则请用户描述字段/示例）     - 是否有特殊字段或命名要求（例如“用例编号规则”等）2. **解析 DOCX 文档**   - 视情况选择方式：     - 在简单场景下，可以直接请求用户粘贴关键章节内容到对话中     - 使用/docx解析需求文档3. **生成测试用例草稿**   - 根据中间结构，自动扩展出多个测试用例，并填充默认字段：     - 缺省情况下可自动生成 `用例ID`（如 `TC-001`, `TC-002`…）   - 对于每个需求，至少生成：     - 1 条正常流程用例     - 若有明显边界/异常场景，再补充 1–N 条边界/异常用例4. **在对话中展示预览**   - 先展示少量代表性的用例（例如每个模块 2–3 条），采用 Markdown 表格形式   - 如用户满意结构与风格，再生成完整列表并导出 Excel5. **生成 Excel 文件**   - 使用/xlsx导出测试用例6. **结果说明**   - 完成后，简要总结：     - 本次根据 DOCX 解析出多少条需求     - 共生成多少条测试用例     - Excel 中有哪些列     - 任何值得用户注意的假设或未覆盖点## 示例交互（示意）- **用户意图示例**  - “这是需求文档（.docx），帮我生成测试用例并放到 Excel 里。”  - “根据这份需求说明书，输出测试用例，最好能给一个 xlsx 表格。”- **Agent 处理示例（高层）**  1. 确认 DOCX 文件和输出需求（是否必须 .xlsx）  2. 解析文档，按模块/需求 ID 拆分需求  3. 生成覆盖正常/边界/异常场景的测试用例  4. 以 Markdown 表格预览部分用例，征求用户确认  5. 将最终用例导出到 Excel（或提供可直接粘贴到 Excel 的表格）  6. 说明生成情况与后续可调整点
```

SkILL.md的经验分享

* description要写清楚: AI 是否调⽤Skill，很⼤程度取决于description描述
* 提供模板示例：如果你给出⼀个表格模板，AI 会按这个结构输出，在日常工作中可将要生成的excel测试用例模版通过markdown文件描述放在examples下
* Skill.md文件不要太⼤：建议小于500 ⾏，可将复杂内容放references下的markdown文件中
