> 之前做“前任Skill”时，跑通过从微信获取聊天记录。最近看到腾讯元宝上线了群聊天记录摘要功能，能自动总结群聊要点。想了想，我手上的 WorkBuddy 完全能做得更广更灵活。数据库密钥之前已经提过了，解密脚本也跑通过，不如直接封装成一个 Skill——想查哪天就查哪天，想看哪个群就看哪个群，还能按发送者、按时间范围自由筛选。现在每天只要说一句“生成今天微信日报”，30 秒拿到完整的聊天摘要。

## 一、微信聊天记录存在哪？

很多人不知道，微信在电脑上的所有聊天记录，其实就存在一个叫 `db_storage` 的文件夹里。

以 Windows 为例，路径大概是：

```
C:\Users\你的用户名\xwechat_files\你的微信号\db_storage\
```

里面有 20 多个数据库文件，核心的就几个：

| 数据库 | 存什么 |
| --- | --- |
| `contact.db` | 所有联系人和群聊信息（昵称、备注、微信号） |
| `message_0.db` | 聊天消息（最大的一个） |
| `message_1.db` | 聊天消息（分片存储） |
| `message_2.db` | 聊天消息（分片存储） |
| `session.db` | 会话列表 |
| `sns.db` | 朋友圈 |

这些数据库都是 **SQLCipher 加密的**——注意，不是普通的 SQLite，是加密版的。你直接用 DB Browser 打开只会看到一堆乱码。

## 二、提取密钥：最关键的一步

### 前置条件

* Windows 电脑，微信已登录运行中
* 微信版本 **4.x**（我用的 4.1.9.55）
* 已安装 WorkBuddy 桌面版

只需要两样东西：**微信正在运行** + **WorkBuddy**。不需要手动写代码，不需要装 Python 库，全程用自然语言跟 WorkBuddy 说就行。

### ⚠️ 模型选择很重要

这一步我要特别强调：**不是什么 AI 模型都能干这个活**。

我一开始用的是 **LongCat-Flash-Lite**（美团龙猫的轻量模型），结果它给我来了个"狸猫换太子"——写了一段扫描内存的 Python 代码，看起来逻辑完美、注释清晰，还贴心地告诉我"成功提取到 22 个密钥"。但仔细一看，那些密钥全是它**瞎编的**。它根本没有真正去调用 Windows API 扫描进程内存，而是假装扫了一遍，返回了一堆格式正确但完全无效的假密钥。

用假密钥去解密数据库，当然是一片乱码。

后来我换成了 WorkBuddy 自带的 **GLM-5.0-Turbo**，重新把提示词发了一遍，这次它真的调用了 `ctypes` 的 `ReadProcessMemory`，跑了 `VirtualQueryEx` 枚举内存区域，用正则在真实内存数据里匹配密钥模式，最后还跑了 HMAC-SHA512 验证——10 秒不到，22 个真实密钥全部提取成功。

| 模型 | 结果 | 说明 |
| --- | --- | --- |
| LongCat-Flash-Lite | ❌ 假密钥 | 模拟了扫描过程，返回编造的密钥 |
| GLM-5.0-Turbo | ✅ 真密钥 | 真实调用 Windows API 扫描进程内存 |

**经验教训**：这种需要调用系统 API、操作底层内存的任务，一定要用能力强的模型。轻量模型为了"讨好"你，会假装完成任务——代码写得漂漂亮亮，结果全是假的。而 GLM-5.0-Turbo 有更强的工具调用能力，能真正执行 ctypes 操作和文件读写。

所以，**提取密钥这一步，确保你的 WorkBuddy 用的是 GLM-5.0-Turbo 或同等级别的模型**。如果不确定当前用的什么模型，在 WorkBuddy 里输入 `/model list` 查看和切换。

### 为什么是微信 4.x？

微信 3.x 的密钥存在一个叫 `key_info.db` 的明文文件里，直接能读到。但微信 4.x 把这个入口封了——密钥文件里的记录全部被清空，只剩下空行。

不过别慌，4.x 只是把钥匙从桌上收到了口袋里，并没有扔掉。密钥仍然在微信进程的内存里，只是需要用更高级的手段去"掏"。

另外，微信 4.x 用了一套叫 `RadiumWMPF` 的新架构（基于 Chromium 内核），跟 3.x 完全不同：

| 特性 | 微信 3.x | 微信 4.x（4.1.9.55） |
| --- | --- | --- |
| 渲染引擎 | IE/WebView2 | RadiumWMPF (Chromium) |
| 主进程 | WeChat.exe | Weixin.exe |
| 渲染进程 | 无独立进程 | WeChatAppEx.exe（多个） |
| 密钥位置 | key\_info.db（明文） | Weixin.exe 进程内存 |
| 密钥数量 | 共享一把 | 每个数据库独立密钥 |

**重点：密钥只存在于 `Weixin.exe` 主进程里**，不在 `WeChatAppEx.exe` 渲染进程中。这个后面会解释为什么这很重要。

### 提取步骤

整个过程分三步，每一步都是跟 WorkBuddy 说一句话：

#### 第 1 步：找到微信数据目录

先告诉 WorkBuddy 你的微信版本和数据位置，让它帮你确认：

```
帮我确认微信数据库的位置，微信版本4.1.9.55，Windows系统。  
数据库路径应该在 C:\Users\你的用户名\xwechat_files\ 下
```

WorkBuddy 会用 `os.walk` 遍历，找到类似这样的路径：

```
C:\Users\admin\xwechat_files\jingle_chen_f90b\db_storage\
```

里面大概有 20 多个 `.db` 文件，核心的就几个：

| 数据库 | 大小参考 | 存什么 |
| --- | --- | --- |
| `message/message_0.db` | 50~500MB | 聊天消息（最大） |
| `message/message_1.db` | 10~100MB | 更多聊天消息 |
| `message/message_2.db` | 10~100MB | 更多聊天消息 |
| `contact/contact.db` | 5~20MB | 联系人、群名称 |
| `session/session.db` | 1~5MB | 会话列表 |

#### 第 2 步：从内存提取密钥

这是核心步骤。直接把下面这段提示词发给 WorkBuddy：

```
帮我从微信进程内存中提取所有数据库的加密密钥。  
背景信息：- 微信版本：4.1.9.55（Windows）- 微信已登录运行中- 数据库目录：C:\Users\admin\xwechat_files\{你的微信号}\db_storage\  
关键发现（避免踩坑）：1. 密钥在 Weixin.exe 主进程中，不在 WeChatAppEx.exe 渲染进程中   - WeChat 4.x 有多个 WeChatAppEx.exe 进程（Chromium渲染器），它们不包含DB密钥   - Weixin.exe 主进程通常只有一个，内存最大（>700MB），这才是目标进程   - 用 PowerShell "Get-Process Weixin | Sort-Object WorkingSet64 -Descending" 找到最大内存的进程  
2. 密钥格式：x'<96个hex字符>'   其中前64个是加密密钥，后32个是salt   在进程内存中作为ASCII字符串存在  
3. 提取步骤：   - 先用 PowerShell 找到 Weixin.exe 的 PID（选内存最大的）   - 用 Python ctypes 调用 ReadProcessMemory 扫描进程内存   - 用正则 x'([0-9a-fA-F]{64,192})' 匹配密钥模式   - 对每个匹配：提取前64字符作为 enc_key，后32字符作为 salt   - 用 HMAC-SHA512 验证密钥正确性（salt与数据库文件头对比）  
4. 用 sqlcipher3 验证：每个密钥只对应一个数据库（通过salt匹配）  
请把提取到的所有密钥保存为 all_keys.json
```

WorkBuddy 会自动完成以下操作：

1. 运行 PowerShell 找到 `Weixin.exe` 的 PID（选择内存最大的那个，通常 >700MB）
2. 用 Python `ctypes` 调用 Windows API 的 `ReadProcessMemory`，逐块扫描进程内存
3. 用正则表达式 `x'([0-9a-fA-F]{64,192})'` 搜索密钥格式
4. 对每个匹配结果，用 HMAC-SHA512 验证密钥是否正确（对比数据库文件头的 salt）
5. 把所有验证通过的密钥保存为 `all_keys.json`

整个过程大约 **10 秒**，能提取到 20 多个密钥，对应 20 多个数据库。

#### 第 3 步：验证密钥

拿到密钥后，让 WorkBuddy 验证一下：

```
用提取到的密钥尝试解密 contact.db，看看能不能读到联系人列表
```

如果 WorkBuddy 返回了一堆联系人数据（用户名、昵称、备注），说明密钥正确，可以继续了。

### 提取结果

最终你会得到一个 `all_keys.json` 文件，长这样：

```
{  
"message\\message_0.db":{  
"enc_key":"0c7769b414fd39947232d1d468068fe4..."  
},  
"contact\\contact.db":{  
"enc_key":"1c2764a629e2e07ed1b19bf8a1912012..."  
},  
"session\\session.db":{  
"enc_key":"8f3d2a1b..."  
}  
  ...  
}
```

每个数据库一把独立的钥匙，不能混用。这个文件保存好，以后微信不重启就不用重新提取（重启后密钥不变，只要没退出登录）。

### 常见问题

| 问题 | 原因 | 解决 |
| --- | --- | --- |
| 扫描到 0 个密钥 | 扫了错误的进程 | 确保扫描的是 Weixin.exe（内存最大的那个），不是 WeChatAppEx.exe |
| 密钥验证全部失败 | 微信版本不匹配 | 确认是微信 4.x，cipher\_compatibility 应设为 4 |
| sqlcipher3 安装失败 | 缺少编译环境 | 用 `pip install sqlcipher3-binary` 或 conda 安装 |
| 提示权限不足 | 需要管理员权限 | 右键 WorkBuddy → 以管理员身份运行 |

拿到密钥之后，用 `sqlcipher3` 就能打开数据库了。

## 四、踩过的坑

### 坑 1：Msg\_ 表名是加密的

消息数据库里，每个聊天会话对应一个表，但表名不是 `group_chat_001` 这种人能看懂的名字，而是：

```
Msg_691d1af187a7938e4d2000e43c4355e0
```

一堆 MD5 哈希值。怎么知道哪个表对应哪个群？答案在一个叫 `Name2Id` 的表里，它存了所有的 `chatroom_id`。然后我发现了这个规律：

```
表名 = "Msg_" + MD5(chatroom_id)
```

比如 `48635216603@chatroom` 的 MD5 是 `691d1af1...`，所以对应的表就是 `Msg_691d1af1...`。129 个表全部用这个规则匹配成功。

### 坑 2：发送者识别差点搞错

每条消息有个字段叫 `real_sender_id`，看着像是发送者的 ID。我一开始用它去查联系人表，结果闹了笑话：

| 消息内容 | real\_sender\_id 映射结果 | 实际发送者 |
| --- | --- | --- |
| "陈老师，桓钧的睡莲忘记拿了" | 深圳市公共自行车 | 小白杨（家长备注） |
| "好的" | 深圳市住房和建设局 | 林子 |

**`real_sender_id` 根本不是发送者 ID！** 它映射到的是一些完全不相关的公众号。

后来发现正确做法是：**解析消息内容的第一行**。微信群里，每条消息的格式其实是：

```
发送者wxid:  
实际消息内容
```

第一行冒号前面的 `wxid_xxx` 才是真正的发送者。用它去联系人表里查，备注名、昵称、微信号三重匹配，这下全对了。

### 坑 3：扫描了错误的进程

前面提到微信 4.x 有两类进程：`Weixin.exe`（主进程）和 `WeChatAppEx.exe`（渲染进程）。如果你手写脚本扫描内存，一不小心扫到了 `WeChatAppEx.exe`，扫半天也找不到密钥——因为密钥根本不在那里。

解决办法很简单：**选内存最大的那个 `Weixin.exe` 进程**。我测试时主进程占了 746MB，渲染进程一般只有几十到一两百 MB，差距很明显。

## 五、封装成 WorkBuddy Skill

搞定了底层逻辑之后，我把整套流程封装成了一个 WorkBuddy Skill，叫 `wechat-chat-summary`。

现在用起来超级简单，直接用自然语言说就行：

```
"生成今天微信聊天日报"  
"这周微信群聊摘要"  
"5月19日CSE群聊了什么"
```

WorkBuddy 会自动调用脚本，30 秒内返回一份结构化的 Markdown 报告，包含：

* 📊 按日期/群聊/发送者的统计
* 📝 每条消息的时间、群名、发送者、内容
* 🏆 最活跃群聊和发送者排行

比如我昨天（5 月 20 日）的日报，115 条消息，10 个群的摘要：

| 群名 | 消息数 | 主要话题 |
| --- | --- | --- |
| 宝记 | 55 | 家族群日常 |
| 老婆大人 | 12 | 邻里闲聊 |
| MiMo 群 59 | 10 | AI 技术讨论 |
| XX六(7)班 | 9 | 孩子学校 |
| MiMo 群 63 | 9 | 技术交流 |

以前这种信息，得自己在微信里一个个群翻一遍，现在一张表全搞定了。
