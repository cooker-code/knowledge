# workBuddy

## 知识点入口

- 本模块先看宏观流程，再看文章：[知识地图](020506_核心知识点/知识地图.md)。
- 新文章必须先归入 workBuddy 的流程节点，再判断是补充、冲突、不同层次还是降权。
- `文章/` 只保留原文锚点，长期知识必须沉淀到 `020506_核心知识点/`。

## 技术定位

| 项 | 内容 |
|---|---|
| 技术名 | workBuddy |
| 一级类目 | Agent 与 AI 工程 |
| 二级类目 | AI 编程工具 |
| 技术本体 | 面向个人和工程工作流的 AI 助手/Agent 工具，重点是把模型、工具、记忆和自动化流程接入真实任务 |
| 全局架构位置 | 位于用户任务和本地/云端模型工具之间，负责组织任务输入、工具调用、上下文、记忆和产物交付 |
| 主要使用者 | 开发者、数据分析人员、知识工作者、个人自动化用户 |
| 主要产出 | 代码或文档产物、数据分析结果、任务摘要、自动化执行记录、可复用经验 |

## 归类边界

| 相邻方向 | 归类规则 |
|---|---|
| AI Coding 工作流 | 文章主问题是通用 AI 编程方法，而不是 workBuddy 产品能力时，归 AI Coding 工作流 |
| Memory Management | 文章主问题是通用记忆机制、长期记忆、知识生命周期时，归 Memory Management |
| 工具调用 / ComputerUse | 文章主问题是浏览器控制、外部工具调用或 MCP/Skill 机制时，归对应工具调用目录 |
| 数据分析与 BI | workBuddy 只是执行数据分析任务的工具，文章重点是指标、报表、ChatBI 或分析方法时，归数据分析与 BI |
| 个人成长 / 效率工具 | 只有主观效率体验、场景清单、无机制和边界时，不直接沉淀为技术知识点 |

## 排重准则

问题指纹：

```text
workBuddy 功能层 + 任务场景 + 上下文/工具/记忆机制 + 解决问题 + 验证或权限边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是场景清单 | 没有机制、失败案例或验证方式时，只保留原文锚点 |
| 都是基础设置 | 比较是否新增权限、上下文、记忆、工具配置边界 |
| 都是数据分析场景 | 判断主问题是 workBuddy 自动化能力，还是数据分析方法 |
| 都是长任务/工作流 | 比较状态保存、恢复、人工接管和结果验收是否有新增信息 |
| 只有“更省时间、更聪明” | 没有证据和适用边界时降为略读 |
