这次不是“大版本重构”，但每一项更新都很贴近日常使用场景：**更灵活的工作区切换、更强的钩子控制、更实用的插件能力，以及更顺手的自动化入口**。

如果你平时就在 Claude Code 里处理多分支任务、维护插件，或者依赖自动化工作流，这次更新值得重点关注。

## 一句话看懂这次更新

**2.1.105 的核心关键词只有四个：Worktree 更灵活、Hook 更可控、插件更能跑、自动化入口更顺手。**

## 核心亮点

### 1. `EnterWorktree` 新增 `path` 参数：切换工作区更自由

官方为 `EnterWorktree` 工具新增了 `path` 参数，意味着你可以更灵活地指定 worktree 位置。

这对经常并行处理多个任务的人很实用：

* 工作区位置更清晰
* 多任务切换更直观
* 更适合复杂项目里的分支隔离

**对开发者的直接价值：** 把“进入哪个工作区”这件事从固定模式里解放出来，工具链更贴合真实工作习惯。

### 2. 新增 PreCompact Hook 支持：压缩前也能拦截

这次更新加入了 **PreCompact hook** 支持，hooks 现在可以在 compaction 之前介入，甚至阻止压缩执行。

这件事的重要性在于：

* 对上下文压缩有了更细的控制权
* 可以在关键场景下避免不合时宜的压缩
* 更适合构建复杂的自动化策略

**对开发者的直接价值：** 让 Claude Code 的上下文管理不再只是“系统自动做”，而是可以被你的规则接管。

### 3. 插件支持后台监控：`monitors` 让插件更“常驻”

官方为插件新增了 `monitors` manifest key，用于支持后台监控能力。

这意味着插件不只是被动响应，还可以：

* 持续观察状态变化
* 在后台执行监控逻辑
* 承担更多常驻型自动化任务

**对开发者的直接价值：** 插件从“工具扩展”进一步走向“工作流组件”，能力边界明显更大了。

### 4. `/proactive` 成为 `/loop` 的别名：自动化入口更好记

官方把 `/proactive` 作为 `/loop` 的别名加入进来。

虽然这是个小改动，但它降低了使用门槛：

* 命令更好理解
* 入口更符合“主动执行”的语义
* 对新用户更友好

**对开发者的直接价值：** 把原本偏工具化的命令，变成更贴近工作意图的表达。

## 这次更新最值得关注的地方

如果只挑一个结论，那就是：

**Claude Code 2.1.105 在继续强化它的“代理式工作流”能力。**

它不是在堆表面功能，而是在补齐三个关键环节：

* **工作区管理**：更灵活
* **上下文控制**：更可控
* **插件自动化**：更可持续

这三点组合起来，意味着 Claude Code 更像一个可以深度嵌入开发流程的协作平台，而不仅仅是一个聊天式助手。

## 适合公众号的简短结论

Claude Code 2.1.105 这次更新很克制，但方向非常明确：**让开发者在 worktree、hook、插件和自动化入口上拥有更强的控制力。**

对于经常使用 Claude Code 的团队来说，这些升级虽然不炸裂，但都很实用，而且是会真正影响效率的那种。
