最近。

我越来越强烈地感觉到一件事。

**AI 编程这条路，开始变了。**

以前大家最爱聊的，是谁最强。

Opus 强不强。
Codex 猛不猛。
Gemini CLI 能不能打。

但现在，真正值得聊的，已经不是“单模型谁赢”。

而是：

**不同模型，怎么配合。**

而在这件事上。

**Claude Code + Codex**，我愿称之为当前最狠的一组搭配。

不是因为它们谁绝对无敌。

而是因为这套组合，终于开始有点“真正工程工作流”的味道了。

---

## 先给结论

如果你本来就在用 Claude Code。

那 Codex，值得接。

而且不是“有空再看”的那种值得。

是**现在就可以装上试试**的那种值得。

原因很简单。

第一，**它能帮你分摊成本。**

第二，**它能帮你做第二轮审查。**

第三，也是最关键的一点。

**它终于让 Claude Code 这套工作流，不用再老是自己写，自己审，自己给自己打分了。**

兄弟们。

这个区别，真的很大。

---

## 为什么这事很重要

AI 编程到今天，最大的问题之一，不是不会写。

而是**太敢写了。**

一个模型。

自己规划。
自己生成。
自己修。
最后再自己判断“我写得挺好”。

这事，听着就不太对。

你总不能让一个学生。

自己出卷。
自己答题。
再自己判卷。

而很多 AI coding workflow，过去本质上就是这么干的。

所以问题就来了。

很多代码，表面看起来没毛病。

能跑。
能编译。
甚至功能都能点通。

但一旦上强度。

一旦进生产。

坑，就开始往外冒。

这种时候，你最缺的，不是“再来一个更会写代码的模型”。

你缺的是：

**一个会找茬的模型。**

而 Codex，正好能补这一刀。

## 这套组合，真正值钱的地方在哪

我觉得，最值钱的地方，就两件事。

### 1. Opus 负责想，Codex 负责干

这是一种非常自然的分工。

复杂任务里，最贵的部分，往往不是把代码敲出来。

而是：

* 怎么拆任务
* 怎么定架构
* 怎么判断方向
* 怎么选实现方式

这些事，交给 Opus 这种强模型，很合理。

但后面的执行。

比如：

* 补实现
* 推细节
* 跑一些具体生成
* 承接更多执行型工作

这部分，Codex 完全可以接过去。

这样一来，整个工作流会更顺。

也更省，说白了就是。

**贵的模型，干贵的活。**

**便宜一点的模型，干重体力活。**

---

### 2. Codex 最狠的，不是执行，是审查

这个我真的想单独拎出来讲。

很多人一听 Codex 接进 Claude Code，第一反应是：

哦，又多了一个模型能跑代码。

不是。

真正狠的，是它那个思路。

**Adversarial Review.**

翻成人话就是：

**别对这段代码太客气。**

**默认它有坑。**

**带着敌意去审。**

这个思路，我特别喜欢。

因为大模型写代码最大的幻觉之一，就是：

**它很容易对自己太宽容。**

但一到“自查”这一步，很多时候就开始放水了。

而对抗性审查的意思是。

我不信你。

我先默认你写砸了。

---

## 这种审查，到底在盯什么

我看了一下这套逻辑，最让我有感觉的，是它盯的不是表面问题。

它重点盯的是那些，最容易在真实项目里搞死你的东西。

比如：

* 认证
* 数据丢失
* 回滚
* 竞态条件
* 依赖降级
* 版本偏差
* 可观测性缺口

你看这几个词，就知道它不是在帮你挑格式问题。

也不是在提醒你变量名不优雅。

它盯的是那种：

**本地一切正常。**

**线上开始出血。**

**出了问题还很难排查。**

这种坑，才最贵。

也最烦。

而这种问题，恰恰是同一个模型自己审自己时，最容易漏掉的。

因为它写的时候，天然就会带一种路径依赖。

它容易觉得：

“我当时这么写，是有道理的。”

但第二个模型不会。

第二个模型上来就会想：

“你这玩意儿，八成有坑。”

这就是价值。

---

## 我为什么会觉得这套组合特别有前途

因为这已经不是单纯的“多一个模型入口”了。

这更像是，Claude Code 开始从一个“会写代码的 AI 工具”，往“工作流平台”走。

这两个东西，看上去差不多。

其实差很多。

前者的逻辑是：

* 我接了什么模型
* 我自动化程度有多高
* 我写代码有多快

后者的逻辑是：

* 谁负责规划
* 谁负责执行
* 谁负责审查
* 谁负责交叉复核
* 谁负责在最后兜底

这就是两个层级的事情了。

而我越来越相信。

AI 编程接下来的分水岭，不是谁 benchmark 再多 3 个点。

而是谁先把**多模型协作工作流**这件事，搭明白。

---

## 这对普通开发者，意味着什么

如果你是普通开发者。

或者是现在已经在用 Claude Code 的人。

这事的意义，其实非常现实。

### 1. 你终于有了便宜一点的第二大脑

很多人不是不用强模型。

而是用不起一直全程高强度地用。

现在有了 Codex 这个搭子之后。

你至少可以开始做分工。

不是所有步骤，都得让最贵的模型从头扛到尾。

### 2. 代码审查这件事，开始更像真的“多人 Review”了

这个特别重要。

因为 AI 编程最容易翻车的地方，往往不是第一版生成。

而是你以为它已经差不多了。

结果一上生产，问题才开始显形。

这时候，如果能让第二个模型来做交叉复核。

价值非常高。

### 3. “让 AI 审 AI” 这件事，终于靠谱一点了

过去很多自动化工作流，其实有一个底层漏洞。

就是同一个模型，干了太多角色。

现在，至少有机会把这些角色拆开。

哪怕还不完美。

哪怕不同模型也会各自漏东西。

但只要你不再让一个模型，从头到尾闭环自嗨。

整个系统就已经更稳了。

---

## Opus 和 Codex，到底谁更强

说实话。

我觉得这个问题，没那么重要了。

至少在这件事上，不重要。

因为现实里，很多项目根本不需要你选边站。

真正有效的做法，往往不是：

* 只用 Opus
* 或者只用 Codex

而是：

**让它们在不同位置，干最适合自己的活。**

有些问题，Opus 看得更全。

有些问题，Codex 看得更毒。

有些时候，Opus 更适合定框架。

有些时候，Codex 更适合做外部视角的复核。

所以重点根本不是谁把谁替代了。

重点是：

**你终于有机会，别再让同一个模型，既当运动员，又当裁判。**

---

## 这套东西，最适合谁

如果你问我，谁最该马上试这套组合。

我觉得有三类人。

### 第一类，是重度 Claude Code 用户

你本来就在里面干活。

那现在多一个 Codex，不管是做执行分流，还是做对抗性审查，收益都很直接。

### 第二类，是对成本特别敏感的人

很多人不是不想上强模型。

而是预算卡得死。

这时候，能把规划和执行拆开，能把不同模型按工种分配，性价比会舒服很多。

### 第三类，是已经开始做真实项目的人

如果你做的不是 demo。

不是 landing page.

而是已经有：

* 状态管理
* 数据写入
* 定时任务
* 第三方集成
* 数据库
* 推送
* 后台面板

这些东西。

那你就会非常清楚。

最吓人的 bug，从来都不是代码写不出来。

而是**写得出来，看着也能跑，结果埋了雷。**

这种时候，第二个模型来交叉审查，真的很值。

---

## 最后

我最近最大的感受是。

AI 编程这件事，正在从“找最强模型”，慢慢走向“搭最强工作流”。

这两个阶段，完全不是一回事。

前一个阶段，大家比的是谁更聪明。

后一个阶段，大家比的是谁更会协作。

谁负责想。

谁负责干。

谁负责找茬。

谁负责最后兜底。

而 **Claude Code + Codex**，现在看，就是这条路上一个非常有代表性的组合。

一个负责把上限拉高。

一个负责把成本打下来。

一个负责生成。

一个负责挑刺。

它未必是最终答案。

但方向，已经非常清楚了。

**AI 编程，不再只是拼单模型。**

**而是在拼，谁能把不同模型，组织成一支真正会打仗的队伍。**
