## 先说结论

用 Claude Code 写应用，现在已经能做到 100% 编程了。但真正的效率分水岭，不在于“能不能写出来”，而在于“跑通一次之后，能不能让下一次直接到位”。

这个分水岭的关键，就是**快速形成规则和技能**。

## 小步慢跑 vs 大步流星

刚开始用 Claude Code 写应用时，你会发现自己在做这些事：

* 一点点调流程
* 一个个改 bug
* 反复跟 CC 说“不对，应该这样”
* 来回确认部署细节
* 重复解释同一个业务逻辑

这是正常的。这叫**小步慢跑**。

但如果每次开发相似功能都要重复这个过程，那就太慢了。

真正的效率提升，来自于**跑通一次之后，立刻整理出规则和技能**。下次遇到类似场景，直接大步流星。

## 什么是“规则”和“技能”

简单说：

* **规则(rules)**：编码和部署的规范。告诉 CC“在这个项目里，我们就是这么干的”。
* **技能(skills)**：具体工作的操作细节。告诉 CC“遇到这类任务，按这个流程走”。

举个例子：

你用 CC 开发了一个 API 接口，跑通了。这时候你应该立刻整理：

**规则层面**：

* API 路由统一放在 `routes/` 目录
* 参数校验用 Zod
* 错误处理统一返回 `{ success, data, error }` 格式
* 数据库操作必须加事务

**技能层面**：

* 新增 API 的完整流程：定义路由 → 写 schema → 实现 handler → 写测试 → 更新文档
* 部署检查清单：环境变量、数据库迁移、依赖安装、服务重启
* 常见 bug 排查步骤：先看日志、再查数据库、最后检查配置

有了这些，下次开发类似接口，你只需要告诉 CC：“按照我们的 API 规则，新增一个用户注册接口”。CC 就能一步到位，不用你再一点点调。

## 为什么要“快速”形成

关键在于**趁热打铁**。

刚跑通一个流程时，你脑子里最清楚：

* 哪些地方容易出错
* 哪些细节容易遗漏
* 哪些步骤必须按顺序来
* 哪些配置必须对齐

这时候不整理，过两天就忘了。下次遇到类似问题，又要重新踩坑。

更重要的是，**规则和技能是会积累的**。

第一次跑通 API 开发，整理出规则。
第二次跑通前端组件，再整理一套规则。
第三次跑通部署流程，又是一套规则。

三个月后，你的项目就有了一套完整的开发规范体系。这时候 CC 就像一个熟悉你项目的老员工，不用你反复解释，直接就能干活。

## 怎么整理规则和技能

### 1. CLAUDE.md：项目的总纲

在项目根目录创建 `CLAUDE.md`，记录项目目标和整体架构。

这是 CC 理解你项目的第一份文档。

结构建议：

> # 项目概述
>
> ## 目标
>
> 这是一个用户管理系统，提供注册、登录、权限管理功能。
>
> ## 技术栈
>
> * 后端：Node.js + Express + PostgreSQL
> * 前端：React + TypeScript
> * 部署：Docker + Nginx
>
> ## 核心架构
>
> * 三层架构：routes → services → models
> * 认证方式：JWT
> * 数据库：主从分离，读写分离
>
> ## 关键设计决策
>
> * 为什么用 PostgreSQL：需要事务和复杂查询
> * 为什么用 JWT：无状态，方便扩展
> * 为什么三层架构：业务逻辑和数据访问分离

**关键原则**：

* 说清楚项目是什么、为什么这么设计
* 让 CC 理解整体架构和技术选型
* 记录重要的设计决策和权衡

### 2. /rules 目录：具体的代码和部署规范

创建 `rules/` 目录，分类记录具体规范。

**rules/coding.md** - 编码规范：

> # 编码规范
>
> ## 命名
>
> * 文件名：kebab-case (user-service.ts)
> * 类名：PascalCase (UserService)
> * 函数/变量：camelCase (getUserById)
>
> ## 代码组织
>
> * 一个文件只做一件事
> * 函数不超过 50 行
> * 复杂逻辑必须加注释
>
> ## 错误处理
>
> * 统一格式：{ success: boolean, data?: any, error?: string }
> * 异步操作必须 try-catch
> * 数据库操作必须加事务
>
> ## 参数校验
>
> * 使用 Zod 定义 schema
> * API 入口必须校验
> * 校验失败返回 400

**rules/deployment.md** - 部署规范：

> # 部署规范
>
> ## 环境变量
>
> * 必须在 .env.example 声明所有变量
> * 生产环境变量不能提交到代码库
> * 敏感信息用密钥管理服务
>
> ## 数据库
>
> * 迁移文件放在 migrations/
> * 每次部署前先跑迁移
> * 回滚脚本必须测试过
>
> ## 依赖管理
>
> * 锁定版本，不用 ^
> * 定期更新安全补丁
> * 新增依赖必须评审
>
> ## 部署流程
>
> 1. 运行测试
> 2. 构建镜像
> 3. 数据库迁移
> 4. 灰度发布
> 5. 全量发布

**关键原则**：

* 规范要具体、可执行
* 每条规范都要说明“为什么”
* 规范要能直接指导 CC 写代码

### 3. /skills 目录：具体工作的操作流程

创建 `skills/` 目录，记录具体任务的完整操作流程。

比如 `skills/api-development.md`：

> # API 开发流程
>
> ## 适用场景
>
> 开发新的 RESTful API 接口
>
> ## 完整流程
>
> 1. 在 src/routes/ 创建路由文件
> 2. 定义请求/响应 schema (Zod)
> 3. 实现 handler 函数
> 4. 添加错误处理
> 5. 写单元测试
> 6. 更新 API 文档
>
> ## 常见问题
>
> * 忘记加参数校验 → 必须先定义 schema
> * 错误没有统一格式 → 使用 errorHandler 中间件
> * 测试覆盖不全 → 至少覆盖正常流程和主要错误场景

**关键原则**：

* 一个技能解决一类问题
* 流程要完整，不要跳步骤
* 记录常见坑和解决方案

### 4. 整理时机：跑通就整理

不要等“以后有时间再整理”。以后永远没时间。

**最佳实践**：

* 项目开始 → 先写 CLAUDE.md 说清楚目标和架构
* 跑通一个功能 → 立刻更新 rules/ 中的相关规范
* 解决一个难题 → 立刻记录到 skills/ 对应文件
* 发现一个坑 → 立刻补充到常见问题

这 10 分钟的投入，下次能省 1 小时。

## 规则和技能的进化

规则和技能不是一次性写完的，而是**持续进化**的。

**第一版**：跑通一个流程，记录基本规范。
**第二版**：遇到新问题，补充边界情况。
**第三版**：发现更好的做法，更新最佳实践。

比如 API 开发规则：

* 第一版：定义基本的目录结构和错误格式
* 第二版：补充参数校验和事务处理
* 第三版：加入性能优化和安全检查

每次更新，都让规则更完善，让 CC 更懂你的项目。

## 从小步慢跑到大步流星

有了规则和技能之后，开发流程会变成这样：

**以前**：

* 你：“帮我写一个用户注册接口”
* CC：写了一版
* 你：“错误格式不对，改成 { success, data, error }”
* CC：改了
* 你：“参数校验要用 Zod”
* CC：又改了
* 你：“还要加事务处理”
* CC：再改……

**现在**：

* 你：“按照 CLAUDE.md 的架构和 rules/coding.md 的规范，参考 skills/api-development.md 流程，写一个用户注册接口”
* CC：一次到位，包含所有规范和最佳实践

这就是**大步流星**。

## 更重要的是什么

规则和技能的价值，不只在于“让 CC 更懂你”，还在于：

1. **降低认知负担**：不用每次都从头解释
2. **保证一致性**：整个项目的代码风格和结构统一
3. **积累经验**：把踩过的坑变成可复用的知识
4. **加速迭代**：相似功能的开发速度指数级提升

从整体来看，这是一个**正反馈循环**：

跑通流程 → 整理规则 → 下次更快 → 跑通更多流程 → 规则更完善 → 速度更快

## 下一步该怎么做

如果你现在正在用 Claude Code 开发项目，建议：

1. **先写 CLAUDE.md**：说清楚项目目标、技术栈、核心架构
2. **创建 rules/ 目录**：分别记录编码规范(coding.md)和部署规范(deployment.md)
3. **创建 skills/ 目录**：把已经跑通的流程整理成技能文件
4. **养成习惯**：每次跑通新功能，花 10 分钟更新对应文件
5. **持续优化**：遇到问题就补充，发现更好的做法就更新

不要等“项目稳定了再整理”。现在就开始，边开发边整理。

## 以上

用 Claude Code 写应用，100% 编程只是起点。

真正的效率提升，来自于**把每一次成功的经验，快速沉淀成可复用的规则和技能**。

小步慢跑是学习阶段，大步流星才是工作状态。

而从小步到大步的关键，就是**跑通一次，规则化一次**。
