Task 系统是 Claude Code 内置的任务管理机制。(注意不要和命令/tasks混淆了，这是两个东西)

当你触发它时，Claude Code 会在内部把复杂任务拆解成结构化的工作单元，追踪每个任务的执行状态，按照依赖关系决定执行顺序。

它不是一个手工维护的外部看板系统，Claude Code 提供了内置 task list：你可以按 **`Ctrl+T`** 查看，也可以直接让 Claude 展示全部任务。你还是用自然语言和 Claude Code 对话——Task 系统是它在后台运作的方式。

你可以理解为：你是项目经理，Claude Code 是工程师，Task 系统是它内部的工单追踪系统。你不需要直接操作工单，但你可以通过对话随时查看任务状态、了解进度、调整计划。

---

## 怎么触发 Task 系统

Task 系统不需要用户手动进入。在复杂、多步骤工作中，Claude Code 会按需创建 task list；大多数情况下，显式要求‘先拆解任务再执行’，通常能让它更早、更清晰地进入任务化流程

**方法一：直接在提示词里要求**

这是最简单最可靠的方式：

> "把这个需求拆解成独立任务，建立执行计划，再开始写代码。"
>
> "先列出任务清单，设置好依赖关系，然后按顺序执行。"
>
> "用任务管理的方式来处理这个功能，我想看到执行进度。"

这类提示词明确告诉 Claude Code：你期望它结构化地管理这次工作，而不是直接开始写。

**方法二：在 CLAUDE.md 里配置**

如果你希望所有复杂任务都自动使用 Task 系统，可以在项目的 `CLAUDE.md` 里加上规则：

```
## 任务执行规范

对于包含 3 个以上步骤的需求，必须先创建任务清单，设置依赖关系，
然后按顺序执行，每完成一个任务立即更新状态。
```

把这类规则写进`CLAUDE.md`后，Claude 更可能稳定地采用任务化流程；但`CLAUDE.md`属于上下文指令，不是强制开关，所以最好还是在终端里面显式提示一句。

**方法三：在复杂任务里要求中途切换**

有时候你一开始没想到，做到一半发现事情比预期复杂。这时候也可以随时告诉它：

> "这个任务比我想的复杂，先停下来，把剩下的工作拆解成任务，再继续。"

除了自然语言提示，最实用的可见操作有两个：按`Ctrl+T`查看任务列表，以及直接对 Claude 说 ‘show me all tasks’ / ‘clear all tasks’。任务还会在 context compaction 后保留，必要时也能通过 `CLAUDE_CODE_TASK_LIST_ID`跨会话共享

## Task 系统在背后做什么

理解了怎么触发，再来看它具体在做什么。

Claude Code 内部有四个任务管理工具，当你引导它使用 Task 系统时，它会在背后调用这些工具：

**创建任务（TaskCreate）：**把每个工作单元创建为独立的任务，包含标题和详细描述

**查看任务列表（TaskList）：**随时检查所有任务的当前状态——哪些完成了，哪些在进行，哪些被阻塞

**更新任务状态（TaskUpdate）：**开始执行时标记"进行中"，完成时标记"已完成"，设置任务间的依赖关系

**查看任务详情（TaskGet）：**在执行某个任务前，读取完整的任务描述确认目标

你可以在对话里随时问它：

> "现在任务进度怎么样了？"
>
> "第三个任务做完了吗？"
>
> "现在哪些任务被阻塞了？"

Claude Code 会查看内部任务状态，然后用自然语言回答你。

## 实战：用 Task 系统实现用户认证

假设我们要实现一个功能：给项目加上用户认证，包括注册、登录、权限校验、token 刷新。

那么你可以跟Claude Code说：

> "帮我实现用户认证功能，包括注册、登录、权限校验、token 刷新。先把这个需求拆解成独立任务，定义好执行顺序，再开始写代码。"

Claude Code 开始工作后，内部首先进行任务拆解：

```
任务 #1：分析现有代码结构
任务 #2：设计数据库 schema [等待 #1]
任务 #3：实现注册接口 [等待 #2]
任务 #4：实现登录接口 [等待 #2]
任务 #5：实现 token 刷新 [等待 #4]
任务 #6：实现权限中间件 [等待 #3, #4]
任务 #7：编写集成测试 [等待 #3, #4, #5, #6]
```

然后开始按顺序执行，每完成一个任务立即更新状态，并开始下一个。

**中途你可以随时插话：**

> "登录接口做完了吗？"

Claude Code 会查看任务状态，回答你：

> "#4 登录接口已完成。token 刷新（#5）正在进行中。"

**如果需要调整，**直接告诉它：

> "token 刷新这个任务先跳过，先把权限中间件做完。"

它会根据你的指令调整执行顺序。

## 任务依赖的三种模式

理解依赖关系的设计，能帮你更好地引导 Claude Code 规划任务。

### 线性流水线

```
任务 1 → 任务 2 → 任务 3 → 任务 4
```

严格按顺序，前一个完成才能开始下一个。适合有明确先后顺序的场景：比如分析需求 → 设计方案 → 写代码 → 写测试。

### 并行任务池

```
任务 1
任务 2  （互相独立，可以同时处理）
任务 3
```

没有依赖关系，理论上可以并行。如果你在用多 Agent 模式，这些任务可以同时进行。单个 Claude Code 会顺序处理，但调度更灵活。

### 菱形依赖

```
        任务 1（基础工作）
       /             \
   任务 2          任务 3   （并行子任务）
       \             /
        任务 4（汇总）
```

先做基础，再并行处理多个子任务，最后汇总。这是实际项目里最常见的模式。上面的用户认证案例就是这种结构的变体。

## 最佳实践

### 怎么描述任务才清晰

当 Claude Code 在内部创建任务时，任务描述的质量决定了后续执行的准确性。

你可以在提示词里要求它描述得更详细，或者在对话里告诉它验收标准是什么：

> "注册接口的验收标准：注册成功返回 201，邮箱已存在返回 409，格式错误返回 400。"

给了清晰的验收标准，任务执行完后它会主动验证，而不是"差不多了"就算完成。

### 任务粒度：不要太大，也不要太小

**太大：**"实现整个用户系统" — 这是项目，不是任务，无法有效追踪

**太小：**"导入 bcrypt 库"、"添加 email 字段" — 管理成本超过执行成本

**合适：**一个可以独立验证的功能单元，完成后你能明确说"这个功能做好了"

通常，一个任务对应 5-30 分钟的执行量比较合适。

### 何时用 Task 系统，何时不用

**适合用：**

* 需求涉及 3 个以上步骤
* 步骤之间有明确的先后顺序
* 你想追踪执行进度
* 多个 Agent 协作

**不适合用：**

* 简单的单步操作（"解释这段代码"）
* 一次性的快速修改（"把这个变量名改一下"）
* 探索性的问答（"这个架构有什么问题"）

Task 系统有管理成本。任务本身很简单时，引入 Task 系统反而增加不必要的复杂度。

---

### 和 Plan Mode 配合使用

Task 系统和 Plan Mode 是互补的：

* **Plan Mode：**

  负责规划，分析需求，设计方案，确认方向——在动手之前
* **Task 系统：**

  负责执行，拆解任务，追踪进度，管理依赖——在执行过程中

典型的工作流：

```
Plan Mode（想清楚做什么）→ 生成计划 → Task 系统（把计划拆解执行）→ 追踪进度
```

对于复杂功能，先用 Plan Mode 确认方向，再用 Task 系统来执行，是最稳的组合。

---

## 核心要点

Task 系统的价值不是让 AI 更聪明，而是让你对 AI 的工作更有掌控力。

**你需要记住的：**

1. **Task 系统大多数情况下不会自动启动。**

   要明确告诉 Claude Code 用任务结构来管理这次工作。
2. **你用自然语言交互。**

   任务的创建、更新、查询，都是 Claude Code 在背后完成的，你不需要直接操作工具。
3. **可以随时查看进度。**

   直接问它"任务现在进行到哪了"，它会给你回答。
4. **任务描述越清晰，执行越准确。**

   在提示词里提供验收标准，能大幅减少返工。

**最实用的触发方式：**

> "先把这个需求拆解成独立任务，建立好依赖关系，然后按顺序执行。"

就这一句，能让 Claude Code 从"一口气往前冲"变成"有条不紊地推进"。
