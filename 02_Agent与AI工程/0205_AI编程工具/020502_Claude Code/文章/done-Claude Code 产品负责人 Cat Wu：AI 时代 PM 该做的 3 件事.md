Claude Code 的产品负责人 Cat Wu 上周上了 Lenny 的播客，整整一个多小时讲他们怎么干活的，我听完最大的感受——人家已经在玩另一个游戏了

我抽出三件事跟你聊

## 一、那个 6 个月的 roadmap，已经死了

这种事你不陌生：年初定全年 OKR，季度初定季度路线图，月初定月度排期，每一层往下拍，到最底下的工程师手里时，已经是被反复“对齐”过的 v17 版本

Cat 讲得很直白：以前你能做长 roadmap，前提是写代码很贵、模型能力变化也慢，所以提前规划是有价值的，现在两个前提同时没了——写代码不再是瓶颈，模型能力每 3-6 个月跳一档，6 个月前规划的功能，等做完时已经不需要了

她举的例子最戳：早期 Claude Code 里他们专门做了一个 to-do list 功能，因为模型不会自己记得改完所有调用点，等 Opus 4 出来，模型自己就会做了，那个 to-do list 反而成了多余的东西

在 AI 时代，你越是认真规划 6 个月后的功能，你就越可能是在做无用功

Anthropic 的应对叫 Research Preview——他们和用户之间的那个“承诺契约”被重新谈了一遍

旧时代的契约是：“我们说要发的东西必须发，发了的东西必须达到 GA 质量”，新时代的契约是：“这是个实验，可能两周后就改主意了”

契约一变，整个内部流程都被解放了——工程师周末写完一个想法可以直接上，不用过 GTM；PM 不用再花一半时间“管理预期”；用户也心甘情愿当反馈源，因为人家本来就告诉你这是早期

旧 PM 一半时间在管理“我们承诺了什么”，新 PM 一半时间在管理“怎么让承诺更模糊”

## 二、PM 现在不是去开会的人，是让别人不用开会的人

Cat 在播客里描述了一个具体场景：Anthropic 的一个工程师在 Twitter 上看到用户反馈，周末写完功能，周一上线，全程没 PM 什么事

PM 没被取消，是工作内容变了，Cat 说她现在每天的活是这三件

第一件：定团队原则，不写 PRD

写一份很短的清单：核心用户是谁、为什么是他们、什么场景必须开箱即用，让团队任何人面对决策时都能自己拍板，不用回来问 PM

过去 PM 是这个“决策源头”，什么事都得来找他确认，这一条做法把这个源头拆掉了——团队任何人都能自己拍板

第二件：搭一条流水线叫 Launch Room

工程师觉得功能能发了，把它贴进这个常驻“发布间”，第二天文档、PMM、DevRel 全部自动接进来，第三天对外发布

PM 不在这条链路上跑，PM 在搭这条链路

第三件：每周整团队对一次指标

不是为了 KPI 考核，是为了让所有人对业务有同样深的理解，这样下次决策时，任何人都能直接拍板，不用跑回来问“我们这季度的 north star 是啥来着”

三件事的共同点：PM 这个人，从协作链路里消失了，PM 做的是让协作不再需要 PM 这个节点

公司里那个一天跑五个会、十个对齐 IM、二十封邮件抄送列表里的 PM，在 Anthropic 的语境里是低效的，他越忙，说明他搭的基础设施越烂

## 三、最值钱的人，是那种“既能想又能做”的工程师

Anthropic 招人的偏好他们自己都不藏：有 product taste 的工程师

Cat 说得很直白——当一个人既能判断做什么、又能直接动手实现，沟通成本就被消灭了，代码越来越便宜，“决定写什么”才是稀缺品

这件事重新定义了三个角色：

| 角色 | 旧定义 | 新定义 |
| --- | --- | --- |
| 工程师 | 实现 PRD 里的需求 | 自己判断做什么 + 直接实现 |
| PM | 写 PRD + 协调资源 | 搭流水线 + 把“想法到用户”的时间压到最短 |
| 设计师 | 独立设计环节 | 介入工程师早期，不做独立交付物 |

注意一个关键判断——Anthropic 没把 PM 这个角色取消，而是把 PM 的能力打散，让每个人都长一点，工程师长 product taste，PM 长技术理解，设计师长用户判断，每个人向中心融合，但角色没消失

最让我有体感的是 Cat 接下来这句话：以前 PM 工作里很大一块是“和合作团队对齐多季度路线图”，因为代码贵、决策成本高，现在多季度计划没意义了，PM 的时间就被释放出来——转去做 product taste、写 evals、看用户信号、判断模型能力的边界

靠“流程能力”立足的 PM，最熟悉的工作正在被淘汰，靠“判断能力”立足的 PM，杠杆反而被放大

你品一下你身边的 PM 各自是哪一种

## 一个特别值得偷的设计：双核分工，按时间切

最后讲一个我觉得最值得直接抄的东西——Cat 和她搭档 Boris 的分工

传统组织里，一条产品线配一个产品负责人 + 一个技术负责人，按“职能”切：你管产品决策，我管技术实现

Cat 和 Boris 不是这样：

* Boris 看的是 3-6 个月之后的世界长什么样、模型能力会跳到哪一档、那个时候产品形态应该是什么
* Cat 看的是从今天到 3-6 个月之间这条路怎么走通——营销、销售、财务、容量、跨团队对齐全部捋顺

他们是按“时间尺度”切的，不是按“职能”切的

这个分工方式我看完琢磨了好几天——它在 AI 时代特别合理：远期愿景需要的是技术远视和大判断，根本不需要操心落地，近期落地需要的是协同打通和细节耐心，根本不需要重新构想未来

两个尺度的工作模式完全不同，逼一个人同时干两件事，结果一定是两头都不到位——白天被周会吃掉，晚上想愿景已经累得不会想了

如果你正好在带一个 AI 团队或一条 AI 产品线，下次回看组织设计的时候，把“PM + 技术 leader”换成“3 个月外 + 3 个月内”试试，我赌它比传统配法管用得多

最后

我觉得自己看到了未来——产品工作未来该是什么样、那些可能性长什么样，Cat 这一个半小时全摊开在我面前

可一回到身边的现实，我就卡住了，身边大部分人连科学上网都还没搞明白

让我焦虑的不是 AI 会替代谁，是这种巨大的方差
