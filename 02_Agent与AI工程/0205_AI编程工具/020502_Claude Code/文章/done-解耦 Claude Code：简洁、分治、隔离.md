工具与执行

### s01Agent循环

这个是整个 agent loop 的基础地基，像是一个递归的循环，Deep Research 也是这种理念，第一个 search query 没搜到，那么llm 就自动改写 query 再去调用搜索工具去执行搜索，搜索结果返回给 llm，llm 这个时候说，我靠还是没搜到结果，那咋办，快点，llm 我再给你改写 query 再去给我调用搜索插件去执行搜索，那要还是搜不到呢，继续反馈 llm-llm 改写-llm 调用-反馈……，循环往复的一个循环中，直到搜到了目标的结果或者循环次数达到了这个任务设定的次数限制，因为如果不设限制，那么可能搜几十次、几百次、几千次，token 和搜索工具调用量是扛不住的，"until"这个词的具象化了。

在 claude code 里面工程化的东西是，user prompt 的指令来了，llm 意识到要去调用搜索工具，，搜索 tools 调用完了，llm 会获得result来思考，触发stop\_reason，这个你就可以认为是 llm 在思考的一个状态，结果是 ok，那就不用再搜了直接break out输出结果，如果思考的结果是 no，那么就再去调用 tool 再去执行。

### s02工具

工具tool 看起来和我们以往看的那些 api 接口、mcp等等都差不多，但 claude code 这里给出的是四个底层的 tool，分别是 bash、read\_file、write\_file 和 edit\_file，覆盖了大约 95% 的编程任务。

* • Bash 处理执行和任意命令；
* • read\_file 提供带行号的精确文件读取；
* • write\_file 创建或覆盖文件；
* • edit\_file 做精确的字符串替换。

工具越多，模型的认知负担越重——它必须在更多选项中做选择，选错的概率也随之增加。更少的工具也意味着更少的 schema 需要维护、更少的边界情况需要处理。

规划与协调

### s03 TodoWrite

这个其实就是一个 todolist，我们在做事情的时候因为事情太多了、太杂了，往往会列一个 todolist 来提醒自己，做完一件事情勾掉一件事，那么 llm 也是如此，做完了这件事情往往会忘记接下来该做什么，以往都是通过我们来一遍遍、一轮轮的通过 prompt 的方式来提醒他、push 它接下来该做什么了，因为 llm 的记忆力并不令人满意，事情一多往往就遗忘了他接下来要做什么了。

TodoWrite 是能让 llm 自己列一个 todolist 并且做完了一件事情就回过头来看看这个 list，给做完的事情打一个勾，然后看看第二件事情是什么，阅读了上下文之后，再去做第二件事情。

spec 的 coding 模式下，llm 先就对你告诉他的需求做一个 plan，输出一个他的实现计划，你也可以看做是一个长长的 todolist，这里的每一个 todo 可能都比较大，在领取这里的大 todo 之后，llm 根据任务复杂度等实际情况，来自主决定要不要继续拆分，拆出来一个小的 todolist。

todowrite 这个我记得好像是不会永久存储在某个架构文件中的，需求来了拆解 todolist 这个及时文件，做完了这个大需求，todolist 的这个及时文件就会被及时清空，llm 会把你完成需求的大 todo 或者小 todo 自主决定把哪些塞到 context（memory） 里面去，只把完成这个需求的主要核心步骤/todo 塞到 context 上下文中，给到下一轮的prompt 中，其他细枝末节的东西通通删掉，少即是多，上下文越多越乱，这是 claude code 的一个核心设计理念：简洁。

### s04 子 Agent

整个 claude code 的 agent 设计我们可以分为三类：主 agent、子 agent（subagent）、agent mates，这里我们先讨论主 agent 和 subagent，主 agent 你可以理解为就是那个唯一的 agent，最开始就存在的agent，资历最老，然后呢subagent 是这个主 agent 自己忙不过来的时候Spawn召唤出来的，你可以想象为：主 agent=lead 老板，subagent=外包人员，老板忙不过来了就找几个外包帮他干活，把大活拆到最小的原子todo来让 subagent 来做，给 subagent 的只有和分配给他活相关的一部分 context 上下文信息，给他一些可能用到的 tools，其他啥也不给，然后就push subagent 快点干，subagent 只能接收主 agent 的 todo 中分给他的活，没有权限再去新建 todolist，也不能再去创建 subagent，否则成无限套娃了。然后subagent 干完了，自己也跑过测试了，就和主 agent 说一下我完成了也测试完了，主 agent可能去验收一遍，也可能不验收，直接去 todolist 那里把这个 todo 给勾掉，表示已经完成了，至此 subagent 的寿命就结束了，就会被主 agent 给开除，一脚踢掉说拜拜👋，一点memory记忆上下文都不能留，直接清空且火化。subagent 唯一留下的就是主 agent 那长长的 todolist 中其中的一个勾选"✅：xxxxxxx"。

```
Parent agent                     Subagent
+------------------+             +------------------+
| messages=[...]   |             | messages=[]      | <-- fresh
|                  |  dispatch   |                  |
| tool: task       | ----------> | while tool_use:  |
|   prompt= ...    |             |   call tools     |
|                  |  summary    |   append results |
|   result =  ...  | <---------- | return last text |
+------------------+             +------------------+

Parent context stays clean. Subagent context is discarded.
```

subagent 就是"外包工人"，它只负责执行具体代码工作，不要让它自己去碰 todowrite / task 工具，全交给主 Agent 来协调最稳。

### s05 技能

对于 skills 我没有太多好解释的，skills=api tools=mcp，在 llm 层面稍微宏观的来看，这3个东西 没差别，都是被 llm 调用的；往里看是有差别的，skills 可以包括另两个，skills=api tools+mcp+**info**，我把 skills 看作是完成一件事的完美 sop 手册，里面写了怎么样做好一件事，该在什么时候就调用什么 tools 工具，什么时候用 mcp，然后把返回的结果要怎么怎么用，是一个教你完美做好这件事的执行手册，这个手册在我们实际生活中是前辈老师们给我们的经验，或者我们自己在实践中吃到苦头了、失败了总结而来的教训和踩坑经验，我们把这些东西总结复盘成 sop，下次我们在做这些事情我们就拿出这本 sop 执行手册，我们就不会再犯错了，对于 llm 来说也是这样，我们把人类的经验传授给 llm，告诉他这本手册里面写的是《如何写出小红书百万爆款文章》、《如何把运营工作自动化》等等，当你让 llm 来给你写一篇文章的时候，llm 在 skills.md这个文件里面扫了一遍，愕然发现卧槽《如何写出小红书百万爆款文章》这本堪比九阴真经的武林奇术，立马学习跟着书里面说的那样一步步执行-调用-执行来写出小红书百万爆款，完美的学习了如何写出百万爆款。

skills 这种东西，把我们的各种 sop 都给套出来了，要是大模型厂商再拿人类各行各业的优秀 skills去训练llm， 提升能力，那我感觉有点鸡贼，阳谋啊。

### s07任务系统

现在 claude code 版本已经用 task 取代了 todowrite，task 其实同样也是一个清单，但是里面有逻辑有前置条件，比如 任务 B 开启的条件必须是任务 A 已经完成，任务 D 开启的条件必须是 B 和 C 都已经完成了，单单 B 完成了是无法启动任务 C 的。举个例子，任务 A 比如是需求调研，任务 B是撰写 prd 文档，任务C 是需求评审会议，那么单单的 A 或者 B 完成了，是不足以开任务 C需求评审会的，被 block 堵塞了，满足前置条件了再给你开；同样的，这三个任务在 todolist 里面是怎么样的？全部是统一层级，完全扁平，罗列出来，也没有任务的先后顺序，没有优先级，没有依赖、状态只有做完没做完，任务之间没有任何逻辑。而 task是有结构的 -- 任务 B 依赖任务 A, 任务 C 和 D 可以并行, 任务 E 要等 C 和 D 都完成。

每一个 task 任务都是有状态的：Status（pending / in\_progress / completed）

1. 1. pending：没人领，但不代表你可以领，还得看这个任务的前置条件（`blockedBy`）都完成了，没有被阻塞
2. 2. in\_progress：被领走了，正在干活
3. 3. completed：已经干完了

把扁平清单升级为持久化到磁盘的**任务图**。每个任务是一个 JSON 文件, 有状态、前置依赖 (`blockedBy`) 和后置依赖 (`blocks`)。任务图随时回答三个问题:

* • **什么可以做?** -- 状态为 `pending` 且 `blockedBy` 为空的任务。
* • **什么被卡住?** -- 等待前置任务完成的任务。
* • **什么做完了?** -- 状态为 `completed` 的任务, 完成时自动解锁后续任务。

```
.tasks/
  task_1.json  {"id":1, "status":"completed"}
  task_2.json  {"id":2, "blockedBy":[1], "status":"pending"}
  task_3.json  {"id":3, "blockedBy":[1], "status":"pending"}
  task_4.json  {"id":4, "blockedBy":[2,3], "status":"pending"}

任务图 (DAG):
                 +----------+
            +--> | task 2   | --+
            |    | pending  |   |
+----------+     +----------+    +--> +----------+
| task 1   |                          | task 4   |
| completed| --> +----------+    +--> | blocked  |
+----------+     | task 3   | --+     +----------+
                 | pending  |
                 +----------+

顺序:   task 1 必须先完成, 才能开始 2 和 3
并行:   task 2 和 3 可以同时执行
依赖:   task 4 要等 2 和 3 都完成
状态:   pending -> in_progress -> completed
```

以上是 task 和 todo 在功能性上的区别，在使用方式和存储上也不一样，
task以 JSON 文件形式持久化在 .tasks/ 目录中，而非像 todo 那样存在内存里。这有三个关键好处：
(1) 任务在进程崩溃后仍然存在——如果 agent 在任务中途崩溃，重启后任务板仍在task的那个 JSON文件里；
(2) 多个 agent 可以读写同一任务目录，无需共享内存即可实现多代理协调；
(3) 用户可以查看和手动编辑任务文件来调试。文件系统就是共享数据库。

内存管理

### s06 上下文压缩

三层压缩策略，上下文一旦多起来那就很难管理了，llm 每次都去100% 读取的话，成本高，而且效果并不好，大水漫灌污染 llm 的大脑。

(1) 微压缩每轮都运行，几乎零成本：它截断旧消息中的 tool\_result 块，去除不再需要的冗长命令输出。
这一条是之前 llm 每次调用 tools 时，tool 工具把调用的结果返回给 llm ，现在既然活已经干完了，那么我要保留这些工具返回的 result 干嘛，通通删除，我只看结果，不要过程。

(2) 自动压缩在 token 数超过阈值时触发：调用 LLM 生成对话摘要，代价高但能大幅缩减上下文。
llm 一次读取的 context 上下文历史信息，你可以设定一个值，比如超过 20 万 token，就启动自动压缩，把历史的所有上下文全部压缩概括成一个新的上下文给到后面的 llm 对话用，之前的被压缩的 20 万 token 的上下文不会删除，而是给你换个地方存起来，但是 llm 不会再去用这个 20 万的上下文了，等于是被锁进了柜子，留个案底。

(3) 手动压缩由用户触发，用于明确的'重新开始'场景。分层意味着低成本操作持续运行（保持上下文整洁），而高成本操作很少触发（仅在真正需要时）。
通过用户/compact 触发。

### s08后台任务

* • 慢操作丢后台, agent 继续想下一步 \* -- 后台线程跑命令, 完成后注入通知。有些命令要跑好几分钟: `npm install`、`pytest`、`docker build`。阻塞式循环下模型只能干等。用户说  装依赖, 顺便建个配置文件 , 智能体却只能一个一个来。

后台任务结果通过 threading.Queue 传递，而非直接回调。后台线程在工作完成时向队列放入通知，主 agent 循环在每次 LLM 调用前轮询队列。这种解耦很重要：后台线程无需了解主循环的状态或时序，只需往队列放入消息然后继续。主循环按自己的节奏取出消息——永远不会在 API 调用中途或工具执行中途。没有竞争条件，没有回调地狱。

协作

### s09 Agent团队

前面说到主 agent 和 subagent（子 agent）的关系，主 agent 像是个老板，subagent 是老板请来的外包，而这里说的 agent 团队可以看成是老板下面的长期合同工，签了劳务合同的那种，比 subagent 这种可以随时一点不留消失的外包强多了，起码是正式员工，这些 agentmates 和主 agent 之间都可以通讯，发送/接收消息，每个员工都有一个小邮箱，但是 agentmates 不能像主 agent 那样聘请 subagent，毕竟是员工而不是老板，请不了外包。

subagent是一次性的: 生成、干活、返回摘要、消亡。没有身份, 没有跨调用的记忆。后台任务能跑 shell 命令, 但做不了 LLM 引导的决策。
真正的团队协作需要三样东西:
(1) 能跨多轮对话存活的持久智能体,
(2) 身份和生命周期管理,
(3) 智能体之间的通信通道。

然后既然有这多agent 了，那么哪些东西是老板交代的，得开会透明啊，坦诚清晰，于是这个团队就有了一块白板，上面写着大家接下来的 okr 任务，该做什么task，谁负责这部分等等，这块共享的白板叫做 shared task list，agent 员工们各自从上面领取自己的 task 任务，这里的任务仍然是有逻辑的有秩序的，如下：

Shared Task List 这块白板上，每个任务都有这些字段（像小卡片一样）：

* • **Task ID**（任务编号）
* • **描述**（要做什么）
* • **Owner**（谁领了）
* • **Status**（状态：pending / in\_progress / completed）
* • **blockedBy**（我必须等谁干完？）
* • **blocks**（我干完后会解锁谁？）

领取任务时的规则（超级重要）
当一个 AgentMate（员工）去看白板想 claim（领取）任务时，Claude 会自动检查：

* **必须满足两个条件才能领：**

1. 1. 状态是 pending（没人领）
2. 2. **没有被阻塞**（blockedBy 里的所有前置任务都已经 completed）

* • 如果一个任务写了 blockedBy: [ task-003 ,  task-007 ]，那它就会显示成 **blocked** 状态（灰色或加锁），**谁都不能领**，直到 task-003 和 task-007 都被完成。
* • 一旦前置任务完成，白板会**自动解锁**（unblock）它，后面的 AgentMate 就能看到了。

举个例子，白板上现在有 4 个任务：

* • Task 1：重构数据库连接 → 无依赖
* • Task 2：写用户登录接口 → **blockedBy: Task 1**（必须等数据库连好）
* • Task 3：做登录页面 UI → **blockedBy: Task 2**
* • Task 4：写单元测试 → 无依赖（可以和别人并行）

**领取顺序就会自然变成这样**：

1. 1. 大家只能先领 Task 1 和 Task 4（这两个没被 block）
2. 2. 等 Task 1 干完，Task 2 自动解锁，大家才能去领 Task 2
3. 3. 等 Task 2 干完，Task 3 才解锁

这样就**不会出现后面的人抢先干活**，也不会出现"数据库还没连好，就去写登录页面"的低级错误。

谁来决定这些 block 关系？

* • **主要由主 Agent（老板）** 在创建任务或做计划时设定。
* • AgentMates（员工）在执行过程中也可以建议增加 block（比如"我这个任务需要等前端先做好样式"），但最终还是 Lead 老板来确认。

**领取 task 不是随便挑的**，白板会自动帮你"上锁"：
只有**前面依赖的任务全部完成**，这个任务才会变成可领取状态。
这就是 Claude 设计 **blockedBy / blocks** 的核心目的 —— 让整个团队的进度**有条不紊、不会乱序**。

### s10 团队协议

Agent team 里面有这么多 agent，那他们怎么通信呢怎么聊天呢？每一个 agent 都有一个自己的小邮箱，用小邮箱来发送/接收信件，
比如：

* • 前端代理对后端代理说："我需要用户ID字段才能继续写登录页面，你能给我API接口定义吗？"
* • 后端代理回复："好的，这是我设计的接口 schema……"
* • 或者领导代理广播："测试代理，代码改完了，请更新测试用例。"

消息是**自由的、自然语言的交流内容**，用来讨论、澄清、协调、汇报进度，就像真实团队里的聊天。系统有一个 **Mailbox（邮箱系统）** 来传递这些消息，可以是私信（发给某个特定队友），也可以是广播（发给所有人）。

**但是团队协议不一样：**
这不是"聊天内容"，而是 **一套结构化的规则和交互格式**，用来让团队协作更有秩序、不乱套。它相当于给团队定了一套"工作纪律"和"沟通模板"，避免大家乱发消息导致混乱、重复或遗漏。

简单的对比就是： **消息（message） = 他们说了什么（内容），一般用于平常的不太重要的内容。****协议（protocol） = 他们应该怎么说、按什么流程说、用什么格式说（规则 + 模板），一般是较为重要的关键操作，比如关闭任务、终止程序、任务移交、正式请求等。**

打个比方，比如你和朋友平常聊天吹牛逼就是普通的 message，大白话说就可以了，但是突然有一天你要结婚了，那你不能随口说吧，你得发一个请柬吧，庄重的发给朋友，这个请柬里面就有：

```
{
  "type": "婚礼request",
  "request_id": "req-20260409-001",
  "from": "老 K（我本人）",
  "to": "朋友老 A",
  "action": "来参加婚礼",
  "content": "请你于 xxxx 来参加我和 xxx的婚礼，举办在 xxx 地方，请你准备好充足的礼金红包来……",
  "priority": "high"
}
```

cc 里面 agent team 之间的 protocol 是这样的方式，有一个独一无二的 request-id，发出这个 id，对面接收这个 id，这个 request 在 cc 里面被叫做ProtocolEnvelope，对面收到了ProtocolEnvelope再来回复。

那么你怎么知道你发了这个request 消息呢，你这里有留存，叫做RequestRecord，记录着说我在什么时候把这个请柬发给了谁谁谁，后面结婚他来不来也会记录在上面，他的回复也都会在上面，就像是我提了一个工单，后面关于这个工单的所有进展记录和状态都在这上面了。

```
request = {
  "request_id": "req-20260409-001",
  "kind": "婚礼",
  "from": "老 K（我本人）",
  "to": "朋友老 A",
  "status": "approved（同意来了）"
}
```

这里的status状态机非常简单，有三种：

```
pending -> approved同意
pending -> rejected拒绝
pending -> expired超时（可能已读不回🐶）
```

### s11 自主Agent

这个自主的 agent 很好解释，就是老板期望的永不停歇的牛马，没活也给自己找活干。最清楚的理解方式，是把每个agent想成在两个阶段之间切换：

```
WORK
  |
  | 当前轮工作做完，或者主动进入 idle空闲
  v
IDLE空闲
  |
  +-- 看邮箱，有新消息 -> 回到 WORK
  |
  +-- 看任务板，有 ready task -> 认领 -> 回到 WORK
  |
  +-- 长时间什么都没有 -> shutdown
```

这里的关键不是"让它永远不停想"，而是： **空闲时，按规则检查两类新输入：邮箱（mail）和任务板（shared task list）。**

**什么任务算"当前这个队友可以安全认领"的任务。**
判定不是单纯看 `pending`，而是：

```
def is_claimable_task(task: dict, role: str | None = None) -> bool:
    return (
        task.get("status") == "pending" and not task.get("owner")
        and not task.get("blockedBy")
        and _task_allows_role(task, role)
    )
```

这 4 个条件缺一不可：

* • 任务还没开始
* • 还没人认领
* • 没有前置阻塞
* • 当前队友角色满足认领策略

最后一条很关键。
因为现在cc 更新后的任务带了以下两个参数：

* • `claim_role`
* • `required_role`

例如：

```
task = {
    "id": 7,
    "subject": "Implement login page",
    "status": "pending",
    "owner": "",
    "blockedBy": [],
    "claim_role": "frontend",
}
```

> 这条任务不是"谁空着谁就拿"，而是要先过角色条件，得看谁能看，前端的活我肯定不会给一个专门做后端的 agent。

### s12 Worktree+任务隔离

如果你熟悉 git，可以把 worktree 理解成：

> 同一个仓库的另一个独立检出目录。

如果你还不熟悉 git，也可以先把它理解成：

> 一条属于某个任务的独立工作车道。

什么叫隔离执行
隔离执行就是：

> 任务 A 在自己的目录里跑，任务 B 在自己的目录里跑，彼此默认不共享未提交改动。

什么叫绑定
绑定的意思是：

> 把某个任务 ID 和某个 worktree 记录明确关联起来。

最小mvp：最容易理解的方式，是把这一章拆成两张表：

```
task 面板
  负责回答：做什么、谁在做、状态如何

worktree 注册表
  负责回答：在哪做、目录在哪、对应哪个任务
```

两者通过 `task_id` 连起来：

```
.tasks/task_12.json
  {
    "id": 12,
    "subject": "Refactor auth flow",
    "status": "in_progress",
    "worktree": "auth-refactor"
  }

.worktrees/index.json
  {
    "worktrees": [
      {
        "name": "auth-refactor",
        "path": ".worktrees/auth-refactor",
        "branch": "wt/auth-refactor",
        "task_id": 12,
        "status": "active"
      }
    ]
  }
```

**任务记录工作目标，worktree 记录执行车道。**

那么为什么要这么设计呢？
因为 A 在配置文件改，B 可能也在这个配置文件里改，甚至拿的配置文件是 A 改了一半的，我靠，那不乱套了啊，改完之后还直接推上去了，可是你可能要问不是前面任务都分配好了吗，认领好了吗，怎么还会乱套抢活干呢？

1. 1. 同时触发抢了一个任务，几率不大
2. 2. 任务分的不清晰
3. 3. 比如 task a 是修改配置文件，被小明领走了；
4. 4. task b 是修改 llm api 的接入方式和参数，被小红领走了；
5. 5. 但是问题在于，task a 改的是配置文件，task b 的接入方式和参数他妈的也在配置文件里面啊，小明和小红开始改同一个文件，那不乱套了啊，所以才有分隔来干活的必要性，别乱套，各干各的，最后可以住 agent 来统一验收，来合并。
