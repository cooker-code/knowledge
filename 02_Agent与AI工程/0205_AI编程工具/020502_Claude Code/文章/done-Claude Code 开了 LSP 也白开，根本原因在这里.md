#

你用的 AI 编程工具，可能根本不会"读代码"。

不是在说它不好用。

是说，它理解代码的方式，和你以为的完全不一样。

我最近深挖了一件事：Cursor、Claude Code、OpenCode，这三款工具在"读懂代码"这件事上，走的是三条完全不同的路。差距之大，超出我的预期。

---

**先说一个你可能没意识到的问题。**

当你让 AI 帮你找一个函数定义，它是怎么找的？

Claude Code 的默认做法，是在整个代码库里跑 grep，把所有包含这个字符串的地方都找出来，然后让模型自己判断哪个才是你要的。

这个过程，快的话 30 秒，慢的话 60 秒。

而且它不知道"调用"和"定义"的区别。两个同名函数分属不同作用域，它也分不清。

说白了，就是把代码当纯文本在搜。

OpenCode 也差不多，工具集更结构化一点，但底层逻辑是一样的：让模型自己充当代码理解引擎，工具只负责把文件内容送到模型面前。

然后你再看 Cursor。

Cursor 在你打开项目的时候，就在后台启动了一套索引流水线：用 tree-sitter 按语法边界把代码切成块，对每个块生成向量嵌入，存进向量数据库。你问它"用户注册的完整流程是什么"，它做的是语义搜索，不是字符串匹配。

实测数据显示，语义搜索比关键词搜索的代码检索准确率高出约 12.5%。

这就是为什么同样一个问题，Cursor 的回答往往更准。不是模型更聪明，是它拿到的上下文质量更高。

---

**但这里有一个很多人没想到的反转。**

LSP（语言服务器协议）这个东西，对 Claude Code 和 OpenCode 的价值，远大于对 Cursor 的价值。

为什么？

因为 Cursor 已经有了向量索引，语义理解这一关它过了。LSP 对它来说是锦上添花，提供精确的实时诊断反馈。

但对 Claude Code 来说，LSP 是从"文本盲"到"语义理解"的质变。

开了 LSP，AI 就能直接调用 goToDefinition、findReferences，知道这个函数在哪里定义、被哪些地方调用，而不是靠 grep 猜。

问题是，Claude Code 的 LSP 默认是关闭的。

原因有三个：

**第一，资源消耗太大。** pyright 在大型代码库上可以吃掉 2~4GB 内存，对一个 CLI 工具来说，默认开启代价太高。

**第二，模型不会主动用。** 这是最有意思的一点。即使你开了 LSP，Claude 也经常习惯性地用 grep 找代码，而不去调用 LSP 工具。因为训练数据里 grep 更常见，这是模型的"肌肉记忆"。除非你在 CLAUDE.md 里明确指示，否则大概率视而不见。

**第三，早期版本有严重 bug。** v2.0.74 发布后，LSP 因为竞态条件完全不可用，直到 v2.1 才修复。

---

**OpenCode 的情况又不一样。**

它把 LSP 作为第一方基础设施内置，支持 30+ 种语言服务器，按文件扩展名自动触发，零配置开箱即用。

但它有一个已知问题：退出之后，那些被启动的语言服务器进程不会跟着退出。

它们会以"孤儿进程"的形式继续在后台跑，无限期占用内存。多项目场景下，这些进程会随使用次数不断累积。

有用户实测，残留进程可以累积到 1.2GB 内存占用。

根本原因是五层缺陷叠加：信号处理器缺失、强制退出跳过清理、实例缓存永不过期、进程未做进程组管理，加上部分 LSP 本身的独立 bug。

这个问题的 Memory Megathread 在 GitHub 上已经开了，截至 2026 年 5 月仍是 Open 状态，还在持续有新用户报告。

---

**Cursor 的方案工程上最复杂，但对用户最透明。**

它用的是"影子工作区"：AI 提出一个代码编辑建议，这个编辑先被发到一个用户完全不可见的隐藏 Electron 窗口，在那里跑完整的 LSP，拿到诊断结果，再返回给 AI。

用户什么都感知不到，AI 已经自动完成了"生成→诊断→修正"的闭环。

Cursor 官方对这套机制的评价是：在底层语言模型固定不变的前提下，让 AI 为其编辑结果获取 lint，是提升代码生成效果"最有影响力的方法之一"。

代价是约 2 倍的内存占用。

---

**所以，三款工具对 LSP 的态度，折射出三种完全不同的产品哲学。**

OpenCode 认为 LSP 是 Agent 的核心基础设施，选择了"显式工具化"，让 AI 能主动感知和调用诊断能力。工程实现务实，但遗留进程问题还没解决。

Cursor 选择了对用户透明的"影子工作区"，AI 无需知道 LSP 的存在就能自动获得诊断反馈。代价是更高的架构复杂度和内存开销。

Claude Code 目前处于最保守的状态。功能存在，但默认关闭，依赖社区插件，模型本身的工具调用习惯也还没跟上。LSP 对它来说更像是"给高级用户的可选增强"，而不是开箱即用的能力。

你会发现，这不只是技术选择的差异。

它背后是三个团队对"AI Agent 应该怎么工作"这个问题的不同回答。

OpenCode 说：让 AI 主动感知，工具要透明可调用。

Cursor 说：让 AI 自动获得反馈，用户不需要知道细节。

Claude Code 说：先把基础做稳，高级功能交给社区探索。

从长远来看，随着 AI Agent 工具调用能力的成熟，这两种范式之间的边界会逐渐模糊。理想状态是两者兼备：全局索引负责找到正确上下文，LSP 负责验证生成结果的正确性。

但现在，如果你在意代码理解的准确性，Cursor 的向量索引是它的护城河。

如果你在意 AI 的主动感知能力，OpenCode 的 LSP 集成更成熟，但要做好进程管理的心理准备。

如果你在用 Claude Code，开 LSP 之前，先在 CLAUDE.md 里告诉它"优先使用 LSP 工具而不是 grep"。否则开了也白开。
