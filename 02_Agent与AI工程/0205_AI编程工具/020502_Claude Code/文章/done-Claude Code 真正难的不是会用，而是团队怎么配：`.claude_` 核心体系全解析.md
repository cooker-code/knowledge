很多人已经开始用 Claude Code，但真正落地到团队协作时，常常还是会卡住：

* 哪些配置应该提交 Git？
* 哪些东西只能保留在本地？
* `CLAUDE.md`、`rules`、`skills`、`subagents`、`settings.json` 到底分别负责什么？
* 为什么明明写了限制，Claude 还是可能“越界”？
* 多套配置叠在一起，优先级到底怎么算？

这篇文章将把这些问题讲得很透。

它不是简单再介绍一遍 `.claude/` 文件夹里有什么，而是把 **Claude Code 团队配置真正该怎么设计** 这件事讲明白了。

如果你正准备把 Claude Code 从“个人玩具”升级成“团队生产工具”，这篇很值得看。

## 一句话先说结论

最核心的价值，不是教你认文件名，而是告诉你：

**Claude Code 的团队落地，必须分层设计。规范、权限、自动化、上下文隔离，不能混在一起。**

这才是很多团队一直没跑顺的根本原因。

## 最值得关注的 5 个重点

## 1. `.claude/` 不是一个配置点，而是一整套作用域体系

很多人以为 Claude Code 的配置就集中在一个地方，实际上不是。

文章把 Claude Code 的配置体系拆成了四层：

* 组织级（policy）
* 用户级（user）
* 项目级（project）
* 本地覆盖（local）

而实际最常用的是这三层：

### 项目级：团队共享配置

项目目录下的 `.claude/`

这里通常放：

* `settings.json`
* `rules/`
* `skills/`
* `agents/`

这些应该提交 Git，用来保证团队成员在同一个仓库中的行为一致。

### 用户级：个人全局偏好

`~/.claude/`

这里适合放：

* 你个人的习惯
* 跨项目复用的资产
* 不希望影响团队其他人的配置

### 本地覆盖：只对自己生效

`.claude/settings.local.json`

这个文件通常应该被 `gitignore`，适合用来做：

* 本地实验
* 临时权限放宽
* 个性化调整

**这部分特别关键。**

因为如果你不把“团队规范”和“个人偏好”分开，仓库配置会很快变得混乱。

---

## 2. `CLAUDE.md` 很重要，但它不是“硬约束”

这里有一个很重要的认知纠偏：

很多人会把 `CLAUDE.md` 理解成“Claude 必须绝对遵守的规则中心”，但实际上，它更像是：

* 自动注入的长期上下文
* 项目默认工作方式说明书
* 团队规范与架构边界的长期记忆

也就是说，`CLAUDE.md` 最适合承载的是：

* 常用命令
* 项目结构
* 协作约定
* 开发习惯
* 容易踩坑的点

比如：

* 改动后默认补测试
* 错误处理统一走哪个模块
* 哪些目录分别负责什么
* 本地依赖有什么特殊要求

这些内容都很适合写进去。

但问题是：

**`CLAUDE.md` 本质上是上下文注入，不是客户端强制执行规则。**

这意味着：

* 如果指令本身含糊，Claude 不一定完全遵守
* 如果多个要求互相冲突，Claude 可能会权衡
* 它更适合表达“应该怎么做”，不适合单独承担“绝对不能怎么做”

这是很多人第一次真正理解 Claude Code 配置体系时最容易忽略的一点。

---

## 3. `rules` 的真正价值，是把规范做成可维护模块

当 `CLAUDE.md` 越写越长，最后变成“团队 Wiki”时，问题就来了：

* 有效信息被稀释
* 模型遵循度下降
* 维护成本越来越高
* 团队多人协作容易互相覆盖

这里给出的建议：

**把规范拆到 `.claude/rules/`。**

一个主题一个文件，递归发现，多人维护，互不干扰。

更重要的是，`rules` 不是简单的“拆文件”，它还有两种加载方式：

### 全局规则

没有 `paths` 的规则，会在启动时加载，作用类似 `CLAUDE.md`。

### 路径触发规则

带 `paths` frontmatter 的规则，只在 Claude 处理匹配路径的文件时触发。

这意味着你可以做到：

* 只有在处理 `src/api/**` 时才注入 API 规范
* 只有在处理 `src/components/**` 时才注入前端组件规范
* 只有在处理测试目录时才注入测试规则

这件事的价值非常大。

因为它让模型拿到的不是一坨大而全的规范，而是**当前任务真正相关的规范**。

对团队来说，这种设计会比把所有内容都塞进一个大 `CLAUDE.md` 更稳定。

---

## 4. `skills` 和 `subagents`，才是 Claude Code 真正拉开差距的高级能力

最值得团队负责人和高频使用者重视的，不只是配置层，还有工作流层。

其中最关键的两个能力是：

* `skills`
* `subagents`

---

## 4.1 `skills`：把重复工作流封装成一条命令

`CLAUDE.md` 和 `rules` 解决的是“Claude 应该知道什么”。

而 `skills` 解决的是：

**Claude 能快速完成什么。**

文章提到，定义一个 `SKILL.md` 后，就可以：

* 手动通过 `/skill-name` 调用
* 也可以在匹配任务时自动触发

更值得关注的是它的几个能力点：

### `description`

决定自动触发的适用场景。

### `disable-model-invocation`

禁止模型自动调用。

这个很适合有副作用的操作，比如会写文件、发请求、改配置的技能，避免模型误触发。

### `allowed-tools`

明确限制这个 skill 能用哪些工具。

这能让技能变得更可控，而不是把所有能力都开放给一次调用。

### 参数化

支持类似 `/fix-issue 123` 这种调用方式。

也就是说，你可以把常见操作封装成真正可复用的工作流命令。

### 动态上下文注入

有一个很实用的能力：

可以先执行 shell 命令，把输出结果插进 prompt 再交给 Claude。

这意味着你可以做出类似：

* 自动带入 `git diff` 的 code review
* 自动带入错误日志的 debug
* 自动带入变更列表的发布检查

这已经不是“会聊天的 AI 助手”了，而是一个开始具备工程可复用性的工作流层。

---

## 4.2 `subagents`：把重活扔进隔离上下文

这里另一个非常关键的功能点。

复杂任务最容易遇到的问题是：

* 主线程上下文被塞爆
* 检索过程污染对话主线
* 成本高
* 结果不够聚焦

而 `subagents` 的意义就在于：

* 跑在独立上下文
* 有独立系统提示词
* 可以单独限制工具集
* 可以单独指定模型
* 最后把摘要结果带回主线程

这里总结了 3 类特别适合交给 subagent 的任务：

### 第一类：大量文件检索

比如让 research agent 去读很多文件、跑 grep、整理结论，最后只返回重点。

### 第二类：强约束审查

比如安全审计 agent 只允许 `Read/Grep/Glob`，禁止 `Write/Edit`。

这时候约束不是“希望它克制”，而是“工具权限上就做不到乱改”。

### 第三类：成本控制

一些适合并行、适合概括、适合快速扫描的任务，可以下沉给更快更便宜的模型；主线程只保留关键决策。

这背后的本质其实是：

**把复杂任务拆层，把高成本思考和低成本检索分开。**

很多团队在 Claude Code 上迟迟没建立出真正效率优势，就是因为所有工作都堆在主对话里。

---

## 5. `settings.json` 才是安全边界的核心，但它也有一个关键盲区

我认为最应该被工程负责人画重点的，是这一部分。

需要明确区分两类东西：

* 提示词层面的期望
* 客户端层面的强制规则

而 `settings.json` 属于后者。

它的意义在于：

**把“希望 Claude 别乱来”，升级为“客户端层面不允许乱来”。**

这包括：

* 哪些命令允许执行
* 哪些工具允许调用
* 哪些危险操作要拒绝
* 哪些能力必须被限制

这比单纯在 `CLAUDE.md` 里写一句“不要执行危险命令”有效得多。

但同时指出了一个非常重要的安全盲区：

如果你只限制了 `Read(./.env)`，但仍然允许 `Bash`，那么 Claude 依旧可能通过 `cat .env` 读取敏感内容。

这意味着：

* `Read/Edit` 的 deny 规则，只约束内置文件工具
* **并不自动约束 Bash 子进程**
* 如果你真想要路径级别的隔离，仅靠 deny 不够，还需要引入 `sandbox`

这部分非常容易让团队产生错觉：

“我都写 deny 了，应该安全了吧？”

实际并不一定。

真正严肃的安全边界设计，必须把：

* 工具权限
* Shell 权限
* sandbox 隔离
* hooks 拦截

这些东西结合起来看。

## 最有价值的一点：它帮你把 Claude Code 的分层方法论讲清楚了

最后其实给出了一套非常清晰的分层框架，我觉得这是全文最值钱的部分。

### 第一层：行为建议与规范引导

用于告诉 Claude“推荐怎么做”：

* `CLAUDE.md`
* `rules`

### 第二层：安全边界与确定性自动化

用于确保“不能乱来”和“某些流程必须执行”：

* `settings.json`
* `sandbox`
* `hooks`

### 第三层：工作流复用与上下文隔离

用于提升效率、降低成本、减少主线程污染：

* `skills`
* `subagents`

这个框架特别重要，因为它回答了一个很多团队都会碰到的问题：

**Claude Code 在团队里，到底怎样才能既高效，又不失控？**

答案不是继续堆提示词，而是把不同职责分开：

* 规范归规范
* 权限归权限
* 自动化归自动化
* 重任务归隔离上下文

一旦把这些东西混写在一起，团队就很容易陷入：

* 配置越来越乱
* 权限越来越模糊
* 工作流不可复用
* 安全边界靠“运气”

而一旦分层清楚，Claude Code 才可能真正变成工程化能力，而不是零散技巧。

## 最值得团队立刻借鉴的地方

如果你准备在团队里正式用 Claude Code，我觉得至少可以马上借鉴这几件事：

### 1. 先划清团队共享和个人本地的边界

不要把个人偏好混进项目仓库。

### 2. 把 `CLAUDE.md` 控制在“关键约定”级别

别把它写成超长百科全书。

### 3. 把细则拆进 `rules`

尤其是按路径触发的规则，性价比非常高。

### 4. 把高频重复工作封装成 `skills`

让团队开始积累真正的 AI 工作流资产。

### 5. 用 `subagents` 处理重检索、重归纳、重审查任务

避免主对话被复杂任务拖垮。

### 6. 不要把安全边界寄托在提示词上

真正的限制要通过 `settings.json`、`sandbox` 和 `hooks` 去落。
