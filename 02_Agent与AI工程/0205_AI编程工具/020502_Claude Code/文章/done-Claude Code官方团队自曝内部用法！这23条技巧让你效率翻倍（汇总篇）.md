大家好，我是噪点

前两天，claude code创始人Boris Cherny在推上分享了他们团队内部使用cc的一些技巧；一个月前，他就分享过他本人使用claude code的10条高效技巧；我看了下，觉得很多都非常受用，毕竟是官方团队的经验，就把这2次分享统一翻译梳理了下，整合成了7部分，分享给大家，如果有觉得适合自己的可以直接跟着官方抄作业；

这2次分享，主要涵盖7部分：Plan模式、并行任务处理、claude.md管理、自动化工作流、工具效率配置、claude验证反馈、提示技巧；

****Plan模式****

Boris认为**好的计划等于一次搞定，所以**他大部分任务会话都会先从plan模式开始；

他的逻辑很简单，他认为花10分钟把计划想清楚，claude就能一次搞定，否则多次来回修改，反而最终会更慢；

团队里则有人更进一步：会**用两个claude来交叉审核；**一个claude写计划，另一个claude来审核这个计划，这样能避免单个claude一条路走到黑的情况；

还有一点就是，他们认为一旦发现任务执行跑偏了，就要立刻切回plan模式重新规划，而不是来来回回多次修改浪费时间；

****并行******任务处理**

cc团队内部公认的**最大效率提升点就是****并行****任务处理，也就是多开才是王道，不要一个个任务串联执行，而是多开几个并行跑，把任务等待时间就变成生产时间；**

Boris的做法是同时开5个终端窗口跑claude code，分别编号1-5，通过系统通知来提醒自己哪个窗口需要输入；同时他还会开5-10个网页版claude配合使用，用--teleport命令在本地和网页之间无缝同步任务会话记录；

团队更多用的是**git****worktrees，**开3-5个worktree，每个跑独立任务，每个任务完全隔离，互相不影响；有人还会给worktree起名字，设置快捷键来一键切换；还有人专门留一个worktree只用来看日志和跑查询，效率最大化；

****claude.md****

claude.md是给claude的“项目说明书"，可以告诉claiude这个项目怎么跑、有什么规则限制等，是任务执行时所遵守的唯一准则；

所以Boris他们团队会共享一个claude.md，并且将其提交到仓库，全组人一起维护，每次发现claude执行有什么问题时，就加进去，让它下次不要再犯；

在用claude时，团队还有一个小技巧：每次纠正claude之后，会提示“Update your cladue.md，so you don't make that mistake again"，也就是说他们建议让claude自己去写被纠正的规则，效果会更好；随着时间的推移，不断修改claude.md，会越用越顺；

****自动********化******工作流**

如果一个操作每天都要重复做好几次，就应该将它自动化掉，让它自己干，可以通过3种方式：

**Slash****Commands**：创建快捷命令，比如Boris分享他每天有几十次提交、推送这些操作，于是就建立了/commit-push-pr这个快捷命令，一键完成提交、推送；

**自定义Skills**：把流程sop变成skills，每次只需让claude 调用skills就能执行整个流程，不用一步步操作；

**长任务自动化**：跑很久的任务可以用ralph插件，让Claude自己循环跑，一直到任务完成，而不用我们一直盯着；

****高效配置****

**模型选择**：Boris只用opus 4.5 ，并且开启思考模式；虽然比sonnet慢，但需要人工介入的少，整体下来反而更快；

**权限管理**：不要用--dangerously-skip-permissions命令，而是用/permissions设置安全权限，并且把这些权限白名单存在.claude/settings.json里，给团队共享；

**终端**：团队很多人用Ghostty；用/statusline自定义状态栏，显示上下文用量，方便及时判断上下文管理；

**语音输入**：Mac上按两次fn可以语音输入，说话比打字快3倍，prompt也会写得更详细；

****claude自己验证反馈****

**Boris认为这是最重要的一条，**让**claude自己去验证自己的任务成果**，让它有及时反馈循环，输出质量能提升2-3倍；

比如Boris自己的做法就是用claude in chrome扩展，让claude可以打开浏览器自己跑、自己看、自己改；

当然其他验证方式也可以，关键是让Claude自己能看到自己的输出结果，形成闭环；

******提示技巧******

团队分享了几个很实用的prompt技巧：

**直接说“fix”**：遇到bug时不需要解释太多，启用Slack MCP，直接把报错信息粘贴进去说“fix”就行，让claude自己搞定；意思就是直接把报错给claude，只需对他说”修复”就行，无需过多解释；

不通过审核就别提pr：让claude来自己审查代码，告诉他不通过你的审核就不要提pr；

如果claude给的方案不够好，别在烂方案上修修补补，直接告诉他推翻重来，往往能给出更好的实现方案；
