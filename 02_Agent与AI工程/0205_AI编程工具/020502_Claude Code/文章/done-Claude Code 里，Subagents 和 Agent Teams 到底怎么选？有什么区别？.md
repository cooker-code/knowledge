# Claude Code 里，Subagents 和 Agent Teams 到底怎么选？有什么区别？

> Agent Team 这个功能，Claude Code 发布也有一阵子了。早先oh-my-opencode我认为也是这种思路，这篇笔记，我只是希望自己能在这上边探索一下。实际场景的开发需求，我还是认为不一定完全要交给AI全程解决，AI瞬间做的工作越多，人类审核的过程也就越复杂。而脱离人类的监督审核，就等于脱缰的野马。还是觉得不太好。

## 核心概念：Multi-Agent, Agent Team, Sub-agent 的关系

在深入之前，我们先用一个简单的比喻来理解本文中三个核心概念的关系，这会让你对接下来的内容有更清晰的认识。

简单来说，**Multi-Agent 是一个总的、宽泛的概念，而 Agent Team 和 Sub-agent 是这个概念下两种不同的、具体的实现模式。**

它们的关系就像 **“交通工具”** 、 **“汽车”** 和 **“摩托车”** 的关系一样：

* **Multi-Agent 系统 (多智能体系统) 就好比是 “交通工具”**：

+ 这是一个非常广泛的分类。任何“由多个智能体协同工作来完成任务”的系统，都可以被称为“多智能体系统”。

* **Agent Team (智能体团队) 就好比是 “汽车”**：

+ 它是“交通工具”里的一种**具体类型**。它有自己鲜明的特点，如：成员持久、可双向沟通、共享状态等，适合需要紧密协作的复杂任务。

* **Sub-agent (子智能体) 就好比是 “摩托车”**：

+ 它也是“交通工具”里的另一种类型。它的特点是：临时、快速、用完即弃、点对点沟通，适合执行独立的、并行的子任务。

所以，一个 **Agent Team** 或 **Sub-agent** 系统 **是** 一个 **Multi-Agent 系统**，但一个 Multi-Agent 系统不一定就是 Agent Team 或 Sub-agent 模式。

理解了这个层级关系后，我们再来深入探讨 Multi-Agent 系统的背景。

## 背景知识：什么是 Multi-Agent 系统？

在我们深入探讨 Sub-agents 和 Agent Teams 之前，先花一分钟理解它们所属的更大概念：**Multi-Agent 系统**。

简单来说，Multi-Agent 系统就是“**让多个 AI 智能体（Agent）协同工作，共同完成一个复杂任务**”。

想象一下，单枪匹马的“超级英雄”（单个 Agent）虽然强大，但面对一个极其复杂的全球性危机（例如，从零开始开发一个完整的软件应用），可能也会力不从心。这时，一个由具备不同能力的英雄组成的“复仇者联盟”（Multi-Agent 系统）就显得至关重要。

在这个联盟里，你可能有：

* **规划师 (Planner)**：负责理解任务目标，并将其拆解成一步步可执行的计划。
* **程序员 (Coder)**：负责根据计划编写代码。
* **测试员 (Tester)**：负责验证代码的正确性，并报告 Bug。
* **项目经理 (Manager)**：负责协调所有成员的工作，确保项目顺利进行。

这就是 Multi-Agent 系统的核心思想：**分而治之，协同作战**。通过为每个 Agent 分配专门的角色和工具，它们可以更高效、更专业地解决各自领域的问题，最终组合成一个强大的解决方案。

### 业界有哪些典型的 Multi-Agent 实现案例？

了解一些流行的框架，可以帮助我们更好地理解 Multi-Agent 的设计思想：

1. **AutoGen (微软)**：一个功能强大的多 Agent 对话框架。它允许你定义具有不同角色和能力的 Agent，并通过自动化的 Agent 间对话来解决问题。它的核心是 `UserProxyAgent`（你的代理人）和 `AssistantAgent`（AI 助手）之间的交互。
2. **CrewAI**：一个专注于“角色扮演”的 AI Agent 协作框架。它通过定义 `Agents`（角色）、`Tasks`（任务）和 `Crew`（团队）来编排工作流。值得注意的是，CrewAI 提供了两种核心流程：

* **顺序流程 (`Process.sequential`)**：这是默认模式，任务会按照定义的顺序严格地一个接一个执行，适用于有清晰步骤的工作流。
* **层级流程 (`Process.hierarchical`)**：在此模式下，会指定一个“管理”Agent，由它来动态决定下一步应该执行哪个任务，从而打破固定的顺序，实现更灵活的任务调度。 因此，虽然 CrewAI 的强项是自动化协作流程而非并行计算，但它通过层级流程提供了一定程度的动态性和灵活性。

3. **BabyAGI**：自主 Agent 的早期探索之一。虽然它的核心通常是一个在循环中不断创建、优先排序和执行任务的单 Agent，但其“任务驱动”的架构思想为后来的多 Agent 系统提供了重要启发，展示了如何通过任务列表来驱动 Agent 的工作流程。

理解了 Multi-Agent 这个基本概念后，你就可以把 Claude Code 中的 **Sub-agents** 和 **Agent Teams** 看作是实现这一思想的两种不同风格的“团队管理模式”。

## 1. 什么是 Sub-agent？把它想象成“临时专家”

**一句话总结 (根据官方文档):** Sub-agent 是一个拥有**独立上下文、独立工具和独立权限**的、专门处理特定任务的 AI 助手。把它想象成一个“用完即弃”的临时专家，非常精确，但不够准确——它更像是一个可以被**反复调用**的、拥有**持久化配置**的专家。

当主会话（Claude）遇到一个与 Sub-agent 职能描述 (`description`) 相匹配的任务时，会自动将工作**委托 (delegate)** 给它。Sub-agent 在自己的独立环境中完成工作，然后只将最关键的结果返回给主会话。

---

### 核心价值与特点

根据官方文档，Sub-agent 的核心价值在于：

* **隔离上下文 (Isolate Context)**：将高信息量的操作（如运行测试、搜索代码、文件读写）放在 Sub-agent 的独立“小黑屋”里进行。只有最终的摘要结果会返回给主会话，这能极大地节省主会话宝贵的上下文空间，避免被琐碎的过程信息淹没。
* **专业化与约束 (Specialize & Constrain)**：你可以为每个 Sub-agent 设置专门的系统提示（System Prompt）、受限的工具集和独立的权限。这让它能成为特定领域的专家（比如一个“只读的数据库查询员”或“精通Go语言的API开发者”），并确保它不会越权操作。
* **成本控制 (Control Costs)**：可以将特定类型的任务（如简单的代码搜索）路由给更便宜、更快的模型（如 Haiku）来执行，而将核心的、复杂的任务保留给更强大的模型（如 Opus）。
* **可复用性 (Reusability)**：定义好的 Sub-agent 可以被保存在项目或用户级别，方便在不同会话中反复调用，甚至通过版本控制与团队共享。

---

### 如何使用 Sub-agent (官方方式)

与 Agent Teams 类似，Sub-agent 的创建和使用也有一套标准方法。

#### **1. 创建 Sub-agent**

有三种主要方式来创建和定义 Sub-agent：

* **推荐方式 - 使用 `/agents` 命令**：在 Claude Code 终端中输入 `/agents`，会打开一个交互式界面，引导您完成创建、配置和管理 Sub-agent 的所有步骤。
* **手动方式 - 创建 `.md` 文件**：在特定目录（如项目下的 `.claude/agents/` 或用户目录 `~/.claude/agents/`）中创建 Markdown 文件。文件使用 YAML frontmatter 来定义 Sub-agent 的配置，正文则是它的系统提示。

  **`.md` 文件示例 (`code-reviewer.md`)：**

  ```
  ---
  name:code-reviewer
  description:Reviewscodeforqualityandbestpractices
  tools:[Read,Glob,Grep]
  model:sonnet
  ---

  Youareacodereviewer.Wheninvoked,analyzethecodeandprovide
  specific,actionablefeedbackonquality,security,andbestpractices.
  ```
* **命令行方式 - 使用 `--agents` 标志**：通过 `claude --agents '{...}'` 命令行标志，可以在启动会话时临时定义一个或多个 Sub-agent，主要用于快速测试或自动化脚本。

#### **2. 调用 Sub-agent**

创建后，有几种方式可以调用它：

* **自动委托 (Automatic Delegation)**：这是最常见的方式。当您的指令与某个 Sub-agent 的 `description` 字段高度匹配时，Claude 会自动决定调用它。
* **自然语言指定 (Natural Language)**：在您的指令中直接说 “*使用 `code-reviewer` agent 来检查我的代码*”，Claude 通常会采纳这个建议。
* **`@` 精确调用 (Explicit Mention)**：这是最可靠的方式。使用 `@` 符号，如 `@"code-reviewer (agent)" look at my recent changes`，可以**保证**`code-reviewer` 这个 Sub-agent 在当前任务中被调用。

这个经过官方文档验证的流程，才是使用 Sub-agent 的正确方式。它远比之前那个单一的 Python SDK 示例要强大和灵活。



## 2. 什么是 Agent Team？把它想象成“全职项目组”

**一句话总结：** Agent Team 是一群长期存在的、可以互相沟通协作的全职团队成员。

如果说 Sub-agent 是临时外包，那 Agent Team 就是你公司内部的一个固定项目组。这个项目组有：

* **项目经理 (Team Lead)**：负责拆解任务、分配工作、最终汇总成果。
* **团队成员 (Teammates)**：每个成员都是一个独立的 Agent，有自己的“记忆”和专长，可以并行工作。
* **共享任务板 (Shared Task List)**：所有人都能看到项目的进展，谁在做什么，谁的工作依赖谁。

**核心特点：**

* **持久存在**：Agent 在项目周期内一直存在，可以累积上下文和经验。
* **双向沟通**：团队成员之间可以直接沟通！比如前端 Agent 发现 API 接口有问题，可以直接 @ 后端 Agent 让他修改，无需事事都通过项目经理。
* **共享状态**：整个团队共享一个任务列表，任务依赖关系清晰明了。

### CLI 实战：如何使用 Agent Teams (根据官方文档)

**核心要点：Agent Teams 是一个实验性功能，需要手动启用，并且通过自然语言指令来创建和管理。**

---

#### **第一步：启用 Agent Teams 功能**

这是一个实验性功能，默认是关闭的。您需要先在 `settings.json` 配置文件中添加一个环境变量来启用它。

**`settings.json` 文件示例:**

```
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

---

#### **第二步：用自然语言创建团队**

启用功能后，您不需要使用特殊的命令或语法，而是直接用**自然语言**向 Claude 描述您想要完成的复杂任务，以及您希望的团队结构。

**官方文档中的一个优秀示例：**

```
> I'm designing a CLI tool that helps developers track TODO comments across their codebase. Create an agent team to explore this from different angles: one teammate on UX, one on technical architecture, one playing devil's advocate.
```

**对应的中文示例：**

```
> 我正在设计一个 CLI 工具，用来帮助开发者追踪代码库里的 TODO 注释。请创建一个 Agent Team 从不同角度来探索这个问题：一个队友负责用户体验（UX），一个队友负责技术架构，另一个队友扮演“魔鬼代言人”（专门挑刺和质疑）。
```

当 Claude 收到这样的指令后，它会自动理解并开始创建团队、生成“队友”(Teammate)实例，并协调它们开始工作。

---

#### **第三步：与团队互动**

团队创建成功后，您可以与“项目经理”（您当前所在的会话）和“队友”进行互动。

1. **与“项目经理”(Team Lead) 沟通**：

* 您可以继续用自然语言给它下达指令，比如“让架构师队友先提交一份草案”或“让所有人都把他们的发现汇总一下”。

2. **直接与“队友”(Teammate) 沟通**：

* 这是 Agent Teams 的核心特性。在默认模式下，使用 `Shift + Down` 快捷键可以在“项目经理”和各个“队友”的会话之间循环切换。切换到某个队友后，您就可以直接输入文字和他单独对话。

---

#### **第四步：清理团队**

当任务完成时，您可以告诉“项目经理”来结束整个团队任务。

**指令示例：**

```
> Clean up the team
```

“项目经理”会负责关闭所有“队友”并移除相关资源。

---

#### **总结**

根据官方文档，正确的 Agent Teams 使用流程是：

1. **一次性启用**：通过修改配置，启用实验性功能。
2. **自然语言驱动**：用一段清晰的自然语言来描述任务和团队结构，从而**创建团队**。
3. **直接互动**：用 `Shift + Down` 快捷键**切换并直接与每个队友对话**。
4. **指令清理**：任务结束后，用明确指令**清理团队**。

这才是经过官方文档验证的、准确的操作方式。

## 3. 开发实战：到底怎么选？

| 特性 | Sub-agents (临时专家) | Agent Teams (全职项目组) |
| --- | --- | --- |
| **通信** | 只能与主 Agent 单向通信 | 成员间可直接、双向通信 |
| **状态** | 无共享状态，即用即弃 | 共享任务列表，持久存在 |
| **复杂度** | 简单，易于管理 | 复杂，但能处理依赖性强的任务 |
| **适用场景** | 独立的、可并行的研究或查询 | 需要持续协商、协作的复杂项目 |

**选择指南：**

1. **问问自己：这个任务可以被完美地拆分成几个互不相干的子任务吗？**

* **是** -> 用 **Sub-agents**。
* 例如：分析一篇文章的“语法”、“情感”和“关键信息”。

2. **问问自己：子任务之间是否存在依赖，或者需要互相“商量”着来？**

* **是** -> 用 **Agent Teams**。
* 例如：开发一个新功能。前端需要等后端定义好 API 接口，后端可能需要根据前端的需求调整数据结构。这种需要来回沟通、协调的场景，正是 Agent Team 的用武之地。

## 4. 总结：从简单开始，按需选择

* **Sub-agents** 适合执行可以 **“极致并行”** 的独立任务，核心价值在于 **“压缩”** 上下文，保持主 Agent 的清晰。
* **Agent Teams** 适合处理需要 **“持续协商”** 的复杂任务，核心价值在于 **“协作”** 和状态共享。

**最重要的原则：** 不要为了用多 Agent 而用多 Agent。先尝试用一个调教得当的单 Agent 解决问题。只有当你发现任务确实可以从“上下文隔离”或“团队协作”中显著受益时，再引入相应的多 Agent 模式。

始终围绕 **“上下文边界”** 来设计你的 Agent，而不是角色。如果两个任务需要高度重叠的信息，强行拆分给两个 Agent 反而会增加沟通成本。

选择标准很简单：

任务是可以"embarrassingly parallel"（极致并行）的吗？独立的研究流、代码探索、查询，这类场景用 Sub-agent，主 Agent 只需要最终汇总。

任务需要持续协商吗？Agent 之间要互相协调（reconcile）输出，或者一个线程的发现会影响另一个线程该做什么，这类场景用 Agent Teams。

## 5. 大部分Multi-Agent 设计失败的原因

大部分多智能体（Multi-Agent）系统之所以设计失败，是因为错误地按照“角色”来拆分任务，而正确的做法应该是按照“上下文”来拆分。

#### 1. 错误的方式：按角色拆分 (Splitting by Role)

* **是什么**：这是一种很直观的分工方式，就像组建一个公司团队一样，分为`规划师 (Planner)`、`代码实现者 (Implementer)`、`测试员 (Tester)`等。
* **为什么会失败**：这种方式会变成一场“传话游戏”（Game of Telephone）。每个环节都在丢失关键的“上下文”，导致最终产出的质量层层下降。

#### 2. 正确的方式：按上下文拆分 (Splitting by Context)

* **是什么**：在分配任务前，先问一个核心问题：“**完成这个子任务，到底需要哪些信息和知识（也就是上下文）？**”
* **如何操作**：

+ 如果两个子任务需要高度重叠的信息，那它们就应该由**同一个 Agent** 来完成，以避免信息传递的损耗。
+ 只有当一个子任务可以用**完全独立、隔离的信息**来完成时，才应该把它拆分出去。

#### 总结

设计的核心应该是**信息流的效率**。让掌握信息最多、最全的 Agent 去完成所有依赖这些信息的任务，才能最大化地保证质量和效率。

---

实话说，大部分Multi-Agent设计失败不是因为开发技术不行，而是因为按角色拆分工作，而不是按上下文拆分。

直觉上按角色拆分很合理：Planner、Implementer、Tester，看起来很清晰。但这就成了"传话游戏"，信息在每个交接点都会衰减。

* Implementer 没有 Planner 知道的东西
* Tester 没有 Implementer 决定的东西
* 质量在每个边界都在下降



**正确的思路是"以上下文为中心"拆分。**

可以问问自己：这个子任务实际需要什么上下文？如果两个子任务需要高度重叠的信息，它们可能该属于同一个 Agent。如果它们可以用完全隔离的信息和干净的接口运作，那就应该拆分。

一个实际的例子是：实现某个功能的 Agent，也应该同时写这个功能的测试。它本来就有上下文。把这两个拆到不同的 Agent，产生的交接成本比并行的收益还大。

只有当上下文能够真正隔离的时候，才应该拆分。

## 6. 五种常见模式

不管用哪种范式，下面这五种模式覆盖了大部分实际需求：



| 模式 (Pattern) | 核心逻辑 (Core Logic) | 适用场景 (Applicable Scene) |
| --- | --- | --- |
| **1. 提示词链 (Prompt Chaining)** | **顺序执行** ：前一步的输出是后一步的输入，像流水线一样。 | 有明确前后依赖关系的任务，如“先总结文章，再根据总结写一封邮件”。 |
| **2. 路由 (Routing)** | **智能分发** ：一个“分类器”Agent 根据任务的复杂度或类型，决定将其发送给哪个专门的“处理者”Agent。 | **成本与性能优化** ：用廉价、快速的模型处理简单任务，用昂贵、强大的模型处理复杂任务。 |
| **3. 并行化 (Parallelization)** | **同时运行** ：将多个独立的子任务分发给不同的 Agent 同时处理。 | 独立的研究、为同一问题生成多种解决方案（投票选优）、或分块处理问题的不同部分。 |
| **4. 编排者-工作者 (Orchestrator-Worker)** | **总分结构** ：一个中心的“管理者”Agent 负责分解和分派任务给多个“工作者”Agent，并最终汇总结果。 | **通用基础模式** ：这是大多数复杂多智能体系统（包括 Sub-agent 和 Agent Team）的基础架构。 |
| **5. 评估者-优化者 (Evaluator-Optimizer)** | **循环迭代** ：一个“生成者”Agent 负责产出内容，另一个“评估者”Agent 负责检查质量并提供反馈，循环往复直到满足要求。 | 对产出**质量要求极高**的场景，如代码优化、文稿润色、或生成需要严格遵守格式的复杂回答。 |

## 7. 什么时候才需要多 Agent

很多团队花了几个月构建复杂的多 Agent 流水线，最后发现用更好的 Prompt 调教单个 Agent 就能达到同等效果。

从简开始。 **只有能够明确衡量出复杂性带来的收益时，才往上加。**

真正需要多 Agent 的场景：

* **某个子任务产生的上下文信息对主任务无关，保持在 Sub-agent 里防止上下文膨胀**
* **独立的研究或搜索任务，确实受益于同时覆盖**
* **任务需要互相冲突的系统提示词，或者单个 Agent 工具太多导致性能下降**

不需要多 Agent 的场景：

* Agent 之间需要频繁共享上下文
* 跨 Agent 的依赖产生的管理开销比执行价值还大
* 任务足够简单，一个 Prompt 调教得好的单 Agent 就能搞定

涉及代码的场景，并行 Agent 写代码会产生不兼容的假设。合并工作时，那些隐式决策会产生难以调试的冲突。写代码相关的 Sub-agent 应该用来回答问题和探索，而不是和主 Agent 同时写代码。

## 8. 三个常见的失败模式

| 失败模式 (Failure Mode) | 核心原因 (Root Cause) | 解决方案 (Solution) |
| --- | --- | --- |
| **1. Agent 重复劳动** (Agent Duplicates Work) | **任务描述模糊** ：Agent 没有被告知明确的目标、输出格式、工具和边界，导致它们可能会研究同一个问题却互不知晓。 | **指令必须清晰** ：在分配任务时，要提供明确的目标、期望的输出格式、推荐使用的工具/信息来源，以及不应触及的范围。 |
| **2. 假装任务已完成** (False Completion) | **验收标准模糊** ：负责验证的 Agent 没有得到具体的验证指令，导致它没有真正地去验证就草率地宣布任务完成。 | **验收标准必须具体** ：给出明确的验证指令，例如“运行完整的测试套件，覆盖 A、B、C 三个场景，必须全部通过才算完成”。 |
| **3. Token 成本失控** (Token Costs Spiral) | **模型使用不当** ：所有任务，不论简易，都使用了最强大、最昂贵的模型，导致成本迅速攀升。 | **实施智能分层策略** ：  1. 在最关键的地方使用最强模型。  2. 将常规、简单的任务**路由**给更快、更便宜的模型处理。  3. 建立预算控制和监控机制。 |

## 9. 最后的行动指南

我们已经走过了从概念到实战的全过程，现在，让我们将所有关键智慧，提炼成一套简洁而强大的行动指南。

1. **从“一”开始，而非从“多”开始**始终从一个调优良好的**单 Agent** 开始。在遇到它的能力或上下文边界之前，不要过早地引入多智能体的复杂性。很多时候，一个设计精巧的 Prompt 比一个复杂的 Agent 系统更有效。
2. **以“上下文”，而非“角色”为拆分依据**当你不得不增加复杂性时，请牢记黄金法则：**围绕上下文边界进行设计，而不是角色。** 问自己：“这个子任务能否在信息完全隔离的情况下完成？”

* 如果答案是**是**，那么它适合成为一个独立的 **Sub-agent**。
* 如果答案是**否**，任务之间需要共享大量信息，那它们应该留在**同一个 Agent** 中。

3. **明确场景，选择正确的“交通工具”**

* 当任务需要多个部分**持续协商、共享状态、紧密协作**时，选择 **Agent Team** 模式（就像一个项目团队开着一辆“汽车”）。
* 当任务可以分解为多个**并行、独立、用完即回**的子任务时，选择 **Sub-agent** 模式（就像派出多个“摩托车手”分头行动）。

4. **只为“收益”而非“炫技”增加复杂性**最后，请记住，多智能体系统本身不是目标。每增加一个 Agent，都应该是为了解决一个真实且可衡量的问题——无论是为了隔离上下文、突破单一模型的认知上限，还是为了并行提速。**只在收益明确大于管理开销时，才值得引入新的 Agent。**
