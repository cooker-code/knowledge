## dmux到底是什么？简单说就是AI代理的“并行管家”

dmux全称有点长：dev agent multiplexer，翻译过来就是开发者AI代理多路复用器。

它把tmux（终端多路复用工具）和git worktree（工作树）结合在一起。每个AI代理跑在一个tmux窗格里，每个窗格自动创建一个独立的git工作树，分支也单独生成。

结果呢？几个代理同时改代码也不会冲突。想让Claude写前端逻辑，让另一个代理优化后端API？直接一起上，互不打架。

比起传统方式——手动开N个终端、手动切分支、手动merge——dmux把这些全自动化了。

它还支持多项目同时塞到一个tmux session里。前端项目和后端项目一起管，键盘一按就能切换。

官网上写得清楚：提供11个AI编码代理支持，包括Claude Code、Codex、OpenCode、Cline CLI等等。如果你装了OpenRouter API Key，它还能自动生成分支名、commit消息，省掉更多手动操作。

一句话总结：dmux把“多个AI代理一起干活”这件事，从折腾变成了享受。

## dmux 5.5三大杀手级特性，直接解决你的日常痛点

这次更新重点就三件事，每一件都戳中开发者心窝。

**第一，macOS系统通知 + 一键跳回窗格**

以前代理在后台跑，你切到浏览器或者其他窗口，它突然需要你确认或者有输出，你根本不知道。

dmux 5.5内置了原生macOS通知。作者在回复里解释得很清楚：它背后有个原生app bundle，能精确知道哪个tmux窗格、哪个终端窗口需要关注。

如果窗格没聚焦，就闪背景；如果整个窗口在后台，就直接弹出macOS通知，点一下就能精准跳回那个代理窗格。

太贴心了！再也不用每隔几分钟切回终端看进度，通知主动喊你回来。

**第二，每个项目的独立文件浏览器，基本可以扔掉大IDE了**

新特性里最实用的就是“per-project file browser”。

按一下`f`键，弹出只读文件浏览器。支持按文件名、修改时间、git状态排序，还能预览文件内容和diff对比。

作者在推文里直呼“dump your ide”，意思就是你大可以把VS Code或者其他IDE关掉，只用dmux的浏览器就够看代码、看改动。

对多项目场景特别友好：不同项目有自己的浏览器，不会混在一起。想快速检查某个代理改了哪几个文件？秒开秒看。

**第三，窗格和项目的显示/隐藏控制**

按`h`隐藏选中窗格，按`H`隐藏其他所有窗格。隐藏后代理继续后台运行，侧边栏还有标记。

多项目模式下，按`P`就能只显示当前项目的窗格，其他项目暂时收起来，界面瞬间清爽。

这三个特性组合起来，dmux从“能用”直接进化到“舒服到飞起”。

## 实战上手：5分钟安装 + 跑起你的第一个多Agent会话

想试试？操作超级简单，不用配置一堆环境变量。

**步骤1：安装**
打开终端，执行：
`npm -g i dmux`

前提是电脑已经装了tmux和Node.js（大部分Mac开发者都有）。

**步骤2：准备AI代理**
确保PATH里能找到你喜欢的AI编码代理，比如Claude Code或者Cursor CLI。

推荐再加个OpenRouter Key：
在`~/.zshrc`里加一行
`export OPENROUTER_API_KEY=你的key`
然后`source ~/.zshrc`

**步骤3：启动dmux**
`cd`到你的git仓库目录
输入`dmux bash`

它会自动创建一个以项目命名的tmux session，打开TUI界面。

**步骤4：创建第一个代理窗格**
按`n`新建窗格
输入你的prompt
选择代理（支持A/B对比模式，两个代理同时跑同一个任务）
回车，代理就开始干活了！

**步骤5：常用操作**

* • `f` 打开文件浏览器，查看代码和diff
* • `j` 直接跳到某个窗格
* • `h` / `H` 隐藏窗格
* • `m` 触发merge（两阶段安全合并：main→worktree→main，AI自动写commit消息）
* • `P` 只显示当前项目窗格
* • `t` 打开纯终端窗格
* • `x` 关闭窗格
* • `s` 打开设置

整个流程键盘优先，鼠标都几乎用不上。

我按照官网步骤测试了一下，从安装到第一个代理跑起来，真就5分钟不到。隐藏窗格后还能收到通知，体验拉满。

## 为什么dmux能让AI开发效率直接起飞？

核心在于“隔离 + 自动化 + 可视化”。

git worktree让每个代理都有完整独立的工作目录，共享.git却不冲突。tmux负责窗口管理，dmux在中间当管家：自动创建分支、自动merge、自动清理、甚至提供11个生命周期hook让你写自定义脚本。

实际场景里特别好用：

* • A/B测试：两个不同代理同时改同一个功能，对比结果
* • 并行开发：一个代理写新特性，另一个fix bug
* • 全栈项目：前端后端塞同一个session，随时切换项目浏览器
* • 长时间任务：代理在后台跑，你切去开会，通知会喊你回来确认

比起单个代理单打独斗，dmux直接把生产力乘以N。

而且它是完全开源的，GitHub仓库standardagents/dmux，MIT协议，想改就改。
