# 概念

子代理是 Claude Code 可以委派任务的专用助手。您可以将它们视为专注的助手：每个子代理都在其独立的对话上下文窗口中运行，执行其工作，并将摘要返回给主线程。中间步骤——所有文件读取、搜索和工具调用——都保持隔离，不会干扰主对话。

# 为什么需要子代理

每次你和 Claude Code 聊天，都会向主上下文窗口添加内容。每次工具调用、每次文件读取、每次搜索结果都会存储在那里。这个空间是有限的，一旦填满，Claude 就会开始丢失对话的早期内容。

子代理通过启动一个独立的上下文窗口来解决这个问题。子代理接收两项信息：

* 配置文件中的**自定义系统提示，用于定义子代理的角色和行为**
* 由上级代理根据您的要求编写**的任务描述**

子代理随后独立运行。它读取文件、运行搜索、编辑代码——执行任何需要执行的操作。完成后，只会向主对话返回一个摘要。**整个子代理对话随后会被丢弃。**

这意味着你的主要上下文保持清晰。你只需获取答案，无需了解寻找答案过程中的所有干扰信息。但缺点是，你无法了解子代理是如何得出结论的。

子代理能为您带来三大好处：

* 他们将工作分解成若干个专注的小部分，让每个子代理人专注于一项特定的任务。
* 它们通过隔离所有中间工作来保持主上下文窗口的简洁。
* 他们会带回你需要的全部信息，并以简洁明了的方式进行总结。

# 什么才算有效的子代理

有效的子代理具有以下几个特征：

1. 具体描述——该描述控制子代理何时启动以及接收哪些指令， 以便更好的控制子代理。
2. 结构化输出——在系统提示符中定义输出格式，以便子代理知道何时完成并返回主线程可以使用的信息。
3. 障碍报告——在输出格式中包含一个部分，用于记录解决方法、怪癖和问题，以便主线程不必重新发现它们。
4. 限制工具访问权限——仅向子代理提供其实际需要的工具。研究人员只能读取工具，审阅人员只能使用 bash 脚本，需要修改代码的代理只能编辑/写入工具。

选择是否需要使用子代理的方法：
 在决定是否使用子代理时，请问自己一个问题：**中间工作重要吗？**

如果答案是否定的——你只需要最终结果——那就把它委托给子代理。如果答案是肯定的——你需要了解并应对过程中发生的情况——那就把它放在主线程中。

使用子代理进行以下操作：

* 研究与探索
* 代码审查
* 需要自定义系统提示的任务

避免使用子代理的情况：

* 那些无法真正提升能力的“专家”角色
* 多步骤流水线，其中每一步都依赖于前一步。
* 运行需要完整输出以进行调试的测试
