> **这篇文章会带你过一遍 Claude Code 最近几个版本新增的 Slash 命令，包括它们分别能做什么、适合在什么场景下使用，以及它们该怎么接进你的工作流。**

本文会讲到：

* • `/effort`：控制 Claude 每个任务的思考深度
* • `/loop`：按固定时间间隔重复运行提示词
* • `/copy`：从回复里挑选并复制指定代码块
* • `/color`：给当前会话的提示栏加颜色
* • `/simplify` 和 `/batch`：面向常见工作流的组合命令
* • `/reload-plugins`：无需重启即可热重载插件变更

下面快速过一遍。

这条命令很适合拿来平衡速度和成本。

**在有**`/effort`**之前，你得先打开**`/model`**菜单，再用方向键去调 effort 滑块。**

能用是能用，但它藏在负责模型切换的菜单里，不够顺手。

现在你只需要输入：

```
/effort
```

你会看到三个等级：low、medium 和 high，对应的标识也很直观：

* • `_○_`*Low：跳过深度推理，速度快，成本低。*
* • `_◐_`*Medium：平衡模式，适合大多数编码任务。*
* • `_●_`*High：完整推理能力，适合更复杂的问题。*

如果你想恢复默认设置，可以运行：

```
/effort auto
```

这样 Claude 就会根据任务复杂度自行决定使用哪一档 effort。

对 Max 和 Team 订阅用户来说，Opus 4.6 默认使用 medium effort，这基本就是大多数工作的最佳平衡点。

> **但如果你正在排查一个棘手的多文件问题，就会想开 high effort；而如果你只是改变量名、整理格式，low effort 能省下不少 token 和时间。**

`/effort` 现在还能在 Claude 回复过程中切换。所以如果你在它回答到一半时意识到下一轮需要更深的推理，也不用等当前回复结束。

之前你也可以在提示词里直接输入 “ultrathink”，让 Claude 在单轮对话里进入高 effort 模式。

这个方式现在依然有效，但 `/effort` 给了你贯穿整个会话的持续控制能力。

### 适用场景

* • **Low：** 简单修改、快速提问、格式整理、变量重命名
* • **Medium：** 日常编码、常规功能开发、通用任务
* • **High：** 架构决策、复杂调试、多文件重构、代码审查

## `/loop`：自动化重复提示词

`/loop` 可以按固定时间间隔重复执行一个提示词或 Slash 命令。你设定周期、给出任务，Claude 就会自动按时运行。

```
/loop 5m check the deploy
```

这会让 Claude 每 5 分钟检查一次部署状态。

你也可以把它和任意提示词或 Slash 命令搭配使用：

```
/loop 10m /stats
```

这会每 10 分钟运行一次 `/stats`，方便你在长会话里跟踪 token 消耗。

### 适用场景

它很适合用在这些任务上：

* • *在长时间编码会话里盯着正在进行的部署*
* • *改完代码后定时查看测试结果*
* • *持续观察 CI 流水线里的构建失败*
* • *对项目做周期性的健康检查*

底层上，`/loop` 接入了同一版本里新增的 cron 调度能力。

> **Claude Code 现在已经可以在单个会话里处理周期性提示词，不再需要额外写 cron 任务或脚本。**

如果你想停掉循环，可以设置这个环境变量：

```
CLAUDE_CODE_DISABLE_CRON=1
```

这会在当前会话里终止所有已调度的 cron 任务。

### 小提示

> `/loop`**最适合轻量、只读的检查任务。把重量级代码生成塞进循环，很快就会把 token 烧光。它适合做监控和状态检查，不适合反复构建或大规模重构。**

## `/copy`：精确复制你真正需要的内容

在 `/copy` 出现之前，从 Claude Code 的回复里拿代码并不轻松。

> **你要么往上翻，手动框选需要的代码块；要么把整段回复全复制下来，再自己删掉多余部分。**`/copy`**用一个交互式选择器把这件事解决了。**

当 Claude 的回复里包含代码块时，输入 `/copy` 就会弹出选择菜单。

> **每个代码块都会单独列出来，你只要选中自己真正需要的那一个。**

```
/copy
```

你会看到类似这些选项：

* • *代码块 1（React 组件）*
* • *代码块 2（API 路由）*
* • *代码块 3（测试文件）*
* • *整条回复*

选中之后，内容就会被复制到剪贴板。

### 通过 SSH 直接写入文件

如果你是在 SSH 环境下工作，剪贴板能力通常比较受限，这时 `/copy` 就特别实用。

> **当焦点停在某个选项上时，按下**`w`**，它会直接把内容写入文件，绕过剪贴板。**

这是个很小的细节，但如果你曾在 tmux 或远程会话里被剪贴板折腾过，就会知道它有多省事。

### “始终复制整条回复”模式

如果你发现自己几乎每次都在复制整条回复，而不是单独复制代码块，也有更省事的做法。

> **在选择器里选中 “Always copy full response” 后，之后的**`/copy`**就会直接跳过代码块菜单，每次都复制整条回复。**

这个行为随时都能在选择器里重置回来。

### 适用场景

* • *从一个包含多文件内容的回复里抓出某个特定函数*
* • *只复制一段配置，而不要周围的说明文字*
* • *在 SSH 环境里工作，因为手动选择内容不够可靠*
* • *快速提取代码片段，粘贴到另一个文件或工具中*

## `/color`：给会话加颜色标记

这个命令很简单，但当你同时开着多个 Claude Code 会话时会非常有用。

```
/color
```

它会为当前会话的提示栏加上颜色。

> **如果你同时开着三个终端，一个管前端，一个管后端，一个管基础设施，就可以给每个会话分配不同颜色。**

0

只看一眼，你就知道自己当前正在和哪个会话交互。

### 重置颜色

如果你想恢复默认颜色，下面这些命令都可以：

```
/color default
/color gray
/color reset
/color none
```

1

它们都会把提示栏恢复成原来的样子。

### 和 `/rename` 搭配使用

`/color` 和 `/rename` 一起用会更顺手。

> **现在提示栏里还会显示会话名称，所以如果再配上像 “auth-service” 或 “frontend-v2” 这样的描述性名字，多会话管理会轻松很多。**

## `/simplify`：并行调用 Agent 审查你的代码

我在 `/simplify` 刚发布时就专门写过一篇，到现在它依然是 Claude Code 里最被低估的命令之一。

`/simplify` 更像是实现完成之后的清理步骤。你先把功能写出来、跑通，然后执行：

```
/simplify
```

3

它会先查看 `git diff`，弄清楚你改了哪些内容，然后并行拉起 3 个专门的 agent：

* • **代码复用 agent：** 查找重复逻辑并进行合并
* • **代码质量 agent：** 捕捉代码异味，并执行你在 `_CLAUDE.md_` 里定义的规则
* • **效率 agent：** 指出性能问题和结构优化点

这 3 个 agent 会同时运行。

> **它们结束后，**`/simplify`**会汇总结果，并一次性把修复应用上去。**

你也可以让它只聚焦某个特定方向：

```
/simplify focus on error handling
```

> 在我的测试里， `/simplify`**在不到 90 秒里，把 69 行混乱代码压缩到了 33 行。它识别出原始 SQL 字符串拼接，把 CommonJS 迁移到 ES modules，用数组方法替换嵌套循环，还删掉了死代码；而这一切都依据 CLAUDE.md 里的规则完成。**

## `/batch`：大规模并行迁移

4

如果说 `/simplify` 是代码质量整理步骤，那么 `/batch` 就是迁移引擎。

> **它处理的是那种通常会让你花上一整个下午，对着几十个文件做查找替换、还生怕漏掉什么的工作。**

```
/batch migrate src/ from Jest to Vitest
/batch replace all uses of lodash with native equivalents
/batch add type annotations to all untyped function parameters
```

### 它是怎么工作的

* • **阶段 1：研究和规划。**`/batch` 会先进入 plan mode。它会遍历整个目录、扫描每一个文件，并生成一张迁移表，列出哪些文件需要修改，哪些文件已经符合要求。随后它会把完整计划展示给你，在真正写入任何一行代码之前等待你的批准。
* • **阶段 2：并行执行。** 在你批准之后，它会把任务拆成相互独立的单元，并在隔离的 git worktree 中拉起 agent。每个 agent 都在自己独立的代码副本里工作。
* • **阶段 3：验证与 PR。** 每个 worker 在改完后都会跑验证检查。它能区分真正的失败，以及演示环境里缺少依赖这类预期错误。worker 在创建 PR 前还会先跑一遍 `/simplify`，所以每个 pull request 都像是提前做过一次审查。

### `/simplify` 和 `/batch` 的区别

> `/simplify`**是针对你最近改动做质量检查的命令，适合在功能完成后运行。**`/batch`**则适合做大规模迁移，也就是你需要把某种模式应用到几十个文件的时候。**
>
> `/batch`**会先思考、展示计划，然后停下来等待。最后这一点非常关键，因为你可以在它真正动手之前，先审查它的拆解方案。**

## `/reload-plugins`：无需重启即可热重载插件变更

如果你写过 Claude Code 插件，就知道这个痛点。

> **你改了一个插件，但在重启会话之前什么都不会发生。这意味着你得丢掉当前上下文、重新恢复工作状态，还要等所有内容重新加载。**

`/reload-plugins` 就是为了解决这个问题。

```
/reload-plugins
```

5

> **它会在不重启当前会话的前提下启用待生效的插件变更。你改完插件代码后，执行**`/reload-plugins`**，改动就会立刻生效。**
