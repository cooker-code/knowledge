很多人用 Claude Code 做内容时，最容易先冲去堆 Skill、调提示词、改工作流。但这篇材料真正点破的是另一件事：如果每次运行都像第一次见客户，AI Agent 再勤快，产出也很容易“像是干了活，但谁都不像”。

这 4 个 `.md` 文件之所以值得看，不是因为它们神秘，而是因为它们在帮 Claude Code 建一层共享上下文。读完你会更清楚：为什么有些团队越用越顺，有些团队却总在重启同一场对话。

## **1** 很多 Claude Code 工作流的问题，不是不会写，而是每次都从零开始

原文一开头用了一个很好懂的类比。1995 年皮克斯做《玩具总动员》时，团队并不是没有天赋，而是每做一部片子，都在重复踩相似的坑。一个导演组刚解决完叙事问题，另一个组又会在下一部片子里重新撞上去。经验散落在个人脑子里，没有变成共享系统。

后来 Ed Catmull 推了 **Braintrust**。你可以先把它理解成一个“共享创作脑”。导演会拿半成品来过会，团队一起判断哪里有效、哪里失灵、为什么会失灵。迪士尼 2006 年收购皮克斯后，又把这套方法迁到了迪士尼动画的 **Story Trust**。至少从原文给出的例子看，同样一批人、同样的预算，换了一层共享上下文，结果就开始变了。

这正是很多人今天在 **Claude Code** 里遇到的问题。大家会做很多 Skill：写 LinkedIn 帖子、写 newsletter、写广告文案，看起来每个都能跑。但它们经常是各写各的，每次都从零起步，不共享受众认知、不共享表达风格、不共享竞品判断，也不共享哪些内容更容易转化。

于是结果常常不是不能用，而是 **“能交差，但很像模板”**。内容平平、语气发虚、谁都能套，像任何一家 SaaS 公司都能发出来的那种话。原文的核心判断也落在这里：问题未必是 Skill 不够多，更可能是缺少一层先于 Skill 的公共底座。

> **Foundation Layer**，可以先理解成 Claude Code 开工前的“预备知识层”。它不是多写几个提示词，而是先把“这家公司在跟谁说话、怎么说、和谁竞争、用户卡在哪里”这些前提条件放进去。

## **2** 所谓 Foundation Layer，本质上是在 Skill 下面再铺一层“共享语境”

原文专门强调了一点：真正让这套系统跑起来的，不是 Skill 本身，而是 Skill 下面那一组 `.md` 文件。它们教给 Claude 的，不是抽象的品牌口号，而是更具体的东西：你的受众是谁、他们怎么说话、你的市场位置在哪、用户从认识你到下单中间会卡在哪一步。

这里很容易和大家熟悉的 **`CLAUDE.md`** 混淆。原文说得很清楚，这不是那类系统配置文件，而是营销团队自己整理出来的业务上下文文件。换句话说，`CLAUDE.md` 更像“这个 Agent 在这个项目里该怎么做事”，而这里讨论的 4 个文件，更像“这个品牌在现实世界里到底是谁，在和谁说话”。

这些文件被统一放在一个 `/foundation/` 目录里，Skill 执行前先去读它们。这样一来，Claude Code 就不必每次都重新猜一遍你的 audience、tone、service 和 product。对大众读者来说，这件事可以简单理解成：**你不是每次临时给 AI 塞一句提示，而是在给它准备长期有效的背景资料。**

原文还特别提到，有没有这层底座，效果差异往往不是“小修小补”，而是接近天壤之别。没有 foundation files 时，一个定位 Skill 很容易写出谁都能套用的空泛文案；有了它之后，Claude Code 才知道你的受众说的是 **“agentic workflows”**，而不是泛泛的 “AI tools”；你的语气是直接、克制、**anti-hype**，而不是一股过度兴奋的营销腔；你的竞争对手也不是抽象的“其他产品”，而是几个明确的替代方案。

原文还补了一个非常重要的设计细节：Skill **不会** 每次把所有 foundation 文件全读一遍。因为那样做，反而会把上下文冲淡。作者给了三个例子：写 LinkedIn 帖子的 Skill，会读 `Audience Delight` 和 `Creator Style`；产品发布文案会读 `Market Positioning` 和 `Customer Journey`；培育邮件则可能同时读前三者中的多个。这个逻辑很像写代码时的 `import`，**按需引入，而不是一把全塞。**

> 这一步很关键。大模型的**上下文窗口**，简单说就是它一次性能带着走的信息量。不是塞得越多越聪明，很多时候是塞得越杂越容易跑偏。

## **3** 这 4 个 `.md` 文件，分别在补 Claude Code 最容易缺的 4 块信息

### **3.1** Audience Delight Profile：别再只写 ICP，先把用户的“原话”交给它

原文第一个文件叫 **Audience Delight Profile**，它想替换的，是传统的 **ICP**。ICP，也就是 Ideal Customer Profile，常被写成“200 到 500 人 SaaS 公司的 VP，关注 pipeline efficiency”这种幻灯片语言。问题在于，这种话给人看也许还能忍，给 AI Agent 写内容时就太空了。

这个文件真正要抓的，是受众会为什么东西兴奋、会自发分享什么、会在品牌不在场时抱怨什么，更重要的是：**他们到底怎么说话。** 原文举的例子非常典型。用户吐槽差的 AI 结果时，常说的是 “slop”“garbage output”，而不是 “inaccurate results”；谈上线时会说 “ship it”“running in prod”，而不是 “implement the solution”。

这背后的意思很直接：你把词用错了，内容就像厂商在讲话；你把词用对了，内容才像圈内人、像同类、像真的理解他们。原文甚至给了 Notion 的一个完整示例。比如用户会说 “my Notion”“my setup”“my system”，而不是 “the platform”；会说 “second brain”“single source of truth”，而不是 “knowledge management system”；会说 “tool sprawl”“tab hell”，而不是 “technology stack consolidation”。

这类信息为什么重要？因为很多面向用户的 Skill，本质上不是在“生成句子”，而是在“判断什么词会被听进去”。如果 Claude Code 手里拿到的是用户原话，它写广告、邮件、脚本时就更容易贴着真实语境走，而不是飘在企业黑话上面。

原文给的构建方法也很实用：把 Reddit 讨论、销售通话记录、客服工单、竞品评论区喂给 Claude，让它从用户语言里抽取画像，而不是从市场部自己写的介绍里倒推画像。这个顺序不能反，因为前者更接近现实，后者更接近自我想象。

### **3.2** Creator Style：不要再写“温暖、专业、可信”，要把风格拆成模式

第二个文件是 **Creator Style**，它替代的是传统品牌手册里那种“我们的声音是 warm、professional、empowering”式描述。原文的判断很尖锐：这种词几乎适用于任何公司，所以对 AI 来说基本没有校准价值。

更有用的写法，是从你已经验证过的内容里，反推出你“实际上怎么写”。原文说，这个文件抓的不是形容词，而是 **pattern**。比如“短句先出手，长句负责解释”“允许用碎片句制造节奏”“先讲用户能做什么，再讲你做了什么”。这些描述听上去不花哨，但对模型来说反而最有用，因为它们是可以模仿的。

[图片：image/the-4-md-files-that-turn-claude-code/03.webp｜03.webp，Creator Style 中 “Sounds Like Us vs. Doesn’t Sound Like Us” 的校准示例]

原文最值得借鉴的一部分，是“像我们”和“不像我们”的并列表。以 Notion 为例，“Your workspace should adapt to how you work, not the other way around” 就更像品牌本人的说法；而 “Our platform provides a streamlined solution for team collaboration” 虽然语法没问题，但一股标准企业宣传册味。再比如，“Build your own CRM. Or a wiki. Or a project tracker. Or all three.” 这种句式，和 “Notion offers comprehensive project management capabilities” 带给人的感觉，几乎不是同一种物种。

这也是为什么原文强调，**风格不是 vibe，而是可以被比较、被验证、被排除的。** Claude Code 一旦能看到这种“对照样本”，它就不只是知道“要更口语化”，而是知道“哪种句子像你，哪种不像你”。

作者给的方法是：找出你表现最好的 10 篇内容，让 Claude 去归纳它们共同的语气、节奏和结构；然后再拿效果不好的内容反向验证。能同时解释“赢家为什么像你”和“输家为什么不像你”，这个风格文件才算站住。

### **3.3** Market Positioning Map：不是写一页竞品对比，而是持续更新“你站在哪”

第三个文件叫 **Market Positioning Map**。它替代的，不是单次发布会前那张竞品对比表，而是“上季度做完就再也不打开”的定位幻灯片。原文把它定义成一份持续更新的 `.md` 文件，用来跟踪竞品现在到底在 claim 什么、哪些位置是你真正拥有的、哪些已经变成所有人都在喊的拥挤地带、还有哪些位置你干脆不争。

这里最有价值的是原文提出的三块区域。第一块是 **你拥有的 territory**，也就是别人不太能可信复制的位置。以 Notion 为例，作者认为它拥有的是 “flexibility without complexity” 以及社区生态，这种模板、视频、用户自发创作一起堆出来的外溢效应，不是竞品喊一句就能拿走的。第二块是 **竞争中的 territory**，像 “all-in-one workspace” 或 “AI-powered” 这种，谁都能讲，谁也很难独占。第三块是 **你主动让出去的 territory**。原文举例说，Notion 并不和 Asana 去硬拼深度项目管理方法论，知道自己不抢什么，和知道自己抢什么一样重要。

这类文件为什么会影响 Claude Code？因为所有涉及发布信息、差异化表达、对外主张的 Skill，都默认在“替品牌占一个位置”。如果它不知道你拥有什么、不拥有什么，就很容易把你写到一块根本不属于你的地盘里去。

原文建议的构建方式也很朴素：去审竞品官网首页，看他们现在到底在讲什么；去看 G2 评价，看客户自己是怎么比较你和别人的；再回看自己的赢单、输单记录。这里最重要的提醒不是技巧，而是频率。**市场定位不是季度 PPT，它更像一张需要月更的活地图。**

### **3.4** Customer Journey Intelligence：真正决定转化的，往往不是阶段图，而是卡点

第四个文件是 **Customer Journey Intelligence**。它想替换的，是墙上那张“认知、考虑、转化、留存”的漏斗图。原文认为，传统 journey map 最大的问题是静态，只画阶段，却抓不住用户在哪一刻真的决定购买、在哪一刻停住、在哪一刻流失。

所以这份文件记录的，不只是流程，而是 **真实机制**：用户怎么发现你、评估时会问什么、哪些证据最能推动成交、他们通常卡在哪里、什么事件会触发扩容。原文特别强调，这里关注的不是“你希望用户做什么”，而是“用户实际上会做什么”，尤其是他们会在哪里卡壳。

Notion 的例子也很具体。原文说，它一个典型卡点是 **blank canvas paralysis**，也就是用户注册后看到一张空白工作区，不知道从哪开始；另一个高频流失原因是团队采用失败，一个内部 champion 把空间搭起来了，但其他人根本没跟上。于是，如果一个 nurture email Skill 不知道这些卡点，它写出来的就只是泛泛而谈的“你可以用 Notion 做很多事”；如果它知道，邮件就可以直接冲着焦虑去，比如给出一个 2 分钟可复制模板，先帮用户跨过“第一步不知道怎么走”的门槛。

原文还列了几个非常实际的转化触发器，比如用户搭出第一个真正能跑的 workspace，看到数据库、视图、模板连起来的 “aha moment”；比如团队里已经有人在用，内部采纳阻力会明显下降；比如正好有一个能立即套用的模板；比如免费版的空间或访客限制被撞到，升级动机会自然增强。对普通读者来说，这段最有价值的翻译是：**转化很多时候不是被说服，而是被具体场景推了一把。**

这份文件要怎么做？原文建议从三类材料里取：分析数据，看真实转化路径和流失点；销售通话，听哪些异议会成交、哪些异议会直接劝退；客户访谈，让用户按时间顺序讲“你是怎么发现我们的，为什么最后决定买”。比起脑补流程，这种回放式材料更接近真实决策。

## **4** 真正高明的地方，不是把 4 个文件都写出来，而是让它们自己被正确调用

原文最后一部分说的，是整套方法里最像系统设计的地方：每个 foundation 文件开头都有一个 header，告诉 Skill 在什么情况下应该加载它、在什么情况下不该加载它。

比如，一个文件可以写明：“当任务涉及博客、社媒、邮件、落地页、广告文案等任何文字输出时加载”；同时再写明：“如果任务只需要竞品信息，或者只需要买家路径，就不要加载我。” 这样做的好处是，Skill 先扫一遍 `./foundation/`，只读每个文件头部，再决定要不要把全文读进来。

```
Step 1: Load Foundation Context
- 扫描 `./foundation/` 里的 `.md` 文件
- 先只读每个文件 `---` 之前的 header
- 对照当前任务，看是否满足 “Load this file when”
- 只有匹配时才加载全文，其余跳过
```

这套方法的关键，在于把“智能”更多地放在文件里，而不是写死在单个 Skill 里。这样一来，你新建一个 Skill，不必手工重新接线；你下个月新增第 5 个 foundation 文件，老 Skill 也能在相关任务里自动吃到它。你改的不是某一个提示词，而是在改整套系统的底座。

当然，边界也要说清楚。Foundation Layer 不会凭空替你创造洞察，也不会自动保证高转化。它仍然依赖你手里有没有真实用户语言、有没有表现好的内容样本、有没有足够新的竞品和转化数据。只是和不停微调单条 prompt 相比，至少从这篇材料给出的框架看，**先把共享上下文写清楚，往往是更有复利的一步。**

## **5** 这篇文章真正想提醒的，是别让 Claude Code 一直“重新认识你”

把全文串起来看，作者真正想谈的其实不是 4 个模板，而是一种组织 AI 的方式。皮克斯当年补上的不是人才缺口，而是共享语境；今天很多团队在 Claude Code 里补的，也未必是模型能力，而是“每次开始工作前，AI 到底知不知道自己在替谁做事”。

如果你的工作流经常重复做内容、发 campaign、写发布、做 nurture，那这 4 个 `.md` 文件的意义就不只是“写得更顺”，而是让 Claude Code 从“会生成”慢慢变成“知道该往哪边生成”。这不神奇，但很关键。

真正拉开 Claude Code 使用体验的，很多时候不是多一个 Skill，而是少一点“每次都从零猜起”。 当共享上下文被写进系统，AI 才更像一个逐步熟悉业务的同事，而不是永远需要重新 brief 的实习生。 问题也许不再是“你有没有用 Claude Code”，而是“你有没有把该告诉它的东西，真的写下来”。
