## 你的 Claude Code 为什么"不听话"？

大多数人用 Claude Code 遇到的问题都很类似：

* • 让它改个文件，结果它不知道项目结构，改错了地方
* • 让它写测试，结果它不知道用什么框架，写出来跑不通
* • 让它提交代码，commit message 格式跟团队规范完全不一样
* • 同一个需求反复解释，每次对话都像从零开始

**根本原因只有一个：你没给它上下文。**

Claude Code 不是通灵的——它不读你脑子里的想法，也不会自动理解你的项目约定。但好消息是，Anthropic 设计了一套很简单的机制来解决这个问题。

## 核心实践一：CLAUDE.md——给 AI 一份"项目说明书"

这是 Anthropic 官方最佳实践的第一条，也是改变最大的一条。

CLAUDE.md 是放在项目根目录的一个文件，Claude Code 每次启动时会自动读取。你可以把它理解为**写给 AI 的项目手册**——告诉它这个项目怎么构建、怎么测试、代码风格是什么、目录结构长什么样。

### 应该写什么？

* • Bash 命令（构建、测试、部署怎么跑）
* • 核心文件和工具函数的位置
* • 代码风格指南
* • 测试执行方法
* • Git 提交规范（commit message 格式、PR 规则）
* • 开发环境设置

### 改进前 vs 改进后

**改进前**（大多数人的 CLAUDE.md，如果有的话）：

```
## 命令  
npm run dev  
npm run build
```

就这？Claude 看完这个文件，跟没看差不多。

**改进后**（Anthropic 推荐的写法）：

```
## 测试指南  
  
### 类型检查及验证  
npm run astro check   # 推荐：代码编写后始终执行  
npm run build         # 生产构建测试  
npm run preview       # 预览构建结果  
  
### 测试检查清单  
1. ✓ npm run astro check 通过  
2. ✓ npm run build 成功  
3. ✓ Content Collections 模式遵守  
4. ✓ SEO 元数据完整性验证
```

区别一目了然。改进后的版本不光告诉 Claude 有哪些命令，还告诉它**什么时候该跑、跑完该检查什么**。

实际效果：配置了详细的测试指南后，Claude 在每次修改代码后**自动执行 `npm run astro check` 进行验证**——不用你提醒。

### Git 提交规范也能教会它

很多团队的痛点是 Claude 生成的 commit message 乱七八糟。解决方案很简单——在 CLAUDE.md 里写清楚：

```
## Repository Etiquette  
  
### Git Commit Messages  
**格式**：<type>(<scope>): <subject>  
**Types**：  
- feat: 新功能  
- fix: Bug 修复  
- docs: 文档修改  
- refactor: 代码重构
```

写完之后，Claude 自动生成的提交信息就变成了这样：

```
feat(blog): add claude code best practices post  
docs(claude): update workflow guidelines
```

规规矩矩，不用手动改。

## 核心实践二：四步工作流——Explore → Plan → Code → Commit

Anthropic 官方推荐的工作流不是"想到什么让它干什么"，而是一套**结构化的四步法**。

### 第一步：Explore（探索）

在让 Claude 动手写代码之前，先让它**阅读相关文件**，理解项目上下文。

```
"请阅读 CLAUDE.md 并理解项目结构"  
"请分析现有博客文章的结构"
```

这一步很多人跳过了，但它直接决定了后面生成代码的质量。

### 第二步：Plan（计划）

让 Claude 用 TodoWrite 工具把任务拆解成可执行的步骤：

```
// Claude 自动细分任务  
1. [pending] 确认博客文章模式  
2. [pending] 生成英雄图片  
3. [pending] 编写韩语版本  
4. [pending] 编写英语/日语版本  
5. [pending] 优化 SEO 元数据
```

有了这个计划，Claude 就不会东一榔头西一棒子，而是按部就班地执行。

### 第三步：Code（实现）

以小单位工作，每次变更后立即验证。不要一口气让它改十个文件——改一个验证一个，出了问题才知道是哪步出的。

### 第四步：Commit（提交）

以有意义的单位提交，编写清晰的 commit message。如果 CLAUDE.md 里配好了规范，这一步完全自动化。

**对比一下**：

```
# 改进前：直接甩需求  
"编写博客文章"  
  
# 改进后：四步工作流  
1. Explore: 分析现有文章结构  
2. Plan: 用 TodoWrite 创建任务项  
3. Code: 分步实现并验证  
4. Commit: 以有意义的单位提交
```

看起来多了几步，但实际上**总时间反而更短**——因为返工少了 60%。

## 核心实践四：子代理系统——专人干专事

这是进阶玩法。Anthropic 的核心理念是：**把特定任务委托给专门的代理，能提高上下文集中度和 Token 效率**。

简单说就是"术业有专攻"——与其让一个通用 Claude 干所有事，不如给不同的任务分配专门的"角色"。

### 子代理怎么组织？

在项目的 `.claude/agents/` 目录下创建专用代理文件：

```
.claude/agents/  
├── content-planner.md   # 内容策略  
├── writing-assistant.md # 博客写作  
├── editor.md            # 语法/风格审查  
├── seo-optimizer.md     # SEO 优化  
├── image-generator.md   # 图片生成  
└── analytics-reporter.md # 流量分析
```

### 怎么调用？

```
@writing-assistant "编写关于 TypeScript 5.0 功能的博客"  
@seo-optimizer "优化最近文章的内部链接"  
@image-generator "生成博客英雄图片"
```

每个代理只关注自己的领域，不会被无关信息干扰——结果就是 **Token 使用量平均节省 25%**，回答质量也更高。

## 核心实践五：MCP 集成——让 Claude 连接一切

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，让 Claude Code 能够连接外部工具和数据源。

实际能接入的东西包括：

* • **Context7**：查询最新库文档（不用再担心 Claude 的训练数据过期）
* • **Playwright**：Web 自动化和测试
* • **Chrome DevTools**：性能分析
* • **Google Analytics**：流量数据分析

使用示例：

```
"请使用 Context7 查询 Astro 5.0 的最新图片优化功能"
```

有了 MCP，Claude Code 的能力边界不再局限于"读写文件"——它能查文档、跑浏览器、分析数据，变成一个真正的全栈助手。

## 实战技巧速查

### 1. 把需求说具体

```
# ❌ 不好的例子  
"编写博客文章"  
  
# ✅ 好的例子  
"请编写关于 TypeScript 5.0 装饰器功能的博客文章。  
必须包含以下内容：  
1. 语法说明及代码示例  
2. 3 个实际使用案例  
3. 与遗留装饰器的差异  
4. 韩语、英语、日语版本  
5. SEO 优化的元数据"
```

### 2. 指明具体文件

```
# ❌ 不好的例子  
"修改头部组件"  
  
# ✅ 好的例子  
"请在 src/components/Header.astro 文件的导航菜单中  
添加多语言切换按钮"
```

### 3. 迭代改进，别想一步到位

```
第 1 次: "编写博客文章"  
第 2 次: "将 description 修改为更 SEO 友好"  
第 3 次: "在代码示例中添加中文注释"  
第 4 次: "将日语版本的技术术语调整得更自然"
```

### 4. 对话太长时用 /clear 重置

对话过长会导致上下文过载、回答质量下降。当主题改变时，果断用 `/clear` 重置——Claude 还是能从 CLAUDE.md 里恢复上下文。

### 5. 给 Claude 看截图或设计稿

提供视觉参考（截图、设计稿、线框图）可以大幅提高 Claude 对需求的理解度，比纯文字描述有效得多。

## 常见问题

**Q: CLAUDE.md 跟普通的 README 有什么区别？**

A: README 是写给人看的，CLAUDE.md 是写给 AI 看的。它的内容更聚焦于"AI 执行任务时需要知道的信息"——构建命令、测试方法、代码规范、目录结构。Claude Code 每次启动时会自动读取这个文件来建立上下文。

**Q: Think 工具什么时候用、什么时候不用？**

A: 简单任务（改个样式、加个字段）不需要 Think。复杂任务（架构决策、多文件联动修改、需要权衡取舍的场景）强烈建议用。判断标准很简单：如果你自己做这个决策都需要想一想，那就让 Claude 也 Think 一下。

**Q: 子代理系统学习成本高吗？**

A: 不高。本质就是在 `.claude/agents/` 目录下放几个 Markdown 文件，每个文件定义一个角色的专长和行为规范。写好之后用 `@agent-name` 调用就行。初期可以只建 2-3 个最常用的代理，后续按需扩展。

**Q: 这些最佳实践适用于所有类型的项目吗？**

A: 核心原则（CLAUDE.md、四步工作流、具体化需求）适用于任何项目。子代理和 MCP 集成更适合中大型项目或需要频繁执行重复任务的场景。个人小项目从 CLAUDE.md 开始就够了。

**Q: 国内用户怎么用上 Claude Code？**

A: 官方订阅需要海外信用卡和合适的网络环境。觉得麻烦的话可以通过 Code80 更方便地使用，不用操心支付和网络问题。
