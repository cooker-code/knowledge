如果你直接裸跑，大概率会卡在某个环节不知所措。所以，别急着让它帮你写代码，我们先把工具本身“武装”到牙齿。接下来，直接上实操：从零开始安装 /feature-dev，搞定环境配置，并带你完整走一遍首次调用的真实体验

## 安装和使用

---

### 安装

```
/plugin install feature-dev@claude-plugin-directory
```

装完立即可用：`/feature-dev` 命令 + 三个 Agent（`code-explorer`、`code-architect`、`code-reviewer`）。

---

### 步骤 1：从干净状态开始

开始前 commit 或 stash 掉进行中的工作。插件会探索并修改多个文件，干净基线让你能精确审变更，想扔掉输出时也有恢复点。

---

### 步骤 2：带描述启动

```
/feature-dev Add rate limiting to the public API endpoints
```

预先给的上下文越多，Phase 3 要问的越少。但别因为规格不完美就犹豫——澄清问题阶段的存在就是为了填空。

---

### 步骤 3：认真参与 Phase 3 和 Phase 4

这是**杠杆效应最高的两个节点**。Phase 3 要具体回答：错误处理怎么做、边界情况是什么、集成点在哪。具体答案比"你觉得怎么好怎么来"能产生好得多的实现。Phase 4 选方案前读权衡，Claude 的建议基于你的实际代码库，但你可能有团队或路线图的上下文应该覆盖它。

---

### 步骤 4：认真对待质量审查

Phase 6 的置信度过滤会浮出真问题。Agent 只报 80%+ 置信度的发现，出现的内容值得看。可以现在修、稍后记、或者接受，但要**有意识地决定**，别直接跳过。

---

## 最佳实践：怎么拿高质量结果

清晰的上下文、具体的目标、紧凑的范围带来更好的结果。Claude 理解你的系统和期望时，产出的代码是"你高质量的代码库"而不只是"能跑"。

---

### 保持 CLAUDE.md 及时且详细

`CLAUDE.md` 给每次 Claude Code 会话（包括 `/feature-dev` 运行）提供上下文。`code-explorer` 和 `code-architect` 会把它纳入分析。

记下你的架构模式、命名约定、测试配置、目录结构。`CLAUDE.md` 越好，输出越贴近你的约定，不用你每次重复。

---

### 把 Phase 3 当需求审查

把澄清问题当成**需求审查**，而不是打扰。Claude 问的问题反映规格里的真实歧义。如果某个问题让你意外，这就是信号——答案本会影响实现——在这里确定比在代码写完后发现缺口好得多。

---

### 约束范围防止 Agent 漂移

**Agent 漂移**发生在宽松范围导致插件做出超纲决策：重构相邻代码、加你没要的功能、碰范围外的文件。具体的初始描述 + 具体的 Phase 3 答案能让实现保持紧凑。

---

### 大功能分阶段跑

substantial 功能拆阶段，分别跑插件。先数据模型，审完；再 API 层，审完；再 UI。每次运行在可管理范围，前进前有验证过的检查点。

---

## /feature-dev与意图识别

每次提示 LLM，模型都得从你说的内容里解析意图。开放式提示留下更多解析空间。`/feature-dev` 这类结构化插件通过在**工作流层面编码意图**来减少这种解析负担。

定义好的流程告诉 Claude 按什么顺序做什么、派什么 Agent、何时暂停等你，把你从**对话式表达意图**变成**结构性编码意图**，结果更确定。

---

### 用任务框架减少幻觉

代码生成里的幻觉跟歧义和范围宽度有关。`/feature-dev` 在多节点解决这个问题：Phase 1 清需求歧义，Phase 2 用实际文件读取代代码库假设，Phase 3 填规格缺口，Phase 6 的置信度过滤审查抓漏网之鱼。

---

### 插件即"工作流层面的意图编码"

`/feature-dev` 不是缩短提示——它编码了整套开发方法论：**探索先于构建，澄清先于设计，设计先于实现，审查先于交付**。每次调用，这套方法论自动跑在你的特定代码库上。这是**工作流层面而非提示层面的意图工程**。

---

## 常见错误

`/feature-dev` 的问题大多来自**使用方式**，不是工具限制。

---

### rushed 过 Phase 3

"加认证"够发现阶段启动，但 Phase 3 的澄清问题才是你**真正确定要建什么**的地方。把这些问题当形式、 rushed 过去，会产出"逻辑自洽但方向错了"的实现。输出的具体程度反映你回答的具体程度。

---

### 用来搞全系统重构

插件为边界明确的功能设计。单次调用里做完整架构大修会超载上下文，产出内部不一致。拆小，分别跑。

---

### 跳过 Phase 6 质量审查

Phase 6 给你置信度过滤的真问题列表，Agent 只报 80%+ 置信度的发现。剩下的值得看。完全跳过审查，违背把它做进工作流的初衷。

---

## /feature-dev是软件开发的未来吗？

Claude Code 插件生态反映开发者工具的 broader 方向：把完整工作流打包成可安装单元，而不是让开发者自己组装提示和命令。

正如 Anthropic 工程团队解释的，Agent Skills 和插件设计为**可组合、可移植、可共享**。`/feature-dev` 把这套应用到完整功能开发生命周期，验证安装已超 8.9 万次，结构化方法确实找到了市场。

---

### 人类工程师仍领先的地方

结构化 Agent 工作流加速**执行**。它们不替代**判断**。跨多系统的架构决策、竞争性长期优先级之间的权衡、需要组织上下文的推理——这些还是工程师的。

插件擅长在定义边界内实现功能。它不擅长决定该建什么、某功能是否属于给定服务、或者架构在规模上是否成立。会用它的工程师，把更多时间花在需要判断的决策上，更少花在不需要判断的实现上。

---

## 最后

`/feature-dev` 是个设计精良的答案，回应真问题：怎么在不微管理每一步的情况下，从 LLM Agent 拿一致、感知代码库的产出？Anthropic 的做法是把整套开发方法论：探索、澄清、设计、实现、审查，打包成可安装插件，在每个关键决策点嵌入人机检查点。

7 个阶段的存在，有经验的工程师会认同：不理解代码库就设计不好，没有清晰设计就实现不好，没有审查就交付不好。插件强制执行这个顺序，而不是让你在时间压力下跳过步骤。这种强制执行正是大部分价值的来源。

认真过澄清问题和架构阶段，把质量审查报告当真，最终检查。这是设计好的工作流，产出结果明显优于直接跳去写代码。

## FAQ

**`/feature-dev` 是什么？**

Anthropic 官方验证的 Claude Code 插件，提供结构化 7 阶段工作流，含三个专职 Agent：代码探索、架构设计、质量审查。安装命令：`/plugin install feature-dev@claude-plugin-directory`

**什么时候用 `/feature-dev` 而不是普通提示？**

功能需要代码库探索、架构决策或多文件变更时用。简单编辑、快速调试、一次性问题用普通提示。

**什么算"中等复杂度"？**

加新 API 端点、写 UI 组件、集成认证逻辑、扩展现有功能。不是单文件小修，也不是完整架构重写。

**怎么减少幻觉？**

Phase 1 清需求歧义，Phase 2 用实际文件读取代代码库假设，Phase 3 填规格缺口，Phase 6 用 80%+ 置信度过滤审查抓剩余问题。

**能重构整个应用吗？**

不能。拆成有界功能任务，分别跑插件。

**会自动改我仓库吗？**

不会。Phase 5 的实现只在明确批准 Phase 4 选的架构后才开始。

**初始指令怎么写？**

随意，短或具体都行——发现和澄清问题会处理歧义。预先指定硬性约束能减少 Phase 3 的问题。

**适合原型设计吗？**

适合。Phase 3 的澄清问题会浮出你还没完全定义的需求，正好在假设固化进代码之前。

**跟 AI coding Copilot 什么区别？**

Copilot 逐行或逐文件协助。`/feature-dev` 在功能级别运作，跨 7 阶段，配专职 Agent 做探索、设计和质量审查。

**会取代资深开发者吗？**

不会。它加速定义范围内的结构化实现，但架构判断和生产责任仍属人类。
