说句实话，**Claude Code / OpenCode 好不好用，大部分取决于你怎么写 Prompt**。

很多人用着用着觉得它“有点笨”，其实不是模型不行，而是 Prompt 还停留在「聊天式」。 我现在给 Claude Code / OpenCode 的 Prompt，有一个原则：

> ❝
>
> **它不是来陪我聊天的，是来帮我干活的。**

下面这些模板，都是我在真实项目里反复用、不断微调后留下来的。

---

## 一、通用底座模板（我几乎每次都会用）

这是一个**万能前置 Prompt**，我经常在脑子里默认加上它：

```
你现在是一个参与当前项目的开发者，
请基于已有代码和项目结构回答，
不要引入新的技术栈或依赖，
如果有不确定的地方，请明确指出。
```

📌 **作用：**

* 防止它“自由发挥”
* 防止它推荐一堆你根本不会用的库
* 回答会明显更克制、更工程化

---

## 二、读代码 / 理解项目类（接手项目必备）

### 1️⃣ 快速理解项目结构

```
请帮我梳理当前项目的整体结构：
1. 核心模块分别是做什么的
2. 主流程从哪里进入
3. 哪些目录是我最应该优先阅读的
用工程师视角回答，不要泛泛而谈
```

📌 **适用场景：**

* 新项目 / 老项目接手
* 看别人写的屎山（尤其有用）

---

### 2️⃣ 解释“看不懂的代码”

```
请解释这段代码的真实意图，
重点说明：
- 为什么要这么写
- 可能解决了什么问题
- 有没有隐藏的副作用
```

📌 我很少再用「逐行解释」，**直接让它讲“动机”和“代价”更有价值。**

---

## 三、代码修改 / 重构前必用模板（防翻车）

### 3️⃣ 重构前风险扫描（强烈推荐）

```
我准备对这部分代码进行修改，
在不改变现有功能的前提下：
1. 帮我指出潜在的风险点
2. 哪些地方一改就容易出问题
3. 有没有隐含的业务假设
```

📌 **这条真的救过我几次命**尤其是老系统、多人维护的项目。

---

### 4️⃣ 小步重构（不求完美）

```
请在保持现有行为不变的前提下，
给出最小成本的重构建议，
目标是提升可读性而不是性能
```

📌 这条会明显减少：

* “建议你整体重写”
* “建议引入新架构”

---

## 四、定位问题类（Debug 场景）

### 5️⃣ 不稳定问题分析（我很常用）

```
当前问题表现为：XXX
在什么情况下会发生：XXX
已知事实：XXX

请基于现有代码分析：
- 可能的触发路径
- 哪些假设最值得优先验证
```

📌 关键词是：**“优先验证”**不是让它直接给结论，而是帮你缩小范围。

---

### 6️⃣ 日志 / 异常反推问题

```
以下是错误日志和相关代码，
请帮我反推出：
- 最可能的问题来源
- 是否存在多种可能性
- 我下一步最该加哪里的日志
```

📌 这一条比“帮我修 bug”靠谱太多。

---

## 五、测试相关（别直接让它写完）

### 7️⃣ 测试点设计（比写代码重要）

```
请基于当前代码列出：
- 必须覆盖的核心测试点
- 容易遗漏的边界情况
- 不测也容易出事的地方
不要直接生成测试代码
```

📌 **测试点 > 测试代码**

---

### 8️⃣ 补漏型测试辅助

```
我已经写了一部分测试，
请帮我检查是否存在：
- 覆盖不足的分支
- 错误但可能被忽略的输入
```

---

## 六、Git / 日常开发效率类（高频）

### 9️⃣ 生成靠谱的 commit message

```
基于当前 git diff，
请生成一个符合约定的 commit message，
要求：
- 准确描述变更意图
- 不夸大、不模糊
```

📌 我现在基本不自己想 commit message 了。

---

### 🔟 PR / 代码评审辅助

```
请站在代码评审者的角度，
指出这次改动中：
- 可能被质疑的地方
- 需要提前解释的设计决策
```

📌 **这条对减少来回沟通非常有用。**

---

## 七、我用 Prompt 的几个真实习惯

这几条比模板还重要：

* ✅ 永远说明「当前目标」
* ✅ 多用“在不改变现有行为的前提下”
* ✅ 少用“帮我写一个”，多用“帮我分析/评估”
* ❌ 不指望一次 Prompt 得到最终答案
* ❌ 不让它替你做架构决策

---

## 结语

这些 Prompt **不是一次写完的**， 而是你用着用着自然会进化出来的。

当你发现自己开始反复用同一句 Prompt， 说明它已经变成了你**工作方式的一部分**。
