# Claude Code Agent 设计解析

> 逆向分析 Claude Code TypeScript 源码（~512K 行，v2.1.88），提炼 Agent 工程的核心设计决策。基于 arXiv:2604.14228v1 "Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems" (Zhiqiang Shen et al., VILA Lab)。

---

## 1. 核心论点：5 个价值 + 13 条设计原则

### 1.1 Claude Code 在乎什么

Claude Code 的架构不是拍脑袋拼出来的。每一个设计决策背后，都能追溯到五个价值取向：

| 价值 | 意味着什么 | 怎么体现在代码里 |
| --- | --- | --- |
| **人类说了算** | 最终拍板权始终在人手里——先是 Anthropic，再是运维，最后是用户 | Deny-first 权限模型、分级信任、审计日志可追溯 |
| **安全兜底** | 哪怕用户没在看，系统也要兜住底线 | 多层安全叠加、ML 分类器、沙箱隔离 |
| **说到做到** | Agent 要完成"用户真正想做的事"，干久了也不能跑偏 | 五层压缩管道、Append-only 状态、故障自动恢复 |
| **真正提效** | 不是换个方式做老事情，而是同样的精力产出更多东西 | 最小推理脚手架、可组合扩展、确定性基础设施 |
| **因人而异** | 适配不同项目、不同工具、不同水平的用户 | 四级 CLAUDE.md、渐进式信任梯度 |

**一个扎心的发现**：论文还引入了第六个视角——**长期能力会不会退化？** 研究数据很吓人：使用 AI 辅助的开发者，在理解力测试中得分降低了 17%，代码复杂度增加了 40.7%，EEG 脑电扫描显示神经连接减弱，而且移除 AI 后这种影响还在持续。换句话说，"帮你提效"这件事本身，可能正在悄悄削弱你的基本功。

### 1.2 十三条设计原则

每条原则都在回答一个反复出现的设计选择：

| 原则 | 服务于什么价值 | 回答的问题 |
| --- | --- | --- |
| **Deny-first，拿不准就升级给人** | 权威、安全 | 不认识的操作：放行、拦住、还是问人？ |
| **分级信任，渐进解锁** | 权威、适配 | 固定权限级别，还是随时间慢慢放开？ |
| **多层纵深防御** | 安全、权威、可靠 | 一个安全边界够不够？还是多层叠加更保险？ |
| **策略外置，生命周期钩子接管** | 安全、权威、适配 | 硬编码规则，还是用 hook 动态配置？ |
| **上下文是稀缺资源** | 可靠、能力 | 约束瓶颈在哪？截断还是分级压缩？ |
| **Append-only 持久状态** | 可靠、权威 | 可变状态、检查点快照、还是只追加日志？ |
| **少搭脚手架，多建操作框架** | 能力、可靠 | 投资让模型"想清楚"，还是投资让它"跑得稳"？ |
| **价值优先于死规则** | 能力、权威 | 固定规则决策，还是有护栏的上下文判断？ |
| **可组合的扩展机制** | 能力、适配 | 一个统一 API，还是按上下文成本分层？ |
| **可逆性加权风险评估** | 能力、安全 | 所有操作同等监管，还是只读/可逆的放宽点？ |
| **配置写文件，用户看得见** | 适配、权威 | 不透明数据库、向量检索、还是纯文本文件？ |
| **子 Agent 隔离运行** | 可靠、安全、能力 | 子 Agent 继承父级权限，还是沙箱隔离？ |
| **优雅恢复，别动不动就挂** | 可靠、能力 | 报错就崩，还是静默恢复、实在不行再通知人？ |

## 2. 架构总览：七个组件 · 五层分解

### 2.1 七个核心组件

整个系统的数据流可以用一句话概括：

```
用户 → 接口 → Agent 循环 → 权限系统 → 工具 → 执行环境
+ 状态 & 持久化 
```

七个组件各管一摊事：

1. 1. **用户**：提 prompt、批权限、审输出
2. 2. **接口**：交互式 CLI、无头 CLI (`claude -p`)、Agent SDK、IDE/桌面/浏览器都能接入
3. 3. **Agent 循环**：`queryLoop()` 这个异步生成器（在 query.ts 里）是整个系统的心脏
4. 4. **权限系统**：Deny-first 规则 + auto-mode ML 分类器 + hook 拦截，三层叠加
5. 5. **工具**：最多 54 个内置工具（19 个始终可用 + 35 个条件加载），还能通过 MCP 接入外部工具
6. 6. **状态 & 持久化**：Append-only JSONL 会话记录、全局 prompt 历史、子 Agent 侧链
7. 7. **执行环境**：Shell 执行（可选沙箱）、文件系统、网页抓取、MCP 连接

### 2.2 五层架构

从上到下，系统分成五层：

```
Surface 层：入口和渲染（CLI、SDK、IDE 等）               
Core 层：Agent 循环 + 压缩管道（queryLoop + 5 层 shaper）
Safety/Action 层：权限 + 钩子 + 扩展 + 工具 + 沙箱       
State 层：上下文组装 + 运行状态 + 会话持久 + 记忆      
Backend 层：实际执行（Shell、文件系统、远程、MCP）         
```

### 2.3 最核心的发现：1.6% vs 98.4%

这是整篇报告最值得记住的一个数字。

Claude Code 有大约 512K 行代码。其中真正涉及"AI 做决策"的逻辑——比如模型该调哪个工具、下一步怎么走——大概只占 **1.6%**。

剩下 **98.4%** 的代码干的都是什么？权限检查、工具路由、上下文压缩、状态管理、故障恢复、日志记录……全是"跑工具、管权限、压上下文"这类基础设施活。

这个比例不是巧合。它揭示了一个设计哲学：**Claude Code 把工程量砸在了确定性基础设施上，让模型在一个搭好的、丰富的操作框架里自由发挥，而不是用一堆规划图来约束模型的每一步。**

打个比方：模型是发动机，但它只需要 1.6% 的代码来"思考往哪开"。剩下的 98.4% 是修路、装红绿灯、设护栏、画车道线——确保不管发动机往哪个方向转，都在安全可控的范围内。

## 3. Agent 循环：极简 while-loop 哲学

### 3.1 一切的核心就是一个 while 循环

Claude Code 的核心逻辑可以浓缩成十几行伪代码：

```
while not stopped:
    context = assemble(system_prompt, tool_schemas, history, hook_additions)
    action = model(context, tools)
    if action.is_text_only():
        stopped = run_stop_hooks(action)
        continue
    if not permitted(action):
        continue
    action = run_pre_tool_hooks(action)
    result = execute(action)
    result = run_post_tool_hooks(result)
    history.append(action, result)
```

就是这么简单：组装上下文 → 模型选动作 → 权限检查 → 执行工具 → 拿结果 → 塞回历史 → 下一轮。

整个 51 万行代码库里，这个循环本身以及直接支撑它的决策逻辑，大概只占 **1.6%**。剩下的 98.4% 是什么？是权限系统、工具路由、上下文压缩、状态持久化、错误恢复、沙箱、钩子……一句话：都是让这个循环能稳定跑起来的基础设施。

换句话说，Claude Code 把工程量花在了"跑工具、管权限、压缩上下文"这些确定性基础设施上，而不是花在帮模型做规划决策上。它不维护什么显式的规划图或者状态机来约束模型怎么想——模型自己推理，框架只管执行。这跟 Devin 那种维护显式规划结构的路线完全相反。

有个类比挺贴切的：模型是 CPU，这个 while 循环是取指-解码-执行的时钟周期，周围的代码是操作系统内核——管调度、管权限、管 IO。时钟周期极简，但内核很重。

### 3.2 四个设计问题

这个 while-loop 架构背后其实是在回答四个根本问题：

| 问题 | Claude Code 的答案 | 其他方案怎么做 |
| --- | --- | --- |
| 推理在哪？ | 模型自己推理，框架只管执行 | Devin 维护显式规划结构，LangGraph 用状态图路由 |
| 几套执行引擎？ | 一套 `queryLoop()`，只有渲染层不同 | 有些系统 IDE 和 CLI 走不同代码路径 |
| 默认安全策略？ | Deny-first + 升级到人工审批 | SWE-Agent 用 Docker 隔离，Aider 用 Git 回滚 |
| 绑定资源约束？ | 上下文窗口（200K/1M），五层压缩管道 | 有些系统限制计算预算或工作内存 |

第二点值得展开。不管是交互式 CLI、无头模式（`claude -p`）、Agent SDK 还是未来的 IDE 集成，底层跑的都是同一个 `queryLoop()`。只是接入层变了，核心引擎没变。这意味着安全策略、上下文管理、工具执行这些关键逻辑不需要在多个入口点之间保持同步。

### 3.3 工具调度：并发读、串行写

模型在流式输出的时候，`StreamingToolExecutor` 就已经开始执行工具了，不用等模型完全生成完。这是在"完全串行等待"和"PASTE 式投机执行"之间的中间地带——PASTE 会在模型还在生成时就投机预执行它预测的未来工具调用，更激进但风险也更高。

具体规则：

* • **只读操作可以并行**跑（比如同时读三个文件）
* • **修改状态的操作串行化**（比如 shell 命令）
* • 任一 Bash 工具出错，**兄弟中止控制器**立即终止其他进行中的子进程
* • 结果按工具接收顺序缓冲——模型期望结果的顺序跟请求顺序一致

### 3.4 五层上下文压缩管道

这是 Claude Code 最有意思的技术亮点之一。每次调模型之前，有五个 shaper 依次跑一遍：

| 层级 | 名称 | 做什么 | 什么时候触发 |
| --- | --- | --- | --- |
| 1 | **Budget Reduction** | 每条工具结果给个大小上限，超了就换成内容引用 | 始终激活 |
| 2 | **Snip** | 轻量裁掉比较旧的历史段 | `HISTORY_SNIP` 门控 |
| 3 | **Microcompact** | 细粒度、缓存感知的压缩 | `CACHED_MICROCOMPACT` 门控 |
| 4 | **Context Collapse** | 读时虚拟投影，不改存储的历史 | `CONTEXT_COLLAPSE` 门控 |
| 5 | **Auto-compact** | 用模型生成完整摘要（最贵） | 默认启用，可禁用 |

**为什么需要五层而不是一层？** 因为没有单一压缩策略能应对所有类型的上下文压力。

想象一下：上下文快满了但其实只是几条工具输出特别长——这时候第一层 Budget Reduction 把大输出换成引用就够了，成本几乎为零。如果还是不够，可能是历史太长了，第二层 Snip 裁掉旧的段落。再不够，第三层 Microcompact 做细粒度压缩，它会感知哪些部分已经被缓存了（在 prompt cache 里），优先压缩没缓存的部分以最大化缓存命中率。如果连这都不够，第四层 Context Collapse 做读时虚拟投影——它不改存储的历史，只是让模型看到一个压缩视图，全历史随时可以重建。最后实在没办法，第五层才用模型生成完整摘要，这是最贵的操作。

设计哲学很清楚：**先用便宜的策略，不够再升级到更贵的**。这跟 CPU 缓存的 L1/L2/L3/内存分级是一个思路——大部分时候命中 L1 就够了，极少情况才需要访问内存。

Context Collapse 特别巧妙。源码注释说的是"不 yield 任何东西；折叠视图是对 REPL 完整历史的读时虚拟投影"。意思是：原始历史一行都没动，只是在读取的时候投影出一个压缩版本给模型看。这让你可以随时重建完整上下文，同时模型当前看到的是精简过的。

### 3.5 五种终止条件

循环可以在五种情况下停下来：

1. 1. **模型不再调工具**——只输出文本，这是主要的终止方式
2. 2. **到达最大轮次**——`maxTurns` 可配置
3. 3. **上下文溢出**——API 返回 `prompt_too_long`
4. 4. **钩子干预**——PostToolUse 钩子设置了 `hook_stopped_continuation`
5. 5. **用户主动中止**——`abortController` 信号触发

### 3.6 恢复机制

循环不只是终止就完了，还得能从各种异常中恢复：

* • **输出 token 触顶**：可以重试并提升限制（最多 `MAX_OUTPUT_TOKENS_RECOVERY_LIMIT` 次），受 GrowthBook 标志控制
* • **上下文快满了**：响应式压缩（`REACTIVE_COMPACT` 门控）会在接近容量时压缩刚好够的空间，每轮最多触发一次
* • **上下文真的溢出了**：先试 context-collapse 溢出恢复，再试响应式压缩，都失败才以 `reason: 'prompt_too_long'` 正式终止
* • **流式 API 出问题**：`onStreamingFallback` 回调兜底
* • **主模型挂了**：`fallbackModel` 参数支持切换到备选模型

这些恢复策略跟"优雅恢复与弹性"的设计原则一致：错误时不要硬失败，能静默恢复就恢复，只有真正不可恢复的时候才通知人。

## 4. 权限系统：七层纵深防御

### 4.1 一个数据驱动的架构决策

Anthropic 发现了一个让安全团队不太舒服的数据：用户批准了大约 **93% 的权限提示**。

这意味着什么？意味着交互式确认作为唯一的安全机制，在行为层面是不可靠的。用户不是在认真审查每个操作，而是在习惯性地点"允许"。这不是用户的问题——是人性。当一个人每天被问几百次"要不要允许这个操作"，他一定会疲。

这个 93% 的数据直接驱动了 Claude Code 权限系统的架构走向：不能只靠交互式审批这一层。必须有更底层的、确定性的安全机制来兜底。这就是 deny-first 策略的核心动机——先默认拒绝，只有明确允许的操作才能通过，不依赖用户的判断力。

### 4.2 七种权限模式

Claude Code 提供七种权限模式，覆盖从最严格到最宽松的全谱：

| 模式 | 行为 |
| --- | --- |
| `plan` | 模型必须先出计划，人批准了才执行 |
| `default` | 标准交互模式，大部分操作要人批 |
| `acceptEdits` | 工作目录内的编辑和特定文件命令自动批 |
| `auto` | ML 分类器评估安全性（功能门控） |
| `dontAsk` | 不弹提示，但 deny 规则照样生效 |
| `bypassPermissions` | 跳过大部分权限提示，安全关键检查还在 |
| `bubble` | 内部用的，子 Agent 把权限升级到父终端 |

### 4.3 七层授权管道

一个工具调用从模型提出到真正执行，要经过七道关卡：

```
工具请求
  → ① 预过滤（批量拒绝工具直接从模型视图里移除）
  → ② PreToolUse 钩子（可以阻止或改写）
  → ③ Deny-first 规则评估
  → ④ 权限处理器（四条路径分叉，见下文）
  → ⑤ Auto-mode ML 分类器
  → ⑥ Shell 沙箱（文件系统 + 网络隔离）
  → ⑦ 会话恢复时不恢复权限
```

第一层预过滤很有意思：它在模型看到工具之前就把被 deny 的工具整个移除了。模型甚至不知道有这个工具可用，所以也不会尝试调用它。这比调了再拒绝更高效。

### 4.4 权限处理器的四条路径

`useCanUseTool.tsx` 里的处理器会根据运行时上下文分叉成四条路径：

| 路径 | 适用场景 | 怎么工作 |
| --- | --- | --- |
| **Coordinator** | 多 Agent 协调模式 | 先尝试自动解决（分类器、钩子、规则），搞不定再问人 |
| **Swarm Worker** | 多 Agent swarm 里的工作 Agent | 有自己的解析逻辑 |
| **投机分类器** | 开启了 `BASH_CLASSIFIER` 且工具是 BashTool | 投机预启动分类结果跟超时赛跑；高置信度时**直接批准，不问人** |
| **Interactive** | 标准兜底路径 | 弹出终端 UI 让用户审批 |

有个关键设计：当分类器或 deny 规则拒绝了一个动作时，系统把拒绝当作**路由信号**而非硬停止。模型会收到拒绝原因，然后在下一轮迭代里调整策略，尝试更安全的替代方案。换句话说，权限系统不只是拦住危险操作，它还在**塑造 Agent 的行为模式**——让模型学会什么是被允许的。

### 4.5 Auto-mode ML 分类器

`yoloClassifier.ts` 里的分类器在启用时参与权限决策。当 `TRANSCRIPT_CLASSIFIER` 打开时，分类器加载三个 prompt 资源：

* • 基础系统 prompt
* • 外部权限模板
* • Anthropic 内部用户的单独内部模板（通过 `USER_TYPE` 和 `forceExternalPermissions` 选）

分类器根据对话转录和权限模板评估拟议的工具调用，输出允许、拒绝或请求手动审批。

有个铁律：**Deny 规则永远优先于 Allow 规则**，即使 Allow 写得更具体。比如你设了一条"拒绝所有 shell 命令"的 deny 规则，又设了一条"允许 npm test"的 allow 规则——deny 赢。广泛的拒绝不能被窄的允许覆盖。这是 deny-first 哲学在规则层面的体现。

### 4.6 纵深防御的隐患：共享故障模式

纵深防御有一个基本假设：如果一层失败了，其他层能兜住。但这个假设有个前提——各层之间是独立的，失败模式不相关。

现实没那么理想。安全研究人员发现了一个问题：包含超过 50 个子命令的复合命令，会降级成单一通用审批提示，而不是逐个子命令跑 deny 规则检查。原因？逐子命令解析会导致 UI 冻结——这是性能约束。

这意味着纵深防御的两层（规则检查和 UI 响应）共享了同一个性能约束。当性能压力足够大时，两层同时退化。这给我们的启发是：评估纵深防御不能只看任何单层能不能被绕过，要看**几层需要同时失败、它们是否共享故障模式**。

### 4.7 预信任初始化时序漏洞

这是独立安全研究者发现的一个真正有意思的漏洞，涉及到**时间维度**——权限管道描述了安全检查的**空间排序**（哪层在前哪层在后），但没捕捉**时间维度**（每个机制在会话初始化过程中什么时候变成活跃的）。

具体怎么回事？

当你启动一个 Claude Code 会话时，系统会做一系列初始化工作：加载 hooks、连接 MCP 服务器、解析设置文件。问题是：这些初始化代码在交互式信任对话框弹出来给用户看**之前**就已经执行了。

换句话说，存在一个"预信任执行窗口"——在这个窗口里，deny-first 评估管道还没有生效，沙箱还没启动，ML 分类器还没加载。但代码已经在跑了。如果你的项目里有一个恶意的 hook 或者一个恶意的 MCP 服务器连接，它在这个窗口里可以不受限制地执行。

这不是某一层的安全检查出了问题，而是安全检查的**启动顺序**本身成了攻击面。扩展架构（hooks、MCP 服务器）在安全架构完全启动之前就已经开始工作了。用通俗的话说：你家的防盗门装得很好，但小偷在你装门之前就已经进来了。

这个发现提醒我们：安全不只是关于检查规则的设计，还关于**这些规则什么时候变得有效**。初始化顺序本身需要安全分析。

### 4.8 信任是怎么随时间演化的

Anthropic 的纵向数据揭示了一个很有意思的模式：

* • 用户使用不到 50 次会话时，自动批准率大约 **20%**
* • 到了 750 次会话时，自动批准率升到 **>40%**
* • 会话持续时间大幅增加
* • 沙箱把权限提示频率降低了约 **84%**

这意味着分级信任梯度不是用户有意识地选择"我要从 default 切到 acceptEdits"——而是**渐进习惯化**。用户用得越多，越信任系统，越懒得审查每个操作。

架构对此的回应很务实：减少人类必须做的决策数量。不是要求人类变得更警惕（93% 证明这不现实），而是通过 ML 分类器和沙箱这些自动化层来补偿人类注意力的自然衰减。93% 的批准率不是一个需要修复的数据——它是一个需要被架构吸收的约束条件。

## 5. 扩展性：四种机制，按上下文成本排队

### 5.1 为什么需要四种？

一句话：不同扩展带来的上下文开销差异太大，一种机制搞不定。

Hooks 几乎不占上下文，MCP 工具却要把完整的 tool schema 塞进 prompt。如果只用一种机制，要么便宜的扩展被过度约束，要么昂贵的扩展吃光上下文窗口。

Claude Code 的答案是分层——按上下文成本从低到高排：

| 机制 | 上下文成本 | 能做什么 | 插入点 |
| --- | --- | --- | --- |
| **Hooks** | 接近零 | 生命周期拦截，事件驱动自动化 | `execute()` : pre/post tool |
| **Skills** | 低（只有 frontmatter 描述） | 领域指令 + 元工具调用 | `assemble()` : 上下文注入 |
| **Plugins** | 中等（可变） | 多组件打包 + 分发 | 三个注入点都能 |
| **MCP 服务器** | 高（完整 tool schema） | 外部服务集成 | `model()` : 工具池 |

什么意思？便宜的扩展可以大量加，不心疼上下文。贵的扩展留给真正需要新工具接口的场景。

### 5.2 三个注入点

整个 Agent 循环只有三个地方可以插入扩展：

* • **`assemble()`**：控制模型看到什么——CLAUDE.md、技能描述、MCP 资源、钩子注入的上下文
* • **`model()`**：控制模型能调什么——内置工具、MCP 工具、SkillTool、AgentTool
* • **`execute()`**：控制动作怎么跑——权限规则、PreToolUse 钩子、PostToolUse 钩子、Stop 钩子

所有扩展最终都落在这三个点上，没有第四个。

### 5.3 工具池组装

`assembleToolPool()` 是"工具池唯一的真相来源"。流程是：

1. 1. 列出基础工具（最多 54 个，含条件性工具）
2. 2. 模式过滤——`CLAUDE_CODE_SIMPLE` 模式下只保留 Bash、Read、Edit
3. 3. Deny 规则预过滤——被拒绝的工具在模型看到之前就移除了
4. 4. MCP 工具合并——deny 规则过滤后跟内置工具合在一起
5. 5. 去重——内置工具优先级高于 MCP 同名工具

关键点：deny 规则在工具池组装阶段就生效了。模型压根看不到被拒绝的工具，连尝试的机会都没有。

### 5.4 Hooks 系统：27 种事件

源码里定义了 27 种钩子事件，按类别分：

* • **工具授权**（5 种）：PreToolUse、PostToolUse、PostToolUseFailure、PermissionRequest、PermissionDenied
* • **会话生命周期**：SessionStart、SessionEnd、Setup、Stop、StopFailure
* • **用户交互**：UserPromptSubmit、Elicitation、ElicitationResult
* • **子 Agent 协调**：SubagentStart、SubagentStop、TeammateIdle、TaskCreated、TaskCompleted
* • **上下文管理**：PreCompact、PostCompact、InstructionsLoaded、ConfigChange
* • **工作空间事件**：CwdChanged、FileChanged、WorktreeCreate、WorktreeRemove

27 种事件覆盖了从用户敲下 prompt 到子 Agent 完成任务的全生命周期。Hooks 是最便宜的扩展机制，但能力覆盖最广。

## 6. 上下文构建与记忆

### 6.1 CLAUDE.md 四级层级

| 层级 | 示例 | 谁能看到 |
| --- | --- | --- |
| **系统级** | `/etc/claude-code/CLAUDE.md` | OS 上所有用户 |
| **用户级** | `~/.claude/CLAUDE.md` | 你自己的全局指令 |
| **项目级** | `CLAUDE.md` 、`.claude/CLAUDE.md`、`.claude/rules/*.md` | 跟着代码库走 |
| **本地级** | `CLAUDE.local.md` | gitignored，只有你看到 |

从上到下，粒度越来越细，覆盖范围越来越小。

### 6.2 一个很精妙的设计选择

CLAUDE.md 的内容是作为**用户消息**塞进上下文的，不是作为 system prompt。这意味着什么？模型对这些指令的遵从是**概率性的**——它大概率会照做，但不保证。

而 deny-first 权限规则是**确定性**的——不管模型怎么想，规则就是规则。

换句话说：**CLAUDE.md 是建议，权限规则是法律**。建议可以听，法律必须遵守。

这种分离很聪明。如果把所有约束都做成确定性的规则，系统会变得极其僵硬；如果都做成概率性的指令，安全就没了。一半一半，刚柔并济。

### 6.3 记忆检索：不用向量，用文件

Claude Code 不搞嵌入索引那套。它的做法更朴素：让 LLM 扫描记忆文件的头部，按需选最多 5 个相关文件。

权衡很明显：嵌入系统可以更精确地检索单条记录，但需要维护索引基础设施。文件方式更粗暴，但每条指令都可读、可编辑、能跟代码库一起提交 review。

对开发团队来说，"能放进 git diff 里 review"这件事可能比"精确到条目的检索"重要得多。

### 6.4 上下文窗口组装顺序

模型每次调用前，上下文按这个顺序组装：

1. 1. System prompt（含 output style 和 `--append-system-prompt`）
2. 2. 环境信息（git 状态）——每个会话只读一次
3. 3. CLAUDE.md 层级——也只读一次
4. 4. 路径作用域规则（延迟加载，需要时才读）
5. 5. 自动记忆（异步预取相关条目）
6. 6. 工具元数据（技能描述、MCP 工具名、延迟工具定义）
7. 7. 对话历史（受压缩约束）
8. 8. 工具结果（文件读取、命令输出、子 Agent 摘要）
9. 9. 压缩摘要（替换较旧的历史段）

前面几项是相对静态的——git 状态和 CLAUDE.md 都做了记忆化，不会每轮重新加载。后面几项是动态的，每轮都可能变化。

## 7. 子 Agent 委托与编排

### 7.1 六种内置子 Agent

| 类型 | 干什么 | 特点 |
| --- | --- | --- |
| **Explore** | 只读调查、搜索代码 | 写和编辑工具被 deny |
| **Plan** | 制定结构化计划 | 走标准权限模型 |
| **General-purpose** | 通用任务 | 用户显式请求时用 |
| **Claude Code Guide** | 入门和文档辅助 | 有自己的 permissionMode 覆盖 |
| **Verification** | 跑测试、lint 验证 | — |
| **Statusline-setup** | 配终端状态行 | 专用 |

### 7.2 隔离是核心设计

子 Agent 的隔离支持几种模式：

* • **Worktree**：创建临时 git worktree，子 Agent 有自己的仓库副本
* • **Remote**：在远程 Claude Code 环境中启动
* • **In-process（默认）**：跟父级共享文件系统，但对话上下文隔离

`SkillTool` 和 `AgentTool` 的根本区别在于：SkillTool 往**当前上下文窗口**注入指令，AgentTool 生成**全新的隔离上下文窗口**。

代价是子 Agent 的 prompt 必须自包含——默认路径不继承父级的对话历史。你不能指望子 Agent 自动知道前面聊了什么。

### 7.3 7 倍 token 消耗

有个数据值得注意：Agent 团队消耗的 token 大约是标准会话的 **7 倍**（plan 模式下）。这也是为什么子 Agent 只返回摘要而不是全历史——如果不这样做，上下文窗口会被撑爆。

每个子 Agent 写自己的转录到单独的 `.jsonl` 文件。返回给父级的只有最终响应文本和元数据。完整的子 Agent 历史永远不会进入父级的上下文窗口。

### 7.4 多实例协调：用文件锁，不用消息队列

Agent 团队之间用**文件锁**协调，不用消息代理或者分布式协调服务。任务通过基于锁文件的互斥从共享列表中认领。

好处是零依赖部署——不需要 Kafka、不需要 Redis。而且完全可调试——任何 Agent 的状态都能通过读纯文本 JSON 文件检查。

简单粗暴，但对 CLI 工具来说够用了。

## 8. 会话持久化与恢复

### 8.1 三条独立通道

Claude Code 有三条互不干扰的持久化通道：

1. 1. **会话转录**：项目作用域，每个会话一个 JSONL 文件
2. 2. **全局 prompt 历史**：只存用户 prompt，存在 `history.jsonl`
3. 3. **子 Agent 侧链**：每个子 Agent 单独的 `.jsonl` + `.meta.json`

三条通道各管各的，互不耦合。

### 8.2 权限不跨会话恢复

`--resume` 和 fork 操作**不恢复会话作用域的权限**。这是刻意的安全选择。

会话被当作隔离的信任域。恢复时把之前的权限带过来虽然方便，但可能把过时的信任决策带入已经变化的上下文。架构选择让你重新授权，而不是隐式地记住之前的决定。

简单来说：每次会话都是白纸，之前的信任关系不延续。麻烦一点，但安全不变量更强。

### 8.3 Append-only：只加不改

压缩操作从不修改或删除之前写入的转录行，只追加新的边界和摘要事件。`compact_boundary` 标记记录 `headUuid`、`anchorUuid`、`tailUuid`，让会话加载器能在读取时修补消息链。

好处？追溯和审计不需要修改之前写入的状态。你可以随时回放整个会话的历史，因为没有任何行被改过。

这跟数据库的 event sourcing 思路一样——append-only 日志 + 读时重组，比 in-place 更新更容易审计。

## 9. Claude Code vs OpenClaw：同样的问题，不同的答案

### 9.1 六个维度对比

| 维度 | Claude Code | OpenClaw |
| --- | --- | --- |
| **系统范围** | CLI/IDE 编码框架，每次会话是临时进程 | 持久 WebSocket 网关守护进程，多通道控制平面 |
| **信任模型** | 每动作 Deny-first + ML 分类器 + 7 种权限模式 | 单受信任操作者；DM 配对和允许列表；可选沙箱 |
| **Agent 运行时** | `queryLoop()` 是系统中心 | Pi-agent 运行器嵌入网关 RPC 分发内 |
| **扩展架构** | 4 种机制按上下文成本分层 | Manifest-first 插件系统，12 种能力类型 |
| **记忆和上下文** | CLAUDE.md 四级 + 5 层压缩 + LLM 扫描 | 工作空间引导文件 + 独立记忆系统 + 可选混合搜索 |
| **多 Agent** | 任务委托子 Agent + worktree 隔离 | 多 Agent 路由 + 子 Agent 委托 |

### 9.2 三个核心观察

**1. 设计问题是稳定的，答案是环境依赖的**

推理在哪里做、安全态势怎么定、上下文怎么管、扩展性怎么组织——这些问题在不同 Agent 系统中反复出现。但答案随部署上下文变化。

**2. 两个系统在多个维度做了相反的赌注**

Claude Code 投资分级的每动作安全评估，OpenClaw 投资边界级的身份和访问控制。Claude Code 以 Agent 循环为架构中心，OpenClaw 以网关控制平面为中心。

没有谁对谁错——因为部署环境不同。Claude Code 是用户本地的 CLI 工具，攻击面是用户的终端。OpenClaw 是多用户的网关系统，攻击面是网络边界。

**3. 它们可以组合，不是互斥的**

OpenClaw 可以通过 ACP 把 Claude Code 当作外部编码框架托管。Gateway 级系统负责身份和访问控制，任务级执行框架负责具体的工具运行和安全。Agent 设计空间是分层的，可以组合。

## 10. 分类与定位

### 10.1 自主度四级

| 类别 | 代表 | 模式 |
| --- | --- | --- |
| **内联补全** | Copilot、Tabnine | 编辑器插件 |
| **对话集成** | Cursor、Windsurf、Cody | IDE 内嵌 |
| **Agent CLI** | Claude Code、Codex CLI、Aider | 工具使用循环 |
| **完全自主** | Devin、SWE-Agent、OpenHands | 沙箱 + 规划 |

Claude Code 有高自主度 Agent 的特征（auto-mode 分类器、后台 Agent 执行），但**默认保留交互式审批**。它卡在 Agent CLI 和完全自主之间。

### 10.2 上下文管理的五个流派

| 方法 | 机制 | 粒度 |
| --- | --- | --- |
| 简单截断 | 丢掉最旧的消息 | 粗 |
| 滑动窗口 | 固定大小的最近历史 | 中 |
| RAG | 检索相关片段 | 细 |
| 单次摘要 | 一次性压缩 | 粗 |
| **分级压缩** | **多层管道** | **极细** |

Claude Code 的五层压缩管道属于"分级压缩"——先用便宜的策略，不够再升级。它有缓存感知压缩和读时虚拟视图语义，不是简单的截断或一次性摘要。

ACE 论文（Zhang et al., 2026）发现了两种压缩失败模式：摘要丢失领域细节、迭代上下文重写导致细节丢失。Claude Code 的应对方式是 CLAUDE.md 层级（累积结构化指令而不是反复总结）和多层渐进压缩。

### 10.3 安全架构的三个轴

生产级编码 Agent 的安全架构沿三个轴变化：

* • **审批模型**：逐动作提示、分类器自动化、无提示事后审查
* • **隔离边界**：OS 容器、文件系统沙箱、权限限定工具池、无
* • **恢复机制**：版本控制回滚、会话权限重置、检查点回退

Claude Code 组合了逐动作 Deny-first、ML 分类器自动审批、可选 Shell 沙箱和会话权限不恢复——多机制分层，不押宝在单一隔离边界上。

## 11. 架构讨论：五对价值在打架

### 11.1 核心张力

| 冲突对 | 张力在哪 | 证据 |
| --- | --- | --- |
| **权威 × 安全** | 审批疲劳 vs 保护 | 93% 批准率说明用户在疯狂点"是"，安全得靠分类器和沙箱补 |
| **安全 × 能力** | 性能 vs 防御 | >50 子命令的复合命令回退到单一通用审批——逐子命令检查会导致 UI 冻结 |
| **适配 × 安全** | 可扩展性 vs 攻击面 | 多个 CVE 利用钩子和 MCP 服务器的预信任初始化 |
| **能力 × 适配** | 主动性 vs 干扰 | 任务完成率 +12~18%，但高频 Suggest 模式下用户偏好从 80-90% 骤降到 47% |
| **能力 × 可靠** | 速度 vs 一致性 | 有界上下文限制全代码库感知，子 Agent 隔离限制跨 Agent 一致性 |

### 11.2 第六对张力：能力放大 vs 长期可持续性

这组数据让人不安：

* • 有项 RCT（16 名开发者，246 任务）发现 AI 工具让开发者**慢了 19%**，但自我感觉快了 20%
* • Cursor 的因果分析（807 仓库）发现代码复杂度增加了 **40.7%**
* • EEG 研究（54 人）发现 LLM 用户的**神经连接减弱**，而且移除 AI 后还没恢复
* • 2023-2024 入门级技术招聘下降了 **25%**

Agent 架构的"能力放大"可能在以长期能力退化为代价。这是架构层面的问题，不是产品层面能解的。

### 11.3 Feature Flags 是构建时变体

一个容易被忽略的架构特性：Feature flags 创建的是**构建时变体**，不是运行时开关。

比如 `TRANSCRIPT_CLASSIFIER=false` 时，整个 auto-mode 分类器代码在构建时就被消除掉了。Feature-gated 模块用动态 `require()` 而非静态 `import`，因为 `feature()` 只在 if/三元条件下有效，受 bun:bundle tree-shaking 约束。

换句话说：**不同的构建目标会产生功能不同的应用**。你看的代码和实际运行的代码可能不一样。

### 11.4 三个贯穿性设计模式

回看整份报告，三个设计选择反复出现：

1. 1. **分级分层优于单体机制**：安全、上下文管理、扩展性都用独立机制的分级栈，不押宝单一方案。
2. 2. **Append-only 优先审计性**：会话转录是 append-only JSONL，权限不跨会话恢复，上下文压缩用读时虚拟投影而不是破坏性编辑。追溯永远比查询重要。
3. 3. **确定性执行框架内的模型判断**：架构信任模型在丰富的执行框架内做判断，而不是用规划图约束模型的选择。98.4% 的代码是基础设施，只有 1.6% 是 AI 决策逻辑。不是巧合。

### 11.5 可以预料到的问题

有界上下文窗口从架构设计上就能预料到一个后果：Agent 生成的代码会展示更高的模式重复和约定违规率。子 Agent 隔离会加剧这个效果——并行 Agent 可以独立重新实现已经存在于其他地方的解决方案。

Cursor 的因果分析已经验证了这一点：代码复杂度增加 40.7%，初始速度提升在第三个月消散到基线。304,000 个 AI 编写 commit 的审计发现约 1/4 的 AI 引入问题持续到最新版本。

## 12. 六个未来方向

### 12.1 静默失败：78% 的 AI 失败你看不到

有个数据挺吓人的：78% 的 AI 失败是不可见的。而且可观测性的采用率（~89%）和离线评估的采用率（52.4%）之间存在巨大差距。

缩小这个差距可能需要额外的架构支撑——生成器-评估器分离、sprint 合约、事后检查——而不是光靠模型改进。

### 12.2 记忆：从上下文管理副产品到独立认知层

Agent 记忆正在变成独立的认知基础层。当前架构有事实层（CLAUDE.md）和工作层（对话窗口）。下一个自然的步骤是经验层——从过去会话自动策展的策略库。

不是"记住用户说了什么"，而是"记住什么策略管用"。

### 12.3 执行框架的四个边界

| 边界 | 方向 | 例子 |
| --- | --- | --- |
| **Where** | 会话、执行框架、沙箱作为可独立替换的接口 | Managed Agents |
| **When** | 持久后台 Agent + 心跳信号 | KAIROS |
| **What** | 执行框架扩展到文本工具返回之外 | 视觉-语言-动作（VLA） |
| **With Whom** | 角色分化的多 Agent 系统 | 多 Agent 协作 |

### 12.4 时间跨度：Agent 能可靠工作多久

METR 研究测量了前沿 Agent 以固定可靠性完成任务的时间跨度。这是个很有意思的度量——不是"能不能做"，而是"能做多久不出错"。

### 12.5 监管在逼近

EU AI Act（2026 年 8 月全面适用）等框架可能会对日志记录、透明度和人工监督施加外部约束。当前架构的 deny-first 评估可以通过会话转录做内部审计，但还没有达到 GPAI 实践准则设想的外部审计形式。

### 12.6 长期人类能力：从诊断到设计

当前的问题是"AI 辅助是否让开发者能力退化"。更长远的问题是：未来的系统能否把可持续性差距当作一等设计问题，而不是下游评估指标？

## 13. 启发总结

### 给 Agent 架构师的八条建议

**1. 投资基础设施，别投资决策脚手架**

前沿模型在编码任务上的能力正在趋同。差异化的不是模型有多聪明，而是周围的操作框架有多扎实。权限、工具路由、上下文管理、恢复逻辑——这些才是真正拉开差距的地方。

**2. 上下文是绑定资源，分级压缩管道是标配**

没有单一压缩策略能应对所有上下文压力。先用便宜的（裁剪、budget reduction），不够再升级到贵的（auto-compact 摘要）。渐进退化，别一刀切。

**3. 安全层必须真正独立**

纵深防御的前提是各层独立。如果安全层共享性能约束（比如都因为 UI 卡顿而被跳过），独立性假设就失效了。评估标准不是"任何单层能否被绕过"，而是"需要多少层同时失败"。

**4. 扩展机制按上下文成本分层**

钩子接近零成本，可以广泛扩展。MCP 工具成本高，留给真正需要新工具接口的场景。中间的 Skills 和 Plugins 填补过渡带。成本低的先上，成本高的谨慎用。

**5. 子 Agent 返回摘要，别返回全历史**

全转录共享会导致上下文爆炸。侧链转录（独立的 JSONL 文件）+ 摘要返回是关键的上下文保护策略。7 倍 token 消耗摆在那里，不控制的话撑不住。

**6. Append-only 优先审计性**

追溯、分叉和审计不需要修改先前写入的状态。事件溯源思路用在 Agent 会话管理上同样好使——append-only 日志 + 读时重组，永远能重放。

**7. 权限不应跨会话恢复**

会话是隔离的信任域。便利性代价换安全不变量。每次重新授权虽然麻烦，但避免了把过时的信任决策带入新上下文。

**8. 概率性指导 + 确定性强制**

用 CLAUDE.md 这种概率性指令指导行为，用 deny-first 这种确定性规则强制安全边界。两者缺一不可——只有指令系统太软，只有规则系统太硬。

### 对 OpenClaw/Hermes Agent 开发的直接启发

如果你在做类似的系统，几个值得借鉴的点：

* • **Gateway 级 vs 执行框架级**：OpenClaw 可以把 Claude Code 当外部编码框架通过 ACP 托管。Gateway 负责身份和访问控制，执行框架负责任务级安全和上下文管理。分工明确。
* • **记忆系统**：OpenClaw 的 structured long-term memory（dreaming、daily notes、memory search）比 Claude Code 的文件扫描更成熟，但两者都坚持文件型透明度。这个方向是对的——用户能读、能改、能 review。
* • **插件架构**：OpenClaw 的 12 种能力类型比 Claude Code 的 4 种机制更丰富，但缺少按上下文成本分层的组织方式。可以在插件系统中引入类似的分层——便宜的能力大量用，贵的能力谨慎用。
