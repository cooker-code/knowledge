## 一、为什么 Anthropic 自己下场写插件？

如果你用过 Claude Code，一定知道它的插件市场（Plugin Marketplace）。在这个市场上，有成百上千个社区贡献的插件，覆盖代码审查、前端设计、测试驱动开发等各种场景。

但有一个插件有点特别——**它不是社区开发者写的，而是 Anthropic 自己官方出品的**，名叫 **feature-dev**。

截至目前，这个插件的安装量已经超过 **20万次**，在官方 marketplace 中属于第一梯队。Anthropic 给它打上了 **"Anthropic Verified"** 的认证标识，和 Frontend Design、Code Review 等官方插件并列。

这释放了一个明确的信号：**Anthropic 认为结构化的功能开发流程，是 AI 辅助编程中最重要的基础设施之一**。

> 想想看，这其实有点反直觉。大多数开发者用 AI 写代码的方式是：描述需求 → AI 直接生成代码 → 人工检查修改。整个过程快则几十秒，慢则几分钟。
>
> 但 Anthropic 说：**别急着写代码**。

feature-dev 的核心理念是：在动键盘之前，先搞清楚**要做什么、代码库长什么样、有哪些模糊点、架构怎么设计、写完之后质量如何**。

这 7 个阶段走完，代码的质量和可维护性会远超"直接开写"。这不是一个帮你"写得更快"的工具，而是一个帮你"想得更清楚"的工具。

## 二、feature-dev 是什么？

### 一句话定义

**feature-dev 是一个为 Claude Code 设计的官方插件，提供了一套结构化的 7 阶段新功能开发工作流。**

使用方式极其简单：

```
/feature-dev 添加 OAuth 认证
```

或者直接输入 `/feature-dev`，它会引导你一步步完成整个流程。

### 核心哲学（四句话）

| 顺序 | 原则 |
| --- | --- |
| 1 | 先理解代码库，再动手改代码 |
| 2 | 先问清楚问题，再开始做设计 |
| 3 | 先想好架构，再写第一行代码 |
| 4 | 先审查质量，再提交代码 |

这四句话听起来像是软件工程的常识，但在 AI 辅助编程的场景下，它们被赋予了新的意义——**因为 AI 太容易"自信地写错代码"了**。

## 三、7 个阶段拆解

feature-dev 的核心是一套 **7 阶段工作流**，每个阶段都有明确的目标、动作和决策点。

### Phase 1：Discovery —— 理解需求

搞清楚到底要构建什么

|  |  |
| --- | --- |
| **目标** | 搞清楚到底要构建什么 |
| **动作** | 追问核心问题、约束条件、模糊点 |
| **关键设计** | 结束时 Claude 会总结自己的理解并和你确认，确保双方对需求理解一致 |

**举例**：你说"加个缓存"，它会追问——

* 缓存什么数据？
* 性能要求是什么？
* 有没有偏好的缓存方案？

### Phase 2：Codebase Exploration —— 探索代码库

深入理解现有代码，找到相关的模式和架构

|  |  |
| --- | --- |
| **目标** | 深入理解现有代码，找到相关的模式和架构 |
| **动作** | 并行启动 2-3 个 code-explorer agent，从不同角度探索 |

**三个 Agent 的分工**：

| Agent 职责 | 探索内容 |
| --- | --- |
| Agent A | 类似功能的实现方式 |
| Agent B | 目标区域的架构和抽象层 |
| Agent C | 相关功能的当前实现 |

**每个 Agent 返回**：

* 入口点和调用链
* 数据流和转换过程
* 架构层次和设计模式
* 关键文件的具体行号引用

### Phase 3：Clarifying Questions —— 澄清所有模糊点

在设计之前，消除所有歧义

|  |  |
| --- | --- |
| **目标** | 在设计之前，消除所有歧义 |
| **动作** | 系统性识别需要澄清的问题 |

**典型问题清单**：

* **边界情况**

  ：异常输入怎么处理？
* **错误处理**

  ：失败时的降级策略？
* **集成点**

  ：和现有模块如何交互？
* **向后兼容**

  ：会不会破坏现有功能？
* **性能需求**

  ：有没有延迟或吞吐量要求？

**关键设计**：这个阶段会**停下来等你回答**，不会自作主张继续往下走。

### Phase 4：Architecture Design —— 设计多种实现方案

给你选择权，而不是只给一个答案

|  |  |
| --- | --- |
| **目标** | 给你选择权，而不是只给一个答案 |
| **动作** | 并行启动 2-3 个 code-architect agent，从不同角度设计方案 |

**三种方案对比**：

| 方案 | 策略 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 最小改动 | 尽量复用现有代码 | 快速、低风险 | 耦合度高、难测试 |
| 干净架构 | 重新设计抽象层 | 可维护、可测试 | 文件多、重构量大 |
| 务实平衡 | 在速度和质量间取平衡 | 复杂度适中 | 仍有部分耦合 |

> Claude 会基于代码库分析给出推荐方案，但**最终选择权在你手上**。

### Phase 5：Implementation —— 开始写代码

按照选定的架构实现功能

|  |  |
| --- | --- |
| **目标** | 按照选定的架构实现功能 |
| **关键设计** | 不会自动开始，**等你明确批准后才动手** |

实现过程中会：

* 严格遵循 Phase 2 发现的代码库模式
* 严格遵循 Phase 4 设计的架构方案
* 持续跟踪进度

### Phase 6：Quality Review —— 质量审查

确保代码简洁、DRY、优雅且功能正确

|  |  |
| --- | --- |
| **目标** | 确保代码简洁、DRY、优雅且功能正确 |
| **动作** | 并行启动 3 个 code-reviewer agent，分别关注不同维度 |

**三个审查维度**：

| 审查维度 | 关注点 |
| --- | --- |
| 简洁性/DRY/优雅度 | 代码质量和可维护性 |
| Bug/正确性 | 功能正确性和逻辑错误 |
| 规范/抽象 | 项目标准和模式一致性 |

**输出**：

* 审查结果按严重程度分类
* 给出具体的文件行号引用和修复建议
* 由你决定：立即修复 / 稍后修复 / 直接通过

### Phase 7：Summary —— 总结交付

记录完成了什么

|  |  |
| --- | --- |
| **目标** | 记录完成了什么 |
| **交付物** | 构建的功能、关键决策、修改的文件、后续步骤建议 |

## 四、三大 Agent 深度解析

feature-dev 的核心能力来自三个**专业化 Agent** 的协作。它们不是简单的 prompt 模板，而是有明确职责分工的深度分析工具。

### 1. code-explorer：代码考古学家

|  |  |
| --- | --- |
| **核心职责** | 深度分析现有代码库 |
| **触发阶段** | Phase 2 |
| **关键能力** | 追踪执行路径、映射架构层 |

**工作任务**：它不只是搜索文件名或关键词，而是追踪完整的执行路径——

* 从入口点出发，沿着调用链逐层深入
* 分析数据在每个环节如何被转换和传递
* 识别架构层次之间的依赖关系
* 标记出关键的实现细节和设计决策

**输出不是"找到了这些文件"**，而是"这个功能从入口到出口经历了这些步骤，涉及这些组件，遵循这些模式"。

### 2. code-architect：架构设计师

|  |  |
| --- | --- |
| **核心职责** | 基于代码库现状设计实现方案 |
| **触发阶段** | Phase 4 |
| **关键能力** | 多方案对比、文件级路线图 |

**核心能力**：

* 不是凭空设计，而是先分析代码库中已有的模式和惯例
* 理解现有的架构决策及其原因
* 设计与现有风格一致的组件
* 给出具体的实现路线图（精确到文件级别）

**最独特的地方**：会同时生成多种方案，让你在"快但耦合"和"慢但优雅"之间做出**知情决策**。

### 3. code-reviewer：质量守门员

|  |  |
| --- | --- |
| **核心职责** | 审查代码质量 |
| **触发阶段** | Phase 6 |
| **关键能力** | 置信度过滤、多维度审查 |

**杀手锏：置信度过滤机制**

* 对每个发现给出置信度评分（0-100）
* **只报告高置信度（≥80%）的问题**
* 按严重程度分级：

+ 关键问题：75-100
+ 重要问题：50-74

* 提供具体的修复建议和文件行号引用

> 意味着着你看到的**不是一堆噪音**，而是真正需要关注的问题。

### Agent 能力对比表

| Agent | 核心职责 | 触发阶段 | 关键能力 |
| --- | --- | --- | --- |
| code-explorer | 深度分析代码库 | Phase 2 | 追踪执行路径、映射架构层 |
| code-architect | 设计实现方案 | Phase 4 | 多方案对比、文件级路线图 |
| code-reviewer | 审查代码质量 | Phase 6 | 置信度过滤、多维度审查 |

## 五、实战体验：添加 OAuth 认证

让我们用一个具体场景来感受完整流程。

**输入**：

```
/feature-dev 添加 OAuth 认证
```

| 阶段 | 输出 |
| --- | --- |
| **Phase 1** | Claude 确认需求——你要添加 OAuth 认证，需要明确使用哪些 OAuth Provider、是替换还是补充现有认证、如何处理会话等 |
| **Phase 2** | 三个 explorer agent 并行工作后返回：现有认证系统使用 JWT + 中间件模式（src/auth/AuthService.ts:45）、会话管理基于 Redis 24 小时过期（src/session/）、API 安全层已有限流和 CORS 配置（src/api/middleware/） |
| **Phase 3** | 需要你回答 5 个问题——支持哪些 OAuth Provider？存储 Token 还是只存用户信息？和现有认证的关系？会话如何集成？OAuth 失败如何处理？ |
| **Phase 4** | 三种方案——最小改动（扩展 AuthService）、干净架构（新建 OAuthService）、务实平衡（新建 OAuthProvider 抽象层）。**推荐务实平衡方案** |
| **Phase 5** | 你批准后，Claude 按照选定方案实现代码，创建新文件、修改现有文件，全程遵循代码库惯例 |
| **Phase 6** | 三个 reviewer agent 发现——OAuth 回调缺少错误处理（高优先级）、OAuth state 未清理导致内存泄漏（高优先级）、Token 刷新逻辑可简化（中优先级） |
| **Phase 7** | 完整的功能总结，包括修改的文件列表和建议的后续步骤（添加测试、支持更多 Provider、更新文档） |

### 对比"直接让 Claude 写 OAuth 代码"

feature-dev 的优势在于：**你不仅得到了能跑的代码，还得到了**：

1. 对代码库的深度理解
2. 经过权衡的架构决策
3. 有针对性的质量反馈

## 六、什么时候该用，什么时候不该用

feature-dev **不是万能的**。官方也明确给出了适用和不适用场景：

### 适合使用

| 场景 | 说明 |
| --- | --- |
| 涉及多个文件的新功能开发 | 需要跨文件修改 |
| 需要做架构决策的功能 | 涉及技术选型 |
| 和现有代码有复杂集成关系的场景 | 需要理解已有架构 |
| 需求本身还不够清晰的功能 | 需要逐步澄清 |

### 不适合使用

| 场景 | 说明 |
| --- | --- |
| 单行 bug 修复 | 直接告诉 AI 改哪就行 |
| 简单的配置改动 | 不需要架构分析 |
| 完全明确的简单任务 | 需求已经非常清晰 |
| 紧急的热修复 | 需要快速完成 |

### 最佳实践建议

| 建议 | 说明 |
| --- | --- |
| 在 feature request 中尽量提供细节 | 可以减少 Phase 3 的提问轮次 |
| 信任流程 | 每个阶段都在为下一个阶段打基础 |
| 认真阅读 agent 的输出 | 它们对代码库的分析本身就很有价值 |
| 不要跳过阶段 | 尤其是 Code Review |

## 七、总结

### 一个重要的趋势

**feature-dev 代表了一个重要的趋势：AI 辅助编程正在从"代码生成器"进化为"开发流程引擎"。**

| 过去的协作模式 | feature-dev 提出的新模式 |
| --- | --- |
| 我告诉你做什么，你帮我写代码 | 我告诉你做什么，你帮我理解代码库、澄清需求、设计方案、审查质量——最后才是写代码 |

这种方式快，但质量不稳定，尤其是在复杂场景下。

feature-dev 提出了另一种模式：这不是效率的倒退，而是**质量的跃迁**。

> 对于真正重要的功能开发，**"想清楚"比"写快点"重要得多**。

### Anthropic 的示范意义

Anthropic 作为 Claude 的创造者，**亲自下场写这个插件**，本质上是在示范一种最佳实践：

**AI 时代的软件开发，不是让 AI 替代你的思考，而是让 AI 增强你的思考。**

### 如何安装

如果你正在使用 Claude Code，**强烈建议安装 feature-dev 试试**。

两种安装方式：

1. **命令安装**

   ```
   /plugin install feature-dev@claude-plugins-official
   ```
2. **图形界面**

在 Claude Code 中运行 `/plugin`，进入 **Discover** 标签页搜索 "feature-dev" 即可
