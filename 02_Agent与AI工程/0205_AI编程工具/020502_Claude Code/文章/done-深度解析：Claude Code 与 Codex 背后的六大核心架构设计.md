**为什么同样的模型，在 Claude Code 或 Codex 里表现神勇，在普通聊天窗口里却常常“智障”？**

**很多开发者在使用过 Claude Code 或 OpenAI 的 Codex CLI 后，都会有这种直观的感受。这并非错觉，而是因为****编程工作的核心，远不止“生成下一个词”那么简单。**

近日，知名 AI 研究者 Sebastian Raschka 博士发表了最新文章，深度剖析了编码智能体的设计哲学。他指出：**当前 LLM 系统的进步，很大程度上不在于模型本身，而在于我们如何使用它们。**

本文将为你拆解编码智能体背后的六大核心组件，看看这套“智能体套具”是如何让大模型真正变成“程序员”的。

---

### 01 LLM、推理模型与智能体：不仅仅是概念的区分

在深入架构之前，我们需要先厘清几个常被混淆的概念。Raschka 博士用了一个非常形象的类比：

* **LLM（大语言模型）**：这是**引擎**。它是核心的“下一个词预测”模型。
* **推理模型**：这是**增强版引擎**。本质上还是 LLM，但通过训练或提示，使其在推理时投入更多算力进行中间步骤推理、验证或搜索。
* **智能体**：这是**控制系统**。它是在模型之上的一个控制循环层。给定一个目标，智能体负责决定下一步检查什么、调用什么工具、如何更新状态以及何时停止。
* **智能体套具**：这是**软件脚手架**。它围绕智能体，负责管理上下文、工具使用、提示词、状态和控制流。
* **编码套具**：这是**专用工装**。它是智能体套具的一个特例，专门为软件工程任务设计，管理代码上下文、工具、执行和迭代反馈。

**结论很简单：** 顶级的 LLM 原始能力已趋同，**套具的质量往往是决定系统表现的关键区分因素。** 一个好的编码套具，能通过卓越的上下文管理和工具支持，显著放大模型的能力。

### 02 编码智能体的六大核心组件

Raschka 以其开源的 **Mini Coding Agent** 为例，详细拆解了编码套具的六大支柱。

#### 1. 实时仓库上下文

这是最基础但也最重要的一环。当用户说“修复测试”或“实现某功能”时，模型不能是“瞎子”。

它需要知道：

* 这是不是一个 Git 仓库？
* 当前在哪个分支？
* 有没有 `AGENTS.md` 或 `README` 文档指明运行命令？

**核心逻辑：** 智能体在工作前，会主动收集工作区信息，形成“工作区摘要”。这确保了智能体不是从零开始，而是拥有完成任务所需的项目背景知识。

#### 2. 提示词构建与缓存复用

有了上下文，怎么喂给模型才高效？

如果每次交互都重新构建巨大的 Prompt，既浪费算力又增加延迟。编码套具通常将 Prompt 分为两部分：

* **稳定提示前缀**：包含通用指令、工具描述和工作区摘要。这部分变化不大，运行时会对其进行缓存和复用。
* **变化部分**：包括短期记忆、最近的对话记录和最新的用户请求。

**技术亮点：** 这种设计不仅节省了计算成本，还保证了长会话中的稳定性。

#### 3. 结构化工具访问与使用

这是从“聊天机器人”跨越到“智能体”的关键。

普通的模型只能建议你运行命令，而编码套具里的模型能**真正执行**。但为了安全，套具不会让模型随意发挥，而是提供预定义的、有明确边界的工具集（如列出文件、读取文件、运行 Shell 等）。

**执行流程如下：**

1. 模型输出结构化的“动作”。
2. 套具验证该动作（检查工具是否已知、参数是否有效、是否需要用户批准、路径是否在工作区内）。
3. 验证通过后执行。
4. 将结果反馈给循环。

这种“有限自由度”的设计，在赋予模型执行力的同时，极大提升了安全性和可靠性。

#### 4. 上下文管理：对抗“膨胀”

**上下文膨胀** 是编码智能体面临的最大挑战之一。反复读取文件、冗长的工具输出、日志记录……这些都会迅速填满上下文窗口。

优秀的编码套具会采用复杂的策略来“瘦身”：

* **裁剪**：缩短过长的文档片段或工具输出，防止单一内容占用过多预算。
* **记录摘要/压缩**：将完整的会话历史压缩成更小的摘要。通常对近期事件保留细节，对较早事件进行激进压缩。
* **去重**：避免模型因早期多次读取而反复看到相同的文件内容。

**一句话总结：** 很多时候你看到的“模型变笨了”，其实是上下文管理没做好。

#### 5. 结构化会话记忆

智能体需要将状态分层管理，通常分为两层：

* **完整记录**：存储所有用户请求、工具输出和 LLM 响应的完整历史。作为持久化状态，支持会话恢复。
* **工作记忆**：一个更小、经过提炼的状态，明确维护跨轮次的重要信息（如当前任务、重要文件、近期笔记）。

前者是为了“存档”，后者是为了“连续性”。这类似于人类程序员的工作方式：我们不需要记住每一秒的细节，但我们会记笔记（工作记忆），同时拥有完整的 Git 历史（完整记录）。

#### 6. 委托与有界的子智能体

当任务变复杂时，主智能体可以将工作委托给子智能体，实现并行化。

例如，主智能体正在重构代码，它可以委托一个子智能体去“查找某个符号的定义”或“检查配置文件”。

**关键设计：** 子智能体必须继承足够的上下文才有用，但同时必须受到限制（例如，设置为只读模式、限制递归深度、或进行任务范围限定），以防止多个智能体重复工作、冲突或产生无限递归。

### 03 国内开发者的机遇与挑战

结合国内的大模型落地现状，我有以下几点思考：

**1. 上下文工程是新的“提示工程”**
    在国内，很多开发者还在纠结如何写好 Prompt（提示工程），但 Raschka 的文章告诉我们，在 Agent 时代，**上下文工程** 才是护城河。
国内大模型（如 DeepSeek、Qwen 等）的上下文窗口越来越长，但“能装进去”不代表“能用好”。如何像文章所述，通过裁剪、压缩、去重策略，在有限的窗口内保留最关键的信息，是构建高质量 Coding Agent 的核心难点。这需要极强的工程化能力，而非单纯的算法优化。

**2. 安全与权限的“中式”考量**
   工具调用的验证和批准机制。在国内的企业级落地场景（特别是金融、国企）中，**安全合规** 是红线。
    让 AI 直接执行 Shell 命令或修改代码库，对于很多企业来说是不可接受的风险。因此，国内的 Coding Agent 开发者可能需要更侧重于“只读分析”或“人机协同”模式。例如，Agent 生成 Patch，必须经过人工 Code Review 后才能合并，这种**强管控的沙箱机制**可能是国内产品落地的必经之路。

**3. “套具”将成为模型厂商的新战场**
    目前国内模型厂商（如智谱、百度、阿里）大多提供 API 或简单的 Chat 应用。但随着竞争加剧，**谁能为模型配套更优秀的“套具”**（如 IDE 插件、CLI 工具），谁就能占据开发者的心智。
    如果把 GLM-4 或 Qwen 放入一个精心设计的 Coding Harness 中，其体验可能并不逊色于 GPT-5 在 Codex 中的表现。这对于国内厂商来说，是一个弯道超车的机会——**不仅要卷模型参数，更要卷工程架构。**

### 结语

编码智能体并非魔法，它是强大的模型与精密的软件工程架构的结合体。

在 AI 时代，不仅要关注模型有多聪明，更要关注我们为模型搭建的“舞台”有多稳固。
