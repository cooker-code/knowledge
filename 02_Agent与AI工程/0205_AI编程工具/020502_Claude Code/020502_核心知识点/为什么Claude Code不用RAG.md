## 一句话结论

这篇文章对用户有价值的地方不是“RAG 已死”，而是帮助校准一个判断：代码库上下文获取更接近动态搜索和验证闭环，不应该直接套文档知识库 RAG 的思路。

## 纵向理解

| 维度 | 判断 |
|---|---|
| 全局架构 | 用户目标 -> Agent 循环 -> 动态搜索/读取文件 -> 模型推理 -> 修改/验证 |
| 本文位置 | 只讲上下文获取路线，没有完整展开权限、执行、压缩和评估 |
| 核心机制 | 动态搜索、按需读取、工具反馈，而不是离线向量索引 |
| 使用链路 | 搜索文件或文本 -> 读取相关片段 -> 修改代码 -> 运行验证 -> 继续补上下文 |
| 边界 | 大型陌生代码库仍可能漏召回；搜索词质量、文件结构和测试覆盖会影响效果 |

## 横向对标

| 对标技术 | 实现方式 | 优势 | 劣势 | 适合场景 |
|---|---|---|---|---|
| 传统代码 RAG | 建索引、向量召回、把片段塞进上下文 | 对稳定代码和文档检索快 | 索引过期、结构关系弱、权限泄露风险 | 稳定文档库、历史知识库 |
| Agentic Search | 动态使用 Glob/Grep/Read 等工具 | 与最新代码一致，证据可追查 | 依赖搜索策略，可能多轮消耗上下文 | 高频变化代码库、调试、局部修改 |
| AST/图索引 | 建符号、调用关系、依赖图 | 结构化理解强 | 构建成本高，语言和框架适配复杂 | 大型代码库、重构、影响面分析 |
