Spec-Driven Development（规范驱动开发）是跟随着AI编程的浪潮发展起来的方法论。它的核心逻辑很简单：先通过结构化规范明确定义“做什么”和“为什么”，再交由 AI 自动生成代码。

目前的代表工具有 Spec Kit（GitHub）、OpenSpec（Fission AI）以及 Kiro.dev。这套方法论的价值显而易见：它能消除需求歧义，降低沟通成本，确保代码与需求高度一致，并利于团队协作与长期维护。

在实际落地中，这两个工具各有侧重。

Spec Kit 适合“从零开始”的项目，主打“specify → plan → tasks → implement”的闭环。

而 OpenSpec 则专为“存量项目”的增量开发设计，工作流为“proposal → review → apply → archive”，强调在现有系统基础上的安全迭代与规范同步。

然而，理想很丰满，现实却有点骨感。AI编程也正在走当年高级语言编程发展的老路。先是结构化编程，再到面向对象编程。同样，这两个工作模式在实际执行中也是各种不顺手。“过分细致、不够灵活”、“不适合国内”等等问题。它们更多提供的是方法论和高阶框架，却难以解决开发者在工位上遇到的具体痛点。

* • 如何确保 AI 严格按照预想执行代码？
* • 面对海量的生成代码，我们该如何有效审查？
* • 如何让 AI 具备自省能力而不是盲目执行？

最关键的是，在追求极致的规范文档与紧张的开发排期之间，我们该如何找到那个平衡点？这不仅是工具的问题，更是我们在增量开发中必须面对的挑战。

今天咱们不谈理论，直接聊聊在增量开发中，怎么用OpenCode把这些方法论落地，把AI用得顺手又顺心。

### 需求澄清

很多时候，AI写出来的代码不对路，通常问题的根源不在于AI，而在于使用它的人。李飞飞讲：“人工智能不是用来替代人类，而是让人类的能力更强大”，所以讲AI强不强大，要看人类想要有多强大。

OpenSpec那种一次性生成design、proposal、task的方式，在存量项目中改动起来确实麻烦。

在实际项目中，我喜欢使用superpowers:brainstorming，对着设计文档或执行计划来个3-5轮次的问答。

比如我们直接说：“给我写个订单导出功能。”

然后试着让使用superpower:brainstorming先发散一下：“我需要优化订单系统的导出性能，目前的痛点是导出几万条数据会超时。你作为资深架构师，先问几个关键问题，帮我澄清需求。”

这时候AI通常会问：

1. 1. 数据量级大概是多少？
2. 2. 是否支持异步导出？
3. 3. 导出格式是Excel还是CSV？
4. 4. 是否需要包含关联的明细数据？

通过这几轮对话，你把“What”和“Why”嚼碎了，让你更了解需求的同时，也让AI接下来制定的执行计划才不会跑偏。这叫“磨刀不误砍柴工”，避免了“一步错，步步错”的惨剧。

### 任务拆分

不管是人写代码还是AI写代码，面对一个复杂任务，直接一句“给我实现一个用户中心”，结果通常都是灾难。

**任务越明确，粒度越小，AI完成度越高。**

在使用OpenCode时，不要试图在一个Prompt里干完所有事。我的习惯是分步走：

1. 1. 第一步：让AI写数据库Model和DTO定义。
2. 2. 第二步：确认无误后，让它生成DAO层接口。
3. 3. 第三步：再生成Service层业务逻辑。

这样做的好处是**时间计划可控**。如果第一步生成的Model字段类型就错了，你可以立马纠正，不至于等它写完了两千行业务代码再推倒重来。

### CRUD开发

对于存量项目里的常规CRUD（增删改查），其实是最适合AI发挥的场景，但抄作业要抄得有水平。项目里已经有了现成的样板代码，虽然说传统的代码生成器也能应付，但有AI我们只要说几句话的事，何必要自己亲自上手设计数据库模型呢？

但这并不意味着可以当甩手掌柜。我的工作流通常是这样的：

**第一步：找参照物**

```
1

  

请先查找一下“A功能”（如用户管理）的增删改查接口及其关联的service、dao、dto、vo、model有哪些，给我列出清单。
```

**第二步：下达模仿指令**

```
1

2

  

我需要完成“B功能”（如角色管理），需求是：支持角色的增删改查，且支持角色与菜单的多对多关联。  
请仿照A功能的代码结构，遵循项目现有的代码规范（如命名规则、异常处理），制定实现功能B的技术方案与实施计划，写入docs/plans目录，等我确认。
```

**第三步：审查关键点**

AI生成的方案里，最需要人工干预的是**数据库设计**。虽然它能模仿表结构，但字段长度、索引策略这些还是得靠经验把关。确认没问题后，再让它执行计划。

**第四步：最后防线**

代码写完提交前，必须看一眼`git diff`。不用逐行读，重点关注那些它“自作聪明”修改的地方，确保没有改坏原本的逻辑。

### 四、 非CRUD需求

遇到复杂的非CRUD需求，比如对接硬件设备（密码机、支付网关等），环境问题（开发没机器）往往是个大麻烦。这时候，AI不仅能写代码，还能帮你做架构设计。

举个对接密码机的例子。为了保证在没有实体密码机的开发环境也能跑通测试，我们需要“软实现”。  
给OpenCode的Prompt可以这样写：

```
1

2

3

4

5

6

7

  

我需要完成对接XX密码机的需求。请根据以下要求帮我设计技术方案与实施计划。  
1. 核心能力（数字摘要、对称加解密、非对称加解密）需要同时支持“硬件密码机”和“软件模拟”两种实现。  
2. 软件实现必须使用国密算法（SM2、SM3、SM4）。  
3. 请采用合适的设计模式（建议策略模式+工厂模式），满足迪米特法则（最少知识原则），调用方无需感知底层使用的是硬件还是软件。  
4. 硬件/软件实现方式可通过配置文件切换，默认使用软件实现。  
参考文档：  
- 密码机对接文档见：docs/hardware/hsm-guide.md
```

这里的关键在于**指定设计模式**。如果不提“策略模式”和“工厂模式”，AI可能会写出一堆`if-else`来判断配置，那代码可读性、扩展性就崩了。

### 五、 接手“屎山”

如果你不幸接手了一个维护了多年的老项目，连文档都没有，别急着写代码。先把AI当“文档工程师”用，先来个提示词问一遍。

```
1

2

3

4

5

6

7

8

  

本项目是一个xxx系统。请基于现有代码，帮我梳理并总结：  
1. 系统整体架构  
2. 核心领域模型（关键概念、对象、业务术语）  
3. 主要模块划分及依赖关系  
4. 技术栈清单（框架、中间件、数据库）  
5. 核心业务流程与数据流向  
6. 数据模型（ER关系简述）  
请将这份分析报告写入 docs/reports/system-architecture-analysis.md。
```

这份报告虽然不能替代真正的架构文档，但能帮你快速建立认知地图。在了解了这些背景后，你再提出的增量开发需求，才不会因为违反了原有系统的某些“潜规则”而导致Bug。

### 结语

用好OpenCode做增量开发，核心不在于它能帮你写多少行代码，而在于你如何驾驭它去理解现有的复杂系统。

**定架构、拆任务、给参照、严审查。** 这十二个字，就能让AI变成“得力助手”。
