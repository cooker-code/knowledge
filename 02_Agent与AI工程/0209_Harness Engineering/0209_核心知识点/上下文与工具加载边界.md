## 核心问题

如何在不撑爆上下文、不增加工具误选成本的前提下，让 Agent 拿到足够、准确、可执行的信息。

## 判断准则

| 输入面 | 适合放什么 | 不适合放什么 |
|---|---|---|
| System Prompt | 全局身份、权限边界、安全规则、输出契约 | 大量项目知识和低频规则 |
| AGENTS / Skill 索引 | 稳定流程、目录入口、触发条件、渐进披露入口 | 单篇文章摘要和一次性结论 |
| Skill 正文 | 任务特定 SOP、模板、脚本调用方式 | 所有任务都要加载的通用常识 |
| MCP / 工具声明 | 高风险、需要外部系统、需要审计的能力 | 数十个相似 API 的平铺清单 |
| 代码模式 | 搜索、批处理、复杂转换、一次性组合工具 | 无边界地执行高危操作 |
| SubAgent | 隔离上下文、并行探索、独立评审 | 用角色扮演替代状态和权限 |

## 关键原则

- 工具越多，选择成本、误用成本和上下文占用越高；少而硬的工具层优先。
- Skill 先加载摘要和触发条件，只有命中任务时再加载正文和模板。
- MCP 适合声明稳定、高风险、需要审计或用户确认的操作；低风险组合动作可以用代码模式压缩工具面。
- Prompt Cache 改变的是成本和延迟，不改变事实正确性；不能用缓存收益掩盖信息过期。
- System Prompt 是全局神经中枢，但项目事实必须留在可更新的文件、索引和外部证据中。

## 待验证缺口

- 需要对比：大量 MCP 工具声明、单个代码执行工具、Skill 渐进披露三种方式在任务成功率、成本和误用上的差异。
- 需要建立工具归并规则：何时保留独立工具，何时下沉为脚本，何时只保留文档入口。
