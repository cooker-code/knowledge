# 020900_Harness Engineering知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 02_Agent与AI工程 / 0209_Harness Engineering |
| 解决的问题 | 给 Agent 增加可执行的工程约束层，避免任务只靠模型自觉和长提示词推进 |
| 典型输入 | 任务规格、项目规则、环境、工具权限、测试门禁、评估集、运行日志 |
| 典型产出 | Harness 模板、状态机、质量门禁、评估报告、回滚策略、交付证据 |
| 用户阅读目标 | 判断文章补的是环境、状态、门禁、评估、观测、回滚还是知识沉淀 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| HE00 任务规格 | Agent 接到的任务是否可执行 | Harness 从明确任务、输入、验收开始 | Spec、PRD、任务分解文章进入 |
| HE01 运行环境 | Agent 能在哪些环境和权限里执行 | 环境是安全和可复现的基础 | 沙箱、浏览器、CLI、依赖环境文章进入 |
| HE02 状态机 | 任务如何推进、暂停、恢复、失败 | 状态机比长提示词更稳定 | 状态机、长任务、Workflow 文章进入 |
| HE03 工具与权限 | 工具如何声明、审批、限制和审计 | Harness 需要管理工具，而不是只暴露工具 | Skill、MCP、CLI、浏览器工具文章进入 |
| HE04 质量门禁 | 什么时候允许继续、回滚或交付 | 没有门禁就只是自动化脚本 | 测试、评估、Lint、Review 文章进入 |
| HE05 观测评估 | 如何看到多个 Agent 的过程和结果 | Trace、Monitor、坏例集是生产化证据 | Monitor、自动评测、LLM-as-judge 文章进入 |
| HE06 知识沉淀 | 经验如何进入下一次运行 | Harness 要沉淀规则、失败和修复方式 | Skill、Memory、团队知识文章进入 |
