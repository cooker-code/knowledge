> 说明：本文聚焦“工程方法论”而非复现细节。源码分析接下来会写。

很多人把 AI 编程工具的体验差异，归因于“模型更强”。

但如果你用过足够多工具，就会发现：同一代模型，体验也能天差地别。

这次围绕 Claude Code 的“源码泄露”讨论，一个更值得行业学习的点是：它让我们更清楚地看到——一个 AI Agent 想从“会回答”变成“能交付”，靠的不是一句 prompt，而是一整套工程系统。

工程圈这两年开始流行一个说法：harness engineering。

先给一个更精确的工作定义，避免误会：

harness engineering = 围绕提示词系统 + 工具层 + 权限/审计 + 记忆/压缩 + 多代理/验证 + 产品化工作流，做的一整套“让 AI 稳定交付”的系统工程。

下面用 8 个模块，把这套方法论拆开讲清楚。

# 模块一：系统提示词不是一段话，而是一套“说明书装配线”

用户看到的是一句话输入；模型实际拿到的，往往是一整本“说明书”：身份定义、安全边界、工具规则、风格约束、以及和当前项目相关的动态上下文。

工程上最关键的不是“写一句更长的 prompt”，而是把提示词当作可维护的系统：

* 静态部分：所有用户共享、可缓存，降低成本、提升一致性
* 动态部分：跟随用户/项目变化，保证个性化与实时性

这会直接决定：

* 模型的行为边界是否稳定
* 输出风格是否一致
* 工具使用是否“像个专业同事”

# 模块二：权限系统不是弹窗，而是一条“风险分流流水线”

很多人以为 Auto 模式就是“全自动放行”。但更成熟的做法是：把风险判断从主模型里拆出去，让系统在后台做分层拦截。

一个不敏感、但非常好理解的例子（你可以拿它当团队内部的权限分级草案）：

* 读文件/查状态（只读）→ 通常可自动放行
* 跑测试/拉依赖/改配置（会执行/会改环境）→ 通常需要确认
* 写入代码/批量改文件（会改产物）→ 视范围与仓库敏感度决定是否确认
* 发版/推送/触发 CI/CD（会影响外部）→ 通常要更严格的确认与审计

更进一步的做法，是用一个“独立判官”模型/分类器来评估操作风险：主模型负责做事，判官负责把关。

一句更像人话的总结：

> 你不是在“限制 AI”，你是在给它装护栏——护栏装好了，它才跑得快。

常见误区：只靠“弹窗确认”。弹窗多了人会麻，麻了就会一路点“允许”，最后等于没护栏。

# 模块三：工具层要“少而硬”，并且每个工具都有认知成本

工具不是越多越好。每加一个工具，系统就得多回答三件事：它是什么、怎么用、什么时候用。然后你会付出三笔成本：

* 上下文占用：工具说明会挤占有效推理空间
* 选择成本：工具多了，模型更容易“犹豫/乱选”
* 误用成本：工具一旦被误用，错的不是结果，是过程

一个很常见的“翻车现场”：

> 工具越多，越容易把关键约束挤出上下文；最后你以为模型变笨了，其实是它被你塞了一堆工具说明书。

高质量的工具层，通常具备三个特征：

1. 基础能力齐全：读写文件、搜索、diff、运行命令、查看状态
2. 语义清晰：输入输出稳定、可预期
3. 带限制：哪些参数不能用、哪些操作必须确认，写进系统规则

很多时候，最先进的工具用的就是最朴素的文本搜索（例如 grep/ripgrep）。原因很简单：当“理解大脑”足够强，搜索不必花里胡哨。

# 模块四：记忆系统的关键不是“记更多”，而是“记对的东西”

记忆做不好，会变成误导。

成熟系统往往把记忆拆成两类：

* 长期偏好：例如风格、习惯、约束、决策口径
* 短期事实：例如当前改到哪、有哪些未解决问题

尤其重要的一条工程决策是：

> 记忆更适合存“人的偏好与判断”，不适合存“会变化的代码事实”。

因为代码会重构、文件会移动、行号会漂移。把代码事实写死进长期记忆，几乎必然过期。

# 模块五：上下文压缩要结构化，否则“总结=遗忘”

长对话的敌人不是 tokens 不够，而是“关键约束被压缩丢了”。

最容易发生的悲剧是：你让它“总结一下”，它把你最在意的禁用项一起总结没了。

所以好的压缩策略不是自由总结，而是结构化提取。给你一个 5 行模板（通用、不敏感）：

* 目标：这次到底要交付什么？
* 约束：不能做什么？必须遵守什么？
* 证据：我们基于哪些文件/日志/结果在判断？
* 进度：已经做了什么、哪里失败过？
* 下一步：接下来最小步行动是什么？

常见误区：只压缩“结论”，不压缩“约束”。结论忘了还能再推一遍，约束忘了就会直接踩坑。

# 模块六：多 Agent 不是炫技，是吞吐与可靠性的工程解法

复杂任务很少能靠一个“全能模型”一次做对。多 Agent 的意义，是把“会做”变成“做得完、做得对”。

一个最通用、也最不敏感的分工例子是：

* 主代理：对话、确认范围、维护全局状态
* 研究 worker：查资料/读文档/列方案
* 实现 worker：改代码/改配置/补脚手架
* 验证 worker：跑测试/做对抗性检查/挑错

你会发现：真正让团队省心的不是“实现 worker 很能写”，而是“验证 worker 专门负责说不”。

常见误区：把多 Agent 当成“开并发”。并发只是速度，多 Agent 的难点在于一致性：最后要有一个统一的答案、一套统一的改动、一条统一的交付链。

---

# 模块七：内部版 vs 外部版：同一工具的“两套标准答案”

很多公司会在内部环境里跑一套更严格、更克制的配置。一个不敏感但很典型的场景是：

* 内部：默认要求“给证据”——引用 commit/日志/测试结果；不确定就明确写“不确定”；并尽量避免暴露内部代号、路径、仓库名
* 外部：默认要求“更易读、更少上下文依赖”——假设用户会丢失上下文，输出要自洽；同时把内部信息暴露降到最低

如果你在团队里频繁遇到这类现象：

* AI 在外部协作时“顺嘴”带出内部术语/路径/代号
* 或者输出看起来很自信，但一问证据就开始含糊

那基本就说明：你需要一套“内外两套策略”，把口径与暴露面写进系统规则，而不是靠人肉提醒。

---

# 模块八：产品化层：除了能力，还要考虑长期关系

当工具开始变成“持续在线的助手”，它就不再是一次性调用。

这里有一个非常好用的“产品化不足信号”（不敏感但很真实）：

* 长任务没有可回溯的进度与状态
* 用户只能反复追问“现在到哪了、还要多久、出了什么问题”
* 追问越多，信任越低，最后只剩“把它当搜索框用”

所以产品要回答的问题会变化：

* 用户怎么信任它、怎么授权它
* 系统怎么解释它为什么这么做
* 长任务怎么可视化、怎么可回溯
* 团队协作怎么接入（权限、同步、审计）

甚至会出现一些“看起来不重要”的设计（例如更轻的提示与反馈机制）。它们未必是主护城河，但往往决定用户是否能长期用下去。

围绕“源码泄露”的讨论很热，但真正有价值的是这套启示：

模型能力决定上限，harness engineering 决定稳定性与可交付性。

最后用 5 个自检问题收口（不提供步骤，只帮你判断系统短板在哪）：

1. 我的系统提示词是“能维护的系统”，还是“堆出来的一段话”？
2. 我的风险控制是“分流流水线”，还是“弹窗祈祷学”？
3. 我的记忆是在记“偏好与判断”，还是在记“注定过期的事实”？
4. 我的压缩是“结构化保约束”，还是“总结式遗忘”？
5. 我的验证是“独立角色说不”，还是“让同一个模型自证清白”？

当你把这 5 个问题回答清楚，你就已经开始做 harness engineering 了。
