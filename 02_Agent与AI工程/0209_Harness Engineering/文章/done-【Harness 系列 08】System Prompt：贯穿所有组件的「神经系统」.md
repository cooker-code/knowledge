> 在讨论完六大组件后，还有一个贯穿始终的要素值得单独讨论——System Prompt。

## 一、System Prompt 不是第七个组件

System Prompt 不是 Harness 的「第七个组件」，但它是**贯穿所有六个组件的「神经系统」**。

它在 Harness 架构中扮演着四个关键角色：

### 1. 定义角色边界

System Prompt 告诉模型「你是谁」和「你不是谁」。这不是简单的角色扮演，而是在定义 Agent（SubAgent） 的能力边界和行为约束。

**示例：**

```
  你是一个前端开发 Agent，专注于 React 生态系统。
当用户提出后端相关需求时，建议他们使用专门的后端 Agent。

你可以：
- 编写 React 组件
- 优化前端性能
- 调试浏览器问题

你不应该：
- 编写数据库 Schema
- 配置服务器
- 处理 DevOps 任务
```

这样的定义帮助 Agent 聚焦在自己擅长的领域，避免在不擅长的领域产出低质量结果。

### 2. 注入领域知识

System Prompt 是「零成本」知识注入的第一入口。项目背景、技术栈选型、核心业务逻辑——这些信息直接写在 System Prompt 中，确保 Agent 从第一轮对话开始就具备必要的领域知识。

**示例：**

```
  项目背景：
- 这是一个电商后台管理系统
- 技术栈：React 18 + TypeScript + Ant Design
- 后端 API 使用 REST 风格
- 部署在 Kubernetes 集群

核心业务流程：
1. 用户登录 → JWT 认证
2. 商品管理 → CRUD 操作
3. 订单处理 → 状态机驱动
4. 数据分析 → 异步报表生成
```

当然，正如前文讨论的，System Prompt 的容量有限。当领域知识量超过 System Prompt 能承载的范围时，AGENTS.md 就接管了这个职责。

**两者是互补关系：**

* • System Prompt 装「必须知道的」（角色定义、安全规则、核心约束）
* • AGENTS.md 装「最好知道的」（项目细节、最佳实践、历史经验）

### 3. 约束安全规则

System Prompt 是 Agent 安全策略的「第一道防线」。哪些操作是允许的，哪些是禁止的；哪些数据可以读取，哪些是敏感的；遇到不确定的情况应该怎么处理——这些安全规则通过 System Prompt 直接「刻」进 Agent 的行为模式。

**示例：**

```
  安全规则：

1. 文件操作
   - 只能访问 /workspace 目录及其子目录
   - 禁止访问 /etc, /root, /home 等系统目录
   - 删除文件前必须确认

2. 网络访问
   - 允许访问：npm registry, PyPI, GitHub
   - 禁止访问：内部 API, 数据库直连

3. 代码执行
   - 禁止执行：rm -rf /, mkfs, dd
   - 敏感操作需要用户确认

4. 数据处理
   - 敏感数据必须脱敏
   - 禁止记录用户密码
   - 日志中不包含 PII
```

### 4. 贯穿所有组件

System Prompt 的影响范围不限于模型本身，它通过定义 Agent 的行为模式，间接影响了所有 Harness 组件的工作方式：

| 组件 | System Prompt 的影响 |
| --- | --- |
| **文件系统** | 定义文件的组织规范和命名约定 |
| **Bash 执行** | 限制可执行的命令范围 |
| **记忆写入** | 规定知识记录的格式和标准 |
| **搜索行为** | 设置搜索的优先来源和可信度标准 |
| **上下文管理** | 定义信息的优先级和压缩策略 |
| **编排逻辑** | 规定任务分解的粒度和协作规范 |

**在这个意义上，写 System Prompt 就是在设计 Harness 的行为规范。**

一个好的 System Prompt 就像一部好的公司章程——它不直接做任何事，但它决定了所有事怎么做。

## 二、写好 System Prompt 的艺术

### 1. 结构清晰

一个好的 System Prompt 应该有清晰的结构：

```
  # System Prompt

## 角色定义
[你是谁，能做什么，不能做什么]

## 领域知识
[项目背景、技术栈、业务逻辑]

## 工作规范
[编码规范、文档规范、沟通规范]

## 安全规则
[允许/禁止的操作，敏感数据处理]

## 输出格式
[期望的输出结构和风格]

## 异常处理
[遇到问题的处理方式]
```

### 2. 具体明确

避免模糊的描述，要具体明确：

| ❌ 模糊 | ✅ 具体 |
| --- | --- |
| "写高质量的代码" | "遵循 Airbnb JavaScript 规范，使用 ESLint 检查" |
| "注意性能" | "首屏加载时间 < 1s，使用 Lighthouse 评分 > 90" |
| "做好错误处理" | "所有异步操作使用 try-catch，错误信息包含堆栈" |

### 3. 优先级明确

当规则冲突时，明确优先级：

```
  优先级（从高到低）：
1. 安全规则（最高优先级，不可违背）
2. 用户需求（用户明确要求的）
3. 项目规范（AGENTS.md 中的约定）
4. 最佳实践（通用编程规范）
5. 个人偏好（System Prompt 中的风格偏好）
```

### 4. 可验证

规则应该是可验证的：

```
  可验证的规则：
✓ "使用 2 空格缩进"（可通过 lint 验证）
✓ "函数长度不超过 50 行"（可通过工具统计）
✓ "所有 API 必须包含错误处理"（可通过代码审查验证）

不可验证的规则：
✗ "写简洁的代码"（什么是简洁？）
✗ "保持良好的可读性"（如何衡量？）
```

## 三、System Prompt 的演进

System Prompt 不是一成不变的，它会随着项目演进不断优化。

### 演进示例

**V1.0：项目启动时**

```
  你是一个前端开发工程师，使用 React 和 TypeScript。
```

**V1.1：第一周**

```
  你是一个前端开发工程师，使用 React 18 和 TypeScript。
技术栈：
- React 18
- TypeScript
- Tailwind CSS
- Vite

编码规范：
- 使用函数式组件
- Props 必须定义类型
```

**V1.2：第一个月**

```
  你是一个前端开发工程师，使用 React 18 和 TypeScript。

技术栈：
- React 18
- TypeScript
- Tailwind CSS
- Vite
- React Query（数据获取）
- Zustand（状态管理）

编码规范：
- 使用函数式组件
- Props 必须定义类型
- 异步操作必须使用 try-catch
- 组件文件使用 PascalCase

安全规则：
- 禁止在代码中硬编码 API Key
- 用户输入必须验证
```

**V2.0：第三个月**

```
  你是一个前端开发工程师，专注于 React 生态系统。

## 技术栈
- React 18 + TypeScript
- Tailwind CSS + Headless UI
- Vite + SWC
- React Query + Zustand
- React Router v6
- React Hook Form + Zod

## 编码规范
- 函数式组件 + Hooks
- Props 必须定义 TypeScript 接口
- 异步操作必须使用 try-catch
- 组件文件使用 PascalCase
- 工具函数使用 camelCase
- 常量使用 UPPER_SNAKE_CASE

## 性能要求
- 首屏加载 < 1s
- Lighthouse 评分 > 90
- 使用 React.memo 优化重渲染
- 图片使用懒加载

## 安全规则
- 禁止硬编码敏感信息
- 用户输入必须验证（Zod schema）
- XSS 防护：dangerouslySetInnerHTML 需要审查
- CSRF 防护：使用 SameSite Cookie

## 测试要求
- 组件必须有基本测试
- 关键逻辑单元测试覆盖 > 80%
- 使用 React Testing Library

## 异常处理
- 遇到不确定的问题，先询问用户
- 发现安全漏洞，立即上报
- 性能问题需要量化数据支撑
```

可以看到，System Prompt 随着项目深入不断丰富，从简单的角色定义发展为详细的行为规范。

## 四、System Prompt 与其他组件的关系

### System Prompt vs AGENTS.md

| 维度 | System Prompt | AGENTS.md |
| --- | --- | --- |
| **更新频率** | 低频（随项目阶段更新） | 高频（随时更新） |
| **内容类型** | 角色定义、安全规则、核心约束 | 项目细节、最佳实践、历史经验 |
| **谁维护** | 项目初始化者 | Agent 自己 + 团队 |
| **容量** | 有限（4K-8K tokens） | 理论上无限 |
| **优先级** | 高（直接影响行为） | 中（补充知识） |

**协作关系：**

* • System Prompt 定义「必须遵守的规则」
* • AGENTS.md 提供「应该知道的信息」
* • 两者结合，构成完整的知识体系

### System Prompt vs 上下文工程

上下文工程决定「什么信息在上下文中」，System Prompt 决定「模型如何理解这些信息」。

**示例：**

```
  上下文工程：
- 决定加载哪些 AGENTS.md
- 决定历史信息的压缩策略
- 决定工具输出的处理方式

System Prompt：
- 告诉模型如何解读 AGENTS.md
- 告诉模型历史信息的优先级
- 告诉模型工具输出的期望格式
```

## 五、System Prompt 的调试技巧

### 1. A/B 测试

对关键任务，测试不同 System Prompt 的效果：

```
  测试组 A：详细规范
测试组 B：简洁指令
测试组 C：示例驱动

评估指标：
- 任务完成率
- 代码质量评分
- 用户满意度
```

### 2. 版本控制

System Prompt 应该版本化：

```
  system-prompts/
├── v1.0.md
├── v1.1.md
├── v2.0.md
└── current.md -> v2.0.md
```

### 3. 效果监控

监控 System Prompt 的效果：

```
  监控指标：
- 规则违反次数
- 用户纠正次数
- 任务成功率
- 输出一致性
```

### 4. 渐进优化

不要一次性大幅修改，而是渐进优化：

```
  优化流程：
1. 识别问题（规则被违反）
2. 分析原因（规则不清晰）
3. 小步修改（明确规则表述）
4. 验证效果（监控指标）
5. 继续迭代
```

## 六、总结

System Prompt 是贯穿 Harness 所有组件的「神经系统」。

**四个关键角色：**

1. 1. **定义角色边界**——明确能做什么，不能做什么
2. 2. **注入领域知识**——零成本知识输入
3. 3. **约束安全规则**——第一道安全防线
4. 4. **贯穿所有组件**——影响整个 Harness 的行为

**写好 System Prompt 的关键：**

* • 结构清晰
* • 具体明确
* • 优先级明确
* • 可验证

**演进原则：**

* • 随项目深入不断丰富
* • 版本控制，A/B 测试
* • 监控效果，渐进优化

## 结语：你就是 Harness 的一部分

让我们回到文章开头的那个公式：

**Agent = Model + Harness**

如果你是一名 AI 工程师、产品经理、技术负责人——如果你不是模型本身——那你就是 Harness 的一部分。

* • 你写的 System Prompt 是 Harness 的神经系统
* • 你搭建的工具链是 Harness 的四肢
* • 你设计的记忆机制是 Harness 的长期记忆
* • 你优化的上下文策略是 Harness 的注意力管理
* • 你编写的 Hook 规则是 Harness 的质量底线

过去两年，行业的聚光灯一直打在模型身上——参数规模、基准分数、推理能力。但真正在生产环境中创造价值的，往往不是那个最强的模型，而是那个最好的 Harness。

**模型会继续进步，但进步的速度在放缓。与此同时，Harness 的工程空间还几乎没有被充分探索。**

文件系统怎么设计才能最大化 Agent 的效率？上下文工程有没有最优的压缩算法？编排模式怎样才能在成本和质量之间取得完美平衡？记忆机制能否实现真正的「终身学习」？

这些问题的答案，不在模型的论文里，而在 Harness 的工程实践中。

**模型决定了 Agent 能力的下限，Harness 决定了 Agent 能力的上限。**

下一次当你在调试 Agent 的表现时，不要急着换一个更强的模型。先看看你的 Harness：

* • 文件系统完善吗？
* • 沙箱安全吗？
* • 记忆机制在工作吗？
* • 上下文腐烂了吗？
* • 编排合理吗？
* • Hooks 在兜底吗？
* • System Prompt 清晰吗？

也许答案就在那里。
