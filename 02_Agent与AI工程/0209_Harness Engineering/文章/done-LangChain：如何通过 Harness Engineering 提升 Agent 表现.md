我们之前读过 OpenAI 和 DeepMind 工程师的博客，他们都在强调一个趋势：Agent 的上限，越来越取决于 Harness，而不是模型本身。

本文将精读 LangChain 的博客《Improving Deep Agents with Harness Engineering》，看看他们如何在不改变模型基座的前提下，通过调优 Harness，让 Deep Agents 的跑分从 52.8 提升到了 66.5，从榜单第 30 开外一路进到前五。

他们刻意把 Harness 优化空间压缩到三类：系统提示词、工具体系，以及围绕模型和工具调用的中间件（即挂在模型和工具调用周围的 Hook）。

这三类改动，都属于 Harness。LangChain 对 Harness 的目标定义是：把模型那种“尖刺式、不稳定”的智能，塑造成适合完成特定任务的稳定执行系统。在模型外围打造系统结构，去优化任务表现、Token 效率、延迟等指标。

## 01 LangChain 用 Trace 数据优化 Harness

关键问题是：他们靠什么数据，发现 Agent 在哪里出了问题？

大模型是黑盒。Agent 跑偏、报错、超时之后，留下的是成千上万行运行轨迹（Trace）。人工逐行排查，成本极高，几乎不可能形成快速迭代。

于是 LangChain 把“读日志找问题”这件事，做成了一个可复用的系统组件：Trace Analyzer Skill。要想持续迭代，这套分析流程必须是可重复的，而不是每次靠人工手动排查。

这个流程分为三步：

**第一步，抓取数据。**

从 LangSmith 日志平台，把这次实验里所有 Agent 的运行轨迹原始数据全部拉下来。

**第二步，并行分析。**

系统按批次把这些 Trace 切分开，同时启动多个分析子 Agent 并行跑。每个子 Agent 负责一批，专门做错误分析。找到报错点之后，所有子 Agent 的结论汇总到一个主 Agent 那里，由主 Agent 统一提炼成结构化的发现和改进建议。

**第三步，人工审查。**

主 Agent 给出的是"下一次实验要改什么"的完整建议清单。工程师在这里参与，核对建议是否合理，批准后才进入下一轮改动。很重要的一点是：改动必须是通用的，对特定任务过度拟合的修改，可能让其他题目的表现退步。

这三步加在一起，把原来需要几个小时的日志排查，压缩成一个可以快速循环的工程流程。追踪记录（Traces）是整个改进闭环的核心信号。很多所谓“模型不够聪明”的问题，其实是系统没有在合适的时机，把正确的上下文和约束交给模型。没有 Trace，就无法定位这种缺口，问题只会被归结为“模型能力不足”。

Agent 的提升，不只是模型能力问题，更是系统设计问题。接下来，我们看看 LangChain 具体做了哪些 Harness 设计。

## 02 强制 Agent 自检

第一个真正有效的 Harness 改进点是：强制模型验证。

LangChain 在 trace 记录里发现，最常见的失败模式是这样的：Agent 写完代码，回头读了一遍自己刚写的内容，觉得逻辑上看起来没问题，然后直接停工宣告完成。它根本没有去运行测试，拿实际的执行结果来验证。

大模型的生成内容在语言层面可以自洽，但在实际运行层面完全可能出错。只有跑测试，才能拿到真实的反馈信号；有了真实的报错，模型才能修复并得到正确答案。

针对这个问题，LangChain 上了两道防线。

**第一道是提示词层面的约束。**

他们在系统提示词里，明确规定了一个四步走的解题框架：

1. **规划**

   读任务、扫代码库、理解要求，想好验证方案。
2. **构建**

   写代码，同时要考虑测试——包括正常路径和边缘用例。
3. **验证**

   运行测试，读完整的输出结果。拿结果去对照最初的任务要求，而不是去对照自己写的代码。模型很容易拿结果去跟自己的代码做比对，从而得出"我的代码是对的"这种自我封闭的结论。必须强制要求它回到原始的任务要求上做验证。
4. **修复**

   分析报错，回头改。

**第二道是系统层面的卡口。**

只有提示词往往不够，模型还是可能忘掉或绕过。所以加入了完结前检查中间件（PreCompletionChecklistMiddleware）。

当 Agent 准备退出、宣告完成时，系统会直接拦截它，强制要求它对照任务规格做一次验证，确认跑过测试后才允许退出。模型天然偏向接受自己的第一个答案，必须主动推一把，让它真正面对客观的测试结果，而不是主观判断。

## 03 上下文注入：给 Agent 提供环境和约束信息

第二个有效的 Harness 改进点是：上下文注入。

Agent 对自己所处的环境、约束条件和评估标准了解得越多，它就越能自主地指导自己的工作。很多失败并非能力不足，而是环境信息缺失。围绕这个原则，他们做了三件事：

**第一，注入工作环境地图。**

通常的做法是让 Agent 自己去探查环境（比如调用 ls 查看目录）。但这本身就是一个高错误率环节。路径理解错误、文件层级判断失误，都会让任务跑偏。

LangChain 的做法是由系统替 Agent 完成这一步。他们加入了 LocalContextMiddleware，Agent 一启动，系统自动扫描当前工作目录、父子文件夹结构、工具路径以及 Python 安装信息，把这些环境信息结构化后直接注入上下文。Agent 拿到的不是“去找环境”，而是一份整理好的运行地图，从而减少无效搜索空间。

**第二，明确自动评估标准。**

模型本身并不知道什么叫“可测试性”，它只是生成代码。LangChain 在系统提示词中明确声明：产出会直接被自动化测试评估，任务规格里提到的文件路径必须严格遵守，否则自动评分会失败；同时强调必须覆盖边缘用例。这是把评估标准提前显性化，防止代码只在“理想情况”下通过。

**第三，注入时间预算。**

Terminal Bench 有严格的超时限制，如果 Agent 在某个问题上反复尝试，可能会一直消耗时间直到任务失败。LangChain 在提示词中加入持续的时间预算提醒。当接近截止时间时，系统引导 Agent 收敛问题，进入验证与收尾阶段。模型天然不擅长时间估算，这种约束必须由外部系统注入。

这三件事背后的逻辑是：不要让 Agent 自己去发现和组装运行条件。Harness 的职责是提前整理好环境信息、评估规则和约束边界，再主动递给模型。外围喂的信息越精准，模型就越能把算力集中在真正的解题步骤上。

## 04 防止“死循环”：Harness 加了哪些护栏？

第三个 Harness 改进点是：循环探测中间件，用来防止 Agent 进入死循环。

在大量 Trace 日志里，存在一个典型问题：当 Agent 确定了一个方向后，会高度执着地推进。即使路径已经走不通，它仍会在同一个文件上反复做微小修改，尝试局部修补，而不是整体重审。这种现象被称为 Doom Loops。模型陷在局部最优里出不来，不断消耗算力却无法推进任务。

LangChain 的解决方式是加入 LoopDetectionMiddleware。原理很直接：通过工具调用的钩子，持续记录每个文件的编辑次数。当某个文件的修改次数超过阈值 时，系统向 Agent 上下文中注入提示，要求它重新审视整体方案，而不是继续局部修补。

需要注意的是，这种机制不能保证模型一定会换方向，它是一种启发式干预，在模型陷入循环时提供一次“强制反思”的机会。

原文对此有清醒的说明：这是针对当下模型缺陷设计的工程护栏，未来可能变得多余，但在今天能显著提高成功率。这体现了 Harness 工程的现实主义哲学：Harness 本质上是在为当前模型的短板提供临时支撑，随着模型进化，这些补丁会逐步消失。

## 05 合理分配算力：Harness 提升表现的算力策略

最后，是算力分配以及 Harness 和模型之间的适配关系。

后台多步思考的推理型模型通常有不同的推理强度档位。以 gpt-5.2-codex 为例，有 low、medium、high、xhigh 四档。档位越高，思考越深入，耗时和 token 消耗也越大。

如果全程开 xhigh，很容易在简单步骤上消耗过多时间导致超时失败。LangChain 采用了一个直观有效的策略：“推理夹心”（xhigh-high-xhigh）。

**规划阶段使用 xhigh。**

理解任务、制定整体方案最容易决定成败，值得投入更高算力。

**中间实现阶段降到 high。**

按计划写代码，不需要每一步过度推演，节省大量时间。

**最后的验证与收尾阶段，再拉回 xhigh。**

查错、确认结果需要谨慎。

这种分段式算力分配，把最终分数推到了 66.5%。这表明推理强度不是越高越好，而是要分配得对。Harness 还在管理算力预算。

此外，LangChain 用 Claude Opus 测试同一套 Harness，得分低于 Codex。这带出一条重要工程原则：**为特定模型定制 Harness**。不同模型在提示风格、推理节奏上都有差异，若想把某模型的表现推到极限，需为它单独跑几轮迭代。

## 总结

回顾整个实验，可以提炼出五条核心实践：

第一，追踪即反馈。没有 Traces，就没有改进方向。

第二，代劳上下文工程。把模型需要的环境信息提前整理好。

第三，强制自我验证。让模型真正面对客观的测试结果。

第四，识别并修补坏模式。循环探测是阶段性补丁，但非常有效。

第五，为特定模型定制 Harness。

一句话概括：模型能力决定上限，而 Harness 决定你能逼近上限多少
