## 一个类比

想象你刚招了一个顶级程序员。

履历完美，面试拉满，算法手撕行云流水。入职第一天，你把他领到工位，说："干活吧。"

他打开电脑，发现：

* • VPN 没开通，连不上代码仓库
* • 数据库权限没申请，查不了数据
* • 测试环境没搭好，代码写完没地方跑
* • 内网 Wiki 打不开，不知道项目规范

能力再强，也是白搭。

**今天的 AI Agent，就是这个处境。**

我们给了它最好的模型、最精心的 Prompt、最丰富的工具。然后把它丢进一个什么都没准备好的环境里，期待它能交付结果。

这不是 Agent 的问题，是我们的问题。

## 为什么环境这么重要？

因为 Agent 不是一次性生成答案的聊天机器人。它的工作方式是一个循环：**思考 → 行动 → 观察结果 → 再思考**。

写一段代码，跑一下测试，看哪个用例挂了，改掉，再跑——这跟人类程序员的工作方式一模一样。

但这个循环能转起来，有一个前提：**行动必须能被执行，结果必须能被观察到。**

如果 Agent 写了一段代码却没有地方跑，它就不知道对不对。如果它发了一个数据库查询却连不上数据库，它就拿不到反馈。没有反馈，就没有修正。没有修正，就不可能交付。

**环境不是锦上添花，它是 Agent 自我纠错的基础设施。** 没有一个能执行、能反馈的环境，Agent 的智力再高，也只是在空转。

## 环境，到底难在哪？

2026年第一季度，我们团队一直在做一件事：把 AI Agent 真正用到企业的日常工作里——不是聊天机器人，而是能写代码、做审查、跑数据分析、执行运维操作的"AI 员工"。这个项目叫 **LinkWork**，一个开源的企业级 AI 员工平台，前几天刚完成**全部开源**。

在这个过程中，我们投入精力最大的地方，不是模型选型，不是 Prompt 调优，也不是怎么去构建自己的Agent 运行时，而是**环境**。

当我们说 Agent 的"环境"时，其实在说一个非常复杂的东西。它不是一个 Docker 容器就能解决的事。

我把它拆成五个维度，聊聊我们是怎么一关一关过来的。

## 第一关：Agent 在哪里跑代码？

这是最直觉的问题。Agent 要写代码、跑测试、执行命令——它需要一个"工作台"。

最朴素的想法：给它一个容器，让它在里面折腾。

但问题马上来了：**这个容器和业务环境是什么关系？**

* • 如果 Agent 直接在业务容器里跑，那它一条 `rm -rf` 就能把你的服务删了
* • 如果 Agent 的容器完全隔离，那它又碰不到业务代码和依赖

这是一个经典的"既要又要"：**既要隔离，又要可达。**

我们的做法是**双容器模式**——同一个 Pod 里放两个容器：一个是 Agent 的"大脑"，负责思考和决策；一个是 Runner，是真正执行命令的"手脚"。Agent 告诉 Runner 该干什么，Runner 在受控环境里执行。

这不是简单的两个进程。两个容器的用户身份不同、权限不同、甚至互相看不到对方的存在。Agent 全程以为自己在直接执行命令，其实中间隔着一整层安全检查。

有人会问，为什么不直接用业务的现有容器？

因为业务容器的运行时环境千差万别——Java 的、Python 的、Node 的，版本各异，依赖各异。而业务需要一个**确定性的执行环境**，它自己的工具链、自己的依赖、自己的安全策略，都得是确定的，甚至都是非常老旧的，没办法安装当下的所有 Agent 工具。

所以答案是两个容器各司其职：Agent 容器的环境是标准化的，Runner 容器可以按业务需求定制。两者通过原子调度绑在一起——要么同时启动，要么都不启动，不会出现"脑子在但手不在"的尴尬局面。

## 第二关：Agent 怎么连上外面的世界？

Agent 部署在服务器上（通常是 K8s 集群里），但它要干的活几乎都在"外面"：

* • 调用 LLM 的 API——在公网
* • 查 Email、发企业微信——在办公网
* • 读写 GitLab——在内网
* • 访问数据库——在业务网

四个不同的网络域。一个 Agent，怎么都够得着？

最不安全的做法是全部放通。最安全但没用的做法是全部关掉。

**我们选择的是"默认关闭，统一代理"。**

Agent 的容器默认是不能上网的。它不需要知道外面的世界长什么样。当它需要调用外部工具的时候，所有请求都走一个统一的网关——我们叫它 MCP Gateway。

这个网关是 Agent 通往外部世界的唯一出口。它部署在能够触达各个网络域的位置，帮 Agent 代理所有的外部调用。Agent 甚至不需要持有任何凭证——API Key、Token、SSH 密钥，全部由网关代为注入。

这样做的好处不只是安全。当你的 MCP 工具需要访问办公网的 Jira，或者外网的某个 SaaS 服务，你只需要让网关能到达那里就行了。Agent 侧的网络拓扑完全不用动。

**网关不是一个简单的代理，它更像是 Agent 的"外交部"——统一出口、统一鉴权、统一计量。**

至于网关本身怎么连接外网或办公网？这就回到了传统的网络方案——VPN、专线、Ingress——运维同学比较熟悉的东西。关键是，这个复杂性被封装在网关层，不会泄漏到 Agent 侧。

## 第三关：不敢让 Agent 碰生产，但它又需要真实环境

这可能是最纠结的问题。

Agent 如果直接操作线上数据库、线上 API，出了事怎么办？但如果只给它一个 Mock 环境，它做出来的东西又不可信。

**解法是：给 Agent 一整套 Staging 环境。**

不是那种缩水版的测试环境，而是一个"受控的完整环境"——

* • 数据库里有脱敏的生产数据快照
* • API 可以录制回放真实的请求响应
* • 网络拓扑跟生产一致，只是流量走的是镜像

这本质上是给 Agent 一个**安全的练兵场**。

你可以把它理解为 Agent 的"实习期"。新来的 Agent（或者说新配置的岗位），先在 Staging 环境里跑几轮任务。你观察它的行为——它执行了什么命令？修改了什么文件？调用了什么工具？结果是否符合预期？

验证通过了，再把它"转正"到生产环境。就像我们对待新员工一样。

更深一层想：这不只是 Agent 的需求。任何一个会在生产环境里执行操作的自动化系统，都需要一个这样的验证环节。只不过 Agent 的行为更不可预测，所以这个需求更迫切。

## 第四关：环境漂移——今天能跑，明天就炸

这个问题更隐蔽，但杀伤力更大。

假设你的 Agent 今天跑得好好的。过了一周，同事升级了某个插件的版本，Agent 就开始报错了。你排查了半天，发现是运行时动态拉取的插件版本变了。

这就是**环境漂移**。比如你的 Agent 昨天跑得好好的，今天因为某个基础开源库发了带 Bug 的次要版本更新，所有动态拉取依赖的 Agent 瞬间全部宕机。

传统的 Agent 框架几乎都有这个问题——运行时动态安装插件、动态拉取配置、动态加载工具。这意味着：

* • 不同时间启动的 Agent 实例，环境可能不一样
* • 昨天能跑的任务今天可能失败，而且你不知道什么变了
* • 哪个实例用了哪个版本的工具？没人说得清

**我们在 LinkWork 里的做法是把这个问题彻底消灭在"构建阶段"。**

我们叫它 **Harness Engineering**——一岗位一镜像。

意思是：每一个 AI 岗位，对应一个独立的容器镜像。这个镜像在构建的时候，就把所有东西锁死了——用哪些 Skills、用哪些工具、安全策略是什么、每个组件的版本号是多少。全部写入镜像，运行时只读。

你不能在运行时给 Agent 加装新插件。你不能在运行时改它的工具配置。你要变更任何东西，就必须重新构建镜像。

这听起来很"重"，但它带来的好处是巨大的：

* • **可复现**：同一个镜像，无论什么时候启动，行为都是一样的
* • **可追溯**：出了问题直接看镜像版本，所有配置一目了然
* • **可锁定**：今天跑通的版本组合，可以永远锁住不动

而且构建过程是 Fail-fast 的——任何一个步骤出错（插件拉取失败、配置生成异常、策略文件格式错误），整个构建立刻中断。绝不会让一个"半成品"上线。

**环境即代码。这是 Harness 系统对环境问题最根本的回答。**

## 第五关：多 Agent 并行时的资源争抢

前面四关解决的是"一个 Agent"的环境问题。但在企业里，你可能同时有几十个 Agent 在跑——有做代码审查的，有做数据分析的，有做运维巡检的。

它们都在同一个集群里，争抢 CPU、内存、网络、存储。

一个跑飞了的 Agent 可能把整个集群的资源吃光，让别的 Agent 全部饿死。

**这就需要 Sandbox（沙箱）级别的资源隔离。**

每个 Agent 都在自己的沙箱里运行，有明确的资源配额——最多用多少 CPU、多少内存。超出配额就被限制，不会影响到别人。

但资源隔离只是一半。另一半是**调度**——当资源不够时，谁先谁后？高优先级的任务能不能抢占低优先级的资源？闲置的 Agent 能不能自动释放资源？

这背后是一整套编排系统在工作。LinkWork 用了 Volcano 调度器来解决这个问题——它能保证一组容器要么全部调度成功，要么全部不调度。不会出现"Agent 的脑子启动了但手没启动"的半拉子状态。

**沙箱不只是一个容器。它是文件隔离 + 进程隔离 + 网络隔离 + 资源配额的组合。** 每一层都有它存在的理由。

## 更深一层的思考

回过头看这五个维度，其实可以归纳成一个核心洞察：

> **环境不是 Agent 系统的配套设施，它是 Agent 系统本身。**

模型能力决定了 Agent 的"思考上限"，但环境决定了它的"行动下限"。一个模型能力 90 分但环境 30 分的 Agent，产出一定不如一个模型能力 70 分但环境 80 分的 Agent。

这也是为什么我们把 LinkWork 定位成 **AI 员工平台**而不是 Agent 框架。框架关心的是推理链路、工具调用、Prompt 模板。平台关心的是——这个 AI 员工要在哪里工作？它需要什么权限？它的工作环境是否安全、确定、可观测？

这恰恰是从 "Demo 能跑" 到 "生产可用" 之间，最大的鸿沟。

## 来看看行业里怎么做的

对比几个当前主流的方案：

**Agent 运行时（如当红的 OpenClaw）**——OpenClaw 在本地运行、跨平台消息接入和 Skills 生态上做到了极致，是目前炙手可热的方案。但当企业试图将其推向云端生产环境时，依然会撞上“环境隔离”和“安全管控”的墙。近期大量直接暴露在公网的 Agent 实例被曝出安全漏洞，正是因为缺乏严格的沙箱和网络统一收口。再强大的运行时，依然需要一个企业级的环境底座来支撑。

**个人 AI 助手（Cursor、Claude Code 等）**——环境就是你的笔记本。一个人用，不存在隔离和调度问题。但也正因如此，它无法直接搬到企业场景。

**评测框架（SWE-bench、SWE-agent 等）**——它们有 Sandbox 的概念，但主要是为了评测隔离，而不是生产级治理。容器用完就扔，不需要考虑持久化、权限、审计。

**Agent 框架（LangChain、CrewAI 等）**——它们关注的是推理和编排，环境基本靠用户自己搞定。这在研究和原型阶段没问题，但一旦要上生产，你会发现 80% 的工作都花在环境治理上。

LinkWork 想做的，就是把这 80% 的工作标准化、产品化。让你把精力放在岗位设计和业务逻辑上，而不是天天跟环境搏斗。

## 写在最后

Agent 的环境问题，本质上是一个**工程化问题**。

它不性感，不酷，不会出现在论文里。但它是每一个想把 Agent 用到生产环境的人，都绕不过的坎。

好的雇主不只看员工的能力，还会提供完善的工作环境——明确的职责边界、安全的操作空间、充足的资源支持、清晰的协作流程。

对 AI 员工也是如此。
