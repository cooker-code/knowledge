系列第三篇 · 记忆与上下文管理

前两篇我们拆了 Claude Code 的骨架（Agent 主循环）和铠甲（权限安全系统）。这篇讲最难也最容易被低估的部分：它如何在 200K token 的"脑容量"限制下，依然表现得像一个有记忆、有判断力、反应灵敏的助手？

## 开场：一个反直觉的事实

200K token 的上下文窗口听起来很大。让我算给你看为什么它远远不够。

一个中等复杂度的编程任务，Claude Code 的上下文消耗大概是这样的：

* System Prompt（身份、安全指令、规则）：5-15K tokens
* 工具描述（40+ 工具的 schema 和使用说明）：3-8K tokens
* CLAUDE.md + 记忆文件：2-10K tokens
* Skill 发现列表（1% 预算）：~2K tokens
* 输出预留：20K tokens
* 自动压缩缓冲：13K tokens

**还没开始对话，已经花掉了 45-68K tokens。** 剩下大约 130-155K 给对话历史。一轮工具调用通常消耗 2-5K tokens，做一个中等复杂的重构任务可能需要 30-50 轮。**一个中等任务就能填满上下文窗口。**

Claude Code 的回答是：**像操作系统管理内存一样管理上下文**。有预算分配、有回收策略、有告警系统、有多级压缩。

## 第一道防线：精心设计的 Prompt Cache 边界

Anthropic 的 API 支持 Prompt Cache——重复前缀的 token 可以从缓存读取，成本降低 90%。Claude Code 的消息排布就是围绕这个特性设计的：

💡 关键洞察一

**消息的排列顺序不只影响 LLM 理解，还影响缓存命中率和 API 成本。** 最稳定的内容放最前面，最易变的放最后。即使对话持续了 50 轮，前面的 System Prompt 部分仍然可以从 Prompt Cache 命中。

## 4 级压缩：不是截断，是理解

当对话历史逼近上下文上限时，Claude Code 不是简单地丢掉最早的消息。它有 4 级压缩策略：

LEVEL 1 · SESSION MEMORY 压缩（最智能）

优先尝试的方式。系统用一个分叉代理分析对话内容，提取关键决策、未完成的任务、重要的代码变更，写入会话记忆文件。然后把早期对话替换为一句摘要 + 记忆文件引用。

LEVEL 2 · 标准压缩（LLM 摘要）

如果 Session Memory 压缩失败，回退到标准压缩：用 LLM 重写历史对话为简要摘要。压缩 prompt 会明确告诉 LLM 保留关键决策、代码变更、未完成的任务和用户偏好。

LEVEL 3 · 微压缩（移除冗余）

移除中间工具调用的详细输出，只保留摘要。比如 10 次 Grep 搜索的完整输出可能占 20K tokens，微压缩后只保留"搜索了这些模式，找到了这些文件"。

LEVEL 4 · 反应式压缩（紧急裁剪）

实验性功能。当上下文接近爆满时，更激进地裁剪。

⚠️ 你可能忽略的细节：压缩后清理

压缩后，系统会自动**重新注入**：最近编辑的文件内容（最多 5 个文件，每个最多 5K tokens）、活跃的 Skill 指令、MCP 工具描述的增量更新。

为什么？因为压缩会丢失"LLM 正在操作哪些文件"的上下文。如果不恢复，LLM 在压缩后的第一次操作可能会读一个刚编辑过的文件——浪费一次工具调用。连续 3 次压缩失败后停止重试，避免加剧 API 过载。

## 四类记忆：模仿人类的分层存储

Claude Code 的记忆系统不是一个简单的键值存储，而是一个**四类分层系统**，严格对标人类的记忆结构：

| 类型 | 含义 | 示例 | 类比 |
| --- | --- | --- | --- |
| `user` | 用户角色和偏好 | "用户是 Go 高手但 React 新手" | 你对某人的了解 |
| `feedback` | 工作方式指导 | "不要在末尾总结刚做的事" | 被纠正后的习惯 |
| `project` | 项目动态 | "3月5日代码冻结" | 项目日历 |
| `reference` | 外部系统指向 | "bug 在 Linear 项目跟踪" | 书签 |

💡 关键洞察二

**记忆系统明确排除了"可从当前状态推导的信息"。** 代码风格？读代码就知道了。Git 历史？`git log` 就行。这个排除清单防止记忆系统变成"什么都往里塞"的垃圾桶。人类的长期记忆也是这样：你不会记住今天编辑了哪些文件，但你会记住"这个同事喜欢简洁的代码风格"。

## LLM 驱动的智能召回

面对最多 200 个记忆文件，Claude Code 不是全部加载。它用了一个两阶段检索：

阶段一：只读 FRONTMATTER

扫描所有 .md 文件，但只读取每个文件的前 30 行（YAML frontmatter）。不读正文——节省 I/O 和 token。

阶段二：LLM 选择器

把所有记忆的"目录"发送给 Sonnet，让它选出最多 5 个相关记忆。LLM 选择器的 prompt 里有一条关键指令：如果 LLM 刚用完某个工具，不需要再加载使用文档——但**仍然需要**加载"已知坑"。这个区分非常细腻。

💡 关键洞察三

**记忆不是真相，记忆是"曾经为真的声明"。** Claude Code 强制 LLM 在基于记忆行动之前验证记忆的当前有效性——"The memory says X exists" is not the same as "X exists now." 这避免了 LLM 根据过时的记忆做出错误决定。

## 后台记忆提取：分叉代理 + Prompt Cache 共享

记忆不是 LLM 在对话中主动写的，而是有一个**后台分叉代理**自动提取。

这个分叉代理的消息历史和主对话完全相同（fork），所以**共享父级的 Prompt Cache**——几乎零额外 Token 成本。但它有受限的工具权限：只能读文件、搜索、写记忆目录。

⚠️ 你可能忽略的细节：主代理与分叉代理的互斥

如果用户在对话中主动说"记住这个偏好"，主 Agent 会直接写记忆文件。此时分叉代理检测到"主人已经写过了"，就跳过自动提取。避免重复写入。

## Dream 机制：记忆整合的"睡眠"

Claude Code 有一个类似人类睡眠整合记忆的机制——**Dream**。它在后台自动运行，对记忆进行清洗、合并和修剪。

## 团队记忆同步：一个被低估的工程挑战

Claude Code 还支持**团队记忆**——同一项目的不同用户共享的记忆，通过 Anthropic 服务器同步。

几个工程细节值得关注：

**永久失败检测**：曾经有一个设备在 2.5 天内发出了 **167K 次推送事件**——因为没有永久失败检测，它一直在重试一个永远不会成功的请求。现在 4xx 错误（除了 409 冲突和 429 限流）会被视为永久性错误，立即停止重试。

**Secret 扫描**：推送前自动扫描记忆文件中是否包含 API 密钥等敏感信息。包含 secret 的文件会被跳过，防止"有人不小心在记忆文件里存了密码，然后同步给整个团队"的噩梦。

## 多协议通信架构

Claude Code 需要在四种环境中运行：CLI 本地、IDE 集成、SDK、远程执行。WebSocket Transport 处理了大量边界情况：

睡眠检测

笔记本合盖后打开，WebSocket 连接已经断了。Claude Code 能检测到"这不是网络问题，是设备睡眠了"（两次重连间隔超过 60 秒），重置重连策略而不是放弃。

双层 KEEP-ALIVE

环形缓冲重放

断线重连时，客户端告诉服务器"我收到的最后一条消息 ID 是 X"，服务器从 X 之后继续发送。如果差距超过缓冲区大小（1000 条），服务器会发送完整快照。

💡 关键洞察四

**后台请求不应该重试 529（过载）错误。** 当 API 过载时，后台请求（如自动生成标题、补全建议）立即失败而不重试——用户看不到这些操作，重试只会加重过载。前台请求的重试机会更有价值。这是典型的**优雅降级**：保核心，舍辅助。

## 总结：Agent 的认知基础设施

回顾这三篇文章：**第一篇**讲了 Agent 的骨架——"怎么跑"。**第二篇**讲了 Agent 的铠甲——"怎么安全地跑"。**这一篇**讲了 Agent 的认知基础设施——"怎么聪明地跑"。

如果要用一句话概括 Claude Code 的架构哲学：

通过精心设计的约束来释放 Agent 的能力。

上下文有窗口限制 → 所以有 4 级压缩。网络会断开 → 所以有睡眠检测和环形缓冲。记忆会过时 → 所以有验证机制和漂移警告。每一个约束，都催生了一个精巧的工程解法。**好的 Agent 框架不是消除约束，而是优雅地应对约束。**
