# 公众号稿件·第二稿

**我给 Agent 喂了 300 条推文，它的选题分析从 6 分变成了 9 分**

obsidian-researcher 给我跑了几个月的选题分析，每次输出都在及格线附近——道理没错，角度说得通，但读完没有那种"对，就是这个"的感觉。我一直以为是 Prompt 写得不够好，直到我改了知识来源，才发现问题根本不在提示词，在于它能读到什么。

**改造前后的差距，一次对比就看清楚了**

我让 Agent 分析同一个选题：「一人公司的知识输出效率」。

改造前，五个来源（个人感悟、文章剪藏、X 热文、Claude 对话存档、操作系统笔记），它给出的是这样的分析："当前平台上知识管理类内容竞争激烈，差异化角度可以从工作流具体性切入，结合个人实践数据会更有说服力。"——正确，无聊，我自己也能写出来。

改造后，新增「收藏作者推文」来源，它在同一个选题下给出："某位长期关注生产力系统的作者在过去三个月里有11条推文涉及'输出频率与系统成熟度'的关系，其中有一个判断和你在4月14日感悟里写的方向高度重叠，可以作为外部印证……"——有具体的人、具体的时间窗口、具体的交叉点，这才是有用的分析。

差别不是 Agent 变聪明了，是它的阅读范围变了。

**缺口是被 Agent 逼出来的，不是我主动发现的**

我长期关注的几个作者，他们在推文里持续输出的判断逻辑、产品发布前后的观察碎片，这些内容有时间维度，是某个人完整认知体系的切面。靠 X 热文抓不到——热文是当天的大众反应，不是某个人两年里的持续思考。

这些内容存在我记忆里，从来没进过知识库。Agent 读不到，给不出来，这很正常。

这是知识库设计最容易忽略的盲区：你以为重要的信息，不等于已经被系统化存入。

**Playwright 做的事情：帮我搬运记忆**

解决方案不复杂。我用 Playwright 写了一个脚本，读取浏览器里已登录的 Cookie，去指定作者主页把推文逐条抓下来，存成 `日期_推文ID.md` 格式，保存到 Obsidian 对应目录，图片单独存到 `images/` 子目录并用 Obsidian 引用格式嵌入。转推和已存过的条目直接跳过。

跑的时候用显示浏览器模式，不用无头模式——方便处理偶尔出现的登录验证，也更接近正常访问行为。每条推文之间随机等待 3 到 8 秒，每 20 条额外停 15 到 35 秒。不是技术上跑不快，是没必要冒风险。第一次跑完，一位作者的历史推文存了 300 条进来。

**把新来源接入 Agent 配置**

光有数据文件不够，Agent 还要知道这些文件的存在和用法。

我在 obsidian-researcher 的配置文件里，把知识来源从五个扩展到七个，新增了「收藏作者推文」和「书本重点内容」。「收藏作者推文」这个来源我标注了它的适用场景：具体作者的一手判断、时间跨度内的观点演变、垂直话题的密集输出。这些是它和「文章剪藏」的区别——文章是经过编辑整合的，推文是原始的、即时的，那种半生不熟有时候反而更真实。

Agent 现在在跑素材收集时，会多走这两个目录，结合另外五个来源判断选题方向和差异化角度。

**自动化解决的不是能力，是执行惰性**

这套东西配好之后，我设了三个 Windows 计划任务：上午 11 点自动跑 X 热文抓取；12 点弹窗提醒手动触发收藏作者推文同步（这一步我保留手动，因为我想自己决定今天关注谁）；下午 5 点弹窗提醒开始内容生产，顺带打开 VS Code。

不做这个，我知道自己会怎么做：某天想起来手动跑一次，然后忘掉两周。

**工具升级不是能力升级，是积累通道的升级**

300 条推文进来，不等于 Agent 自动变聪明。真正有价值的，是我选择关注哪些人、为什么关注、他们的哪些判断和我的内容方向有交叉——这个判断是我做的。

工具做的事情，是让我的判断可以被系统积累下来，让 Agent 在每次写作辅助时都能调用这些积累。

知识库升级的本质不是「多存了几百个文件」，是对密度更高的认知来源建立了持续接入的通道。想清楚了这件事，工具只是执行层的问题。
