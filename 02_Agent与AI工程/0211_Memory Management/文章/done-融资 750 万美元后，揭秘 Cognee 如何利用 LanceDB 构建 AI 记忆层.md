# 公司概览

#

Cognee 助力智能体（Agents）通过具备结构化和时间特性的上下文进行检索、推理与记忆。它摄取诸如文件、API 和数据库等数据源，随后进行内容切片与嵌入，增强实体及关系，并构建可查询的知识图谱。

基于 Cognee 构建的应用可以同时查询图谱和向量，从而以更清晰的出处溯源回答多步问题。该平台专为在知识密集型领域构建自主智能体、智能助手以及搜索功能的团队而设计。

# 挑战

#

智能体团队常因无状态的上下文和自行搭建的 RAG 技术栈而面临困境。他们不得不兼顾分布在各处的图谱、向量存储以及其间权宜的临时规则。

这增加了可靠性风险并降低了迭代速度。Cognee 需要一种向量存储，既能匹配其隔离模型，支持按工作空间开发，又能保持日常工作的简洁性。

## 隔离问题

##

Cognee 的隔离模型通过分离存储空间的方式，确保每位开发者、用户或测试实例都能获得其完全独立的工作空间。

Cognee 并非在不同上下文之间共享单一的向量数据库或图谱，而是为每个工作空间启动一套专用的后端——其中最核心的是 **LanceDB****存储**。

这种方法解决了 AI 开发中的一个关键问题：当多名开发者在同一个项目上工作，或者并行运行测试时，共享状态可能会导致不可预测的行为、数据损坏以及调试噩梦。传统的向量数据库通常需要复杂的编排才能实现这种程度的隔离。

# LanceDB 作为解决方案

#

Cognee 选择 LanceDB 是因为它契合了工程师构建和测试记忆系统的实际方式。由于 LanceDB 是**基于文件**的，Cognee 可以为每个测试、每个用户以及每个工作空间启动独立的存储。

## 为什么基于文件的存储至关重要

##

开发者可以并行运行实验而无需共享状态，并避免了为每个沙盒管理独立向量数据库服务的开销。这种方法缩短了拉取请求周期，提高了演示的可靠性，并使多租户开发更具可预测性。

与需要运行服务器的传统向量数据库不同，**LanceDB****基于文件的方法**意味着每个工作空间在磁盘上都有自己的目录。这消除了在开发和测试期间管理数据库连接、端口和服务依赖项的复杂性。

> 数据与嵌入的统一存储：
>
> LanceDB 通过 Lance 列式格式将原始数据与嵌入存储在一起。由于无需额外的粘合代码来保持有效负载与向量的一致性，数据管道变得更加简洁。

# 各组件如何协同工作

#

Cognee 通过将知识图谱与高性能向量搜索相结合，为 AI 智能体提供持久的记忆层。

它摄取文件、API 和数据库，随后应用 **提取–认知–加载（Extract–Cognify–Load）模型** 进行内容切片、增强实体与关系、添加时间上下文，并写入用于检索的图谱结构和嵌入。

**LanceDB** 是该技术栈中默认的向量数据库，它使嵌入与有效负载保持紧邻，并消除了额外的编排工作。

由于 LanceDB 是**基于文件**的，Cognee 可以为每个用户、每个工作空间和每个测试提供一个干净的存储库，从而使团队在无需共享状态或沉重基础设施的情况下实现快速迭代。

## 从开发到生产

##

这一架构在不改变思维模型的情况下，实现了从笔记本电脑到生产环境的无缝扩展。开发者可以先在本地使用 **Cognee** 的 UI 启动，针对 **LanceDB** 构建并查询记忆，并通过隔离的沙盒环境验证其行为。

Cognee 的 **Memify 管道**在部署后通过清理过期节点、加强关联以及重新权衡重要事实来保持记忆的实时性。这在无需全量重建的情况下提升了检索质量。

## 记忆处理管道

##

Cognee 将非结构化和结构化输入转换为不断演进的知识图谱，并将其与用于检索的嵌入相结合。其结果是形成了一个随时间推移不断完善，并支持图谱感知推理的记忆系统。

该管道通过三个不同的阶段处理数据：

**1. 提取**：从各种数据源（文件、API、数据库）拉取数据。

**2. 认知**：将原始数据转换为带有嵌入的结构化知识。

**3. 加载**：存储图谱和向量数据，以实现高效检索。

**图 2：Cognee 记忆管道**

在此，“提取、认知和加载”阶段将原始输入转化为**知识****图谱****与****嵌入**，随后通过 **Kuzu**（图谱）和 **LanceDB**（向量）提供双重搜索服务。

用户可以从中获得具备出处溯源和时间上下文的高质量检索，由于有效负载与向量始终保持对齐，数据管道也变得更加简洁，且随着记忆的演进，系统更新也更加迅速。

# 为什么选择 LanceDB

#

LanceDB 与 Cognee 的**隔离模型**高度契合。每个测试和用户工作空间都可以拥有一个纯净的数据库，且其创建与删除都极其简单。

这减少了持续集成（CI）中的摩擦，确保了并行运行的安全，并加快了新成员的上手速度。

当团队需要更多控制权时，LanceDB 提供了现代化的索引选项，例如 **IVF-PQ** 和 **HNSW 风格的图索引**，使工程师能够针对不同的工作负载优化召回率、延迟及存储占用。

除了隔离性之外，**LanceDB** 还提供了几项技术优势，使其成为 Cognee 用例的理想选择：

**2. 零拷贝演进**：模式变更不需要复制数据。

**4. 高性能**：针对交互式查询和批处理进行了优化。

**5. 部署灵活性**：支持本地、容器或云环境运行。

## Memify 管道实战

##

**Memify** 管道代表了 AI 记忆管理领域的重大进步，它实现了持续的自我完善，且无需耗费高昂的全量重建成本。

以下是一个典型的 **Memify 管道** 用于知识图谱后处理的示例：

```
import asyncioimport cogneefrom cognee import SearchType
async def main():    # 1) Add two short chats and build a graph    await cognee.add([        "We follow PEP8. Add type hints and docstrings.",        "Releases should not be on Friday. Susan must review PRs.",    ], dataset_name="rules_demo")    await cognee.cognify(datasets=["rules_demo"])  # builds graph        # 2) Enrich the graph (uses default memify tasks)    await cognee.memify(dataset="rules_demo")        # 3) Query the new coding rules    rules = await cognee.search(        query_type=SearchType.CODING_RULES,        query_text="List coding rules",        node_name=["coding_agent_rules"],    )    print("Rules:", rules)
asyncio.run(main())
```
