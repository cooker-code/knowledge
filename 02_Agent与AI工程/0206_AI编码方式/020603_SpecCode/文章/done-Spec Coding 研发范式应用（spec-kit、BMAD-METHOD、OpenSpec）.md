在今年上半年 Vibe Coding 的概念被 Andrej Karpathy 提出后在网上非常火爆，在这一年我们看到了很多有编程基础或者没有编程基础的人，用 AI Vibe coding 的方式做出了有趣的东西。

这期间我使用 Cursor、Claude Code 等工具做出了数十款 iOS App 以及网页工具，在这个过程中我发现这些项目会随着 AI 持续的迭代（包括一些**从0到1的项目**以及公司内**存量系统的业务需求**）逐渐失去**可维护性**，怎么**用 AI 持续迭代下去**是一个很大的问题

前段时间 Amazon Kiro IDE 中推出了 Spec 能力，可以将模糊需求的 prompt 转化为**需求文档（requirements.md）**、**系统设计文档（design.md）**和**任务清单（tasks.md）**。

这是一种 **规范驱动开发**（Spec-Driven Development，SDD）的开发范式，在 Kiro Spec 中的本质其实是**上下文构建**的过程，它能够将很多 Vibe Coder 模糊不清的一句话需求进行很好的分析拆解，但一句模糊不清的话或者一个看似描述清晰的 PRD 如何保证 AI 能正确理解，其中缺失的业务含义、产品黑话和看似描述恰当的业务逻辑，都是影响最终交付产物的因素。

这是一种新的 AI Coding 范式，不同于 Vibe Coding 的“放荡不羁”，拿起锤子🔨就是干，Spec Coding 更加强调 **specification**（参考 Specification\_(technical\_standard)  的通识标准，spec 可以有 **requirement specification（需求规范）、functional specification（功能规范）、design or product specification（产品或设计规范）**等内容）

**Vibe Coding 与 Spec Coding 对比**

| 维度 \ 类型 | Vibe Coding （直觉驱动开发） | Spec Coding （规范驱动开发） |
| --- | --- | --- |
| 开发模式 | 跟着感觉走，探索式发现，“干中学” （偏直觉判断）快速迭代、创意驱动，先做出来 | 工程化管理，“计划后执行” （逻辑分析，系统性规划）复杂系统需要明确架构，避免技术债务累积 |
| 交付效率 | 短期：极高的初始开发速度，快速启动长期：可能因重构和返工导致总体时间增加“永远在调试”的体验感 | 短期：前期规划阶段耗时较长，延迟启动长期：减少返工，总体项目时间可能更短过度设计可能导致不必要的复杂性 |
| 可维护性 | 隐性知识 → “Vibe PM”的个人理解 → 难以文档化 | 显性知识 → 文档规范 → 易于标准化 |

### Spec Coding 实践方式

> 目前有很多开发套件or工具支持进行 Spec Coding，下面主要介绍下它们都做了什么。

#### 一、Spec Kit

**链接**: https://github.com/github/spec-kit

**介绍**: 由 GitHub 官方开源，定位是辅助进行 Spec Coding 的工具包。（不搞 AI 读心术😼）

> cli 使用方法在 spec-kit 中介绍的很详细，怎么使用不做详细介绍。

**规范是真理的来源，代码是其表达方式。** 仔细回想没有 AI 前的开发流程，最核心的是写好设计文档，在有优秀设计文档指导的前提下，编码工作就像是在操作打字机打字，不需要动太多脑子，这里的设计文档就是 Specification，现在的 Spec Coding 是对整个流程有了更高的要求，如何编写和维护高质量的规范。

**SDD 的目标是：**

1. **消除规范与实现之间的鸿沟**：通过使规范可执行，直接生成代码，从而确保两者始终保持同步。
2. **提升开发效率和质量**：自动化机械性工作，让开发者专注于创造性、批判性思维和实验。
3. **应对软件复杂性和快速变化**：提供系统化的方法来管理复杂系统，并支持快速、系统化的迭代和修改。

而 spec-kit 的几个核心命令分别对应着：

| 命令 | 作用 |
| --- | --- |
| /specify | **想法到PRD的转化。** 通过与 AI 的迭代对话，将模糊的想法转化为全面、结构化的 PRD。AI 提问、识别边缘情况、定义验收标准。重点强调**“用户需要什么”和“为什么需要”**，避免过早关注**“如何实现”**。利用模板强制标记不明确之处 [NEEDS CLARIFICATION]。 |
| /plan | **PRD到实现计划的转化。** 从已共识的 PRD 生成详细的实现计划，将业务需求映射到技术方案。这一步可以得到**技术架构、数据模型、API 接口规范、测试用例等** (plan.md, data-model.md, contracts.md 等) |
| /tasks | **实现计划到可执行任务的转化。** 根据实现计划和相关设计文档，生成一个可执行的任务列表。 |

在生成代码后，遵循“测试优先”原则，先生成单元测试、集成测试、端到端测试，确保 AI 生成的代码能够通过。

然后是人工的持续检查反馈，将相关信息补充到规范中驱动其演进，达到 SDLC 循环。

> SDD 的工作流通过 /specify、/plan 和 /tasks 等命令实现高效自动化。从一个模糊的想法开始，/specify 命令通过与 AI 的迭代对话，将其转化为结构化的 PRD，并自动管理版本控制。接着，/plan 命令依据 PRD 生成详细的实现计划，将业务需求转化为技术架构、数据模型和 API 契约，并遵循预设的相关原则。最后，/tasks 命令根据计划生成可执行的任务列表，为代码生成做好准备。

**工作流程**

**SDD 核心思想**

* **规范即通用语言**：规范是主要产物，代码是其表达，维护软件即演进规范。
* **可执行规范**：规范必须足够精确、完整、无歧义，以便生成工作正确进行。
* **持续精化**：AI 持续分析规范的歧义、矛盾和漏洞，进行持续改进。
* **研究驱动的上下文**：研究代理在规范过程中收集技术选项、性能影响、组织约束等关键信息。
* **双向反馈**：生产现实反馈到规范演进中。
* **分支探索**：从同一规范生成多种实现方法，以探索不同的优化目标（性能、可维护性、用户体验、成本）。

spec-kit 中为用户提供了一些比较精妙的模版，其内是一些“元提示”，它们通过结构化输入来约束 LLM 的行为，从而产出更高质量的规范。

其主要约束如下：

* **防止过早的技术细节**：强制 LLM 关注“是什么”而非“如何实现”。
* **强制标记不确定性**：要求 LLM 明确指出模糊不清之处，避免臆断。
* **通过清单进行结构化思考**：内置的清单作为 LLM 的“质量保证框架”，促使其自我审查。
* **通过门禁强制合规**：将原则转化为具体的检查点，确保架构一致性。
* **分层细节管理**：指导 LLM 保持文档的高层次可读性，并将详细实现细节移至专门文件。
* **测试优先思维**：强制在实现代码之前定义契约和测试。

> **为什么 TDD测试必须先失败？**
>
> 1、**验证测试本身是正确的**，如果测试一开始就通过了，说明 ① 测试可能写错了（测试逻辑有问题） ② 测试可能测试了错误的东西 ③ 测试可能根本没有测试到预期的功能。
>
> 2、**虚假的安全感**，如果测试一开始就通过，可能会让我们误以为功能已经正确实现了，产生代码已经工作的错觉。
>
> 3、**确保测试覆盖了正确的代码路径**，保证测试代码确实在检查你想要的功能，能够检测到功能缺失，而且当我们实现功能后，测试通过是因为我们的代码，而不是巧合。

#### 二、BMAD-METHOD

**链接**: https://github.com/bmad-code-org/BMAD-METHOD

**介绍**: Breakthrough Method for Agile Ai Driven Development，敏捷AI驱动开发的突破方法。与 spec-kit 类似，解决复杂业务场景的需求开发。

> BMAD-METHOD 源自敏捷开发，它专注于协调多个代理，使其能够以灵活的、基于角色的方式协同工作。BMAD-METHOD 强调实时协调和团队合作，并根据在动态环境中部署 AI 工具的实践者的反馈不断发展。

Spec Kit 的设计理念主要以规范为导向，注重项目规范的一致性和单一事实来源。这种方法强调了清晰、定义明确的规范的重要性，可以保障 AI 生成需求代码与开发者理解间尽可能的一致性。

相比之下，BMAD-METHOD 则更注重**灵活性和适应性**。该方法允许团队动态地定义代理之间的角色和职责，从而提高适应变化的能力并实现更有效的沟通。它体现了敏捷迭代开发的原则，促进了持续的反馈和调整。

**工作流程**

> 每个节点代表与不同专业 Agent 的不同 Claude Code 会话。需要在每个步骤之间清晰地总结出上下文（过程产物），向前推进项目。

**BMAD-METHOD 的一些关键能力**

1、文档分片

文档分片可以将复杂的项目文档分解为原子的、AI 可理解的部分。我们不再是给 AI 提供一份长篇大论的 PRD，而是将其细分为执行摘要、用户角色、单个用户需求、功能规范、技术需求等。

每个分片都成为一个上下文，防止 AI 幻觉并保持整个项目的一致性。

2、多 Agent 团队协同

包括市场研究与业务分析、项目管理、产品所有者验证、用户体验设计、系统设计等等，不同角色间职能、关注点划分明确。

3、上下文工程

Vibe Coding 与 Spec Coding 的关键差异，规范产物的保留（上下文），在我们使用 BMAD 时每个 Agent 都会创建相关的产物，作为下一个 Agent 的输出，保证价值传递。

> 同 spec-kit 这里不介绍怎么用，具体可见仓库说明。以及相关用户手册: https://github.com/bmad-code-org/BMAD-METHOD/blob/main/docs/user-guide.md
>
> **推荐官方的视频课程**: https://www.youtube.com/watch?v=H4k3AdU-ckw

#### 三、OpenSpec

**链接**: https://github.com/Fission-AI/OpenSpec

**介绍**: OpenSpec是一个\*\*规范驱动开发(Spec-driven Development)\*\*工具，专门为 AI Coding 助手。核心价值是解决 AI 开发的不确定性问题，建立人机协作标准，增强项目的可见性与可持续迭代性。整体与 spec-kit 作用相似。

**工作流程**

**第一步：草拟变更提案**：AI 根据需求创建完整的变更文件夹结构，包含：proposal.md（提案）、tasks.md（任务）、specs（规范增量）

**第二步：审查与对齐**：人和 AI 共同审查提案和规范，反复修改直到双方达成一致。

**第三步：实现任务**：AI 根据已确认的规范实施具体任务，按照tasks.md中的检查清单逐项完成。

**第四步：归档与更新**：完成后将变更归档到archive/目录，将批准的更新合并回源规范文件。

对于一次人工提交的新请求，OpenSpec 会基于需求的类型进行不同程度的处理，如果是修改BUG等会基于规范直接修改；如果是更新功能、架构调整或未知因素等会重新创建提案走上面流程，分类如下：

#### 四、建议与思考

本次一共分析了三个工具（spec-kit、BMAD-METHOD、OpenSpec），三者的本质都是对 Context Engine 的应用，从关注点上来说 spec-kit 与 OpenSpec 的作用类似，都是在规范产物上做功（细节怎么做）；而 BMAD-METHOD 主要是角色划分以及流程协作中上下文如何使用。因此，是可以搭配使用的，推荐 BMAD-METHOD + spec-kit，**BMAD** 以 Agent 能力负责需求、设计等，**Spec Kit** 对 BMAD 内容再进行精炼输出规范。（理论上可行，但二者框架暂时没有尝试进行搭配）。

Context Engine 在 AI 相关工作中的重要程度愈发明显，Spec Coding 的本质也是上下文工程，任何维护好有价值的上下文对于 AI 来说至关重要，软件开发的未来不是编写更多代码，而是指导 AI 编写更好的代码。
