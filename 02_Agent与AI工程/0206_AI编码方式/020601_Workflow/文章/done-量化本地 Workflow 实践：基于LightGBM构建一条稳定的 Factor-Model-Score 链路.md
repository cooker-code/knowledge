## 为什么会想到做这个项目

最直接的原因，其实是我发现自己花了很多时间在重复搬运流程。

比如一轮研究里，真正有效的信息通常是这些：

* 这次用了什么 universe
* 这次算了哪些 factor
* 标签怎么定义
* 训练窗口怎么切
* 模型参数是什么
* 输出的 score 文件能不能直接接回测

但在没有把流程整理清楚之前，实际做起来往往不是这样。更多时候是在不同脚本之间来回切，文件格式靠约定，时间对齐靠记忆，结果能跑出来，但链路是松的。

而量化研究偏偏又是一个特别依赖重复实验的事情。你不会只跑一次 `LightGBM`，也不会只试一个 factor 组合。只要开始做第二轮、第三轮实验，前面那些临时方案就会变成明显负担。

我后来就慢慢意识到，这里面真正值得单独做成项目的，不是某个“厉害模型”，而是把研究链路里的几个关键节点标准化。

## 这个项目现在主要做了什么

目前我把项目里比较核心的部分收敛成了下面这条链路：

```
SQLite / Parquet market data
    -> factor generation
    -> training dataset
    -> LightGBM walk-forward training
    -> score generation
    -> score backtest / simulation
```

对应到仓库里，当前比较关键的能力主要有这几块。

### 1. 本地数据同步和标准化存储

项目里现在把数据底座拆成了两层：

* `storage/source/` 下的 SQLite 源数据
* `storage/parquet/` 下的标准化 Parquet 快照

这么做的原因很实际。

SQLite 适合做同步落地和增量维护，Parquet 更适合后续研究和分析。量化里很多数据天然就是列式读取的场景，factor、score、面板特征这些东西，用 `Parquet` 会比散落的 CSV 顺很多。不管是压缩率、读取效率还是后续按列处理，都更适合研究环境。

所以项目第一步不是先谈模型，而是先把数据同步和存储层整理清楚。后面的 factor build、训练和回测，都是建立在这个底座上的。

### 2. factor 文件生成

这一层我现在越来越觉得是研究链路里最容易被低估的部分。

很多时候，大家会把 factor 看成训练前临时产出的一张表，用完就算了。但只要你开始认真做迭代，就会发现 factor 其实应该是一个稳定的中间层。字段怎么命名、时间怎么对齐、索引怎么组织、缺失值怎么处理、截面的去极值与标准化怎么做，以及更重要的，股票池怎么约束，比如对 ST、退市股、停牌和一字涨跌停等不可交易状态的剔除，这些都直接影响后面的训练和回测真实度。

所以我在项目里单独做了 factor build 流程，先基于指定 universe 构建 factor snapshot，再把输出组织成后面可以直接消费的 factor panel。

这里的重点不是“因子多不多”，而是 factor 文件能不能稳定复用。

如果这一层不稳定，后面训练和回测都会不断补丁；如果这一层稳定，很多实验其实就只是换配置、换窗口、换特征组合的问题。

### 3. 基于 `LightGBM` 的模型训练

模型这层我当前主要用的是 `LightGBM`。

原因也比较直接。对这类表格型量化特征来说，`LightGBM` 足够实用，训练速度快，做 baseline 很合适，也适合做横截面回归类任务。很多研究的早期阶段，最重要的不是立刻上复杂模型，而是先把 factor、label、切分方式和验证链路跑顺。

项目现在已经支持基于 factor panel 和未来收益标签做离线训练，并按 walk-forward 的方式逐月滚动训练和预测。当前主线更接近这样一套逻辑：

* 从因子面板读取训练样本
* 按时间窗口切训练集和测试集
* 严格遵循 A 股 T+1 的可见性边界，用 `LightGBM` 回归未来超额收益标签，例如基于 T 日的因子，去拟合 T+1 日开盘到 T+N 日开盘的收益，确保底层逻辑不引入未来函数
* 产出预测分数和训练指标

这一步看上去没什么花哨的地方，但我觉得量化研究里真正重要的也不是“训练代码有多炫”，而是输入、输出和切分逻辑是不是清楚。

因为过几周之后你真正要回答的问题往往不是“模型跑没跑”，而是：

* 这次用的标签是什么
* 训练窗口是多久
* 哪些 factor 进了模型
* 输出的分数文件能不能直接复用

如果这些东西回答不清，实验的复盘价值就会很有限。

### 4. score 文件生成

这是我后来特别在意的一层。

很多研究代码做到模型输出就结束了，但实际研究里，模型文件本身并不能直接进入回测。真正往后接组合构建、分层分析、选股逻辑和策略验证的，通常是标准化之后的 score 文件。

所以项目里我把 score generation 单独收出来了。训练完成之后，会把模型输出落成结构稳定的 score 文件，而不是停在一个一次性的中间结果上。

这样做有几个很直接的好处：

* 训练和回测之间不会断层
* 不同模型和不同参数的输出更容易横向比较
* 回测入口会更稳定，不需要每次再写一层临时转换脚本
* 后续做 latest inference 和 simulation 也更自然

对我自己来说，这一步其实是把“模型实验”真正变成“策略研究”的关键。

### 5. walk-forward、latest inference 和 score backtest

当前项目里，研究流程不只是把模型训练完，还会继续往下走。

现在已经有一条最小可复现链路：

1. 从 Parquet 日线数据构建 factor 面板
2. 使用横截面未来超额收益标签训练 `LightGBM`
3. 按月做 walk-forward 滚动训练和预测
4. 对预测分数做标准的 Alpha 检验，包括 IC / RankIC 衰减、分层收益表现等
5. 用预测分数驱动回测引擎构建组合，兼顾换手率和基础交易摩擦，并评估结果

除了 walk-forward 之外，项目里也有 latest 相关的流程，用来维护当前最新可用日期的分数工件，并进一步接到盘前准备和模拟执行上。

这也是我当时做这个项目时比较在意的一点：研究不能只停在“回顾性训练”这一层，还得能把最新分数和后续动作组织起来，不然整套链路还是会在最后一公里断掉。

### 6. simulation 和本地 Web 控制台

除了 CLI 之外，我还补了一套本地 Web 控制台。

它不是为了做成一个完整平台，而是为了把几个关键状态集中起来看，包括：

* 数据状态

* 回测结果

* latest 工件
* 模拟账户计划
* simulation 执行历史

对我自己这种个人研究场景来说，这种本地可视化其实刚好够用。它不需要很重，但能明显降低“状态分散在不同目录和脚本里”的管理成本。

## 这个项目是怎么一步步踩坑踩出来的

如果只看最后的结构，好像这条链路很自然。但实际上中间踩过的坑不少，而且很多坑都不是单点 bug，而是设计上迟早会遇到的问题。

### 坑 1：总以为“先跑起来再说”

这是最常见的一类坑。

一开始为了快，很多环节都想先用临时方案顶一下，比如 factor 先随手落一份 CSV，训练脚本先直接读，模型输出先写成另一份表，回测的时候再临时拼回去。

短期看这很省事，长期看几乎一定会失控。

因为这些临时方案会向后传递：

* factor 不稳定，训练就要补清洗逻辑
* 训练输出不稳定，score generation 就会很别扭
* score 结构不稳定，回测入口又得继续写转换层

最后你会发现每一段单独都能跑，但整个 workflow 很松，实验稍微一多，维护成本就会超过训练本身。

### 坑 2：低估了“文件流”和“结果复盘”的难度

量化研究不是一次性任务，而是持续比较。

你会不断换 factor、换 label、换训练窗口、换股票池、换调仓参数。只要输入输出没有标准化，最后就一定会掉进“知道自己跑过很多东西，但说不清哪份结果对应哪次配置”的状态。

这也是我后来越来越强调 `Parquet`、标准输出目录和中间工件管理的原因。不是因为这些东西高级，而是因为研究一旦进入多轮迭代，它们就是最低限度的秩序。

### 坑 3：训练和策略验证之间天然有一层断点

很多时候，训练部分和回测部分是分开写的。两边都没问题，但接口不稳定。

比如训练那边输出的是某种预测表，回测那边需要的是另一种格式；或者训练按样本切得很顺，到了策略验证才发现 `trade date`、`execution date`、可见性边界没对齐。

score 文件这层，本质上就是为了解决这个断点。

我后来越来越明确地觉得，训练结果不能直接等于策略输入，中间一定要有一个规范的 score 层，不然后面很容易越接越乱。

## 量化里有些问题，工具做得再好也解决不了

这个项目可以把研究流程整理清楚，但量化里一些根本问题并不会因为 workflow 更顺就消失。

这一点我觉得最好一开始就讲清楚。

### 1. 滞后性是不可避免的

只要你在用历史数据做研究，滞后性就是绕不过去的。

数据可得时间、信号生成时间、调仓时间、实际执行时间，这几件事如果没有严格拆开，结果就很容易好看得不真实。很多所谓“有效信号”，最后一拆时间边界，可能只是把未来信息提前用了。

所以这个项目虽然在工程上尽量把 factor、score、backtest 之间的时间边界理清，但这不代表滞后性问题被自动解决了。这个问题仍然需要研究者自己持续盯着。

尤其是在：

* 标签构造
* latest inference
* 分数生效日
* 调仓执行日

这些地方，时间定义只要有一点松动，后面的结果就会变形。

### 2. 拟合几乎是一定会发生的

只要开始反复试 factor、调参数、换标签窗口、换股票池，拟合就很难完全避免。

这和你用的是不是 `LightGBM` 关系没那么大，更多是研究流程本身决定的。工具层做得越顺，实验成本越低，某种意义上你就越容易不自觉地过度搜索。

所以我现在对这个项目的定位也很清楚：

* 它可以让实验更容易复现
* 可以让输入输出更清晰
* 可以让训练、打分、回测更顺畅地接起来

但它不能替你解决泛化能力的问题。

像样本外验证、时间切分、结果解释、控制实验次数、避免在结果上反复追参数，这些事本质上还是研究纪律，不是工程封装能替代的。

### 3. 回测结果和真实执行之间永远有落差

即便已经考虑了手续费、印花税、滑点、成交参与率上限和挂单保留天数，回测和真实执行之间还是会有差异。

原因也很简单：

* 流动性约束不可能被完全简化
* 冲击成本通常很难建模得足够真实
* 极端行情下的成交行为会偏离日线近似
* 组合容量一放大，很多结果会明显变化

所以我现在更倾向于把 backtest 当成筛选和诊断工具，而不是结果终点。项目里已经往 simulation 和盘前准备延伸，也是因为我希望它不只是停在“纸面回测”这一层。

## 目前已经实现了什么

如果按“已经能稳定使用的部分”来总结，目前大概可以概括成这些：

* 本地 A 股日线数据同步到 SQLite
* 从 SQLite 导入并标准化到 Parquet 分析层
* 基于 universe 构建 factor snapshot 和 factor panel
* 基于 `LightGBM` 做横截面未来收益标签训练
* 按月 walk-forward 滚动训练和打分
* 生成标准化 score 文件
* 基于 score 做分层分析和分数驱动回测
* 维护 latest 工件并衔接盘前准备
* 提供 simulation 和本地 Web 控制台查看关键状态

这些能力还远远谈不上“完整平台”，但对我自己的研究节奏来说，已经把最常反复折腾的那条主线搭起来了。

## 后面还想继续补什么

这个项目后面我还会继续做，但方向基本还是围绕“把研究链路补扎实”，而不是盲目加功能。

当前比较明确的 todo 包括：

* 扩大 walk-forward 和回测验证区间
* 继续补 simulation 和真实执行约束之间的衔接
* 做更清晰的实验归档和结果追踪
* 支持更系统的 factor / label / universe 组合管理
* 增加更细的收益来源、风险暴露和阶段拆解
* 继续评估流动性、容量和冲击成本的更合理接入方式
* 提升模型层稳定性，而不是只追求某次回测收益更高

这一部分我自己其实也比较克制。因为量化项目很容易越做越大，最后变成“功能很多，但主链路反而不清楚”。我更希望它保持一个比较直接的状态：核心流程可见，数据流清楚，边界明确，方便继续改。

## 最后

我做这个项目，并不是想证明某套模型有多强，而是想先把量化研究里最容易拖慢效率、最容易越写越乱的那几段流程整理清楚。

如果只用一句话概括它现在的价值，大概就是：

> ❝
>
> 它把 factor、训练、score、backtest、simulation 这几段原本很容易分散的事情，收敛到了一条更可维护的本地工作流里。
>
> ❞

它当然解决不了量化里最核心的问题。滞后性不会消失，拟合不会消失，真实交易和回测之间的偏差也不会消失。

当然，现在的状态还远不是终点，后续我也会在这个底座上继续做日常的迭代和优化，毕竟学习就是一个不断除错和精进的过程。
