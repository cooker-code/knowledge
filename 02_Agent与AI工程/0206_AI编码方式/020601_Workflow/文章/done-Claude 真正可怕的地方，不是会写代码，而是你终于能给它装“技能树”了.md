很多人每个月花 20 美元订阅 Claude，结果真正用到的，往往只有最表面那一层：补一段代码、改一处 bug 、润一下文案、顺手问个思路。

这不能说错，但确实太浅了。

因为当你还把 Claude 当成一个更聪明的问答工具时，另一批人已经开始把它当成一整套可复用的工作流系统来用。两边用的是同一个模型，最后拉开的却不是一点点体验差，而是生产方式上的代差。

真正的分水岭，不在模型会不会回答，而在你有没有给它装上一层“技能树”。

## 多数人为什么把 Claude 用浅了

问题不在模型不够强，而在使用方式还停留在“临时提问”这一级。

很多人和 Claude 的关系，其实像在用一个昂贵的自动补全器：想到什么问什么，卡在哪儿补哪儿，解决的是眼前这一步，但并没有把背后的流程沉淀下来。于是每开一个新会话，都要重新解释上下文，重新说明规则，重新讲清楚你到底想让它怎么做。

这样当然也能得到答案，但很难得到稳定、可复用、可协作的结果。

开发工作最消耗人的，往往也不是某一行代码本身，而是在不同阶段之间来回切换：需求还没说透，就开始写实现；实现刚写一半，又发现测试没补；测试补完之后，review 标准又不统一；最后文档没人整理，知识再次散掉。

如果 Claude 只是一个聊天框，它最多帮你在某个点上加速一下；但如果它能理解并执行整条链路里的不同工作方法，事情就完全不一样了。

## Skill 和 Prompt，根本不是一回事

很多人第一次听到 Skill，会以为它只是“更高级的 Prompt”。

其实不是。

Prompt 更像一次性口头交代。你临时把需求说给模型听，希望它在当前会话里按你的意思完成任务。只要会话换了、上下文丢了、表达方式变了，结果就很容易漂移。

Skill 则是把某类工作的做法，正式固化下来。它的本质，是一个带 `SKILL.md` 的文件夹，里面写清楚这类工作的步骤、约束、示例，以及必要时的辅助脚本或模板。

这个差别非常关键。

你不是每次都重新告诉 Claude“怎么做代码评审”，而是把代码评审这件事本身定义成一个可复用的执行单元。你也不是每次都临时解释“我们怎么写 PRD 、怎么拆计划、怎么做调试”，而是把这些方法沉淀成一套长期可调用的技能层。

从这个角度看，Prompt 解决的是“这一次怎么说清楚”，Skill 解决的则是“以后都按什么方法做”。

前者是对话技巧，后者是流程资产。

## 项目介绍

为什么说这件事重要？因为它直接改变了 Claude 在工作流里的角色。

当 Skill 机制建立起来之后，Claude 不再只是一个被动响应问题的模型，而更像一支可以切换角色的团队。它可以在不同环节里扮演 architect 、 reviewer 、 debugger 、 docs writer，各自按照不同的工作方法运转。

而且这不是抽象概念。现在已经有人把大量 skill 按 use case 做了系统整理。你会发现，这套生态覆盖的根本不是某个零散小技巧，而是完整的软件生产链路。

大致看，可以分成几层：

第一层是 Meta skills，比如 `skill creator`、`write a skill`、`find skills`。这一层做的不是直接交付业务结果，而是帮你管理“技能系统”本身：怎么创建、怎么迭代、怎么查找已有方法。

第二层是规划和设计层，比如 `grill me`、`write a PRD`、`PRD to plan`、`PRD to issues`、`design an interface`、`request refactor plan`。这类 skill 解决的是“先别急着写，先把问题定义对”。

第三层是代码开发层，比如 `TDD`、`triage issue`、`QA`、`improve codebase architecture`、`auto-commit`、`code review`、`systematic debugging`。这部分开始真正进入实现、排错、验证和质量控制。

第四层是工具链和工程护栏，比如 `setup pre-commit`、`git guardrails`、`dependency auditor`、`git work trees`。这些 skill 的价值，在于把那些容易被忽视、但又经常出事的工程细节，变成默认执行的一部分。

第五层是写作与知识沉淀，比如 `edit article`、`ubiquitous language`。这意味着工作流并不在代码提交那一刻结束，而会继续延伸到表达、文档和团队语言统一。

这时候你再看 Claude，就会发现它已经不是“一个模型能做很多事”，而是“很多种工作方法可以被同一个模型调用”。这两者看起来只差一句话，实际差的是系统级能力。

## Skill 真正补上的，是工作流里最贵的那一层

开发者最贵的，不只是写代码的时间，而是切换上下文和重复解释流程的成本。

你让 Claude 单次生成一段代码，确实也能省时间。但如果每一个环节都要你重新讲一遍标准、重新说一遍顺序、重新盯一遍输出，那它本质上仍然只是个辅助工具。

Skill 的价值就在这里：把那些最容易在会话里流失的经验，变成长期稳定存在的方法层。

需求澄清不是靠今天灵感好不好，PRD 拆解不是靠这次提问问得够不够准，调试也不是碰运气式地乱试。你把这些动作分别写进 skill 之后，Claude 每次被调用时，执行的就是你定义好的套路，而不是随机发挥。

所以很多人会误以为“模型升级”是效率跃迁的核心，其实更大的跃迁，常常来自你终于把自己的做事方式标准化了。

## 使用教程

如果你以前没碰过 skill，最简单的入门方式不是自己从零写，而是先装现成的。

命令格式很直接：

```
  npx skills@latest add mattpocock/skills/[skill-name]
```

真正关键的，不是装了哪个名字，而是装完之后，你开始按“工作流分层”来组织 Claude 。

一个比较自然的起步顺序是这样的。

先从规划层入手。比如需求还模糊的时候，用 `grill me` 去逼自己把边界说清楚；需要成文时，用 `write a PRD` 把目标、约束、范围先固定下来；再用 `PRD to plan` 或 `PRD to issues` 把方案拆成可执行阶段。

然后再进入开发层。实现时用 `TDD` 控制节奏，出了问题用 `triage issue` 或 `systematic debugging` 去定位原因，快交付前再走 `QA` 和 `code review` 做质量兜底。

如果你已经有一套工程规范，就把工具链也接上去。`setup pre-commit` 负责把基础检查前置，`git guardrails` 负责减少高风险操作，`dependency auditor` 帮你盯住依赖问题，`git work trees` 则适合管理并行分支和实验环境。

最后别忘了知识层。功能做完，如果表达和文档不跟上，团队还是会反复踩坑。`edit article` 和 `ubiquitous language` 这种 skill，本质上就是在帮你把“做出来”继续推进到“讲清楚、传下去”。

很多人一开始会觉得这套东西有点重，但真正用起来你会发现，它不是变复杂了，而是把原本散在脑子里的隐性流程显性化了。你第一次整理会花点时间，后面每次调用都在回收这笔成本。

## 真实案例

对个人开发者和一人公司来说，Skill 的意义尤其大。

因为这类人最稀缺的，从来不是“会不会写一点代码”，而是能不能在一个人身上同时稳定承担产品、架构、开发、测试、评审、文档这些角色。

以前一个人做项目，最痛苦的不是不会，而是角色切换太频繁。上午还在想需求边界，中午开始写实现，下午被 bug 卡住，晚上回头补文档，第二天又发现上一个决定没有留下标准，整个过程反复断线。

有了 skill 之后，Claude 承担的就不再只是某个零散任务，而是整套协作节奏的一部分。

你可以先让它按 `write a PRD` 的方式把需求收束，再交给 `PRD to plan` 去拆执行阶段；开发时切到 `TDD`，出问题时切到 `systematic debugging` 或 `triage issue`；快交付时再用 `QA`、`code review`、`auto-commit` 这类方法收尾；最后让 `edit article` 或 `ubiquitous language` 帮你把成果沉淀成别人也能复用的表达。

这时候最本质的变化就出现了：你不再是一个人硬扛所有步骤，而是在调度一套已经被定义过的方法系统。

也正因为如此，真正的门槛慢慢从“会不会用模型”转向了“有没有能力把自己的流程抽象出来”。谁能把流程封装成可复用 skill，谁就能把 Claude 从聊天窗口接进生产系统；谁还停留在临时提问，谁就更容易把它用成一个高配版自动补全器。

## 最后一句判断

所以这件事真正值得重视的地方，不是又多了多少个 skill 名字，也不是谁整理了多少条清单。

真正重要的是：Claude 终于不只是一台会回答问题的机器，而开始具备了承载方法论、复用流程、分角色协作的能力。

当模型之上长出“技能层”之后，竞争就不再只是模型参数和回答质量的竞争，而是谁先把自己的工作方式系统化、工程化、可复用化。

很多人看到的是一个聊天窗口，少数人看到的，已经是一套随时待命的开发团队。
