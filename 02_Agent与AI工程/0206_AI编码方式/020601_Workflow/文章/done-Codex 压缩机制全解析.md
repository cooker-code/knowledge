> 💡 一句话先说清：Codex 的压缩，不是把历史聊天“简单做个摘要”这么朴素。它更像是一次上下文重建：先判断是否真的撑到上限，再按 Local / Remote 路径生成新的 `replacement_history`，把旧的 `active history` 直接替换掉。

如果你最近在看 AI Coding Agent，应该会越来越明显地感受到一个现实：真正决定 Agent 能不能跑长任务的，不只是模型聪不聪明，而是它怎么“忘”、怎么“记”、怎么“交接班”。

基于你给的原始文章，我把 Codex 的上下文压缩过程重新整理成了一篇更适合公众号传播的版本。我们不讲晦涩的源码细枝末节，而是聚焦两个问题：

1. 1. Codex 到底是怎么压缩上下文的？
2. 2. Remote Compact 和 Local Compact，实际效果差在哪？

## 为什么上下文压缩，决定了 Agent 的“长程战斗力”？

一个 Coding Agent 在处理复杂任务时，会不断累积三类东西：

* • 用户的新要求
* • 工具调用结果
* • 自己前面几轮已经做过的判断与动作

这些内容一旦越积越多，就会逼近上下文窗口上限。这个时候，系统面前只有两条路：

* • 不压缩：继续堆，直到超窗
* • 做压缩：保留最重要的信息，把历史重新整理成一个更短、还能继续工作的版本

问题在于，压缩做得不好，Agent 就会“失忆”；压缩做得太粗暴，它下一轮看起来还在工作，其实已经不是刚才那个自己了。

> ⭐ 真正厉害的上下文系统，不是“记住一切”，而是“在该忘的时候，仍然保留继续工作的能力”。

## 先记住 Codex 的核心思路：它不是“摘要流”，而是“换底稿”

很多人第一次理解 Codex compact，会以为它只是：

> 历史太长 → 让模型写个摘要 → 把摘要塞回去继续聊

但从原文梳理出的机制看，Codex 更接近一种 Checkpoint / Rebase 设计。

意思是：

* • 旧的 `active history` 不再继续原样使用
* • 系统会构造一份新的 `replacement_history`
* • 后续所有推理，都建立在这份新基线上

这和“保留原始长聊天，只是额外挂一个摘要”是两种完全不同的哲学。

> 💬 你可以把 Claude Code 想成“投影视图”：原始日志多数还在，只是模型眼前看到的是压缩后的视图。

> 🥇 你可以把 Codex 想成“重新交接一份底稿”：历史不是继续累加，而是压缩后生成一份新的工作起点。

## Codex 的压缩过程，到底怎么走？

先看一张图，把主链路看清：

这张图里最重要的，不是三条分支本身，而是一个判断顺序。

### 第一步：先判断是不是真的该压缩了

Codex 不会无脑压缩。它首先会检测当前上下文是否已经接近模型上限，比如：

* • pre-turn 阶段已经快超窗
* • mid-turn 还要继续追问，但 tokens 已经危险
* • 切换到更小上下文窗口的模型后，现有历史塞不下了

也就是说，compact 是一个“到线才触发”的动作，不是每轮都做。

### 第二步：判断当前 provider 支不支持 Remote Compaction

这里是 Codex 路由分流的关键：

* • 如果当前 provider 支持 remote compaction，就走远程分支
* • 如果不支持，就退回 Local Compact

换句话说，Local 并不是“默认更高级”的方案，它很多时候更像一个兜底路径。

### 第三步：远程路径里，再判断走 V1 还是 V2

如果 provider 支持远程压缩，Codex 还会继续判断是否启用了 RemoteCompactionV2。

于是最终形成三条路径：

* • Local Compact
* • Remote V1
* • Remote V2

## 三条路径，分别在做什么？

### 1）Local Compact：本地自己写一份“交接摘要”

Local Compact 的逻辑最容易理解。

它大致做的是：

1. 1. 把 compact prompt 作为一次新的输入发给模型
2. 2. 让模型根据完整历史生成 summary
3. 3. 从最近的真实 user messages 里，保留一部分高价值内容
4. 4. 拼成新的 `replacement_history`

压缩后的结果，核心长这样：

| 部分 | 含义 |
| --- | --- |
| `summary_user_message` | 模型生成的明文摘要，人类可以直接读、直接检查 |
| `recent_user_messages` | 最近一部分真实用户消息，按 token 预算从新到旧保留 |
| `replacement_history` | 新的有效历史基线，后续推理直接建立在它上面 |

Local 的最大特点是：它非常透明。

你基本可以知道：

* • 它保留了什么
* • 它删掉了什么
* • 它写出来的摘要长什么样

这意味着它非常利于调试、审计、复盘。

### 2）Remote V1：把压缩动作交给服务端专用端点

Remote V1 会调用专门的 `/responses/compact` 端点。

它的做法不是简单返回一句摘要，而是：

* • 把完整 prompt 发给服务端
* • 让服务端生成一份 compacted transcript
* • 再由客户端做一轮过滤，只保留适合继续工作的那部分内容

所以 Remote V1 的结果，已经不只是“写一段总结”，而是更像：

> 服务端直接帮你整理出一份新的压缩历史。

和 Local 比，它已经往“平台级接管压缩”迈出了一步。

### 3）Remote V2：最关键，也最有意思

真正让 Codex 和很多第三方 Agent 拉开差距的，是 Remote V2。

它不再走专用 compact 端点，而是在普通 `/responses` 流里，额外追加一个 `CompactionTrigger`。然后服务端返回一个特殊对象：

```
Compaction { encrypted_content }
```

注意这四个字：`encrypted_content`。

这意味着，客户端只能知道：

* • 这里有一段压缩后的核心状态
* • 它必须被保存下来
* • 后续请求里要继续把它带上

但客户端不知道里面到底是什么。

可能是摘要，可能是更高密度的压缩表示，可能是某种隐藏状态封装。至少从文章里整理出的源码可见部分来看，客户端无法解码，也无法审计。

> 💭 这就是 Codex Remote V2 最值得注意的一点：它不是“把摘要交还给客户端”，而是“把压缩后的关键状态封成一个黑盒，再交给客户端继续携带”。

## 那么，Remote Compact 和 Local Compact，效果上到底差在哪？

这也是本文最值得展开的地方。

很多人会把差异理解成：

* • Local = 本地压缩
* • Remote = 云端压缩

但这只是“部署位置”的差异，不是“效果差异”的本质。

真正的差别，在于压缩结果的形态不同、可见性不同、恢复方式不同。

### 差异一：Local 给你的是“能看懂的摘要”，Remote V2 给你的是“看不懂但可能更强的状态封装”

Local Compact 的核心产物，是一段明文 summary。

这有两个直接效果：

* • 人类可以审查它是不是写漏了
* • 第三方框架也比较容易复现这套逻辑

但它也有局限：

* • 摘要本质上还是自然语言重述
* • 一旦摘要漏了，后面就真的漏了
* • 连续多次 compact，容易出现“摘要套摘要”的信息衰减

Remote V2 则不一样。

它把最关键的压缩结果放进 `encrypted_content`。客户端看不懂，但这反而给了服务端更大空间：它不一定非得把信息退化成一段给人看的自然语言。

从效果上看，这通常意味着 Remote V2 更有机会保留“继续工作所需的高密度状态”。

但要注意，这里我说的是“更有机会”，不是“源码已经证明它一定更强”。因为 `encrypted_content` 的内部结构，客户端侧看不到。

### 差异二：Local 更可控，Remote 更连续

如果你是开发者、框架作者，或者你特别在意系统到底丢了什么，Local 会让你更安心。

因为 Local 的每一步都更像一个公开流程：

* • 触发了 compact
* • 生成了 summary
* • 保留了哪些 recent user messages
* • 最终 replacement history 长什么样

这意味着它更适合：

* • 排查“为什么 Agent 忘了这件事”
* • 对压缩质量做人工验证
* • 在不同 provider 之间迁移

但如果你更关心的是：

> 压完之后，Agent 能不能更顺地继续干活？

那 Remote，尤其 Remote V2，理论上会更有优势。因为它把压缩权交给了更靠近模型协议的一侧，服务端能做的事情比客户端大得多。

原文里对这一点的概括很重要：Codex 在 compact 后，不是只靠 summary 继续，而是通过 replacement history + initial context scaffolding 一起恢复工作能力。

也就是说，Codex 并没有把全部希望压在“摘要质量”上。

### 差异三：Local 的损失是“可见的”，Remote 的损失可能是“不可见的”

Local 的问题很朴素：

* • 摘要如果漏了，你能看出来
* • 最近用户消息如果保留得不够，也能看出来

Remote 的问题则更“现代”一些：

* • 它可能压得更好
* • 也可能丢了某些你无法观察的细节
* • 但客户端不具备验证能力

这就是两者在“效果”上的一个深层分歧：

| 维度 | Local Compact | Remote Compact（尤其 V2） |
| --- | --- | --- |
| 压缩产物 | 明文 summary | 服务端 transcript / `encrypted_content` |
| 人类可读性 | 高 | 低，V2 基本不可读 |
| 可审计性 | 强 | 弱，尤其 V2 |
| 连续工作潜力 | 取决于摘要质量 | 更依赖服务端压缩能力与协议能力 |
| 适合谁 | 框架作者、调试者、需要透明控制的人 | 追求长任务连续性、使用原生 provider 的用户 |

## 一个更接地气的比喻：你在给下一位同事交班

如果把 Codex 的 compact 想象成“项目交接”，那：

### Local Compact 像什么？

像你自己把整个项目整理成一份交接文档：

* • 这件事做到哪了
* • 哪些决定已经拍板了
* • 还有什么坑没填
* • 接下来第一步该干什么

优点是清楚、透明、谁都能读。

缺点是，一旦你交接文档写漏了，后面接手的人就真的不知道。

### Remote Compact 像什么？

像公司内部有一个很强的“项目档案系统”，你把项目移交过去，它给下一位同事的，不只是交接文档，还可能包括系统内部保留的一些结构化状态。

优点是，接手的人可能更容易快速进入状态。

缺点是，你本人不一定看得见这套内部档案到底长什么样。

> 🥈 所以，Remote Compact 和 Local Compact 的差异，不是“谁在压缩”，而是“压缩后的记忆，到底是公开笔记，还是平台内部封装状态”。

## 为什么 Codex 还要保留 `replacement_history` 这个概念？

因为它解决的是一个很多人会忽略的问题：

> 压缩完之后，系统到底该以什么为“新的过去”？

Codex 的答案很明确：

* • 不是继续沿用旧 `active history`
* • 不是简单把摘要贴在最前面
* • 而是直接生成新的 `replacement_history`

这一步非常关键。它相当于告诉系统：

> 从现在开始，你的“上下文过去”已经更新了，后续全部推理都建立在这个新历史上。

这就是为什么我更愿意把 Codex 的 compact 理解成重建工作基线，而不是简单的“上下文瘦身”。

## 对普通用户来说，最重要的结论是什么？

如果你不是源码分析者，而只是一个日常使用 AI Agent 的用户，真正值得记住的其实只有三点。

### 1. 别把 compact 理解成“自动摘要”

在 Codex 里，compact 更像一次上下文换底稿。压缩完以后，系统是在新的历史基线上继续工作。

### 2. Remote Compact 的价值，不只在“省 tokens”

它更大的价值是：把压缩这件事推进到更靠近模型协议的一层。尤其是 Remote V2，客户端不再持有完整可读摘要，而是持有一个 opaque state。

这可能带来更强的连续性，但也带来更低的可解释性。

### 3. 任务太长时，开新线程往往比反复压缩更稳

无论是 Local 还是 Remote，多次 compact 都不是没有代价的。压缩再聪明，也不是无限无损。任务真的跑得过长时，重新开一个更干净的新线程，往往比反复“压了再压”更靠谱。

## 最后，用一句话收尾

Codex 的 compact，本质上不是“把长对话缩短”，而是：

> 在上下文即将爆掉的时候，重新生成一份还能继续工作的历史基线。

而 Local Compact 与 Remote Compact 的真正差异，也不只是本地和远程，而是：

* • Local 更像可读、可审计、可复盘的交接文档
* • Remote 更像平台内部保留更多状态的系统级交接

前者透明，后者强大；前者容易检查，后者更像黑盒能力。

理解这一点，你就会明白：AI Agent 的竞争，已经不只是模型回答得好不好，而是谁更擅长在超长任务中“体面地记住自己”。
