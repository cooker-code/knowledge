运维人自救搭了一套 AI 告警根因分析工具。记录一下我亲手搭的草台班子。这套工具主打轻量化，上手毫无难度，新手也能一分钟玩明白。

    运维人肯定都经历过半夜被告警吵醒、对着日志逐行排查？这套工具帮你把半小时的排查工作，压缩到 2 分钟完成。（目前只完成demo,后续待扩充更多场景）

先拿最常见的**磁盘占用过高告警**举例子

告警内容：节点根目录使用率突破 91%，触发阈值告警。

### 传统排查流程

### 

登录服务器 → `df -h` 查分区 → `du -sh` 找大文件 → 逐个排查容器日志 → 核对 Docker 配置

一套流程走下来，可能**半小时**。遇上半夜，心态直接崩一半。

### 换上我的 Alert Agent（草台班子上线）

### 

粘贴告警文本，剩下的全交给系统自动跑：

(后续优化直接接入alertmanager，最大化减少手动部分）

接下来的步骤是：

1. （1）自动连接服务器执行排查命令
2. （2）收集日志与系统数据
3. （3）AI 智能分析定位根因
4. （4）输出完整结构化报告

（5）全程3-5**分钟**出结果，报告写得明明白白：

* 现场现象：根分区总容量 222G，已使用 201G，Docker 容器日志独占 190G；
* 问题根因：系统已配置 Docker 日志轮转策略，但仅对**新建容器**生效，存量容器未继承配置，日志持续膨胀撑满磁盘；
* 附带所有排查命令，拿来就能用。

不用手动敲一条命令，不用逐行翻日志，这个 “草台班子”，干活效率还行。

### ✅ 核心能力

* （1）多方式接入告警，自动解析关键信息
* （2）远程执行排查命令，全程日志可追溯
* （3）大模型智能定位故障根因，深挖隐性问题
* （4）自动生成标准化报告，复盘、同步一键搞定

### 🧩 整体架构

    从告警接入→智能排查→AI 分析→报告输出，全链路自动化，逻辑简单清晰，落地性拉满，很容易上手

### 📲 4 步快速使用

### 

1. （1）提交告警（手动 / 粘贴文本均可）
2. （2）系统自动执行排查
3. （3）查看 AI 分析报告
4. （4）处理故障并归档记录

     搭建这套工具，只为解决日常告警多、排查慢、重复劳动多的痛点。虽然是个人的 “草台班子”，但完全服务实战场景。

    也许你会说，我直接在告警产生的时候触发自动化编排的脚本，查一下服务器上面的数据不就可以了嘛，有些场景确实是可以这么做，这也是以前我们平台辅助告警诊断的一种方式。但是这样场景落地使用会比较受限，为了更好的应用模型的分析能力，就可以分2步走。

（1）固化的告警处理就总结成skills,做成告警与skills的映射。

（2）服务异常的告警，让模型去分析服务的日志，通用模型的能力从今年的故障处理来看，准确率是非常高了。
