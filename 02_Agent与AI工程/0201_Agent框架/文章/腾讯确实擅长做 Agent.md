OpenAI 姚顺雨加盟腾讯的消息，被辟谣了。但能感觉出来，公众还是非常看好腾讯在 Agent 领域的前景。

无论是在整个微信生态系统中打造 AI Agent，还是其他全新的 Agent 产品，腾讯在产品体系和用户基盘上的优势都极为突出。元宝的反超就是最直接的证明。

这段时间，我也有感触。如果说 AI 真的已经到了下半场，那腾讯这家公司的积累和能力，会慢慢显现出来。

最近我打算给公司搭几个 Agent，选来选去，最后还是觉得腾讯云的智能体开发平台（ADP）最合适。

和市面上不少同类产品相比，它从一开始就把重点放在 ToB 场景上，更关注企业复杂流程和个性化需求的落地。

这半年接触了越来越多智能体应用，我的感觉也愈发确定：企业场景才是智能体的主战场。

消费级产品可以靠创意和流量迅速起势，但真正能跑得久、跑得深的，还是那些能嵌进企业业务里的产品。毕竟，AI 的核心价值还是提升生产力。

过去的十年，行业内诞生了非常多的 SaaS 产品，SaaS 公司创始人最常回答的问题是：在定制化和标品之间怎么做权衡。无论对方的回答是什么，都无法掩盖这样的事实：客户是有很多定制化需求的。

软件完全可以更个性化，而这些需求过去都被忽视了。腾讯云智能体开发平台，我觉得核心就是在解决这个问题。

******#01******

**案例测评**

废话不说多，接下来就搭两个应用看看效果。

### 文章 Review Agent

每天改文章改到头大，几乎成了我的日常。

我的写作流程大致分三步：先自己写完初稿，再检查错别字，最后调整整体结构。后面两步，我其实早就总结出一套提示词和标准，让 AI 来代劳。特别是查错别字，人真比不过 AI。

最近团队来了新同事，我把自己的提示词分享给了他，但发现他用得还是很生疏。我没生气，也没爹味。

本着“能流程化就流程化，能产品化就产品化”的思路，我干脆开始搭建团队第一个真正能工作的智能体。

打开腾讯云智能体平台：

https://www.tencentcloud.com/zh/products/tcadp

我先新建了一个应用，选择单工作流模式。

这件事的流程本身就很清晰，我希望这个 Agent 能沿着我规定的路径一步步执行，不需要额外的发挥或创意，只要按部就班地跑完流程，稳定交付结果就行。

查错字这一块，我们团队有一套自己的校验规则。

除了常规的错别字，这些年我还整理了一份专有名词对照表，比如 Agent 要写成大写 A，LLM 也必须是大写。

写初稿的时候偶尔着急，我常常顾不上这些细节，但在最后环节，我想让它们全部对齐标准。不能让用户觉得我们的文字乱七八糟。

之前在 GPT 里，每次我都得上传一个 PDF 附件。现在想想，也不知道自己是怎么有那么多耐心的，也不怕麻烦。

这次，我用腾讯云开发完一个完整的 Agent 后，以后这就是一个开箱即用的产品，只要把自己的文章扔进去就行，其他的一概不用操心。让团队用起来的最好方式就是降低门槛。

腾讯云支持接入文档、问答对和数据库三种数据源。因为我之前已经有现成的 PDF，所以直接导入文档就行。或者如果你有网页，那直接输入链接也行。

总之，这部分私有数据的导入腾讯云做的还是比较完善，毕竟这也是一个 Agent 能否胜任的关键。

接下来我开始搭自己的工作流程。

平台提供了一张空白画布，我把流程里需要的每个步骤依次放上去，再用连线确定执行顺序。操作步骤挺直观的，基本上就是拖动、连接、设置参数。

如果你没做过开发，那也完全能够搞定。虽然是开发平台，但使用门槛已经非常低了。

下面是我的工作流，我在文稿里故意埋了几个错别字，比如把 LLM 写成 LLm，Agent 最后能够识别出来。润色也很顺，一键点击就能优化整篇。

有意思的是，我提了个特殊要求，把文章缩减到 1000 字。一般情况下，这种需求很容易让流程乱掉，要么从头跑一遍，要么干脆报错。

但这次并没有，它精准的跳回了自定义优化这个节点，从那里重新开始。像个熟练的编辑，整个过程干净利落，没有多余的折腾。

换句话说，就是我没有额外的配置，直接得到了一套流程控制能力。感觉就像有一个隐形管理员在背后打理好一切，我只管看结果。

### Timeline Agent：一分钟生成网页排期

我还试了个 Timeline Agent，先把需求排期塞进知识库，再让它自动生成时间线，再部署到网页。

这个部署的部分用到了插件，使用插件也很简单，同样是直接拖过去。现在插件广场，有很多限时免费的插件，大家可以去玩一玩，挺有意思的。

Agent 的整体思路是：输入需求-> RAG 检索数据 -> 大模型生成 HTML -> 插件 Deploy 部署 -> 生成网页 URL。

过去要做这个，得拉个前端写半天代码；现在拖拽一下插件，连个大模型，几分钟就生成一个能用的网页。部署环节也很省心，基本不需要我操心什么。

直接看效果：我让它根据知识库里的需求信息，生成一个 Timeline 网页。它执行后立马给我生成了网页链接，点开链接，需求时间轴、数据分析和洞察都在里面，清清楚楚。

这一刻最直观的感受是，技术门槛被拉低了，这种效率感才是企业真正需求的。

### Multi-Agent 协作：主 Agent 当调度员

最后我还试了 Multi-Agent 模式，接到业务系统里，用来做问数任务。

我设置了一个主 Agent（智能问数 Agent），作为应用的核心调度者，来负责分配任务。比如问收益问题的时候，它就把问题转给商单 Agent。问需求时间节点问题时，它就转给需求 Timeline Agent。

调试满意了，直接发布。平台还自带一整套评测、发布和运营功能。你不用再折腾一堆额外工具，直接在一个平台里闭环跑完，省心省力。

******#02******

**几个亮点**

事后回想，有几个细节让我印象挺深。

先是 RAG。以前做知识库最让我头疼的，就是文档一多就乱，新旧版本打架，AI 又一本正经地胡说。

比如下面我特意混了几份旧文档进去，结果它在上传时直接把冲突点标出来，我点一下就能决定保留还是覆盖。

更重要的是，遇到模糊问题它会追问，实在不懂就拒绝，而不是装懂乱编。这对于企业的应用场景来说挺重要的。最近 OpenAI 提到模型的幻觉，核心也是希望能够让模型在不懂的时候直接说不懂，而不是瞎说。

再是 Multi-Agent。在腾讯云上搭多智能体协作的时候，体验比我预想的要顺很多。

智能体之间可以自由转交任务，像团队里拍一下肩膀说一句交给你了；如果流程比较长，也可以在画布上事先编排好转交的顺序，让它们按节点依次接力。

平台还提供了 P&E 协同模板，把规划和执行拆成两个角色协作，拖拽几下就能跑出一条完整的链路，用来做网页生成、报告生成这种复杂任务也不费力。整个过程几乎不用写逻辑。

另外，腾讯云内置了一百多个插件，想加什么功能直接拖进去就行，它们会自动接入多智能体的流程里。这种扩展方式让我第一次觉得，做复杂应用也可以搭得很轻松。

Workflow 是不是智能体，年初很多人都在争论。我倾向于认为预置的 Agent，其实不是智能体。但怎么讲，放到自己的场景中，我确实需要一个预置流程的 Agent。

所以，我也不管定义了，先满足自己的需求。

这次，腾讯云智能体开发平台上，Agent 的稳定性也让我印象深。流程一长最怕的就是崩掉，以前这种情况太多了。

这次我看到它在流程里安排了一个全局 Agent，当调度员一样盯着全场。用户说“改时间”，它能自动回退到预约节点；用户临时说“算了”，就能立刻退出。

我完全没写条件判断，但流程衔接得很自然，整个过程兜得很稳。企业做应用，最怕的就是不稳定，这块让我挺放心。

******#03******

**写在最后**

这两天，我开始在公司内部推广 Agent。

也开始认真反思，还有哪些重复的工作可以交给 Agent，还有哪些流程我反复强调、大家却总是记不住，那我也该干脆把它们做成 Agent。

说来惭愧，作为一个每天写 AI 的人，直到现在才真正意识到：公司内部的流程和工作，其实完全可以一步步 Agent 化。只是以前没往这个方向想，多少有点后知后觉。

回头想想，我们太习惯那套旧逻辑了。

过去要上一个 SaaS 产品，动辄一年得花上万块，久而久之也就默认，个性化的小需求没办法被满足。

但时代已经变了。像腾讯云这套智能体开发平台，已经把成本压得足够低，让任何规模的公司都能快速搭建自己的 Agent，把那些日常的琐碎和重复流程自动化重做一遍。

也许这就是下一阶段的起点：不再是我们去适应工具，而是工具开始适应我们。企业里那些最不起眼的细枝末节，也能用 Agent 去重塑一遍。

就像今天我举的各种细碎的小例子一样。其实，写完文章，我又已经在优化我们的 Agent 了。

比如，文章 Review 的 Agent 和 Timeline Agent 完全可以合并。我只需输入文章标题，Agent 先生成大纲初稿和一份清晰的时间线，方便随时同步给同事或客户。

文章定稿后，再把全文交给它，自动完成校对、结构调整、格式统一和素材打包等后续步骤。

这都是后面的事。慢慢迭代，慢慢优化。更重要的是先动起来，把思路切换到用 Agent 解决具体问题，从一个小流程切进去，让它在团队里真正跑起来。

等到第一个可用的 Agent 上线，第二个第三个就不难了。下半场的工作方式，大概就是这样被一点点重写的。

我好像理解了什么是 AI 原生公司了。惭愧，晚了一些。
