先把一件事说清楚：

**Agent和ChatGPT并不是一类东西。**

AI Agent并不能够被简单理解为“更聪明的ChatGPT”——前者是Agent，后者是chat box。

我们现在用的豆包、Kimi、ChatGPT，还有网页版的Claude，其实他们都属于ChatBox。

而博科尼这门课教的Claude Code，是一个Agent。

Claude Code作为一个Agent：

  -它在电脑的终端（Terminal）运行你的指令

  -访问权限（Access Level）是你的所有本地文件

  -它可以产出文件、图表、代码……

  -如果Chatbot更适合brainstorm和回答问题，那么ClaudeCode就更适合去handle那些带着数据的真实projects。

总结起来，你可以这样理解：

**"ChatGPT talks about your work,**

**Claude Code does your work."**

多提一嘴，

那么Agent是怎么工作的呢?

它的工作核心其实是一个循环(agent loop):

**Step1 先收集上下文**

这一步里，它会阅读你的prompt、阅读当前所处文件夹里面的所有文件以及在当前聊天窗口里的一些历史记录。

**Step2 开始执行**

它会根据你的指令需求来去查找文件，编写代码，跑命令，然后生成结果。

**Step3 检查结果**

比方说，它会自己去跑test，自己检查结果，自己确认根据你命令做出的改变是works的。

总之，上面三步就是一个loop，

Claude Code会一直repeat这个loop，

直到完成你给的任务。

OK, 现在你大概明白Claude Code是怎么工作的了。

但是在我给你展示它能做什么之前，

我还想先告诉你：

怎么样才能让Claude Code不发疯。。

这里会涉及三个概念，分别是：

CLAUDE.md、Context Window和Plan Mode

**第一个是CLAUDE.md**

每次你在VSCode terminal里面启动Claude Code，

你都会获得一个全新的“渣男”——

他不会记得你上次说过什么，不会记得你的项目是做什么的，不记得你喜欢什么格式，也不记得有哪些文件是不能动的……

每次都会像是你们刚相处的第一天。。

那么CLAUDE.md就是用来解决这个问题的！

CLAUDE.md是一个放在你项目文件夹里的文本文件，你会在里面写清楚那些不希望每次启动Claude Code都要跟它重复一遍的基本信息：比方说这个项目的背景是什么，你有哪些固定的偏好，有什么规则是每一次都要遵守的……

Claude Code每次启动都会先读CLAUDE.md。

这意味着你只需要编辑一次这个文档，后面第二次，第三次第n次，只要你在这个项目文件夹里面启动Claude Code，它自己就会先复习一遍你的要求。

特别省事。。-

这是一个我在每个月的月底跟父母报销分账时通用的CLAUDE.md：我会用excel记录开支，claude code会根据这个CLAUDE.md里面的规定来帮我算明白到底谁要转给谁多少钱

但是，

有些人可能不善于做取舍，

可能他们觉得好多东西都懒得再重复一遍，

所以他们可能会想：

“那我把所有东西都写进CLAUDE.md不就好了？”

那你就错了！

就引出了我们的**第二个概念，叫做context window。**

在课堂上，教授把这个概念比作是抽屉；

不过我更喜欢把它理解成claude code在一次工作里的脑容量——

Claude Code每次能“记住”的信息量是有上限的。

但凡是在你跟它这次对话里涉及到的内容，

包括但不限于你的CLAUDE.md、你说的话、它读过的文件、它自己的回复……这些东西都会占据Claude Code的脑容量。

所以如果你的CLAUDE.md写得太长，

那么Claude Code每次启动光是读这份说明书就已经用掉了很多脑容量，

那留给让它真正做事的脑容量就会变得更少——

**它就会变蠢！发疯！失焦！气死你！**

孩子发疯怎么办呢？

多半是context window满了！

那么这件事情将教会我们什么呢？

1. CLAUDE.md是一个永远不会被压缩，并且必然会在每次启动claude code的时候都先占据你的初始context window的存在——所以它不能太长。

CLAUDE.md里面只需要放稳定的全局的规则，至于临时的任务需求，在对话里说就够了。理想状态下，它占据的context window应不超过10%~15%

2. 不要用同一个聊天窗口做所有事情。

对话越来越长，Claude Code的表现就会越来越偏，越来越难聚焦。更好的方式是一个任务，一个新窗口，做完关掉，再开新的。

3. 主动管理context window

其实Claude Code自己在意识到“脑容量”快到上限时，也会自动压缩那些已经存在的占context的聊天记录。

但这样带来的坏处是，你无法控制它什么时候进行压缩。这就容易打断你的workflows。

如果你能够在结束workflows里面一段环节的工作之后，主动运行**/compact**这个命令来去进行压缩，就能够更好的把context window的管理主动权掌握在自己手里。

让我们回到最后一个能够避免你的cc（Claude Code的简称，后文就这样叫它了）发疯的东西：

**Plan Mode**

我还是蛮喜欢这个东西的！

我觉得plan mode有两个作用：  
第一个是，你在让cc动你的文件之前，可以用plan mode确认它理解的方向是对的。  
第二个是，如果你自己也不确定“正确的方向”是什么，你自己也比较迷茫，那么你同样可以用plan mode来去进行一个探索。

你可以通过**两次Ctrl+Shift**来使cc切换到Plan模式。

开启plan mode之后，

cc会在动任何文件之前先给你列出他的行动计划：

比方说它打算如何实现你的这个任务，每一步会做什么，会改哪里，等等等等。

它还会通过采访你，给你一些选项，从而和你对齐颗粒度（这是我特别喜欢plan mode的一个点）

比方说我在数据可视化的期末作业那儿有一个选题想法，但是具体不知道怎么实施，以及是否可以实施。那么我就可以在plan mode里说“我想就微软最新发布的working index相关报告，来去找一下我数据可视化作业的选题方向”。然后cc就会帮你web search, 然后用interview的方式来去跟你对齐颗粒度.

你看完后，觉得方向对，再让cc执行；

如果觉得哪里不对，那么就可以当即跟它说，这会比等cc做完之后再返工要省事的多。

整个过程就好像你雇了一个人来帮你做项目，

你不会让ta闷头先做个两周再给你看结果，

ta也会先主动地来跟你进行一次方案确认，

你们对齐方向之后，ta再动手。

这不就很好地规避了cc趁你不注意突然发疯，然后用他的雷霆output把你惊掉下巴的局面！？
