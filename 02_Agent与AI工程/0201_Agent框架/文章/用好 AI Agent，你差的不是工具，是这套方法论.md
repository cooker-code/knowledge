**你有没有遇到过这些情况？**

花了半小时调教 AI，结果它帮你"搭建了一个认证系统"——但用的是你完全没听过的库，和你的技术栈毫无关系。

让它找代码里的 Bug，它洋洋洒洒列了 8 条，你逐一排查，发现 6 条根本不是问题。

你的 CLAUDE.md 越写越长，规则越加越多，刚开始效果还不错，但跑了几周之后，AI 的表现反而越来越奇怪，开始做一些莫名其妙的事情。

这些问题，你大概率不是第一次遇到。

很多开发者的直觉反应是：换个更好的工具、装更多插件、找更多提示词模板。但折腾一圈之后，你会发现问题依然存在。

**根源不是工具，是你跟 AI 协作的方式出了问题。**

---

## 核心问题：上下文膨胀

在深入讲方法之前，先要理解一个概念——**上下文膨胀（Context Bloat）**。

简单说：你给 AI 塞的无关信息越多，它的表现就越差。

这听起来很反直觉。你以为给 AI 更多背景信息是负责任的做法，但实际上，一个正在写 Hangman 小游戏的 Agent，如果它的上下文里还装着"71 次会话前某个进程 hang 死"的记录、"怎么管理内存"的笔记、以及 14 个 markdown 规则文件——它大概率会开始混淆任务边界。

**Agent 需要的是精准的信息，不是完整的信息。**

这是本文所有方法的底层逻辑。

---

## 四个让 Agent 真正好用的原则

### 原则一：研究和实现，必须分开

这是最常被忽略、但影响最大的一个点。

很多人习惯给 AI 一个模糊的大任务，比如：

> "帮我搭建一个用户认证系统"

AI 接到这个指令后，会先"研究"——什么是认证系统？有哪些方案？JWT 好还是 Session 好？OAuth 要不要加？研究完，上下文里已经塞满了各种方案的对比细节。等它真正开始写代码，脑子里装的是 5 种不同实现方式的混合体，幻觉概率大幅上升。

**正确做法是两段式：**

**第一步：研究任务（单独开一个对话）**

```
你是一个后端架构师。我需要为一个 Node.js + PostgreSQL 项目  
设计用户认证方案。请对比以下维度：  
- JWT vs Session-based  
- 密码加密方案（bcrypt rounds 建议）  
- Refresh token 策略  
- 安全性风险点  
  
给我一个明确的推荐方案和理由，不需要写代码。
```

**第二步：实现任务（新开一个窗口）**

```
请实现以下用户认证模块，技术栈 Node.js + PostgreSQL：  
- JWT 认证，access token 15 分钟过期  
- bcrypt rounds=12 密码加密  
- Refresh token 7 天有效，支持 rotation  
- 登录失败超过 5 次锁定账户 30 分钟  
  
只实现认证核心逻辑，不需要考虑其他模块。
```

两段式的好处在于：实现阶段的 Agent 上下文只有"怎么做"，没有"要不要做"的干扰，输出质量会明显提升。

---

### 原则二：用中性提示，对抗 AI 的"讨好本能"

AI 被训练成尽力满足你的要求——这是它好用的原因，也是它会坑你的原因。

你说"帮我找代码里的 Bug"，它就一定会找到——哪怕它需要把一个正常的边界判断解读成"潜在空指针风险"。你给了它一个预设结论，它会帮你圆这个结论。

对比两种提示方式的差异：

| 场景 | ❌ 有偏见的提示 | ✅ 中性提示 |
| --- | --- | --- |
| 代码审查 | "帮我找这段代码里的 Bug" | "梳理这段代码每个函数的逻辑，汇报所有发现" |
| 方案评估 | "这个架构有什么问题？" | "分析这个架构的优势和潜在风险，客观评估" |
| 性能分析 | "为什么这个接口这么慢？" | "分析这个接口的执行链路，记录每个阶段的耗时情况" |
| 需求评审 | "这个需求可行吗？" | "列出实现这个需求的技术路径和每条路径的约束条件" |

中性提示的核心逻辑：**不给 AI 一个需要证明的结论，而是给它一个需要完成的观察任务。**

---

**进阶：多 Agent 博弈，逼出更高精度**

如果你需要高可信度的输出，比如安全审计、关键业务逻辑 review，可以用三 Agent 博弈系统：

```
Agent A（攻击者）：  
你是一个安全研究员，专门寻找代码中的漏洞。  
发现低危漏洞得 1 分，中危得 5 分，高危得 10 分。  
目标：最大化总分。列出所有发现，包括你认为值得关注的边界情况。  
  
Agent B（防御者）：  
你是一个代码审核专家。下面是另一位研究员提交的漏洞报告。  
你的任务是逐条反驳——证明这些不是真实漏洞。  
成功反驳一条得对应分值，但如果反驳错误，扣该分值的 2 倍。  
  
Agent C（裁判）：  
你是最终裁判。我手中有标准答案。  
你的评分越准确得 +1，越不准确得 -1。  
基于攻防双方的陈述，对每条漏洞做最终裁决。
```

这套机制利用了 AI 的"讨好本能"——让三个 Agent 分别在不同方向上努力"表现"，最终通过博弈逼近真实答案。

### 原则三：给任务一个明确的终点

对人来说，"任务完成"是一种感觉。对 Agent 来说，它需要一个**可验证的终止条件**，否则它会在某个"差不多"的节点停下来——比如实现了接口但没写测试，或者写了测试但没跑通。

**方法一：测试即终点**

```
实现用户注册接口。  
  
完成标准：  
- 以下测试全部通过（不允许修改测试文件）  
- test/auth/register.test.js  
- test/auth/register-edge-cases.test.js  
  
在所有测试通过之前，任务未完成。
```

**方法二：CONTRACT 文件**

对于复杂任务，创建一个 `TASK_CONTRACT.md`：

```
# 用户认证模块 - 完成合同  
  
## 功能验收  
- [ ] POST /auth/register 注册接口  
- [ ] POST /auth/login 登录接口（返回 access + refresh token）  
- [ ] POST /auth/refresh 刷新 token  
- [ ] 登录失败 5 次账户锁定逻辑  
  
## 测试验收  
- [ ] npm run test:auth 全部通过  
- [ ] 覆盖率 > 80%  
  
## 安全验收  
- [ ] 密码字段不出现在任何响应体中  
- [ ] token 过期逻辑正确  
  
## 终止条件  
以上所有 checkbox 完成前，禁止结束任务。
```

这个 CONTRACT 文件可以配合 stop hook 使用：Agent 在终止前必须核对合同，未完成则继续执行。

### 原则四：CLAUDE.md 是目录，不是说明书

很多人的 CLAUDE.md 是一个越来越长的"总规则文档"，把所有偏好、规则、背景全堆在一起。这是上下文膨胀的重灾区。

**正确的定位：CLAUDE.md 是一个逻辑路由表。**

它的工作不是"告诉 AI 所有事情"，而是"告诉 AI 在什么情况下去读哪个文件"。

```
# CLAUDE.md  
  
## 开始任何任务前  
1. 读取 rules/core-principles.md  
2. 读取当前任务对应的 CONTRACT 文件（如果存在）  
  
## 写代码时  
→ 读取 rules/coding-standards.md  
  
## 写测试时  
→ 读取 rules/testing-rules.md  
  
## 测试失败时  
→ 读取 rules/debug-checklist.md  
  
## 涉及数据库操作时  
→ 读取 rules/database-rules.md  
  
## 遇到不确定的设计决策时  
→ 先提问，不要自行假设
```

**Rules vs Skills 的区别：**

|  | Rules（规则） | Skills（技能） |
| --- | --- | --- |
| **用途** | 编码你的偏好和禁忌 | 编码具体的操作方法 |
| **内容** | "永远不要直接操作生产数据库" | "如何写一个 database migration" |
| **触发方式** | 场景触发，由 CLAUDE.md 路由 | 任务触发，显式调用 |
| **维护频率** | 随使用不断迭代 | 需要时新增，不常改动 |

当你发现 Agent 开始"做奇怪的事"，通常有两个原因：规则之间出现了矛盾，或者 Skills 太多导致命名混淆。这时候需要做一次"清理"——让 Agent 整合现有规则，消除矛盾，重新梳理命名。

## 深度思考：方法论背后的核心能力

以上四个原则都是可以立刻上手的技巧，但如果你想真正拉开差距，需要建立三个更底层的能力。

**① 任务拆解能力，才是真正的壁垒**

把"搭建一个电商平台"拆解成 30 个边界清晰、上下文精准的 CONTRACT 文件——这件事，没有任何工具能帮你做。这需要你对系统架构、业务逻辑和 Agent 能力边界同时有深刻理解。

这是区分"普通 AI 用户"和"世界级 Agent 工程师"的真正分水岭，不是你会不会写提示词。

**② 建立你的失败日志**

优秀的工程师不只关注"让 Agent 成功"，更要系统记录"它在哪里失败"。比如：

* 哪类任务最容易产生幻觉？（通常是"研究 + 实现"混在一起的任务）
* 什么时候 Agent 会绕路？（通常是终止条件不清晰时）
* 哪些 Skill 文件被频繁误触发？（命名不够精准的信号）

这些数据会让你的规则和技能文件越来越精准，形成真正属于你的"协作体系"。

**③ 培养人机协作的节奏感**

什么时候该接管，什么时候放手——这是最难用文字描述、但最值钱的一种直觉。

经验法则：**任何涉及"假设"的环节，Agent 最容易出问题。** 只要你发现它在做推断而不是执行，就该介入。提前在 CLAUDE.md 里写明"遇到不确定的设计决策时，先提问"，能大幅减少这类漂移。

---

## 最后

用好 AI Agent 是一个持续做减法的过程。

不是安装更多工具，而是精简不必要的上下文；不是写更长的 CLAUDE.md，而是让它成为更清晰的路由表；不是让 AI 做更多决策，而是把模糊的目标拆解成它能精准执行的任务。

**你给 AI 的信息越精准，它给你的结果就越好。**
