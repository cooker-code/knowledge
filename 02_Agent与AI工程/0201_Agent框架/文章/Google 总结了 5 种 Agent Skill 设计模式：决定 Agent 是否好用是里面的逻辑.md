---

谈到 SKILL.md，开发者往往专注于格式:正确使用YAML格式，结构化目录，并遵循规范。

但随着30多个代理工具（如Claude Code、Gemini CLI和Cursor）统一采用同一布局，格式问题几乎已经过时。

**现在的挑战是模式设计。**

规范解释了如何打包技能，但对如何构建其中的逻辑完全没有指导。例如：

> 一个包裹FastAPI惯例的技能，尽管它们的 SKILL.md 文件外观相同，但其操作方式与四步文档流水线完全不同。

通过研究整个生态系统中技能的构建方式：从Anthropic的仓库到Vercel和谷歌的内部指南，有五个反复出现的设计模式可以帮助开发者构建代理。

本文介绍了每个带有正常 ADK 代码的版本：

* **工具包装器**：让你的经纪人瞬间成为任何图书馆的专家
* **生成器**：从可重复使用的模板生成结构化文档
* **评审员**：根据严重程度对应清单评分代码
* **采访**：经纪人在行动前会面试你
* **流程**：强制执行严格的多步骤工作流程，并设有检查点

## 模式1：The Tool Wrapper(工具包装器)

工具包装器为你的代理提供特定库的按需上下文。你不是把API惯例硬编码到系统提示中，而是打包成技能。你的代理只有在实际使用该技术时才加载这个上下文。

这是最简单的模式。

SKILL.md 文件会监听用户提示词中的特定库关键词，动态加载参考文献/目录中的内部文档，并将这些规则作为绝对真实应用。

这正是你用来将团队内部编码指南或特定框架最佳实践直接分发到开发者工作流程的机制。

这里有一个工具包装器的示例，它教代理如何编写 FastAPI 代码。注意说明明确指示代理只在开始审查或编写代码时加载 conventions.md 文件：

```
skills/api-expert/SKILL.md  
## 名称：api-expert  
描述：FastAPI 开发最佳实践与规范。在构建、审查或调试 FastAPI 应用程序、REST API 或 Pydantic 模型时使用。  
  
## 元数据：  
模式：工具包装器 (tool-wrapper)  
领域：fastapi  
你是一位 FastAPI 开发专家。请将以下规范应用于用户的代码或提出的问题。  
  
## 核心规范  
加载 references/conventions.md 以获取完整的 FastAPI 最佳实践列表。  
  
## 代码审查时  
1.加载规范参考文档。  
2.对照各项规范检查用户的代码。  
3.对于每一处违规，引用具体规则并提供修复建议。  
  
##编写代码时  
1.加载规范参考文档。  
2.严格遵守每一项规范。  
3.为所有函数签名添加类型注解 (Type Annotations)。  
4.使用 Annotated 风格进行依赖注入。
```

## 模式二：The Generator(生成器)

工具包装器应用知识，生成器则强制输出一致。如果你遇到代理每次生成不同文档结构的问题，生成器通过编排填空过程来解决这个问题。

它利用两个可选目录：资产/存储输出模板，和引用/保存样式指南。说明书起到项目经理的作用。

他们会让代理加载模板，阅读样式指南，询问用户缺失的变量，然后填充文档。

这对于生成可预测的API文档、标准化提交消息或构建项目架构非常实用。

在这个技术报告生成器的示例中，技能文件并不包含实际的布局或语法规则。它只是协调这些资产的检索过程，并强制代理一步步执行：

```
skills/report-generator/SKILL.md  
名称：report-generator（报告生成器）  
描述：生成结构化的 Markdown 技术报告。当用户要求编写、创建或起草报告、总结或分析文档时使用。  
## 元数据：  
模式：生成器 (generator)  
输出格式：markdown  
你是一位技术报告生成专家。请严格执行以下步骤：  
第1步：加载 references/style-guide.md 以获取语调和格式规则。  
第2步：加载 assets/report-template.md 以获取所需的输出结构。  
第3步：向用户索取填充模板所需的任何缺失信息：  
  ·主题或对象  
  ·核心发现或数据点  
  ·目标受众（技术人员、高管或普通读者）  
第4步：遵循风格指南规则填充模板。模板中的每个章节都必须在输出中呈现。  
第5步：将完成的报告作为一份完整的 Markdown 文档返回。
```

## 模式三：Reviewer（评审员）

Reviewer 模式将检查内容和检查方式分开。你不必写一个详细描述每个代码气味的长系统提示，而是把模块化评分标准存储在 references/review-checklist.md 文件里。

当用户提交代码时，代理会加载该检查表，并按严重程度对提交进行系统评分。

如果你用OWASP安全清单替换Python风格的检查表，你得到的审计完全不同，且使用完全相同的技能基础设施。

这是一种非常有效的方法，可以自动化公关审查或在人工查看代码前发现漏洞。

以下代码审查技能展示了这种区分。指令保持静态，但代理会动态加载外部检查清单中的具体审查标准，并强制输出结构化、基于严重度的输出：

```
skills/code-reviewer/SKILL.md  
名称：code-reviewer（代码审查员）  
描述：审查 Python 代码的质量、风格及常见漏洞。当用户提交代码进行评审、寻求代码反馈或需要代码审计时使用。  
  
##元数据：  
模式：审查器 (reviewer)  
严重程度等级：错误 (error)、警告 (warning)、提示 (info)  
你是一位 Python 代码审查专家。请严格执行以下审查协议：  
第1步：加载 references/review-checklist.md 以获取完整的审查标准。  
第2步：仔细阅读用户代码。在进行评价之前，务必先理解代码的意图。  
第3步：对照清单中的每一项规则检查代码。对于发现的每一处违规：  
  ·注明行号（或大致位置）  
  ·分类严重程度：错误（必须修复）、警告（应该修复）、提示（建议考虑）  
  ·解释为什么这是一个问题，而不只是指出哪里错了  
  ·提供具体的修复建议及修正后的代码  
第 4 步：生成包含以下章节的结构化评审报告：  
  ·摘要 (Summary)：代码的功能实现、整体质量评估  
  ·发现项 (Findings)：按严重程度分组（错误优先，其次是警告和提示）  
  ·评分 (Score)：1-10 分，并给出简短的评分依据  
  ·前三大建议 (Top 3 Recommendations)：最具影响力的改进点
```

## 模式4：Inversion（反向提问）

代理本质上想要立即猜测和生成。反向提问模式颠覆了这种动态。用户不再驱动提示，代理执行，而是代理充当访谈员。

反向提问依赖于明确且不可协商的门控指令（如“在所有阶段完成前不要开始构建”）来强制代理先收集上下文。

它会依次提问结构化的问题，等待你的回答后再进入下一阶段。代理拒绝综合最终输出，直到它对你的需求和部署约束有完整了解。

要了解具体操作，请查看项目规划技能。这里的关键元素是严格的相位分配和明确的门槛提示，阻止代理在收集完所有用户答案前合成最终方案：

```
skills/project-planner/SKILL.md  
名称：project-planner（项目规划师）  
描述：通过结构化提问收集需求，并据此规划新的软件项目。在用户表达“我想构建”、“帮我规划”、“设计一个系统”或“开始一个新项目”时使用。  
  
##元数据：  
  
模式：反向提问 (inversion)  
交互：多轮对话 (multi-turn)  
你正在进行一场结构化的需求访谈。在所有阶段完成之前，严禁开始构建或设计。  
第一阶段 ：问题发现（每次只提一个问题，并等待回答）  
按顺序询问以下问题，不得跳过：  
Q1：“这个项目为用户解决了什么问题？”  
Q2：“主要用户是谁？他们的技术水平如何？”  
Q3：“预期的规模是多少？（每日活跃用户、数据量、请求频率）”  
  
第二阶段：技术约束（仅在第一阶段完全回答后进行）  
Q4：“你将使用什么样的部署环境？”  
Q5：“你对技术栈有任何具体要求或偏好吗？”  
Q6：“有哪些不可逾越的硬性要求？（延迟、可用性、合规性、预算）”  
  
第三阶段：综合汇总（仅在所有问题回答后进行）  
加载 assets/plan-template.md 以获取输出格式。  
  
使用收集到的需求填补模板中的每个章节。  
向用户展示完整的规划方案。  
询问：“这份规划是否准确记录了你的需求？有哪些地方需要修改？”  
根据反馈进行迭代，直到用户确认。
```

## 模式五：Pipeline（管道模式）

对于复杂的任务，你不能跳过步骤或忽视指令。管道模式强制执行严格的顺序工作流程，并设有硬检查点。

指令本身就是工作流程的定义。通过实现显式的钻石门条件（例如从文档字符串生成到最终汇编前要求用户批准），流水线确保代理无法绕过复杂任务，呈现未经验证的最终结果。

该模式利用所有可选目录，仅在特定步骤拉入不同的参考文件和模板，保持上下文窗口的干净。

在这个文档流水线示例中，请注意显式的门条件。在用户确认前一步生成的文档字符串之前，代理明确禁止进入组装阶段：

```
skills/doc-pipeline/SKILL.md  
名称：doc-pipeline（文档流水线）  
描述：通过多步流水线从 Python 源代码生成 API 文档。当用户要求为模块编写文档、生成 API 文档或根据代码创建文档时使用。  
  
##元数据：  
模式：管道模式 (pipeline)  
  
步骤：共4步  
  
你正在运行一个文档生成流水线。请按顺序执行每个步骤。严禁跳过步骤，若当前步骤失败，不得继续后续操作。  
  
第一步：解析与清单整理  
分析用户的 Python 代码，提取所有公共类（Classes）、函数（Functions）和常量（Constants）。将提取结果以清单形式展示。询问：“这是你想要记录的完整公共 API 吗？”  
  
第二步：生成文档字符串 (Docstrings)  
针对每个缺失文档字符串的函数：  
加载 references/docstring-style.md 以获取要求的格式。  
严格遵循风格指南生成文档字符串。  
将生成的每个文档字符串展示给用户以供确认。  
在用户确认之前，不得进入第三步。  
  
第三步：汇编文档  
加载 assets/api-doc-template.md 以获取输出结构。将所有类、函数和文档字符串汇编成一份完整的 API 参考文档。  
  
第四步：质量检查  
对照 references/quality-checklist.md 进行审查：  
  ·每个公共符号是否均已记录？  
  ·每个参数是否都有类型说明和描述？  
  ·每个函数是否至少包含一个使用示例？  
报告检查结果。在展示最终文档前修复所有发现的问题。
```

## 模式混合

这些模式并不互斥，管道模式可以在结尾加入反转提问步骤，用以再次核查自己的工作。生成器可以在一开始依赖反演来收集必要的变量，然后再填充模板。

别再试图把复杂脆弱的指令塞进一个系统提示符里。拆解你的工作流程，应用正确的结构模式，打造可靠的代理。
