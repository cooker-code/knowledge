Anthropic 是靠卖 token 赚钱的。

你使用它的工具搭的 Agent 系统越复杂，调用链越长，token 消耗越多，他们赚得越多。按这个逻辑，他们应该疯狂鼓励所有人上 Multi-Agent——框架越重越好，Agent 越多越好。

但最近，他们跟几十个实际落地团队合作之后，主动发了一篇文章，自己给行业浇了一盆冷水：

> **"For most use cases, optimizing single LLM calls with retrieval and contextual examples is sufficient."**
>
> 对大多数场景，优化单次 LLM 调用就够了，不需要 Agent。

这话不是某个"反 AI"的评论者说的，不是竞争对手说的，是那个靠 Agent 赚钱最多的公司说的。

这说明什么？

说明他们在合作中见过太多**因为架构太重而翻车的案例**——重到他们宁愿少赚一些 token 费，也要把这件事说清楚。

**AI 落地最大的坑，不是模型不够强，而是架构太重了。**

---

## 造 Agent 的公司，劝你别用 Agent

这篇文章不长，但有一个细节值得注意：它不是技术教程，不是框架文档，而是 Anthropic 把自己和几十个落地团队合作的经验提炼出来的**方法论**。

它真正想说的不是"怎么搭 Agent"，而是——**你得先搞清楚自己的需求在哪一层，然后停在那一层，不要往上爬了。**

这个判断框架，就是文章的核心：六层架构阶梯。

---

## 六层架构阶梯：从"一次调用"到"自主 Agent"

Anthropic 在文章中给出了一个清晰的架构分层。我根据自己的实践理解重新整理成一个"阶梯"，从下往上依次递进。

**核心原则：从底层开始，逼不得已才往上爬一层。**

---

### 第一层：增强型 LLM（Augmented LLM）

**一句话概括**：一次模型调用，但给它配好检索、工具和记忆。

这是最简单、最稳定、最便宜的方案。但有一个反直觉的地方——

**"简单"不是退而求其次，而是把复杂度从系统架构转移到了 prompt 设计和上下文管理上。** 这个转移通常是合算的，因为 prompt 好改，架构难改；prompt 出了问题一眼就能看见，中间层出了问题你根本不知道该查哪里。

**真实场景举例**：

* • 客服机器人回答常见问题：用 RAG 挂上知识库，一次调用搞定
* • 代码生成助手：把相关文件内容塞进上下文，一次调用生成代码
* • 文档摘要/结构化提取：单次调用配合精确的 prompt 模板

**很多人低估这一层的原因**，是觉得单次调用"不够体现技术含量"。但换个角度想：单次调用能做好，说明你真的理解了任务的本质——你知道模型需要什么信息、需要多少信息、怎么表达问题才能拿到好答案。这比搭一堆 Agent 把问题"分包出去"要难得多。

Anthropic 在文章里说，**这一层配合好的检索和 few-shot 示例，可以覆盖"大多数场景"**。这不是在贬低其他层，而是在说：在确认单次调用真的不够用之前，不要往上爬。

---

### 第二层：Prompt 链（Prompt Chaining）

**一句话概括**：把任务拆成多个固定步骤，前一步的输出喂给下一步。

注意关键词：**固定步骤**。这意味着你在写代码的时候，就已经知道第一步干什么、第二步干什么、第三步干什么。步骤之间的顺序不变，路径不变。

这一层和第五层（编排者模式）最容易被混淆，但区别非常清楚：**Prompt 链的结构是你写死的，编排者的结构是模型运行时决定的。** 如果你的任务步骤是确定的，用 Prompt 链就够了，完全不需要编排者。

**真实场景举例**：

* • 长文写作：大纲 → 质量检查 → 正文 → 润色
* • 数据分析报告：理解结构 → 生成代码 → 运行获取结果 → 文字报告
* • 代码重构：分析 → 生成方案 → 执行 → 对比差异

**Anthropic 强调在步骤之间加 gate（门控检查）**，这个设计被严重低估了。

gate 的本质是：**在错误扩散前把它拦住**。大纲生成完了，先让模型自评一下质量，不合格就打回重做，合格了才继续写正文。这和软件工程里 CI pipeline 的逻辑完全一样——你不会等代码部署到生产环境才跑测试。

还有一个容易忽略的细节：**gate 不一定要用 LLM 来做**。很多时候，一个简单的规则检查就够了——输出长度是否合理、格式是否符合、关键字段是否存在。规则检查比 LLM 调用便宜几个数量级，能用规则的地方不要用 LLM。

---

### 第三层：路由（Routing）

**一句话概括**：先分类，再走不同的处理流程。

这一层解决的核心问题是：**不同类型的输入需要不同的处理方式**，而且这些类型你是提前知道的。

**真实场景举例**：

* • 智能客服：用户消息进来，先分类——退款问题走退款流程（可能需要查订单系统），技术问题走技术文档检索，情绪激动的转人工
* • 代码问题诊断：报错信息进来，先判断类型——编译错误、运行时错误、性能问题，分别走不同的诊断链
* • 多语言翻译：检测输入语言，路由到对应的翻译 prompt（不同语言对的最优 prompt 差异很大）

路由还有一个隐藏优势：**你可以把简单问题路由给小模型**。比如客服场景里，80% 的问题是"我的订单到哪了"这种简单查询，用 Haiku 就够了，没必要每个请求都调 Opus。光这一条，就能把成本砍掉一半以上。

路由的另一个深层价值是**可观测性**。当你把"退款问题"和"技术问题"分成两条独立的链路，你就能分别衡量每条链路的表现，单独优化、单独监控。如果是一条大而全的链路处理所有问题，你根本不知道哪类问题表现差、差在哪里。

**到这里，请注意——前三层已经覆盖了绝大多数的业务场景。**

Anthropic 在文章里的判断是：**对大多数用例，前三层足够了**。这不是谦虚，而是他们跟几十个落地团队合作后得出的结论。现实是，很多团队直接跳到了第五层甚至第六层，绕过了真正应该花时间的地方。

---

### 第四层：并行化（Parallelization）

**一句话概括**：同一个任务交给多个 LLM 同时处理，再合并结果。

并行化有两种模式：

**模式一：投票型。** 同一个问题扔给多个不同 prompt 的模型，各自独立回答，然后投票或综合。Anthropic 在文章里举的例子是代码安全审查——三个模型分别从不同角度扫描漏洞，最后综合判断。

**模式二：分片型。** 一个大任务拆成互不依赖的小块，分给多个 worker 同时处理。比如要审查一个 10 万行的代码库，按模块拆开，5 个 worker 同时扫描。

**投票型有一个经常被忽略的前提**：投票能提高准确率，依赖的是"每个模型的失败是独立的"这个假设。如果三个模型用同一个 base model、处理同一类问题，它们很可能在同样的地方犯同样的错——这种情况下投票给你的是"错误的自信"，不是更高的准确率。

所以**投票型并行化真正有效的场景**，是让模型从不同角度审视同一个问题（而不是用不同 prompt 问同一个问题）：一个模型检查安全性，一个检查性能，一个检查可读性——这三者的失败模式才是真正独立的。

**并行化的成本是线性增长的**——三路投票就是三倍 token，用之前先想清楚有没有必要。

---

### 第五层：编排者-执行者（Orchestrator-Workers）

**一句话概括**：一个中心 LLM 动态分析任务，决定要派哪些 worker 去干活，然后合并结果。

跟第四层最大的区别在于"**动态**"这两个字。第四层的任务分片是你写代码时确定好的，第五层的任务分配是模型自己决定的。

**真实场景举例**：

* • 复杂代码重构：编排者分析需要改动的范围，动态决定要修改哪些文件，把每个文件的修改分配给不同的 worker
* • 研究报告生成：编排者分析主题，决定需要从哪几个角度搜集信息，分别派 worker 去做不同方向的调研
* • 多步骤数据处理：编排者解析用户的自然语言需求，判断需要调用哪些 API，组合出处理流程

这一层开始有 Multi-Agent 的味道了。但有一个关键区别必须说清楚——**这里的 worker 是无状态的**。编排者给 worker 一个明确的小任务，worker 完成后返回结果，就结束了。Worker 不需要记住之前做了什么，不需要跟其他 worker 交流。

这跟很多人想象中的"多个 Agent 互相讨论、自主协作"是完全不同的。

**无状态为什么是个优势而不是局限？** 无状态的 worker 本质上是纯函数——给定相同的输入，始终产出相同类型的输出。纯函数可以独立测试、可以独立替换、失败了可以直接重试。一旦 worker 有了状态，就引入了"这个 worker 此刻处于什么状态"的问题，调试难度指数级上升。

编排者-执行者模式的核心约束是：**编排者的"动态"指的是它决定做什么，而不是它如何做**。编排者自身的决策逻辑应该是可预测的——给定同类任务，它应该做出同类决策。如果编排者的行为本身也难以预测，你已经在向第六层滑落了。

---

### 第六层：自主 Agent（Autonomous Agent）

**一句话概括**：感知环境 → 自主规划 → 执行行动 → 获取反馈 → 调整计划，循环往复。

这是阶梯的顶层，也是**最贵、最不稳定、最难调试**的一层。

Anthropic 在文章中对 Agent 的定义非常克制：只有当任务的路径是开放的、不可预知的，需要模型自己判断下一步做什么的时候，才该用 Agent。

**真正适合 Agent 的场景其实很少**：

* • 开放域的安全渗透测试（不知道会遇到什么系统、什么漏洞）
* • 复杂的端到端软件开发（从需求分析到部署，中间的路径高度不确定）
* • 科研探索（假设驱动的实验，每一步取决于上一步的实验结果）

注意这些场景的共同特征——**人做这些事的时候也很难提前规划好每一步。** 如果你的场景是人能提前写出 SOP 的，那大概率不需要 Agent。

**一个简单的自测方法**：把你的任务用决策树画出来。如果你能画出来，哪怕树很大很复杂，都不需要 Agent——用 Prompt 链或编排者就够了。**只有当你根本画不出来、因为你不知道会遇到什么分支时**，才真的需要 Agent。

大多数被冠以"Agent"名字的产品，其实是可以画出决策树的。它们真正需要的是第四或第五层。

## 一张表看清：你该停在哪一层

| 层级 | 架构 | 单次请求 token 量级 | 成本倍率 | 适用场景 | 典型延迟 |
| --- | --- | --- | --- | --- | --- |
| 1 | 增强 LLM | 1-4K | 1x | 问答、摘要、简单生成 | <3s |
| 2 | Prompt 链 | 4-15K | 2-4x | 固定步骤的多阶段任务 | 5-15s |
| 3 | 路由 | 2-8K | 1-2x | 多类型输入的分流处理 | 3-8s |
| 4 | 并行化 | 10-30K | 3-5x | 高准确率要求、大规模处理 | 5-10s |
| 5 | 编排-执行 | 20-80K | 5-15x | 动态任务分解 | 15-60s |
| 6 | 自主 Agent | 50-500K+ | 10-100x | 开放域、路径不可预知 | 分钟级 |

这个表格里最值得关注的是**成本倍率**那一列。

从第一层到第六层，token 消耗不是线性增长，而是指数级跳跃的。看一下数字：第五层的成本是第一层的 5-15 倍，第六层是 10-100 倍。这还只是单次请求。如果你的系统每天处理几万次请求，这个倍率的差距会直接反映在月账单上。

**更关键的问题是**：这些额外的成本买来了什么？

往上爬每一层，你买到的不是"更好的效果"，而是"更大的灵活性"——处理更复杂、更不确定任务的能力。如果你的任务结构是确定的（绝大多数任务都是），那这部分灵活性对你毫无用处，你只是在为用不到的东西付钱。

**每往上爬一层，先问自己：我的任务真的需要这一层才有的那种灵活性吗？**

## Agent 系统的两个致命代价

成本还不是最可怕的。更可怕的是**错误累积**。

在单次调用中，模型犯错了，你看到的就是一个错误的输出，改一下 prompt 就好了。

但在 Agent 系统中，错误是会**传播和放大**的。

Agent A 对任务的理解偏了 10%，它把这个偏了 10% 的结果传给 Agent B，Agent B 在错误的基础上又偏了 15%。最后的结果可能跟你的预期南辕北辙，而你回溯的时候根本不知道到底是哪一环出了问题。

这里有一个数学问题值得想清楚：**如果单次 LLM 调用有 95% 的准确率，串联 5 个 Agent 后，整体准确率是多少？**

不是 95%，而是 0.95⁵ ≈ **77%**。

77% 意味着每 4 次里就有 1 次输出是有问题的。对于很多业务场景，这是不可接受的。而且这还是假设每个 Agent 的失败是独立的——如果错误会传播和放大（实际上往往如此），最终准确率会更低。

这就是 Anthropic 说的：**复杂度是有代价的。** 不是说复杂的系统一定不好，而是你得确保增加的复杂度所带来的收益，大于它引入的风险和成本。

大多数时候，答案是否定的。

## "但 Multi-Agent 才是未来啊！"

我知道你在想什么。

社区里到处都在说 Multi-Agent，论文一篇接一篇，框架一个接一个。AutoGen、CrewAI、LangGraph……不搭个多智能体系统好像就落伍了。

说句不好听的：**社区的声量不等于工程的真相。**

学术界研究 Multi-Agent 是合理的——推进前沿本来就是学术界的工作。但作为工程师，你的 KPI 不是发论文，是把功能稳定地交付到线上。

Anthropic 原文里有一句话，我觉得被大多数人忽略了：

> **"The most successful implementations we've seen use simple, composable patterns — not complex frameworks."**
>
> 我们见过的最成功的实现，用的都是简单、可组合的模式——不是复杂的框架。

注意"composable"这个词。Anthropic 不是说你不能组合多个调用，而是说组合的方式应该是**简单的、可拆的、每一层都能独立测试的**。不是用一个大框架把所有东西糊在一起。

Multi-Agent 会不会是未来？可能是。但那个未来的前提是模型的可靠性还要再上几个台阶——当每个 Agent 的输出准确率能稳定在 99% 以上的时候，错误累积的问题才不那么致命。

**在模型可靠性真正达到 99%+ 之前，每多串联一个 Agent，你就在主动稀释整个系统的可靠性。**

## 给已经"搭重了"的团队：迁移路径

如果你的团队已经搭了一个复杂的 Agent 系统，怎么办？全部推翻重来不现实，但可以逐步降级。

**Step 1：画出你的调用链路图。**

把每一次 LLM 调用都标出来。调用了几次？每次传了什么上下文？哪些是串行的？哪些是并行的？

**Step 2：用三个问题审视每个节点。**

* • 这个 Agent 能被一段好的 prompt 替代吗？→ 合并掉
* • 这个 Agent 的输入是固定的还是动态的？→ 固定的就不需要编排者
* • 去掉这个 Agent，整体效果会下降多少？→ 实测，而不是猜

**Step 3：从叶子节点开始砍。**

不要一上来就动核心逻辑。先砍最外围的、最容易验证的节点。比如一个"反思 Agent"，试试去掉它，看效果有没有明显下降。很多时候你会发现——根本没区别。

**Step 4：给每个 Agent 设一个"存在的必要性"门槛。**

不是"它有没有用"，而是"它带来的增量效果，是否值得它引入的成本和复杂度"。这个门槛因业务而异，但一定要量化，不能凭感觉。

> 砍系统最大的心理障碍是"万一效果变差怎么办"。Anthropic 的建议是：**实测，不要猜**。很多团队实测之后发现——去掉某个 Agent，效果根本没变。那个 Agent 存在的理由，只是当初"感觉应该有这一层"。

## 三个问题判断法：你的需求该停在哪一层

最后给一个可以反复使用的决策框架。拿到任何一个 AI 功能需求，问三个问题：

**问题一：这个任务能不能拆成固定步骤？**

能 → 停在第一到第三层。用增强 LLM、Prompt 链或路由就够了。

**问题二：子任务是提前能预知的，还是需要动态决定的？**

提前预知 → 用并行化。动态决定 → 才考虑编排者-执行者模式。

**问题三：任务路径是开放的、根本没法提前规划吗？**

是 → 才用自主 Agent。不是 → 回到前面的层级。

三个问题问完，你会发现——**你以为需要 Agent 的那些场景，80% 在第二个问题就停下来了。**

---

## 写在最后

我越来越觉得，**AI 工程的核心能力不是搭复杂系统的能力，而是克制的能力。**

知道什么时候该用 Agent，什么时候一次调用就够了——这比你会用几个框架重要得多。

Anthropic 的这篇文章之所以值得反复读，不是因为它给了什么新技术，而是它给了一个**判断框架**。有了这个框架，下次产品经理跟你说"我们搞个 AI Agent 吧"的时候，你能冷静地回一句：

> "Agent 是第六层。你确定你的需求不是第二层能解决的事？"

**能克制住往上爬的冲动，才是真正的架构能力。**
