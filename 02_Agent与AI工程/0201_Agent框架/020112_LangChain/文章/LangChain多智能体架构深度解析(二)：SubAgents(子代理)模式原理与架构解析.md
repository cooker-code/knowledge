在上一篇[LangChain多智能体架构深度解析(一)：SubAgents模式的设计哲学与实战指南](https://mp.weixin.qq.com/s?__biz=MzYzOTM4NzA1NA==&mid=2247484118&idx=1&sn=18a44d29b2a2fe09b651899643dc9874&scene=21#wechat_redirect)，讲了一个实际案例。在设计 AI Agent 系统时，经常会遇到一个问题：

> 如果一个AI既要能安排日历事件，又要能发送邮件，该如何设计系统结构？

   一种简单的方法是把所有能力都放进**一个Agent**中，但随着系统能力越来越多，这种方式很快会遇到问题：

* Prompt 会变得非常长
* 工具越来越多
* 推理过程变复杂
* 系统越来越难维护

   为了解决这个问题，LangChain在多智能体体系中提出了一种经典架构：**Subagents（子代理模式）**。

简单来说：通过一个主代理协调多个专业子代理完成任务。

一、什么是 Subagents

Subagents 可以理解为一种 **“AI 专家团队”** 架构，整体结构如下：

```
User  ↓Main Agent（主代理）  ↓Subagents（多个子代理）
```

在这个结构中：主代理负责整体调度，它主要承担几个职责：

* 理解用户问题
* 判断需要什么能力
* 选择合适的子代理
* 汇总子代理结果
* 生成最终回答

而**子代理只负责某一个专业能力，**例如个人助理系统可能包含这些子代理：

* calendar\_agent：负责处理日程安排、可用性检查和事件管理
* mail\_agent：负责管理通信、起草邮件和发送通知

主代理就像一个“项目经理”，负责协调这些专家完成任务。

二、Subagents 的运行流程

    在 Subagents 架构中，一个典型流程如下：

```
用户提问   ↓主代理理解问题   ↓调用子代理   ↓子代理返回结果   ↓主代理整理答案   ↓返回给用户
```

例如用户提问：“帮我下周二下午2点为设计团队张三和李四安排一个1小时的会议，并给他们发送一封关于审阅新样稿的邮件提醒。”

主代理判断：

这个问题需要安排日程和发送邮件，于是先后调用calendar\_a**gent、mail\_agent**，执行完之后，再将结果整理成自然语言回答给用户，整个过程中，用户只会看到 **主代理的回复**。

三、子代理通常以 Tool 的形式存在

在LangChain中，Subagents 通常会被封装成 **Tool，系统结构大致可以理解为：**

```
Main Agent   │   ├── schedule_agent_tool   ├── mail_agent_tool
```

当主代理判断需要某种能力时，就会调用对应的Tool，而这个Tool的内部，其实就是一个**子代理(Subagent)，这种设计方式有一个明显优点：系统可以按照能力模块进行拆分，不同子代理可以独立开发、维护和调试，从而提升系统的可扩展性。**

**四、Subagents 的三个关键特点**

**1：子代理不会直接和用户交互**

   在 Subagents 模式中：只有主代理和用户进行对话，系统结构可以理解为：

```
User  ↔  Main Agent            ↓        Subagents
```

子代理只负责返回结果，最终回答始终由主代理生成。

## 2：子代理通常是无状态的

   大多数情况下，Subagent是 **无状态(stateless)**的,也就是说，每次调用子代理时都会重新开始一个新的上下文，对话历史通常由**主代理统一管理，这样做有几个好处：**

* 避免上下文不断膨胀
* 子代理逻辑更简单
* 子代理更容易复用

## 3：子代理只接收必要信息

   主代理在调用子代理时，可以控制输入信息，只传递与任务相关的上下文，例如：

* 用户问题
* 相关数据
* 必要参数

这样可以避免：

* 传递过多无关信息
* Token 消耗过高
* 模型产生不必要的推理错误

在 LangChain中，这种设计也被称为 **Context Engineering（上下文工程）**。

五、Subagents模式设计要点

    在设计 Subagents模式架构时，有几个关键问题需要我们提前考虑：

**1：子代理能力划分**

每个子代理都应该有清晰的职责，例如：

* 日程管理等
* 邮件操作等

能力划分越清晰，主代理越容易选择合适的子代理。

**2：子代理需要什么上下文**

调用子代理时需要决定传递哪些信息：

* 如果上下文太少，子代理可能无法完成任务。
* 如果上下文太多，又会增加 Token 消耗。

因此通常只传递**与当前任务相关的信息**。

3：子代理输出格式

子代理返回的结果最好结构清晰，例如：

* JSON 数据

这样主代理在整合结果时会更加稳定。

# 六、总结

   Subagents是LangChain 多智能体体系中的一种经典架构模式，其核心思想是：通过一个主代理协调多个专业子代理完成复杂任务，在这种设计中：

* 主代理负责决策与调度
* 子代理负责具体能力
* 子代理通常以 Tool 的形式存在
* 系统上下文由主代理统一管理

从而让 AI 系统形成类似 **“专家团队”** 的结构，更好地处理复杂任务。
