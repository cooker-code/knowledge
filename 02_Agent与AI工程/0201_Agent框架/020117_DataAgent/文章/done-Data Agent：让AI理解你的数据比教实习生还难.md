最近被一个问题折腾得不轻——**Data Agent老是“错误“的理解用户问题、生成错误的查询分析！**虽然你进行了“较好”的意图理解，任务拆分与规划、较“详细"的tools的使用说明书和MCP，但准确率就是不高，你想想，要是你的BI系统有一半概率给老板错误的报表，那不得被骂死？

LLM为啥搞不定你的分析操作呢？

咱先说下，在BI分析场景中，**Data Agent作为连接用户的自然语言需求与底层数据的智能中****介**，其核心挑战在于如何准确理解和映射用户意图到具体的数据查询操作。

这事儿其实挺有意思。LLM就像一个特别聪明但没见过世面的实习生——它语言能力超强，啥都能聊，但你让它去查数据库？对不起，它连表结构都看不懂。

**问题的根源在于：LLM天生就不是为处理结构化数据设计的**

怎么说呢，LLM的训练数据主要是文本，它习惯了"从左到右"地阅读。但数据库表格呢？行列交错，还有各种关联关系。这就像让一个只会看小说的人去读Excel表格，能不懵吗？

之前在做的Data Agent时就踩过坑。用户问"上个月的销售额是多少"，结果LLM生成的SQL里，把 order\_amount （订单金额）当成了销售额，实际上销售额应该是 quantity \* unit\_price 的总和。更离谱的是，它还忘了加时间过滤条件，查出来的是所有历史数据的总和。

坦白讲，这种"幻觉"问题在实际项目里特别常见。TigerData团队做过统计，**42%的SQL查询要么遗漏了关键过滤条件，要么误解了表关****系**。这数据一点都不夸张，我们当时的失败率也差不多这个水平。

元数据：让AI理解数据的"说明书"

后来我发现，解决这个问题的关键是**给LLM配一份详细的"数据说明书****"**——也就是元数据。

这就像你招了个新人，不能上来就让他干活，得先给他讲清楚：咱们这个表是干啥的，每个字段啥意思，哪些表之间有关联，业务上怎么定义指标的……

### 1. 表结构信息：最基础但最重要

首先得让LLM知道你有哪些表、哪些字段。但光有名字不够，还得加注释：

```
SQL-- 不要这样（太简陋）CREATE TABLE so (    oid INT,    cid INT,    amt DECIMAL);-- 要这样（信息丰富）CREATE TABLE sales_order (    order_id INT COMMENT '订单唯一标识',    customer_id INT COMMENT '客户ID，关联customer表',    amount DECIMAL COMMENT '订单金额（含税）') COMMENT = '销售订单主表，记录所有客户订单信息';
```

这个改动看起来简单，但效果立竿见影。我们当时就是加了这些注释后，SQL生成准确率提升了15%左右。

### 2. 业务语义映射：让AI听懂"人话"

但是吧，光有表结构还不够。用户说的是"销售额"、"活跃用户"这些业务术语，数据库里存的是 amount 、 login\_count 这些技术字段。中间这个鸿沟咋填？

我的做法是建立一个业务术语映射表：

```
JSON{  "销售额": {    "technical_field": "sales_amount",    "calculation": "SUM(quantity * unit_price)",    "synonyms": ["营收", "收入", "GMV"],    "note": "不含退款金额"  },  "活跃用户": {    "definition": "30天内至少登录1次",    "sql_condition": "DATEDIFF(NOW(), last_login_date) <= 30"  }}
```

这个映射表特别有用。有一次产品经理问"这个月的GMV是多少"，要是没有这个映射，LLM根本不知道GMV是啥。有了映射后，它立马就能理解GMV就是销售额，还知道具体怎么算。

### 3. 表间关系：别让AI瞎JOIN

还有个大坑就是表关联。你要是不告诉LLM表之间怎么关联，它就会瞎JOIN，要么漏了关联条件造成笛卡尔积，要么用错了字段。

我们的做法是把常用的关联路径都定义清楚：

```
Yamlrelationships:  - name: "订单关联客户"    from: sales_order    to: customer      join_type: LEFT JOIN    condition: "sales_order.customer_id = customer.id"    usage: "查询订单的客户信息时使用"
```

### 4. 统计特征与数据样本

统计特征元数据提供了数据本身的概况，帮助它-LLM了解数据的分布、质量和规模，从而更好地评估查询的可行性和结果的可靠性。比如表的行数、列的基数等信息，让LLM了解数据规模，有助于其优化查询策略。对于数值型或日期型数据，提供其行数、最小值、最大值、平均值、中位数等统计信息，帮助LLM理解数据的分布特征。

```
{  "data_quality": {    "sales_amount": {      "completeness": 0.98,      "min_value": 0,      "max_value": 1000000,      "outlier_threshold": 500000,      "null_percentage": 0.02    }  }}
```

一些行业内的案例

最有意思的是，Oracle和AWS的方案跟我们不谋而合。Oracle的Select AI会自动提取数据库的所有元数据来增强Prompt；AWS则建议把数据按业务域划分，每个域都配上详细的DDL和业务规则说明。

* **Oracle Auto****nomous Database Select AI**：Oracle的内置SQL生成器自动提取数据库的模式元数据（表名、列名、注释等）来构建增强提示  ，用户无需手动准备元数据。该系统专注于将自然语言转换为SQL，并通过“增强的提示”指导LLM生成与数据库模式一致的查询。

* **AWS企业级Text2****SQL方案**：AWS建议将复杂的企业数据划分为“数据域”，按查询意图映射到相应域，并“增强数据域的SQL DDL定义，添加表、列以及业务规则的描述” 。通过聚焦于相关域并附加业务约束（如计算公式和过滤规则），提示规模更小、模型推理更准确。这种**领域化和元数据增强**的组合策略，有效提高了复杂环境下NL2SQL的准确率。

* **ChatBI等对话式B****I系统**：多家企业在构建对话BI系统时都强调构建**知识库**来存储元数据。京东ChatBI项目中，数据中台资产（包含元数据、指标SQL、业务文档）被汇总为模型输入 。用户提问时，系统先识别出相关的时间、指标、维度等，并结合用户权限、部门等背景信息；对业务性查询则直接在知识库中检索答案。若是数据分析问题，则调用后台接口，并最终由LLM生成可视化报告。这种架构体现了“**先有知识再让模型作答**”的原则，即用结构化的元数据和先验洞察引导LLM推理。

* **TigerData语义目录实验**：TigerData团队尝试为PostgreSQL数据库构建“自描述”模式，为每个表、列、函数甚至示例查询和业务规则都写入自然语言描述 。结果显示，使用这些生成的语义描述后，自动生成SQL的准确率提升了27%（从58%提升到86%） 。他们证实：\*\*“没有额外上下文，LLM无法做出正确决策。”\*\*只有在给出富含语义的描述后，模型才能理解列名背后的含义  。

* **IBM LLM元数据增强**：IBM用LLM自动为企业数据生成业务相关的元数据。IBM Knowledge Catalog的新功能可以根据数据元素生成**表描述、列描述、标签和扩展的列名**，并将其映射到组织本体或词汇表 。例如，字段ACT\_END\_DT经元数据增强后，标签会写明它是“实际通讯结束日期” 。这些自动化的元数据生成技术帮助分析师快速理解数据含义，提高了数据可发现性和分析效率。

## 一些实用的小技巧

基于近期的经验，分享几个实用的tips：

**1. 别一次性塞太多信息**

不要把整个数据库的元数据都扔给LLM，它会懵的。用RAG的思路，根据用户问题动态检索相关的元数据。比如用户问销售相关的，就只给销售域的表结构。

**2. 提供统计信息很有用**

告诉LLM每个表大概有多少数据，字段的取值范围是什么。这能帮它判断查询的合理性：

```
JSON{  "sales_amount": {    "min": 0,    "max": 1000000,    "avg": 5000,    "null_rate": 0.02  }}
```

**3. 业务术语与行****业黑话一定要写进去，并且给一些案例case**

**4. 建立反馈循环**

**把生成错误的日志记录下来，分析原因，持续优化元数据。我们就是这样一点点把准确率提上去的。**
