读完论文 *Externalization in LLM Agents*，最值得带走的不是“记忆、技能、协议、Harness 很重要”这个表层结论，而是它背后的判断：LLM Agent 的可靠性提升，正在从“让模型在内部想得更好”，转向“把任务重新表征成模型更容易完成的形式”。

这篇论文把这种变化称为 `externalization`，可以译作“外化”或“认知外化”。它借用了 Donald Norman 的认知工件思想：购物清单没有增强人的生物记忆，但它把“回忆要买什么”变成了“看清单识别什么还没买”；地图没有让人脑里的导航模块变强，但它把不可见的空间关系变成了可见结构。

关键不是能力被简单放大，而是任务被改写。

## Agent 的进展，不只是模型参数的进展

过去几年，我们习惯把 Agent 能力理解成模型能力的外延：模型更大、推理更强、上下文更长、工具调用更准，Agent 就更强。这当然没有错，但解释不了一个现实：很多生产级 Agent 的可靠性提升，并不是来自换了一个更大的模型，而是来自模型周围的运行时变得更严密。

一个软件工程 Agent 要完成真实任务，光会写代码不够。它要知道项目结构、历史决策、当前分支状态、测试命令、团队约定、可写路径、哪些操作需要审批、失败后如何恢复、什么时候停止。这些东西如果全靠 Prompt 临时塞进去，系统会脆弱、冗长、不可审计，也很难跨会话延续。

论文用“从权重到上下文再到 Harness”描述这个迁移：

* 权重阶段：能力主要被看作模型参数中的知识、推理习惯和行为倾向。
* 上下文阶段：能力转移到 Prompt、RAG、示例、CoT、ReAct 等运行时输入组织方式中。
* Harness 阶段：能力进一步进入持久记忆、技能文件、协议接口、沙箱、权限、观测、反馈和评估循环。

这不是否定模型能力，而是重新划分能力的位置。模型仍是推理核心，但它不再是智能的唯一载体。Agent 的有效能力来自模型与外部认知基础设施的耦合。

## 记忆不是存更多，而是让正确的过去在当下可用

论文把记忆定义为“跨时间状态的外化”。这个定义比“长期记忆”更准确。记忆系统的目标不是保存一切，而是把当前决策需要的历史状态变成可检索、可判断、可治理的形式。

它区分了四类记忆：工作上下文、情景经验、语义知识和个性化记忆。

工作上下文保存当前任务状态、打开文件、检查点、部分计划，让任务中断后能恢复；情景经验保存过去运行、决策、失败和结果，用于避免重复错误；语义知识保存稳定事实、项目约定和领域规则；个性化记忆保存用户、团队和环境偏好。

这里有一个容易被忽略的点：记忆质量的核心不是容量，而是表征。一个系统可以存下全部历史，却在关键时刻检索出错误片段；这时“记住了”并不等于“可用”。反过来，一个较小但索引、摘要、更新和选择策略更好的记忆系统，可能更可靠。

## 技能不是工具，而是过程专长的外化

工具告诉 Agent 可以做什么；技能告诉 Agent 一类任务应该怎么做。

论文对技能的定义很清晰：技能外化的是程序性专长。一个好技能不只是几条说明，而应包含三种内容：

* 操作程序：步骤顺序、依赖关系、停止条件。
* 决策启发：分支时先试什么、什么时候退回、证据到什么程度算够。
* 规范约束：哪些边界不能越过，哪些验证必须完成，哪些输出才算合格。

这解释了为什么真实 Agent 系统里 `AGENTS.md`、`SKILL.md`、项目 SOP、发布流程、代码审查清单会越来越重要。它们不是 Prompt 装饰，而是在降低执行方差。

没有技能时，模型每次都要重新综合一套做法。即使它原则上知道怎么做，也可能漏步骤、顺序错、过早结束、局部正确但全局失败。有了技能，模型的任务从“重新发明流程”变成“判断是否适用、加载相关部分、按约束执行并处理例外”。

## 协议不是胶水，而是交互结构的外化

Agent 最容易被低估的困难，是交互。一个模型要调用工具、委托另一个 Agent、向用户展示状态、处理错误和权限，它不仅要“知道想做什么”，还要知道格式是什么、字段怎么填、状态怎么转移、失败怎么表达、谁有权批准。

如果这些都靠自然语言临场推断，外部行动就会变成脆弱的 Prompt-following。协议的价值是把交互从“自由文本协商”变成“结构化契约”。

因此，MCP 这类工具协议不只是让工具“接得上”，而是把工具调用的语法和能力发现变成可检查表面；A2A、ACP、ANP 这类 Agent-Agent 协议不只是发消息，而是把委派、身份、状态和交接显式化；AG-UI、A2UI 这类 Agent-User 协议则把用户界面和执行状态事件流协议化。

协议的深层作用，是把模型输出边界收窄成可治理动作。模型仍然负责判断意图，但不必每次重造交互格式。正确行为更容易发生，错误行为更容易被拦截。

## Harness 是 Agent 的认知环境

论文最重要的概念是 Harness。它不是记忆、技能、协议之外的第四个模块，而是让这些外化结构共同运作的运行时环境。

一个成熟 Harness 至少要处理六个维度：Agent loop 与控制流、沙箱与执行隔离、人类监督与审批门、可观测性与结构化反馈、配置权限与策略编码、上下文预算管理。

这套东西的共同作用，是给模型安排一个更可控的认知空间。模型不是在无限开放环境中自由生成，而是在一个有文件边界、有权限、有可调用接口、有状态快照、有观察日志、有审批点的环境里工作。

所以 Harness 不只是基础设施，而是 Agent 的认知环境。它决定模型能看见什么、能记住什么、能调用什么、必须如何行动、出了错怎样被发现和修复。

## 真正的工程问题：哪些能力应该留在模型里

这篇论文没有落入“模型 vs 基础设施”的二分。它更接近一个系统分区问题：给定某个负担，它应该留在模型参数里，放进上下文里，还是外化成记忆、技能、协议或 Harness 策略？

稳定通用能力，例如语言理解、常识和通用推理，更适合留在模型内部。快速变化的状态、需要审计和回滚的规则、需要跨 Agent 复用的流程、需要确定性约束的权限和 schema，更适合外化。

这引出一个比“尽量外化”更重要的原则：最小充分外化。

外化不是越多越好。过多记忆会污染上下文，过长技能会让模型失去全局目标，过多协议和工具会把选择本身变成负担。好的 Harness 应该降低模型的决策难度，而不是制造第二个需要模型理解的复杂系统。

## 结语

这篇论文的最终判断可以压缩成一句话：Agent 的下一层竞争，不只是模型聪明程度，而是认知基础设施的组织质量。

记忆让过去以合适形式进入当下；技能让经验以可复用过程进入执行；协议让意图以可治理契约进入外部世界；Harness 则把这些结构组织成一个能持续行动、可观察、可恢复、可约束的运行时。

这并不意味着模型不重要。恰恰相反，强模型让外化结构更有价值，因为它能在这些结构提供的表征中做更好的判断。但如果没有外部认知基础设施，模型能力会反复浪费在记忆、重构流程、猜接口、处理脆弱上下文这些低杠杆工作上。

未来的 Agent 工程，核心问题会越来越像系统设计，而不是单轮 Prompt 设计：哪些能力内化，哪些能力外化；哪些状态持久化，哪些状态丢弃；哪些流程写成技能，哪些继续临场推理；哪些交互必须协议化；哪些决策需要人类 gate；哪些失败要进入反馈循环。

更好的 Agent 不只是更好的推理器，而是组织得更好的认知系统。
