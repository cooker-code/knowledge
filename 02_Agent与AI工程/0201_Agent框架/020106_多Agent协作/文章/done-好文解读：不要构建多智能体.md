# 要说2025年哪个Agent的效果最亮眼，Claude Code应该能排进前三，在Claude Code带火了Sub Agent的情况下，多智能体基本上已经成为了构建强大智能体系统的标配，但依然有公司建议不要构建多智能体，今天这篇文章是Cognition公司（就是推出全自动AI软件工程师Devin，后又收购了Windsurf的那家公司）于今年6月份发布的这篇文章，虽然看标题非常反直觉，但内容还是有真知灼见的，非常值得一读，读完后你就会发现，作者其实说的是构建多智能体时，你应该遵循的基本原则。

先说核心结论：**凡是不满足「共享上下文」和「避免隐含决策冲突」的多智能体架构，大部分场景下都应该默认排除掉，而倾向用单线程线性 agent 或少数严格受控的子任务 agent。**

## 上下文工程原则

我们将逐步阐述以下原则：

1. 共享上下文
2. 操作隐含决策

**为什么要思考原则？**

HTML 于1993年推出。2013年，Facebook 向世界发布了 React。如今已是2025年，React（及其衍生品）主导了开发者构建网站和应用的方式。为什么？因为 React 不仅仅是一个编写代码的脚手架，它更是一种哲学。通过使用 React，你接受了以响应式和模块化的模式构建应用，如今人们认为这是标准要求，但对早期的 Web 开发者来说，这并非显而易见。

在大语言模型（LLM）和构建 AI 智能体的时代，我们感觉自己仍在摆弄原始的 HTML 和 CSS，并试图将它们拼凑起来以创造良好的体验。除了某些绝对基础的内容外，尚未有一种构建智能体的单一方法成为标准。

> 在某些情况下，OpenAI 的 https://github.com/openai/swarm 和微软的 https://github.com/microsoft/autogen 等库积极推动了一些我认为是错误的智能体构建理念。即，使用多智能体架构，我将解释原因。

话虽如此，如果你是智能体构建的新手，有很多资源可以教你如何搭建基本的脚手架。但是，当涉及到构建严肃的生产级应用时，情况就完全不同了。

## 构建长期运行智能体的理论

让我们从可靠性开始谈起。当智能体需要在长时间运行中保持可靠并维持连贯的对话时，你必须采取某些措施来控制错误累积的可能性。否则，如果不小心，事情会迅速崩溃。可靠性的核心在于上下文工程（Context Engineering）。

**上下文工程**

在2025年，市面上的模型极其智能。但即使是最聪明的人，如果没有被明确告知任务的上下文，也无法有效完成工作。“提示工程”（Prompt engineering）一词最初是指为 LLM 聊天机器人以理想格式编写任务所需的努力。“上下文工程”（Context engineering）则是其进阶版。它关乎在一个动态系统中自动完成这项工作。这需要更多的细微差别，实际上也是构建 AI 智能体的工程师的首要任务。

以一种常见的智能体为例。该智能体

1. 将其工作分解为多个部分
2. 启动子智能体来处理这些部分
3. 最终将这些结果合并

这是一种诱人的架构，尤其是在你所处的任务领域包含多个并行组件时。然而，它非常脆弱。关键的故障点在于：

> 假设你的**任务**是“构建一个 Flappy Bird 克隆版”。该任务被分解为**子任务1**：“构建一个带有绿色管道和碰撞箱的移动游戏背景”和**子任务2**：“构建一个可以上下移动的小鸟”。
>
> 结果，子智能体1误解了子任务，开始构建一个看起来像超级马里奥兄弟的背景。子智能体2为你构建了一只鸟，但它看起来不像游戏素材，移动方式也与 Flappy Bird 中的完全不同。现在，最终的智能体面临着一个不理想的任务：将这两个因沟通失误产生的结果结合起来。

这可能看起来有些牵强，但大多数现实世界的任务都包含许多层细微差别，所有这些都有可能被误解。你可能会认为一个简单的解决方案就是将原始任务作为上下文也复制给子智能体，这样它们就不会误解自己的子任务。但请记住，在真实的生产系统中，对话很可能是多轮次的，智能体可能需要进行一些工具调用来决定如何分解任务，任何数量的细节都可能影响对任务的解读。

> **原则1**: 共享上下文，并共享完整的智能体轨迹，而不仅仅是单条消息

让我们再次修改我们的智能体，这次确保每个智能体都能获得之前智能体的上下文。

不幸的是，我们仍未完全脱离困境。当你给智能体分配相同的 Flappy Bird 克隆任务时，这次你可能会得到一只鸟和一个背景，但它们的视觉风格完全不同。子智能体1和子智能体2无法看到对方在做什么，因此他们的工作最终彼此不一致。

子智能体1所采取的行动和子智能体2所采取的行动都是基于未事先明确规定的冲突假设。

> **原则2**: 行动暗含决策，而矛盾的决策会带来糟糕的结果

```
注：此处作者的意思是，在使用多智能体的情况下，每个子智能体都可能会包含一些隐含决策（就是没有明确说明的决策），拿Flappy Bird来说，就是风格、交互方式、技术路线等，一旦这些隐含决策不一致，在合成阶段，就会造成灾难性的后果。所谓隐含决策，大致可以理解为任务没有明说，但子智能体基于模型内置知识做做出的决策。
```

我认为，原则1和原则2至关重要，几乎不值得违反，因此你应默认排除任何不遵守这些原则的智能体架构。你可能会觉得这很受限制，但实际上，你仍然可以为你的智能体探索多种不同的架构。

## 应用这些原则

如果你是一名智能体构建者，请确保你的智能体的每一个操作都受到系统其他部分所做所有相关决策的上下文指导。理想情况下，每个操作都能看到其他所有内容。不幸的是，由于上下文窗口有限以及实际的权衡，这并不总是可行的，你可能需要根据你所追求的可靠性水平，决定愿意承担多大的复杂性。

当你思考如何设计你的智能体以避免冲突的决策时，这里有一些现实世界的例子供你思考：

**Claude Code 子智能体**

截至2025年6月，Claude Code 是一个会生成子任务的智能体的例子。然而，它从不会与子任务智能体并行工作，而且子任务智能体通常只负责回答一个问题，而不是编写任何代码。为什么？因为子任务智能体缺乏主智能体的上下文，而这些上下文是执行超出回答明确定义问题之外的任何任务所必需的。如果它们要运行多个并行的子智能体，可能会给出相互冲突的回应，从而导致我们之前例子中看到的可靠性问题。在这种情况下，拥有子智能体的好处是，所有子智能体的调查工作都不需要保留在主智能体的历史记录中，从而在耗尽上下文之前允许更长的轨迹。Claude Code 的设计者们采取了一种刻意简化的方法。

**编辑应用模型**

在2024年，许多模型在编辑代码方面表现很差。编码智能体、IDE、应用构建器等（包括 Devin）中的一种常见做法是使用“编辑应用模型”。其核心思想是，让一个小模型根据你想要的更改的 Markdown 说明重写整个文件，实际上比让一个大模型输出格式正确的 diff 更可靠。因此，构建者们会让大模型输出代码编辑的 Markdown 说明，然后将这些说明输入给小模型来实际重写文件。然而，这些系统仍然会非常容易出错。例如，小模型经常会由于指令中最细微的歧义而误解大模型的指令，从而做出错误的编辑。如今，编辑决策和应用更多地由单个模型在一次操作中完成。

**多智能体**

如果我们真的想从系统中获得并行性，你可能会想到让决策者之间“交谈”并解决问题。

这正是我们在（理想情况下）意见不合时人类的做法。如果工程师A的代码与工程师B的代码发生合并冲突，正确的协议是讨论差异并达成共识。然而，如今的智能体还无法以比单个智能体更可靠的方式进行这种长上下文的主动对话。人类在向彼此传达最重要的知识方面非常高效，但这种效率需要非同寻常的智慧。

自 ChatGPT 推出后不久，人们就开始探索多个智能体相互交互以实现目标的想法（参考https://arxiv.org/abs/2304.03442和https://github.com/FoundationAgents/MetaGPT）。虽然我对智能体长期协作的可能性持乐观态度，但在2025年，很明显，让多个智能体协作只会导致脆弱的系统。决策过程过于分散，上下文无法在智能体之间充分共享。目前，我还没有看到有人在致力于解决这个困难的跨智能体上下文传递问题。我个人认为，随着我们让单线程智能体在与人类沟通方面变得越来越好，这个问题将会迎刃而解。当这一天到来时，它将释放出更大程度的并行性和效率。

**迈向更普适的理论**
这些关于上下文工程的观察只是我们未来可能视为构建智能体标准原则的开端。这里没有讨论到的挑战和技术还有很多。在 Cognition，智能体构建是我们思考的一个关键前沿。我们围绕这些原则构建内部工具和框架，这些原则是我们不断重新学习的，以此来强化这些理念。但我们的理论可能并不完美，我们也期望随着领域的发展，事情会发生变化，因此也需要一定的灵活性和谦逊态度。
