很多团队在做 Multi-Agent 系统时，会不自觉地往复杂的方向走——觉得模式越高级，系统越可靠。但 Anthropic 的工程师们观察到了另一个更普遍的问题：团队选模式，往往选的是"听起来更厉害的"，而不是"最适合眼前问题的"。

这篇文章梳理了他们总结的 5 种 Multi-Agent 协调模式，每种模式的工作机制、适用场景和常见失效点，以及在什么条件下应该从一种模式演进到另一种。文章最后附有一张决策速查表。

核心建议只有一句话：从能解决问题的最简单模式开始，观察它在哪里卡壳，再向更复杂的方向演进。

模式一：生成-验证（Generator-Verifier）

这是五种模式里最简单、也是部署最广泛的一种。

逻辑很直接：一个 Agent 负责生成输出，另一个 Agent 负责审核。审核通过则结束，审核不通过则把反馈返回给生成方，让它重新生成。这个循环一直持续，直到输出通过审核，或者达到最大迭代次数。

一个典型场景是客服邮件系统：生成方根据产品文档和工单内容起草回复，验证方对照知识库检查内容准确性、品牌语气，并逐条确认每个问题都有被回应。如果验证发现某个功能归属了错误的定价档位，或者某条工单问题没被回答，就把具体反馈传回生成方重新修改。

这个模式适合的场景是：**输出质量关键、评估标准可以被显式定义**。代码生成（一个 Agent 写代码，另一个写测试并运行）、事实核查、合规审核、基于评分标准的内容评估，都是典型用法。错误输出的代价越大，这个模式就越值得用。

它最常见的失效点：

验证方的质量取决于评估标准的清晰程度。如果只告诉验证方"检查输出是否够好"而没有具体标准，验证方会直接放行生成方的任何输出——营造了质量控制的假象，却没有实质。

另一个坑是迭代循环卡死：生成方如果始终无法解决验证方的反馈，系统就会来回震荡而不收敛。必须设置最大迭代次数，并配好降级策略（升级给人工、返回最佳尝试并附加说明）。

模式二：编排-子 Agent（Orchestrator-Subagent）

这个模式的核心是层级关系：一个 Agent 扮演"项目负责人"，负责规划任务、分派工作、汇总结果；子 Agent 各自承担特定职责，完成后向上汇报。

Claude Code 就采用了这个模式。主 Agent 负责写代码、编辑文件、运行命令，同时在需要搜索大型代码库或调查独立问题时，在后台派发子 Agent——子 Agent 在各自的上下文窗口里工作，返回提炼后的结论，主 Agent 的上下文始终专注在主要任务上。

一个经典用例是自动化代码评审系统：PR 到来时，系统需要检查安全漏洞、测试覆盖率、代码风格、架构一致性。每项检查相互独立、需要不同上下文、产出清晰的结论。Orchestrator 把每项检查派给专门的子 Agent，收集结果后综合输出统一的评审报告。

推荐在这个模式里使用的条件是：**任务分解清晰、子任务之间依赖最少**。Orchestrator 维持对整体目标的全局视角，子 Agent 保持对各自职责的专注。

它的短板是 Orchestrator 本身会成为信息瓶颈。如果安全子 Agent 发现了一个影响架构设计的认证漏洞，这条信息必须先经过 Orchestrator 再传给架构子 Agent——经过多次这样的中转之后，关键细节往往在摘要中被稀释或丢失。另外，如果没有显式并行化，子 Agent 是串行执行的，付出了 Multi-Agent 的 token 成本，却没能获得速度收益。

模式三：Agent 团队（Agent Teams）

当工作可以拆分成多个可以**长时间独立推进**的并行子任务时，Orchestrator-Subagent 就显得过于束缚了。这时候 Agent Teams 更合适。

与 Orchestrator-Subagent 的核心区别在于 Worker 是否持续存活。在 Orchestrator-Subagent 里，Orchestrator 为单一的有界子任务派发子 Agent，子 Agent 返回结果后就终止了。

而在 Agent Teams 里，团队成员在多个任务间保持存活，积累上下文和领域专业知识，表现随之持续提升。Coordinator 负责分配工作并收集结果，但不会在任务之间重置 Worker。

一个典型例子是将大型代码库从一个框架迁移到另一个框架。每个服务可以由一个团队成员独立迁移，包括依赖更新、代码修改、测试修复、部署配置变更。Coordinator 把每个服务分配给团队成员，每个成员自主完成整个迁移过程——对各自服务的依赖关系、测试模式、部署配置越来越熟悉，效率远高于每次从零开始的派发式执行。

这个模式的前提是子任务真正独立——这也是它最大的风险所在。团队成员自主工作，无法方便地共享中间发现，他们的输出可能存在冲突。多个成员操作同一个代码库、数据库或文件系统时，还可能出现并发写入冲突。任务分区和冲突解决机制是使用这个模式的必要前提。

模式四：消息总线（Message Bus）

随着 Agent 数量增加、交互模式变复杂，直接协调越来越难以维护。

消息总线引入了一个共享通信层：Agent 通过发布（publish）和订阅（subscribe）事件来交互。新的 Agent 只需订阅相关话题，不需要改动现有连接，就能开始接收对应的工作。

安全运营自动化系统是这个模式的理想场景：告警从多个来源涌入，一个分类 Agent 按严重性和类型做分类路由，高严重性网络告警流向网络调查 Agent，凭证相关告警流向身份分析 Agent，各调查 Agent 可能发布补充信息请求，由上下文采集 Agent 来响应，最终汇聚到响应协调 Agent 决定处置方案。

消息总线适合这个场景，因为**流程由事件驱动而非预设固定序列**，新的威胁类型出现时只需新增 Agent 而不用改写现有逻辑，各 Agent 也可以独立开发部署。

它的核心挑战是可观测性差。一个告警触发了跨 5 个 Agent 的事件级联，追踪完整链路需要完善的日志和关联机制，调试难度远高于跟踪 Orchestrator 的顺序决策。路由准确性也至关重要：如果 Router 分类错误或丢弃了一个事件，系统会悄悄失效——什么都不处理，却也从不报错。

模式五：共享状态（Shared State）

前面四种模式都有某种形式的中心调度——Orchestrator、Coordinator 或 Router。共享状态去掉了这个中间人，让各 Agent 通过一个所有人都能读写的持久化存储来协调。

没有中央 Coordinator，各 Agent 自主运行：读取存储里的相关信息，采取行动，把发现写回存储。工作流通常由一个初始化步骤（向存储里写入问题或数据集）触发，由终止条件结束——时间限制、收敛阈值，或由某个指定 Agent 判断存储中的答案已经足够充分。

研究综合系统是这个模式最典型的应用：多个 Agent 分别调查复杂问题的不同侧面——学术文献、行业报告、专利档案、新闻动态。一个 Agent 的发现可以即时被其他 Agent 看到并加以利用，无需等待 Coordinator 转发。共享存储成为一个不断演进的知识库，Agent 们彼此构建、相互启发。

共享状态还消除了单点故障：任何一个 Agent 停止，其他 Agent 继续读写，不受影响。而在 Orchestrator 和消息总线模式中，核心组件一旦宕机，系统就全部停止。

这个模式最棘手的失效点是反应性循环：Agent A 写入一个发现，Agent B 读到后写了跟进，Agent A 看到跟进再次响应……系统持续消耗 token 却无法收敛。重复工作和并发写入有工程解（加锁、版本控制、分区），但反应性循环是行为问题，必须用一等公民的方式设计终止条件：时间预算、收敛阈值（N 个周期内没有新发现），或者专门指定一个"收尾 Agent"来判断答案已经足够。把终止条件当成事后补丁的系统，往往会无限循环或在某个 Agent 上下文被填满时任意停止。

如何在模式之间做选择

五种模式的核心差异，本质上是**上下文边界的管理方式**不同。选择时可以按以下几个问题逐步推断。

**Orchestrator-Subagent vs. Agent Teams：**子任务是否需要跨多轮积累上下文？如果子任务短暂且边界清晰，用 Orchestrator-Subagent；如果 Agent 需要在多个任务周期里保持对某个领域的持续认知，用 Agent Teams。

**Orchestrator-Subagent vs. 消息总线：**工作流步骤是否可以预先确定？如果流程固定，用 Orchestrator-Subagent；如果流程由运行时事件决定（比如不知道下一条告警是什么类型）、且 Agent 生态还在持续扩展，用消息总线。当 Orchestrator 里开始堆积大量条件判断来处理各种情况，消息总线就是把路由逻辑显式化、可扩展化的时机。

**Agent Teams vs. 共享状态：**各 Agent 的工作是否互相影响？如果可以严格分区、互不相关，用 Agent Teams；如果一个 Agent 的中间发现对其他 Agent 有价值、需要实时共享，用共享状态。

**消息总线 vs. 共享状态：**工作是以离散事件形式触发并流过各阶段，还是在一个知识库上持续积累？事件驱动的流水线选消息总线；知识积累型的研究任务选共享状态。如果消息总线里的 Agent 发现自己在发布"发现成果"而不是"触发动作"，共享状态会是更自然的选择。另外，消息总线仍然有 Router 作为中心组件；如果消除单点故障是首要目标，共享状态能更彻底地做到去中心化。

生产系统往往会组合使用多种模式——比如用 Orchestrator-Subagent 管理整体流程，但在协作密集的子任务上使用共享状态；或者用消息总线做事件路由，但每种事件类型由一个 Agent 团队来处理。这些模式是构建块，不是互斥选项。

决策速查表

| 场景 | 推荐模式 |
| --- | --- |
| 输出质量关键，评估标准可以显式定义 | 生成-验证 |
| 任务分解清晰，子任务有界且依赖少 | 编排-子 Agent |
| 子任务相互独立，需要长期、多步骤自主推进 | Agent 团队 |
| 事件驱动的流水线，Agent 生态持续扩张 | 消息总线 |
| 协作式研究，Agent 间实时共享发现 | 共享状态 |
| 不能有单点故障 | 共享状态 |

对大多数场景而言，推荐从 Orchestrator-Subagent 入手——它覆盖了最宽的问题范围，协调开销也最小。在实际运行中观察它在哪里卡壳，再按需演进到其他模式。
