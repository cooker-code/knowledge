## 核心问题
多 Agent 的增量来自任务隔离、并行探索、专长分工和独立验证；如果只是把一个可线性完成的任务拆成多个聊天角色，通常会增加通信成本和错误合并风险。

## 判断准则
- 优先用单 Agent；只有任务天然可分、上下文互相污染或需要独立验证时再拆。
- 模式选择按依赖关系：顺序、路由、并行、Supervisor、网络协作，不按角色数量。
- 每个子 Agent 必须有输入边界、输出契约、失败语义和合并策略。

## 认知偏差
| 常见错误认知 | 正确理解 |
|---|---|
| Agent 越多越智能 | Agent 越多越需要通信、协调和合并治理 |
| 多 Agent 是默认架构 | 多 Agent 是复杂度工具，默认应先证明单 Agent 不够 |
