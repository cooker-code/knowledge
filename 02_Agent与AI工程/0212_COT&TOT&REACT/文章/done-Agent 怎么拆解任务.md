**涌现 · 第六篇**

上一篇结尾留下了一个问题：如果单次"感知-行动"能形成闭环，那把这个闭环重复——让模型看到行动结果、再推理、再行动——是否可以处理更复杂的任务？

答案是肯定的。这正是 ReAct 框架和 Agent Loop 要做的事：把单次的预测-行动，变成可以迭代推进的循环。

但"可以处理更复杂的任务"，和"在真实环境里可靠地完成复杂任务"，是两件不同的事。这个差距，在早期 Agent 演示里几乎看不出来——在真实部署里，它会以各种方式出现。

**循环扩展了模型能完成的任务范围，但没有改变每一步的工作方式。理解这个区别，是判断 Agent 能力边界的前提。**

## **单次闭环的天花板**

Function Call 解决了一类问题：让模型能访问当下的信息、触发真实操作。但这个能力有一个内置的限制——它是单次的。

想象一个任务：帮我整理这周所有未回复的邮件，按优先级排序，然后起草回复。这个任务包含多个步骤，每一步的结果都会影响下一步的判断：需要先知道有哪些邮件，才能判断优先级；需要读过邮件内容，才能起草有针对性的回复；起草完成后，可能还需要根据上下文调整语气。

单次 Function Call 能做其中一步，但无法把这些步骤串联成一个有内在逻辑的推进过程。每一步都需要人手动触发下一轮，把上一步的结果复制进去，再等模型输出。

这不是工具调用能力的问题，而是任务结构的问题：有些任务本身就是多步骤的，后面的步骤依赖前面的结果，而且这个依赖关系在任务开始时无法完全预见。这类任务需要的不是更多工具，而是一个可以自己推进的循环。

## **ReAct：让推理和行动交替发生**

ReAct 是 2022 年一篇论文提出的框架，名字来自 Reason 和 Act 的组合。核心思路很简单：让模型在每次行动之前，先输出一段推理；行动之后，再根据结果继续推理；然后再行动。推理和行动交替，形成循环。

这和纯粹的 Function Call 有一个关键的不同。单纯的工具调用是这样的：看到任务 → 输出工具调用 → 得到结果 → 输出回复。中间没有显式的推理步骤。

ReAct 在每一步之间插入了一个"想"的环节：

**Thought（推理）：**我现在知道了什么，还缺少什么，下一步应该做什么，为什么。

**Act（行动）：**调用某个工具，用特定的参数。

**Observation（观察）：**工具返回了什么结果。

然后再回到 Thought，基于新的 Observation 更新推理，决定下一步。循环，直到任务完成。

这个结构的好处，和第三篇讲过的 chain-of-thought 逻辑相通：显式的推理步骤改变了模型预测的上下文，使它更容易在训练数据里找到高质量的问题解决模式，而不是直接跳到一个可能不合适的结论。

**ReAct 不是给模型加装了一个"思考引擎"，而是通过结构化的输出格式，把每一步的上下文拉向训练数据里"高质量的逐步推进"的模式。循环在跑，但每一个 Thought，依然是预测。**

## **Agent Loop：循环是怎么运作的**

把 ReAct 的结构放进一个持续运行的系统里，就是所谓的 Agent Loop。它的基本形态，比听起来要简单：

**AGENT LOOP 的基本结构**

**1**

接收任务目标和初始上下文（工具列表、历史信息、用户指令）

**2**

**Thought：**模型预测当前状态下的推理输出——现在知道什么、还缺什么、接下来做什么

**3**

**Act：**输出工具调用（或判断任务已完成，输出最终回复）

**4**

*宿主系统执行工具调用，把结果作为 Observation 写入上下文*

**5**

返回第 2 步，基于更新后的上下文继续循环——直到触发终止条件

紫色步骤：模型在做的（预测）。灰色步骤：外部系统在做的（执行）。终止条件由外部设定，不由模型自行判断。

这个结构有一个很重要的细节：每一轮循环，模型看到的上下文都在增长——它包含了所有之前的 Thought、Act 和 Observation。这意味着随着循环推进，模型在每一步都能利用前面积累的信息，而不是从零开始。这是它能处理多步骤任务的根本原因。

但这个机制也带来了一个内置的约束：上下文窗口是有限的。循环越长，上下文越大，直到撑满窗口的上限。这不只是工程问题——它意味着 Agent 处理真正长任务的能力，存在一个结构性的天花板。

## **认知插入：演示为什么总是成功，部署为什么总会失败**

2023 年前后，Agent 演示是 AI 领域最流行的内容形式之一。一个模型，给定一个模糊的任务，自己搜索、自己分析、自己写代码、自己执行——然后交出一个完整的结果。看起来像是一个真正在"独立思考"的系统。

这些演示里，Agent 通常表现出色。然后同样的系统被部署到真实环境里——开始出各种问题。这不是个案，是一个系统性的规律。理解它，需要看清演示和真实部署之间有哪些结构性的差异。

**差异一：任务是被设计过的，还是真实提出的。**

演示里的任务，通常由演示者本人设计——它被设计成工具能覆盖所需的信息、任务边界足够清晰、成功标准可以直观展示。这不是造假，而是一种自然的选择偏差：你不会拿一个不确定能跑通的任务去做演示。

真实用户提出的任务，不会为 Agent 的能力边界做任何让步。它可能需要的工具没有被定义，成功标准是模糊的，包含着 Agent 根本无法处理的隐含假设。

**差异二：误差在受控环境里不累积，在真实环境里会。**

Agent Loop 的每一步都是独立的预测。单步的小误差——选错了工具、参数稍微偏了、对结果的解读不够准确——在受控演示里影响有限，因为后续步骤通常还在预期的轨道上。

真实环境里，第一步的小偏差会改变第二步的上下文，第二步的偏差进一步影响第三步。这种误差累积没有自动纠正机制——模型在每一步都基于当前上下文预测，它不会回头检查前面的判断是否走偏了。

结果是：演示里三步能完成的任务，在真实环境里可能在第二步就开始偏离，然后用四步、五步继续沿着偏差方向推进，最终交出一个看起来完整但从根本上就不对的结果。

**差异三：停止条件在演示里是隐含的，在真实部署里必须显式设定。**

在演示里，演示者知道任务完成了什么样。这个判断有时候通过系统提示传递给模型，有时候只是因为任务结构足够清晰，模型恰好在对的地方输出了"Final Answer"。

真实部署里，不是每个任务都有清晰的完成信号。模型没有独立的"任务完成检测器"——它判断"该停了"，是因为当前上下文下，训练数据里这类模式通常会在这里总结然后结束。这个判断在任务结构和训练分布高度吻合时是准确的；不吻合时，它可能在任务还没完成的地方停下来，也可能在任务已经完成之后继续做多余的操作。

**演示成功，是因为任务、工具、环境三者是匹配的。部署失败，是因为真实世界不为这种匹配做让步。这个落差不是 Agent 的 bug，而是基于预测的循环在真实环境里工作时的系统性特征。**

这不是在说 Agent 不可用。它的意思更准确：Agent 在受控的、边界清晰的、工具覆盖完整的任务上可以非常可靠；在开放的、边界模糊的、需要判断何时停止的任务上，需要人在外部托底——设定清晰的终止条件、监控循环是否走偏、在关键节点介入判断。

## **循环扩展了什么，没有扩展什么**

把这一篇的内容汇聚成一张图景：Agent Loop 相比单次预测，真正扩展的是什么，边界依然在哪里。

|  |  |
| --- | --- |
| **循环真正扩展的**  可处理任务的复杂度——多步骤、依赖中间结果、需要迭代调整的任务。可以在过程中积累信息，而不是只依赖初始上下文。 | **循环没有扩展的**  每一步的工作方式——仍然是预测。误差累积的问题——甚至比单步更严重。需要人托底的部分——停止条件、异常判断、后果承担。 |

还有一件事值得在这里点出来。ReAct 循环里的 Thought 步骤——那段看起来像在"推理"的文字——在视觉上非常像一个真正在思考的系统："我现在知道了 X，还缺少 Y，所以下一步应该做 Z。"这段话读起来像是内省，像是真正的自我监控。

但它和 chain-of-thought 的本质一样：模型在预测"在这个上下文下，一个正在解决这类问题的系统通常会输出什么样的推理步骤"。它不是真正的内省，是对"推理输出"的模式合成。区别在边界处会显现：真正的内省可以检测到自己的推理走偏了；预测出来的 Thought，会在走偏之后继续以同样的确信度推进，下一步仍然会输出一个听起来合理的推理，沿着已经偏离的路径继续走下去。

**循环让模型看起来更像在"独立思考"。这种视觉上的相似性，是演示特别有说服力的原因，也是部署期望特别容易被高估的原因。**

ReAct 和 Agent Loop 是真实的进展：它们让语言模型能处理需要多步骤推进的复杂任务，让每一步的行动结果可以反馈回推理过程，让"感知-推理-行动"形成可以迭代的循环。

但循环没有改变的，同样清晰：**每一步仍然是预测，误差仍然会累积，停止条件仍然需要从外部设定，异常判断仍然需要人来兜底。**演示和部署之间的落差，不会随着循环变长而自动消失。

下一个自然的问题是：既然 Agent Loop 这么复杂，有没有办法把这个复杂性封装起来，让开发者不需要关心每一步的细节，只需要描述任务，框架来处理其余的事？有。那就是 LangChain、LangGraph 这类框架要做的事。**但封装带来便利的同时，也带来了一件事：你开始看不见里面发生了什么。**
