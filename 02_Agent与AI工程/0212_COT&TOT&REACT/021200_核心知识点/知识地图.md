# 021200_COT&TOT&REACT知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 02_Agent与AI工程 / 0212_COT&TOT&REACT |
| 解决的问题 | 判断复杂任务应使用线性推理、多路径搜索，还是推理-行动-观察闭环 |
| 典型输入 | 用户任务、已有上下文、候选工具、输出格式、验证标准 |
| 典型产出 | 推理路径、候选方案、工具调用轨迹、观察结果、最终答案和验证结论 |
| 用户阅读目标 | 读到推理范式文章时，先判断它补的是任务拆解、路径搜索、外部反馈还是工程迁移边界 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| R00 任务判定 | 任务是否真的需要推理范式 | 简单任务不需要 CoT/ToT/ReAct，先看复杂度、外部依赖和验证要求 | 只讲“让模型更聪明”的文章降权 |
| R01 线性拆解 | 能否沿一条清晰步骤求解 | CoT 适合多步但路径明确的问题 | 有步骤、输入输出和错误传播边界才吸收 |
| R02 多路径搜索 | 是否需要探索多个候选思路 | ToT 适合路径不确定、需要试探和回溯的问题 | 必须说明候选生成、评估和剪枝方式 |
| R03 外部反馈 | 是否必须查工具或环境结果 | ReAct 的核心是把推理和真实观察绑定 | 有工具结果、观察反馈和循环退出条件才吸收 |
| R04 验证收敛 | 如何判断答案可信 | 推理轨迹不是证据，最终仍要靠输出契约、工具结果或评测 | 缺验证标准的案例只能略读 |
| R05 工程迁移 | Prompt 范式什么时候迁到 Tool Calling / Harness | 显式 ReAct 文本格式脆弱，生产系统应把状态、工具和门禁结构化 | 讨论状态机、结构化工具调用、Trace 的文章和 Agent 框架互链 |

## 待补缺口

| 流程节点 | 缺什么 | 后续文章进入时怎么判断 |
|---|---|---|
| R00 任务判定 | 缺少本地任务级对照实验 | 看文章是否给出普通 Prompt、CoT、ToT、ReAct 的对照结果 |
| R02 多路径搜索 | 缺少 ToT 评估器和剪枝策略的工程细节 | 看是否说明候选数量、评分标准、回溯成本 |
| R04 验证收敛 | 缺少推理轨迹与最终正确性的验证关系 | 看是否有评测集、工具结果、人工标注或单元测试 |
| R05 工程迁移 | 缺少结构化 tool calling 替代显式 ReAct 的本地实验 | 看是否能复现格式偏移、工具误调、状态丢失等失败模式 |

## 使用方式

- 新文章先归入 R00-R05 的一个节点，再判断是补充、冲突、不同层次还是降权。
- 只讲 Prompt 模板而没有任务边界、验证标准或失败场景的文章，不进入核心知识点。
- ReAct 相关文章必须区分“设计思想”和“显式文本协议”；后者不能直接当生产实现准则。
