# 0700_工程与架构知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 07_工程与架构 |
| 解决的问题 | 先定义软件系统工程流程，再把文章吸收到后端、前端、工程保障等具体路线 |
| 典型输入 | 本地工程文章、已有 AGENTS、已有核心知识点、后续新增资料 |
| 典型产出 | 技术路线、流程节点、判断准则、风险边界、待补证据 |
| 用户阅读目标 | 读到一篇工程文章时，先判断它属于系统实现、前端交付、后端服务、工程保障，还是应该迁出到其他一级类目 |

## 宏观流程

| 节点 | 节点问题 | 主要落点 | 文章吸收规则 |
|---|---|---|---|
| E00 边界确认 | 系统职责、输入输出、用户场景、非功能约束和不做什么是什么 | 后端架构、前端工程 | 没有边界和约束的“架构”文章降权 |
| E01 实现组织 | 模块、分层、依赖、状态、数据访问和扩展点如何组织 | 后端架构、前端工程 | 按技术本体进入 Python、Java、SpringBoot、React、Vue、Nuxt 等路线 |
| E02 交互契约 | API、协议、网关、幂等、认证和错误语义如何设计 | 后端架构、工程实践与质量保障 | 区分框架内接口写法和跨系统接口契约治理 |
| E03 质量验证 | 单测、集成测试、契约测试、回放、压测和验收标准如何覆盖 | 工程实践与质量保障、前端工程化 | 必须有可复现输入、断言、失败信号或回归动作 |
| E04 运行保障 | 日志、指标、链路、告警、部署、回滚、容量和故障处置如何闭环 | 工程实践与质量保障 | 运行文章要落到证据、排障路径和回滚动作 |
| E05 复盘沉淀 | 缺陷、事故、性能瓶颈和架构偏差如何变成可复用准则 | 各路线核心知识点 | 不扩大文章池，优先沉淀到具体路线 |

## 当前已吸收知识点

| 流程节点 | 知识点 | 关系 | 来源 |
|---|---|---|---|
| E01 | Java 与 Spring Boot 需要拆开维护，避免语言机制和应用框架互相污染 | 纠偏 | 本轮 SpringBoot 拆分 |
| E01 | FastAPI/Pydantic 属于 Python 后端路线，不应留在工程顶层池 | 纠偏 | 本轮顶层文章池迁移 |
| E02 | 前后端分离接口规范属于接口契约与网关，不是前端或后端单方文章 | 补充 | 本轮接口文章迁移 |
| E03 | AI 测试平台主问题是测试质量，不是 FastAPI 后端框架 | 纠偏 | 本轮测试文章迁移 |
| E05 | 顶层文章池会掩盖技术本体，应删除并强制进入具体路线 | 规则沉淀 | 本轮目录治理 |

## 待补缺口

| 缺口 | 后续文章进入时怎么判断 |
|---|---|
| 后端路线之间的横向对比 | 看文章是否能说明 Python / Java / SpringBoot / Node 在同一问题下的不同边界 |
| SpringBoot 生产治理 | 看文章是否补安全、测试、观测、配置、部署和回滚 |
| 前端与后端接口协作 | 看文章是否给出契约、错误语义、联调、Mock、契约测试和版本兼容 |
| 工程保障与具体框架的边界 | 框架内实现留在对应路线，跨系统质量/安全/部署/观测进入 `0703` |

## 使用方式

- 新文章不要放到 `07_工程与架构/文章`。
- 先按技术本体进入 `0701`、`0702` 或 `0703` 的具体路线。
- 如果主问题是数据分析、调度、数据库、智能问数或电脑工具，直接迁出 `07_工程与架构`。
