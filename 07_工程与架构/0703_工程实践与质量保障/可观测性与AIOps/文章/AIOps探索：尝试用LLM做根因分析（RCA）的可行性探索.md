这两天我看了一篇文章，是关于做根因分析的算法，整了一堆公式，而且还涉及图数据库、流计算任务，甚至调参等等。看起来确实很牛逼，但对我这种非科班出身的人来说，实在有点懵。

后来我在想一件事：我们真的非要“自己造一个根因分析算法”吗？

如果换个思路，其实完全可以用：LLM + RAG（检索增强生成），在工程上把这件事做得又快又稳。

核心思想一句话总结就是：

> 不要让 LLM 去大海捞针，  
> 我们先用工程手段把“针”捞出来，  
> 再把这几根“针”喂给 LLM，让它去判断哪根是“根因”。

整体方案可以概括为：  
**RAG（负责“找对信息”） + CoT（负责“推理因果”）**

## 第一步：数据清洗与模板化（决定成败的 80% 工时）

**这一步不是可选项，而是整个方案能否成立的前提。**

### 1. 为什么原始日志不能直接进LLM？

因为日志里 90% 都是“对分析没意义、但对 Token 很致命”的东西：

* 用户ID
* 订单号
* IP
* 时间戳
* 各种随机数字

对 LLM 来说：

```
```
User1001 loginUser1002 login
```

是两个完全不同的token序列，但对人类来说，它们的语义是100% 一样的。
```

如果不做处理，结果就是：

* Token被大量浪费
* 相同语义被拆成无数个“孤立事件”
* 后续的相似度检索（RAG）基本失效

### 2. 怎么做？（不用发明新算法）

直接用**成熟的日志模板化算法**即可：

* Drain
* Spell
* IPLoM

这些都是**纯Python实现**，几分钟就能跑起来。

#### 示例

```
```
原始日志： ERROR: Connection timeout to 10.0.0.1 after 3000ms模板化后： ERROR: Connection timeout to <*> after <*> ms
```
```

### 3. 这一步和 RAG 的关系是什么？

**这里其实已经在为RAG做准备了。模板化之后，每一条日志就变成了：**

* 稳定的语义单元（Template）
* 可被向量化
* 可被统计、聚合、检索

**落地动作：**

* 对日志模板生成embedding
* 存入向量数据库（Milvus / pgvector / FAISS）
* 额外存储：模板ID、服务名、时间、级别等元信息

到这里为止，RAG的 “Retrieval数据源” 已经准备好了。

## 第二步：极简异常检测（目的只有一个：缩小问题空间）

很多根因分析系统一上来就搞：

* 全链路拓扑
* 因果图
* 概率推断

但实际上，**80%的线上故障根因，都出现在“异常最早、变化最大”的地方**。

### 我们的目标不是“算因果概率”，而是：找到“不正常的时间段 + 不正常的日志模式”

### 1. 一个非常工程化的做法

以5分钟为一个时间窗口：

1）统计当前5分钟内，每个日志模板的出现次数

2）统计过去24小时内，同一时间段的历史平均值

3）应用非常简单的规则：出现次数暴涨（比如 >10 倍）或者历史从未出现过（新模板）

### 2. 输出是什么？

假设最终你只筛选出了**20条异常日志模板**。

注意这个点非常重要：从“上百万条日志” → “20条嫌疑模板”。这一步，本质上就是**RAG中的第一次粗筛检索**，而且是用传统工程方法完成的，稳定、可控、可解释。

## 第三步：真正的RAG —— 构建“可推理的上下文”

很多人说自己在用RAG，其实只是：把一堆文本拼进Prompt。但在根因分析里，**你给LLM的不是“资料”，而是“案发现场”**。

### 1. RAG在这里到底检索了什么？

检索的不是“知识库文章”，而是：

* 当前时间窗口内的异常日志模板
* 与这些模板语义相近的历史故障日志（可选）
* 相关服务的基础依赖关系
* 同一时间段的关键指标（CPU、QPS、延迟、错误率）

这才是**真正和根因分析相关的上下文**。

### 2. Prompt设计（这是整个方案的核心）

示例：

```
你是一个拥有10年经验的SRE专家，现在发生了一起线上故障，请帮我分析最可能的根因。  
【故障现象】时间：2023-10-27 10:05订单服务P99延迟从20ms上升至2s错误率从0.1%上升至5%  
【异常日志摘要（已按时间排序）】[10:05:01] ERROR: Connection timeout to <*> after <*> ms (Template ID: T_882)[10:05:02] WARN: Redis connection pool exhausted, waiting... (Template ID: T_103)[10:05:05] ERROR: Payment service unavailable (Template ID: T_552)[10:05:10] ERROR: Connection timeout to <*> after <*> ms (Template ID: T_882)  
【系统组件依赖】订单服务 -> 支付服务订单服务 -> Redis  
请按以下步骤进行思维链推理：1. 找出最早出现的异常信号2. 结合依赖关系判断是否存在级联故障3. 给出最可能的根因4. 给出具体、可执行的排查建议
```

注意这里做了几件非常“反算法”的事：

* 明确告诉模型 **要按步骤思考（CoT）**
* 把时间顺序、依赖关系显式给出
* 不让它“自由发挥”，而是**约束推理路径**

## 第四步：LLM推理（让模型做它最擅长的事）

当你把问题空间压缩到这个程度之后，LLM的优势才会真正体现出来。它并不是在“算公式”，而是在做：

* 常识推理
* 经验匹配
* 因果排序

比如：

* 它知道**connection pool exhausted 往往是更早的信号**
* 它知道**Redis挂了，下游服务一定会超时**
* 它能根据时间戳，判断哪个是“源头”，哪个是“被波及”

最终给出的结论可能是：

> 最早出现的异常是Redis连接池耗尽，  
> 这导致订单服务在访问Redis和支付服务时发生大量超时，  
> 从而引发延迟和错误率的整体上升。

并且还能给出**工程上可执行的建议**：

* 检查Redis maxclients配置
* 查看是否有连接泄漏
* 排查同时间段是否有流量突增或慢查询

## 最后总结一下

这套方案的本质不是：“用LLM替代传统根因分析算法”，而是：用工程手段做信息压缩，用LLM做因果判断。

你不需要复杂的数学背景，也不需要维护一整套难以调优的推理系统。

只要记住一句话：RAG负责把问题缩小到“人也能看懂的规模”，LLM负责把“人类经验”规模化。
