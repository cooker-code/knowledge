其实做根因分析大多时候我们不需要做全链路拓扑、复杂因果图、概率推断以及多模态关联，因为线上超过95%（甚至99%）以上的故障并不复杂。在真实生产环境里，我越来越坚定一个判断：大多数线上问题，根因就藏在「最早异常、变化最大的那一小撮日志里」。

所以在任何LLM、因果推断、Agent介入之前，我都会强制要求系统先完成一步：极简异常检测（目标只有一个：缩小问题空间）

## 一、为什么我要“反对”一上来就做因果推断？

先讲一个非常现实的问题。你现在有的是什么？

* 每分钟几万到几十万条日志
* 上百个服务
* 成千上万种日志模板
* 大量噪声（重试、超时、告警风暴）

如果你把这些**原始日志**直接丢给LLM，让它“分析根因”，结果通常只有两种：Token直接爆炸、模型给你一堆“听起来很合理，但不可验证”的废话。这不是模型不行，而是输入空间太大。

所以我给这一步定的目标非常单一、非常冷酷：不求找根因，只求把问题空间缩到人类和LLM都能处理的规模。

## 二、核心思想：异常 ≈ 变化最大 + 出现最早

这一步我不谈什么复杂算法，只用一句话概括：异常日志，通常具备两个特征：① 在当前时间窗口里“突然变多”；②历史上几乎没出现过。

注意：我们不是在做精确异常检测，而是在做“嫌疑人筛选”。

## 三、一个可以直接落地的工程化方案

下面是一个我在多个系统里反复用过、非常稳定的方案。

### 1️⃣ 时间窗口切分：5 分钟

为什么是 5 分钟？

* 1 分钟：波动太大，噪声多
* 10 分钟以上：异常被平均掉
* **5 分钟是一个非常工程友好的折中**

我们定义：

* **当前窗口**：最近 5 分钟
* **对照窗口**：过去24小时中，相同时间段的历史数据

比如现在是19:30–19:35，对照的是过去24小时内所有xx:30–xx:35的日志

### 2️⃣ 前提条件：日志模板必须先做好

这里必须说一句大实话：没有日志模板，一切 RCA 都是空谈。

日志模板的作用只有一个：把“无限变化的文本”压缩成“有限的模式空间”。

### 3️⃣ 当前窗口统计（5 分钟）

在当前 5 分钟内，对每个日志模板做一件极其简单的事：

```
(template_id) -> count
```

例如：

| 模板 ID | 描述 | 5 分钟出现次数 |
| --- | --- | --- |
| T\_001 | DB connection timeout | 2 |
| T\_017 | User login failed | 1,240 |
| T\_089 | Cache miss | 32 |

### 4️⃣ 历史对照统计（过去 24 小时）

对同一个模板，统计历史同时间段的**平均值**或**中位数**：

```
```
(template_id) -> historical_avg
```

例如：
```

| 模板 ID | 历史均值 |
| --- | --- |
| T\_001 | 1.8 |
| T\_017 | 80 |
| T\_089 | 30 |

### 5️⃣ 极简异常规则（非常重要）

这里我**刻意不用复杂模型**，只用三条规则：

#### 规则一：出现次数暴涨

```
current_count > historical_avg * K
```

* K 通常取 **5~10**
* 不是为了精确，是为了**宁可多抓，不要漏抓**

例如：

* 历史 80 → 当前 1240
* 放大 15 倍，直接入选

规则二：历史从未出现过（新模板）

```
historical_avg == 0 && current_count >= threshold
```

* threshold 可以是 3、5
* **新模板极其危险**，通常意味着：新代码路径、新异常分支、未覆盖的错误场景。

#### 规则三（可选）：持续多个窗口

如果你想再稳一点，可以加一条：连续N个5分钟窗口都异常。但注意：不要在这一步追求“准确率”。

四、这本质上就是一次“工程化RAG粗筛”

如果你从RAG的角度看，这一步本质是：第一次检索（Recall），而且是用传统工程方法完成的。

它有几个巨大优势：① 稳定：不受模型波动影响；② 可控：阈值、窗口、规则全在你手里；③ 可解释：每一条异常都能说清楚“为什么被选中”。

LLM真正该做的是下一步：对这N条模板结合上下游日志做语义归纳、因果推理、根因总结。

## 五、一个很重要但常被忽略的认知

很多人会问：“这么简单的方法，真的够吗？”我的回答通常是：如果这一步都做不好，后面的所有‘智能分析’都是在噪声上做文章。

根因分析不是比谁算法复杂，而是比谁：更早缩小问题空间和更少把LLM浪费在无关信息上。

## 六、总结一句话

**不要一上来就让LLM当侦探。先用最土、最硬的工程方法，把嫌疑人名单缩到 20个以内。后面的事情，才值得用“智能”。**
