### 来源

### 核心判断

测试文章的价值不在“生成脚本更快”，而在测试资产是否能闭环：风险识别、用例设计、自动化执行、环境隔离、报告归因、质量门禁和缺陷回流。

| 环节 | 准则 |
|---|---|
| 用例设计 | 决策表适合多条件业务规则，要先规范条件、动作、规则和优先级 |
| AI 生成用例 | 必须补断言、测试数据、依赖隔离、环境配置和可重复执行标准 |
| 流量回放 | 录制流量要脱敏、归一化、基线对照、差异解释，不能只重放请求 |
| E2E 自动化 | Playwright/Groovy 等动态脚本要有权限边界、执行沙箱和失败截图/日志 |
| 压测 | 并发数字不等于容量模型，要有指标、退出条件、资源边界和故障注入 |
| 报告平台 | 报告要服务失败归因、趋势判断和门禁，而不是只展示通过率 |

### AI 测试降权规则

| 文章表现 | 处理 |
|---|---|
| 只说“AI 一键生成测试” | 降权，除非给出输入 IR、断言、依赖隔离和验收方式 |
| 只列 Skills/工具清单 | 降权为工具锚点，不写核心知识点 |
| 只展示平台页面 | 降权，除非说明测试资产、权限、报告和门禁流程 |
| 有 HAR/IR/Case JSON/fixture/断言生成 | 可吸收为自动化用例生成链路 |
| 有真实故障、压测、云故障复盘 | 优先吸收为质量基础设施准则 |
