决策表是一种把“复杂业务规则”系统化、结构化的测试设计方法，用一张表清晰地描述\*\*条件（inputs/facts）× 动作（actions/outcomes）\*\*之间的对应关系，使缺失规则、冲突行为、优先级错误等问题变得一目了然。

这份文档面向：

* 希望更系统化编写测试用例的测试工程师、QA、业务分析师
* 正在构建规则密集系统（风控、优惠、定价、物流、权限等）的产品与开发同学
* 想快速从“业务描述 → 可执行测试”的初学者

目标是让你掌握一套能直接落地的\*\*“决策表测试设计工作流”\*\*，并提供示例、检查清单与可复用模板。你可以把决策表看作是：“在动手写用例前，先把规则讲清楚的那张核心真相表”。

# 1. 决策表能解决什么问题？

许多场景存在大量组合式业务规则，例如：

* 多种优惠叠加与互斥
* 运费、配送方式 eligibility（可用性）
* 复杂的 KYC/AML 校验
* Feature Flag / 灰度发布策略
* 权限控制（AuthZ）
* UI 状态 / 按钮可用性
* 重试（retries）、超时、并发控制策略

这些规则在真实项目里常出现：

* **不完整**

  ：某些输入组合没有被定义（代码里要么走默认分支，要么抛异常）。
* **不一致**

  ：多个规则同时满足，但动作互相冲突（前端允许、后端拒绝）。
* **优先级模糊**

  ：靠“代码写在前面的先运行”或业务同学的口头理解。
* **隐式条件**

  ：例如“VIP 用户”这种词没有任何可度量的定义。
* **测试缺失**

  ：测试用例只覆盖了“典型路径”，而没有覆盖规则本身。

决策表的优势在于：

* 一张表清晰呈现规则，**强制你写清“在什么条件下做什么事”** 。
* 有助于识别“空洞区域”：哪些组合没有任何规则覆盖。
* 条件与动作被结构化后，可以直接映射为高覆盖的测试用例集。
* 方便评审与协作：产品、开发、测试可以围绕同一张表讨论，而不是各自脑补。

在测试设计层面，可以简单理解为：

> **一条规则 = 至少一条测试用例**，
> 决策表就是“测试场景的索引”。

# 2. 决策表的核心概念

## 条件（Condition）

促成不同业务结果的输入因素。要求：

* **归一化（normalized）**

  ：拆成若干相对独立的布尔或枚举条件，而不是一个模糊句子。
* **语义清晰**

  ：每个取值的含义可口头复述、不会产生歧义。
* **可实现**

  ：条件可以在代码层面被计算出来（有明确来源：请求字段、配置、用户画像等）。
* **可枚举**

  ：取值范围是有限、可列举的（布尔或有限集合）。

示例对比：

* ❌ 模糊条件：`C1: 用户是否是 VIP`
* ✅ 规范化后：

+ `C1: past_90d_spend_ge_1000`

  — 过去 90 天消费 ≥ 1000 元（Y/N）
+ `C2: membership_tier`

  — `none/silver/gold`

## 动作（Action）

系统在满足某条规则时会产生的**可观察结果**，包括：

* 接受 / 拒绝 / 回退 / 降级
* 报错 message ID（API / UX）
* 价格变化（字段 `final_price`）、运费、折扣金额
* 标记变化：如 `is_eligible=true/false`、`shipping_method=EXPRESS`
* UI 状态：按钮是否可点、选项是否显示

关键点：动作必须是**可以通过测试验证的东西**，而不是“模糊意图”。

示例：

* ✅ `APPLY 10% DISCOUNT`（最终价格可计算且可断言）
* ✅ `REJECT code=VALIDATION.code.expired`（错误码可验证）
* ❌ `给予适当优惠`（无法测试、每个人理解不同）

## 规则（Rule）

**一列**就是一个“条件组合 → 动作”的安排：

> Rule\_i: 在 C1=… 且 C2=… 且 … 条件下，执行 A1/A2…

常见两种规则执行策略：

* **首匹配生效（first-match wins）**

  ：从上到下依次检查，命中第一条后停止，不再往下匹配。
* **多规则同时生效（all-match apply）**

  ：所有被满足的规则动作都生效（例如多种优惠叠加）。

在测试设计和实现层面，应清晰写明当前表使用的是哪一种，否则很容易出现“业务理解”和“实现行为”不一致的情况。

# 3. 决策表设计流程（8 步可执行）

下面这套流程可以在几乎所有规则型场景中复用：

1. **列出条件**：

* 从 PRD/现有逻辑/业务对话中提取所有可能影响结果的关键因素。
* 优先拆成布尔/枚举条件，并给出明确取值定义。

2. **规范化模糊词**：

* 所有“VIP、大单用户、高风险”等模糊描述，都要写成可计算的条件。
* 例如：“大单 = 小计 ≥ 1000 元且商品数 ≥ 3”。

3. **列出动作**：

* 只保留“系统真正会做”的动作：返回什么、收多少钱、展示什么、调用哪个下游。
* 把“设计意图”转成“可观察行为”。

4. **草拟规则列**：

* 先不要追求穷举所有笛卡尔积。
* 从“业务中最常见 / 风险最大的几种情况”开始写规则列。
* 逐渐补齐，会比一开始列出所有组合更容易落地。

5. **加入优先级机制**：

* 明确规则的评估顺序。
* 一般先写“拒绝类、错误类、禁运类”等安全相关规则，再写“正常通过类”。

6. **检查完整性与一致性**：

* 全局地看一下：有没有明显的“组合空洞”？
* 有没有输入组合会命中多条规则且动作冲突？
* 是否需要显式写一条“默认规则”。

7. **最小化规则**：

* 使用 `–`（don’t-care）合并完全相同行为的列。
* 减少重复，只保留能区分逻辑分支的列。

8. **生成测试**：

* 至少一条测试对应一条规则。
* 对包含数值/日期的条件，额外补边界值测试（配合边界值与等价类方法）。

# 4. 记号规范

建议在团队内统一一套简洁的记号规范，避免每个人“各写一套”：

* 条件布尔取值：

+ `Y`

  = Yes（条件满足），`N` = No（条件不满足），`–` = don’t-care（与该条件无关）。

* 枚举取值：

+ 如地区：`EU/US/ROW`（欧洲/美国/其他地区）；
+ 用户级别：`none/silver/gold`。

* 动作记法：

+ 简洁同时可测试，如：`ACCEPT`、`REJECT code=...`、`SET shipping=FREIGHT`、`APPLY 10%`。

* 规则评估：

+ 若是首匹配策略，在表头或说明中标明：**Rules evaluated top-down, first-match wins**。

这套规范可以写入团队内部的 `decision-table-style-guide.md`，长期复用。

# 5. 示例 A：优惠码可叠加规则

以下示例展示如何从“业务政策”一步步落到：条件 → 动作 → 规则 → 测试。

## 5.1 业务政策（已规范化）

假设业务给出的要求是：

* 优惠码格式：长度 1–16，字符 `[A-Z0-9-]`。
* 优惠码可标记为“可叠加”或“不可叠加”。
* 赠品（gift card）与百分比折扣不可叠加。
* 订单小计需达到最低消费（例如 ≥ 50 元）。
* 冲突时，提示信息必须能指导用户下一步操作，而不是简单“调用失败”。

这里你可以看到，既有“验证类规则”（格式、过期），也有“业务冲突类规则”（是否可叠加、是否达到最低消费）。

---

## 5.2 条件

| ID | 条件 | 取值 |
| --- | --- | --- |
| C1 | 输入格式有效 | Y/N |
| C2 | 是否过期 | Y/N |
| C3 | 是否可叠加 | Y/N |
| C4 | 购物车含 gift card | Y/N |
| C5 | 是否满足最低消费 | Y/N |
| C6 | 优惠码类型 | `percent` / `fixed` |

注意几点：

* C1 通常由“边界值与等价类”测试负责（长度、字符集、trim 等）。
* C2 需要明确“过期”定义，例如是否含当天末尾。
* C3/C4/C5/C6 代表更高层的业务状态，往往来自 DB 或配置。

---

## 5.3 动作（可观察）

| A# | 动作 |
| --- | --- |
| A1 | 接受并应用优惠 |
| A2 | 拒绝：码已过期 |
| A3 | 拒绝：格式非法 |
| A4 | 拒绝：不可叠加 |
| A5 | 拒绝：未达最低消费 |
| A6 | 提示用户移除冲突商品 |

实际实现时，A2–A5 通常会对应具体错误码和前端提示文案，
例如 `VALIDATION.code.expired`、`CONFLICT.code.not_combinable` 等。

---

## 5.4 规则（首匹配）

假设：该决策表使用 **首匹配生效（first-match wins）** 策略。

| 规则 | C1 | C2 | C3 | C4 | C5 | C6 | 动作 | 说明 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| R1 | N | – | – | – | – | – | A3 | 输入非法优先 |
| R2 | Y | Y | – | – | – | – | A2 | 已过期直接拒绝 |
| R3 | Y | N | N | Y | – | percent | A4 + A6 | 不可叠加 + 有礼品卡 |
| R4 | Y | N | Y | – | N | – | A5 | 可叠加但金额不够 |
| R5 | Y | N | – | – | Y | – | A1 | 其他满足条件时接受 |

可以看到：

* 先处理“输入非法”、“已过期”这类 **通用拒绝条件**。
* 然后处理“不可叠加”这类特定业务冲突。
* 再处理最低消费。
* 最后落到“接受”作为默认通过路径。

---

## 5.5 示例测试

针对上述规则，可以设计如下测试（每条规则至少一例）：

* **R1**

  ：输入 `"SAVE 10"`（内部空格） → 格式非法 → 返回 `VALIDATION.code.charset`，不调用业务验证。
* **R2**

  ：输入合法 `"SAVE10"`，但已过期 → 返回 `VALIDATION.code.expired`。
* **R3**

  ：优惠码是百分比折扣且不可叠加 + 购物车包含 gift card → 返回 `CONFLICT.code.not_combinable`，前端提示“请先移除礼品卡后再使用优惠码”。
* **R4**

  ：可叠加优惠码，但小计只达到 40 元（未达 50） → 拒绝并提示“未达最低消费”。
* **R5**

  ：小计≥50，格式合法，未过期，组合合法 → 正常应用，更新订单总价并展示“已应用”。

> 可以配合：
> 边界与等价类：`../20-techniques/boundary-and-equivalence.md`
> 错误分类 → UX 映射：`../40-api-and-data-contracts/error-taxonomy.md`

# 6. 示例 B：运送方式可用性

这个例子展示决策表在“配送规则”、“可用配送选项”类场景中的应用。

## 条件

| ID | 条件 | 取值 |
| --- | --- | --- |
| C1 | 地区允许 | Y/N |
| C2 | 是否国内 | Y/N |
| C3 | 小计 ≥100 | Y/N |
| C4 | 重量 >20kg | Y/N |
| C5 | 冷链商品 | Y/N |
| C6 | 是否周末 | Y/N |

这些条件是配送可用性的主要影响因素。
如果真实业务中还涉及“偏远地区 surcharge”、“当日达”等条件，可以继续补充 C7、C8。

## 动作

| A# | 动作 |
| --- | --- |
| A1 | 提供免费配送 |
| A2 | 提供标准配送 |
| A3 | 提供货运配送 |
| A4 | 全部拒绝 |
| A5 | 隐藏“周末发货”选项 |

## 规则（优先级：安全 > 可选项）

| 规则 | C1 | C2 | C3 | C4 | C5 | C6 | 动作 | 说明 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| R1 | N | – | – | – | – | – | A4 | 区域禁运，全部拒绝 |
| R2 | Y | – | – | Y | – | – | A3 | 重货，只允许货运 |
| R3 | Y | Y | Y | N | – | – | A1 + A2 | 国内且满 100，免邮 + 标准 |
| R4 | Y | – | – | N | Y | Y | A2 + A5 | 冷链 + 周末，不允许周末发货 |
| R5 | Y | – | – | N | – | – | A2 | 默认标准配送 |

优先级安排：

1. 安全 / 合规优先（R1：禁运）。
2. 不可逆的物流限制（R2：重货必须走货运）。
3. 优惠或加值策略（R3：满额免邮）。
4. 特殊限制（R4：冷链 + 周末限制）。
5. 最终兜底规则（R5：标准配送）。

## 示例测试

* **R1**

  ：地区禁运（C1 = N） → 不展示任何配送选项，返回“无法配送到该地区”。
* **R2**

  ：地区允许，重量 >20kg → 只展示货运选项，费用按重货计费。
* **R3**

  ：国内用户，小计 ≥ 100，无重货 → 展示“免费配送 + 标准配送”。
* **R4**

  ：冷链商品 + 周末 → 只展示标准配送，但隐藏“周末发货”类选项或说明无法周末发货。
* **R5**

  ：其他情况 → 仅标准配送。

# 7. 检查完整性与一致性

在完成初版决策表后，推荐做一次“结构化自查”：

* **完整性**

+ 对照条件的所有可能取值组合，看看是否存在落空的情况。
+ 特别注意“边缘组合”：例如同时重货 + 冷链 + 小计≥100 等极端情况是否被覆盖。

* **一致性**

+ 不应出现同一组合命中多条规则且动作冲突的情况。
+ 若使用多规则同时生效，需要明确动作如何合并（如多个折扣叠加顺序）。

* **优先级**

+ 验证类规则、禁运类规则，建议放在表的上方优先判断。
+ 把“默认规则”显式写出来，而不是依赖代码里的默认分支。

* **域互斥（MECE）**

+ 如果发现某些组合因为排序 / tie-breaker 不明确而导致不稳定行为，例如列表排序规则依赖 `created_at` 而没有 `id` 作为次级排序，可以在条件中增加“排序稳定条件”，避免结果随时间漂移。

# 8. 从决策表到最小测试集

决策表不等于“测试用例列表”，但可以直接派生出一个**最小测试集**：

* 每条规则（每一列）至少对应一条测试。
* 对包含数值 / 时间的条件，为关键规则额外补齐**边界值测试**（如`99/100/101`）。
* 对重要条件，可以采用简化版 MC/DC（修改条件 / 决策覆盖）思路：

+ 为每个条件构造一组测试，使得在其他条件不变的情况下，**只改变该条件就能翻转结果**（如果业务允许）。

在运送方式场景下，可以：

* 以 R3（国内满额免邮）为基础：

+ 改变 C3（满额 → 不满额） → 应从 R3 跳到 R5。
+ 改变 C1（允许区域 → 禁运） → 应从 R3 跳到 R1。
+ 改变 C4（非重货 → 重货） → 应从 R3 跳到 R2。
+ 改变 C5/C6（普通 → 冷链 + 周末） → 应从 R3 跳到 R4。

这类“单条件翻转”测试可以确保条件真的影响决策，而不是“写在表里但实现中并没有使用”。

# 9. 可验证依据（Oracles）与证据

决策表转为测试后，还需要明确**如何验证规则确实被命中**。

可验证依据包括：

* **API / DB**

+ 响应体中的字段：可用配送列表、最终折扣金额、错误码。
+ 数据库记录：例如 `applied_rule_id`、`discount_amount`。

* **UX**

+ 哪些按钮或选项可见/不可见。
+ 显示的 message ID 或文案 key（避免使用纯文案匹配）。

* **合约（schema）**

+ 是否符合预期的错误码 schema。
+ 是否缺少必选字段、重复字段。

* **非功能**

+ 规则执行是否在可接受性能范围内（p95）。
+ 批量规则评估是否稳定（例如千级 / 万级订单场景）。

* **证据（Evidence）**

+ 结构化日志（建议记录 rule\_id、条件快照）。
+ 监控指标，如某条规则被命中的次数。

示例日志：

```
{
  "event": "discount.evaluate",
  "rule_id": "R3",
  "conditions": {
    "input_valid": true,
    "expired": false,
    "combinable": true,
    "gift_card_present": false,
    "subtotal_ge_50": true,
    "code_type": "percent"
  }
}
```

通过 `rule_id` + 条件快照，可以在事故复盘时快速还原“当时命中的到底是哪条规则”。

# 10. 反模式（Anti-pattern）

在实践中，下面这些做法会显著降低决策表的价值：

* **隐式优先级**

+ 业务逻辑“默认上面的先处理”，但在表中没有任何说明。
+ 建议：在表中明确写明评估顺序，并在实现层保持一致。

* **冲突规则**

+ 同一输入组合命中多条规则，却指向不同动作。
+ 典型表现：前端允许使用优惠码，后端却拒绝。

* **含糊条件**

+ “VIP 用户”、“活跃用户”等未被量化。
+ 建议：全部转化为具体数值/区间条件。

* **无默认规则**

+ 某些组合因为没有显式规则，落入“不确定行为”。
+ 建议：显式写出“否则 → 拒绝”或“否则 → 使用默认行为”。

* **动作不可观察**

+ 动作描述为“给予一定优惠”、“冻结账户一段时间”等模糊说法，缺乏可验证的字段。
+ 测试无法判断是否正确执行。

* **范围条件缺少边界值**

+ 只写“≥100”的条件，而没有在测试中覆盖 99/100/101 等边界值，容易产生 off-by-one 问题。

# 11. 快速检查清单（Review Gate）

在评审决策表或相关测试用例时，可以使用以下清单：

* 条件完整、规范化、取值域清晰（无模糊词）
* 每个动作均可观察、可通过测试断言
* 规则有明确优先级说明，无冲突或歧义
* 存在显式“默认规则”或兜底逻辑
* 数值 / 时间条件在测试中有边界值覆盖
* 每条规则至少有 1 条对应测试，关键条件有 MC/DC 翻转用例
* 日志中包含 `rule_id`、条件快照或等效信息，便于排查与审计
