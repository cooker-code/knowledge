**一、背景**

在数字化转型的浪潮中，企业对软件质量和交付效率的要求日益提高，然而传统 UI 自动化测试平台面临着高维护成本和低适应性的瓶颈（比如：界面元素频繁变更导致脚本失效），人工维护效率低下，难以匹配持续交付需求。与此同时，随着技术团队逐步推进试点功能测试由研发主导的模式，亟需通过“功能测试左移”将质量保障手段前置到需求设计与编码阶段，提前发现缺陷问题以降低缺陷修复成本，提升研发效率。

**（一）**

**传统UI自动化平台的局限性**

传统的 UI 自动化测试主要依赖于元素定位和脚本编写来实现自动化测试。这种方法在一些简单、稳定的业务场景中表现良好，但在面对以下挑战时则显得力不从心：

|  |  |
| --- | --- |
| **01** | **产品 UI 变更频繁：**随着产品迭代速度加快，UI 界面的变化较频繁，传统的 UI 自动化测试高度依赖元素定位，需要持续投入大量时间和精力来维护和更新脚本，但实际的落地运用效果却不尽如人意。 |
| **02** | **跨平台兼容性问题：**不同平台（如 Web、H5、iOS、Android 等）的 UI 自动化框架存在差异，即使产品在不同平台上的操作相差无几，传统的自动化测试也需要针对不同平台分别编写和维护脚本，这导致大大增加了测试工作量。 |
| **03** | **复杂界面元素识别：**对于一些复杂的 UI 界面和交互元素（如：动态生成的内容、图片中的文字、iframe 框架、画布等），传统的元素定位方法难以准确识别，需要特殊处理，导致测试结果不可靠。 |
| **04** | **可读性和可维护性：**复杂页面的 UI 交互逻辑可能导致自动化脚本冗长、结构混乱，降低了自动化脚本的可读性，同时也增加了后续理解和维护脚本的难度。 |

**（二）**

**基于大模型AI+UI自动化的创新方案**

ChatGPT 的横空出世引发了全球关注，同时在今年年初爆火的 DeepSeek 国产 AI 热潮影响下，为了应对上述传统 UI 自动化测试的诸多局限性，我们开始探索引入基于大模型 AI 的 UI 自动化测试方案，通过 AI 视觉理解与自然语言处理能力，动态解析 UI 元素逻辑关系，自动生成可执行的脚本指令，减少对元素定位的依赖，降低各角色使用自动化的成本以及传统 UI 自动化对元素定位的依赖。对比传统 UI 自动化测试，大模型 AI+UI 自动化具有以下几个核心优势：

* **UI 元素智能识别：**利用 AI 视觉技术，能够智能识别 UI 界面中的各种元素，包括文本、图片、图标等。即使产品界面发生变化，也能快速准确地找到对应的元素，极大地提升了测试的灵活性和准确性，降低人工维护的成本。
* **跨平台通用性：**由于基于页面截图进行识别和操作，故在不同平台上的自动化脚本编写调试体验一致，同时针对不同平台的细微 UI 差异，也可以实现一次编写、多平台运行，极大地简化了跨平台测试的工作量。
* **高精度与鲁棒性：**AI 视觉技术具有强大的图像处理和模式识别能力，能够处理各种复杂的 UI 场景（如：动态内容、模糊图片等），确保测试结果的高精度和鲁棒性。
* **可读性和可维护性：**基于自然语言描述的自动化脚本，不仅整体可读性和可维护性较好，同时对各产研角色使用自动化功能也更加友好。
* **共性 UI 问题检测：**运用大模型 AI 的能力，通过定制的 Prompt 提示词，可以实现共性 UI 问题的巡检发现，例如：UI 样式问题诸如白屏、报错、空坑、元素堆叠、异常内容 NaN/Null 等；页面数据问题诸如：小数点精度。
* **大模型自主规划：**基于大模型 AI 的能力，能够逐步实现基于意图识别的大模型自主规划能力，降低用例编写成本，未来甚至可以基于大模型对页面的理解来自动做测试。

**二、方案选型**

时间回到今年 2 月，在初步明确了未来自动化测试的方向后，我们启动了基于大模型 AI + UI 自动化创新方案的调研工作，旨在寻找能够通过大模型 AI 来辅助提升软件质量与研发效率的技术路径，期望运用大模型 AI 带来的技术红利，降低自动化测试的成本。本次调研主要聚焦于两种具备大模型和视觉识别能力的方案，即：基于 OmniParser + Qwen2.5-VL + Playwright 的方案 以及 基于 Midscene 框架 + Qwen2.5-VL + Playwright 的方案。

经过近两周的突击调研，我们发现上述两种方案均展示了大模型应用于 UI 元素识别和意图理解的潜力和优势（尤其是视觉语言模型），而执行层均选择 Playwright 作为底层的执行引擎，初步调研结果与我们期望通过“功能测试左移”提升研发效率的目标高度契合。然而，两者在**识别精度与鲁棒性、系统集成度、扩展性和开发维护成本**方面各有侧重和局限。我们意识到，要实现高精度、高鲁棒性且易于维护的基于大模型 AI 驱动的 UI 自动化平台，必须在视觉识别的准确性与操作执行的灵活性之间找到最佳平衡点。

项目组对初步调研结论进行了深入探讨与权衡，同时结合“基于大模型 AI+UI 自动化创新方案”的核心目标，重点聚焦于智能识别 UI 界面中的各类元素和操作，以实现跨平台通用性以及自动化操作的高精度与强鲁棒性，我们最终决定采用 **Qwen2.5-VL + 文本描述 来识别当前浏览器页面，让大模型给出要操作的内容和坐标，再转化成 Playwright 进行操作的方案，并且设计了类似 ReAct 的架构，能够基于目标多次拆解动作**。选择此方案主要有以下几点考量：

01

**充****分发挥大模型的核心能力**

* **直接视觉理解与意图识别：** 我们的方案将 Qwen2.5-VL 作为核心“大脑”，直接输入浏览器页面的视觉截图和用户提供的文本描述（如“点击提交按钮”），由大模型进行综合理解，直接输出要操作的元素内容及其精确坐标。这规避了像 OmniParser 那样需要独立前置模块进行繁琐元素标注的复杂性，减少了中间环节的识别误差累积。
* **降低元素定位维护成本：** 传统 UI 自动化依赖硬编码的元素定位器，易受 UI 界面变化影响。通过 Qwen2.5-VL 的视觉识别能力，可以显著降低对特定元素定位符的依赖，即使界面布局或元素细节发生微小变化，大模型也能基于其强大的视觉理解能力找到并识别目标元素，从而大幅减少维护工作量，提升平台的适应性。

02

**实现高度的灵活性与解耦**

* **“大脑”与“肢体”分离：** Qwen2.5-VL 专注于“思考”（理解用户意图、识别目标、给出坐标），而 Playwright 专注于“执行”（按照给定坐标和操作类型精确执行）。这种高内聚低耦合的设计使得系统更易于扩展和维护。当需要支持新的操作类型或适应新的浏览器环境时，可以分别优化 Qwen2.5-VL 的理解能力或 Playwright 的执行封装，而不会相互影响。
* **规避框架绑定：** 相较于 Midscene 框架的强绑定特性，我们的方案避免了对特定框架的过度依赖。Midscene 虽然提供集成能力，但其在复杂操作扩展、执行调度、缓存机制等方面的局限性，需要大量的二次开发才能弥补。而我们的选择允许我们根据实际需求，更自由地封装 Playwright 的强大操作能力，并构建更灵活的执行调度和断言机制。

03

**满足跨平台与高精度要求**

* **视觉驱动的跨平台性：** Qwen2.5-VL 直接处理页面截图，使其天生具备跨平台通用性。无论是 Web、H5、还是移动端视图，甚至是桌面应用（通过截图方式），只要能够捕获其 UI 视觉信息，理论上都能进行识别和操作。这与项目背景中“一次编写，多平台运行”的目标高度吻合。
* **提升鲁棒性：**大模型的 AI 识别能力远超传统基于 DOM 结构或图像特征的识别方法，能够处理更复杂的 UI 场景，如动态内容、图片中的文字、乃至模糊图像等，从而保障测试结果的高精度与鲁棒性。

在方案初步确定之后，我们快速做了初步的 DEMO 并跑通了核心流程（主要使用 Qwen2.5-VL 7B 模型和 Playwright 来模拟操作），开始前期的 DEMO 验证，在DEMO 验证阶段我们发现针对简单的页面，DEMO 落地的效果令人满意，但稍微复杂一些的页面，DEMO 的效果却差强人意，但随着进一步的调研尝试，我们发现通过使用更大参数的模型（Qwen2.5-VL 72B 模型）以及更精确的自然语言来描述操作，复杂页面所遇到的问题基本也都能解决。

经过初步的 DEMO 验证后，参与前期调研的项目组同学也基本有了一个共识，尽管 Qwen2.5-VL 作为通用大模型，在特定 UI 识别场景下仍可能面临挑战，但其直接的视觉理解能力和强大的语义理解潜力，与 Playwright 的稳定执行能力结合，为构建一个智能化、高适应性、低维护成本的基于大模型 AI 驱动的 UI 自动化平台提供了可行性较高的技术路径。

**三、方案挑战**

虽然我们在前期做了很多调研和相对充分的 DEMO 验证，而且 Qwen2.5-VL + 文本描述 + Playwright 的方案也具备显著优势，但在实际落地过程中，仍然面临一系列技术挑战，需要投入精力进行攻克，主要核心挑战如下：

|  |  |
| --- | --- |
| **01** | **Qwen2.5-VL 的视觉识别精度与鲁棒性：**   * **复杂 UI 元素识别：** 尽管 Qwen2.5-VL 强大，但对于高度自定义、动态生成、或在视觉上高度相似但功能不同的 UI 元素（例如，多个同名按钮或不同筛选条件下的相似输入框），大模型能否始终准确识别其意图并提供正确的坐标，仍是核心挑战。调研中 OmniParser 遇到的“遗漏选项”、“定位偏差”等问题，Qwen2.5-VL 也可能会存在类似的问题。 * **微小差异与上下文理解：** 对于需要精确定位到像素级的操作（如拖拽特定区域），或需要根据上下文（如某个表单内部的“保存”按钮与全局的“保存”按钮）进行选择时，Qwen2.5-VL 的识别精度和上下文理解能力将经受考验。 * **假阳性与误识别：**大模型可能将非操作性元素误识别为可操作目标，或将错误的元素与用户意图关联，导致输出错误的操作坐标。 |
| **02** | **大模型输出结果与Playwright 操作的映射及转换：**   * **标准化输出格式：**Qwen2.5-VL 需要输出一个清晰、标准化、易于 Playwright 解析和执行的结构化数据（例如，{"action": "click", "coordinates": [x, y], "text": "按钮文本"}），确保其能够覆盖 Playwright 支持的所有操作类型（点击、输入、拖拽、滚动、悬停、键盘操作等）。 * **坐标精确度与操作类型：** 大模型给出的坐标可能是一个 bounding box，而 Playwright 的操作（如click）可能需要更精确的中心点或其他特定位置。如何将大模型输出的宽泛坐标精确转化为 Playwright 所需的交互点，并确保操作类型的正确性（例如，是fill 还是type，是单次click 还是dblclick），是需要细致处理的环节。 * **复杂操作的语义化：** Playwright 支持一些高级操作，如文件上传、多元素选择等。如何通过文本描述让 Qwen2.5-VL 理解这些复杂操作，并生成 Playwright 能够执行的指令，是转换层的又一挑战。 |
| **03** | **用户意图理解与自然语言处理（NLP）的鲁棒性：**   * **意图识别与歧义：**用户输入的自然语言描述可能存在歧义、不完整或口语化。Qwen2.5-VL 需要准确地从这些描述中抽取核心意图，并将其与页面上的具体元素关联起来。例如，“点一下”可能指点击按钮，也可能指点击链接，大模型如何根据视觉上下文判断出真实意图。 * **步骤扩写与规划：** 对于一些隐含的且需要多步操作才能完成的意图（如“登录系统”），大模型能否将其分解为具体可执行的子步骤，并进行合理的执行规划，将直接影响自动化用例的编写效率和覆盖度。 * **上下文保持与记忆：** 在连续的自动化步骤中，大模型需要能够理解并保持上下文信息，例如，前一步骤的某个输入值可能影响下一步骤的可用元素，这需要在设计时考虑如何将历史交互信息有效传递给大模型。 |
| **04** | **性能与响应时间：**  **推理延迟：** Qwen2.5-VL 作为大型模型，其推理过程可能存在一定的延迟，在需要快速交互或执行大量步骤的场景下，如何优化模型推理速度，减少端到端响应时间，以保证用户体验和测试效率，是一个重要考量。  **资源消耗：**大模型运行对计算资源（GPU、内存等）的需求较高，需要合理规划部署和资源分配，以支持并行执行和高并发场景。 |
| **05** | **调试与问题排查：**   * **AI 黑盒特性：**基于大模型 AI 驱动的自动化测试执行失败时，定位问题根因将比传统 UI 自动化更加复杂（失败可能是模型识别错误、意图理解偏差、Playwright 执行以及环境因素等），需要设计可视化工具和详细日志，以便深入洞察模型决策过程和执行细节。 * **迭代优化与反馈循环：** 如何建立有效的反馈机制，将失败案例或误识别情况反馈给模型，进行持续的训练和优化，以提升模型的能力，是维护该系统的关键。 |

为了应对上述核心挑战，我们采取敏捷迭代、持续优化的策略，重点围绕在AI Agent、输出解析逻辑、Playwright 操作封装和可视化调试工具方面进行建设，逐步提升平台的稳定性、鲁棒性、性能以及用户体验。

**四、技术实现**

平台架构图

如上图所示，我们通过一套产品架构设计，既支持 PC 端的 AI+UI自动化，也支持 APP 端的 AI+UI自动化，确保用户在跨平台的自动化场景下，产品能力和用户体验上能够保持一致，具体技术方案实现细节介绍如下：

**（一）**

**用例平台**

在传统 UI 自动化平台中，每次修改用例后通常需要重新执行整个用例，而执行过程仅能通过录屏或截图查看，这种方式不仅调试成本较高，还显著增加了时间消耗，导致整体效率较低，基于此痛点我们设计了实时交互和单步调试功能来提升调试操作体验，并有效降低问题排查成本，具体功能点介绍如下：

**1、实时交互**

通过集成浏览器渲染执行环境实现动态执行过程可视化，允许用户在本地浏览器页面中实时观测远程浏览器引擎的执行轨迹，同时支持通过鼠标、键盘与嵌入式浏览器实例进行双向交互，在保障执行环境隔离性的同时，实现了毫秒级的渲染状态更新与输入反馈延迟，为用户提供端到端的浏览器执行过程分析与调试能力。

实时交互示例：

实现原理：通过在本地浏览器页面嵌入 Chrome DevTools 调试器前端（Web 应用程序），远程 Playwright 无头浏览器执行环境实例启动时，暴露特定调试器后端端口，通过 WebSocket 协议提供前后端实时通信的能力，处理 Chrome DevTools Frontend 的操作指令，并返回相关信息，从而实现远程浏览器的实时渲染。另外，在平台实际落地实践过程中，我们发现 Chrome DevTools 在特定情况会发生自动断开 WebSocket 链接的问题，严重影响用户调试操作体验，针对此问题我们增加了无感自动重试的能力。

**2、单步调试**

基于远程浏览器实时交互能力，平台实现了单步骤调试功能，进一步降低用例脚本修改及调试成本，同时单步调试完成后可以实时预览步骤的执行过程和执行结果，辅助快速定位失败原因，提升用户调试操作体验。

**（二）**

**执行调度**

在产品架构设计初期，我们确定了支持上万级别自动化测试用例并发执行的核心目标。为此，我们在分布式任务调度层也做了深度优化。通过采用微服务化任务编排引擎与容器化资源调度框架，结合动态负载均衡算法和分布式锁机制，实现了自动化任务的智能分片与资源池化管理；在系统设计上，我们实现了基于优先级的弹性扩缩容策略，并结合多维度资源隔离技术与稳定性设计，从而有效确保高并发场景下的任务调度吞吐量与执行稳定性。此外，我们还引入了分布式事务补偿机制和故障自愈能力，以便在大规模自动化执行过程中，保障平台的高可用性与线性扩展能力。核心功能点介绍如下：

**1、调度能力**

我们构建了自动化实验室功能，支持用户将编写的自动化用例配置为实验室任务。通过配置定时任务，可按需定时触发自动化用例集的执行。此外，自动化实验室还支持灵活设置用例的执行顺序，以及用例和步骤失败后的重试执行次数，从而提升用例执行的稳定性，减少因用例执行冲突或稳定性问题等因素导致的用例失败。另外，为了避免高峰期用例调度给执行环境造成压力，导致用例执行阻塞，我们设计了动态限流以及动态扩缩容能力。

* 当有大量用例需要执行或存在用例因资源不足而等待时，系统会自动感知环境并无感完成机器扩容，扩容完成后通知系统调度层，调度层增加可同时执行用例数量，从而提升执行效率。
* 当系统检测到存在较多空闲执行机器时，会触发执行环境自动缩容操作，缩容启动后，系统将通知调度层，调度层随即减少可同时执行的用例数量，从而释放空闲机器资源，优化资源利用率。

2、**心跳检测**

在自动化用例执行过程中，可能会因机器重启、服务异常等各类异常场景而被迫中断，最终导致用例无法顺利完成执行，针对这种情况我们设计了用例心跳机制，通过定期写入心跳记录来监测用例执行状态是否存活。当系统检测到某条用例无心跳时，将判定该用例的执行已被中断。在系统重新调度时，会自动捞取上下文和执行实例，并尝试从中断步骤恢复执行。如果执行实例已被销毁，系统将回退相关记录并重新启动用例执行。

**（三）**

**AI Agent**

在系统设计中，我们抽象出统一的 Agent 层，用于集中管理大模型相关的核心能力，包括模型调用、Prompt 管理、RAG、历史数据沉淀等，同时Agent 层不仅支持 PC 端 和 APP 端 等跨平台的自动化用例执行，还能够扩展支持基于大模型AI的内容检测、UI样式检测、页面数据检测等多样化检测能力。核心功能点介绍如下：

**1、自动化执行**

平台支持各业务线自定义配置规划、断言的 Prompt，在用例执行时由调度层传递设备信息、自定义 Prompt、用户步骤描述等信息，AI Agent主要负责意图识别、规划执行以及结果断言。

**意图识别：**Agent通过设备信息获取执行环境执行前的截图，拿到后结合上下文信息，汇总给大模型，大模型基于这些识别用户意图，并转换成内部结构。

**规划执行：**基于执行前截图，根据意图识别结果转换成需执行的坐标和操作类型，然后传递给执行环境执行具体的操作，在执行结束后，执行环境返回页面执行状态。

AI 输出：Next Action（动作） + coordinate（坐标） + value（输入 value，可空）；Reasoning（思考过程）

**结果断言：**在需要断言时，我们拿到用户断言目标以及执行前截图、执行后截图，汇总交给大模型，由大模型判定断言结果。

**2、页面内容及UI样式检测**

AI+UI 自动化用例编写过程中，无论是业务操作结果还是页面状态，我们都需要明确对每个步骤执行结果断言，才可能发现特定的问题，我们思考是否能够形成一些基础通用的检测规则，在自动化用例执行过程中，自动对执行结果进行检测，这样不仅能减少编写用例校验点成本，还能避免编写用例时校验点遗漏的情况。所以我们设计了基于大模型 AI 的页面内容和UI样式检测机制，在自动化用例执行完成后，我们会收集用例执行过程产出的页面截图等上下文信息，通过系统或者用户自定义检测的 Prompt，异步检测执行过程中潜在的页面内容及UI样式问题。

**3、页面数据检测**

基于 Agent 对页面的感知能力，我们在通用的自动化能力基础上，进一步扩展了页面数据巡检能力。该能力用于检测页面可见数据的正确性，帮助我们快速发现数据质量问题，适用于页面数据指标检测、跨页面或跨步骤数据关系检测等多种场景下的数据质量问题检测。而页面数据巡检用例与通用的自动化用例的主要区别在于用例的系统配置的不同，其执行流程与通用的自动化执行基本一致，我们主要在自动化执行基础能力的基础上扩展了数据提取和校验能力，同时支持子 Agent 实现，适配自定义需求，下图仅展示差异部分。

**机器实例上限与资源控制：**

Playwright 实例并非轻量级对象，其底层会启动多个 Chromium 进程，对系统资源的消耗显著。单个Playwright实例进程结构如下：1个浏览器主进程（Browser Process）、1~5个渲染进程（Renderer Processes，取决于页面复杂度）、1个 GPU 进程（可选）、1个网络服务进程、1个插件/沙箱进程（可选），而单个Playwright实例对内存和 CPU 的占用情况取决于页面的复杂性，通常占用 300MB至1GB 不等，如果不限制单机上的实例数量，可能会迅速耗尽机器内存或CPU资源，导致容器 OOM 被 kill，从而引发浏览器崩溃并影响自动化测试执行。因此，需要对 Playwright 实例进行管理，设置服务器实例上限。

**Playwright实例负载均衡：**

在 AI+UI 自动化测试平台中，Playwright 实例根据使用场景的不同，分为 Debug 模式 和 非 Debug 模式 两种运行形态。两者在执行方式、资源占用、功能特性及调度策略上存在显著差异，需进行差异化管理与资源配置。

因此，Playwright 实例的创建不仅需要单独判断实例的初始化情况，还需要考虑两种不同模式在不同情况下的实例上限配置，然而，HSF服务自带的负载均衡能力无法满足这一要求，需要设计一套适配 Playwright 实例创建的定制化负载均衡能力，具体实现简述如下：

**六、****一点心得**

01

**方案选型**

基于大模型 AI 的 UI 自动化，业界有很多种不同的实现方案，但每种方案都会存在各自的问题和局限性，本质的区别主要体现在对页面信息的获取方案不同（比如：DOM 树/图像/DOM+图像等 ；提取页面部分信息/全量信息等），每种方案都有需要解决的技术难点，没有“完美”的方案，选择最适合的方案。

02

**大模型选型**

不同方案选型，甚至是方案中的核心技术选型，对大模型 AI 的能力要求各不相同，需要根据方案和技术选型来选择适合的大模型，如果大模型的效果不佳，不建议投入时间做 SFT（ROI 不高），因为目前模型的迭代升级速度很快，后续通过模型升级基本都能够解决，无论是通过升级模型尺寸还是版本。

03

**Prompt 提示词**

这一点非常关键，一份高质量Prompt提示词能够解决大部分问题。在实际应用大模型的过程中，如果遇到大模型效果不佳的情况，建议优先尝试优化 Prompt 提示词，相比投入资源进行 SFT，优化Prompt提示词往往能够带来更高的 ROI，同时成本更低、见效更快。

04

**大模型幻觉**

目前，大模型的幻觉问题尚无法完全避免。为了确保系统的稳定性，在系统设计上可以对大模型的返回结果增加一层工程兜底机制。通过这种方式，能够有效规避大模型幻觉带来的潜在问题，从而进一步提升平台的稳定性和可靠性，提升自动化执行的通过率。
