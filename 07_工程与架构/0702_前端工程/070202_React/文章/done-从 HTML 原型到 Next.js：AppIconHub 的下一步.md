## 缘起：原型跑起来了，但不够用

##

上一篇聊到用 DeepSeek V4 写了个纯 HTML 的图标管理原型，一个文件打开就能用，左侧应用列表、右侧图标网格，查看起来比翻文件夹直观多了。

但用了几天发现，原型毕竟是原型。图标多了没有分页，搜索只能搜应用名，主题也只有一套亮色。最难受的是数据存在内存里，刷新页面全没了。每次想加新图标，得改 JSON 数据再刷新，效率实在跟不上。

## 当时想：要不找个现成的开源项目改改？省得从零写。

##

---

##

第一步：找到 svgl，差点直接 Fork

GitHub 上搜了一圈，发现 svgl 这个项目，界面确实漂亮——三栏布局、卡片式图标展示、分类筛选，跟我想做的方向很接近。

看了下技术栈：SvelteKit + Tailwind CSS，代码结构也挺清晰。当时差点直接 Fork 开改，但仔细看了数据层发现不对劲：svgl 是展示第三方品牌 Logo 的。

我要的是"APP图标管理"，随时上传、随时分类、随时搜索，这跟 svgl 的设计目标完全不同。强行改的话，数据层要重写，交互逻辑要重写，最后可能改动的代码比保留的还多。

算了，借鉴 UI 思路，自己用熟悉的技术栈重新做。

---

第二步：技术选型，不追新只追顺手

svgl 用 SvelteKit，性能确实好，但最近用 Next.js 更多，shadcn/ui 也用得顺手。做个人工具没必要为了技术而技术，能快速迭代、遇到问题能自己解决才是关键。

最终定了这套：

* **Next.js 15 + TypeScript**：熟悉，生态成熟
* **shadcn/ui v4**：组件即拿即用，主题切换现成的
* **Tailwind CSS**：写样式快，不用切文件
* **next-themes**：白天黑夜切换，几行代码搞定

---

## 第三步：把原型验证过的东西，做成产品级

##

## 上一篇原型验证了核心交互：左侧应用列表 + 右侧图标网格。这次在此基础上补全了实际需要的功能：

##

**三栏布局保留，但加了响应式**

桌面端左侧分类、中间内容、右侧可以补操作区；移动端侧边栏收进 Sheet 抽屉，不占用屏幕空间。这个思路跟 svgl 一致，但实现是用自己熟悉的组件库。

**图标卡片重新设计**

原型里只有图标和名称，正式版加了尺寸标签（1024×1024、512×512）、状态标识（在用/备用/废弃）、收藏按钮。底部放了复制、下载、链接三个操作，常用功能不用点进去。

**分页和搜索**

原型没分页，图标多了卡顿明显。这次做了分页器，支持调整每页数量。搜索从只搜应用名，扩展到搜图标名称和标签。

**主题切换**

原型只有亮色，这次加了深色模式，用 next-themes 实现，跟随系统或手动切换，状态持久化。

**添加图标弹窗**

原型改 JSON 的方式太原始，这次做了表单弹窗：上传图标、选择分类、填写尺寸和用途，提交后直接入库刷新列表。

---

## 现在的状态

##

目前功能基本跑通，但还有几个地方没完善：

* 数据持久化还没接，现在刷新页面还是会丢
* 拖拽上传和自动压缩在计划里，还没实现
* 多尺寸导出功能待开发
* 代码结构需要再整理，暂时没打算开源

算是从"能看"到"能用"的中间阶段，先用着，边用边改。

如果你在管理 APP 图标或投放素材时也遇到过类似的痛点，欢迎评论区聊聊，看看有没有更好的工具推荐给我。

---

## 一点感受

##

1. **原型验证的价值**

   上一篇 HTML 原型虽然简陋，但帮我确认了"左侧列表 + 右侧网格"的交互是顺手的。如果没有原型直接写 Next.js，可能做到一半发现布局不合适，返工成本更高。
2. **借鉴设计，不借鉴实现**

   svgl 的三栏布局和卡片样式确实好看，但数据层、技术栈完全不适合我的需求。看开源项目时，先想清楚"我要借什么"，比直接 Fork 更重要。
3. **熟悉的技术栈省时间**

   用 Next.js 而不是 SvelteKit，不是因为哪个更好，是因为我能更快写出没有 bug 的代码。个人项目，顺手比先进重要。
