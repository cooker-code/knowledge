## 核心问题

Java 工具类文章容易把“能少写代码”包装成架构价值。真正要沉淀的是两类边界：扩展点如何被发现、加载和隔离；本地缓存与远端缓存如何控制一致性、容量和失效。

## 扩展点判断

| 机制 | 适用 | 风险 |
|---|---|---|
| 枚举工具 / Guava 集合 | 降低局部样板代码 | 不改变模块边界，不能作为架构结论 |
| 反射工具 | 框架适配、元数据读取、兼容旧接口 | 类型安全、性能、访问权限、升级兼容风险仍在 |
| SPI / ServiceLoader | 主程序定义接口，外部实现按约定接入 | 只解决发现，不解决生命周期、隔离和卸载 |
| 动态模块 / Celix 类方案 | 平台型系统需要运行时装配能力 | 类加载、依赖冲突、状态迁移、资源释放和审计 |
| Project Panama | Java 与原生库互操作 | 部署、内存安全、平台差异、观测和安全边界 |

## 缓存组合判断

| 问题 | 准则 |
|---|---|
| 为什么加本地缓存 | 必须有命中率、延迟、QPS 或成本目标；不能只因为“性能炸裂” |
| 本地 + Redis 如何读 | 通常先查本地，再查 Redis，再回源；每层都要有超时和失败处理 |
| 如何失效 | 要明确 TTL、主动删除、消息通知或版本号；否则本地缓存可能长期脏读 |
| 如何防击穿 | 热点 key 需要单飞、锁、异步刷新或限流；Function 链不能掩盖并发问题 |
| 如何观测 | 至少记录命中率、回源次数、加载耗时、缓存大小、淘汰次数和异常数 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| SPI 就是插件化 | SPI 只是扩展发现，插件化还要生命周期、隔离、配置、审计和回滚 |
| 反射工具让反射变安全 | 工具只降低调用成本，不消除类型、权限和性能风险 |
| 双缓存天然高性能 | 没有失效、一致性、容量和回源保护，双缓存会放大错误 |
| Guava/Commons 工具值得单独沉淀 | 只有改变设计判断时才沉淀；纯 API 清单降权为工具锚点 |

## 待验证缺口

- SPI 与 Spring 自动配置、Java Module System、插件框架的边界。
- 双缓存的一致性策略需要真实指标和故障案例支撑。
- Project Panama 需要官方版本、平台兼容和生产部署证据，不能只凭示例下结论。
