# 标准项目架构与代码组织

## 目标

这个知识点要回答：

> 新建一个 Java 项目时，目录怎么建，代码怎么放，每一层写什么，不写什么。

## 推荐分层

### 轻量项目

适合后台管理、小型服务、原型项目。

```text
src/main/java/com/example/app/
  web/
    controller/
    dto/
    advice/
  service/
    model/
    exception/
  repository/
  integration/
    external/
    messaging/
  config/
```

| 层 | 写什么 | 不写什么 |
|---|---|---|
| controller | 接收请求、触发参数校验、调用 service、返回 response | 不写业务规则，不直接访问数据库 |
| dto | request、response、参数校验注解 | 不承载领域行为 |
| service | 用例编排、事务边界、调用 repository/integration | 不堆入口参数校验，不写 HTTP 细节 |
| model | 简单业务模型和状态 | 不依赖 Web 框架 |
| repository | 数据访问接口和实现 | 不反向决定业务规则 |
| integration | 第三方 API、消息、文件、缓存适配 | 不把外部模型传进业务核心 |
| config | 框架配置和 Bean 装配 | 不写业务逻辑 |

### 核心业务项目

适合订单、库存、计费、工作流、规则平台、数据治理等长期维护系统。

```text
src/main/java/com/example/app/
  interfaces/
    rest/
    scheduler/
    messaging/
  application/
    command/
    query/
    service/
    port/
      incoming/
      outgoing/
  domain/
    model/
      aggregate/
      entity/
      valueobject/
    service/
    repository/
    event/
  infrastructure/
    persistence/
    external/
    messaging/
    config/
```

| 层 | 写什么 | 不写什么 |
|---|---|---|
| interfaces | REST、定时任务、消息消费者等入口适配 | 不写业务规则 |
| application | 一个用例的流程、事务、调用领域对象和端口 | 不写 SQL、HTTP、MQ 细节 |
| domain | 业务不变量、实体行为、值对象约束、领域服务 | 不依赖 Spring MVC、JPA、第三方 SDK |
| infrastructure | JPA/MyBatis、HTTP Client、MQ、缓存、文件等技术实现 | 不把外部结构反向污染领域模型 |

## 接口代码书写准则

```java
@RestController
@RequestMapping("/orders")
public class OrderController {
    private final CreateOrderUseCase createOrderUseCase;

    @PostMapping
    public OrderResponse create(@Valid @RequestBody CreateOrderRequest request) {
        CreateOrderCommand command = request.toCommand();
        return OrderResponse.from(createOrderUseCase.create(command));
    }
}
```

| 准则 | 说明 |
|---|---|
| Controller 只做入口适配 | 不直接写业务规则和数据库访问 |
| Request 转 Command | 外部入参不要直接进入应用层和领域层 |
| Response 从 Result 转换 | 不把领域对象直接暴露给外部 |
| 参数校验放 DTO | 入口形态合法性在 DTO 解决 |
| 业务合法性放领域或应用服务 | 例如状态流转、库存、额度、权限上下文 |

## 请求进入后的代码流动

一次普通写接口不要只理解为“Controller 调 Service”。更完整的流程是：

```text
HTTP 请求
  -> Controller 接收 DTO
  -> Bean Validation 校验请求形态
  -> 全局异常处理统一返回错误
  -> 判断是否需要幂等
  -> DTO 转 Command
  -> Application Service 开启事务并编排用例
  -> Domain 执行业务规则
  -> Repository / ExternalClient 端口访问外部资源
  -> Adapter 落到 JPA / MyBatis / HTTP / MQ / Redis
  -> 外部依赖设置超时和 fallback
  -> Result 转 Response
```

对应文章吸收：

| 流程节点 | 文章内容进入哪里 |
|---|---|
| Bean Validation 校验请求形态 | 参数校验文章的 `@Valid`、`@Validated`、嵌套校验、统一异常 |
| 判断是否需要幂等 | 幂等文章的唯一键、乐观锁、Token、下游序列号 |
| Application Service 和 Domain 分工 | 六边形架构文章的输入端口、输出端口、领域模型、适配器 |
| 外部依赖设置超时和 fallback | Resilience4j 文章的 `TimeLimiter` 和 fallback |
| 批处理进入 Job/Step | Spring Batch 文章的 Job、Step、JobRepository、chunk、skip |
| 扩展能力走扩展点和生命周期 | SPI 文章、动态 jar 文章的 ServiceLoader、ClassLoader、Bean 注册和卸载 |
