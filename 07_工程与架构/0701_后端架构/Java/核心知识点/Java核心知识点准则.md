# Java 核心知识点准则

## 一句话目标

Java 核心知识点应该先回答“怎么开发一个可维护的 Java 系统”，再吸收文章。

不是先把文章分组，再从文章里凑出知识点。

## 核心知识点清单

| 知识点 | 必须回答的问题 | 产出形态 |
|---|---|---|
| 标准项目架构 | 项目怎么分层？包怎么命名？哪些代码放在哪一层？ | 项目目录骨架、分层职责、禁止事项 |
| 代码管理方式 | 单模块还是多模块？Maven 父子工程怎么组织？依赖版本怎么管？ | Maven 结构、模块边界、依赖规则 |
| DDD 开发规范 | 什么时候用 DDD？实体、值对象、聚合、仓储、领域服务怎么写？ | DDD 包结构、类职责、代码模板 |
| 接口开发规范 | Controller、DTO、Command、Response、异常、参数校验怎么写？ | 接口书写准则、请求响应模型 |
| 事务与一致性 | 事务放哪层？幂等怎么选？重复提交和并发更新怎么防？ | 事务边界、幂等选择表 |
| 外部依赖治理 | 第三方接口、消息、文件、数据库慢调用怎么隔离？ | 端口适配器、超时、fallback、重试边界 |
| 批处理能力 | 导入、对账、迁移、日终任务怎么写才可重启、可追踪？ | Spring Batch 作业模型 |
| 扩展机制 | SPI、插件、动态任务、动态 jar 怎么设计生命周期？ | 扩展点、加载、注册、卸载、审计 |
| 测试与质量 | 单元测试、集成测试、契约测试、架构约束怎么放？ | 测试分层和质量门禁 |
| 安全与权限 | 认证、授权、审计、接口权限怎么设计？ | 权限模型和安全边界 |

## 判断顺序

做 Java 项目时，按这个顺序思考：

```text
业务复杂度
  -> 项目架构
  -> Maven 模块
  -> 领域模型
  -> 接口契约
  -> 事务和幂等
  -> 外部依赖
  -> 批处理/插件化等专项能力
  -> 测试、安全、部署和观测
```

## 文章如何进入准则

文章进入知识库时，只能有四种处理方式：

| 处理方式 | 含义 | 示例 |
|---|---|---|
| 补充 | 文章补上某条准则的具体写法 | 参数校验文章补“接口入口防线” |
| 冲突 | 文章观点和准则冲突，或适用范围更窄 | 动态 jar 文章不能被当成通用插件化最佳实践 |
| 不同层次 | 文章讲的是专项能力，不是主架构 | Spring Batch 是批处理能力，不是普通 Web 项目主线 |
| 降权 | 标题夸大、证据不足、不能指导代码 | 只列场景但没有失败模式和生产证据的内容 |

## 当前缺口

| 缺口 | 影响 |
|---|---|
| Maven 系统化资料不足 | 还不能完整指导父工程、子模块、BOM、插件、依赖收敛 |
| DDD 资料不足 | 还不能完整指导聚合边界、领域事件、仓储接口和反腐层 |
| 安全权限资料不足 | 还不能指导认证授权和接口权限 |
| 测试质量资料不足 | 还不能指导测试分层和架构约束验证 |
| 部署可观测资料不足 | 还不能覆盖日志、指标、链路追踪、健康检查和发布回滚 |
