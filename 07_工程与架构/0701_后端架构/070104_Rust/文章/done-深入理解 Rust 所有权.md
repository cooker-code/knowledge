Rust 的“所有权”据称是导致 Rust 学习曲线陡峭的一个重要原因。本文我们以《Rust 程序设计语言》一书的第 4 章节作为蓝本介绍一下这部分知识，在介绍的过程中会补充一些重要的解读便于读者更好地理解。

# 1. 知识储备：堆 & 栈

先解释一下：为什么了解所有权需要储备堆栈的知识？这一点，《Rust 程序设计语言》一书并没有给读者一个“前情提要”式的解释，而是一上来就介绍堆栈的工作原理，从讲解的角度看这做得不够好。

实际上，通过后续的学习我们会发现：在 Rust 语言层面上，一个变量在离开作用域时是否会自动释放，以及所有权的 move 与变量是在栈上还是堆上没有必然的关系，但是，如果不从更底层去了解变量创建和销毁的过程很难理解建立这些机制的动机以及它们能发挥作用的根本原因。这就是为什么所有介绍 Rust 所有权的文章都要求读者具备一定堆栈知识的原因。

关于堆栈，对于 C/C++ 程序员来说会比较熟悉，因为 C/C++ 中的指针、智能指针、对象的创建和销毁等机制都要求开发者必须了解堆栈，对于 Rust 也是一样的。而对于 Java、Python 等语言的程序员来说，由于这些语言拥有 GC 机制，不需要开发者负责对象释放时的资源清理工作，所以对堆栈的了解可能就薄弱一些。

由于堆栈是一个比较大的话题，我们不能在本文展开介绍，这会让读者失去本文关注的焦点。好在，此前的《编程底层概念回顾：虚拟内存、栈、栈帧、堆》一文已经对堆栈进行了必要的介绍，这篇文章中的堆栈知识足以支撑读者理解本文的内容。如果你认为有必要补习一下堆栈知识，可以先阅读一下此文，然后再回来。

别外，本文还会高频提及一个术语：按位复制 ( bitwise copy )，这里提前解释一下，在别的语言里似乎不怎么使用这个词，其实它就是我们常说的“浅拷贝”。按位复制的意思是：按比特位复制，就是 C 语言里的 memcpy 函数的行为：纯纯地内存字节复制，与之对应的就是“深拷贝”。如果没有特别说明，本文所有说的“复制”一般是指“按位复制”。

# 2. “所有权”的由来

## 2.1 问题的产生

考虑以下代码，不管在什么语言里，它们的行为基本上一致：

```
let x = 5;let y = x;println!("{x}");
```

变量 x 被赋予了整型值 `5`，x 的值“复制”了一份给到了 y，y 的值也变成了 `5`， x 和 y 都在栈上，这很平常，无需过多解释。在 Rust 中，“事情开始变得有趣起来”是从下面这个例子开始的：

```
let s1 = String::from("hello");let s2 = s1;println!("{s1}"); // 该行程序报错！
```

几乎和上面是一样的代码，只是类型从 i32 变成了 String，程序就报错了！因为 s1 已不再是一个有效的变量！当执行完 `let s2 = s1` 时，s1 就已失去了 “hello”  字符串的所有权，而不再可用！这是怎么发生的呢？以及为什么要这样处理呢？

首先，作为一个字符串对象，s1 的值（对象实例）是分配在堆上的！无论是在 C/C++ 还是 Java 中，都是如此，Rust 也是这样，因为：String 类型的长度不固定且无法预知，你无法将其分配到栈上，所以 `String::from` 函数会向堆上申请空间来存放字符串 “hello” ，与此同时，在栈上有一个指针会存放字符串 “hello” 在堆上的地址，以便程序可以获取到字符串，到这里，Rust 的处理与 C++ 是非常像的，以下是 s1 被赋值 “hello” 字符串后内存中的情形：

当出现第二个变量 s2 时，我们有两种选择，第一种选择是在栈上复制 s1（指针） 的同时也把它指向的字符串复制一份，形成两个完全独立的变量：

一般，大家把这种处理方式称为“浅拷贝”，也就是按位复制，这和 C++ 中复制一个指针变量的处理方式是一样的，也是大多数语言对于分配在堆上的对象默认的处理方式。不过，**这种处理方式会带来很多麻烦，那就是：由于现在有两个“指针”指向了同一个位于堆上的“值”，程序应该在什么时候销毁它才是“恰当”的？以及以其中一个变量的“名义”销毁之后，另一个变量会怎样？会不会还在别处正被使用着？你看，问题一下就变得复杂起来**。

针对这个问题，主流编程语言使用了两种完全不同的处理方式：一种是以 Java 为代表的 “GC 派”，由语言自己（jvm）追踪和分析变量的使用情况，并在适当的时机销毁它们，这样做的好处是完全不需要程序员操心，不会出现内存泄漏，另一种则是以 C++ 为代表的 “责任越大能力越大”派，即完全依靠程序员自己在代码中管理对象的释放，但这并不是一件简单的事情，C++ 应用中的大量 bug 皆因此而起。

## 2.2  Rust 给出的方案

Rust 选择了自己独有的“第三种”方案：基于所有权的内存管理。这个方案制定了一系列严格的规则，靠编译期的严格检查“硬控”变量的内存使用，这规避了纯手工管理内存的各种潜在内存安全问题，同时又保证了极高的效率。这些规则都是围绕“所有权”这一概念构建的，官方给出的是三条核心原则：

1. Rust 中的每一个值都有一个所有者。
2. 值在任一时刻有且只有一个所有者。
3. 当所有者离开作用域，这个值会被自动销毁 (drop)。

但其实可以浓缩成**两个核心原则**：

1. **每个值都有且只有一个所有者**
2. **所有者离开作用域，值会被自动销毁 (drop)**

这种处理模式深受 C++ 的 RAII（资源获取即初始化）思想的影响，我们详细解释一下。回看前面的示例：

由 `{}` 包裹起来的部分构成了一个作用域（更多的场景是在一个函数体内），基于第一个规则，“hello” 字符串一开始的所有者是 `s1`，当执行到 `let s2 = s1` 时，发生了所有权的转移，`s2` 成了 “hello” 字符串的唯一所有者，此时 `s1` 已不再可用！所以程序才会报错。注意，此时 `s1` 是无效，不是被销毁，真正的销毁（drop）只会在作用域结束时发生，是发生在 `s2` 上。最后，在离开作用域时，Rust 会自动调用一个 `drop` 函数来释放 `s2` 拥有的值。

# 3. 支撑“所有权”的语言机制

Rust 所有权的两个核心原则需要靠语言层面上的硬核支撑才能“落地”，我们来深入了解一下这些相关机制。

## 3.1 自动析构：保证所有者离开作用域时值能被自动释放

Rust 编译器保证：当一个变量离开它的作用域时，会自动对其执行析构操作（Copy 类型除外），有以下几个重要的规则：

* **如果变量的类型实现了 Drop trait，则调用它的 Drop::drop 方法，但这并不妨碍自动析构机制按自己的方式递归销毁它的字段**。
* **如果变量的类型没有实现 Drop trait，则自动生成析构代码 (drop glue)， 递归地销毁变量的所有字段**。
* **不管有没有实现 Drop trait，Rust 编译器始终都会自动生成析构代码并执行，这是一个托底机制**。
* **你不需要（也不应该）在 Drop::drop 里手动 drop 字段，只需专注编写释放资源（文件、连接、套接字等）的代码**。

解释几个地方：

* `Drop` 特质有一个 `drop` 方法，在变量离开作用域时会被 Rust 编译器自动调用，所以，`drop` 方法其实就是一种特定的析构函数。由于有自动析构机制兜底，实际上，大多数情况下，我们都不需要实现 Drop，真需要自己手写 drop 逻辑的场景一般都是需要释放某种类型的资源，比如：关闭文件、数据库连接、网络套接字等。这是很典型的应用析构函数的场景，在 C++ 中也是如此。
* Drop Glue 是指：自动生成的析构代码，是代码，不是机制，对应它的机制是：自动析构。
* 即使变量的类型实现了 Drop trait 但什么都没写，编译器依然会递归地 drop 所有字段，也就是说 drop glue 与 开发者在 drop 函数中自定义的代码无关，只是如果有 drop 函数，在自动析构的时候会“顺带”调用一下。

## 3.2 编译期检查：保证所有权的转移

不像自动析构机制那样有很多的隐式行为，所有权的转移仅从代码上就可以判断出来。所以，自动析构机制是一种运行期机制，而所有权转移则是在编译期就可以检查和确定下来的。这不难理解，在一个作用域中，一个变量被用于二次赋值时，是很容易判断的，编译器直接报错终止编译即可。

并不是什么类型的变量在复制时都会遵循 move 语义，文章开头的 x、y 两个整型变量的复制遵循的就是 copy 语义。导致这种差异的原因是：整型实现了实现了 Copy trait，而 String 类型没有。在 Rust 中，如果一个类型实现了 Copy trait，在复制时执行的就是 copy 语义，如果没有，则执行默认的 move 语义。关于 Copy trait下文会详细介绍。此外，所有权的转移并不只发生在复制时，在向函数传递参数和接收函数返回值时，也会涉及所有权的转移，这部分内容也会在下文单独介绍。

## 3.3 两者的关系

“转移所有权”和“所有者离开作用域时值被自动释放”可以认为是两种独立的语言机制，但是，它们必须联合起来才能实现 Rust 追求的“基于所有权”的内存管理。其中最主要的一个关系链条是：**只有通过 move 始终保持一个值只有一个所有者，才能在让所有者离开作用域时自动释放它所持有的值，否则就会有 double free 的问题**。

# 4. 将所有权语义和类型绑定

让我们重新梳理一下 Copy 特质的作用：实现了 Copy 特质的类型在复制时执行的是 copy ( 按位复制 ) 语义，没有实现 Copy 特质的类型在复制时执行的是 move 语义，也就是说：**Rust 把所有权的 move / copy 语义绑定到了“类型”上**，这是很需要认真思考的问题：为什么所有权的语义和类型是强相关的？原因是：**Rust 认为：“对一个类型做一次按位复制（memcpy），能否得到一个完全合法、独立、语义等价的值”是类型的内在性质，而不是某一次使用时的临时选择**。

**观察不同的数据类型可以发现：不同的类型在进行“按位复制（memcpy）”后表现是不同的。像整型、浮点、布尔这些数据类型，它们“按位复制（memcpy）”就是没有问题，这很直观，很难再进一步解释了，你可以理解为它们在内存中的“形态”太简单了，“按位复制（memcpy）”就是复制了它们的“全部”，干脆且纯粹；但换成 String 类型，“按位复制（memcpy）”就会带来问题：因为没有完整地复制全部数据（分配在对堆上的字符串），破坏了复制出来的两个变量的“完整性”和“独立性”，使得它们始终相互耦合，相互影响（指针悬空、二次释放等）。正是基于这样的一种根源性认识，Rust 认为：那就索性追加一个“所有权”的语义，把 String 类型的值始终牢牢地绑定在一个变量上，允许将它从一个变量转移给另一个变量，但绝不允许两个以上的变量同时拥有，问题不就解决了吗？这就是为什么 Rust 把所有权的 move / copy 语义绑定到了“类型”上而且能绑定到类型上的根本原因！**

# 5. Copy 和 Drop 之间的关系

**“是否会进行自动析构”和“赋值时是 copy 还是 move”是两个相互独立但又有交叉的维度**，其中在交叉处的逻辑非常烧脑，这个要仔细分析，但跳过这部分内容其实对编程影响不大，只需记住结论就可以：

* **实现了 Drop 的复合类型：允许其字段是任何类型（Copy 或非 Copy均可）**
* **实现了 Copy 的复合类型：其所有字段必须全是 Copy 类型**

这里面的关键逻辑是：**“复合类型里面有没有任何字段在离开作用域时需要被清理？”只要有一个，那整个类型就得进行自动析构**。

# 6. 参数与返回值的“所有权”转移

向函数传递参数以及接收函数的返回值也是一种变量复制的过程，也必然会涉及到“所有权”转移。这部分不复杂，看几个示例就清楚了。

## 6.1 作为参数传递给函数时

当变量作为实参以“按值传递”的方式传给一个函数时，也是一个“复制”的过程，所以会遵循相同的 move / copy 语义规则，下面的示例代码已经阐述的非常直白了：

```
fn main() {  let s = String::from("hello");  // s 进入作用域  takes_ownership(s);             // s 的值移动到函数里 ...                                  // ... 所以到这里不再有效  let x = 5;                      // x 进入作用域  makes_copy(x);                  // x 应该移动函数里，                                  // 但 i32 是 Copy 的，所以在后面可继续使用 x} // 这里, x 先移出了作用域，然后是 s。但因为 s 的值已被移走，  // 所以不会有特殊操作fn takes_ownership(some_string: String) { // some_string 进入作用域  println!("{}", some_string);} // 这里，some_string 移出作用域并调用 `drop` 方法。占用的内存被释放fn makes_copy(some_integer: i32) { // some_integer 进入作用域  println!("{}", some_integer);} // 这里，some_integer 移出作用域。不会有特殊操作
```

## 6.2 作为返回值从函数返回时

当返回值从函数返回时同样如此。但是，有一个细节需要注意，下面的代码中，`takes_and_gives_back()` 函数中的 `a_string` 在离开函数体时，是移交了所有权，不要错误地理解成它将被自动析构，**它不是函数体中的中间变量，它是返回值！作为返回值它要执行 move 语义，把所有权移交出去，而不是被自动析构掉**。

```
fn main() {  let s1 = gives_ownership();         // gives_ownership 将返回值                                      // 移给 s1  let s2 = String::from("hello");     // s2 进入作用域  let s3 = takes_and_gives_back(s2);  // s2 被移动到                                      // takes_and_gives_back 中,                                      // 它也将返回值移给 s3} // 这里, s3 移出作用域并被丢弃。s2 也移出作用域，但已被移走，  // 所以什么也不会发生。s1 移出作用域并被丢弃fn gives_ownership() -> String {           // gives_ownership 将返回值移动给                                           // 调用它的函数  let some_string = String::from("yours"); // some_string 进入作用域  some_string                              // 返回 some_string 并移出给调用的函数}// takes_and_gives_back 将传入字符串并返回该值fn takes_and_gives_back(a_string: String) -> String { // a_string 进入作用域  a_string  // 返回 a_string 并移出给调用的函数}
```

# 7. 引用与借用

## 7.1 引用

考虑上一节函数的传参与返回，由于有了“所有权”的概念，一个变量从当前作用域以参数形式传给一个函数时，当前作用域中的变量随即失去了所有权，但很多时候，我们并不想这样，因为在接下来的处理中，我们还需要继续使用它，这时候，你只能把它作为“返回值”从执行完的函数重新取回来，而如果这个函数还有别的结果要返回，你就得使用一个元组来封装这个变量和其他的返回值，这就很“扭曲”了，正常编写程序没人会这么干。为了解决这类问题，Rust 引入了“引用”：

> let s1 = String::from(“hello”);
>  let s = &s1; // s 是 s1 的一个引用

考虑一个“求字符串长度”的函数，仅从函数的设计目标就可以推断出：

1. 函数仅需以只读方式使用字符串
2. 求出字符串长度后，这个字符串肯定在别处还要使用（你不能仅仅因为计算了一下长度就把它销毁了）

这个以参数形式传进函数的字符串就是一个典型“引用”类型，也就是下面代码中的 `&s1`：

```
fn main() {    let s1 = String::from("hello");    let len = calculate_length(&s1); // 将引用而不是变量本身传到函数中    println!("The length of '{}' is {}.", s1, len);}fn calculate_length(s: &String) -> usize {    s.len()}
```

**引用不同于一般变量的关键在于：它并不获得值的“所有权”，所以当一个引用离开作用域时并不会导致值被自动销毁**！下面的代码说明了这一点：

```
fn calculate_length(s: &String) -> usize { // s 是对 String 的引用    s.len()} // 这里，s 离开了作用域。但因为它并不拥有引用值的所有权，  // 所以什么也不会发生
```

最后，连带性的解释一下“借用”（borrowing），**使用引用的过程和动作就是借用**。当一个函数声明了一个引用类型的参数，而当你想调用它时，就得把变量的引用传给它，从函数的视角看：它是“借用”了这变量，变量的所有权还在你（调用方）的手里。

## 7.2 可变引用

通过引用只能读取值，不能修改值，这是出于安全的考量，不过，有时候，这确实不太方便，所以 Rust 引入了“可变引用”，但追加了很限制条件以保证内存安全。

> let mut s1 = String::from(“hello”);
>  let s = &mut s1; // s 是 s1 的一个可变引用

**限制规则一：在同一时间，一个值只能有一个可变引用**

如果在同一个作用域里，针对同一个值创建了两个或两个以上可变引用，Rust 就会报错。Rust 引入这个限制的初衷就是要：避免“数据竞争”（data race），这是多线程编程中的经典问题。在多线程环境下，如果有两个以上的指针指向同一个数据，且至少有一个指针会写数据，在没有加 synchronized 保护的情况下就会出现这个问题。在多线程中执行转账操作时，“账户余额”就是一个典型的“被竞争的数据”，如果不对“账户余额”施加同步保护，经过多轮的相互转账，最后所有账户的金额总和会改变，这种情况叫“数据腐蚀”，是数据竞争导致的一种错误。所以，Rust 直接“一刀切”，索性不允许两个以上的可变引用同时存在。

**限制规则二：如果一个值已经有了不可变引用，就不能再声明可变引用，除非声明可变引用前所有不可变引用已不再使用**

# 8. Slice 类型：只引用一部分数据但也占用一个引用“名额”

## 8.1 Slice 是一种什么类型？

我们关于所有权的介绍只剩下最后一个话题了：Slice 类型。在初次接触这部分内容时，很多人会产生一个错误的认知：把 Rust 中的 Slice 理解成其他语言里与集合有关的那种 slice，比如在 Python 和 Scala 里都有 slice，把它认为是一种操作或是集合类型，但是**只要你注意到：可以有字符串切片、数组切片等不同类型的切片时，你就应该意识到，Rust 中的 Slice 不是一个简单的东西，也许你会联想到 C++ 中的模板函数，但它不是函数，它是一种数据抽象，而不是行为抽象，所以它更偏向是一种类型而不是函数**。

在《C++ Primer》一书中，把引用和指针归为 **Compound Type**，意思是：它们是基于另一个类型构造出来的类型，稍微想一下指针或引用的声明形式就能理解这个说法的意思，**在入门阶段，我们可以把 Rust 中的 Slice 类型理解成是 C++ 中的 Compound Type，也就是基于另一个类型构造出来的类型，是与指针、引用平行的一种类型**。当然，Slice 类型的真实情况会更复杂，因为涉及到 DST 和 胖指针等机制，我们暂时先不展开。

我们以 “字符串 slice” 为例来了解一下 Slice 类型。从一个字符串中截取一部分内容是很常见的一种操作，在其他语言里都有对应的模板函数，在 Rust 里，有字符串 slice:
