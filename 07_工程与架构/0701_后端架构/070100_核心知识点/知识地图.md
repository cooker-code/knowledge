# 070100_后端架构知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 07_工程与架构 / 0701_后端架构 |
| 解决的问题 | 先定义后端服务的工程流程，再把文章吸收到 Python、Java、SpringBoot、Node 等具体路线 |
| 典型输入 | 本地后端文章、已有 AGENTS、已有核心知识点、后续新增资料 |
| 典型产出 | 后端路线、流程节点、判断准则、风险边界、待补证据 |
| 用户阅读目标 | 读到一篇后端文章时，先知道它补的是语言机制、应用框架、接口、数据、扩展还是生产治理 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| B00 服务边界与非功能约束 | 服务职责、输入输出、性能、安全、稳定性和不做什么是什么 | 后端架构从边界开始，不从框架开始 | 架构、案例、项目结构文章进入，缺边界则降权 |
| B01 技术栈与运行时选择 | Python、Java、SpringBoot、Node 分别承担什么角色 | 技术本体优先于标题关键词 | 按语言/框架主问题路由 |
| B02 项目结构与模块边界 | 包结构、分层、依赖方向、插件和扩展点如何设计 | 代码组织要让变化边界清楚 | Maven、FastAPI 项目结构、Spring Boot 分层、动态模块文章进入 |
| B03 接口入口与请求契约 | API、DTO、校验、认证、幂等和错误语义如何处理 | 入口负责协议适配和合法性，业务规则不能塞进入口 | FastAPI、Spring Boot、接口规范文章进入 |
| B04 业务编排与事务边界 | Service、UseCase、Pipeline、批处理和事务如何组织 | 编排层要明确事务、副作用和补偿 | Pipeline、批处理、规则引擎文章进入 |
| B05 数据访问、缓存与外部依赖 | 数据库、缓存、文件、对象存储、第三方 API 如何接入 | 外部依赖必须有生命周期、超时、失败和资源边界 | SQLAlchemy、HikariCP、MinIO、Resilience4j、缓存文章进入 |
| B06 批处理、异步任务与扩展点 | 后台任务、批处理、SPI、动态 jar 如何可追踪可回滚 | 扩展能力必须管理生命周期和审计 | Spring Batch、SPI、动态加载、Celix 文章进入 |
| B07 测试、安全、观测、部署 | 服务上线后如何验证、监控、回滚和排障 | 生产能力要有可验证信号 | 安全、测试、部署和观测文章进入，不足则转 `0703` |

## 当前已吸收知识点

| 流程节点 | 知识点 | 关系 | 来源路线 | 吸收结论 |
|---|---|---|---|---|
| B01 | Java 与 Spring Boot 需要拆开：一个是语言/机制，一个是应用框架 | 纠偏 | Java、SpringBoot | Spring Boot 已独立成具体技术目录，避免 Java 目录继续膨胀 |
| B03 | FastAPI 和 Spring Boot 都在解决入口契约，但运行时校验机制不同 | 补充 | Python、SpringBoot | 后续可以横向对比 Pydantic 与 Bean Validation |
| B04 | Pipeline、规则引擎、批处理都是业务编排方式，但生命周期要求不同 | 补充 | Java、SpringBoot | 不能把设计模式、规则引擎、批处理混成同一类文章 |
| B05 | 外部依赖治理要看资源生命周期、超时、失败和回滚，不只看调用方式 | 纠偏 | Python、Java、SpringBoot | Resilience4j、连接池、对象存储和缓存文章都要落到失败模式 |

## 使用方式

- 先读本文件的宏观流程，再进具体路线目录。
- 新文章不要单独写摘要，先归入路线和流程节点。
- 如果文章主问题不是后端本体，按 `目录划分.md` 迁出到前端、工程保障、数据分析、数据库、数据工程或 Agent 工程。
