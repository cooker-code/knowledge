# 工程与架构

## 模块定位

本模块只放软件系统本体的设计、实现和运行保障知识，不再承载通用工具产品目录。

| 二级目录 | 职责 | 不放什么 |
|---|---|---|
| [0701_后端架构](0701_后端架构/AGENTS.md) | Java、Python、Node 等后端服务的分层、接口、数据访问、配置、任务、性能和工程化实现 | 单纯工具使用、个人效率工具 |
| [0702_前端工程](0702_前端工程/AGENTS.md) | React、Vue、Nuxt、TypeScript、前端 AI 应用、前端工程化与质量；按具体路线维护组件、渲染、状态、交互、构建、测试和发布 | 纯设计素材、AI 编程 Skill、后端认证框架、跨系统测试质量；这些按技术本体重路由 |
| [0703_工程实践与质量保障](0703_工程实践与质量保障/AGENTS.md) | 接口契约与网关、测试质量、安全权限、部署发布与环境、可观测性与 AIOps、并发运行时；按具体工程保障路线维护验证、排障、回滚和规则沉淀 | 前端框架、AI 编程工具、数据平台、OLAP/数据库、电脑工具、分析方法；这些按技术本体重路由 |

## 目录治理结论

原 `03_接口与网关`、`05_可观测性`、`06_部署发布`、`07_安全权限`、`08_测试质量`、`12_并发与运行时` 已合并到 `0703_工程实践与质量保障`，并在 `0703` 下拆成具体工程保障路线。

合并原因：

- 这些目录都属于工程系统的横切保障能力，单独作为二级目录时文章数量和技术对象都不稳定。
- 多数文章不是在讲一个独立技术，而是在补系统交付流程中的某个节点。
- 后续不按旧目录扩展，统一按流程节点吸收：接口契约、质量验证、安全控制、运行观测、发布回滚、并发运行时。

## 宏观流程

| 节点 | 要回答的问题 | 主要落点 |
|---|---|---|
| E00 边界确认 | 系统职责、输入输出、非功能约束和不做什么是什么 | 后端架构、前端工程、工程实践 |
| E01 实现组织 | 模块、分层、依赖、状态、数据访问和扩展点如何组织 | 后端架构、前端工程 |
| E02 交互契约 | API、协议、网关、幂等、鉴权和错误语义如何设计 | 工程实践与质量保障 |
| E03 质量验证 | 单测、集成测试、契约测试、回放、压测和验收标准如何覆盖 | 工程实践与质量保障 |
| E04 运行保障 | 日志、指标、链路、告警、部署、回滚、容量和故障处置如何闭环 | 工程实践与质量保障 |
| E05 复盘沉淀 | 缺陷、事故、性能瓶颈和架构偏差如何变成可复用准则 | 核心知识点 |

## 与电脑工具的边界

| 文章主问题 | 应归类 |
|---|---|
| 讲某个工具怎么安装、配置、调用、自动化，例如 uv、OpenCLI、Obsidian、Mermaid | `09_电脑工具` |
| 讲工具背后的工程机制，例如构建依赖解析、测试体系设计、接口契约、运行时并发模型 | 按主问题归入 `0701`、`0702` 或 `0703` |
| 讲 AI 编程工具如何组织上下文、规则、评估和多 Agent 工作流 | `02_Agent与AI工程/0205_AI编程工具` 或相关 Agent 目录 |

## 使用规则

- 新文章先判断主问题是在“系统工程机制”还是“电脑工具使用”。
- 能抽出长期工程准则的内容进入 `核心知识点/`；工具安装、命令清单、版本资讯只保留原文锚点。
- 不因为标题包含“架构、实战、最佳实践”就直接精读，仍按问题指纹、证据和可迁移性判断。
