# 知识目录划分

本文件是 4 个基础文件之一，只维护分类准则和目录边界。基础文件包括：根 `AGENTS.md`、`用户画像.md`、`目录划分.md`、`技术定位卡.md`。

## 划分准则

不要按文章标题、原始文件夹名或作者话术划分，要按“文章实际讨论对象”和“技术在系统里承担的职责”划分。

原始材料只是初始化输入，不是最终知识结构。整理时要批判性判断标题、目录、作者观点是否准确。

| 一级类目 | 二级类目 | 核心问题 | 典型技术/主题 |
|---|---|---|---|
| 数据工程与数仓 | 数据集成、离线数仓、实时计算、湖仓表格式、调度编排、元数据血缘与治理、数据质量与治理 | 数据如何进入平台、被加工、建模、治理、调度、交付 | Hive、Spark、Flink、Kafka、Paimon、Iceberg、Hudi、HDFS、YARN、Kyuubi、DolphinScheduler、Airflow、SeaTunnel、DataX、Ranger |
| OLAP 与数据库 | OLAP 引擎、关系数据库、嵌入式分析、搜索与索引、缓存与 KV、存储引擎、查询优化、宽列数据库 | 数据如何被高性能查询、更新、索引、优化和服务化 | Doris、StarRocks、ClickHouse、DuckDB、SQLite、PostgreSQL、MySQL、TiDB、HBase、Redis、Elasticsearch |
| 数据分析与 BI | 语义层与智能问数、指标体系、报表看板、可视化、归因分析、实验分析、统计方法、经营分析 | 指标如何定义、分析、展示、归因和驱动决策 | 指标平台、Superset、Grafana、Power BI、Tableau、Datart、ChatBI、Text-to-SQL、AB 实验、置信区间、双重差分、合成控制 |
| Agent 与 AI 工程 | Agent 框架、工具调用、RAG 与知识库、评估与观测、AI 编程工具、工作流编排、Prompt Engineering、CoT/ToT/ReAct、Context Engineering、Harness Engineering、安全与权限、Memory Management | 如何把模型能力工程化成可执行、可评估、可控的应用或工作流 | LangGraph、DeerFlow、Vercel AI SDK、Open Agents、Pi、Dify、n8n、MCP、RAGFlow、Claude Code、Cursor、OpenCode、Hermes、workBuddy、OpenAI Agents SDK、CoT、ToT、ReAct |
| LLM 与大模型 | 模型能力、微调与训练、模型服务、多模态、模型评测 | 模型本身如何工作、如何优化、如何部署和评测 | GPT、Claude、Gemini、DeepSeek、Qwen、Llama、vLLM |
| 工程与架构 | 后端架构、前端工程、工程实践与质量保障 | 通用软件系统如何设计、实现、验证、交付、观测和稳定运行 | Spring Boot、FastAPI、Java、React、Vue、Nuxt、TypeScript、前端 AI 应用、API 契约、测试质量、安全权限、可观测性、AIOps、部署回滚、ThreadPoolExecutor |
| 电脑工具 | 开发工具与 CLI、文档与知识工具 | 个人电脑和开发环境中的工具如何安装、配置、自动化、验证和长期维护 | Git、gh、uv、npm、Maven、OpenCLI、Playwright CLI、Obsidian、Mermaid、Quarkdown、Doc2Wiki |
| 机器学习 | 特征工程、传统模型、模型评估、可解释性、MLOps、推荐与预测 | 如何用数据训练模型、解释模型、上线模型 | XGBoost、LightGBM、sklearn、MLflow、特征平台、模型评估 |
| 个人成长 | 职业发展、学习方法、效率工具、管理沟通 | 职业、学习、效率和管理沟通类文章如何沉淀 | 职业规划、管理经验、读书方法、心理文章、效率工具 |

当前一级目录已经定型，只能使用文件系统中已有的 9 个一级目录。表中一级类目的职责是稳定技术域，不作为文章临时收纳空间。

当前二级类目以文件系统真实目录为准。二级目录代表用户的技术认知领域，不是文章标题分组。表中没有落地的概念，例如 `LLM 与大模型 / 提示词`、`数据分析与 BI / 因果推断`、`个人成长 / 心理建设`，不能直接新建为正式目录；若后续确实需要新增，必须先补二级类目 `AGENTS.md` 和知识地图，并在路由表中标为“二级认知领域候选”。

三级目录是可由整理者主动发现和细分的层级。新增三级节点必须落在已有二级认知领域下，并能说明它是一个可长期积累的具体技术、稳定专题、关键机制或工程方法；不能把低置信分类、一次性资讯或营销名词创建成三级目录。

## 归类优先级

归类不是按关键词匹配，也不是按工具名固定归类。优先级如下：

| 优先级 | 判断项 | 要问的问题 |
|---|---|---|
| 1 | 技术本体 | 这个技术自己主要解决什么问题 |
| 2 | 文章主问题 | 这篇文章重点讨论什么问题 |
| 3 | 使用场景 | 技术被用在什么场景里，这个场景是否改变了文章主问题 |
| 4 | 关键词 | 标题和正文出现的词只能辅助判断 |

## 低置信与待处理记录

不要为了让目录看起来完整而硬塞文章。遇到以下情况，先标记为低置信，只进入路由表或处理日志，不直接写正式核心知识点，也不创建 `99`、`xx99_待人工判断`、`xx99_待归类` 这类临时目录：

| 情况 | 例子 | 处理 |
|---|---|---|
| 标题和正文主问题不一致 | 标题讲 AI，正文实际讲数据平台或工程工具 | 先标低置信，后续人工确认主问题 |
| 正文关键词抢分类 | Paimon 文章正文出现 Doris、MySQL、HBase | 分类以文章主问题为准，正文只辅助 |
| 产品、工具、方法混杂 | SeekDB、ima、OpenClaw、Warp、Plandex | 拆成产品体验、技术机制、工作流方法再归类 |
| 工程基础过宽 | OkHttp、uv、Maven、ThreadPoolExecutor、structlog | 先分清主问题：系统机制进“工程与架构”，工具安装/命令/自动化进“电脑工具” |
| 统计/分析方法跨域 | 贝叶斯估计、置信区间、双重差分、合成控制 | 如果服务指标/实验/经营决策，归“数据分析与 BI”；如果服务建模训练，归“机器学习” |
| 知识工具跨域 | Mermaid、Quarkdown、Doc2Wiki、代码 Wiki、Obsidian | 如果核心是本机文档/知识工具使用，归“电脑工具 / 文档与知识工具”；如果核心是 Agent 自动维护知识或 RAG 架构，归“Agent 与 AI 工程 / RAG 与知识库” |

低置信不是失败，而是防止误分类进入长期知识库。低置信结论只能作为路由状态保存在表格中，不能落成一个长期目录。

## 全量路由分类准则

批量处理时采用两阶段判断：

```text
第一阶段：标题 + 文件路径 + 明确技术名 -> 初步主问题
第二阶段：正文片段 -> 只辅助判断图片、投入建议、冲突点和置信度
```

原因：10% 灰测发现，如果把正文全部用于分类，相邻技术会抢走主类目，导致文章被误路由。

全量路由允许低置信，但正式沉淀不允许低置信。低置信文章必须先补充：

- 文章主问题。
- 技术本体。
- 不应归入的相邻类目。
- 需要用户或后续文章验证的点。

典型例子：`Airflow`。

`Airflow` 的技术本体是工作流调度、DAG 编排、任务依赖管理，不是机器学习。

| 文章主问题 | 应归类 | 判断 |
|---|---|---|
| DAG、Scheduler、Executor、Worker、补数、任务依赖 | 数据工程与数仓 / 调度编排 | 主问题是调度 |
| 调度 Hive、Spark、Flink、SQL、ETL/ELT | 数据工程与数仓 / 调度编排 | 主问题是数据任务编排 |
| 编排训练、特征生成、模型上线，并与 MLflow 等结合 | 机器学习 / MLOps | 主问题是机器学习流水线，Airflow 是编排工具 |
| Airflow 源码、插件、分布式执行架构 | 工程与架构 或 数据工程与数仓 / 调度编排 | 看文章重点是通用系统架构还是数据调度 |

## 数据工程与数仓 vs OLAP 与数据库

这两个目录容易混，边界按“主职责”判断。

| 判断项 | 数据工程与数仓 | OLAP 与数据库 |
|---|---|---|
| 关注点 | 数据进入、加工、分层、调度、质量、治理 | 查询性能、存储结构、索引、优化器、并发、更新 |
| 典型问题 | 表怎么分层？任务怎么跑？数据怎么同步？血缘怎么追？质量怎么控？ | 查询为什么慢？索引怎么设计？Join 怎么优化？并发怎么扛？ |
| 产出对象 | 数据表、任务、模型、血缘、质量规则、同步链路 | 查询服务、执行计划、索引、物化视图、存储格式 |
| 使用者 | 数据开发、数仓工程师、数据治理、平台工程 | 数据库工程师、OLAP 平台、查询服务、分析应用 |
| 时间特征 | 常见是批处理、准实时、流批一体的生产链路 | 常见是交互式查询、即席分析、服务化查询 |

一句话：

- `Hive` 默认放在“数据工程与数仓”，因为它通常承担离线数仓表、元数据、批 SQL、分区和建模落地。
- `Doris/StarRocks/ClickHouse` 默认放在“OLAP 与数据库”，因为它们通常承担查询加速、交互分析、服务化和高并发。
- 如果一篇文章讲 `Flink CDC -> Doris`，主角是 CDC Pipeline、同步链路和下游写入一致性就放“数据工程与数仓 / 实时计算 / Flink CDC”；主角是 Doris 查询模型、索引、Compaction 就放“OLAP 与数据库”。

## Kubernetes / Docker 归类规则

Kubernetes 和 Docker 是横跨多个目录的基础设施，不按工具名固定归类，按文章主问题判断。

| 文章主问题 | 应归类 | 判断 |
|---|---|---|
| Dockerfile、镜像、Compose、Registry、镜像拉取、基础镜像、容器化部署、发布回滚、环境一致性 | `07_工程与架构/0703_工程实践与质量保障/070306_部署发布与环境` | 主问题是通用软件交付和运行环境治理 |
| Kubernetes 基础对象、Deployment、Service、Ingress、Helm、Operator、集群运维、滚动发布、弹性伸缩 | `07_工程与架构/0703_工程实践与质量保障/070306_部署发布与环境` | 主问题是通用平台工程和部署运维 |
| YARN、YARN vs Kubernetes、Hadoop 生态队列、Application、Container、日志聚合、资源隔离 | `03_数据工程与数仓/0302_离线数仓/030201_Hadoop&HDFS` | YARN 是 Hadoop 生态资源调度层，跟随 Hadoop 沉淀 |
| Spark/Flink/Kafka on K8s、具体计算引擎资源运行、Shuffle、Checkpoint、状态、Consumer Lag | 对应数据技术目录，例如 `0303_实时计算/030301_Flink`、`0302_离线数仓/030205_Spark`、`0303_实时计算/030304_Kafka` | 如果文章聚焦具体引擎运行时机制，放到对应技术目录 |
| Docker/K8s 用于 AI Agent 沙箱、权限隔离、凭证隔离、执行审计 | `02_Agent与AI工程/0210_安全与权限` | 主问题是 Agent 执行安全，不是通用容器教程 |
| Docker/K8s 用于部署 MCP Server、n8n、Dify、RAG 服务等 AI 工程组件 | 对应 `02_Agent与AI工程` 技术目录 | 主问题是 AI 工程组件部署；只有文章重点讲通用容器治理时才转 `0703` |
| Docker/K8s 用于部署 vLLM、Ollama、模型服务、推理网关 | `01_LLM与大模型/0105_模型服务` | 主问题是模型服务运行、成本、性能和部署形态 |
| 本机 Docker Desktop、OrbStack、Colima、容器命令行使用体验 | `09_电脑工具/0901_开发工具与CLI` | 主问题是个人开发环境工具 |

一句话：

- `Docker` 默认不是后端架构语言路线；如果讲镜像和发布治理，进 `0703_部署发布与环境`。
- `Kubernetes` 默认不是单独技术池；如果讲通用集群和应用部署，进 `0703_部署发布与环境`；如果讲 YARN/Hadoop 生态资源调度，进 `0302_离线数仓/030201_Hadoop&HDFS`；如果讲具体数据引擎运行时，进对应技术目录。
- 如果容器只是某个技术的部署手段，优先放到那个技术本体目录。

## 归类示例：Hive

Hive 是一个示例，用来说明跨边界技术如何归类，不代表所有技术都要单独写固定规则。

Hive 多数情况下归入：

```text
knowledge/03_数据工程与数仓/0302_离线数仓/030202_Hive/
```

纵向定位：

```text
数据平台
  -> 数据工程与数仓
    -> 离线数仓
      -> Hive
        -> 元数据：Hive Metastore
        -> 查询入口：HiveServer2 / JDBC / Kyuubi
        -> 存储承载：HDFS / ORC / Parquet / 分区
        -> 计算执行：MapReduce / Tez / Spark
        -> 建模落地：ODS / DWD / DWS / ADS / 拉链表 / 快照事实表
        -> 性能治理：参数、Join、倾斜、小文件、排序、分区裁剪
```

横向对标：

| Hive 维度 | 对标技术 | 关键差异 |
|---|---|---|
| 离线 SQL 数仓 | Spark SQL、Trino、Presto | Hive 更偏传统离线数仓和元数据生态，Spark SQL 更偏计算引擎，Trino/Presto 更偏交互式联邦查询 |
| 元数据服务 | Hive Metastore、Glue Catalog、Iceberg REST Catalog | Hive Metastore 是大数据生态事实标准，现代湖格式逐渐强化独立 Catalog 能力 |
| 表格式/湖仓 | Iceberg、Hudi、Paimon、Delta Lake | Hive 表偏目录/分区管理，湖格式增强事务、增量、快照、演进和流批一体 |
| 查询加速 | Doris、StarRocks、ClickHouse | Hive 不适合高并发低延迟查询，OLAP 引擎更适合报表和即席查询 |
| 数据同步 | SeaTunnel、DataX、Sqoop、Flink CDC | Hive 常是目标数仓或中间层，不是同步引擎本身 |

## 目录组织规则

知识内容按“一级类目 -> 二级类目 -> 具体技术/主题节点 -> AGENTS.md”组织。

| 层级 | 作用 | 示例 |
|---|---|---|
| 一级类目 | 大问题域 | `03_数据工程与数仓` |
| 二级类目 | 用户的技术认知领域，统一使用 4 位前缀：前两位继承一级目录编号，后两位是二级编号 | `0302_离线数仓`、`0306_调度编排`、`0308_元数据血缘与治理` |
| 具体技术/主题节点 | 整理者需要主动发现和细分的技术对象或稳定专题，统一使用 6 位前缀：前四位继承二级类目编号，后两位是该二级类目下的节点编号 | `030202_Hive`、`030203_Kyuubi`、`030205_Spark` |
| `AGENTS.md` | 技术纵向知识地图 | 定位、架构、模块、上下游、对标、官网/GitHub |

`文章/`、`编号_核心知识点/` 是固定内容容器，不属于分类节点；`编号_核心知识点/` 的编号来自父目录，用于排序和定位。

`编号_核心知识点/知识地图.md` 是固定知识地图入口。一级目录使用 `一级编号 + 00`，例如 `0300_核心知识点/知识地图.md`；二级目录使用 `二级编号 + 00`，例如 `030300_核心知识点/知识地图.md`；三级目录使用自身 6 位编号，例如 `030301_核心知识点/知识地图.md`。

一级目录下的 `xx00_核心知识点/` 长期只应保留 `知识地图.md`。该文件辅助一级 `AGENTS.md`，用于描述一级技术域的宏观流程、二级/三级节点吸收位置和待补缺口；单篇文章知识点应进入二级或三级目录，不能长期沉淀在一级 `xx00_核心知识点/` 中。

`AGENTS.md` 不是文章摘抄，而是该技术的长期入口。它应该回答：

- 这个技术是什么，解决什么问题。
- 它在全局架构中处在哪一层。
- 它由哪些核心模块组成。
- 上游、下游、依赖和产出是什么。
- 应该与哪些技术横向对标。
- 官方文档、GitHub、关键架构图在哪里。
- 本地已沉淀了哪些核心知识点。

`编号_核心知识点/` 用来存文章提炼结果。每个文件应快速回答：

- 文章主问题是什么。
- 可复用机制是什么。
- 对技术理解新增了什么。
- 适用边界和风险是什么。
- 可追查锚点是什么。
