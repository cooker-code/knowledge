## 工程与架构 vs 电脑工具的边界

| 文章主问题 | 应归类 |
|---|---|
| 讲某个工具怎么安装、配置、调用、自动化，例如 uv、OpenCLI、Obsidian、Mermaid | `09_电脑工具` |
| 讲工具背后的工程机制，例如构建依赖解析、测试体系设计、接口契约、运行时并发模型 | 按主问题归入 `0701`、`0702` 或 `0703` |
| 讲具体 AI 编程工具如何组织上下文、权限、Hook、插件和运行状态 | `02_Agent与AI工程/0205_AI编程工具` 或相关 Agent 目录 |
| 讲跨工具的 AI 编码方式，例如 Workflow、VibeCode、SpecCode、需求到交付闭环 | `02_Agent与AI工程/0206_AI编码方式` |

## 电脑工具 vs 其他一级类目的边界

| 文章主问题 | 应归类 |
|---|---|
| 工具如何安装、配置、调用、自动化、接入本机工作流 | `09_电脑工具` |
| 工具背后的模型调用、Agent 工作流、上下文工程、评估机制 | `02_Agent与AI工程` |
| 工具背后的系统架构、接口契约、并发、测试、发布、可观测性 | `07_工程与架构` |
| 工具服务的是数据开发、OLAP 查询、指标分析，且主问题是数据链路 | 数据工程、OLAP 或数据分析目录 |
| 只是软件清单、版本资讯、营销推荐 | 略读或低置信记录，不直接沉淀，也不创建临时目录 |

当前一级目录已经定型，只能使用文件系统中已有的 9 个一级目录。表中一级类目的职责是稳定技术域，不作为文章临时收纳空间。

当前二级类目以文件系统真实目录为准。二级目录代表用户的技术认知领域，不是文章标题分组。表中没有落地的概念，例如 `LLM 与大模型 / 提示词`、`数据分析与 BI / 因果推断`、`个人成长 / 心理建设`，不能直接新建为正式目录；若后续确实需要新增，必须先补二级类目 `AGENTS.md` 和知识地图，并在路由表中标为“二级认知领域候选”。

三级目录是可由整理者主动发现和细分的层级。新增三级节点必须落在已有二级认知领域下，并能说明它是一个可长期积累的具体技术、稳定专题、关键机制或工程方法；不能把低置信分类、一次性资讯或营销名词创建成三级目录。
