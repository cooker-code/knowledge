# 知识目录划分

## 划分准则

不要按文章标题、原始文件夹名或作者话术划分，要按“文章实际讨论对象”和“技术在系统里承担的职责”划分。

原始材料只是初始化输入，不是最终知识结构。整理时要批判性判断标题、目录、作者观点是否准确。

| 一级类目 | 二级类目 | 核心问题 | 典型技术/主题 |
|---|---|---|---|
| 数据工程与数仓 | 数据集成、离线数仓、实时计算、湖仓表格式、调度编排、元数据血缘与治理、数据质量与治理、资源与运维 | 数据如何进入平台、被加工、建模、治理、调度、交付 | Hive、Spark、Flink、Kafka、Paimon、Iceberg、Hudi、HDFS、YARN、Kyuubi、DolphinScheduler、Airflow、SeaTunnel、DataX、Ranger |
| OLAP 与数据库 | OLAP 引擎、关系数据库、嵌入式分析、搜索与索引、缓存与 KV、存储引擎、查询优化、宽列数据库、运维排障 | 数据如何被高性能查询、更新、索引、优化和服务化 | Doris、StarRocks、ClickHouse、DuckDB、PostgreSQL、MySQL、TiDB、HBase、Redis、Elasticsearch |
| 数据分析与 BI | 指标体系、语义层与智能问数、报表看板、可视化、归因分析、实验分析、统计方法、因果推断、用户画像、经营分析 | 指标如何定义、分析、展示、归因和驱动决策 | 指标平台、Superset、Grafana、Power BI、Tableau、Datart、ChatBI、Text-to-SQL、AB 实验、置信区间、双重差分、合成控制 |
| Agent 与 AI 工程 | Agent 框架、工具调用、RAG 与知识库、工作流编排、AI 编程工具、上下文工程、评估与观测、安全与权限 | 如何把模型能力工程化成可执行、可评估、可控的应用或工作流 | LangGraph、Dify、n8n、MCP、RAGFlow、Claude Code、Cursor、OpenCode、OpenAI Agents SDK |
| LLM 与大模型 | 模型能力、推理与上下文、提示词、微调与训练、模型服务、多模态、模型评测、成本与性能 | 模型本身如何工作、如何优化、如何部署和评测 | GPT、Claude、Gemini、DeepSeek、Qwen、Llama、vLLM |
| 工程与架构 | 后端架构、前端工程、工程实践与质量保障 | 通用软件系统如何设计、实现、验证、交付、观测和稳定运行 | Spring、FastAPI、React、Vue、Nuxt、TypeScript、前端 AI 应用、API 契约、测试质量、安全权限、可观测性、部署回滚、ThreadPoolExecutor |
| 电脑工具 | 开发工具与 CLI、文档与知识工具 | 个人电脑和开发环境中的工具如何安装、配置、自动化、验证和长期维护 | uv、Maven、OpenCLI、Playwright CLI、Obsidian、Mermaid、Quarkdown、Doc2Wiki |
| 机器学习 | 特征工程、传统模型、模型评估、可解释性、MLOps、推荐与预测 | 如何用数据训练模型、解释模型、上线模型 | XGBoost、LightGBM、sklearn、MLflow、特征平台、模型评估 |
| 个人成长 | 职业发展、学习方法、认知方法、心理建设、效率工具、管理沟通 | 职业、心理、学习和个人效率类文章如何沉淀 | 职业规划、管理经验、读书方法、心理文章、效率工具 |

## 归类优先级

归类不是按关键词匹配，也不是按工具名固定归类。优先级如下：

| 优先级 | 判断项 | 要问的问题 |
|---|---|---|
| 1 | 技术本体 | 这个技术自己主要解决什么问题 |
| 2 | 文章主问题 | 这篇文章重点讨论什么问题 |
| 3 | 使用场景 | 技术被用在什么场景里，这个场景是否改变了文章主问题 |
| 4 | 关键词 | 标题和正文出现的词只能辅助判断 |

## 低置信与待人工判断

不要为了让目录看起来完整而硬塞文章。遇到以下情况，先进入“待人工判断”，不直接写正式核心知识点：

| 情况 | 例子 | 处理 |
|---|---|---|
| 标题和正文主问题不一致 | 标题讲 AI，正文实际讲数据平台或工程工具 | 先标低置信，人工确认主问题 |
| 正文关键词抢分类 | Paimon 文章正文出现 Doris、MySQL、HBase | 分类以文章主问题为准，正文只辅助 |
| 产品、工具、方法混杂 | SeekDB、ima、OpenClaw、Warp、Plandex | 拆成产品体验、技术机制、工作流方法再归类 |
| 工程基础过宽 | OkHttp、uv、Maven、ThreadPoolExecutor、structlog | 先分清主问题：系统机制进“工程与架构”，工具安装/命令/自动化进“电脑工具” |
| 统计/分析方法跨域 | 贝叶斯估计、置信区间、双重差分、合成控制 | 如果服务指标/实验/经营决策，归“数据分析与 BI”；如果服务建模训练，归“机器学习” |
| 知识工具跨域 | Mermaid、Quarkdown、Doc2Wiki、代码 Wiki、Obsidian | 如果核心是本机文档/知识工具使用，归“电脑工具 / 文档与知识工具”；如果核心是 Agent 自动维护知识或 RAG 架构，归“Agent 与 AI 工程 / RAG 与知识库” |

待人工判断不是失败，而是防止误分类进入长期知识库。

## 全量路由分类准则

批量处理时采用两阶段判断：

```text
第一阶段：标题 + 文件路径 + 明确技术名 -> 初步主问题
第二阶段：正文片段 -> 只辅助判断图片、投入建议、冲突点和置信度
```

原因：10% 灰测发现，如果把正文全部用于分类，相邻技术会抢走主类目，导致文章被误路由。

全量路由允许低置信，但正式沉淀不允许低置信。低置信文章必须先补充：

- 文章主问题。
- 技术本体。
- 不应归入的相邻类目。
- 需要用户或后续文章验证的点。

典型例子：`Airflow`。

`Airflow` 的技术本体是工作流调度、DAG 编排、任务依赖管理，不是机器学习。

| 文章主问题 | 应归类 | 判断 |
|---|---|---|
| DAG、Scheduler、Executor、Worker、补数、任务依赖 | 数据工程与数仓 / 调度编排 | 主问题是调度 |
| 调度 Hive、Spark、Flink、SQL、ETL/ELT | 数据工程与数仓 / 调度编排 | 主问题是数据任务编排 |
| 编排训练、特征生成、模型上线，并与 MLflow 等结合 | 机器学习 / MLOps | 主问题是机器学习流水线，Airflow 是编排工具 |
| Airflow 源码、插件、分布式执行架构 | 工程与架构 或 数据工程与数仓 / 调度编排 | 看文章重点是通用系统架构还是数据调度 |

## 数据工程与数仓 vs OLAP 与数据库

这两个目录容易混，边界按“主职责”判断。

| 判断项 | 数据工程与数仓 | OLAP 与数据库 |
|---|---|---|
| 关注点 | 数据进入、加工、分层、调度、质量、治理 | 查询性能、存储结构、索引、优化器、并发、更新 |
| 典型问题 | 表怎么分层？任务怎么跑？数据怎么同步？血缘怎么追？质量怎么控？ | 查询为什么慢？索引怎么设计？Join 怎么优化？并发怎么扛？ |
| 产出对象 | 数据表、任务、模型、血缘、质量规则、同步链路 | 查询服务、执行计划、索引、物化视图、存储格式 |
| 使用者 | 数据开发、数仓工程师、数据治理、平台工程 | 数据库工程师、OLAP 平台、查询服务、分析应用 |
| 时间特征 | 常见是批处理、准实时、流批一体的生产链路 | 常见是交互式查询、即席分析、服务化查询 |

一句话：

- `Hive` 默认放在“数据工程与数仓”，因为它通常承担离线数仓表、元数据、批 SQL、分区和建模落地。
- `Doris/StarRocks/ClickHouse` 默认放在“OLAP 与数据库”，因为它们通常承担查询加速、交互分析、服务化和高并发。
- 如果一篇文章讲 `Flink CDC -> Doris`，主角是同步链路就放“数据工程与数仓”；主角是 Doris 查询模型、索引、Compaction 就放“OLAP 与数据库”。

## Agent 与 AI 工程 vs LLM 与大模型

边界不看是否出现 `Claude`、`GPT`、`DeepSeek` 这些名字，而看文章讨论的是模型本身，还是模型工程化应用。

| 判断项 | Agent 与 AI 工程 | LLM 与大模型 |
|---|---|---|
| 关注点 | 工具调用、工作流、项目规则、上下文管理、评估、权限、安全、人机协作 | 模型能力、推理、上下文窗口、提示词、训练/微调、模型服务、评测 |
| Claude 示例 | Claude Code、Claude Desktop、MCP、SDK/API 应用、上下文压缩、AI 编程流程 | Claude 模型能力、版本对比、推理效果、长上下文、提示词优化 |
| 产出对象 | 可运行的应用、智能体流程、工具链、工程规范、评估体系 | 模型选型结论、提示词策略、推理部署方案、模型评测 |
| 批判点 | 是否可控、可评估、可回滚、权限边界是否清楚 | 模型效果是否被夸大、评测是否可靠、成本延迟是否可接受 |

## 归类示例：Hive

Hive 是一个示例，用来说明跨边界技术如何归类，不代表所有技术都要单独写固定规则。

Hive 多数情况下归入：

```text
knowledge/03_数据工程与数仓/0302_离线数仓/Hive/
```

纵向定位：

```text
数据平台
  -> 数据工程与数仓
    -> 离线数仓
      -> Hive
        -> 元数据：Hive Metastore
        -> 查询入口：HiveServer2 / JDBC / Kyuubi
        -> 存储承载：HDFS / ORC / Parquet / 分区
        -> 计算执行：MapReduce / Tez / Spark
        -> 建模落地：ODS / DWD / DWS / ADS / 拉链表 / 快照事实表
        -> 性能治理：参数、Join、倾斜、小文件、排序、分区裁剪
```

横向对标：

| Hive 维度 | 对标技术 | 关键差异 |
|---|---|---|
| 离线 SQL 数仓 | Spark SQL、Trino、Presto | Hive 更偏传统离线数仓和元数据生态，Spark SQL 更偏计算引擎，Trino/Presto 更偏交互式联邦查询 |
| 元数据服务 | Hive Metastore、Glue Catalog、Iceberg REST Catalog | Hive Metastore 是大数据生态事实标准，现代湖格式逐渐强化独立 Catalog 能力 |
| 表格式/湖仓 | Iceberg、Hudi、Paimon、Delta Lake | Hive 表偏目录/分区管理，湖格式增强事务、增量、快照、演进和流批一体 |
| 查询加速 | Doris、StarRocks、ClickHouse | Hive 不适合高并发低延迟查询，OLAP 引擎更适合报表和即席查询 |
| 数据同步 | SeaTunnel、DataX、Sqoop、Flink CDC | Hive 常是目标数仓或中间层，不是同步引擎本身 |

## 目录组织规则

知识内容按“一级类目 -> 二级类目 -> 具体技术 -> AGENTS.md”组织。

| 层级 | 作用 | 示例 |
|---|---|---|
| 一级类目 | 大问题域 | `03_数据工程与数仓` |
| 二级类目 | 具体问题域，统一使用 4 位前缀：前两位继承一级目录编号，后两位是二级编号 | `0302_离线数仓`、`0306_调度编排`、`0308_元数据血缘与治理` |
| 具体技术 | 一个可持续积累的技术对象 | `Hive`、`Kyuubi`、`Spark` |
| `AGENTS.md` | 技术纵向知识地图 | 定位、架构、模块、上下游、对标、官网/GitHub |
| `核心知识点/` | 有价值文章提取出的核心内容 | `Hive Metastore的实现和优化.md`、`Hive性能调优.md` |

`AGENTS.md` 不是文章摘抄，而是该技术的长期入口。它应该回答：

- 这个技术是什么，解决什么问题。
- 它在全局架构中处在哪一层。
- 它由哪些核心模块组成。
- 上游、下游、依赖和产出是什么。
- 应该与哪些技术横向对标。
- 官方文档、GitHub、关键架构图在哪里。
- 本地已沉淀了哪些核心知识点。

`核心知识点/` 用来存文章提炼结果。每个文件应快速回答：

- 文章主问题是什么。
- 可复用机制是什么。
- 对技术理解新增了什么。
- 适用边界和风险是什么。
- 可追查锚点是什么。
