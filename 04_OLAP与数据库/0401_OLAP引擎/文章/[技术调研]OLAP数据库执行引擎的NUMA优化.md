### 一．基于火山模型的执行引擎

火山模型(Volcano Model)是数据库查询优化和执行中最为常用的执行模型。其优点是（1）模块化设计：文件系统和查询处理功能被明确分开，使得每个模块可以独立开发和优化。（2）：扩展性和灵活性：由于查询处理模块与文件系统层分离，系统能够支持不同的数据模型，并且可以灵活扩展新的查询处理算法和操作符。（3）并行性支持：如交换算子，它可以嵌入到查询树的不同位置，提供并行查询处理的能力，这使得 Volcano 能够在多核或分布式环境中高效运行。

 火山模型执行器由查询处理层和文件系统层组成，每一种操作抽象为一个 Operator，整个 SQL 查询被构建成一个 Operator 树。查询执行时，树自顶向下调用 next() 接口，数据则自底向上被拉取处理，因此这种处理方式也被称为拉取执行模型（Pull Based）。

    

图2执行了关系代数：SELECT name, age FROM employees WHERE age > 30; 执行器在查询时会将查询看作一个实例/算子树，算子之间通过Exchange算子连接，从而实现数据流的交换。每个算子实现 next() 方法。当对算子的 next() 方法进行调用时，该算子将调用其孩子算子的 next() 方法来获取输入的数据，然后对数据进行逻辑加工并输出。而因为算子的 next() 方法是同步方法，在没有数据产生时， next() 方法将会持续阻塞。这时候需要循环调用根节点算子的 next() 方法，直到全部数据处理完毕，即可得到整个 Instance 的计算结果。

数据传输与同步：图5中的查询计划包括多个操作符，其中包括 scan（文件扫描）和 join（联接）操作。exchange 算子插入在这些操作符之间，管理数据的并行流动，还确保数据在传递过程中保持同步。它通过共享内存（例如端口）来实现进程间的数据交换。创建进程：exchange 算子的 open 过程创建新的子进程，并通过进程间的共享内存来传递数据。子进程负责处理父进程分配给它的数据分区。数据流管理：在 exchange 算子的控制下，数据被封装成数据包，并在父子进程之间传递。这些数据包可以包含多个记录，并在目标进程中解包进行进一步处理。    

火山模型有两种并行模式：（1）垂直并行是指将查询的不同阶段（如扫描、过滤、联接等）分配到不同的进程或线程中并行执行。在图6中，查询的多个算子通过 exchange 算子并行处理。exchange 算子不直接处理数据，它只负责 数据的传递与同步，确保数据可以在多个进程之间流动。在这种并行模式下，多个阶段的处理可以在同一时刻进行，而不需要等待前一个阶段完成。例如，扫描操作和联接操作可以并行执行，数据在两个阶段之间通过 exchange 算子交换。但是，尽管查询计划中包含了多个 exchange 算子，联接操作符（Join Operators） 可以选择 在同一个进程内执行，而不是将每个操作符都分配到不同的进程中执行。（2）水平并行将数据集分成多个分区，然后在多个进程或处理单元上并行执行这些分区的操作。exchange 算子 在这个过程中负责将数据分区并分发到不同的处理单元。在图7中，join 操作符 通过三个进程执行，而 file scan 操作符 通常由一个或两个进程执行，每个进程扫描一个文件的不同分区。exchange 算子负责管理这些分区数据的传递。

火山模型问题：主要是对数据局部性感知较差，操作符调用next方法获取数据没有做负载均衡等优化。

### 二．Morsel-Driven NUMA感知的执行引擎

《Morsel-Driven Parallelism: A NUMA-Aware Query Evaluation Framework for the Many-Core Age》发表于SIGMOD2014，提出一种基于火山模型的多核优化执行引擎。    

随着计算机架构的发展，并行查询必须为现代硬件做出优化，以实现良好的加速。该论文指出数个需要被优化的关键问题：

(i)为了利用多核，所有查询工作必须均匀分布在数百个线程中，以实现良好的加速

(ii)由于现代物理核心乱序执行，即使有准确的数据统计，也很难均匀地分配工作。

第一点很好理解，即执行引擎在处理SQL查询时，多个并行的物理核心工作负载最好是均衡的，否则某个任务负载较重的物理核心将成为处理瓶颈。

第二点，由于现代处理器大都使用乱序执行来提高核心并行处理能力，即允许处理器在执行程序指令时，不按照程序的原始顺序。所以，现有的“plan-driven“执行器在乱序执行时即便准确进行数据统计，但也要不停的进行上下文切换，来准确的将任务均匀划分给空闲的核心。

总而言之，现有的plan-driven执行器会遇到负载均衡和上下文切换无法扩展。另外一个挑战是NUMA，这是由于内存控制器分散在现代多核处理器不同的插槽上。本文提出了一种新的细粒度的并行执行框架“Morsel-driven“，其主要竞争者是”plan-driven“的提出与20世纪90年代的Volcano模型，这种模型在查询编译时静态确定应该运行多少个线程，每个线程都实例化一个查询运算符计划，并将它们与交换运算符连接起来。

Morsel-driven的核心思想是一种调度程序：Dispatcher，该调度器序运行固定数量的、与机器相关的线程，这样即使有新查询到达，也不会出现资源超额订阅，并且这些线程固定在核心上，这样就不会因为操作系统将线程移动到其他核心而发生意外的 NUMA 本地性损失。任务分配是在运行时完成的，因此具有完全的弹性。这允许实现完美的负载平衡，即使面对中间结果的大小分布不确定，以及现代 CPU 内核的性能难以预测，即使它们获得的工作量相同，性能也会发生变化。它具有弹性，因为它可以处理在运行时发生变化的工作负载（通过减少或增加正在执行的查询的并行性）。    

第二章主要是根据Figure 1给的三表连接例子来解释morsel-driven的并行执行流程。执行的关系代数如下：

扫描、过滤和构建基本关系 T 的哈希表 HT(T)，2. 扫描、过滤和构建参数 S 的哈希表 HT(S)，3. 扫描、过滤 R 并探测 S 的哈希表 HT(S) 并探测 T 的哈希表 HT(T) 并存储结果元组。

表R的pipeline只能在哈希表HT和HS建立后再进行。因此，文中提到了一个组件叫做，QEPobject，负责观察数据依赖，并将目前可执行的 pipeline 转移到 dispatcher。同时，QEPobject还负责给每个 pipeline 分配临时的存储空间，用于存储 pipeline 的执行结果。为了NUMA本地写，每条pipeline的存储空间都在自己运行的核上。在整条 pipeline 执行完毕后，该临时存储空间会在逻辑上被重新划分为相等大小的块，从而避免不同 pipeline 之间的数据倾斜。QEPobject 被实现为一个被动状态机。只有当一个 pipeline job 完全执行完毕后（找不到下一个Morsel），这段代码才会被 dispatcher 调用，来检查是否有依赖该 pipeline job 输出的任务。    

哈希连接主要包括两个阶段，第一个阶段负责从关系 T 中读数据，过滤后物化中间结果到临时的存储区域，第二个阶段主要读取过滤后的数据并建立哈希表。图中红、绿、蓝三条细线分别代表了三个线程，每个线程每次从表T中读取一块（morsel）数据。由于表T也是按 NUMA 结构存储的，因此每个线程只会读取自己\*CPU\*所在的 socket 的数据，并且临时结果的存储区域也都是基于 socket 分开的。表T的所有数据都被扫描，过滤之后，才会对第一阶段的结果进行扫描，并存储指针到哈希表中，建立好的哈希表中会包含指向临时存储区域的指针用来给probe阶段做数据访问使用。

probe-phase比较简单，过程如图4所示。与图3一样，dispatcher根据 NUMA 分区给对应的线程分配任务，对数据过滤之后再进行探测，注意这里没有再将数据过滤和 probe 操作分成两个阶段。pipeline 的执行结果仍然存在 NUMA local 的存储区域内，等待下一步处理。

morsel-driven 并行执行多条 pipeline，这点是和传统的火山模型相同的。但是不同之处在于，morsel-driven的并行策略下，pipeline之间不是独立的，也就是说，pipeline之间会共享数据结构，以及执行必要的同步。同时，执行线程数也是完全灵活的，不仅不同的 pipeline 之间可能不同，相同的 pipeline 内部也可能不同。    

图5中描述了 dispatcher 的架构。其维护了一个 list，其中存储了目前可执行的所有 任务。每个核心都有一个 morsel 的链表。实际上，我们会给每个核/socket维护存储区域的边界，并且把这些大的存储区域按需划分为 morsel 大小的小块，也就是说，当一个核心从 dispatcher 请求任务时，其所在 socket 存储区域的下一个 morsel 块会被“切下来”。dispatcher 被实现为一个 lock-free 的数据结构。当 worker 线程需要获取任务时，会主动调用 Dispatcher 的代码，从 pipeline job queue 中拿一个任务执行。

Morsel 用于将一个大的 task 切分为小的，固定大小的工作单元来促进 work-stealing 和抢占。因此，morsel 大小并不是性能的决定性因素，只需要在提供良好的响应时间的同时，能够均摊掉调度造成的开销。为了显示 Morsel 大小对查询性能的影响，这里做了一个查询实验select min(a) from R。图中显示当 morsel 大小到达 1万以上时，其性能损耗就可以忽略不计了。最优的设置依赖于硬件。    

此外，Morsel还介绍了一些NUMA优化的并行查询算法，例如图8并行聚合算法。图9并行合并排序算法。

图8并行聚合算法详细描述了一种分两阶段执行的并行聚合流程。在第一阶段，每个线程对数据进行局部预聚合，将结果存储到线程本地的固定大小哈希表中。当哈希表填满时，会将数据溢出到对应的分区中（图片中猜测以键值mod4的值为分区依据，当ht被填满时，后续数据中mod4=0的被分到partition0，以此类推）。在第二阶段，各线程按分区对数据进行逐一扫描并进一步聚合。此方法有效减少了多线程环境下的冲突，并提升了对大量分组或高度聚集的场景的处理性能。

图9展示了一个用于全局排序的并行合并排序算法。该算法首先让每个线程对其局部数据进行排序，然后通过计算局部分隔点（例如：线程1：选取 200 和 800 作为分隔点。线程2：选取 100 和 400 作为分隔点。线程3：选取 600 和 900 作为分隔点。），确定全局分隔点（例如：全局分隔点为 [100, 400, 600, 900]）。全局分隔点用于将排序结果划分为多个区块（例如：区块1（<400；区块2（400~600；区块3（600~900）；区块4（>900），使每个线程能够独立处理分配到的区块。最后，各线程对这些区块进行合并，生成全局排序的输出。这种方法有效利用了多线程并行处理能力，同时避免了线程间的同步需求，进一步提高了排序的效率和内存局部性。    

### 三．用于主存列存储的自适应NUMA感知技术

本文提出了用于主存列存储的自适应numa感知技术。在运行时调整数据放置和socket间任务窃取以适应由处理单个表或表组（共分区表）的算子主导的工作负载。我们的实施和实验基于商业列存储（SAP HANA）。贡献是：

自适应数据放置策略，与总是跨所有socket分区数据的静态策略相比，可以将吞吐量提高2倍。我们提出了一种自适应启发式算法，该算法在运行时移动和重新分区表，以平衡socket之间的利用率。根据任务的内存强度调整socket间任务窃取，与始终允许socket间窃取的静态策略相比，将吞吐量提高1.1 - 4倍。

为了适应数据放置和任务窃取，需要完全了解系统的利用率。我们提出了一种设计，可以在(a)任务、(b)表和表组分区以及(c)socket等级别跟踪使用历史。

1.自适应技术最重要方面的概念示例

 

图1显示了自适应技术最重要方面的概念示例。服务器有四个完全互连的socket。工作负载由三个表上的大量并发内存密集型扫描组成，这些表最初位于服务器的两个socket上。由于扫描的内存强度，任务窃取是不允许的。其中两个socket已被充分利用，其内存带宽已饱和，而其余两个socket处于空闲状态。我们的自适应数据放置检测到利用率不平衡，并采取措施修复它。它将表TBL2移动到socket3，跨socket2和4对TBL3进行分区，最后合并TBL4中未使用的部分。socket利用率变得平衡。总内存吞吐量比最初提高了2倍，将工作负载的吞吐量提高了2倍。    

为了验证论文解决的问题，作为整个论文强有力的制程，作者做实验展示了不必要的数据分区，和任务窃取在聚合和连接操作时的开销，指出了自适应的必要性。

图3展示了文章作者通过实验验证 分区（partitioning） 和 任务窃取（task stealing） 的效果，重点分析这两种策略在不同查询类型和工作负载场景下对系统性能的影响。实验数据来自 聚合查询 和 连接查询 的吞吐量（TP）、内存带宽使用（Mem.TP）、以及缓存行为（本地和远程的LLC Misses）。以下是详细解释：

聚合查询（Aggregation Queries）：低选择率（0.01%：Selectivity=查询返回的记录数/表中的总记录数）的工作负载特征：大部分任务在扫描阶段（IV Scan）耗费内存带宽。内存密集型任务占主导，CPU利用率较低。

观察结果：任务窃取的影响：禁用任务窃取：本地任务处理使得总内存带宽接近插槽极限。启用任务窃取：任务分布在远程插槽，增加了远程内存访问（Remote LLC Misses），导致整体性能下降。    

分区的影响：无分区（1TBP/表）：单插槽可能过载，导致性能受限。启用分区（4TBP/表）：分区提高了插槽间的并行性，使得总内存带宽使用增加，吞吐量提升了 3倍。

           

聚合查询（Aggregation Queries）：高选择率（10%）工作负载特征：随着选择率增加，任务的内存密集程度降低，CPU密集型任务占主导。CPU资源成为瓶颈，内存带宽使用率较低。

观察结果：任务窃取的影响：启用任务窃取：对CPU密集型任务的负载平衡效果显著，使吞吐量提升了 2.5倍。禁用任务窃取：部分插槽CPU空闲，限制了吞吐量。

分区的影响：无分区（1TBP/表）：不利于平衡插槽之间的CPU负载。启用分区（4TBP/表）：带来一定开销，但当负载分布均匀时性能略优。

任务窃取和分区策略对性能影响显著，需根据具体工作负载特征动态调整。在 内存密集型 和 CPU密集型 场景下，采用统一静态策略（如始终允许任务窃取或始终分区）可能导致性能下降。本文提出的自适应策略（动态调整任务窃取和数据分布）有望在不同工作负载下实现更优的性能表现。

本文给出了跟踪具体工作负载，来动态调整数据的放置，和任务窃取：跨三个级别跟踪资源利用率（CPU负载和内存吞吐量）：(a)任务，(b)表和表组分区，以及(c)socket。图4用一个并发低选择性扫描的示例描述了基础结构（主要是扫描IV的内存密集型第一阶段）。    

图4分为三个主要部分：1.任务类别（Task classes）。Scan IV：表示扫描表的索引向量（Index Vector，IV）的任务（内存密集型）。Scan Mat：表示扫描结果物化的任务（CPU密集型）。Aggreg：聚合任务，涉及CPU和内存的混合负载。

2.资源利用率历史（RUH）的层次化结构：跟踪单个任务、表分区/表组分区（TBP/TGP）和插槽的资源利用情况。任务级别：每个任务的内存带宽和CPU使用会更新到其所属分区（TBP或TGP）的RUH中。表分区和表组分区（TBP/TGP）级别：TBP（Table Part）：单张表的分区。TGP（Table Group Part）：多个相关表的分区组合。每个TBP/TGP与一个插槽关联，记录其总资源使用。3.插槽级别：每个插槽（Socket）会聚合所有分区（TBP/TGP）的利用率，形成插槽的整体RUH。

利用率记录机制：利用硬件计数器采集CPU和内存带宽使用，并以滑动窗口的方式维护历史记录。资源使用采集方式： 图4右侧展示了资源利用的采集机制：CPU负载（CPU load）：跟踪当前运行的任务线程数。内存带宽（Mem.TP）：通过硬件计数器（如Intel PCM）测量本地LLC未命中和内存访问速率。    

   图5由两部分组成,左侧为RUH的定义与主要字段。右侧为历史数据存储与更新机制。

RUH是整个自适应NUMA感知技术的核心，主要作用包括：1.实时监控：动态采集任务、表分区和插槽的资源使用情况，提供准确的利用率信息。2.调整依据：DP算法利用RUH数据判断是否需要迁移或分区，并选择合适的调整策略。通过历史平均值与当前使用率的对比，判断资源使用是否稳定。3.性能优化：通过RUH信息，DP能够检测插槽之间的不平衡（如CPU负载过载或内存带宽瓶颈），并采取优化措施。

左侧的字段解释如下：1.pages：分区或插槽所占用的内存页数。用于估算分区迁移或重新分区的时间开销（如图6和算法1所示）。对于表组分区（TGP），pages 记录的是所有相关表分区中最大的页数，而非总和，因为迁移和分区可以并行处理。2.socket：表示当前分区或插槽的所在插槽编号（NUMA节点）。3.owner：指向分区的拥有者，可能是：TablePart（表分区）、TableGroupPart（表组分区）、Socket（插槽本身）。RUH通过owner字段将分区与其资源使用记录关联起来。4.memh（Memory History）：用于跟踪内存带宽的利用率历史（单位：MB/s）。存储当前和历史的内存带宽使用信息。5.cpuh（CPU History）：用于跟踪CPU线程使用的历史数据。存储当前和历史的CPU负载（线程数）。    

右侧的History结构的字段为：1.value：当前的资源利用率值（如当前的内存带宽或线程数）。由任务在运行时动态更新。2.ravg（Recent Average）：最近一个周期内的平均值，通常为 DP运行周期（1秒）。用于优化性能，以避免在频繁读取RUH时反复计算平均值。3.samples（历史样本列表）：一个时间戳与值的列表，用于记录过去一段时间内的资源使用情况。列表大小有限（最多3000条），表示大约5分钟的历史数据。4.entries：记录当前历史样本列表的条目数量，用于管理samples的大小。5.lock：读写锁，用于多线程环境下保护samples的更新和读取操作。History的关键操作包括1.sample()：每100毫秒由后台刷新线程调用一次，将当前值记录到samples列表中。如果列表已满，移除最旧的条目，保证存储容量固定。2.avg(uint64\_t us)：计算指定时间段内的资源利用率平均值。遍历历史样本列表，提取从给定时间点开始的所有样本，并根据时间权重计算平均值。如果时间段超过samples覆盖范围，则假定最旧样本值保持不变。3.reset()：清空历史记录，用于在分区迁移或重新分区完成后重置资源利用率历史。

    

图6展示了自适应数据放置data Placer（DP）线程的抽象工作流。在每个周期，DP都获得活动RUH （tbp和TGP）及其最近使用情况的快照。然后，根据它们过去的利用率是否稳定，计算它们是否适合移动和分区。接着，DP根据每个套接字最近的利用率对它们中的RUH进行排序，并将它们聚合以计算套接字最近的利用率。然后，DP计算所有套接字对之间的CPU利用率不平衡，并对这些套接字对进行排序。对于每一对套接字，DP会调查一个新的位置是否可以减少不平衡。只有当不平衡超过一个阈值，并且热socket饱和时，DP才会进行。如果热套接字未饱和，则TBP和TGP无法通过利用冷套接字上的空闲线程来提高其利用率。DP迭代热套接字的RUH，并检查移动或分区符合条件的RUH的所有者（相应的TBP或TGP）是否减少了不平衡。另外，如果它不会造成内存带宽瓶颈，DP将继续移动或分区RUH的所有者。

步骤可以拆分为：1.等待周期性触发：DP线程以固定周期运行（如1秒），间隔时间由参数 cp 控制。周期性检查插槽、表和分区的资源利用率。2.计算每个插槽的活跃资源使用历史（RUH）：DP获取所有活跃的资源利用历史（Resource Utilization History，RUH），包括表的分区（TBP）和表组的分区（TGP）。RUH指标包括：每个分区的CPU负载（线程数量）、内存带宽使用、以及占用的内存页数量。3.计算插槽利用率：DP聚合每个插槽的RUH数据，计算每个插槽的CPU和内存带宽使用。结果用于后续的插槽之间的负载平衡决策。4.计算插槽间不平衡：DP计算任意两个插槽之间的CPU利用率差异（称为“负载不平衡”）。

如果发现利用率差异超过阈值（由参数 ci 控制），且较热的插槽（hot socket）已经饱和（CPU使用率超过 cs 阈值），则尝试进行数据调整。5.尝试通过数据迁移减少不平衡：对每对插槽（热插槽和冷插槽），DP尝试选择一个可移动的表分区或表组分区。可移动的条件：RUH最近的资源使用模式稳定（判断依据见算法1）。如果迁移可显著减少不平衡且不会在目标插槽引入新的内存带宽瓶颈，则执行迁移。6.如果迁移无效，则尝试重新分区：

如果迁移无法减少不平衡，DP尝试重新分区（partition）表。分区方法：将目标表分区数加倍（从当前分区数增加到2倍）。将新生成的分区重新分配到插槽，其中一个分区移动到冷插槽，其余分区留在原插槽。重新分区是更昂贵的操作，因此优先考虑迁移。7.检查并合并冷数据：如果当前插槽间负载平衡已经达到阈值（负载不平衡在合理范围内），DP进一步检查冷表或冷分区。冷表的定义：过去一段时间（由分区和合并耗时决定）内的资源使用接近0。对冷表或分区执行合并操作，并将其移动到利用率最低的插槽。    

图7展示了两种分区（RUH）的利用率模式，分别是不稳定的RUH和稳定的 RUH，用于说明如何判断一个分区是否满足迁移或分区的条件。

其中图7(a)为不稳定的 RUH1，观察到最近利用率和过去的历史利用率差异显著。利用率的波动幅度超过了阈值 ce。可能因为此分区受到工作负载的短期变化影响，资源利用模式尚未稳定。如果立即迁移或分区，会影响系统性能，并可能导致不必要的操作开销。DP 会认为该 RUH 不满足迁移或分区条件。系统会继续观察此分区的资源利用模式，直到其稳定。

### 四．结论

三篇论文分别针对不同的数据库查询优化需求展开，Adaptive NUMA 和 Morsel-Driven 针对NUMA架构进行了更深入的优化，而 Volcano 则奠定了并行查询框架的基础。Adaptive NUMA的关键技术为自适应数据放置（迁移和分区）和自适应任务窃取。Morsel-Driven的关键技术为Morsel驱动任务分配、NUMA感知调度和动态负载均衡。 Volcano的关键技术为迭代器模型支持操作符解耦和交换器支持并行化任务分发和动态负载均衡。同时Adaptive NUMA 通过历史数据和实时监控自适应调整数据布局和调度策略，结合分析型数据库系统，适用于多种查询操作和数据分布场景。Morsel-Driven动态分配morsels，支持任务窃取实现负载均衡，适用于多核环境，morsel粒度灵活，可适配不同的硬件和工作负载。Volcano静态分配任务，任务调度较为固定，对现代多核系统的适应性不如前两者。
