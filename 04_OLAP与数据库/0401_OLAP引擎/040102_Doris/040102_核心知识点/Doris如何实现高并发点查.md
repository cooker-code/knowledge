## 一句话结论

这篇文章值得精读，核心价值是校准 Doris 的边界：它可以通过点查专门优化进入部分 Data Serving 场景，但不能因此把 Doris 等同于通用 KV 或在线交易数据库。

## 纵向理解

| 维度 | 判断 |
|---|---|
| 全局架构 | 客户端 -> FE SQL 接入 -> 轻量计划或通用计划 -> BE 存储与执行 -> 行存/缓存/索引 |
| 本文位置 | 只覆盖点查优化，不代表 Doris 完整架构 |
| 核心机制 | 行式存储、Row Cache、Short-Circuit、Prepared Statement |
| 使用链路 | 建表启用相关能力 -> 主键条件查询 -> 命中轻量路径和缓存 -> 返回整行 |
| 前置条件 | 查询模式简单、主键或高选择性条件明确、表设计和缓存配置匹配 |
| 边界 | 复杂 Join、聚合分析、冷热不稳定、尾延迟要求极低时不能套用点查结论 |

## 横向对标

| 对标技术 | 实现方式 | 优势 | 劣势 | 适合场景 |
|---|---|---|---|---|
| Redis / KV | 内存或 KV 存储按 key 访问 | 延迟低、在线服务成熟 | SQL 分析和宽表聚合弱 | 在线核心链路、极低延迟 |
| MySQL / PostgreSQL | BTree 索引点查 | 事务和在线查询成熟 | 大规模分析和列式聚合弱 | OLTP、强一致业务查询 |
| Elasticsearch | 倒排索引和文档检索 | 文本检索和过滤强 | SQL 分析和强一致更新复杂 | 搜索和检索服务 |
| Doris | OLAP 引擎加点查短路径 | 分析和 serving 可复用一套数据 | 对表设计、版本、缓存和查询形态敏感 | 分析型宽表点查和报表服务 |
