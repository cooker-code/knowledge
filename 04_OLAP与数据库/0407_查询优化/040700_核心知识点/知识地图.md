# 040700_查询优化知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 04_OLAP与数据库 / 07_查询优化 |
| 解决的问题 | 查询优化器如何把 SQL 转成可执行计划，并通过统计信息、代价模型、计划搜索和运行反馈降低执行成本 |
| 典型输入 | SQL、逻辑计划、物理算子、统计信息、基数估算、运行反馈 |
| 典型产出 | 执行计划、Join 顺序、索引选择、代价估算、优化建议 |
| 用户阅读目标 | 区分通用查询优化器机制与某个数据库的具体 SQL 技巧 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| Q00 问题建模 | 慢查询来自扫描、Join、排序、聚合、分页、统计信息还是优化器误判 | 先识别慢在哪里，再决定调 SQL、调索引还是研究优化器 | 通用慢查询诊断框架文章进入 |
| Q01 统计与估算 | 优化器如何估算基数、选择率和中间结果规模 | 基数估算错误会放大到 Join 顺序和算子选择 | Cardinality Estimation、统计信息文章进入 |
| Q02 计划搜索 | 优化器如何枚举 Join 顺序、访问路径和物理算子 | 搜索空间和剪枝策略决定优化时间与计划质量 | CBO、Join Reorder、动态规划、启发式规则文章进入 |
| Q03 代价模型 | 成本模型如何估算 CPU、IO、网络、内存和复杂谓词成本 | 公式假设在真实复杂查询中可能失效 | Cost Model、JOB-Complex、真实世界基准文章进入 |
| Q04 学习型优化 | AI/ML 如何表示计划、学习成本或提供优化建议 | 学习型方法的关键是计划表示、训练数据和鲁棒性 | QueryFormer、Learned Cost Model、Plan Vectorization 文章进入 |

## 使用方式

- 新文章先问：这是通用优化器机制，还是某个数据库的调优技巧。
- 具体数据库技巧回到技术目录；通用机制才留在本目录。
- 没有执行计划、统计信息、数据规模或基线的“优化效果”文章要降权。
