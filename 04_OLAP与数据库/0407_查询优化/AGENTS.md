# 查询优化

## 知识点入口

- 本模块先看宏观流程，再看文章：[流程化知识点总览](knowledge/04_OLAP与数据库/0407_查询优化/核心知识点/流程化知识点总览.md)。
- 本目录只承接跨数据库的查询优化机制、查询优化器、代价模型、执行计划表示和学习型优化。
- 明确属于某个数据库的调优文章，应放回对应技术目录，如 MySQL、PostgreSQL、Doris、ClickHouse、StarRocks。

## 类目定位

| 项 | 内容 |
|---|---|
| 一级类目 | OLAP 与数据库 |
| 二级类目 | 查询优化 |
| 核心问题 | 查询优化器如何基于统计信息、代价模型、执行计划搜索和运行反馈选择更优查询计划 |
| 不解决什么 | 单个数据库的索引使用技巧、分页 SQL 改写、业务慢 SQL 案例，除非文章抽象成跨数据库优化机制 |
| 用户当前认知假设 | 用户知道 SQL 可以调优，但需要区分“某数据库实践技巧”和“查询优化器机制” |

## 划分准则

| 保留条件 | 归入位置 | 示例判断 |
|---|---|---|
| 讲查询优化器、CBO、统计信息、基数估算、Join Reorder、代价模型 | `查询优化器/` | PostgreSQL/OLAP CBO、JOB-Complex、QueryFormer |
| 讲 AI/机器学习如何表示查询计划、学习成本模型或辅助优化器 | `查询优化器/` | Learned Cost Model、Plan Vectorization |
| 明确是 MySQL/InnoDB 的 SQL 改写或索引技巧 | `02_关系数据库/MySQL/` | 大 offset limit 延迟关联 |
| 明确是 Doris/ClickHouse/StarRocks 的执行计划、统计信息、Join 优化 | 对应 OLAP 引擎技术目录 | 不放通用查询优化目录 |

## 排重准则

问题指纹：

```text
查询优化主题 + 统计信息/基数估算/代价模型/计划搜索/计划表示模块 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都讲慢 SQL | 先判断是否是具体数据库技巧；不能抽象为优化器机制则移到具体技术目录 |
| 都讲 AI 优化 | 比较是计划表示、成本估算、Join Order、鲁棒性还是自动索引 |
| 只有性能数字 | 没有查询结构、基线、执行计划和数据规模时降权 |
| 和 OLAP 引擎混淆 | 如果文章主角是 Doris/ClickHouse/StarRocks 的实现，放对应引擎目录 |
