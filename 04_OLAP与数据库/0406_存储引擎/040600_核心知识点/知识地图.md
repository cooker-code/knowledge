# 040600_存储引擎知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 04_OLAP与数据库 / 06_存储引擎 |
| 解决的问题 | 底层存储引擎如何在写入吞吐、读取效率、空间占用、恢复能力之间做权衡 |
| 典型输入 | 写请求、读请求、范围扫描、删除、TTL、压缩、恢复 |
| 典型产出 | WAL、MemTable、SSTable、Manifest、索引、Bloom Filter、Compaction 产物 |
| 用户阅读目标 | 把 LSM/RocksDB/LevelDB/专用引擎的机制放回读写路径，而不是只记“写快读慢” |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| E00 负载建模 | 是点写、批写、点查、范围扫、时序写入还是删除/TTL | 存储引擎设计从负载约束开始 | 专用引擎、Redis/RocksDB 对比、时序存储文章进入 |
| E01 写入路径 | 写请求如何进入 WAL、MemTable、Immutable MemTable 和 SSTable | 写优化来自顺序写和内存缓冲，但会形成后续整理债务 | WAL、MemTable、Flush、写放大文章进入 |
| E02 读取路径 | 点查和范围扫描如何经过内存、索引、Bloom Filter、SSTable | 读性能取决于层数、过滤、缓存和文件组织 | LevelDB/RocksDB 读路径、Block Cache、Bloom Filter 文章进入 |
| E03 Compaction | 后台合并如何处理层级、重复 key、删除标记、空间回收 | Compaction 是 LSM 的价格标签，影响写放大、读放大和空间放大 | RocksDB Compaction、LSM 策略文章进入 |
| E04 恢复与治理 | 崩溃恢复、Manifest、版本管理、GC、参数治理如何保证可靠 | 存储引擎不是单个数据结构，而是一套恢复和治理机制 | Manifest、恢复、GC、参数调优文章进入 |

## 使用方式

- 新文章先问：它是在讲存储引擎机制，还是某个上层数据库的普通功能。
- 明确技术本体后再进入 LSM-Tree、RocksDB、LevelDB 或专用引擎目录。
- 只讲概念、不讲读写路径和代价的文章降权。
