#

> 如果把 LSM-Tree 想成一条写入流水线，那么最核心的两站通常就是 WAL 和 MemTable。前者负责“先把写入留痕”，后者负责“先把数据接住并维持有序”。一个保证崩溃后还能找回来，一个保证写入足够快、读时还能按 key 顺序组织。



## 一次写入，为什么不能只进内存

假设有一条最简单的写入：

```
put("user:42", "active");
```

如果系统只是把它直接塞进内存结构，比如哈希表或跳表，那么平时当然很快。但只要机器突然掉电、进程异常退出，问题就来了：

* 内存里的内容会丢
* 这次写入对外可能已经返回成功
* 但重启后却找不到这条数据了

这就是很多存储系统写路径的第一原则：

* 先留下可恢复的持久化痕迹
* 再把数据放进高性能内存结构

WAL 的存在，就是为了解决这个问题。

## WAL 到底在保证什么



WAL 是写前日志（Write-Ahead Log）。名字里最关键的是 `Write-Ahead`：

* 数据正式进入主要内存结构之前
* 先把这次写操作顺序追加到日志里

它真正保证的，不是“性能更高”，而是：

* 崩溃后能恢复
* 对一批写入提供原子边界
* 让“返回成功”这件事更有依据

### 最简单的直觉可以理解成“先记账，再入账本”

* WAL：像流水账，先顺序记下来
* MemTable：像当前营业中的有序索引，负责接住最新写入

只要 WAL 已经安全落盘，即使 MemTable 还没来得及刷到磁盘 SSTable，系统重启后仍然可以：

* 重新读 WAL
* 再把操作一条条回放回内存结构

### WAL 的原子性通常体现在“要么整批生效，要么整批不算”

例如一个写批次：

```
WriteBatch batch;batch.put("user:42", "active");batch.put("user:42:last_login", "2026-03-22");write(batch);
```

如果系统已经对外承诺这个批次写成功，那么恢复后就不能只看到其中一半。

这也是为什么很多实现会把一个批次写成：

* 同一条逻辑记录
* 或者同一个 sequence range
* 恢复时按批次边界回放

## WAL 记录里通常会放什么



WAL 不是一段模糊文本，它通常有明确的记录结构。

一个非常简化的想象版本可以写成这样：

```
struct WalRecord {    uint64_t sequence;    uint8_t  op_type;     // put / delete    uint32_t key_size;    uint32_t value_size;    bytes    key;    bytes    value;    uint32_t crc;};
```

真实实现会更复杂，但几个核心信息常常都绕不开：

* 这条记录属于哪个 sequence
* 这是 put 还是 delete
* key 和 value 的长度是多少
* 数据本体是什么
* 如何校验记录完整性

### 为什么 sequence 很关键

sequence 解决的不是“写到哪里”，而是：

* 这些写入的全局先后顺序是什么
* 恢复时该按什么顺序回放
* 同一个 key 的多个版本谁更新

在 LSM 系统里，sequence 常常是“逻辑时间”的一部分。

### 为什么要 CRC 或其他校验

因为崩溃时最怕一种情况：

* 一条 WAL 记录只写了一半
* 日志文件尾部出现不完整残片

如果没有校验，恢复时就很难判断：

* 这是一条合法记录
* 还是崩溃中途留下来的半截垃圾

## 崩溃恢复时，WAL 是怎么把数据找回来的



恢复时最核心的动作其实很朴素：

* 打开最近一段 WAL
* 按顺序读记录
* 把合法记录重新插回新的 MemTable

一个极简伪代码大概是：

```
MemTable mem;for (auto& record : read_wal()) {    if (!record.crc_ok()) {        break;  // 文件尾部可能是不完整记录    }    mem.apply(record);}
```

这个过程里，WAL 起到的作用非常直接：

* 它不是最终查询结构
* 它是恢复时的事实来源

所以很多系统里会出现这种分工：

* WAL 负责可恢复性
* MemTable 负责当前写入和最近读请求
* SSTable 负责更长期的持久化组织

## MemTable 到底是什么，为什么它常常选跳表



MemTable 可以理解成：

* 当前还驻留在内存中的有序写入集合

它承担的目标通常有三个：

* 新写入要快
* 按 key 查找要快
* 后续 flush 成 SSTable 时，最好能顺序遍历

如果只是单纯追求插入快，哈希表当然也很诱人。但很多 LSM 引擎仍然偏爱跳表，原因就在于：

* 跳表天然有序
* 插入、查询、范围扫描都比较平衡
* 实现复杂度比平衡树更可控
* 对并发读写也更容易做工程化优化

### 一个非常简化的节点样子可以想成这样

```
struct Node {    Slice key;    Slice value;    Node* next[level_count];};
```

关键点是：

* 每个节点有多层 forward 指针
* 上层更稀疏，下层更密集
* 查找时先走高层，再逐层下沉

## 跳表为什么能同时兼顾写入和有序遍历



跳表最值得喜欢的一点，是它在工程上很平衡。

### 查找时，先高后低

例如要找 `key=42`：

* 先从最高层开始往右跳
* 一旦发现再往右就超过目标，就往下一层掉
* 最后在最底层定位到目标或插入位置

这让它避免了单链表那种“每次都从头一格格走”的尴尬。

### 插入时，不需要整体搬移元素

和有些数组式有序结构不同，跳表插入更多是：

* 找到各层前驱节点
* 把新节点挂进去
* 更新若干层 forward 指针

这对写密集场景很友好，因为它不会动辄触发大块搬移。

## 并发 MemTable 的跳表设计，真正关键的点是什么



一说“并发跳表”，很多人第一反应是：

* 每一步都加锁？
* 还是全程 lock-free？

真实系统常常没有那么极端，更多是围绕这几个原则做设计。

### 节点一旦发布，内容尽量不再改

这是非常关键的一条。

* key 和 value 写进节点后不再修改
* 节点的可变部分尽量只剩 next 指针

这样做的好处是：

* 读线程看到的节点内容天然稳定
* 并发推理难度会下降很多

### 内存分配尽量走 arena

很多 MemTable 实现会配 arena allocator。

原因很现实：

* 节点很多而且生命周期接近一致
* 与其一个个 `new/delete`
* 不如大块申请、线性切分、统一回收

这样能减少：

* 分配器开销
* 小对象碎片
* 节点释放管理复杂度

### 并发写入时，更新的是“链接关系”而不是对象本体

一个简化插入逻辑可以想成：

```
Node* n = arena.allocate_node(key, value, height);find_predecessors(key, prev);for (int level = 0; level < height; ++level) {    n->next[level] = prev[level]->next[level];    prev[level]->next[level] = n;}
```

如果要进一步支持更强并发，核心变化通常在：

* `prev[level]->next[level] = n` 这一步如何安全发布
* 是否借助 CAS 做无锁链接
* 如何避免两个写线程同时插到同一位置时互相覆盖

### 读线程为什么常常可以更轻

因为一旦节点发布后内容不改，读线程很多时候只需要：

* 顺着已经发布的 next 指针往前走
* 不需要碰写线程的复杂状态机

这也是很多实现会优先做到：

* 读尽量轻
* 写再通过更谨慎的发布协议兜住正确性

## 一次写入，WAL 和 MemTable 是怎么接起来的



如果把前面的所有零件拼起来，一条 `put` 往往会经历这几个关键动作：

```
Status write(WriteBatch batch) {    wal.append(batch);      // 先记 WAL    wal.sync_if_needed();   // 满足持久化策略    memtable.insert(batch); // 再进 MemTable    return OK();            // 最后再对外返回成功}
```

这条顺序背后其实有很强的因果关系：

* 先有 WAL，崩溃后才可恢复
* 再进 MemTable，当前读请求才看得到最新值
* 最后返回成功，外部语义才站得住

如果把顺序反过来，比如先改内存、后写日志，中间一旦崩溃，就会出现：

* 内存改过了
* 日志没落下
* 重启后恢复不出这次写入

## 真正常见的问题，不在“会不会写”，而在边界怎么守

### WAL 先写了，但没真正刷稳

如果系统对“返回成功”的定义太松，比如只是写到 page cache 就算成功，那么它保证的就不是最强持久化语义。

所以工程里一定要想清楚：

* 成功返回是指 append 成功
* 还是 fsync / fdatasync 成功

### 只看 MemTable，不看 WAL 的批次边界

这样会丢掉写批次的原子语义。

因为恢复时真正的事实来源是 WAL，不是崩溃前那一瞬间已经来不及保存的内存状态。

### 并发跳表里最怕“节点发布过早”

如果 next 指针已经对外可见，但节点里的 key/value 还没准备好，就会让读线程看到半初始化状态。

所以并发设计最重要的不是“用了跳表”这几个字，而是：

* 节点何时构造完成
* 指针何时发布可见
* 发布前后哪些字段绝不能再改

## 开源系统里常见的取向

很多 LSM 系统在大方向上都很像：

* 用 WAL 兜住恢复和原子边界
* 用有序 MemTable 接住当前写入
* flush 后再形成 SSTable

常见差异主要在这些地方：

* WAL 记录格式怎么切块和校验
* MemTable 用跳表、树还是别的有序结构
* arena 怎么管理
* 并发写入是单写线程串行、还是多写线程并发插入

也就是说，WAL + MemTable 这条主线很稳定，但具体工程取舍会随着系统目标不同而变化。

## 给团队的落地判断清单



如果把这篇压成一组最值得落地的判断，通常是这些：

* 先把“成功返回”到底意味着什么定义清楚，是 append 成功，还是 flush 成功
* WAL 记录必须有明确边界和校验，崩溃恢复时才能识别尾部残缺记录
* 写批次如果承诺原子，就要在 WAL 层保留批次语义
* MemTable 如果要支持 flush 输出有序结果，优先考虑天然有序结构
* 跳表节点一旦发布，内容尽量不可变，降低并发读写推理难度
* 大量小节点优先评估 arena 分配，别让通用 allocator 成为瓶颈
* 并发设计重点看“节点何时可见”，不要只盯着有没有锁
* 整条写路径要始终记住一个顺序：先 WAL，后 MemTable，最后才是成功返回
