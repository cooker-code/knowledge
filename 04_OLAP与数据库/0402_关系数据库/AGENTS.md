## 用户认知重点

| 项 | 内容 |
|---|---|
| 已知基础 | 用户知道关系数据库做事务、查询、索引、复制，无需重复 SQL 入门 |
| 待补边界 | 与嵌入式分析（SQLite/DuckDB）、OLAP 引擎、分布式 NewSQL 的差异 |
| 易偏差点 | MySQL 与 PostgreSQL 经验互相套用；TiDB 当成 MySQL 集群版 |
| 优先抽取 | 存储引擎、MVCC、WAL、索引、执行计划、复制与高可用 |

## 排重指纹

```text
关系数据库本体 + 存储/事务/索引/查询/复制模块 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 技术本体相同 | 继续比较模块和机制，不直接判重 |
| 解决问题相同 | 没有新增边界、反例、证据或实践动作时合并 |
| 只多了案例 | 案例不改变准则时，只追加原文锚点 |
| 多了观点冲突 | 必须记录为认知校准点 |
| 与嵌入式分析混淆 | 单文件、本地、单写者归 SQLite；服务化治理留本类 |

## 已覆盖问题（按三级节点）

| 节点 | 已覆盖 | 还缺 |
|---|---|---|
| MySQL | 候选文章涉及 MVCC、定时任务、连接池实践 | InnoDB 内核、复制、Binlog、慢查询调优证据 |

## 待补缺口

| 优先级 | 项 | 为什么补 |
|---|---|---|
| 高 | PostgreSQL WAL 与归档管理 | 数据库可恢复性的核心，候选文章已有但未沉淀 |
| 高 | MySQL MVCC 与 InnoDB 锁 | 排障和并发控制的基本盘 |
| 中 | MySQL vs PostgreSQL 选型 | 候选文章已对比，需固化为决策表 |
| 中 | TiDB 新引擎与 HTAP 边界 | 与 MySQL/分析型数据库的差异不清晰 |
