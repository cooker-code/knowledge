我能够找到的最早的关于 TiDB X 的起源的材料，大概在 2021.4 在某个 CCF 的会议上。不过在这个分享之前，已经有这个想法很长时间了。

从 LSM-Tree 到存算分离

最早是来自于 Max (我们 CEO) 大概在 2017 年开始的一个玩具项目，目的是为了方便测试一些事务的优化的想法，也是为了节省跑 SQL 层单元测试的时间，另外当时我们非常欣赏的 WiscKey (FAST16) 解决 LSM-Tree 写放大和 Compaction 的思路（通常简单的方案都是更好的方案），当时 Dgraph 这个公司正好开源了 WiscKey  的 Go 语言开源实现 BadgerDB（目前这个公司已经不在了，不免唏嘘）。作为 Go 语言的粉丝，很自然的我们就基于这个项目开始在上层从新开始实现一个 TiKV 协议兼容的分布式 KV 存储，因为想着万一未来要上手修个 Bug 什么的，我们几个作为 Go 的老手，修 Go 写的代码当时还是比较自信的（后来在证明也是想多了，因为 TiDB X 后来还是用 Rust 写了😂），也就是那个时候，随着 Snowflake 的上市，云原生数据仓库这个概念开始进入大家的视野。

其实早在 Snowflake 上市之前，我就很欣赏 Snowflake 的架构，而且 Snowflake 的论文是对我影响影响非常深远的一篇论文（我 2019 年写的文章为证：https://www.infoq.cn/article/x7hvzr1qtgamwbxow0bl）

但是 Snowflake 的架构好虽好，但是最大的问题是：他们是个数仓，而 TiDB 是个 OLTP 数据库！学他们用 S3 做在线数据库的存储引擎，延迟不得高上天了？

不过我们 PingCAP 的风格一直是敢想敢做的，我们就开始想：既然 WiscKey 都能在 LSM-Tree 层面上搞 Key 和 Value 分离，为什么不搞的彻底一点？把 LSM-Tree 的 SST 干脆放到 S3 上，本地只缓存 SST 和 Raft 日志就好了？

（我在 2021 年 4 月 CCF 的分享）

（2022 年 GTLC 杭州， 在分享前用酒店便签纸手绘的第一个 TiDB X 的架构图）

其实也并不是只有我们观察到这个趋势，至少当时 Dhruba (RocksDB 的原初作者) 的创业公司 RockSet 已经比我们更早一步开始在 RockDB 上魔改嫁接 S3 的尝试, 并开源为 RocksDB-Cloud 这个项目

(https://medium.com/rocksetcloud/remote-compactions-in-rocksdb-cloud-f2424050e575, RockSet 的分享)

而且 RocksDB 官方在 2021 年也引入了 SecondaryCache 的接口，虽然原意大概是为了在 NVME 上实现 Block Cache 缓存，但是对于 LSM-Tree 这样分层结构，其实非常适合使用云存储作为更冷数据的缓存。

RockSet 的出发点是希望能使用 S3 来解决本地 Compaction 带来的抖动和分布式构建覆盖索引（Converged indexing）的问题，因为 RockSet 的定位其实是一个 OLAP 数仓；而 RockDB 官方团队只是想解决二级缓存的问题。当然这些都是很有意义的问题，只不过还不够，我们希望更进一步，通过这条路，构建完整的 Serverless 云 OLTP 数据库。

大概在 2022 年，又一次重新开始云原生的 TiDB 引擎的项目的时候，我们定了些非常具象的目标：

* “我们要给全世界所有的开发者提供免费的 TiDB 云服务”
* “新集群秒级别创建，开箱即用，开发者完全自助”
* “单集群不用介入任何人工操作，分钟级别完成从 0 到 100,000,000 级别的 QPS/TPS 的扩展性，并且能马上缩到 0”
* “目标用户的数据规模从 KB 到 PB 级别全覆盖”
* “不能依赖任何特殊硬件和假设，只能用云提供的公开服务和 API”

...

要知道那个时候一个最基础最小的 TiDB 起步集群的月成本大概是几百美金，全自动化的运维脚本启动一个 TiDB 集群也得是几分钟的级别，还没加上在云上动态的申请资源的时间，扩容一下都是按天起步（很多用户集群大点，都需要好几天的均衡时间。。。）而且我们一直强调的是没有 100 TB 就不是 TiDB 的适用场景。所以这个愿景是相当于在当前系统上实现 1000x 的优化。

人的思维总是有惯性，我们过去犯过的一个错误就是掉进经验主义的陷阱：因为在一个已有系统上进行打补丁型的优化，很容易得到 10%，20% 的效果，牛逼一点 100% 已经很不得了了。尤其是做底层系统出身的工程师，目光向下压榨机器的性能其实是一个习惯性的动作，当目标超出理论的极限的时候，很自然的会觉得不可能。

写到这里，我不禁回想起现在大红大紫的 AI，底层基于连结主义的神经网络的理论基础在上世纪 40 年代就被提出，反向传播网络的发明也都能追溯到 80 年代，短暂的喧嚣之后经历了长久的沉寂，正式因为构建大规模神经网络需要的算力过于恐怖，远远超出了当年 CPU 的能力极限，不可能得到有意义的结果。直到 2012 年 Ilya 和 Hinton 在 AlexNet 上使用 GPU 得到惊人的结果后，才有了这一轮 AI 的复兴，不过要知道，2012 年时 GPU 已经发明出来十几年了（算上 CUDA 也 6 年多了）。

回到 TiDB X，有了上面那样的目标，团队自然很难用线性的思维去思考系统的设计，尤其是 TiDB 已经在传统的 Shared-nothing 上取得了如此的成功，团队有着极大的惯性。所以我们当时的想法是赌一把，通过这个看似不可能的愿景，牵引团队往更创新和大胆的方向尝试，现在回头看，这个目的达到了，我们的团队确实厉害。

如果现在马后炮的总结一下完成这一切的架构的关键点，我认为有几个很重要的指导思想：

1. 首先要清醒的意识到，要完成上面那些目标，我们得忘记我们是数据库的开发者和厂商：我们并不是在做数据库，而是做一个“在线服务”，只不过提供的服务内容是 SQL 数据库而已。这一点会带来深远的影响，从团队配置到技术选型都是，选择 SOA（Service-Oriented Architecture）是必然的，后来实践中发现 SOA 带来的好处还不仅仅是多租户和成本控制，甚至可以说大多数好处都是 SOA 带来的，以后有空再细聊。

2. 谜底就在谜面中：如果需要为全世界的开发者提供免费数据库的话，每个人分配一个真实的数据库实例的方式是完全不可行的，就算不考虑成本，哪怕是上千万个实例的管理成本，就是一个巨大的问题，我们的方案是虚拟集群（Virtual Cluster）。因为只要目光往上（往业务）看，很快就能发现：99.9% 的用户是冷的，只会偶尔来几条请求，但是你也不能不服务这些零星的请求，而且你也不知道哪些客户会突然变大。多说两句，这也是我不认可现在 Supabase 和 NeonDB 的地方，并不能说把 pg 的存储引擎换成 s3 就是 Serverless db了。

3. “Log is the database”，这一洞察并非我原创，而是来自于 AWS Aurora，要说 Aurora 为业界做出的重大贡献，我认为这句话绝对能占其一，其实我认为这句话的精妙之处在于：我们能舍弃什么。现代的硬件（尤其是内存越来越大，NVMe 的普及）和网络加持下，我们事实上可以通过缓存解决读的问题，而 SSD 的特性更加适合 Append 的写入模式，对于 Log 是非常友好的，另外随着硬件成本的降低，可用性也好解决：多写几个副本就好了，即使本地日志丢失，从其他节点拉日志回来重放一下就好。但是这个设计的好处是明显的：首先日志是很简单的数据结构，只需要保证日志的持久化和可用性，你其实就保证了数据库的数据安全性，极大降低了数据库内核开发者的心智负担。事实上也解耦了 Snapshot 和 Log 的存储方式。

当然并不是说我们要抛弃一切重新开始，老的 TiDB 的 Shared-nothing 架构给了 TiDB X 打下来了很好的基础。首先 SQL 层（tidb-server）是无状态，进行服务化的时候，减少了很多工作量，而且关于 SQL parser 和优化器的几乎所有工作都是可以复用的，这个很好理解，而且当时为了实现无感升级我们把 TiDB 的 session management 抽出来做的 tidb-proxy，也被复用到了 TiDB X 的网络层，作为 gateway 使用，一点没浪费。

另外一个可以复用的宝藏是：传统的 TiKV 内部事实上已经实现了数据的动态分片，包括 Range-based 的自动分裂/合并逻辑，只是过去我们的分片和物理的存储绑定的，分片的移动就是数据的移动，这也是扩容的时候速度堪忧的根源。在 TiDB X 里，即使新的 TiKV 变成了无状态的服务，我们仍然需要分片的概念，否则单机的计算和带宽会变成瓶颈。只不过在 TiDB X 中我们的 Key Range 的大小可以切得更大（为什么可以变得更大？变大了有啥好处？这里可以作为读者的思考题）。

当物理分片变成了逻辑分片后 + 所有数据（Snapshot）都存储在 S3 上，带来的好处显而易见：在 TiDB X 上，我们把扩容的速度推向极致，我可以瞬间启动成百上千个 TiKV 实例，然后同时从 S3 上加载各自负责的 Range 的数据文件，而瓶颈只会是 S3 的吞吐和本地计算节点（TiKV）的缓存预热速度，一切都是并行的。缩容就更简单了：调整 Routing 策略，然后直接 shutdown node 就好。老实说，没有蚂蚁搬家式的数据移动，一切都简单了很多，而且扩容的速度提升了百倍都不止。。。。

而且就像上面提到的，因为有分片，我们可以把日志（Raft Log）的写入流也按照分片进行切分，这样就可以避免 Supabase / NeonDB 这些直接拿 Pg 换存储引擎的方案带来的计算和单点写入瓶颈的问题。

另外从 TiDB 一开始我们就选择的 SQL on KV 的抽象让我们的很好实现逻辑的租户隔离，只需要对 KV 的访问加上 Key 的前缀，类似: key -> value 变成 tenantID\_key -> value 即可。所以对于每一个租户来说，在存储上的物理 Footprint 是很少的，这也是对于海量小而冷的用户，我们能提供极致低成本的原因。
