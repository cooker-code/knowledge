| 节点 | 用途 |

## 用户认知重点

| 项 | 内容 |
|---|---|
| 已知基础 | DuckDB 适合本地分析与文件查询；SQLite 适合本地单文件事务型存储 |
| 待补边界 | 进程内分析、嵌入式事务、本地优先应用与服务化 OLAP/OLTP/DataFrame 工具的边界 |
| 易偏差点 | 把 DuckDB 当"小型数据库"；把 SQLite 当 PostgreSQL 低配替代 |
| 优先抽取 | 向量化执行、并行调度、扩展机制、WAL/锁、类型系统、文件扫描 |

## 排重指纹

```text
嵌入式分析 + 执行/存储/扩展/事务模块 + 核心机制 + 解决问题 + 使用边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是 DuckDB 快 | 按向量化执行、列式存储、文件扫描、优化器、并行执行拆分 |
| 都是 SQLite 单文件 | 按 WAL/锁、类型系统、索引、备份迁移、扩展能力和应用集成边界拆分 |
| 与 OLAP 引擎混淆 | 服务化、高并发、集群查询归 OLAP；进程内本地分析归本类 |
| 与关系数据库混淆 | client/server、权限治理、HA、集中运维归关系数据库 |

## 已覆盖问题（按三级节点）

| 节点 | 已覆盖 | 还缺 |
|---|---|---|
| DuckDB | 嵌入式分析定位、向量化执行与 Pipeline | MVCC、优化器、扩展机制、Morsel-Driven 并行、与 Polars/Pandas 对标 |
| SQLite | 单文件嵌入式事务型数据库定位、WAL/单写者边界、类型系统和 STRICT 表候选文章 | B-tree/Pager、WAL/checkpoint、锁与并发、FTS/JSON/扩展、备份迁移、与 PostgreSQL/DuckDB 对标 |

## 待补缺口

| 优先级 | 项 | 为什么补 |
|---|---|---|
| 高 | DuckDB MVCC 与并发边界 | 需要明确它能不能替代服务型数据库 |
| 高 | SQLite WAL/锁/单写者边界 | 区分本地、单应用服务器场景，避免按 PostgreSQL 经验误用 |
| 中 | SQLite 类型亲和性与 STRICT 表 | 影响数据约束、迁移和应用建模 |
| 中 | DuckDB 优化器与文件扫描 | 影响 Parquet/CSV 本地分析效率 |
| 中 | DuckDB vs SQLite vs PostgreSQL | 帮助判断本地分析、嵌入式事务和服务端业务库选型 |
| 中 | DuckDB vs Pandas/Polars/ClickHouse | 帮助判断本地分析和服务化分析选型 |
