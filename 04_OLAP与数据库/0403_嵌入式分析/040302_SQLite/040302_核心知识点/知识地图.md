# 040302_SQLite知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 04_OLAP与数据库 / 0403_嵌入式分析 / SQLite |
| 解决的问题 | 把 SQLite 作为单文件、进程内、事务型嵌入式数据库来整理，避免误当 PostgreSQL 低配版或 DuckDB 分析引擎 |
| 典型输入 | 本地 SQLite 文章、官方文档、应用实践、最小实验 |
| 典型产出 | 选型边界、建模准则、锁与 WAL 判断、备份迁移规则、扩展能力边界 |
| 用户阅读目标 | 读到一篇 SQLite 文章时，先判断它补的是场景边界、写入事务、类型系统、索引查询、可靠性还是扩展能力 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| S00 场景边界 | 数据和应用是否在同一台机器，是否读多写少，是否单应用服务器 | SQLite 是单文件低运维方案，不是 PostgreSQL 低配替代 | 架构降级、本地优先、MVP、内部工具文章进入 |
| S01 数据建模 | 表、rowid、主键、类型亲和性、STRICT 表如何设计 | 动态类型带来灵活性，也带来约束和迁移风险 | 建表、类型系统、约束、迁移文章进入 |
| S02 事务并发 | WAL、单写者、锁、busy timeout 和 checkpoint 如何工作 | WAL 改善读写并发，但不取消单写者边界 | 事务、锁、并发、WAL、崩溃恢复文章进入 |
| S03 查询索引 | B-tree、普通索引、FTS、JSON、查询计划如何服务查询 | SQLite 查询优化要看索引、页访问和 SQL 形态 | 索引、全文检索、JSON、执行计划文章进入 |
| S04 文件可靠性 | 备份、VACUUM、文件复制、损坏恢复和版本兼容如何处理 | 嵌入式数据库可靠性重点是文件生命周期 | 备份、恢复、迁移、文件系统边界文章进入 |
| S05 扩展集成 | load_extension、语言绑定、ORM、本地队列等扩展如何接入 | 扩展可以减少基础设施，但会引入安全、版本和生态成熟度风险 | 扩展、绑定、ORM、队列、桌面/移动集成文章进入 |

## 待补缺口

| 流程节点 | 缺什么 | 后续文章进入时怎么判断 |
|---|---|---|
| S00 场景边界 | 缺少真实应用案例的容量、并发和迁移证据 | 看文章是否给出数据量、写入频率、部署形态和迁移路径 |
| S01 数据建模 | 缺少 rowid、WITHOUT ROWID、外键、CHECK、迁移策略的系统整理 | 看文章是否能形成建模准则，而不是只列命令 |
| S02 事务并发 | 缺少官方锁机制、WAL checkpoint、busy timeout 和崩溃恢复证据 | 看文章是否能解释锁状态、失败模式和验证方法 |
| S03 查询索引 | 缺少查询计划、B-tree、FTS5、JSON1 的边界 | 看文章是否能说明索引适配条件和不适用场景 |
| S04 文件可靠性 | 缺少 backup API、VACUUM、在线备份、文件复制风险和损坏恢复 | 看文章是否能沉淀操作步骤、风险和验收信号 |
| S05 扩展集成 | 缺少 Honker、sqlite-vec、FTS5、ORM 绑定等扩展的成熟度判断 | 看文章是否讲清扩展安全、版本兼容、生态活跃度和替代方案 |

## 文章进入时的对比规则

| 判断项 | 处理方式 |
|---|---|
| 补充 | 文章提供新的锁边界、类型系统、文件可靠性、扩展能力或可验证实践时，写入对应流程节点 |
| 冲突 | 文章把 SQLite 说成 PostgreSQL 替代品或高并发服务端数据库时，先记录冲突并降权 |
| 不同层次 | 入门命令、内核机制、应用实践、扩展生态分属不同层次，必须标注后再吸收 |
| 降权 | 只有“轻量、简单、重新流行”等情绪表达，没有边界和证据时，不进入核心知识点 |

## 使用方式

- 先读本文件的宏观流程，再读具体主题知识点。
- 新文章不要直接总结 SQLite 优点，先归入 S00-S05。
- 精读时优先补“边界和失败场景”，再补命令和用法。
