## 知识点入口

- 本模块先看宏观流程，再看文章：[知识地图](040400_核心知识点/知识地图.md)。
- 新文章必须先判断“搜索技术本体”是否成立，再进入具体技术目录。
- `文章/` 只作为临时入口；正式归档必须落到 `Elasticsearch/`、`040402_Pizza Engine/` 或后续新增的具体搜索技术目录。

## 类目定位

| 项 | 内容 |
|---|---|
| 一级类目 | OLAP 与数据库 |
| 二级类目 | 搜索与索引 |
| 核心问题 | 面向文档、日志、文件、地理空间、知识片段等非结构化或半结构化数据，如何建立索引并提供可检索服务 |
| 不解决什么 | 不承接 Agent 检索策略、RAG 召回编排、Claude Code 搜索代理、代码搜索工具、普通后端文件上传流程 |
| 用户当前认知假设 | 用户知道 ES 是搜索引擎，但需要补倒排索引、分片、副本、一致性、写入链路、空间索引和与数据库索引/RAG 检索的边界 |

## 划分准则

`04_搜索与索引` 不是“凡是出现搜索两个字都放这里”。只有文章主问题满足下面任一条件时才归入本目录：

| 保留条件 | 归入位置 | 示例判断 |
|---|---|---|
| 讨论搜索引擎本体、倒排索引、分片、副本、写入可见性、查询 DSL、搜索集群运维 | `Elasticsearch/` | ES 倒排索引、文档索引过程、空间索引、搜索平台架构 |
| 讨论轻量或嵌入式搜索引擎本体，重点是本地索引、浏览器内查询、边缘搜索 | 具体技术目录，如 `040402_Pizza Engine/` | “像 SQLite 一样无处不在”的搜索能力 |
| 只把 ES 当作后端系统的一个组件，但核心问题仍是搜索索引服务 | `Elasticsearch/`，并在笔记中标记“集成案例” | SpringBoot + Minio + ES 文件内容检索 |

必须移出本目录的情况：

| 误归类型 | 应归位置 | 判断理由 |
|---|---|---|
| RAG 论文、Agentic RAG、检索增强生成流程 | `02_Agent与AI工程/0203_RAG与知识库/` | 主问题是模型应用的召回与推理流程，不是搜索引擎本体 |
| Claude Code 搜索代理、Context Engineering 搜索策略 | `02_Agent与AI工程/0205_AI编程工具/020502_Claude Code/` 或 `02_Agent与AI工程/0208_Context Engineering/` | 主问题是 AI 编程工具如何找上下文 |
| 代码搜索工具、CLI 搜索工具 | `02_Agent与AI工程/0205_AI编程工具/` 或 `09_电脑工具/0901_开发工具与CLI/` | 主问题是开发工具，不是数据库搜索服务 |
| 语音转录、本地知识工具、文档工具中附带全文搜索 | `09_电脑工具/0902_文档与知识工具/` | 搜索只是工具功能的一部分 |

## 排重准则

问题指纹：

```text
搜索技术本体 + 索引/写入/查询/分布式模块 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都讲 ES 基础概念 | 继续比较是否新增倒排索引、写入链路、一致性、分片副本、选主或空间索引边界 |
| 只是接入案例 | 如果没有新增索引设计、查询建模、失败模式或验证指标，只保留路由锚点 |
| 和 RAG 混淆 | RAG 的召回策略归 Agent 与 AI 工程；只有底层搜索引擎机制才留在本目录 |
| 和数据库索引混淆 | 单机关系库 B+Tree、执行计划索引选择归关系数据库或查询优化，不归搜索引擎目录 |
