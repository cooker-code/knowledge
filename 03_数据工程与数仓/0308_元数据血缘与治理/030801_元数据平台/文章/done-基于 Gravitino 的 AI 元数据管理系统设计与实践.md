**01**

**业务背景：AI 时代下的数据存储与元数据管理挑战**

首先探讨一下 AI 时代下数据存储与元数据管理面临的挑战。

**1.****AI****时代的数据增长压力**

**自 2022 年 11 月 ChatGPT 发布以来，生成式 AI 已经在深刻地改变人们的生产生活，例如 Cursor、Windsurf、Github Copilot 等工具，大大提升了工程师的工作效率，这些商业化大模型产品的落地，离不开海量的高质量数据作为支持基础。知乎在生成式 AI 的浪潮中也推出了和面壁合作开发的中文大语言模型“知海图”，以及在线 AI 搜索引擎产品“知乎直达”，收获了不少的用户和关注度。业务蓬勃发展的背后，是存储端训练数据集规模的显著增长。目前知乎使用 Hadoop HDFS 作为主要的数据存储系统，数据规模的增长给我们平台的运维带来了两方面的压力。**

**一方面是存储空间的压力。知乎的 hadoop 集群每天都会有大量离线与在线计算任务在上面运行，这些任务会产生大量的结果文件和临时文件，为保证集群的平稳运行，集群需要保持一定的存储水位。**在结构化的表数据管理方面，业界已经有一套成熟的方法论，包括采用高效的压缩格式、低频访问数据转 EC、设置表的生命周期等策略。然而，针对算法工程师高频使用的非表数据目录，当前尚缺乏行之有效的治理手段。项目实施前的统计数据显示，非表数据在整体存储空间中的占比达 43%，且仍呈持续增长态势。

**另一个方面是元数据对象的管理压力。基于 HDFS 的实现，单个 NameNode 能管理的元数据对象是有限的。对于结构化的表数据，我们可以通过小文件合并，合理的分区过期时间等治理方法去控制小文件个数，减少元数据管理压力；而对于非表数据治理，同样缺乏管理手段。**

总结一下，主要是两个方面：一是业务的不断增长使得算法侧数据存储量暴增，而算法工程师缺乏高效易用的治理工具对这部分数据进行治理；另一方面存储侧虽能根据底层审计日志做低频访问的数据定位，但缺乏更多信息来判断数据是否可以做转冷或者删除等操作，因此需要一个系统对 AI 资产进行元数据管理。

**2.****现有元数据工具的局限性**

**元数据管理在业界其实并不是一个新鲜的话题，已经上有很多开源的元数据管理工具，下面一起来看下这些工具的基本功能与适用场景。**

**首先是 Amundsen,第一代的数据管理工具，是一个可以支持底层多数据源和上层多引擎的统一元数据管理平台。它对于元数据的获取策略是定时请求并保存到自己的状态后端里，存在无法感知底层元数据变更的问题，需要自行通过其他手段保证强一致性。**

**第二代元数据管理工具 DataHub 支持基于推送的架构收集元数据，可以实现比较好的一致性，但是只能保证其源端和 datahub 的一致性，对于下游其他依赖这张表的元数据还是需要手动变更。**

**后来出现第三代元数据管理工具 Open Metadata和MetaCat，他们支持了元数据的变更可实时订阅，保证了整个链路上的变更可以自动完成，降低了人工漏操作的可能。同时，Open Metadata 还提出了开放元数据标准的概念，通过提供统一的元数据模型与 API 来降低上层业务接入的接入复杂度和工作量。**

**上面这些都是在 2022 年之前开源的一些数据管理工具，属于前 AI 时代的元数据管理工具，主要解决的问题是结构化数据的管理问题以及数据孤岛问题，而对于我们当前面临的 AI 资产管理，例如非结构化、半结构化数据的目录管理场景，这类工具力有所不逮。**

近几年，开源社区涌现出一批新的元数据管理工具，这些工具可实现表数据与 AI 数据的统一治理，并提供统一的元数据管理层，不仅能对表数据进行管理，还能对权限、血缘、审计等关键信息进行统一治理。其中，比较典型的有 Unity Catalog 和本文将要重点介绍的 Gravitino**。**

**3.****平台元数据管理现状和痛点**

**在了解了市场上一系列元数据管理工具后，再来了解一下知乎的元数据管理建设现状。**

架构图的左半部分是数仓工程师和数据分析师日常使用的大数据平台，涉及到离线任务和在线自助分析等场景。在业务元数据层的 meta-api 服务，主要管理一些包含业务信息的元数据，比如表的业务层级、数仓层级、所属项目组和生命周期等，在基础元数据，通过 Hive Metastore 组件来进行表的元数据管理，最终管理底层存储层的 Hadoop 目录，或者是一些 Paimon、Iceberg 之类的开放表结构。目前，这一链路的主要问题是接入新数据源的成本较高。每当要接入一个新的数据库作为数据源提供上层查询时，平台工程师都需要根据数据库的元数据访问方式去实现一套 API，再进行接入。复用的可能性较小，上层接入和数据源维护存在较大难度。

右半部分为机器学习平台，主要供算法工程师进行深度学习任务的调度与管理工作。该平台有一个简单的模型管理功能模块，记录了模型存储位置，但训练数据集、模型的生命周期等信息并未进行等级、管理。信息的缺失导致平台侧难以开展有效的数据管理，也增加了与算法工程师协同进行数据治理的难度。

另外，可以看到，整个元数据层其实是各自为战的，缺乏一个中心化的、统一的元数据管理层去实现跨 schema 的统一审计、全链路追踪和权限管理。

基于上面的这些现实情况，我们的短期目标是实现 AI 资产的元数据管理，中长期目标则是搭建统一的元数据层，从而降低数据扩展、数据接入和数据治理的难度。

**4.****Gravitino****的架构与选择理由**

**经过对比，我们最终选择了 Apache Gravitino 作为元数据管理系统的核心服务。**

**Apache Gravitino(incubating)****是一个高性能的地理分布的联邦的元数据湖。通过使用技术数据目录和元数据湖技术，可以对你所有的数据源进行访问和治理，也可以在各种云的厂商通过用各种格式来支持你的计算引擎，像 Spark，Trino，Flink 和 ClickHouse，包括你的一些深度学习的框架，如 Pytorch 和 TensorFlow。**

**我们为什么选择了 Gravitino？大家可以看一看上面的架构图，能够给我们一些答案。**最下面一层是连接层，负责从数据源处获取元数据。这里采用了插件化设计，新增数据源只需要对插件接口进行实现，降低了新数据源接入的难度。

上面是 gravitino 的核心层。Metalake 是元数据的最高层级，向下依次为 Catalog、Schema，以及最底层的 Table/Fileset/Model/Topic 这样的三层架构。Gravitino 通过这样一个标准化的三层架构实现对元数据的统一管理。无论是表格、文件目录、消息队列，还是大模型，均可通过该结构进行标准化访问。

基于核心层的标准化的统一视图，上层的接口层可提供规范的 API 接口，便于上层数据平台的元数据服务与应用通过标准 API 进行数据交互。标准化的优势在于既能降低后期系统变更的风险，又能显著简化新数据源的接入流程。

**另外，Gravitino**支持市面上主流的计算引擎；它还有一个非常活跃的社区；由一个中立的公司来进行孵化，没有和特定的厂商去做深度绑定，这些也是我们选型时的一些考虑的点。

Gravitino 提供了多种基础元数据模型，我们在实现 AI 资产的数据管理系统时，主要是针对非结构化和半结构化数据，在 gravitino 侧，主要通过 Fileset 进行支持。

上图中展示了 Gravitino Fileset 示意图。从明细内容可见，其核心信息是记录数据的存储位置，这也是 Fileset 的主要功能体现。此外，Fileset 支持自定义添加属性，还能通过 Gravitino 内置标签进行标记，以便快速筛选目标 Fileset。Fileset 的另一核心特性在于，通过 Catalog、Schema、Fileset 三级命名空间体系，可直接实现数据访问，而无需关注底层实际存储路径，这一机制将在后续内容中详细解析。在存储兼容性方面，Fileset 支持 HDFS、S3 等多种存储协议，这一特性在数据治理场景中尤为重要，例如当需要将不适合存储于 HDFS 的数据迁移至对象存储时，该功能可提供灵活高效的支持。

**02**

**系统设计：Gravitino 的核心能力与知乎的定制化扩展**

**前文提到，整个系统构建会基于 Fileset 实现，若想将元数据管理能力作为平台功能对外提供服务，我们还需完成其它一些工作。**

**1.****AI****元数据管理系统设计**

**第一，API 能力封装与功能扩展**。基于 Gravitino 提供的 REST API，需在平台层进行二次封装，同时针对实际高频场景中缺失的功能进行拓展开发，以满足个性化业务需求。

**第二，构建数据治理闭环**。围绕 Gravitino 的基础功能，设计并落地覆盖数据采集、存储、管理、使用全生命周期的数据治理流程，形成从元数据标准化到质量管控、安全审计的完整闭环。

**第三，开发文件系统浏览器**。针对算法工程师需通过登录服务器执行 Hadoop 命令访问文件、效率低下的问题，开发文件系统页面浏览器，支持文件快速预览与问题排查，提升操作便捷性。

**2.****AI****元数据管理系统功能展示**

**经过一段时间的开发后，系统顺利上线。下图展示了我们的大数据平台上的一个详情页。**

**可以看到在目录层级方面，参考了底层的设计，实现了三级目录；也可以看到 Fileset 对应的标签，可以记录一些业务相关的信息；存储路径记录实际的底层文件系统存储位置；生命周期规则记录用户设置的生命周期策略等。同时，还可以在页面上查看当前数据集的存储历史、生命周期日志，以及文件浏览器。**

**值得一提的是，按照最初的设计，我们希望能够通过用户输入的目录自动识别文件及 Fileset 名，自动在平台层进行录入。但因为我们的业务方向较多，各个业务方向团队的目录并不是特别规范，所以自动识别方案最终没有落地。**

因此，我们通过与各团队达成约定，统一规范目录格式，避免新增非标准格式。平台侧主动执行目录扫描，将符合标准的日期格式纳入系统支持的分区策略与格式体系。同时，我们也会与机器学习平台协作，推动存量模型主动同步至本系统，减轻用户的手动导入负担。

在业务侧，我们需确保每个目录仅由一个 Fileset 管理。由于此前使用的 Gravitino 版本未对此做限制，因此需增加目录唯一性校验机制。同时，我们会对目录层级进行管控，避免过高层级的目录录入管理系统，从而影响管理效率。

**03**

**生产实践：存储治理闭环与性能优化**

**1.****HDFS****存储治理**

**上线整个系统最主要的目的就是对AI资产数据进行管理。用户在平台上录入完成后，平台侧就可以进行存储的治理。**

**当前平台执行治理的核心流程为：**

* **算法工程师在平台上创建一个数据集，并指定规则。规则包括两种，一种是是否过期，决定了数据集是否会被删除,成为 TTL；另一种是是否低频存储，决定了是否要对数据集采用更高效的编码方式,称为 EC。**
* **平台调用 Gravitino API，创建 Fileset，同时进行 tag 打标。**
* **平台有离线任务，每天会基于 Gravitino 的元数据和 Hadoop 的 HDFS fsimage 计算出当天需要进行操作的一些表，将这些目录下发给具体执行做 TTL/EC 的 worker 进行实际操作。**
* **操作会记录审计日志，并提供给大数据平台进行访问查看。这样用户在第二天就可以通过平台去查看昨天有哪些数据被删掉了，有哪些数据被转成了 EC 的，或者转成了低频存储，整个目录的存储的使用情况是怎样的，以便进一步优化策略。**

**前面提到 Fileset 不仅能实现对元数据的管理，还可以通过三级命名空间做数据的访问。目前我们主要支持了通过 Python 进行访问的场景。社区中其他使用较早的一些公司，也有可以使用 Spark Jar 或 Spark SQL 进行访问的一些例子。这也是我们后续工作的方向。**

**2.****GVFS****在机器学习平台的使用**

基于 GVFS（Gravitino Virtual File System），我们目前正在落地的一个实际场景如下。当前机器学习平台完全基于 K8s 部署，训练任务对数据读取性能要求较高，因此我们在每个机房均部署了缓存服务以加速训练任务的数据读取。目前通过 CSI（容器存储接口）机制启动 Fuse Pod 提供支持，但该方案存在资源占用问题：若要保证较好的性能，需为每个节点的每个 NameSpace 下单独部署一个 Fuse Pod。在有很多 node 的大集群里，这些 Fuse Pod 所用的 CPU 和内存资源也是一个不小的数目。

为解决缓存服务资源占用问题，我们引入了 GVFS 访问方案，从 CSI Fuse 模式，转向以 S3 协议的方式提供支持。用户只需在代码中采用 GVFS 文件协议进行底层数据访问，仅需一次代码调整即可完成适配。而在 GVFS 客户端，我们添加内置智能路由逻辑，结合平台注入容器的一些配置，可以自动路由到容器距离最近的缓存服务。未来即使需要变更缓存服务，也可用户无感的完成。

通过启用 GVFS 方案，可以移除 Fuse Pod 的资源占用，显著降低集群整体资源负载，这是我们当前重点推进的优化方向。

**3.****基于用户名和密码的验证方式**

**当前 Gravitino 提供了三种接入方式：只记录用户信息，不做验证的 simple 模式，基于 Kerberos 的验证方式，以及基于 OAuth2 的验证方式。但三种方式均与公司情况不太匹配，所以我们实现了一个基于用户名密码的验证方式。**

**04**

**总结与展望**

**1.****项目收益**

**项目的主要目标是实现 AI 资产的元数据管理系统，目前随着元数据治理工作的开展，相关问题已经得到收敛，增长趋势得到了有效控制。**

**EC****的收益也初见成效，收益显著，已达到 PB 级别。每月能为公司相关团队降低数万元的存储成本。**

**从长远来看，引入新组件 Gravitino，为后续实现统一的元数据管理层这一大目标提供了可行的路径。**

**2.****后续工作计划**

**后续工作将从以下多个关键方向推进：**

首先，针对社区 0.8 版本提出的大模型支持，我们计划深入研究该功能在生产环境中的适用性，探索将其引入实际生产流程的可行性。

第二，**我们将继续拓展 GVFS 的应用场景，致力于构建统一的数据访问模式，以提升数据使用的便捷性与高效性。**

**第三，积极探索 Iceberg Rest Server 在公司内部的应用场景。**

**第四，尝试接入更多内部数据源，实现统一管理，降低管理的复杂度。**

**最后，尝试构建跨 schema 数据血缘，开展全流程的数据审计。**

**3.****元数据管理愿景**

**从长远愿景来看，我们期望借助 Gravitino 在元数据层实现统一收口，从而在数据血缘追溯、数据审计以及权限管理等方面，构建高效且统一的元数据管理体系，大幅降低整体工作难度。**
