B站通过Neo4j知识图谱与MCP协议构建数仓智能化底座，实现LLM与万级数仓资产的动态交互。针对“找表难、写SQL慢”的瓶颈，以交互式开发重构prd2sql流程，形成需求补全、血缘探查、代码生成与自动校验的半自动闭环，在万级任务规模下实现80%的SQL生成准确率，为自动化治理奠定基础。

B站资深开发工程师陈哲，主导构建了B站商业数仓“智能化底座”，通过Neo4j知识图谱与 MCP 标准化感知协议，打通了 LLM 与万级数仓资产的通信闭环。目前专注于需求交付工作流（prd2sql）的智能化重构，在万级任务规模下实现了高准确率的人机协同开发实践。

面对万级任务、十万字段的超大规模数仓，如何突破“找表难、写 SQL 慢”的效能瓶颈？在2026年4月24-25日举办的DACon上海站大会上，陈哲老师将分享《LLM + 知识图谱驱动的数仓智能化实践：从基建架构到交付流重构》演讲话题，本分享介绍 B 站商业数仓如何构建智能化基建框架：以 Neo4j 知识图谱为底座，通过 MCP 协议赋予 AI 实时感知能力。在此基础上，重点拆解智能化交付实践（prd2sql）：采用 Claude Code 交互模式，将原本碎片化的开发流程重构为“需求补全-血缘探查-代码生成-自动校验”的半自动闭环。该方案实现了 80% 的 SQL 生成准确率，并为后续自动化治理奠定了架构基础。

通过本次演讲，能帮助听众获取“知识图谱 + MCP + LLM”的解耦式智能数仓大框架设计。学习如何通过“交互式开发”而非“黑盒自动化”解决复杂需求的交付质量和效率。同时能掌握在超大规模环境下提升 AI 感知确定性的实战策略。

DACon：您在演讲中会提到通过 Neo4j 知识图谱 + MCP 协议 构建了数仓的“智能化底座”。能否具体解释 MCP 协议 的设计核心是什么？它如何帮助 LLM 实现“按需、分层的元数据感知”，从而解决万级任务环境下的语义漂移问题？

陈哲：MCP 的设计核心在于标准化 LLM 访问元数据的“接口契约”。它不再是将数仓元数据死板地灌入 Prompt，而是为 LLM 提供了一套动态调用的“工具箱”。

* 按需检索（On-demand Fetching）： 只有当 LLM 明确需要查询某张表的 schema 或血缘时，才通过 MCP 触发图数据库的查询。
* 分层抽象： 协议定义了不同粒度的 Context（如：任务层、表层、字段层），定义了不同粒度的实体与关系模型，以支撑多样化的元数据查询需求。

在万级任务下，直接输入元数据会导致 LLM 的 Attention 稀释（即语义漂移）。

* 思维链路驱动的感知： 依托 MCP 的分层能力，LLM 的感知不再是漫无目的的搜索，而是遵循“以表为锚点”的逻辑，定位->通过需求解析，首先调用 MCP 接口定位到核心“锚点表”。扩展-> 以该表为中心，按需调用 MCP 的血缘或字段接口，补充必要的上下文。
* 子图挂载 (Sub-graph Mounting)： 最终注入 LLM 的并非全量图谱，而是通过 MCP 按需拼凑出的“局部子图上下文”。这就像为 AI 提供了一张局部高清地图而非全球简图。通过极高的信息浓度来解决语义漂移。

DACon：在 prd2sql 的智能化重构中，您强调了 “交互式开发” 而非完全黑盒自动化。能否分享一个典型的需求处理场景，说明 Claude Code 如何与开发者协作完成“需求补全-血缘探查-代码生成-自动校验”的闭环？过程中如何平衡效率与人工介入的频度？

陈哲：我们之所以强调“交互式”而非“黑盒自动化”，是因为数仓需求中存在大量的业务歧义。

典型场景：构建“双 11 活动新客留存”指标

1. 需求补全： 开发者输入简短需求，Claude Code 发现“新客”定义模糊，主动询问：“新客指首次下单用户，还是首次曝光用户？”并给出建议。

2. 血缘探查： 确定口径后，LLM通过 MCP 检索知识图谱，定位需求文档中的表元信息、任务生产sql，以及上游表元信息、任务生产sql；并且分析探查特定业务字段的枚举分布。

3. 代码生成： 开发者确认技术方案后， LLM生成 SQL。此时并非结束，而是进入半自动调试。

4. 自动校验：  LLM基于SQL代码规范来校验sql逻辑的准确性。

平衡效率与介入： 人工介入点被精准限制在“口径确认”和“异常终审”，而耗时的“查表、找字段、写 Join、纠错”由 AI 承担。

DACon：您提到在复杂场景下实现了 80% 的 SQL 生成准确率。在调优过程中，哪些类型的任务最容易出现错误？你们通过哪些具体策略提升了生成结果的确定性？

陈哲：易错任务的地方主要是业务逻辑的“隐性缺失”。最容易出错的任务通常不在于语法，而在于业务背景的断层。口径“潜规则”缺失，比如人群包需求看似简单（如“活跃用户”），但历史生产 SQL 中往往沉淀了大量隐性过滤（如剔除测试号、特定黑名单、退款剔除）。AI 若脱离历史代码纯靠 DDL 生成，逻辑必然缺失。

提升确定性的核心策略是，从“创造”转向“迭代”。历史代码约束 (Baseline-Based)， 坚持“不从零开始”。通过 MCP 协议拉取目标任务的历史生产 SQL，约束 AI 必须在现有逻辑框架内进行增量修改。将 AI 的角色从“翻译官”转变为“代码修补匠”，以此继承所有沉淀在代码里的隐性业务背景。

DACon：面对 1.1w+ 任务和 10w+ 字段 的超大规模拓扑，知识图谱的构建与更新如何保持实时性与一致性？在链路血缘过深时，你们的 “任务级关联剪枝” 策略是如何定义“高价值上下文”并实现动态路径压缩的？

陈哲：我们并没有追求图谱全量的实时更新，而是通过“双 MCP”来做信息补充。首先是离线全量图谱（T+1）， 基于 Hive 元数据表定时同步表、任务、字段及血缘至 Neo4j，基于图数据库的元信息构建了覆盖表、任务、血缘等的mcp工具。其次，提供实时表元 MCP，基于平台侧提供的表元数据的实时接口，封装mcp，获取表的最新的元信息。

面对深层血缘，我们不采取一次性拉取的“全量模式”，而是执行贪婪式探测。 LLM 首先基于需求表溯源最多 3 层血缘，获取该范围内的元信息与任务 SQL。 LLM 自行判断当前 3 层内的元信息是否足以闭环 PRD 逻辑。如果有疑问，则反馈人工确认环节，由开发确认是否需要继续血缘探查。

DACon：您展望了从 交付场景横向扩展到自动化治理与分析场景。基于当前的智能化底座，您认为下一步最具潜力的应用方向是什么？在推进更深度的自动化过程中，您认为最大的技术或组织挑战会是什么？

陈哲：我认为最具潜力的应用方向是“自助式根因分析与治理”。当仪表盘指标异动时，LLM 可以沿着知识图谱自动回溯，定位到是哪个底层任务延迟或哪次 SQL 变更导致的，并自动提出修复方案。

这其中最大的挑战主要分为两个点：

* 技术挑战：长链路推理的可靠性。随着自动化深度增加，AI 的每一个微小错误都会在链路中放大，如何构建比 SQL 本身更可靠的“监督模型”是技术关键。
* 组织挑战：研发范式的重塑。传统的“PRD -> 排期 -> 开发”流程将向“Prompt -> 实时反馈 -> 审核”转变。这对数据开发者的要求从“会写 SQL”转向了“会定义标准与逻辑审核”，这种角色转型的阵痛和组织信任的建立将是最大的阻力。
