## 一、Paimon是什么

Paimon是一种**湖表格式**，简单说就是在数据湖（S3、HDFS）上加了一层“货架和目录”，让数据湖里的文件有了“表”的样子。但它跟Iceberg、Hudi最大的不同：Paimon从设计第一天就为**实时更新**而生，出生在Flink社区，目标很明确——让数据湖既能存海量数据，又能像数据库一样实时响应变更。

用一句话概括：**Paimon = 数据湖的廉价存储 + 数据库的更新能力 + 消息队列的流式消费能力**。

今天，我们就从下往上，一层层拆开Paimon的整体架构，看看它内部到底是怎么运转的，为什么能成为实时数仓的“地基”。

### 2.2 文件存储层：数据最终落在哪

Paimon的数据最终存在对象存储（S3、OSS）或HDFS上，文件格式是Parquet或ORC。这层看起来简单，但有两个细节决定了Paimon的底层能力：

**第一，文件是不可变的。** 一旦写入，文件就不会再被修改。这意味着更新操作不是去改旧文件，而是写新文件。这听起来很浪费，但结合上一层的LSM Tree，就成了优势——写新文件是顺序的，比随机修改快得多。

**第二，文件是按“桶”组织的。** Paimon会把数据分成多个桶，每个桶对应一组文件。写入时根据主键哈希决定去哪个桶，查询时如果带上主键，就能直接定位到桶，不用扫全表。这有点像Hive的分桶，但Paimon的桶是动态的，可以随着数据量调整。

举个例子：订单表按order\_id分64个桶，每个桶对应一个物理目录。写入一条订单时，计算`hash(order_id) % 64`，就知道该去哪个桶。查询某个订单时，也只查一个桶，不用扫全表。

### 2.3 数据组织层：LSM Tree到底怎么跑

这是Paimon最核心的设计。LSM Tree（日志结构合并树）这套机制来自Google的BigTable，HBase、RocksDB都在用。但Paimon把它用在了数据湖上。

**LSM Tree在Paimon里长什么样？**

想象一个订单表的写入过程：

2. **第一步：写内存。** 数据进来先写内存里的MemTable。MemTable是一个有序结构，按主键排好。这一步是纯内存操作，所以极快——每秒几万条不在话下。
3. **第二步：刷盘。** MemTable满了（默认256MB），就冻结成一个不可变的文件，刷到磁盘上。这个文件叫SSTable（Sorted String Table）。SSTable也是按主键排好序的，内部还有索引，方便查询。

   关键点：刷盘是顺序写，不是随机写。因为MemTable已经排好序了，刷出去就是一个有序文件，磁盘IO非常友好。
4. **第三步：合并。** 随着时间推移，SSTable文件越来越多。同一个主键可能出现在多个文件里——比如订单状态从“待支付”变成“已支付”，旧版本在一个文件，新版本在另一个文件。查询时就要读多个文件，效率下降。

   这时候后台的Compaction线程开始工作：挑几个SSTable文件，把它们合并成一个新文件。合并时只保留每个主键的最新版本，旧版本丢掉。合并完生成新文件，删除旧文件。

   Compaction的时机可以配置，比如每小时跑一次，或者文件数量到阈值就跑。

**这个机制怎么支撑高频更新？**

假设一个订单状态变了5次。传统数据湖要改文件，得把整个文件读出来、改掉那条、再写回去，成本极高。Paimon呢？每次更新只是追加一个新版本到内存，然后刷成一个新文件。更新本身是顺序写，不涉及随机IO，所以再频繁也不怕。

查询时，Paimon会读所有相关文件，合并出最新版本。虽然读有代价，但通过Compaction可以控制文件数量，保证读性能。

**老张的总结：** LSM Tree就是用“写放大换读放大”，再通过后台Compaction把读放大收回来。ODS层扛得住高峰写入，DWD层稳得住状态变更，靠的就是这个。

### 2.4 元数据层：怎么知道有哪些文件

文件越来越多，总得有人记着“哪些文件属于这张表”。这就是元数据层的活。

Paimon的元数据存在一个独立的目录里，通常是`表路径/metadata`。里面有几个核心文件：

* **schema.json**：表结构，字段名、类型、主键。
* **snapshot-{id}.json**：快照文件。每次提交（commit）都会生成一个新快照，记录这个版本有哪些文件。
* **manifest-{id}.avro**：清单文件。快照里不直接列所有文件，而是指向一批清单文件，每个清单文件记录一组SSTable的路径和统计信息（比如最小值、最大值、行数）。

**这个设计解决了什么问题？**

有了快照，就有了时间旅行能力。想查昨天上午10点的数据？找到那个时刻的快照，读它指向的文件就行。

老张有一次DWD层的清洗逻辑写错了，把一批订单过滤掉了。以前遇到这种情况只能认栽，数据丢了就丢了。现在直接从出问题之前的快照重跑，几分钟就补回来了。

**快照怎么生成？**

每次写入（不管是批还是流）都会触发一次commit。commit时做几件事：

* 把这次写入的新文件列出来
* 更新元数据，生成新快照
* 把快照指针指向最新版本

这个过程是原子的，要么全成功，要么全失败，保证了ACID。

### 2.5 Changelog机制：数据怎么流动起来

这是Paimon区别于其他表格式最特别的地方。它让表不仅能存数据，还能像Kafka一样输出变更流。

**Changelog是怎么产生的？**

每个SSTable文件里，除了数据本身，还记录了每条数据的操作类型——是插入（INSERT）、更新（UPDATE\_BEFORE/UPDATE\_AFTER）、还是删除（DELETE）。

当Flink消费Paimon表时，可以选择两种模式：

* **读快照**：读某个版本的全量数据
* **读变更流**：从某个时间点开始，源源不断接收表的变更

第二种模式就是Changelog。Paimon会把新写入的数据，按操作类型组织成流，推给下游Flink任务。

**举个例子：**

订单表里一条订单状态从“待支付”变成“已支付”。Paimon的Changelog里会产生两条记录：

```
UPDATE_BEFORE: (order_id=1001, status='待支付')  
UPDATE_AFTER:  (order_id=1001, status='已支付')
```

下游的DWS层任务收到这个流，就知道要更新汇总值——把“待支付”的销售额减掉，把“已支付”的加上。

**这个机制解决了什么问题？**

以前实时数仓想分层，DWD层建起来容易，但DWS层怎么实时响应DWD的变化？只能靠批处理每天跑。现在DWD的变更能直接流到DWS，DWS就能实时更新。这才是真正的“流式分层”。

**Kafka vs Paimon的变更流：**

Kafka也能做类似的事——把变更数据写到Kafka topic里，下游消费。但Kafka的问题是：数据流过去就没了，想回溯得提前存好，而且Kafka里不能查历史快照。Paimon的变更流是在数据已经存好的基础上生成的，既保留了存储，又提供了流式消费。

### 2.6 查询引擎层：谁来读数据

最后，数据存好了，总得让人查。Paimon不自己做查询引擎，而是对接各种计算引擎：

* **Spark**：通过DataSource API读取Paimon表，可以做批处理、跑SQL
* **Flink**：通过Table Store Connector读取，既可以读快照也可以读变更流
* **Trino/Presto**：通过插件支持，跑即席查询
* **StarRocks**：可以建外表，也可以把数据导入内部表加速

最关键的是：所有引擎读的是同一份数据，不是拷贝。Spark读的是S3上的Parquet文件，Trino读的也是同一批文件，Flink读变更流时背后的数据也是这些文件。口径天然统一。

以前实时大屏和离线报表对不上账，是因为数据存了两份，口径各自一套。现在都从Paimon读，再也不用扯皮了。

## 三、为什么Paimon是实时数仓的“地基”

把上面的设计串起来，答案就自然浮现了：

* **文件存储层**给了低成本海量存储——ODS、DWD、DWS层想存多久存多久，分层才有物质基础。存一年都不怕。
* **LSM Tree**给了高频更新能力——ODS层扛得住每秒几十万条写入，DWD层稳得住订单状态一天变五次。写放大换读放大，后台Compaction兜底。
* **元数据层**给了可追溯性——快照机制让时间旅行成为现实，历史回溯、错误修复不再是难题。
* **Changelog机制**给了流式流动能力——DWD的变更能实时流到DWS，DWS才能实时更新，这才是真正的“实时”数仓。
* **查询引擎层**给了统一接口——实时和离线读同一份数据，口径一致，再也不用对账。

以前实时数仓分层建不起来，是因为没有这样的存储底座。Kafka是管道，存不久；Hive存得住，但更新不了；Iceberg/Hudi各有擅长，但没有一个像Paimon这样把三样能力凑齐了——**低成本存储 + 高效更新 + 流式消费**。

Paimon把这三点做齐了，才成了真正的地基。

## 写在最后

老张后来在团队里说：

> 以前实时数仓分层是不可能的事情，因为下面没有地基。有了湖表格式Paimon之后，就有了那个地基——它用LSM Tree稳住了高频更新，用Changelog让数据流动起来，用元数据让数据可追溯，用文件存储保障了低成本。

技术的演进，说到底就是找到更合适的抽象层次。Paimon在存储层找到了实时数仓真正的改变灵魂的位置。
