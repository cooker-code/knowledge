## 用户认知重点

| 项 | 内容 |
|---|---|
| 已知基础 | 用户知道血缘用于追踪上下游、元数据用于描述数据资产，无需重复定义 |
| 待补边界 | 采集、解析、图模型、服务层、治理应用之间的边界；与质量/调度的联动 |
| 易偏差点 | 血缘文章易只讲可视化，不讲解析准确性和应用闭环 |
| 优先抽取 | 可验证的治理场景、采集粒度、字段血缘取舍 |

## 排重指纹

```text
治理对象 + 采集/解析/建模/存储/服务/应用模块 + 核心机制 + 解决问题 + 准确性/成本边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都是血缘综述 | 没有实现链路和应用场景时只略读 |
| 都是 SQL 血缘 | 按表级、字段级、算子级、运行时血缘拆分 |
| 都是可视化 | 只有展示不新建，能支持影响分析才沉淀 |
| 有字段血缘争议 | 必须写入认知校准点 |

## 已覆盖问题（按三级节点）

| 节点 | 已覆盖 | 还缺 |
|---|---|---|
| 元数据平台 | 统一 Catalog、统一元数据服务、主动元数据、非表资产和元数据统计边界 | DataHub/OpenMetadata/Atlas/Gravitino 官方架构补证、最小实验 |
| 数据安全治理 | Ranger 权限治理、owner 语义、权限检索、审计和元数据联动 | Ranger 官方插件机制补证、权限回收实验 |
| 数据血缘 | 元数据与血缘实现分层 | SQL 解析准确性、字段血缘边界、算子级血缘 |

## 待补缺口

| 优先级 | 项 | 为什么补 |
|---|---|---|
| 高 | SQL 解析器对比 | 决定血缘准确性和方言覆盖 |
| 高 | 字段级血缘边界 | 成本高、争议大，需要准则 |
| 高 | 数据质量与血缘联动 | 治理闭环核心 |
| 高 | DataHub / OpenMetadata / Atlas / Gravitino 对标 | 决定元数据平台选型和二次开发边界 |
| 高 | 主动元数据治理闭环 | 把元数据从"展示"推进到可执行动作 |
| 中 | 血缘可视化 | 影响分析和溯源必须可用 |
