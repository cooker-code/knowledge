## 核心问题

SQL 审核工具的价值是把高风险写法提前暴露出来，但不能替代工程判断。Hive 优化最终必须能解释：原 SQL 触发了什么物理计划、为什么会产生倾斜/全局排序/全量扫描，以及改写后如何保持结果正确。

## 判断准则

| 优化对象 | 可吸收准则 | 必须验证 |
|---|---|---|
| `ORDER BY + LIMIT` | 全局排序容易单 Reducer 或重 Shuffle，可考虑分区内排序再合并候选 | 结果是否仍是全局 TopN，分区键是否安全 |
| 分区裁剪 | 谓词应能落到分区列或文件扫描前 | Explain 是否真的裁剪分区 |
| 数据倾斜 | 先确认热点 Key 和任务长尾，再选择打散、MapJoin 或分阶段聚合 | 任务指标、热点分布、结果一致性 |
| 自动审核 | 工具建议作为候选，不作为最终结论 | 需要人工确认语义等价和执行计划改善 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 工具说能提升多少倍就可信 | 性能数字必须有数据规模、基线、执行计划和正确性验证 |
| SQL 改写只看跑得快 | 改写必须保持全局语义、排序语义和去重口径 |
| 参数和规则越多越好 | 没有定位瓶颈的规则清单会干扰判断 |

## 架构/流程图

```mermaid
flowchart LR
  SQL["原 SQL"] --> Risk["风险识别\n排序 / Join / 分区 / 去重"]
  Risk --> Explain["Explain / 执行指标"]
  Explain --> Rewrite["候选改写"]
  Rewrite --> Verify["结果对账 / 性能对照"]
  Verify --> Rule["沉淀为 SQL 准则"]
```

## 待验证缺口

- 需要补 Hive Explain 中常见慢计划信号和对应处理动作。
- 需要补自动 SQL 审核建议的误判案例，尤其是语义不等价改写。
