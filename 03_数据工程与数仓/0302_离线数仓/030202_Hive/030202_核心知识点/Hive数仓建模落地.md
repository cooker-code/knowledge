# Hive 数仓建模落地

## 一句话结论

Hive 的数仓价值主要体现在把业务过程稳定落成 ODS、DWD、DWS、ADS 等分层模型；表设计的重点不是“能查”，而是粒度清晰、历史可追、复用稳定、成本可控。

## 快速抓重点

| 主题 | 关键判断 |
|---|---|
| ODS | 是否保留源系统原貌，是否支持回溯 |
| DWD | 是否统一业务过程、口径、粒度 |
| DWS | 是否围绕主题和公共指标沉淀复用层 |
| ADS | 是否面向具体应用和报表交付 |
| 维度表 | 是否稳定描述实体属性 |
| 事实表 | 是否明确业务过程、粒度、度量 |
| 拉链表 | 是否需要记录状态历史变化 |
| 快照事实表 | 是否需要按周期保留状态快照 |

## 纵向定位

```mermaid
flowchart LR
  Source["源系统"] --> ODS["ODS\n贴源层"]
  ODS --> DWD["DWD\n明细事实层"]
  DWD --> DWS["DWS\n主题汇总层"]
  DWS --> ADS["ADS\n应用层"]
  ADS --> App["报表 / 指标 / 数据产品"]
```

## Hive 落地时的重点

- 分区通常按时间或业务周期设计，但不能只为了查询方便牺牲写入和维护成本。
- 拉链表适合记录实体状态变化，不适合所有表都强行拉链。
- 快照事实表适合周期性状态分析，但要注意存储膨胀。
- 宽表可以提升查询便利性，但可能破坏粒度、复用性和变更稳定性。
- 字段新增、口径变更、历史回刷必须设计流程，不应只看当前 SQL 能否跑通。

## 常见风险

- 粒度不清导致指标口径混乱。
- DWS 宽表越做越大，新增维度导致返工。
- ADS 反向污染公共层。
- 分区、文件和任务链路缺少治理，后期维护成本急剧上升。
