## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 口径表达 | 把业务问题转成明确的筛选、分组、窗口和统计逻辑 | 粒度是否一致，是否存在重复计数和口径漂移 |
| Join 关系 | 处理维表关联、多条件关联、不等值关联和长尾问题 | Join 条件是否改变语义，是否引入笛卡尔积或倾斜 |
| 窗口与序列 | 处理排名、累计、留存、路径、日期交叉和缺失日期补齐 | 窗口分区和排序是否稳定，是否能用样例验证 |
| 聚合与去重 | 处理 Count Distinct、Grouping Sets、Cube、去重聚合 | 是否触发数据膨胀，是否可改写为分阶段聚合 |
| 行列转换 | 处理 lateral view、explode、数组、Map、行转列 | 展开后粒度是否变化，是否影响下游 Join |
| 性能前置 | 在写 SQL 时规避全表扫、全局排序、倾斜和小文件放大 | 优化动作是否有执行计划或指标证据 |
| 验证与排障 | 用 Explain、任务指标、抽样对账和边界样例证明 SQL 正确 | 不能只凭跑通判断正确 |

## 横向对标

| 对标对象 | 对标点 | SQL书写关注 | 不应混淆 |
|---|---|---|---|
| Hive / Spark | 执行引擎 | SQL 写法如何触发不同物理计划 | 不把引擎参数清单当 SQL 准则 |
| 数仓建模 | 表结构和粒度 | SQL 是否遵守模型粒度和口径契约 | 不用复杂 SQL 掩盖模型设计错误 |
| 语义层与智能问数 | 自然语言到查询 | SQL 是可审计、可验证的最终表达 | 不把 Text-to-SQL 文章归到 SQL 写法 |
| OLAP 查询优化 | 低延迟查询服务 | 离线加工 SQL 的正确性和资源成本 | 不把索引、物化视图、并发服务归入本模块 |

## 文章处理 SOP

1. 先确定业务口径和表粒度；只有题解没有口径边界的文章降权。
2. 构造问题指纹：`SQL + 表关系/表达模式/执行机制 + 解决问题 + 适用边界 + 认知增量`。
3. 先提取认知校准点：是否改变 Join 语义、是否引入重复计数、是否缺少样例对账。
4. 能沉淀成稳定表达模式、反例、执行风险或验证模板时，写入 `030204_核心知识点/`。
5. 面试题只作为最小样例池；不能抽象成工程准则时只重命名。
6. 有知识贡献的文章改为 `done-` 前缀，知识点来源链接指向 `../文章/done-原文件名.md`；无贡献文章直接删除。
7. 不生成日期化中间产物；批量判断最终沉淀到本文件、知识地图或具体知识点。

## 排重判断

| 判断项 | 排重规则 |
|---|---|
| 都是 Join | 按语义正确性、OR/不等值、多对多放大、热点长尾拆分 |
| 都是窗口函数 | 按排名、累计、区间、缺失补齐、路径序列拆分 |
| 都是调优技巧 | 没有 Explain/指标/数据分布证据时降权 |
| 都是面试题 | 只保留能形成边界样例或验证模板的题型 |
| 都是 Hive/Spark 参数 | 若主问题是执行引擎参数，转 Hive/Spark；若主问题是 SQL 写法，留本目录 |
