## 技术定位

| 项 | 内容 |
|---|---|
| 技术名 | SQL书写 |
| 一级类目 | 数据工程与数仓 |
| 二级类目 | 离线数仓 |
| 技术本体 | 面向离线数仓加工任务的 SQL 表达、口径落地、复杂逻辑拆解和慢 SQL 前置规避方法 |
| 全局架构位置 | 位于数仓建模之后、执行引擎之前，把业务口径、表粒度、Join 关系和聚合逻辑转成可执行 SQL |
| 主要使用者 | 数据开发、数仓工程师、分析工程师 |
| 主要产出 | Hive SQL、Spark SQL、口径 SQL、窗口函数、Join 写法、去重聚合、异常排查线索 |

## 一句话结论

SQL书写不是函数速查表，而是把业务口径、模型粒度和执行引擎约束同时放进 SQL 里的工程能力；只讲语法不讲口径、数据量、执行计划和校验方式的文章应降权。

## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 口径表达 | 把业务问题转成明确的筛选、分组、窗口和统计逻辑 | 粒度是否一致，是否存在重复计数和口径漂移 |
| Join 写法 | 处理维表关联、多条件关联、不等值关联和长尾问题 | Join 条件是否改变语义，是否引入笛卡尔积或倾斜 |
| 窗口与滚动 | 处理排名、累计、留存、路径、日期交叉等序列问题 | 窗口分区和排序是否稳定，是否能用样例验证 |
| 聚合与去重 | 处理 Count Distinct、Grouping Sets、Cube、去重聚合 | 是否触发数据膨胀，是否可改写为分阶段聚合 |
| 行列转换 | 处理 lateral view、explode、数组、Map、行转列 | 展开后粒度是否变化，是否影响下游 Join |
| 性能前置 | 在写 SQL 时规避全表扫、全局排序、倾斜和小文件放大 | 优化动作是否有执行计划或指标证据 |
| 验证与排障 | 用 Explain、任务指标、抽样对账和边界样例证明 SQL 正确 | 不能只凭跑通判断正确 |

## 横向对标

| 对标对象 | 对标点 | SQL书写关注 | 不应混淆 |
|---|---|---|---|
| Hive / Spark | 执行引擎 | SQL 写法如何触发不同物理计划 | 不把引擎参数清单当 SQL 准则 |
| 数仓建模 | 表结构和粒度 | SQL 是否遵守模型粒度和口径契约 | 不用复杂 SQL 掩盖模型设计错误 |
| 语义层与智能问数 | 自然语言到查询 | SQL 是可审计、可验证的最终表达 | 不把 Text-to-SQL 文章归到 SQL 写法 |
| OLAP 查询优化 | 低延迟查询服务 | 离线加工 SQL 的正确性和资源成本 | 不把索引、物化视图、并发服务归入本模块 |
