## 核心问题

SQL 调优不应该等任务跑慢后才开始。写 SQL 时就要识别会触发全表扫、宽 Shuffle、全局排序、热点长尾、小文件放大和无效列扫描的形状，并提前设计可验证的执行边界。

## 判断准则

| 风险 | 写 SQL 时的前置动作 | 验证方式 |
|---|---|---|
| 全表扫描 | 分区过滤、列裁剪、谓词前置 | Explain 扫描范围、输入数据量 |
| Map 端长尾 | 文件大小、Split、压缩是否可切分 | Map 任务耗时分布、输入文件分布 |
| Reduce 端长尾 | Group/Join Key 热点、distinct 去重 | Reduce 输入行数、热点 Key 抽样 |
| 全局排序 | 是否需要全局 TopN，能否分阶段候选 | Sort/Exchange 节点和结果对账 |
| 小文件放大 | 动态分区数、写入 task 数、合并策略 | 输出文件数和下游读取耗时 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| SQL 慢了才调优 | 很多风险在 SQL 形状阶段就能识别 |
| 多加资源就能解决长尾 | 数据分布不变，资源只能缓解平均耗时 |
| 技巧清单可以直接套 | 每条技巧都要绑定适用条件和验证指标 |
