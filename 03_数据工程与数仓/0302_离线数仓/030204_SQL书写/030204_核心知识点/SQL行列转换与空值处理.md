# SQL行列转换与空值处理

## 来源

## 核心问题

半结构化字段、行列转换、空值、除零和稀疏表补齐类 SQL，难点不在函数记忆，而在粒度是否变化、排序是否稳定、异常值是否可解释。写法一旦改变行数、列含义或 Null 语义，就必须补样例验证。

## 判断准则

| 场景 | 判断准则 |
|---|---|
| 除零与异常值 | 先定义业务上分母为 0 的含义，再用 `nullif`、`case when` 或默认值处理；不能为了跑通直接填 0。 |
| Null 处理 | 区分未知、无值、缺失和业务默认值；`coalesce` 只解决展示或补齐，不等于数据真实存在。 |
| 稀疏表补齐 | 必须先定义排序键和“最近一条有效记录”的方向；常见做法是用窗口累计分组后在组内取 `max`。 |
| 分列 / JSON 提取 | 提取后要验证字段是否存在、类型是否稳定、数组下标是否越界；半结构化字段不能只按样例解析。 |
| 行列转换 | 转换前后要对账行数、实体粒度和字段顺序；`collect_list` 等聚合结果若依赖顺序，必须显式排序。 |
| 数值扩充 | 生成完整序列后再左关联事实表；补齐值要标识来源，避免下游把补齐行当真实发生。 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 空值都可以直接填默认值 | 默认值会改变业务含义，必须先确认缺失原因和下游消费方式。 |
| JSON 能取出来就算完成 | 半结构化字段要验证缺字段、类型漂移、嵌套层级变化和解析失败。 |
| 行列转换只是语法问题 | 行列转换会改变粒度、顺序和重复计数风险，必须做前后对账。 |
| 稀疏补齐就是向前填充 | 向前、向后、最近邻补齐分别对应不同业务语义，不能混用。 |

## 待验证缺口

- Hive/Spark 在 `collect_list` 顺序、JSON 函数和 Null 排序上的差异。
- 稀疏表补齐在多实体、多字段同时补齐时的性能和正确性边界。
- 补齐行、真实行、默认行在下游指标里的可审计标记方式。
