## 技术定位

| 项 | 内容 |
|---|---|
| 技术名 | 数仓建模 |
| 一级类目 | 数据工程与数仓 |
| 二级类目 | 离线数仓 |
| 技术本体 | 把业务过程、口径、实体、事件和分析需求组织成可复用、可治理、可演进的数据仓库模型 |
| 全局架构位置 | 位于数据接入之后、SQL 加工和下游指标/报表/服务之前，是离线数仓的结构设计层 |
| 主要使用者 | 数仓工程师、数据架构师、数据开发、分析工程师 |
| 主要产出 | 主题域、ODS/DWD/DWS/ADS 分层、事实表、维度表、宽表、拉链表、命名规范、模型评估准则 |

## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 需求边界 | 明确消费者、业务过程、口径和不做什么 | 防止按报表倒推孤立宽表 |
| 主题域与总线 | 组织业务对象、公共维度和跨系统协作边界 | 主题域服务复用和治理，不是目录美化 |
| 分层与粒度 | 明确 ODS/DWD/DWS/ADS 的职责、粒度和消费者 | 粒度先于字段，层次先于宽表 |
| 事实与维度 | 处理事务事实、周期快照、累积快照、维度变化 | 先定业务过程和粒度，再定事实和维度 |
| 宽表与服务层 | 为下游查询、指标和服务提供便利出口 | 宽表不能替代上游稳定分层 |
| 规范与物理落地 | 命名、字段、分区、生命周期、物理模型 | 规范要可执行，不是格式文档 |
| 模型评估与演进 | 面对新增维度、口径变更、下游影响时减少返工 | 需要复用性、变更成本、兼容和回填证据 |
| 平台化与 Data+AI | 建模平台、元数据、语义层、Agent 访问 | AI 只能增强建模流程，不能替代业务边界判断 |

## 横向对标

| 对标对象 | 对标点 | 数仓建模关注 | 不应混淆 |
|---|---|---|---|
| SQL书写 | SQL 如何表达模型 | 建模先决定粒度和口径，SQL 负责落地 | 不用复杂 SQL 弥补错误模型 |
| Hive / Spark | 表承载和计算执行 | 建模定义表结构和层次，Hive/Spark 负责存储和计算 | 不把引擎调优文章归为建模 |
| 指标体系 | 指标定义和消费 | 建模为指标提供稳定明细和汇总基础 | 指标治理应进入数据分析与 BI |
| 数据质量与治理 | 规则、血缘、权限、准入准出 | 建模要预留治理接口 | 质量闭环不归建模单独解决 |
| OLAP 引擎 | 查询服务化 | 建模沉淀可复用数据资产 | 高并发低延迟查询优化归 OLAP |

## 文章处理 SOP

1. 先判断文章在回答哪个建模节点的问题：需求、主题域、分层、粒度、事实维表、宽表、规范、演进。
2. 构造问题指纹：`建模对象 + 层次/模型类型 + 核心机制 + 解决问题 + 演进边界 + 认知增量`。
3. 优先提取认知校准点：宽表是否被神化、分层是否模板化、AI/平台是否被夸大、是否缺少业务边界。
4. 有模型准则、反例、返工风险、评估指标或可迁移流程时，写入 `030206_核心知识点/`。
5. 纯概念清单、面试题和营销平台文章降权；只保留能沉淀为流程节点的内容。
6. 有知识贡献的文章改为 `done-` 前缀，知识点来源链接指向 `../文章/done-原文件名.md`；无贡献文章直接删除。
7. 不生成日期化中间产物；批量判断最终沉淀到本文件、知识地图或具体知识点。

## 排重判断

| 判断项 | 排重规则 |
|---|---|
| 都是分层 | 只有新增层职责、消费者、粒度、变更责任时才追加 |
| 都是事实表 | 按事务事实、周期快照、累积快照、无事实事实拆分 |
| 都是宽表 | 按消费便利、粒度漂移、字段膨胀、动态字段、下游影响拆分 |
| 都是规范 | 只有能变成可执行约束、检查项或命名模板时才沉淀 |
| 都是平台/AI | 只吸收平台化流程、元数据、语义层和治理接口，不采信效率营销 |
