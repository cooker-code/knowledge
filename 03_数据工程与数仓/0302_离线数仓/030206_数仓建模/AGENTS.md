## 技术定位

| 项 | 内容 |
|---|---|
| 技术名 | 数仓建模 |
| 一级类目 | 数据工程与数仓 |
| 二级类目 | 离线数仓 |
| 技术本体 | 把业务过程、口径、实体、事件和分析需求组织成可复用、可治理、可演进的数据仓库模型 |
| 全局架构位置 | 位于数据接入之后、SQL 加工和下游指标/报表之前，是离线数仓的结构设计层 |
| 主要使用者 | 数仓工程师、数据架构师、数据开发、分析工程师 |
| 主要产出 | 主题域、ODS/DWD/DWS/ADS 分层、事实表、维度表、宽表、拉链表、命名规范、模型评估准则 |

## 一句话结论

数仓建模不是分层名词和事实表定义的集合，而是持续回答“业务过程如何沉淀、表粒度如何稳定、口径如何复用、下游变化如何少返工”的工程设计活动。

## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 业务过程识别 | 从业务事件和实体变化中抽取建模对象 | 不能按现有报表倒推孤立宽表 |
| 主题域划分 | 组织跨业务线、跨系统的数据边界 | 主题域应服务复用和治理，不是目录美化 |
| 分层设计 | 明确 ODS/DWD/DWS/ADS 的职责和交付边界 | 每层要回答输入、产出、粒度、口径和消费者 |
| 事实表与维表 | 处理过程事实、事务事实、周期快照、累积快照和维度变化 | 先定粒度，再定事实和维度 |
| 宽表与服务层 | 为下游查询、指标和服务提供便利出口 | 宽表不能替代上游稳定分层 |
| 规范与评估 | 命名、字段、分区、生命周期、复用性和质量 | 模型好坏要有评估维度和变更成本证据 |
| 模型演进 | 面对新增维度、粒度变化和口径变更时减少返工 | 需要版本、兼容、回填和下游影响评估 |

## 横向对标

| 对标对象 | 对标点 | 数仓建模关注 | 不应混淆 |
|---|---|---|---|
| SQL书写 | SQL 如何表达模型 | 建模先决定粒度和口径，SQL 负责落地 | 不用复杂 SQL 弥补错误模型 |
| Hive / Spark | 表承载和计算执行 | 建模定义表结构和层次，Hive/Spark 负责存储和计算 | 不把引擎调优文章归为建模 |
| 指标体系 | 指标定义和消费 | 建模为指标提供稳定明细和汇总基础 | 指标治理应进入数据分析与 BI |
| 数据质量与治理 | 规则、血缘、权限、准入准出 | 建模要预留治理接口 | 质量闭环不归建模单独解决 |
| OLAP 引擎 | 查询服务化 | 建模沉淀可复用数据资产 | 高并发低延迟查询优化归 OLAP |
