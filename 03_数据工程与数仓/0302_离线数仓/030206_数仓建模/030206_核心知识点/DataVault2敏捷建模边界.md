# Data Vault 2.0 敏捷建模边界

## 来源

- [Data Vault 2.0建模实战：构建企业级敏捷数据仓库的核心方法论](<../文章/done-Data Vault 2.0建模实战：构建企业级敏捷数据仓库的核心方法论.md>)

## 核心问题

Data Vault 2.0 用 Hub、Link、Satellite 分离业务键、关系和属性变化，强调审计追踪、敏捷扩展和企业级集成。它适合复杂源系统和频繁变化环境，但不应被当作所有数仓模型的默认替代。

## 判断准则

| 组件 | 作用 | 边界 |
|---|---|---|
| Hub | 稳定业务键 | 不承载大量描述属性 |
| Link | 业务关系 | 关系设计复杂度会提高查询成本 |
| Satellite | 属性、历史变化、来源审计 | 需要严格的时间和来源治理 |
| Raw Vault | 原始可追溯层 | 不直接服务分析消费 |
| Business Vault | 业务规则加工层 | 仍需面向消费建模 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| Data Vault 比维度建模更先进，所以应全面替代 | 它解决的是敏捷集成和审计追踪问题，消费层仍可能需要维度模型 |
| Hub/Link/Satellite 只是换命名 | 它改变了变化隔离和历史追踪方式 |
| 复杂企业都适合 Data Vault | 团队能力、工具链和治理成熟度不足时会增加复杂度 |

## 待验证缺口

- 需要补 Data Vault 到维度模型/ADS 消费层的转换模式。
