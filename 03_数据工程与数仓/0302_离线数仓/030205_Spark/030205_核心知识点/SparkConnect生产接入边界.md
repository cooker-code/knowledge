# Spark Connect 生产接入边界

## 来源

- [基于Spark Connect的生产实践](<../文章/done-基于Spark Connect的生产实践.md>)

## 核心问题

Spark Connect 的核心是客户端与 Spark Driver/服务端解耦，改善远程接入、轻量客户端和多语言访问体验。它不是调度系统，也不自动解决 Spark 作业性能、权限和资源治理。

## 判断准则

| 判断项 | 准则 |
|---|---|
| 适合场景 | 远程客户端、轻量依赖、Notebook/服务调用、跨语言接入 |
| 不适合替代 | 任务调度、资源队列、SQL 审计、结果对账、数据权限 |
| 生产关注 | 会话生命周期、连接稳定性、服务端资源、兼容 API、错误追踪 |
| 验证方式 | 对比传统 Spark 应用的功能覆盖、性能、失败恢复和运维成本 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| Spark Connect 是新的执行引擎 | 它是接入协议和客户端/服务端形态，执行仍归 Spark |
| Connect 能替代 Kyuubi | Kyuubi 是多租户 SQL 网关，Connect 是 Spark 接入形态，治理层次不同 |
| 轻客户端就是低成本 | 服务端会话、资源和兼容性仍要治理 |

## 待验证缺口

- 需要补 Spark Connect 与 Kyuubi、Spark Thrift Server 的边界对照。
- 需要补生产接入时的鉴权、会话回收和错误追踪模式。
