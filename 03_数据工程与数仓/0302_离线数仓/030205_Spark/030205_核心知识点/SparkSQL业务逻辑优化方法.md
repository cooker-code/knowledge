## 核心问题

Spark SQL 业务逻辑优化不是通用 SQL 技巧复述，而是要判断某种写法会如何影响 Catalyst 规则、物理计划、Stage 并发、笛卡尔积、谓词下推和版本行为。

## 判断准则

| 问题 | 判断准则 |
|---|---|
| 笛卡尔积 | 先看 Join 条件是否缺失、非等值、OR 条件或隐式 cross join |
| 写法差异 | 同一逻辑不同写法要比较逻辑计划、物理计划和结果一致性 |
| Stage 并发 | 并发执行受依赖关系、资源和调度模式约束，不是 SQL 拆分越多越并发 |
| 周期过滤 | 时间过滤要闭合、可分区裁剪、可对账 |
| UDF/表达式 | 注意规则优化是否内联、合并或改变求值顺序 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| Spark SQL 和 Hive SQL 写法经验可以直接互用 | Spark 有 Catalyst、AQE、Stage 和版本规则差异 |
| 笛卡尔积只是性能问题 | 也可能是结果语义错误和重复计数问题 |
| 并行提交多个 Job 就一定快 | 并发会受资源、依赖和 Driver/调度开销限制 |

## 待验证缺口

- 需要补 Spark 3.2/3.3/4.x 对同一 SQL 写法的计划差异样例。
- 需要补笛卡尔积和周期过滤问题的自动检查模板。
