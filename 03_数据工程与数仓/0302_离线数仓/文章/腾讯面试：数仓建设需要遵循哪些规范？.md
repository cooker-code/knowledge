**数仓建设规范**

**第一章：数据仓库分层规范**

数据仓库的分层架构并非技术上的强制要求，而是一种经过长期实践验证的最佳工程方法论。它将复杂的数据加工流程分解为多个逻辑清晰、职责单一的层次，旨在提升数据处理的效率、复用性和稳定性。

**1.1****分层架构的核心价值**

在深入了解每一层的具体职责之前，理解为何要进行分层至关重要。其核心价值主要体现在以下几个方面：

* **清晰的数据结构**：通过分层，数据从原始状态到最终应用的整个生命周期被清晰地划分开来。每一层都有明确的输入和输出，使得整个数据体系的脉络一目了然，极大地便利了数据溯源（血缘追踪）和问题定位 。
* **数据复用与效率提升**：分层架构通过在中间层（如明细层和汇总层）沉淀公共的、可复用的数据模型，避免了不同业务需求的重复计算和开发工作。例如，多个上层应用都可以从同一个经过清洗的明细事实表或公共的汇总宽表获取数据，显著提升了开发效率和计算性能 。
* **隔离源系统变化**：数据仓库的底层（ODS层）直接面向多变的业务系统。分层设计可以将源系统的变化影响隔离在底层，当源系统表结构或数据内容发生变动时，只需调整底层的数据接入和处理逻辑，而不会直接冲击到上层的业务报表和分析应用，保证了整个数据仓库的稳定性。
* **简化任务与协同开发**：复杂的业务需求被分解到不同的层次进行处理，每个层次的任务相对单一和简化。这使得不同角色的开发人员（如数据接入、数据建模、数据应用）可以并行工作，专注于各自的层次，降低了项目的复杂度和管理成本。

**1.2****经典分层模型详解**

业界通用的数据仓库分层模型虽然在命名和细节上可能存在差异，但其核心思想高度一致。一个经典且广泛采用的模型通常包含以下几个层次：

**1.2.1 ODS (Operational Data Store) -****操作数据层**

* **核心职责**：作为数据仓库的“入口”和“缓冲区”，ODS层的首要职责是接收并存储来自各个异构业务系统的原始数据。此层的数据在结构和内容上与源系统保持高度一致，仅进行最基础的加载（如数据格式的统一），而不进行或只进行极少量的清洗和转换。其核心目标是“原汁原味”地保留历史快照，为后续的数据处理提供最原始、最全面的数据基础。
* **交互规范**：ODS层是数据仓库与外部业务系统的唯一接口。它通过ETL（抽取、转换、加载）或CDC（变更数据捕获）等方式，周期性地或实时地从生产数据库、业务日志、第三方API等数据源捕获数据。ODS层的数据是后续所有数据加工的起点，其数据的完整性和及时性直接决定了整个数据仓库的质量上限。

**1.2.2 DWD (Data Warehouse Detail) -****明细数据层**

* **核心职责**：DWD层是数据仓库建设的核心，承担着数据清洗、规范化和建模的重任。此层会对ODS层的原始数据进行深度加工，包括：

+ **数据清洗**：处理空值、去除重复记录、纠正异常值或“脏数据”。
+ **数据转换与规范化**：统一不同源系统的编码（如将多个业务线的“男性”、“男”、“1”统一为“M”）、统一数据类型和长度、统一命名规范。
+ **构建维度模型**：基于Kimball的维度建模理论，将数据组织成事实表（Fact Table）和维度表（Dimension Table）。事实表通常包含业务过程的度量值，保持与业务行为一致的最低粒度；维度表则包含分析业务的上下文环境（如时间、地点、用户等）。

* **交互规范**：DWD层的数据来源于ODS层。它消费ODS层的原始数据，产出干净、规范、易于理解的明细事实表和维度表。DWD层的数据质量和模型设计的合理性，是整个数据仓库能否提供一致、准确分析结果的关键。它是数据仓库中数据复用价值最高的层次之一。

**1.2.3 DWS (Data Warehouse Service / Summary) -****汇总数据层**

* **核心职责**：DWS层也常被称为数据服务层或公共汇总层。它的主要职责是基于DWD层的明细数据，面向特定的业务主题（如交易域、流量域、用户域）进行轻度或重度的聚合操作，构建公共的汇总指标和宽表模型。建立DWS层的目的是为了“预计算”，通过空间换时间的方式，提前将复杂的、常用的计算逻辑固化下来，从而提升上层应用的查询性能和响应速度。
* **交互规范**：DWS层的数据来源于DWD层的明细事实表和维度表。它通过关联多个事实表和维度表，并进行聚合运算（SUM, COUNT, AVG等），形成面向特定分析场景的主题宽表。这些宽表整合了某个主题下的大量指标和维度，使得业务分析师可以方便地进行多维分析，而无需每次都从最底层的明细数据开始计算。

**1.2.4 ADS (Application Data Store) -****应用数据层**

* **核心职责**：ADS层是数据仓库的“出口”，直接面向最终的业务应用和用户。此层的数据完全根据具体的数据产品需求（如BI报表、管理驾驶舱、用户画像标签、推荐系统接口等）进行组织。数据通常是高度聚合和定制化的，可能以指标、标签、或者特定格式的结果集等形式存在。为了满足前端应用的高并发和低延迟查询要求，ADS层的数据有时会存储在专门的查询引擎或键值存储系统中。
* **交互规范**：ADS层的数据主要来源于DWS层的公共汇总数据，有时也会直接来源于DWD层的明细数据（用于需要下钻分析的场景）。它是数据仓库对外的最终交付物，其产出直接决定了数据产品的用户体验和业务价值。

**1.3****数据在各层间的流转与交互原则**

数据在上述分层架构中遵循着清晰的单向流转原则：ODS -> DWD -> DWS -> ADS。这个过程中，数据呈现出以下规律性变化：

* **数据粒度**：从ODS到ADS，数据的粒度由细到粗。ODS是原始交易级粒度，DWD保持明细粒度，DWS是主题汇总粒度，ADS则是高度聚合的应用粒度。
* **数据内容**：从ODS到ADS，数据由原始、可能脏乱的状态，逐渐变得干净、规范、聚合，业务含义也越来越明确。
* **数据流向**：严格禁止跨层调用，例如ADS层应优先从DWS层取数，DWS层从DWD层取数。不允许出现从DWS层逆向依赖DWD层之上的某个自定义数据，或者ADS层直接大量依赖ODS层的情况。这种规范保证了数据依赖关系的清晰和可维护性。

**1.4****分层架构的演进思考：从集中式到****Data Mesh**

值得注意的是，传统的集中式分层架构虽然成熟稳定，但在应对超大规模企业和快速变化的业务需求时，也暴露出一些瓶颈，如中央数据团队成为瓶颈、响应速度慢等。近年来，Data Mesh（数据网格）架构思想应运而生。

Data Mesh提倡去中心化的数据所有权和架构，将数据视为一种“产品”。在Data Mesh理念下，数据的所有权和开发职责被下放到具体的业务领域（Domain）。每个领域负责端到端地生产、维护和提供高质量的“数据产品”。这种模式可能不会严格遵循上述ODS->DWD->DWS的固定分层模式 ，而是更强调数据产品的服务化和互操作性。它将传统的数据仓库或数据湖转变为一个能够提供即时计算能力的服务性数据融合层 。这对传统的分层规范提出了新的挑战和思考，未来的数据架构可能会是集中式分层与去中心化数据产品理念的混合体。

**第二章：数据仓库命名规范**

如果说分层规范是数据仓库的“骨架”，那么命名规范就是数据仓库的“DNA”。一套优秀的命名规范能够让数据“不言自明”，极大地降低沟通成本和理解成本，是数据治理和数据资产化的基础。

**2.1****命名规范的基石：原则与目标**

所有具体的命名规则都服务于以下几个核心原则：

* **见名知意（****Clarity****）** ：任何一个表名或字段名都应能清晰、准确地反映其所承载的业务含义或数据内容 。避免使用无意义的、模棱两可的或过于晦涩的缩写。
* **一致性（****Consistency****）** ：在整个数据仓库体系内，相同的业务概念必须使用相同的命名。例如，“用户ID”在任何表中都应命名为 user\_id，其数据类型和长度也应保持一致 。
* **可维护性（****Maintainability****）** ：结构化的、可预测的命名方式使得开发人员可以快速理解和定位数据，便于后续的迭代开发、问题排查和系统迁移。
* **唯一性与无歧义（****Uniqueness & Unambiguity****）** ：同一个名称在同一个数据库或模式下只能代表一个实体。同一个字段名在不同的表中也应该代表相同的业务含义 。

**2.2****通用命名规则**

这些规则是构建具体对象名称的基础，适用于表、字段、视图等所有数据对象。

* **语言与字符集**：

+ **语言**：必须使用**英文单词**或业界公认的英文缩写进行命名 。严禁使用中文拼音或拼音与英文混合的方式 因为拼音存在多音字和歧义，不利于标准化和国际化协作。
+ **大小写**：推荐全部使用**小写字母** 。这可以避免因不同操作系统或数据库系统对大小写敏感性策略不同而引发的潜在问题。
+ **字符**：名称只能由小写字母、数字和下划线组成。禁止使用空格  和其他特殊字符 。

* **单词分隔符**：

+ 多个单词之间必须使用 **下划线（****\_****）** 进行分隔 。例如，user\_login\_time。

* **长度限制**：

+ 对象名称应在保证清晰表达业务含义的前提下，尽可能简洁。不同的数据存储系统对名称长度有不同的物理限制，例如30、64或128个字符 。为保证通用性和可读性，通常建议**表名总长度不超过****64****个字符，字段名不超过****30****或****32****个字符** 。实际开发中，更严格的内部规范可能会限制在20或30个字符以内 。

* **保留字与禁忌**：

+ 严禁使用所用数据处理框架或SQL语言的**保留字**（如 select, table, group, order 等）作为名称 。
+ 名称**禁止以数字开头** 。
+ 避免使用过于宽泛的、无法体现业务含义的词汇，如 data, info, temp 等。

**2.3****核心对象命名详解**

在通用规则的基础上，针对不同类型的数据库对象，需要设计更具结构化的命名范式。

**2.3.1****表命名规范**

表的命名是数据仓库中最重要的一环，一个结构化的表名应该能揭示其所在的层级、所属的业务域、数据主题、内容和更新周期等信息。推荐的表命名结构如下：

**[****层级****]\_[****业务域****]\_[****主题****]\_[****内容描述****]\_[****可选：更新周期****]**

* **层级（****Layer****）** ：使用标准前缀标识该表所属的数据仓库分层。

+ ods\_：操作数据层
+ dwd\_：明细数据层
+ dws\_：汇总数据层
+ ads\_：应用数据层
+ dim\_：维度表（专用前缀，有时归于DWD/DWS管理）
+ tmp\_：临时表

* **业务域（****Business Domain****）** ：表示该表所属的核心业务板块，如 trade (交易), user (用户), item (商品), log (日志)等。业务域的划分是进行数据治理和实现Data Mesh理念的基础。
* **主题（****Subject****）** ：在业务域下对数据进行进一步的细分，描述一个具体的数据主题。例如，在 trade 业务域下，可以有 order (订单), payment (支付), refund (退款) 等主题。
* **内容描述（****Content****）** ：用一个或多个单词描述表的核心内容。对于事实表，通常描述其业务过程或事件，如 dwd\_trade\_order\_detail\_di (交易域订单明细事实表)。对于维度表，则直接描述维度本身，如 dim\_user\_info\_df (用户信息维度表)。
* **更新周期（****Partition/Update Cycle****）** ：对于增量分区表，需要在表名末尾添加后缀来表明其分区的时间粒度。这对于调度和理解数据时效性至关重要。

+ \_di：每日增量 (Daily Increment)
+ \_df：每日全量快照 (Daily Full)
+ \_wi / \_wf：每周增量/全量
+ \_mi / \_mf：每月增量/全量
+ \_hi：每小时增量
+ \_td：截至当日的全量快照 (Total Daily)

**示例**：

* ods\_trade\_order\_info\_di：ODS层的交易业务库订单信息表，每日增量。
* dwd\_user\_login\_log\_di：DWD层的用户域登录行为明细事实表，每日增量。
* dws\_trade\_seller\_stats\_1d：DWS层的交易域卖家统计主题宽表，统计粒度为最近1天。
* dim\_item\_info\_df：公共维度商品信息表，每日全量快照。
* ads\_user\_profile\_tags\_df：ADS层的用户画像标签表，每日更新。

**2.3.2****字段命名规范**

字段命名追求简洁与规范的平衡。

* **主键与外键**：主键字段通常以 id 结尾，如 user\_id, order\_id。外键字段应与它所引用的主键同名，以保证关联关系清晰。
* **通用指标后缀**：对于常见的指标类型，使用统一的后缀，可以快速理解字段含义。

+ \_cnt / \_count：计数类指标，如 login\_cnt (登录次数)。
+ \_amt / \_amount：金额类指标，如 payment\_amt (支付金额)。
+ \_num / \_number：数量类指标，如 item\_num (商品数量)。
+ \_ratio / \_rate：比率类指标，如 refund\_ratio (退款率)。
+ \_d / \_dur：时长类指标，单位秒，如 session\_d (会话时长)。

* **标志位字段（****Flag****）** ：布尔类型的字段，推荐使用 is\_ 或 has\_ 开头，如 is\_vip, has\_coupon。
* **时间戳字段**：明确表明是时间、日期还是时间戳，并统一格式。如 gmt\_create (通用创建时间), order\_date (下单日期), login\_timestamp (登录时间戳)。
* **枚举类型字段**：字段名应体现其分类属性，如 order\_status, payment\_type。

**2.3.3****视图命名规范**

视图作为虚拟表，其命名应体现其“虚拟”特性，并能描述其功能。

* **前缀**：使用统一的视图前缀，如 v\_ 或 vw\_ 。
* **命名结构**：v\_[层级]\_[业务域]\_[内容描述]。例如 v\_ads\_realtime\_sales\_overview (应用层实时销售总览视图)。

**2.4****命名规范的落地与词根管理**

制定规范只是第一步，更关键的是如何落地。**词根管理**是确保命名规范一致性执行的有效手段 。企业应建立一个公共的“词根库”或“数据字典”，它维护了业务术语、中文含义、英文缩写/全称之间的映射关系。当开发人员需要为一个新业务概念命名时，必须首先查询词根库，使用已有的标准词根，或按规定流程申请添加新的词根。这从源头上保证了命名的一致性，避免了“千人千面”的混乱局面。

**第三章：规范的联动与治理**

数据仓库的分层规范与命名规范并非孤立存在，它们相互关联、相互支撑，共同构成了数据治理的核心框架。将它们与数据质量监控、元数据管理相结合，才能构建一个真正健壮的数据体系。

**3.1****命名规范与数据质量的协同作用**

命名规范本身就是一种“元数据”，它为数据质量的监控和保障提供了重要的上下文信息。

* **指导数据质量规则的设定**：规范的命名直接揭示了数据的业务属性，从而指导了数据质量校验规则的制定。例如，一个名为 user\_mobile\_no 的字段，其命名就暗示了需要对其进行手机号格式的正确性校验。一个在ODS层被标记为唯一键的字段 unique\_order\_id，在后续的DWD层就必须进行唯一性校验 。
* **提升问题定位的效率**：当数据质量出现问题时（如某个核心报表指标异常），结构化的命名和清晰的分层可以极大地加速问题的定位。例如，如果发现 ads\_trade\_gmv\_1d (应用层交易GMV日报) 的数据有误，可以根据命名和血缘关系，迅速回溯到 dws\_trade\_core\_stats\_1d (汇总层交易核心统计)，再到 dwd\_trade\_order\_detail\_di (明细层订单明细)，最终定位到是ODS层的哪个源表数据出现了问题 。
* **自动化治理的基础**：标准化的命名是实现数据治理自动化的前提。治理平台可以通过解析表名和字段名，自动识别其所属的业务域、安全等级、数据主题，并自动关联相应的数据质量规则和权限策略。例如，可以设定规则，所有名称中包含 id\_card 或 mobile 的字段，自动被标记为高敏感等级，并施加脱敏和访问控制策略。

**3.2****分层架构下的元数据管理**

元数据，即“关于数据的数据”，是数据仓库的“说明书”。分层规范和命名规范本身就是元数据最重要的组成部分。统一的元数据管理旨在将这些“说明书”集中管理起来，发挥更大价值。

* **元数据的范畴**：元数据主要分为三类：

+ **业务元数据**：描述数据的业务含义、业务规则、数据所有者等。命名规范中的业务域、主题、字段业务含义等都属于此类 。
+ **技术元数据**：描述数据的物理属性，如存储位置、数据类型、长度、表结构、ETL作业脚本、分区信息等。分层和命名规范中定义的层级、数据类型、分区后缀等都与技术元数据紧密相关 。
+ **操作元数据**：描述数据处理过程的信息，如作业运行时间、读取/写入记录数、数据质量校验结果等 。

* **统一元数据管理的实施策略**：

i.**建立统一元数据平台**：构建一个中央元数据知识库，自动采集或手动录入来自数据仓库各个层次、各个系统的元数据，形成统一的数据视图 。

ii.**制定统一元数据标准**：参考行业标准（如CWM），定义企业内部统一的元数据模型和标准，确保所有元数据能够被一致地理解和集成 。命名规范和分层规范是这个标准的核心。

iii.**打通元数据应用**：基于统一的元数据，可以构建丰富的数据应用，如：

+ **数据地图****/****数据资产目录**：让业务用户可以像逛图书馆一样方便地搜索、发现和理解数据。
+ **端到端血缘分析**：清晰地展示数据从源头到最终应用的完整加工链路。
+ **影响分析**：当某个表或字段需要变更时，能快速评估其对下游所有依赖任务和报表的影响。

**3.3****规范的生命周期管理**

数据仓库的建设规范不是一成不变的文档，而是一个需要持续维护和演进的动态体系。

* **建立治理组织**：成立虚拟或实体的数据治理委员会，负责制定、评审和发布数据仓库建设规范，并对执行情况进行监督。
* **评审与准入机制**：所有新的数据模型（表、字段）在进入生产环境之前，必须经过设计评审。评审的一个重要环节就是检查其是否严格遵守了分层规范和命名规范 。不符合规范的设计应被驳回，直到修改合规为止。
* **定期审计与优化**：定期对数据仓库中的存量模型进行审计，识别和清理不符合规范的、废弃的或冗余的模型。同时，随着业务的发展，规范本身也需要被重新审视和优化，以适应新的业务需求和技术演进。
