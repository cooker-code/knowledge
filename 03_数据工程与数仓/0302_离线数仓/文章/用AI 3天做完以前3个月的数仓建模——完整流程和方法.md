| --- |
| 去年我们做了一个零售会员域的数仓建模，从对齐需求到上线花了将近3个月。  今年做物流域重构，同等复杂度，我用了3天完成了80%的建模工作——剩下的20%是需要跟业务确认的业务规则，这部分AI替代不了。  时间差在哪里？不在于AI多聪明，在于我换了工作流程。 |

下面是我实际用的流程，不是理论，是上线过的方法。分3步。

|  |  |
| --- | --- |
| 1 | 第一步：用AI做需求提炼和实体识别（0.5天） |

以前这步最慢。要开需求会、整理会议纪要、再翻译成数据语言，往往来回拉扯两周都没结论。

**现在的方式：**把业务系统的字段清单、PRD文档、已有报表需求，全部丢给AI，让它完成两件事——

① **实体识别**：从需求描述里提炼出核心业务对象（事实表候选、维度表候选）

② **口径初稿**：对每个指标生成初版定义和计算口径，带出可能存在的歧义点

AI给出的实体清单和口径初稿，80%可以直接用于跟业务确认，会议效率提升了一倍。

|  |
| --- |
| **提示词示例：**"这是我们物流域的业务字段清单和报表需求，请帮我识别核心业务实体，并区分哪些是事实数据（有时间戳的行为数据）、哪些是维度数据（属性描述）。同时列出潜在的口径歧义点。" |

|  |  |
| --- | --- |
| 2 | 第二步：让AI生成DDL初稿和命名规范（1天） |

以前建模最花时间的环节之一——写DDL。光是字段命名、注释、分区设计，一张复杂的宽表能耗掉大半天。

**现在的方式：**把第一步的实体清单和业务字段，加上你的命名规范文档，直接让AI生成DDL初稿。

实测生成一张DWD层宽表的完整DDL（含字段注释、分区字段、生命周期设置），AI大约需要30秒，我人工审查+修改约15分钟。

以前这张表从零写完要2小时。

|  |
| --- |
| **关键要点：**必须把你公司的命名规范一起提供给AI（表名前缀规则、字段命名约定、禁用词汇等），否则AI会按通用规范生成，反而要大改。规范文档准备好后，每次复用即可。 |

一天内，ODS→DWD→DIM三层的DDL初稿全部完成，人工审核通过率约85%。

|  |  |
| --- | --- |
| 3 | 第三步：AI生成ETL SQL骨架 + 数据质量规则（1.5天） |

有了DDL，接下来是ETL。这一步以前最耗人，尤其是多表JOIN、字段清洗逻辑、NULL值处理——很多逻辑工程师写一遍就够了，但每次从零写还是慢。

**现在的方式：**把DDL和ODS源表结构给AI，让它生成ETL的SQL骨架——包括INSERT OVERWRITE结构、字段映射、基础清洗逻辑和注释占位符。

AI生成的SQL骨架约完成70%的工作量，剩余30%是业务特定的清洗规则和异常处理逻辑，需要人工补充。

同步让AI基于字段类型和业务含义，生成数据质量检查规则初稿（主键唯一性、非空约束、值域范围、字段波动性），覆盖率能达到60%-70%，剩余的再人工补充。

|  |
| --- |
| 🔑 关于"AI会不会做错"的问题  AI生成的SQL和规则肯定有错误——但这不是问题，这是正常的。你的工作从"从零开始写"变成了"Review和修正"。同样的输出，后者快5倍。会出错的AI，比让你从零开始的空白文档，还是要香得多。 |

|  |
| --- |
| ⚠️ 这个流程能快起来的前提  ① **你的命名规范文档要完整**——没有规范的AI生成是一场噩梦  ② **ODS层源表结构要干净**——源头混乱，AI只能帮你更快地生产乱  ③ **你要有Review能力**——AI生成的50%结果需要你判断对不对，判断力是核心竞争力 |

用了这个流程，物流域数仓建模，3天完成了以前3个月才能完成的工作量。

但这3天里，我没有让AI做任何一个业务判断——什么数据该有，什么口径代表什么，哪个字段对业务来说是关键，这些都是我决定的。

AI提速了执行，但没有替代判断。这正是数据人的护城河。
