事实表是指保存了大量业务度量数据的表，简单讲就是通过各种维度和一些指标值的组合来确定一个事实的，如通过时间维度、地域维度与指标值可以去确定在某时某地的一些指标值怎么样的事实。它主要特点是包含数字数据（即事实），并且这些数字信息可以汇总，以提供有关单位作为历史的数据。除数字度量外，事实表总是包含外键 ，用于关联与之相关的维度（维度表包含事实记录的特性） ，也包含可选的退化维度饱和日期／时间戳。查询请求的主要目标是基于事实表开展计算和聚集操作。

事实表作为数仓维度建模的核心，紧紧围绕着业务过程来设计，通过获取描述业务过程的度量来表达业务过程，包含了引用的维度和业务过程有关的度量。

**01**

**—**

**几个重要的概念**

1.1 粒度

事实表的每行数据记录代表一个业务事件（如下单、支付、退款、评价等），一条数据记录所表达的业务细节程度被称为事实表的粒度（即事实表的粒度决定了数据仓库中数据的详细程度）。通常，粒度可以通过两种方式来表达：

```
维度属性组合所表示的细节程度，一般适用于 “聚集型事实表”；所表示的具体业务含义。
```

定义事实表的粒度是整个维度建模中最重要步骤，定义了事实表的粒度，则事实表所能表达的数据的详细程度就确定了。在数据分析中，粒度决定着分析范围与细致程度；在数据记录中，粒度则决定着什么情况下增加一条记录。例如：在用户系统中，注册总人数可以按照区域、省市、国家等来进行统计，这就是地域层面上的不同统计粒度；而活跃用户数可以按天、周、月、年等来进行统计，这就是时间层面上不同的统计粒度。通常，为了便捷的下钻分析，我们都会使用到最小粒度（如订单表中一条订单记录就是最小粒度），使用最小粒度的优点：

```
可以频繁的ETL操作；很多数据挖掘需要最小粒度数据；方便向下钻取。
```

当然，使用最小粒度也有缺点：

```
存储和维护代价较高；需要进一步构建汇总事实表来支持汇总数据查询。
```

1.2 事实

作为度量业务过程的“事实”（事实表属性），一般为整型或浮点型的十进制数值，有可加性、半可加性和不可加性三种类型：

|  |
| --- |
| 可加型事实：指可以按照与事实表相关的任意维度汇总，且汇总结果有意义；  半可加型事实：只能按照特定维度汇总，不能对所有维度获得有意义的汇总（即事实可以进行汇总相加，但是汇总结果没有意义）；  不可加型事实：完全不具有可加性，比如比率型事实。对于不可加性的事实，可以分解为可加的组件来实现聚集。 |

例如，一条订单数据包括了“用户ID、会员等级、订单金额、账户余额”。其中：

* “订单金额”：属于可加型事实，它在所有维度下都可以统计且有意义。如对所有记录的订单金额相加汇总就是所有订单总额汇总统计，对日、周、月的订单金额相加汇总就是当日、周、月订单金额汇总统计；
* “账户余额”：属于半可加型事实，它只有在用户维度下的各种相加汇总统计才有意义，其他维度下的相加是没有意义的；
* “会员等级”：属于不可加型事实，它在任何维度下的相加汇总，结果都是没有意义的数字。

## 1.3 空值处理

1. 事实表中可以存在空值度量，所有聚合函数（SUM 、COUNT 、MIN 、MAX 、AVG）都可以对空值事实进行计算；
2. 事实表的外键中不能存在空值，否则会导致违反参照完整性的情况发生；
3. 关联的维度表必须用默认行（代理键）不能用空值外键表示位置的或者无法应用的条件。

## 1.4 一致性事实

1. 如果需要比较或者计算不同事实表中的事实，应针对事实的技术定义是相同的；
2. 如果不同的事实表定义是一致的，则这些一致性事实应该具有相同的命名；
3. 如果不同的事实表定义是不一致的，则应该有不同的命名用于告诫业务用户和BI应用。

**事务事实表**

事务事实表主要用于记录与业务对象相关的事务性数据，它的粒度是每一行数据对应一个事务，或者一行对应事务中的一个条目，它是最细粒度的事实表，因此也称为“原子事实表”。一旦事务被提交，事实表数据被插入，数据就不再进行更改，其更新方式为增量更新。事务可以理解为业务过程中的各个事件节点，如电商交易过程中的创建订单、付款、发货、签收等都可以理解为事务。事务事实表针对这些过程构建，作为数仓的原子明细数据，可以跟踪定义业务过程的事件，提供丰富的统计与分析能力。

由于事实表具有稀疏性质 ，因此只有当天数据才会进入当天的事实表中，相当于每个分区里面都是每天的数据，不包含之前的数据。一般情况下，它又可以分为：单事务事实表和多事务事实表。

## 2.1 单事务事实表

针对于每个业务过程设计一个事实表，方便每个业务过程进行独立分析与研究。这样更方便跟踪业务流程细节数据，针对特殊的业务分析场景比较方便和灵活，数据处理上也更加灵活；其不足之处在于数仓中需要管理太多的事实表，同时跟踪业务流转不够直观。例如在交易流程中把下单和支付分别设计到不同事实表中。

## 2.2 多事务事实表

将不同的事实放到同一个事实表中，即同一个事实表包含不同的业务过程。这样能够更直观的跟踪业务流转和当前状态，流程事实集中，方便大部分的通用分析应用场景，由于和业务侧的数据模型设计思路一致，也是目前最常用的事实表设计。其不足之处在于细节数据跟踪不到位，特殊场景的分析不够灵活。

多事务事实表在设计时有两种方法进行事实的处理：

* 1）不同业务过程的事实使用不同的事实字段进行存放。它一般用于业务相似、粒度相同，但是业务过程的度量差异大的场景。有两个典型的问题：

```
① 在数据中遇到不是当前业务过程的度量，采用零值处理；② 表中存在多个业务，如何标记？给每一个数据加一个属性标识是否当日业务。
```

* 2）不同业务过程的事实使用同一个事实字段进行存放，用一个业务过程标签字段来标识业务类型（如商品的收藏/删除）。它一般用于业务相似、粒度相同，同时业务过程的度量差异不大的场景。但是有一个问题要注意，因为用同一个字段标识不同业务的度量，所以数据一个周期内会有多条记录。

**—**

**周期快照事实表**

周期快照事实表是以具有规律性的、可预见的时间间隔来记录事实，按时间周期进行汇总的事实表，主要用于记录反映与业务对象相关的周期性变化数据，也称为快照事实表或周期事实表。它的特点是随着业务周期性的推进而变化，记录某个时间段内的业务数据度量和状态度量的变化。周期性通常以日、周、月、年等为单位进行数据汇总统计，业务对象可以是各个维度的组合。常用于统计与业务对象相关的周期性数据，如销售日快照表、库存日快照表、月/季/年度利润表等，同时也可以用于分析和预测未来趋势和变化。

周期快照事实表的维度个数比事务事实表要少，但是记录的事实要比事务事实表多。这是因为事务事实表是稀疏表，只有当天发生的业务过程，事实表才会记录该业务过程的事 实， 如下单、支付等；而周期快照表是稠密表，无论当天是否有业务过程发 生，都会记录一行，比如针对卖家的历史至今的下单和支付金额，无论当天卖家是否有下单支付事实，都会给该卖家记录一行。

它有如下特性：

1）用快照采样状态

周期快照事实表通常是使用“周期+状态度量”的组合方式来记录数据，这里的状态指一段时间内某个事实的汇总，如月支付汇总金额就是一个状态值。快照事实表以预定的间隔采样状态度量（周期可以是天，周，月，季度，年，自然年等），这种间隔联合一个或多个维度，将被用来定义快照事实表的粒度，每行都将包含记录所涉及状态的事实（即周期性快照事实表是稠密的）。

2）快照粒度

事务事实表的粒度可以通过业务过程中所涉及的细节程度来描述，但快照事实表的粒度通常总是被多维度声明，可以简单地理解为快照需要采样的周期以及什么将被采样。例如，在淘宝交易卖家快照事实表中，粒度可以被理解为每天针对卖家的历史截至当日的下单支付金额进行快照。

3）密度与稀疏性

快照事实表和事务事实表的一个关键区别在密度上。事务事实表是稀疏的，只有当天发生的业务过程，事实表才会记录该业务过程的事实，如下单、支付等；而快照事实表是稠密的，无论当天是否有业务过程发生，都会记录一行，比如针对卖家的历史至今的下单和支付金额，无论当天卖家是否有下单支付事实，都会给该卖家记录一行。

4）半可加性

在快照事实表中收集到的状态度量都是半可加的。与事务事实表的可加性事实不同，半可加性事实不能根据时间维度获得有意义的汇总结果。

注意事项：

1）事务与快照成对设计

数据仓库维度建模时，对于事务事实表和快照事实表往往都是成对设计的，互相补充，以满足更多的下游统计分析需求，特别是在事务事实表的基础上可以加工快照事实表。

2）附加事实

快照事实表在确定状态度量时，一般都是保存采样周期结束时的状态度量。但是也有分析需求需要关注上一个采样周期结束时的状态度量，而又不愿意多次使用快照事实表，因此一般在设计周期快照事实表时会附加一些上一个采样周期的状态度量。

3）周期到日期的度量

在实际应用中，也有需要关注自然年至今、季度至今的一些状态度量，因此在确定周期快照事实表的度量时，也要考虑类似的度量值，以满足更多的统计分析需求。

**04**

**—**

**累积快照事实表**

在实际应用中，有时候需要统计类似于“买家下单到支付的时长、买家支付到卖家发货的时长”等情况，对于这种类似用于研究事件之间时间间隔的需求，采用累积快照事实表就可以很好地解决这类问题。它主要用于记录与业务对象相关的完全覆盖一个事实的生命周期、度量统计不确定周期的数据，也称为累计快照事实表或时间周期事实表。

相对于周期快照事实表记录的确定周期数据，而累积快照事实表记录的是不确定周期数据，它是针对一个业务对象完全覆盖一个事实的生命周期进行记录，它由多个周期数据组成，每行汇总了过程开始到结束之间的度量。每行数据类似于工作流一样，有事件的起点、过程于终点，并且每个关键步骤都包含日期字段。

例如，订单状态表就是一个典型的累积快照事实表，该表记录了订单的所有状态变化，如下单、付款、出库、发货与签收等。当订单下单时，只有下单时间而没有其他状态时间，则只记录下单时间；当完成支付时则更新支付时间，其他状态依次不断更新。由此可见，为了记录一个事实的生命周期中的关键时间点，累积快照事实表通常是一条记录包含多个时间字段，并针对这条记录不断进行更新。

**05**

**—**

**无事实的事实表**

相对于有事实的事实表，无事实的事实表则表示不包含事实或度量的事实表，它只有维度外键，而没有数字化的度量。表面上看，无事实的事实表是没有意义的，因为作为事实表最重要的就是度量。但在数据仓库中，这类事实表有其特殊用途，可以用在以下两个方便：

```
1）用来跟踪事件的发生，最常见的如日志类事实表。2）用来说明活动的范围， 常被用于回答“什么未发生”这样的问题
```

例如，在促销范围事实表中，一般可以回答如促销商品的销售情况， 但无法回答的一个重要问题是： 处于促销状态但尚未销售的产品包括哪些？

销售事实表所记录的仅仅是实际卖出的产品， 而不包括由于没有销售行为而销售数量为零的行。因为如果将包含零值的产品都加到事实表中， 那么事实表将变得非常巨大。 这时通过建立促销范围事实表，将商场需要促销的商品单独建立事实表保存， 然后通过这个促销范围事实表和销售事实表即可得出哪些促销商品没有销售出去。

**06**

**—**

**聚集事实表**

它是指通过对事务事实表的数据进行聚集汇总，得到一些常用的汇总数据，并将汇总好的这些数据聚集到聚集事实表中。这样，下游用户在需要一些汇总数据时，可以直接从聚集事实表中直接查询已经提前汇总好的数据，而不需要在进行汇总。其作用是：

```
1）减少数据仓库在响应查询时必须的工作量，能够快速响应查询；2）有利于减少不同用户访问明细数据带来的不一致问题。
```

需要注意的是：

1）聚集粒度可不同： 聚集并不需要保持与原始明细粒度数据 一样的粒度，聚集只关心所需要查询的维度。例如，订单涉及的维度有商品、买家、卖家、地域等，可以按照商品汇总一天的交易额、也可以按照商品与地域汇总一月的交易额；

2）聚集不是跨越事实的：聚集是针对原始星形模型进行的汇总，为了获取和查询与原始模型一致的结果，聚集的维度和度量必须与原始模型保持一致，因此，聚集是不跨越事实的。

3）聚集增加了 ETL 维护的难度：例如，当子类目对应的一级类目发生变更时，先前存在的、已经被汇总到聚集表中的数据，需要被重新调整；这一额外工作随着业务复杂性的增加，会导致多数 ETL 人员选择简单强力的方法，删除并重新聚集数据。

**07**

**—**

**合并事实表**

合并事实表遵循一个原则，就是相同粒度，数据可以来自多个过程，但是只要它们属于相同粒度，就可以合并为一个事实表，这类事实表特别适合经常需要共同分析的多过程度量。

与聚集事实表相似，合并事实表也会增加ETL处理过程的负担，但是降低了BI应用的分析代价。
