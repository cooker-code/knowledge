大家好，我是云祁！某大厂资深数据老兵。

中午十二点半，业务方发来一条消息："今天的订单数仓里怎么还没有？我们报表要用这个数据。"

你急急忙忙打开任务监控，看到昨晚那个全量 Merge 任务还在跑。已经跑了九个多小时，今天凌晨的调度窗口全部被它吃掉了。订单系统分了 16 个库、每库 64 张表，大促前流量一上来，合并任务的时间直接翻倍——这次没赶上，整条链路延迟超过 10 小时。

这不是你的配置错了，只是这套方案从一开始就不该用在这里。

## SeaTunnel 做离线同步，其实挺好的

不管怎么聊 Paimon，先把 SeaTunnel 说公道一点。

它本来就是干数据搬运的，Source 和 Sink 覆盖几十种，配置简单，批量写入吞吐量高。纯离线场景下——日终报表、T+1 分析、历史数据迁移——用它没问题，踩过的坑少，维护也简单。

但它的底层逻辑是**批处理**：周期调度触发、全量或增量抽取、写入 Hive 分区、再用 SQL 跑一遍 Merge。这套流程每一步都隐含着一个假设——**你的数据量是可预期的，你的调度窗口是够用的，你的数据主要是 INSERT。**

分库分表一旦上了规模，这三个假设都会失效。

数据量不可预期：大促、秒杀、节假日，写入峰值可以是平日的数倍，全量 Merge 的时间会随着数据量非线性增长，然后把后面的调度窗口全部压垮。

UPDATE 根本无法优雅处理：Hive 不支持行级更新，分库分表场景下同一个业务主键的数据可能分散在不同批次、不同分片里，你写出来的 Merge SQL 既复杂又脆弱，改一次逻辑要测很久。

分片多了以后小文件是另一个问题：1024 张源表，每次同步写几个文件，攒下来小文件数量很快就到了让查询性能崩掉的量级，然后要专门跑 Compaction 任务来清理，而且清理过程还可能和正在跑的查询互相打架。

还有一个更隐蔽的：Hive 写入过程中没有读写隔离，如果下游报表恰好在 Merge 的中间时刻开始跑，读到的可能是只写了一半的分区——数据撕裂，复现困难，让人怀疑人生。

## 为什么 Paimon 在这个场景更合适

Paimon 底层用的是 LSM Tree，跟 RocksDB 是同一类结构。简单理解就是：写入先进内存缓冲区，异步合并到磁盘，合并过程后台完成，不阻塞读写。这让它天然就支持高频 Upsert，不需要你在外面套一层 Merge 逻辑。

FlinkCDC 消费 MySQL Binlog，每一条 INSERT / UPDATE / DELETE 事件直接 Upsert 进 Paimon 主键表，数据延迟从 T+1 压到分钟级，正常情况下 1 分钟以内。多库多表的 Binlog 可以统一消费、写进同一张 Paimon 表，1024 张分片表汇成一张宽表，就是配置一个任务的事。

建表的时候大概是这个样子：

```
CREATE TABLE pm_ods.order_info (
    order_id   BIGINT,
    user_id    BIGINT,
    status     STRING,
    amount     DECIMAL(18, 2),
    updated_at TIMESTAMP,
    dt         STRING
) PARTITIONED BY (dt)
TBLPROPERTIES (
    'primary-key'    = 'dt,order_id',
    'bucket'         = '128',
    'sequence.field' = 'updated_at'  -- 保证乱序事件按时间戳合并，不是按到达顺序
);
```

小文件问题 Paimon 自己在后台处理，后台 Compaction 线程会持续合并，查询侧不需要感知。每次写入完成后生成新的 Snapshot，读操作基于 Snapshot 进行，不会读到中间状态——这是真正的读写隔离，不是靠调度窗口错开时间来"假装没有冲突"。

另一个不太被提到的点是 Schema Evolution。MySQL 源表加字段，Paimon 可以跟上，不需要重建表。Hive 方案你试试，ALTER TABLE 之后历史分区的数据处理够你头疼一阵。

## 有几个地方真的要小心

说了这么多 Paimon 的好，得说清楚它目前还不是一个"随便用"的工具。你可能得考虑到：可用性 99.9%，**不支持 P1/P2 级系统**，必须有兜底方案。

**Bucket 数量一旦定了就很难改**，这是建表前最需要认真估算的参数。Bucket 太少会写入热点，单个 Bucket 文件越堆越多；Bucket 太多，查小表时会扫大量空文件，反而更慢。大概的经验是每个 Bucket 对应 128MB 到 512MB 数据，但分库分表场景下要结合 QPS 和分片分布来算，最好上线前压一下。

**分库分表的主键不能用数据库自增 id。** 各分片各自自增，`id=1` 在 16 个库里各有一条，如果直接用 `id` 当 Paimon 主键，不同分片的数据会互相覆盖。主键必须能在业务上全局唯一，订单表通常是 `(dt, order_sn)` 这样的组合，不是技术键。这个判断失误了，数据会静悄悄地错，而不是直接报错。

**FlinkCDC 任务要常驻**，7×24 小时都在跑，计算资源是持续消耗的，不像 SeaTunnel 只在调度窗口有开销。如果你的业务其实对实时性没有真实需求，这笔资源就是多花的。另外，重要任务必须开 EXACTLY\_ONCE 并配置好 Checkpoint，任务失败如果从头跑而不是从 Checkpoint 恢复，数据会有空洞，还很难排查。

## 什么时候不要动

Paimon 适合的场景很具体：数据时效性真的有要求（小时级以内），源端 UPDATE/DELETE 比较多，多库多表需要汇总。在这个范围内，它比 SeaTunnel+Hive 方案好太多。

但如果只是纯离线分析、T+1 完全够用、数据几乎只有 INSERT，那 SeaTunnel 跑得好好的，没必要折腾。目前 Paimon 还在快速迭代，部分边缘场景还在打磨，引入它需要团队有 Flink 运维能力，并且学习成本还不低。把成本和收益对应到具体场景再做决定——而不是因为 "别人都在用 Paimon" 就跟着切。
