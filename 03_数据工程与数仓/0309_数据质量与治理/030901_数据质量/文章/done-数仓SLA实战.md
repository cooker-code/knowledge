背景

经历了xx业务的进攻期，流量从每日20亿上涨到500亿，记录一下数仓中离线链路的SLA保障思考～

SLA（Service Level Agreement）：服务级别协议，对互联网公司来说是网站服务可用性的保证。数据 SLA，即数据可用性保证，一般以数据产出时间作为 SLA。

**01**

**基线监控&诊断**

首先，我们对xx业务设置了N条核心基线，清晰每条基线服务的人群及定位，同时订阅基线的完成情况，3:00～8:00每间隔30min自动同步N条基线的执行情况（完成情况&链路名称&目标时间&完成时间&余量时间）至值班群。基线负责人对结果负责，监控整条链路并从链路整体推动优化（卡点任务or层级过深等），避免大家只盯着自己的某几个任务疯狂优化（木桶要挑短的补～😂）

基线监控

侧重于“事前预测性”监控，监控整条任务链路的完成情况。

｜告警能力

+ 基线完成

+ 破预警线

+ 破目标线

+ 基线持续恶化：当基线已经破预警线，且基线余量还在不断减小时，通过配置的不同预警梯度线，平台会进行相应的通知告警。

+ 基线XXX任务变慢

+ 基线XXX任务失败

｜诊断能力

+ 最晚链路（也叫关键链路）分析

+ 基线破线分析
+ 运行时长波动分析
+ 慢任务分析

* Tips：基线的目标任务不要设置太多，否则会导致对于某些链路“失聪”

｜名词解释：

+ 基线：用户将重要任务加到基线上并设置承诺时间后，系统将根据任务运行情况计算基线任务的预计完成时间。当系统判断基线任务可能无法在承诺时间前完成，系统将发出告警。

+ 基线目标任务：被监控的数据加工链路最末级的任务(节点)。
+ 基线目标时间：基线上的目标任务的最晚完成时间（即需保障的数据最晚的产出时间），晚于该时间则会出现线上事故。可以调整目标任务的“预计完成时间”，预留一定的时间处理异常。
+ 基线预警时间：即目标任务的预计完成时间。
+ 基线预警余量：基线发生问题后的可处理问题的Buffer。计算逻辑为 “目标时间” - 目标任务最晚“预计完成时间”。
+ 基线实例：系统使用基线实例计算任务每次运行的预计完成时间。基线实例的状态包括：安全、预警、破线。

- 安全：预计完成时间 < 预警时间。

- 预警：预警时间 < 预计完成时间 < 目标时间。

- 破线：预计完成时间 > 目标时间。

+ 最晚链路：影响基线任务的多条路径中，最晚完成的数据加工路径。
+ 基线链路预警时间：平台通过基线的预警时间，反推出上游每个任务的最晚开始的预警时间和最晚结束的预警时间，如果超过该时间，有可能会导致对应基线破最终的预警线。可以配置不同的预警梯度线。
+ 基线链路破线时间：平台通过基线的目标时间，反推出上游每个任务的最晚开始的破线时间和最晚结束的破线时间，如果超过该时间，有可能会导致对应基线超过最终的目标时间。

**02**

任务监控&优化

其次，我们对xx业务的相关核心任务升至最高级别L0并加入核心队列（资源倾斜），对于历史老任务升级执行引擎（hive→spark、spark3.0→spark3.4 ），对于高频失败、波动大、耗时长任务进行调优（单点爆破）。

任务监控

侧重于“事后结果性”监控，监控单个任务成功/失败/异常。

｜配置告警触发条件

+ 任务（环节）成功
+ 任务（环节）失败
+ 任务（环节）超过计划时间未开始
+ 任务（环节）超过计划时间未结束
+ 任务（环节）执行时长超长
+ 任务超过SLA时间
+ 任务运行时长大于平均时长
+ 任务运行时长小于平均时长
+ 任务（环节）每次失败
+ 任务（环节）失败超过N次
+ 任务（环节）开始延迟
+ 任务（环节）结束延迟
+ 任务在运行时间N分钟后未开始
+ 任务在运行时间N分钟后未结束

**03**

质量监控

同时，我们对xx业务增加了大量的数据质量监控，避免任务产出不符合预期的问题数据，影响正常使用和业务决策，同时降低问题处理的时间成本，避免大量下游任务重新运行带来的资源浪费。

质量监控

侧重于“事中准确性”干预，拦截问题任务，阻断脏数据向下游蔓延。

｜执行时机

+ 前置（任务开始前对上游数据）
+ 后置（任务结束后对产出数据）

｜强弱属性

+ 强规则（任务失败，熔断下游任务）
+ 弱规则（任务成功，仅通知负责人）

｜配置告警触发条件

+ 表行数

- 当前值（完整性）

* 当天统计到校验分区对应校验字段的空值个数（null值和空字符串）

- n日对比（准确性）

* 表行数当前值/前n天平均值

+ 表（分区）大小

- 当前值（完整性）

* 固定值规则，当天统计校验分区的表文件大小（字节）

- n日对比（准确性）

* 对比规则，表（分区）大小当前值/前n天平均值

+ 主键校验

- 空值个数/占比（完整性）

* 当天统计到校验分区对应的主键字段中存在空值的行数/占比
* （null值和空字符串）注：如果多个字段，其中1个为空，则认定为空值

- 重复值个数/占比（唯一性）

* 统计表的主键组合的数据中，存在重复的主键数 /占比

+ 字段（数值+非数值）

- 空值个数/占比（完整性）

* 当天统计到校验分区对应校验字段的空值个数/占比

- 重复值个数/占比（唯一性）

* 当天统计到校验分区对应校验字段的（value）重复值个数/占比
* 比如A字段出现了3个a，2个b，重复值个数为2

- 离散值-分组个数/占比（准确性）

* 某字段（数值+非数值）group by后的分组个数/各分组数值占比

- 枚举值（完整性）

* 扫描对应分区的校验字段所在列的数据枚举值与xx获取到的枚举值进行对比校验

- 枚举值占比（有效性）

* 扫描对应字段枚举值的有效占比情况。计算公式为：有效枚举值行数/全部行数 \* 100%

+ 字段（非数值）

- 格式-非全英文大写/小写（有效性）

* 扫描字段所在列的数据格式类型为非英文大写/小写的count值

- 格式-非合规日期行数（有效性）

* 扫描字段所在列的数据格式类型为非日期（YYYY-MM-DD）的count值

+ 字段（数值型）

- 最小值/最大值/平均值/汇总值/方差（有效性）

* 当天统计到校验分区对应校验字段的最小值(min)/最大值(max)/平均值(avg)/汇总值(sum) /方差(VAR\_POP)

- 平均值-n日对比（准确性）

* 字段平均值当前值/前n天平均值

**04**

复盘

最后，复盘流程必不可少～复盘不是为了甩锅定责，核心目的是获取经验教训，避免重复错误，优化流程并沉淀SOP。

事前的预警+事中的干预+事后的复盘，重视技术文档和SOP的沉淀，共同保障SLA产出稳定。
