## 技术定位

| 项 | 内容 |
|---|---|
| 技术名 | 调度补跑 |
| 一级类目 | 数据工程与数仓 |
| 二级类目 | 调度编排 |
| 技术本体 | 数据任务失败、延迟或依赖不满足后，按依赖、周期、资源和一致性要求恢复产出的调度治理能力 |
| 全局架构位置 | 位于调度系统、任务元数据、血缘、资源队列和数据质量之间，承担失败恢复和 SLA 修复 |
| 主要使用者 | 数据平台工程师、数据开发、运维、数据治理负责人 |
| 主要产出 | 补跑计划、补跑实例、状态回写、影响评估、告警和复盘记录 |

## 架构图

```mermaid
flowchart LR
  Fail["失败/冻结/依赖未就绪"] --> Trigger["触发判断"]
  Trigger --> Impact["影响评估\n血缘/SLA/下游深度"]
  Impact --> Strategy["策略选择\n并行/串行/跳过历史/断点"]
  Strategy --> Resource["资源与时窗校验"]
  Resource --> Execute["补跑执行"]
  Execute --> State["状态回写\n产出可用/血缘刷新"]
  State --> Observe["监控告警"]
  Observe --> Review["复盘与策略库"]
  Review --> Strategy
```

## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 触发判断 | 判断失败、依赖未就绪、SLA 偏差是否需要补跑 | 触发阈值、关键路径、误触发 |
| 影响评估 | 计算受影响表、指标、下游深度和恢复窗口 | 依赖血缘是否可信、SLA 是否量化 |
| 策略库 | 根据周期一致性和跨周期依赖选择补跑方式 | 并行、串行、最近一次、首个失败点前一成功点 |
| 资源控制 | 隔离补跑资源池、控制并发和时窗 | 不扰正班、避免资源雪崩 |
| 幂等与断点 | 保证重复执行安全，失败可续跑 | 写入覆盖、分区状态、事务边界 |
| 观测复盘 | 记录补跑成功率、耗时、原因和后续策略优化 | 告警抑制、事件关联、复盘闭环 |

## 上下游

| 方向 | 对象 | 关系 |
|---|---|---|
| 上游 | 调度系统、任务状态、血缘、数据质量规则 | 提供失败信号、依赖关系和影响范围 |
| 下游 | 数仓表、指标、报表、业务 SLA | 消费补跑后的可用产出 |
| 依赖 | 资源队列、元数据、告警系统、权限系统 | 决定能否安全执行和追责 |

## 横向对标

| 对标技术/方式 | 对标点 | 调度补跑优势 | 调度补跑劣势 | 使用判断 |
|---|---|---|---|---|
| 手工重跑 | 恢复失败任务 | 可标准化、可观测、可控资源 | 建设成本更高 | 高频失败、SLA 明确时必须自动化 |
| 普通重试 | 失败恢复 | 补跑能处理跨周期依赖和影响评估 | 比单任务重试更复杂 | 临时网络错误用重试，链路阻断用补跑 |
| 数据质量拦截 | 防止错误数据下游扩散 | 补跑负责恢复，质量负责判断可用性 | 两者边界容易混 | 质量失败触发补跑，但质量规则不替代调度策略 |
| 资源调度器 | 控制计算资源 | 补跑关注恢复计划和状态 | 不负责底层资源分配算法 | 补跑策略要调用资源队列，而不是绕开资源治理 |
