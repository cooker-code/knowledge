## 核心问题

实时业务平台常遇到“事件流在 Kafka 里不可探查、在 Flink State 里不可见、落到 OLAP 又有延迟和链路成本”的问题。Fluss 的场景价值是把实时事件流变成可列裁剪、可点查、可订阅、可被 Flink 继续计算的分析热层。

## 判断准则

| 场景 | Fluss 贡献 | 边界 |
|---|---|---|
| AB 实验实时分析 | 降低重复消费、支持列裁剪和点查，缓解大 State 作业维护压力 | 科学实验评估口径仍在业务模型和统计方法，不在 Fluss |
| Agent 全链路事件 | 接收 Hook 事件流，供 Flink 和模型推理做窗口聚合、链路识别和告警 | 风险识别能力来自规则/模型，Fluss 只负责事件可观测和存取 |
| 实时 OLAP 探查 | 让消息流具备更强的在线探查能力 | 复杂多维分析仍应进入 OLAP 引擎或湖仓查询层 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 场景用了 Fluss，核心问题就是 Fluss | 场景文要拆出 Fluss 在链路中的职责，业务算法和平台治理另归其位 |
| Fluss 能直接解决实时数仓所有问题 | 它主要解决实时热层存取和状态外置，建模、口径、权限、质量仍要由数仓体系治理 |

## 待验证缺口

- Fluss 在高基数点查、列裁剪、更新流订阅下的实际延迟和成本。
- 与 Kafka + Redis、Kafka + OLAP、Flink State 三类方案的同口径压测。
