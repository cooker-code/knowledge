## 文章处理 SOP

### 前置：内容理解四步法

四步法详见根 [AGENTS.md](../../../AGENTS.md#内容理解四步法)，本节只举 Flink 领域典型反例：

- 标题「Flink 写 Doris 性能调优」→ 正文核心是 StreamLoad 两阶段提交机制 → 主问题是 **Connector 写入一致性**（域 G），不是 Doris 性能调优。
- 标题「DolphinScheduler 新增 FlinkSQL 支持」→ 主问题是 DS 调度平台，不是 Flink 引擎 → `⊘已迁移`。
- 配置清单只列参数无机制 → 不单独新建知识点，可作为已有知识点的补充来源。

---

### 步骤一：判断是否属于 Flink 本体

先在上级 [实时计算 AGENTS](../AGENTS.md) 确认文章属于 Flink 引擎本体。
以下类型不在此目录处理：

| 文章主问题 | 正确目录 |
|---|---|
| Docker/K8s 容器化/镜像/集群部署 | `07_工程与架构/0704_K8s&Docker容器化/` |
| DolphinScheduler/Airflow/Streampark 调度 Flink | `03_数据工程与数仓/0306_调度编排/` |
| StarRocks/Doris SMT 同步工具 | 对应 OLAP 目录 |
| CDC 技术综述（Binlog/全量/增量） | `030302_Flink CDC/` |
| 实时数仓整体架构/行业方案 | `03_数据工程与数仓/` 上层 |

### 步骤二：按文章类型处理

| 文章类型 | 蒸馏步骤 |
|---|---|
| 原理/机制文 | ① 查知识地图定位域（A–H）；② 提取判断准则和失败模式；③ 指纹相同无增量 → 只追加来源，不新建；④ 有新边界/失败模式 → 合并已有知识点或新建 |
| 调优/实战文 | ① 提取可操作判断树和参数边界；② 必须有失败信号或指标，否则降权；③ 合并或新建 |
| 案例/复盘文 | ① 只提取与已有准则不同的认知增量；② 无新增判断 → 只追加来源到已有知识点，不新建 |
| 版本发布/FLIP | ① 只摘取影响判断准则的 breaking change；② 追加到[版本记录](030301_版本记录.md)；③ 文章本身不新建知识点 |
| 面试题/入门教程 | 默认`✗跳过`并删除；若含反直觉准则则提取后删除 |

### 步骤三：排重判断

1. 查 [知识地图](030301_知识地图.md)，定位属于 A–H 哪个域
2. 查该域已有知识点文件，比对问题指纹（技术 + 模块 + 机制 + 解决问题 + 边界）
3. 指纹相同且无新增增量 → 只追加本地路径到知识点 `## 来源`，不新建
4. 有冲突 → 写入知识点冲突点，不直接覆盖已有准则
5. 有新边界/失败模式/实践证据 → 合并或新建，更新知识地图覆盖状态

### 步骤四：文章处理后的操作

- `✗跳过` 类文章：**直接从 `文章/` 目录删除**
- `⊘已迁移` 类文章：**移动到对应目录后删除此处文件**
- `✓已蒸馏` 类文章：**保留在 `文章/`，在知识点 `## 来源` 中引用**
- `版本资讯` 类文章：关键信息摘入[版本记录](030301_版本记录.md)后**删除文章**

### 步骤五：知识点文件格式规范

```markdown
## 来源
- `../文章/done-文章文件名.md`   ← 只存本地路径，不存外部 URL

## 核心问题
这个知识点回答什么问题

## 判断准则
可复用的规则，适用场景和边界

## 认知偏差
常见的错误认知或反直觉点

## 待验证缺口
机制未经实验验证的部分
```

> 版本依赖：若结论依赖特定版本，在文件顶部标注 `> 验证版本：Flink x.x`，并查阅[版本记录](030301_版本记录.md)。
