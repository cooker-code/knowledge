作为 Apache Flink 的原始创建者，我们经常被要求提供有关监控 Flink 应用程序的最佳实践，人们想知道他们应该大规模监控他们的应用程序的哪些指标。在这个基于之前的监控网络研讨会的两篇博文系列中，我们想分享我们在监控方面的经验，关注要查看的指标，并解释如何解释它们。

为了让每个人都在同一个页面上，我们将开始详细介绍 Apache Flink 的监控和指标的概念，并回答如何监控 Apache Flink 的问题。然后，我们将注意力转向持续监控的指标，并解释它们的特征以及它们如何帮助了解正在发生的事情。后续博客文章将涉及有助于故障排除的指标，并展示我们在这里和我们的 Apache Flink 训练场使用的仪表板。

### 概念

根据 Technopedia [1]，应用程序监控...

*...是确保软件应用程序以预期的方式和范围处理和执行的过程。该技术定期识别、测量和评估应用程序的性能，并提供隔离和纠正任何异常或缺陷的方法。*

这是一个相当通用的领域，那么监控 Apache Flink 有什么特别之处呢？

首先，*Flink 是一个**分布式**流处理框架*！因此，作业不仅可以在具有跨多个抽象层（虚拟化、Kubernetes 等）的大量不同环境的多台机器上以高并行度运行，而且还可以在不同的部署模式（应用程序、每个作业、会话模式）中运行. 较大的工作通常会给监控和解释底层发生的事情带来一些挑战。

除此之外，Flink 作业运行具有自定义业务逻辑的任意用户代码，并且应用程序行为的巨大差异可能会影响底层资源并暴露不同的瓶颈。

#### 常用工具

用于监控大型应用程序的工具集非常广泛：

*日志记录*对于调试特定的故障非常有用（所以不要为了减小日志的大小而盲目地禁用或减少它）。但是，它并不真正适合大规模监控，因为您通常必须记录太多数据才能连续监控您的应用程序，并且可能难以基于此设置警报。在这方面，通用度量系统通常具有更好的工具。

*系统级和集群指标*可帮助您在粗粒度级别了解您的应用程序，例如，底层 Flink、Kubernetes 等集群的行为方式，但不会让您深入了解应用程序层。这就是为什么我们建议使用反映应用程序关键部分和状态的*自定义应用程序指标*来丰富您的应用程序。基于这些，您可以了解您的应用程序在特定情况下正在做什么，例如，在对问题进行故障排除时。

最后但同样重要的是，*分析器*可以为您提供对应用程序的非常详细的见解，以识别瓶颈、线程锁和拥塞、解决内存泄漏和 GC 问题等等。虽然它们对于故障排除和性能调整非常有用，但它们并不真正适合大规模监控，因为它们积累了相当多的数据，很难在问题发生之前用于警报。

#### Flink 的指标体系

让我们看看 Flink 如何使指标可用，支持哪些指标类型，以及如何使用这些指标。

首先，Flink 度量是将标识符与度量配对的对象。有 4 种不同类型的指标：

* *计数器*对事物进行计数，例如numRecordsIn
* *计量*器计数和测量速率，例如，numRecordsInPerSecond
* *直方图*测量统计分布，例如延迟分布，并且可以轻松提供百分位数

每个指标都被限定在 Flink 运行时的特定上下文中，其中范围成为指标标识符的一部分，如<system scope> [+ <user scope>] + <metric name>。在 JobManager (JM)、JM+job、TaskManager (TM)、TM+job、task 和 operator有度量的系统范围。您还可以微调范围格式以根据您的需要调整指标标识符。

在 Flink 中公开您自己的自定义指标相当容易，并且Flink 文档已经提供了有关此主题的一些详细信息，因此我们不会在这里详细介绍。

指标可以通过MetricReporter、Flink REST API 或 Flink 的 Web UI 导出。虽然很明显 REST API 可用于自动化，但您可能会想使用 Flink 的 Web UI 来监控您的应用程序（和集群）。然而，Flink 并不是一个成熟的指标系统，它也没有尝试提供合适的仪表板进行监控——作为调试体验的一部分，偶尔使用 UI 进行检查是可以的；它缺乏适当的度量系统和仪表板的适当功能。因此，任何严重的应用程序监控都会通过指标报告器，它将度量值写入外部系统。Flink 捆绑了大量此类报告器，例如 Prometheus、Datadoc、Ganglia、InfluxDB 等，如果您需要其他东西，也可以轻松编写自己的报告器。

### 持续监控的指标

Flink 包含许多开箱即用的指标。在这里，我们将介绍一些最有用的示例，您可以使用它们来持续监控（大规模）应用程序、设置警报或用于故障排除。我们将提供指标仪表板的屏幕截图，并将在结束本系列的后续博客文章中分享完整的仪表板。

#### 应用程序运行状况

Flink 应用程序需要注意的明显事项是它是否仍在运行（正常运行时间）或它不运行/重新启动的频率（numRestarts）。您应该为每一个设置适当的警报。

作业可能重新启动的原因有很多。然而，在一个表现良好的系统中，大多数这些都源于瞬态故障，可以忽略不计。但是，由于这些可能随时发生，因此您需要确保恢复速度很快，这就是您可以使用restartingTime指标跟踪的内容。

如果restartingTime很高并且您有一个延迟关键的工作，您可能需要在这方面对其进行调整。

*Flink 监控：你的工作是否正在运行，是否正在优雅地恢复？*

每当发生故障（暂时性或非暂时性）并触发重新启动时，容错的 Flink 应用程序将从最新的检查点或保存点重新启动作业。对于这些情况，您可能希望确保不必返回太远并重新处理大量数据，因为这会有效地增加您的工作停机时间。您可以为此跟踪各种（一般）检查点指标，例如numberOfCompletedCheckpoints、numberOfFailedCheckpoints、numberOfInProgressCheckpoints。对于在预先配置的失败检查点尝试次数上重新启动的作业，您可能还需要提前跟踪检查点故障。警报可以设置在失败检查点数量的某个阈值上，或者在很久以前创建的最后一个成功检查点上设置。\*

\* 不幸的是，“最后完成检查点的时间”没有直接的度量标准。您可以从最后一次增加的numberOfCompletedCheckpoints中提取它，从 Flink 的 REST API 中检索它，或者根据您的检查点频率计算它，并在多个连续检查点失败时设置警报。

#### 检查点

在我们研究检查点时，让我们探讨一些更多细节，例如持续时间、大小和对齐时间：lastCheckpointDuration应该低于检查点超时，否则检查点将失败。如果持续时间不断增加，您可以为此设置警报，但相对于误报，在实际检查点故障时发出警报可能更安全（见上文）。lastCheckpointSize可用于估计当前状态大小。然而，对于增量检查点，这在过去并没有太大意义，Flink 1.15 添加了新的lastCheckpointFullSize指标，它提供完整的检查点大小（包括与以前的检查点共享的文件），而不是增量检查点中的字节数。checkpointAlignmentTime也更多是用于解决检查点故障的指标。

*Flink 监控：用于监控和故障排除的检查点详细信息*

#### 延迟

如果您的工作运行事件时间逻辑，最重要的部分是：“我们取得进展了吗？”。

对于使用事件时间处理的应用程序，水印继续前进很重要。通过观察当前时间戳和水印之间的差异，可以很好地监控对时间敏感的操作符（例如过程函数和窗口）的水印。如果此*事件时间偏差* 或*事件时间滞后*异常高，则表明 (1) 您正在处理较旧的事件，可能是在从中断中恢复期间，或者 (2) 上游的某些东西很长时间没有发送水印时间，例如，源变得空闲。在后一种情况下，您需要找到解决方法或修复您的代码。

要从内置指标中获取事件时间延迟，我们可以绘制收集currentOutputWatermark指标值与实际值的时间戳，并设置适当的警报。

我们可以在不同的层次上这样做，例如，

* 为了收集有关事件时间如何*在整个作业图中取得进展的*信息，我们可以按作业的不同运算符进行分组；或者
* 如果我们想检查子任务之间的*事件时间偏差*，那么我们绘制特定运算符的每个子任务，如图所示。

然而，内置度量方法的一个缺点是处理时间时间戳，至少在 Prometheus 中，只有第二个精度。对于非常低延迟的用例，这可能还不够……

*Flink 监控：Event-time lag（内置）*

一般来说，上面的事件时间滞后定义运行良好，但它还包括事件在外部系统中等待时所花费的时间（在 Flink 之外，例如，在您的 Kafka 主题中，甚至在写入它之前） ）。如果您想要更细粒度的分辨率，只查看在 Flink 中花费的时间，或者监控不同风格的*处理延迟*，例如，通过从异步 I/O、延迟事件处理等中排除延迟，或者想要要包括在端到端完全一次模式下运行时来自事务接收器的延迟，您将需要自己的自定义指标来反映您的定义。

例如，在我们的训练练习中，我们创建了自己的自定义eventTimeLag指标，每当窗口触发时我们都会更新该指标。由于我们使用直方图，我们还可以轻松绘制可能对 SLA 管理有用的百分位数。

或者，您也可以查看 Flink 的内置延迟标记，但它们更像是一个调试工具，并且在定义上有点特殊。

*Flink 监控：Event-time lag（自定义）*

最后但同样重要的是，我想强调一些特定于连接器的指标，以表明我们是否跟上外部系统的步伐。例如，从 Kafka 或 Kinesis 读取时，records-lag-max和millisBehindLatest分别指示消费者（组）在消息队列头部之后的距离。为方便起见，Flink 将这些连接器指标转发到 Flink 的指标系统中。

* records-lag-max 显示任何分区的记录数方面的最大滞后。随着时间的推移，价值的增加是消费者群体跟不上生产者步伐的最好迹象。
* millisBehindLatest 显示消费者落后于流头部的毫秒数。对于任何消费者和 Kinesis 分片，这表明它落后于当前时间多远。

#### 吞吐量

任何性能监控都应该伴随吞吐量测量！

Flink 为每个子任务提供了numRecords(In|Out)PerSecond或numRecords(In|Out)等指标\*。尽管这些可用于您工作中的所有任务，但由于在 Flink 中向上游传播的背压，通常监控源输出的吞吐量并在该源上配置警报就足够了。每个任务和/或子任务的其他详细信息可能会在故障排除和性能调整期间为您提供帮助。

#### JVM指标（一）：持续监控

更深入一层，还有一些系统级指标需要查看以进行持续监控，从 JVM 开始，然后进入更深入的指标，例如来自底层部署系统（例如 Kubernetes）的指标。

在 Kubernetes 上，container\_memory\_working\_set\_bytes是您的 pod 的容器总共使用了多少内存的准确快照，包括 JVM 可以访问的内存以及它无法访问的区域，例如来自 RocksDB 的本机内存。此处的合理警报会将其与container\_spec\_memory\_limit\_bytes的值进行比较，并在机器重新启动之前尽早通知您，或者帮助您解决重新启动的原因。同样，任何Status.JVM.Memory.\*指标都将帮助您密切关注 JVM 内存及其组件。

请参阅Metrics以获取 Flink 提供的系统指标的完整列表。可以在 Flink 之外为底层系统组件收集更多指标。
