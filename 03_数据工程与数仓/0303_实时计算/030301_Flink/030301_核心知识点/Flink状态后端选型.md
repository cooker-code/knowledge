## 纵向理解

| 维度 | 判断 |
|---|---|
| 全局架构 | Flink 算子状态 -> State Backend -> Checkpoint Storage -> 外部持久化存储 |
| 本文位置 | 只讲 State Backend 选型，不讲 KeyGroup、TTL、状态 schema 演进 |
| 核心机制 | 内存状态、文件系统快照、RocksDB 本地磁盘和增量快照 |
| 使用链路 | 估算状态规模 -> 明确延迟/恢复目标 -> 选择后端 -> 观察 checkpoint 和恢复指标 |
| 前置条件 | 有稳定 Checkpoint 存储、监控和状态规模估算 |
| 边界 | 不解决 key 倾斜、业务状态无界增长和下游一致性问题 |

## 横向对标

| 对标技术 | 实现方式 | 优势 | 劣势 | 适合场景 |
|---|---|---|---|---|
| 内存类状态后端 | 状态在 JVM 内存，快照到远端或管理端 | 延迟低 | 规模和可靠性受限 | 开发测试、小状态 |
| 文件系统类状态后端 | 运行时内存，快照写文件系统 | 简单、适合中小状态 | 大状态全量快照可能慢 | 中小状态生产 |
| RocksDB 类状态后端 | 本地 LSM 存储，增量快照 | 大状态、恢复更可控 | 延迟、compaction、内存调优成本 | 大状态和长期状态 |
| Changelog State Backend | 状态变更写日志，状态表异步物化 | 降低 checkpoint 抖动 | 恢复重放和额外存储成本 | 大状态低延迟容错 |
