在 Shopify 中，我们将Apache Flink作为标准的有状态流处理引擎，为我们的BFCM Live Map等各种用例提供支持。我们的 Flink 应用程序部署在利用Google Kubernetes Engine的 Kubernetes 环境中。我们的集群采用配置使用HA模式，以避免作业管理器成为单点故障。我们还使用 RocksDB 状态后端并将我们的检查点和保存点写入 Google Cloud Storage (GCS)。

确保我们的 Flink 应用程序保持高性能和弹性是我们的首要任务之一。这也是我们最大的挑战之一。保持大型有状态应用程序的弹性很困难。一些数据模型需要存储巨大的状态（例如，13 TB 的销售数据，正如我们在我们的“永远存储状态：为什么它对您的分析有好处”演讲中所分享的那样）并且我们在性能调优上花费了很多时间，一路上学到了很多教训。

下面我们将引导您完成优化大型有状态 Apache Flink 应用程序的关键课程。我们将从介绍推荐的工具开始，然后关注性能和弹性方面。

## 1. 找到合适的分析工具

第一件事。手头拥有正确的分析工具是深入了解如何解决性能问题的关键。在部署我们的第一个应用程序时，我们发现使用以下工具集在调试 Flink 时很有用：

* **Async-profiler**：为 Java 虚拟机 (JVM) 构建的分析工具，用于跟踪多种事件，包括 CPU 周期、Java 堆分配以及缓存未命中和页面错误等性能计数器。它对火焰图的支持对于检查任务管理器将时间花在哪里特别有用。这个工具对于帮助我们调试 Kryo 序列化可以产生的性能差异特别有用。
* **VisualVM**：另一个 JVM 分析工具。您可以将此工具连接到正在运行的 JVM 实例，以实时查看堆分配和 CPU 使用情况。它对于交互式调试很有用。我们经常将其用作调查内存问题的初始工具。它有一个友好的用户界面，它不需要很多设置。
* **jemalloc + jeprof**：一个通用的 malloc 实现，从 1.12 版本开始被用作默认内存 Flink 分配器。jeprof 是一个与 jemalloc 一起工作的分析器。结合起来，您可以将任务管理器和作业管理器设置为自动转储内存配置文件，然后您可以使用 jeprof 进行分析。我们发现这对于长时间观察内存趋势很有用，可以帮助我们检测 RocksDB 中某个应用程序的内存泄漏。
* **Eclipse Memory Analyzer** **(Eclipse MAT) :** Eclipse MAT 是一个 Java 堆分析器，用于检查 JVM 堆转储的内存利用率、查找内存泄漏等。它可用于读取 jemalloc 输出的堆转储，以提供额外的分析层和解释。当我们需要调查我们的一个应用程序面临 GCS 文件接收器的内存不足问题时，该工具非常有用，我们将在下面进行描述。

这些工具可以在任何 Flink 应用程序中普遍使用，但它们绝不是唯一的工具——只是为我们工作的工具。JVM 拥有丰富的分析工具生态系统，从jmap等基本内置命令到Java Flight Recorder等现代高级功能，这些工具值得研究。

## 2. 避免 Kryo 序列化

Flink 为其使用的数据结构提供了多种不同的序列化器。大多数时候，我们使用 Flink 支持的 Scala 的案例类或 Avro 记录。它们可能不是性能最高的选择，但它们对开发人员体验非常有用。

当 Flink 无法使用组合案例类或 Aro 序列化序列序列化记录时，它会自动化为目标化。Kryo 序列化通常很慢，比您使用 async-profile 的其他数据类的工具，您实际上不会注意到这种性能下降。例如，当我们与我们不相关的性能问题时，观察 Kryo 类在内存使用时占用了多少空间。我们禁止了调试对 Kryo 的回退（`env.getConfig().disableGenericTypes();`），这会导致各种序列化无法导致回退以下。是我们遇到的一些示例以及我们如何修复它们：

* ***Scala 的 BigDecimal。****Flink 不支持序列化 Scala 的 BigDecimal 值，但它可以序列化 Java 值。默认使用 Java 的 BigDecimal 来避免这种序列化程序失败的实例。当您处理货币价值时，您可能会面临这个问题。*
* ***Scala ADT。*** Flink 不支持序列化使用密封 trait 和一些案例对象实现的 Scala ADT，通常表示类似枚举的数据结构。但是，它确实支持Scala 枚举，因此您可以使用它们。

解决所有这些问题后，我们注意到吞吐量增加了 20%。您可以采用相同的方法：禁用 Kryo 回退并修复弹出的问题，直到 Flink 不再使用 Kryo。

## 3.根据工作负载调整配置

Flink 在配置方面提供了无数的选项，但调优实际上取决于应用程序的状态和负载。例如，在 Shopify，典型的流媒体管道可能会受到不同的系统负载配置文件的影响，具体而言：

* **回填：**从时间开始并消耗输入源中可用的所有历史消息，直到管道赶上当前时间（即源滞后接近于零）。
* **稳定状态：**管道正在消耗接近实时的消息，并且源延迟最小（即秒）。
* **极端或季节性事件：**管道正在消耗接近实时的消息，但资源使用量激增并且可能会增加延迟。Shopify 的一个例子是限时抢购和黑色星期五等大批量销售时刻。

让我们关注前两个配置文件，因为它们定义了我们管道的关键操作模式。在回填开始期间，管道积压处于其最大大小，而在稳定状态期间，管道积压最小。我们希望回填尽快赶上现在，以减少需要从头重新处理所有数据的任务和代码更改的时间成本。

大数据量和速度标准使回填成为一项具有挑战性和计算量大的工作。对于一些大型应用程序，这可能意味着在几个小时内处理数百亿条消息。这就是为什么我们调整我们的管道以优先考虑吞吐量而不是新鲜度。另一方面，对于稳定状态，我们调整我们的应用程序以非常低的延迟运行并最大化所有输出的新鲜度。这会导致根据应用程序的当前状态设置两个不同的配置文件。

虽然您需要考虑您的系统负载以及它如何影响您的调整，但以下是可以应用于两种负载配置文件的一些注意事项：

* **输入源分区**（例如，Kafka 分区）。在稳定状态下，积压是最小的，因此少量的输入分区可能会为健康的管道提供足够的并行度，并且延迟最小。但是，在回填期间，确保您的管道可以从输入源获得尽可能多的吞吐量非常重要——输入分区越多，吞吐量就越高。因此，请务必在创建源时考虑回填场景。
* **背压。**当数据的生成速度快于下游运营商的消费速度时，性能瓶颈可能会对上游运营商造成背压。如果您的管道健康，您不太可能在稳定状态下看到背压。但是，在回填时，管道瓶颈会变得很明显（在作业图 UI 中显示为红色）。借此机会确定管道的缓慢阶段并尽可能优化它们。
* **水槽节流**。即使您的应用程序代码经过高度优化，管道仍有可能无法以您希望的速度快速写入接收器。接收器可能不支持很多连接，或者即使支持，它也可能会因过多的并发写入而不堪重负。如果可能，扩大接收器的资源（例如，向数据库添加更多节点或向 Kafka 主题添加分区），如果不在表上，请考虑减少接收器的并行度或传出的数量连接。
* **联网。** 最初引入网络缓冲区是为了提高资源利用率并增加吞吐量，但代价是延迟缓冲区队列中的消息。为了增加并行度，您可以添加更多的任务管理器并提供更多的任务槽。但请注意，如果管道图相当复杂并且包含多个 shuffle 操作，这通常需要随之增加网络缓冲区的数量（通过）。`taskmanager.memory.network.fraction`
* **检查点。**为了减少从故障中恢复的时间，`execution.checkpointing.interval`在稳定状态下保持较高的检查点频率 ( ) 很重要。但是，在回填期间，最好降低频率并避免相关的开销。根据管道状态的大小，可能需要调整任务管理器堆，以便有足够的内存来上传文件。此外，如果状态大小很大，请考虑使用增量检查点 ( `state.backend.incremental`)。`execution.checkpointing.timeout`最后，如有必要，请考虑增加检查点超时 ( )。

有关可能有用的其他 Flink 部署配置列表的文档，请参阅 Flink 文档。

## 4. 定位堆内存

Flink 提供了一个文件接收器，能够将文件写入文件系统或对象存储，如 HDFS、S3 或 GCS（Shopify 使用）。配置文件接收器非常简单，但让它高效可靠地工作可能会很棘手。

Flink 的 File Sink 在内存中维护一个分区（或桶）列表。每个桶由 BucketAssigner 确定。例如，自定义 BucketAssigner 可以使用提供的记录中的时间戳字段来生成一个看起来像date=2021-01-01. 这是 Hive 使用的一种非常流行的分区格式。

我们配置了一个 File Sink 并天真地将其添加到现有的 DataStream 中：

```
val records: DataStream[Record] = …   
val fileSink: SinkFunction[Record] = …  
records.addSink(fileSink)
```

这在测试中有效，但是当我们将其部署到真实环境并尝试在回填期间处理所有历史数据时，我们立即观察到内存问题。我们的应用程序将很快使用所有可用的 Java 堆并崩溃！即使我们多次增加内存，它仍然会崩溃。我们知道缓冲存储桶中的记录可能需要一些内存，但可能不需要几十 GB。

因此，我们在应用程序快要崩溃时进行了一些堆转储，并使用Eclipse MAT对其进行了分析。结果看起来真的很令人担忧：

Eclipse MAT：概述

在上面的堆转储中，你可以看到两个大对象几乎占据了整个堆。支配树报告清楚地显示（以红色突出显示）两个 HashMap，它们支持 File Sink 存储桶作为违规者：

Eclipse MAT：支配树

在进一步探索堆转储和应用程序日志后，我们意识到发生了什么。由于我们没有应用任何数据重组，每个任务管理器都会消耗可能最终在任何存储桶中的记录，这意味着：

* ***每个任务管理器都需要在内存中保存大量存储桶列表。***我们定期观察超过 500 个桶。
* ***由于上述原因，滚动和刷新文件的时间显着增加：***每个任务管理器上没有足够的数据来快速完成。

通过这样做，我们保证将具有相同分区或存储桶的记录路由到同一个任务管理器，从而使每个任务管理器在内存中保存的东西更少，刷新文件的速度更快。没有更多的内存问题！堆转储分析显示每个任务管理器的活动存储桶数量减少了 90%。

如果所选分区具有良好的分布，则此方法效果很好。但是，如果您的某些日子的数据明显多于其他日子（在进行历史回填时可能会出现），那么您最终可能会出现很大的偏差，从而导致内存问题。通过向分区键添加小时数来稍微更改分区以改善分布可能是解决此问题的好方法。

数据局部性是分布式系统中的一个重要方面，这一经验清楚地证明了这一点。一种简单的逻辑混洗数据以实现良好的数据并行性的技术也可以应用于其他接收器和运算符。

## 5. 使用 SSD 作为 RocksDB 存储

尽管RocksDB（最受欢迎和推荐的 Flink 状态后端）将一些数据保留在内存中，但大部分状态仍保存在磁盘上。因此，在处理大型有状态应用程序时，需要非常高性能的磁盘。这不是很明显，尤其是刚开始使用 Flink 时。例如，最初我们在部署具有少量状态的应用程序（例如，Kafka 消费者偏移量）时，开始将网络文件系统(NFS) 卷用于 RocksDB 状态后端。我们没有注意到任何性能问题，但受益于 NFS 提供的额外弹性。

但是，网上有很多资源推荐本地 SSD 等快速磁盘，因此我们尝试将GCP提供的一种用于我们的状态超过 8 TB 的应用程序。通过使用本地 SSD，我们注意到由于磁盘 I/O 速率的提高，处理速度提高了大约 10 倍。同时，如果实例宕机，GCP 中的本地 SSD 可能会丢失，但由于 Flink 检查点和保存点，可以轻松恢复状态

## 6.避免动态类加载

Flink 有多种方式类以供 Flink 应用程序使用。从调试类加载：

* ***Java 类路径：**这是 Java 的通用类路径，它包括 JDK 库，以及 Flink 的 /lib 文件夹中的所有代码（Apache Flink 的类和一些依赖项）。*
* ***Flink 插件组件：**插件代码文件夹位于 Flink 的 /plugins 文件夹中。Flink 的插件机制在启动时会动态加载一次。*
* ***动态用户代码：**这些都是包含在动态提交作业的 JAR 文件中的所有类（通过 REST、CLI、Web UI）。它们是按作业动态加载（和卸载）的。”*

动态用户代码在每个作业开始时加载，因此如果存在对旧类的延迟引用，可能会发生泄漏。每当 Flink 应用程序需要从暂时性故障中恢复时，它会重新启动作业并从最近可用的检查点恢复，并重新加载所有动态用户代码。

动态动态类加载之前和之后的元空间内存

我们在这些重启期间观察到内存泄漏，以“java.lang.OutOfMemoryError: Metaspace”错误的形式出现。正如您在上面的屏幕截图中所见，每次应用程序在修复之前重新启动时，它都会增加使用的 Metaspace 内存量。

通过将我们的应用程序代码放在 Java 的公共类路径上来禁用动态类加载，解决了这个问题。修复后上面的屏幕截图显示内存不会随着重新启动而增加。

该方案适用于以Application Mode运行 Flink 集群，不需要支持在单个 Flink 集群上运行多个作业。

## 7. 了解 RocksDB 内存使用情况

我们还观察到另一个与内存相关的问题，该问题非常难以调试，只要我们：

* 启动了一个有很多状态的 Flink 应用程序
* 等了至少一个小时
* 手动终止任务管理器容器之一。

我们预计将替换任务管理器添加到集群中（感谢 Kubernetes 部署），并在此之后不久进行应用程序恢复。但相反，我们注意到我们的另一个任务管理器因“内存不足”错误而崩溃，导致崩溃和重启的无休止循环：
