关注我们

**Flink 的网络堆栈是构成****flink-runtime****模块的核心组件之一，也是每个 Flink 作业的核心****，**它连接来自所有TaskManager的各个子任务。这是Flink待处理的数据必须流经的地方，针对吞吐量和延迟而言，对于 Flink 作业的性能至关重要。**与通过 Akka 使用 RPC 的 TaskManager 和 JobManager 之间的协调通道相比，TaskManager 之间的网络堆栈依赖于使用 Netty 的低级别 API。**

@精英打工人们

逻辑视图

Flink 的网络堆栈在相互通信时为子任务提供以下逻辑视图，例如在keyBy()所需的网络shuffle期间。

它抽象了以下三个概念的不同设置：

1、子任务输出类型（ ResultPartitionType ），有以下两种情况：

1. **管道式（有界或无界）：** 数据一生成就向下游发送，可能是作为有界或无界记录流逐一发送。
2. **阻塞****式****：** 仅当生成完整结果时才向下游发送数据。

2、调度类型，有以下三种类型：

1. **all at once (eager)****：**同时部署作业的所有子任务（对于流应用程序），即对于一个flink程序而言，所有的task是同时启动的。
2. **next stage on first output (lazy)****：**一旦有任何子task有结果输出，那么就开始启动下游的task。
3. **next stage on complete output****：**当某个算子的所有子task共同生成完整的输出集时，就开始启动下游的task。

3、传输类型，有以下两种方式：

1. **高吞吐：**Flink 不是逐条发送每个记录，而是将一堆记录缓冲到网络缓冲区中，然后将它们一起发送。这降低了每条记录的成本，但却大大提高了整个程序的吞吐量。
2. **低延迟：**通过超时机制将没有完全填充的缓冲区发送出去，当然虽然这样大大降低了task的延迟，因为缓冲区没有填满，因此吞吐量会有所降低。

另外，重要的是我们要知道**子任务输出类型和调度类型紧密相连，只有两者的特定组合才有效。**

管道化结果分区是流式输出，需要实时目标子任务来发送数据。可以在产生结果之前或首次输出时安排目标。批处理作业产生有界结果分区，而流式作业产生无界结果。

批处理作业也可能以阻塞方式产生结果，具体取决于所使用的运算符和连接模式。在这种情况下，必须先产生完整的结果，然后才能调度接收任务。这使得批处理作业能够更高效地工作并降低资源使用率。

不同任务之间的每个（远程）网络连接都会在 Flink 的网络堆栈中获得自己的 TCP 通道。但是，如果同一任务的不同子任务被调度到同一个 TaskManager 上，则它们与同一 TaskManager 的网络连接将被复用并共享单个 TCP 通道以减少资源使用。在我们的示例中，这适用于 A.1 → B.3、A.1 → B.4，以及 A.2 → B.3 和 A.2 → B.4，如下图所示：

每个子任务的结果称为ResultPartition ，每个子任务分为单独的ResultSubpartitions — 每个逻辑通道一个。此时，在堆栈中，Flink 不再处理单个记录，而是处理一组组装到网络缓冲区中的序列化记录。每个子任务在其自己的本地缓冲池中可用的缓冲区数量（发送方和接收方各一个）最多限制为

```
channels * buffers-per-channel + floating-buffers-per-gate
```

单个TaskManager上的缓冲区总数通常不需要配置。如果需要，请参阅配置网络缓冲区文档以了解有关如何执行此操作的详细信息。

* 造成背压

每当子任务的发送缓冲池耗尽时（缓冲区驻留在结果子分区的缓冲队列中或位于较低的 Netty 支持的网络堆栈内），生产者就会被阻塞，无法继续，并经历背压。接收器以类似的方式工作：较低网络堆栈中的任何传入 Netty 缓冲区都需要通过网络缓冲区可供 Flink 使用。如果相应子任务的缓冲池中没有可用的网络缓冲区，Flink 将停止从此通道读取数据，直到有可用的缓冲区为止。这将有效地对该多路复用上的所有发送子任务施加反压，因此也会限制其他接收子任务。下图说明了过载子任务 B.4 的情况，这会导致多路复用上的背压，并且还会阻止子任务 B.3 接收和处理更多缓冲区，即使它仍然有容量。

为了防止这种情况发生，Flink 1.5引入了自己的流控机制。

基于信用的流量控制

基于信用的流量控制可确保接收器有能力处理“线上”的任何内容。它基于网络缓冲区的可用性，作为 Flink 之前机制的自然扩展。每个远程输入通道现在不再只有共享的本地缓冲池，而是拥有自己的一组**独占缓冲区**。相反，本地缓冲池中的缓冲区称为**浮动缓冲区，**因为它们会浮动并且可供每个输入通道使用。

接收方将向发送方宣布可用的缓冲区作为**信用**（1 个缓冲区 = 1 个信用）。每个结果子分区将跟踪其**频道信用**。仅当信用可用时，缓冲区才会转发到较低的网络堆栈，并且每个发送的缓冲区都会将信用分数减少 1。除了缓冲区之外，我们还发送有关当前**积压**大小的信息，该信息指定有多少缓冲区正在该子分区的队列中等待。接收器将使用它来请求适当数量的浮动缓冲区，以加快积压处理速度。它将尝试获取与积压大小一样多的浮动缓冲区，但这可能并不总是可行，我们可能会获取一些缓冲区或根本没有缓冲区。接收方将利用检索到的缓冲区，并监听更多可用的缓冲区以继续。

基于信用的流量控制将使用buffers-per-channel来指定本地缓冲池有多少个缓冲区是独占的（强制）和float-buffers-per-gate （可选3 ），从而实现与没有流量控制时相同的缓冲区限制。已选择这两个参数的默认值，以便在具有正常延迟的健康网络的情况下，使用流量控制的最大（理论）吞吐量至少与不使用流量控制时一样好。您可能需要根据实际的往返时间和带宽来调整这些。

如果没有足够的可用缓冲区，每个缓冲池将获得全局可用缓冲区的相同份额 (± 1)。

与没有流量控制的接收方背压机制相反，信用提供了更直接的控制：如果接收方无法跟上，其可用信用最终将达到 0，并阻止发送方将缓冲区转发到较低的网络堆栈。仅此逻辑通道上存在反压，无需阻止从多路复用 TCP 通道读取。因此，其他接收器在处理可用缓冲区时不受影响。

由于通过流量控制，多路复用中的一个通道不能阻塞其另一个逻辑通道，因此总体资源利用率应该会增加。此外，通过完全控制“线路上”的数据量，我们还能够改进检查点对齐：如果没有流量控制，通道将需要一段时间才能填充网络堆栈的内部缓冲区并传播接收器不再读取。在此期间，可能存在大量缓冲区。任何检查点屏障都必须在这些缓冲区后面排队，因此必须等到所有这些缓冲区都被处理后才能启动（“屏障永远不会超过记录！”）。

然而，来自接收器的额外通告消息可能会产生一些额外成本，特别是在使用 SSL 加密通道的设置中。此外，单个输入通道无法使用缓冲池中的所有缓冲区，因为独占缓冲区不被共享。它也不能立即开始发送尽可能多的可用数据，因此在启动过程中（如果您生成数据的速度比宣布积分的速度快），可能需要更长的时间才能发送数据。虽然这可能会影响您的作业性能，但由于其所有优点，通常最好进行流量控制。您可能希望通过每通道缓冲区增加独占缓冲区的数量，但代价是使用更多内存。然而，与之前的实现相比，总体内存使用量可能仍然较低，因为较低的网络堆栈不再需要缓冲太多数据，因为我们总是可以立即将其传输到 Flink。

在使用基于信用的流量控制时，您可能还会注意到一件事：由于我们在发送方和接收方之间缓冲的数据较少，因此您可能会更早地遇到背压。然而，这是所希望的，并且缓冲更多数据并不能真正获得任何优势。如果您想缓冲更多但保持流量控制，您可以考虑通过float-buffers-per-gate增加浮动缓冲区的数量。

在 Flink 中，存在三种情况使得缓冲区可供 Netty 服务器使用：

**向缓冲区写入记录时缓冲区已满，或者****缓冲区超时，或者****发送特殊事件，例如检查点屏障。**

缓冲区满后刷新

RecordWriter 使用当前记录的本地序列化缓冲区，并将这些字节逐渐写入位于适当结果子分区队列的一个或多个网络缓冲区。虽然 RecordWriter 可以在多个子分区上工作，但每个子分区只有一个 RecordWriter 向其写入数据。另一方面，Netty 服务器从多个结果子分区中读取数据，并将适当的子分区复用到单个通道中，如上所述。这是一个经典的生产者-消费者模式，网络缓冲区位于中间，如下图所示。在 (1) 序列化和 (2) 将数据写入缓冲区之后，RecordWriter 相应地更新缓冲区的写入器索引。一旦缓冲区完全填满，记录写入器将 (3) 从其本地缓冲池中为当前记录的任何剩余字节（或下一个记录）获取一个新缓冲区，并将新缓冲区添加到子分区队列中。这将 (4) 通知 Netty 服务器数据可用（如果它还不知道） 4 。每当 Netty 有能力处理此通知时，它就会 (5) 获取缓冲区并沿着适当的 TCP 通道发送它。

如果队列中有更多已完成的缓冲区，我们可以假设它已经收到通知。

缓冲区超时后刷新

为了支持低延迟用例，我们不能仅仅依靠缓冲区已满来向下游发送数据。在某些情况下，某个通信通道可能没有太多记录流过，并且不必要地增加了您实际拥有的少量记录的延迟。因此，周期性进程将刷新堆栈中可用的任何数据：输出刷新器。周期性间隔可以通过StreamExecutionEnvironment#setBufferTimeout进行配置，并充当延迟5的上限（对于低吞吐量通道）。下图显示了它如何与其他组件交互：RecordWriter 像以前一样序列化并写入网络缓冲区，但同时，输出刷新器可以 (3,4) 通知 Netty 服务器数据可用（如果 Netty 尚未意识到）（类似到上面的“缓冲区已满”场景）。当 Netty 处理此通知 (5) 时，它将消耗缓冲区中的可用数据并更新缓冲区的读取器索引。该缓冲区保留在队列中 - Netty 服务器端对该缓冲区的任何进一步操作都将在下次继续从读取器索引中读取。

严格来说，输出刷新器不提供任何保证 - 它仅向 Netty 发送通知，Netty 可以随意/容量地拾取它。这也意味着如果通道有背压，输出冲洗器将不起作用。

@精英打工人们

特殊事件后的刷新

一些特殊事件如果通过 RecordWriter 发送，也会立即触发刷新。最重要的是检查点障碍或分区结束事件，它们显然应该快速进行，而不是等待输出刷新器启动。

与 Flink < 1.5 相比，请注意，(a) 网络缓冲区现在直接放置在子分区队列中，(b) 我们不会在每次刷新时关闭缓冲区。这给我们带来了一些优势：

* 更少的同步开销（输出刷新器和 RecordWriter 是独立的）
* 在Netty成为瓶颈的高负载场景下（无论是通过背压还是直接），我们仍然可以在不完整的缓冲区中积累数据
* 显着减少 Netty 通知

但是，你可能会注意到在低负载情况下 CPU 使用率和 TCP 数据包速率会增加。这是因为，随着更改的进行，Flink 将使用任何*可用的*CPU 周期来尝试维持所需的延迟。一旦负载增加，这将通过填充更多缓冲区来进行自我调整。高负载场景不受影响，甚至因为同步开销减少而获得更好的吞吐量。

如果您想更深入地了解 Flink 中生产者-消费者机制是如何实现的，请仔细查看 Flink 1.5 中引入的BufferBuilder和BufferConsumer类。虽然读取可能仅*针对每个缓冲区*，但写入它是*针对每个记录*，因此位于 Flink 中所有网络通信的热路径上。因此，我们非常清楚，我们需要任务线程和 Netty 线程之间的轻量级连接，这并不意味着太多的同步开销。有关更多详细信息，我们建议查看源代码。

延迟与吞吐量

引入网络缓冲区是为了获得更高的资源利用率和更高的吞吐量，但代价是一些记录在缓冲区中等待的时间更长。尽管可以通过缓冲区超时给出此等待时间的上限，但您可能会好奇地想了解有关延迟和吞吐量这两个维度之间的权衡的更多信息，因为显然，您无法同时获得两者。下图显示了从 0（刷新每条记录）到 100 毫秒（默认值）的缓冲区超时的各种值，并显示了具有 100 个节点和 8 个插槽（每个插槽都运行没有业务逻辑的作业）的集群上的最终吞吐率，并且因此仅测试网络堆栈。为了进行比较，我们还绘制了添加低延迟改进（如上所述）之前的 Flink 1.4。
