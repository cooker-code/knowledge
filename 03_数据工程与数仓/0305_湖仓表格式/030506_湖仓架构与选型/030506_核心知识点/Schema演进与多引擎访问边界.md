# Schema演进与多引擎访问边界

## 来源
- [数据开发中的引擎如何访问数据湖中的数据？](../文章/done-数据开发中的引擎如何访问数据湖中的数据？.md)
- [详解数据湖的 Schema Evolution(模式演进)](<../文章/done-详解数据湖的 Schema Evolution(模式演进).md>)

## 核心问题

湖表能否被多个引擎安全访问，取决于 Catalog、表格式元数据、引擎 runtime、存储权限和版本兼容。Schema 演进也不是任意改字段，而是要看表格式和引擎对列增删改、嵌套类型、分区列和历史文件的约束。

## 判断准则

| 判断项 | 准则 |
|---|---|
| 引擎访问湖表 | 必须同时满足 Catalog 可见、表格式依赖匹配、存储路径可访问、权限可通过、引擎支持对应表版本 |
| Schema 增删改 | 添加列通常风险较低；删除/改类型/移动列/嵌套类型改动要看表格式能力和下游读者兼容 |
| 分区列 | 分区演进和分区列变更最容易影响查询裁剪、历史数据解释和下游任务 |
| 版本管理 | 任何 Schema 演进结论都要标注表格式和引擎版本，不能从单篇文章泛化 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 有统一元数据就能让所有引擎访问 | 统一元数据只是入口，表格式 runtime、权限、存储协议和版本才决定能不能正确读写 |
| Schema Evolution 等于随便改表 | 每种改动都有兼容边界，分区列、嵌套类型和类型收窄尤其高风险 |
