## 核心问题

湖仓选型不是在“谁取代谁”的口号里做判断，而是看底层存储、开放表格式、计算引擎、Catalog、治理和查询出口分别承担什么职责。

## 判断准则

| 问题 | 判断准则 |
|---|---|
| 开放表格式解决什么 | 它在数据文件之上维护表状态、事务、快照、Schema、分区和版本，不负责通用计算和高并发 BI 查询 |
| Iceberg 倾向 | 开放标准、多引擎分析、快照/分支/标签、审计和生态中立 |
| Hudi 倾向 | Upsert、Timeline、增量查询、流式摄取和表服务 |
| Paimon 倾向 | Flink 原生、主键表、LSM、Changelog、实时更新和流式湖仓 |
| Delta Lake 倾向 | Spark/Databricks 生态、事务日志、Kernel/UniForm 和 Spark 集成 |
| Hive 是否被替代 | 纯 T+1、历史批处理和成熟生态仍可保留；实时更新、快照和多引擎湖表语义才引入新表格式 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 湖仓就是数据湖加数仓 | 湖仓的关键是让对象存储上的文件具备表语义、事务和多引擎访问能力 |
| Paimon 或 Iceberg 有一个终极答案 | 两者优化目标不同，选型要按实时更新、审计、开放生态、Flink 依赖和查询出口拆分 |
| 表格式能替代 OLAP | 表格式解决数据管理语义，复杂查询服务仍需要引擎和服务层优化 |

## 待验证缺口

- 同一 CDC 数据分别写入 Paimon/Iceberg/Hudi/Delta Lake 的 Schema 演进、删除、增量读取和查询延迟。
- 各表格式与 Spark、Flink、Trino、Hive、Doris/StarRocks 的当前版本兼容矩阵。
