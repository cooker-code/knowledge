# AI Native 团队最小必要的 20 份文档：把共识写进仓库

很多团队一聊 AI Native，最后都容易滑向两个极端：

* **靠脑子记**：谁记得多谁有话语权。然后就是同一件事反复吵、反复返工。
* **文档爆炸**：目录像图书馆，真正能“拿来干活”的不到一成。

更靠谱的路子只有一个：先立一套 **最小必要文档（Minimum Docs）**，把那些“真能要命的共识”写进仓库：协作方式、验收口径、安全隐私、成本边界、事故止血。

> 文档不是写给“看起来专业”的，是写给“下次不再踩坑”的。

下面这套 **20 份文档骨架**，不是官方标准，更像工程团队的作战手册底盘：**能跑、能迭代、能审计**。

---

## AI Native 的坑，往往集中在 4 个地方

### 1）目标漂移：今天 Copilot，明天 Agent，后天平台

不是方向不能变，是**方向变了还用旧口径验收**，团队就开始互相伤害。

### 2）验收虚化：只能靠情绪打分

“效果不错”这种话没有复现路径。没有复现路径，就没有工程化。

### 3）风险漏网：安全、隐私、成本失控

这三项一旦翻车，修复成本通常比写功能高一个量级。

### 4）事故手忙脚乱：开会先于止血

没有固定流程时，出问题第一反应是“拉更多人”。人越多，决策越慢，血越喷。

所以这 20 份文档要解决的不是“写得全”，而是四件事对齐：

* **对齐目标**：使命 / 用户 / 需求 / 术语
* **对齐交付**：完成定义 / 测试 / 评审 / 架构 / 协作契约
* **对齐风险**：安全 / 隐私 / 伦理 / 成本 / 应急 / 备份
* **对齐资产**：数据 / 模型 / Prompt / 决策记录

---

## 这 20 份文档怎么分组（按“落地顺序”来）

别一口气写 20 份。那不是体系，是自残。

下面按“先后顺序”分 4 组：

* **第一组：方向与范围（先写 6 份）** —— 先把战场画出来
* **第二组：交付与质量（再写 4 份）** —— 让产出能验收、能复现
* **第三组：核心资产（再补 4 份）** —— 防止关键东西跑丢
* **第四组：风险与韧性（最后 6 份）** —— 让团队能长期跑

---

## 第一组：方向与范围（先写这 6 个，不然越写越散）

> 目标：团队对“做什么 / 为谁做 / 不做什么”有同一幅图。

### 1) mission.md｜使命宣言

写清三件事就够：

* 为什么存在（要解决什么硬问题）
* 成功长什么样（可被观察的结果）
* **明确不做什么（Non-goals）**

要求：**越短越好**，最好 1 页内。

### 2) people.md｜用户画像

不用做“画像大全”。先写：

* 1 个核心用户（谁是主要服务对象）
* 1 个反画像（谁不是目标用户）

目标：团队脑子里出现同一位“具体的人”。

### 3) needs.md｜需求清单（不是 PRD 合集）

这是“当前要打的仗”，不是需求仓库。

必须包含：

* Top 需求（按优先级）
* 明确不做（Non-goals）

### 4) guide.md｜知识地图（新人第一天只看它）

把信息入口写成“导航页”，解决：

* 文档在哪里
* 工具怎么用
* 关键链接/仓库入口

目标：减少“问人 / 翻群聊 / 靠传说”。

### 5) glossary.md｜术语词典（统一口径）

AI 团队最常见的内耗：**同一个词，不同的人脑子里不是一回事**。

高频雷区词示例：Agent / Tool / Skill / Node / DoD / PII。

### 6) protocol.md｜协作契约（把沟通变成可执行）

约定最小流程：

* 怎么提需求
* 怎么请求/调用 Agent（输入输出格式）
* 怎么做变更（谁能改、怎么评审、怎么记录）
* 怎么开会（哪些会必须、哪些会禁止）

目标：把“感觉对”变成“可落地”。

---

## 第二组：交付与质量（让结果可验收、可复现）

> 目标：主分支不靠祈祷，线上不靠运气。

### 7) definition.md｜完成定义（DoD）

这是验收底座。没有 DoD，最后只剩“情绪验收”。

建议至少覆盖：

* 功能完成
* 测试通过
* 安全与隐私检查
* 可观测性（日志/指标/告警）
* 回滚方案
* 相关文档更新

### 8) test.md｜测试标准（关键链路必须测什么）

不必一上来追求全自动化，但要写清楚“必须测什么”。

AI 专项建议至少包含：

* 幻觉一致性 / 关键事实不漂
* Prompt 回归（改提示不把旧功能打碎）
* 安全/越狱测试
* 隐私泄露风险（日志/输出/缓存）

### 9) review.md｜审查规范（最小清单）

把评审从“拍脑袋”变成“过清单”。

覆盖对象建议包括：

* 代码评审
* 方案评审
* Prompt 评审
* Skill/Tool 接口评审

目标：别把错误带进主分支、带进线上。

### 10) design.md｜技术架构（最小架构图 + 决策点）

写清：

* 系统边界与关键链路
* 关键决策与取舍
* 非功能需求：性能/稳定性/成本/安全

建议：先用 Mermaid 画一张最小架构图，后面再慢慢补。

---

## 第三组：核心资产（AI 团队最容易“跑丢”的东西）

> 目标：不让团队的“有效经验”靠口口相传。

### 11) data.md｜数据资产

必须能回答三件事：

* 数据从哪来（来源与授权）
* 怎么用（用途与访问控制）
* 怎么删（生命周期与审计）

建议包含：数据分级 / 数据字典 / 访问与审计策略。

### 12) model.md｜模型卡片（Model Card）

把模型从“许愿机”拉回工程对象。

建议写：

* 能力边界（擅长/不擅长）
* 失败模式（常见翻车姿势）
* 风险点与缓解措施
* 评测与回归策略

### 13) prompt.md｜提示模板（高频任务固化）

把高频任务沉淀为模板：输入、约束、输出格式、拒答规则。

目标：降低“临场发挥”带来的波动。

### 14) decision.md｜决策记录（ADR/决策日志）

记录关键决策，避免同一件事吵三个月。

建议结构：

* 背景
* 选项
* 取舍
* 结论
* 影响范围
* 回滚方式

---

## 第四组：风险与韧性（决定能不能长期跑）

> 目标：把风险的默认值写死，把事故的操作手册准备好。

### 15) budget.md｜成本预算

写清成本项与阈值：

* 模型调用
* 存储
* 观测（日志/trace）
* 第三方工具

目标：**可告警、可止损**，不是月底对账被教育。

### 16) safety.md｜安全护栏

建议明确：

* 禁止输出/禁止执行的范围
* 工具最小权限（least privilege）
* 红队/越狱测试策略
* 关键操作的二次确认/审批

目标：把“默认风险”锁在仓库里。

### 17) privacy.md｜隐私合规

必须写清：

* PII 定义（什么算敏感）
* 收集最小化原则
* 保存周期
* 访问控制
* 日志脱敏

目标：别在日志里把自己送走。

### 18) ethics.md｜伦理准则

不是讲大道理，是处理灰区的“裁判规则”。

建议包含：

* 灰区决策原则
* 争议升级路径（谁拍板、怎么留痕）

### 19) incident.md｜应急手册

建议用最小流程固定节奏：

* 分级标准
* 止血 → 定位 → 修复 → 回归 → 复盘

目标：出事先做正确动作，而不是先找背锅。

### 20) backup.md｜备份策略

写清：

* 备份对象（代码/配置/数据/模型工件）
* 频率
* RPO/RTO
* 恢复演练计划

目标：能恢复才叫备份。

---

## 建议落地顺序：拆成 3 次迭代

### 迭代 1：先把方向钉住（1 天内）

* mission / people / needs / definition

### 迭代 2：把交付变可控（1 周内）

* protocol / review / test / design

### 迭代 3：把风险补齐（2～4 周内持续完善）

* safety / privacy / budget / incident / backup

其余（glossary / data / model / prompt / decision）可以边做边补，但不要缺席。

## 一个能直接抄的 docs-as-code 目录结构

```
ai-native-docs/  
  README.md  
guide.md：知识地图，快速找信息  
glossary.md：术语词典，统一说话口径  
protocol.md：协作契约，规范沟通方式  
mission.md：使命宣言，对齐团队目标  
people.md：用户画像，理解服务对象  
needs.md：需求清单，指导开发工作  
definition.md：完成定义，明确验收标准  
design.md：技术架构，理清系统结构  
data.md：数据资产，管好核心财产  
model.md：模型卡片，了解AI边界  
prompt.md：提示模板，稳定输出质量  
budget.md：成本预算，控制资金花费  
test.md：测试标准，保证交付质量  
review.md：审查规范，防止错误流入  
safety.md：安全护栏，过滤有害内容  
privacy.md：隐私合规，避免法律风险  
ethics.md：伦理准则，守住道德底线  
incident.md：应急手册，快速恢复服务  
backup.md：备份策略，防止数据丢失  
decision.md：决策记录，避免重复犯错
```

建议配套两条底线：

* 文档必须跟代码一样进评审（PR）
* 任何关键变更必须进 decision.md（否则就是“口头修改”）
* 示例

```
# review.md — 审查规范（Review）  
  
## 评审对象  
  
- 代码评审（PR）  
- 需求/方案评审（PRD/Design）  
- Prompt/Skill 评审  
  
## PR 最小检查清单  
  
- 需求是否对齐 `needs.md`  
- DoD 是否满足 `definition.md`  
- 安全/隐私是否触发更新（`safety.md`/`privacy.md`）  
- 是否新增/修改了数据资产（更新 `data.md`）  
- 是否有回滚/应急说明（`incident.md`）  
  
## AI 输出审查  
  
- 有无编造事实/数据  
- 有无泄露敏感信息  
- 输出是否可复现/可验证
```

## 行动清单：照着做就能启动

* **今天**：先写 `mission.md`：一句话目标 + 3 条不做。
* **明天**：补 `definition.md`：验收标准写成清单，别写形容词。
* **本周**：把 `safety / privacy / budget` 写出骨架：先有底线，再谈优化。

文档不是负担，是团队的**重启按钮**。

## 实操示例：这 20 个 md 文件怎么用（usage-example.md）

> 这套文档的关键不是“写完”，而是把它们接到日常动作里：开需求、提 PR、上线、事故、复盘都必须路过。

下面是一个可照抄的从 0 到上线示例：做一个「客服 Agent 自动生成工单摘要」功能。

### 0）一次性准备：仓库结构（docs-as-code）

```
ai-native-docs/  
  guide.md  
  glossary.md  
  protocol.md  
  mission.md  
  people.md  
  needs.md  
  definition.md  
  design.md  
  data.md  
  model.md  
  prompt.md  
  budget.md  
  test.md  
  review.md  
  safety.md  
  privacy.md  
  ethics.md  
  incident.md  
  backup.md  
  decision.md
```

### 1）立项当天：先把方向钉住

* mission.md：一句话目标 + 3 条不做
* people.md：1 个核心用户 + 1 个反画像
* needs.md：Top 需求 + Non-goals
* definition.md：把验收写成清单（允许“未知/需确认”，禁止情绪验收）

### 2）开始干活前：先把协作和质量写成流程

* protocol.md：需求/Agent 请求/变更/会议怎么跑
* design.md：最小架构图 + 失败点标注
* test.md：黄金集回归 + 越狱 + 隐私 + 幻觉边界
* review.md：评审清单（prompt 版本、黄金用例、预算、隐私红线）

### 3）开发过程中：资产别跑丢

* data.md：数据分级/生命周期/审计
* model.md：模型边界/失败模式/回归策略
* prompt.md：prompt 当代码管（版本号 + schema + 示例）
* decision.md：关键取舍留痕（否则下次还吵）

### 4）上线前：把“翻车成本”提前写死

* budget.md：上限 + 告警 + 自动降级策略
* safety.md：禁止输出/最小权限/越狱防护
* privacy.md：脱敏/保存周期/访问控制
* incident.md：第一小时止血流程
* backup.md：prompt/黄金集/配置/数据快照怎么备
* ethics.md：灰区裁判规则（不确定标注、别诱导要隐私）

### 3 个硬钩子（保证文档不吃灰）

1. PR 模板强制勾选：每个 PR 必填“影响哪些 docs（至少 1 个）”。
2. Prompt 变更必须更新 prompt.md + test.md。
3. 每次事故必须在 incident.md + decision.md 留痕。
