# 为什么很多团队开始上 commit 校验？

很多人第一次看到团队里开始加这些东西时，第一反应都差不多：

* `commitlint`
* `husky`
* `lint-staged`
* 各种 pre-commit / commit-msg hook

心里冒出来的通常不是“这套真专业”，而是：

* 又来形式主义了
* 又在限制开发者
* 为什么连提个交都这么麻烦

这种反应很正常。  
因为你第一眼看到的是“规则”，不是“成本”。

很多团队后来真的上 commit 校验，不是因为突然迷上了规范。  
更常见的原因是：**他们已经被那些本来可以更早拦住的问题折腾烦了。**

比如：

* `fix bug`
* `tmp`
* `debug`
* 忘记跑 lint
* 一堆本地就能发现的低级问题，最后全流进了 PR、CI，甚至流进了共享分支

这时候团队想要的，往往不是“更严格”。  
而是**把这些问题往前挡一层**。

## 先记一句：commit 校验，本质上是在前移问题

很多人以为 commit 校验在解决的是：

* 提交 message 漂不漂亮
* 团队规范齐不齐
* Git 历史看起来整不整齐

这些当然会被影响。  
但它真正解决的，往往是另一类更现实的问题：

* 本来本地就能发现的低级错误，为什么还要拖到 CI 才发现
* 明明一眼看不懂的提交，为什么还要让 reviewer 帮忙猜
* 明明只是临时调试状态，为什么要让它流进共享分支

也就是说，  
commit 校验最核心的作用，不是“约束你提交”，  
而是**把本来迟早会爆的问题，尽量拦在更便宜的环节。**

## 为什么很多人会误判它是形式主义

因为你看到的最表面那层，确实像“规则”：

* message 要符合格式
* 不能随便写 `tmp`
* 某些文件要过 lint
* 某些脚本不过就不让提

但团队真正怕的，通常不是“有人没按格式写提交”。  
而是这些问题如果不在本地拦，后面就会在更贵的地方爆出来：

* 在 reviewer 那里爆
* 在 CI 那里爆
* 在共享分支里爆
* 在回退、回补、追问题时爆

所以很多团队上这套，不是因为喜欢多一道门。  
而是因为他们已经被后面那些更贵的环节教育过了。

## 最值得前移拦截的，通常是这 3 类问题

### 1. 模糊提交

比如：

* `fix bug`
* `tmp`
* `debug`
* `final fix`

这些提交最大的问题，不是“看着不专业”。  
而是后面一旦要：

* review
* revert
* cherry-pick
* 追线上问题

大家都会先卡在一句话上：

**这条提交到底是干嘛的？**

如果一个问题本来就能在提交那一刻被拦住，  
那它就不该等到后面让整个团队一起承担。

### 2. 明明本地就能发现的问题

这类问题最不值钱，也最不该后移。

比如：

* 格式问题
* lint 问题
* 最基础的脚本校验不过
* 某些一跑就能发现的低级错误

这些问题如果最后靠 reviewer 提醒、靠 CI 打回、靠别人替你发现，  
那团队其实是在用更贵的资源帮你补本地就该做完的事。

### 3. 会污染共享分支的临时状态

这类问题最容易被低估。

有些提交不是“错”，  
但它们本来就不该进入共享历史：

* 半截改动
* 调试代码
* 只为了本地先过的临时 patch
* 没整理过边界的实验状态

一旦这些内容进入共享分支，  
后面每个人都要为它付成本。

所以很多团队上 commit 校验，  
其实是在给共享分支减噪。

## 本地、commit-msg、CI，到底该各拦什么

很多争议不是“要不要校验”，  
而是“到底该把什么放在哪一层拦”。

一个更健康的分工，通常是：

* 本地 pre-commit：拦最快、最便宜、最确定的问题
* commit-msg：拦明显会污染历史的模糊提交
* CI：拦需要更完整环境才能验证的问题

如果把所有事情都压到 CI，  
那本地前移就失去意义了。

如果反过来，想靠本地脚本替代所有工程判断，  
那它又真的会变成形式主义。

所以重点不是“校验越多越好”，  
而是**让每一层只拦最适合它拦的问题。**

## 什么时候团队真的该考虑上 commit 校验

### 1. 你们总在反复遇到同一类低级问题

如果某些问题已经不是偶发，  
而是在重复消耗 reviewer、CI 和主线质量，  
那就值得往前移。

### 2. 现在这些问题主要靠人肉兜底

只要还在靠：

* reviewer 顺手提醒
* CI 事后打回
* 同事帮忙修历史

那说明这件事还停在昂贵环节里。

### 3. 你们想拦的是噪音，不是判断

最该前移的是：

* 模糊提交
* 本地可发现问题
* 明显不该进共享分支的临时状态

而不是想用规则代替所有工程决策。

## 最后就记一句

很多团队开始上 commit 校验，  
不是因为突然热爱规范，  
而是因为他们终于受够了那些本来可以更早拦住的问题。

如果一个问题本来就能在 commit 那一刻发现，  
那就别把它拖到 reviewer、CI 或共享分支里，  
再让整个团队一起买单。
