过去一年，我用 AI 写东西的方式经历了三个阶段。第一阶段是惊艳：一句话生成一个页面。第二阶段是失望：项目稍微变大，AI 生成的代码就开始“漂移”——它自信地编造不存在的 API，把我之前明确否决的方案又写了一遍，改好了 A 处又坏了 B 处。第三阶段是醒悟：问题往往不在模型不够强，而在我没有把“要做什么”讲清楚。

这个“讲清楚”的载体，就是 **spec（规格说明）**。

vibe coding 在玩具项目上无往不利，但在真实工程里会撞墙。撞墙的根因有个名字，叫**意图漂移**：你脑子里的目标，和 AI 实际在做的事，随着对话变长越走越远。spec 就是用来锁死意图的锚点。

这篇文章想把一件事讲透：在 AI 深度参与的今天，怎么写一份真正有用的 spec，怎么让 AI 帮你写、帮你审，以及什么时候根本不该写。这个框架对产品和开发通用：对产品经理，spec 约等于需求文档（PRD）；对开发者，spec 约等于技术设计文档。我会用 Claude Code 的实际工作流把它们串成一条线。

## 一、先把概念讲清楚：spec 是什么，不是什么

一句话定义：**spec 是把“要做什么、做成什么样算对”写成结构化、可执行的契约。** 它是人与 AI（以及人与人）之间的共识层，而代码，是这个共识的一种可再生产出物。

这正是 2025 年以来兴起的**规格驱动开发（Spec-Driven Development, SDD）**的核心主张：把 spec 当作项目的主产物，代码是从 spec 生成的结果，而不是反过来。SDD 之所以会出现，恰恰是作为对“vibe coding 失败模式”的回应——agent 生成看似合理、却逐渐偏离意图的代码。

关于 spec，有三个常见误解需要先破掉：

* **误解一：spec 就是越详细越好的需求清单。** 不是。越长不等于越清楚，一份 30 页、把实现细节也写死的 spec，人和 AI 都读不动，还会过度约束。好 spec 的标准是“没参加讨论的人也能照着做对”，而不是“字数多”。
* **误解二：spec 是一次写完就冻结的合同。** 不是。spec 是 **living document（活文档）**，它应该随着你和 AI 在过程中的发现而更新——数据模型变了、某个功能砍了，都要回写，让它始终是事实基准。
* **误解三：spec 只是给人看的文档。** 在 AI 时代，spec 首先是给 AI 看的。它现在有了第二个读者——一个没有你公司背景知识、却要据此动手的 agent。这改变了你写它的方式。

记住这个分层就够了：**人持有意图（why），spec 持有契约（what），代码持有实现（how）。** 三者各司其职，是后面一切工作流的基础。

## 二、为什么 AI 时代 spec 反而更重要

很多人的直觉是反的：AI 这么强，是不是可以少写文档、直接让它干？恰恰相反。**模型越强，spec 的边际价值越高，而不是越低。**

第一个原因是**成本结构反转**。我在[《生产力的狂飙与商业模式的静止》](https://mp.weixin.qq.com/s?__biz=Mzg5Mjc0NDA3OA==&mid=2247484052&idx=1&sn=47f50ebcb1a6145af2b7639923a89bfd&scene=21#wechat_redirect)里聊过这个趋势：当“写”的成本被 AI 压到接近零，瓶颈就从“写”移到了“想清楚要写什么”。过去把 80% 的力气花在实现、20% 花在定义；现在比例倒过来了。spec 正是“定义”这件事的产物。

第二个原因是 **vibe coding 的三种典型失败模式**，而 spec 恰好各个对症：

* **漂移**：对话越长，AI 离你最初的目标越远。spec 把目标固定在一个不随对话滑动的地方。
* **幻觉**：AI 自信地调用不存在的 API、编造不存在的字段。spec 里写明的约束和接口是它的护栏。
* **规模衰减**：项目一大，前后实现互相矛盾。spec 提供一个跨会话、跨 agent 共享的一致事实源。

第三个原因更本质：**AI 没有你的常识背景。** 它不知道你们公司三个月前为什么放弃了方案 X，不知道这个字段为什么不能动，不知道“管理员”在你们系统里到底意味着什么权限。这些隐性知识，平时活在老员工的脑子里；当协作者换成一个每次都“失忆”的 agent 时，你必须把它们显性化——**spec 就是你把隐性知识写下来的地方。**

所以 spec 不是 AI 时代的负担，而是**把澄清做一次、然后无限复用的压缩上下文**。同一份 spec，会被你后续每一个 prompt、每一个 subagent、每一次返工反复读取。写一次，省十次。

## 三、一份好 spec 的解剖：六个必备要素

抛开模板差异，一份能让 AI（和人）做对事的 spec，本质上要回答六个问题。这套六要素综合了 SDD 实践对“给 agent 的好 spec”的总结：

1. **结果与动机**：要解决什么问题，为什么现在做，成功长什么样。这是给 AI 的“北极星”，防止它优化错目标。
2. **范围边界**：明确写出**做什么**，更要明确写出**不做什么**。“不做什么”是反幻觉、反范围蔓延最有效的一句话。
3. **约束**：技术栈、性能指标、合规红线、向后兼容、不能碰的东西。AI 默认是乐观的，不写明它就会自作主张。
4. **既有决策与理由**：哪些方案已经定了、为什么。这一节专门用来**防止 AI 推翻你上周刚否决的方案**。
5. **任务拆解**：把工作拆成有依赖关系、有边界、可独立验证的步骤。便于小步执行、便于 review、便于回滚。
6. **验收标准**：怎么算“做完了”，最好每条都能转化成一个测试。没有验收标准的 spec，等于没有终点线。

每个要素缺失，AI 都有对应的、可预测的翻车方式：

| 要素 | 一句话 | 缺了会怎样（AI 的典型翻车） |
| --- | --- | --- |
| 结果与动机 | 要解决什么、为什么 | 优化错目标，做出“能跑但没用”的东西 |
| 范围边界 | 做什么、明确不做什么 | 自行扩张范围，顺手加你没要的功能 |
| 约束 | 技术 / 性能 / 合规红线 | 乐观默认，踩合规或性能的雷 |
| 既有决策 | 为什么这么定 | 推翻你刚否决过的方案，重复返工 |
| 任务拆解 | 有依赖、可独立验证的步骤 | 一次性大改，难 review、难回滚 |
| 验收标准 | 怎么算做完（可测试） | “做完了”无法证伪，扯皮从这里开始 |

你不需要每份 spec 都把六节写满。但每写一份，过一遍这六个问题：**哪一节我留白了？留白的地方，就是 AI 会替我瞎猜的地方。**

## 四、产品 spec 与技术 spec：两条线，一根脊柱

“spec”这个词对产品和开发意味着不同的东西，但它们共享同一根脊柱（就是上面的六要素）。理解这一点，是写好统一框架的关键。

**产品视角（PRD 化）** 关心的是 why 和 what：用户是谁、痛点是什么、为什么现在做、成功用什么指标衡量。它的作用是**防止 AI 优化错目标**——如果你只给 AI 一句“做个导出功能”，它可能做出一个技术上完美、却根本没解决用户问题的东西。

**技术视角（SRS 化）** 关心的是约束下的 what：数据模型、接口契约、状态流转、边界条件、错误路径。它的作用是**让 AI 生成正确的代码**——把它需要的精确信息钉死，减少它瞎猜。

最佳实践，是**在一份文档里融合 PRD 与 SRS 两个视角**：上半部分用 PRD 的笔法交代 why/what（保证不跑偏），下半部分用 SRS 的精度交代约束与验收（保证能落地）。产品经理主笔上半，工程师主笔下半，但它是同一份、同一个事实源。

这里有一条贯穿始终的原则：**描述行为，而不是描述实现。**

* 好的写法（行为化）：“管理员可以查询某用户近 90 天的活动，结果在 2 秒内返回。”
* 差的写法（实现化）：“用 PostgreSQL 给 activity 表加索引。”

后者不是不重要，而是**它属于 plan 阶段，不属于 spec**。spec 锁定“做什么、做对的标准”，把“怎么实现”留给计划和代码——这样 AI 才有空间提出比你更好的实现，你也不会因为过早写死实现而堵死优化空间。

对关键行为，可以借用 **EARS 句式** 来消除歧义，它也是 AWS Kiro 等工具采用的验收写法。基本句型是：

> **WHEN** <触发条件>，**THE SYSTEM SHALL** <明确的响应>。 （可叠加 WHILE <前置状态>、IF <异常> 等前缀。）

例如：“**WHEN** 导出范围超过 90 天，**THE SYSTEM SHALL** 拒绝请求并返回明确的错误提示。”这种句式天然可以对应成测试用例。但提醒一句：**别全文都用 EARS**，它对关键路径很值，对琐碎小改动则是负担，这也是 Kiro 在小改动上被诟病“仪式感过重”的原因。

## 五、实战工作流：brainstorm → spec → plan → implement → review

这是全文最核心的一节。下面用 Claude Code 把五个阶段串成一条真实可走的线。把它记成一句话：**先对齐，再定契约，再排计划，再小步落地，最后对标验收——每一步都回写 spec。**当然，你也可以用其他AI Agent工具来实践这个方法，比如Codex、Cursor、Copilot等。

**阶段 0 · Brainstorm（头脑风暴）。** 别一上来就让 AI 写代码，甚至别急着写 spec。先发散、对齐问题定义。在 Claude Code 里，这对应让它**先提澄清问题**而不是直接动手。一个好用的开场 prompt 是：“在动手之前，先列出你需要我澄清的 5-8 个问题，尤其是范围边界和验收标准上的歧义。”这一步挡掉的返工，往往比后面所有优化加起来都多。产出：一份你和 AI 都认可的问题陈述。

**阶段 1 · Spec（定契约）。** 把对齐的结论落成结构化文档（就是第三节的六要素）。可以让 AI 起草初稿，但**人要逐条定稿**。把它存进仓库（比如 `specs/` 目录）并纳入版本管理——spec 进了 git，它才会被当回事、才会被维护。产出： `spec.md`。

**阶段 2 · Plan（排计划）。** 让 spec 映射到实现计划。Claude Code 的 **plan mode（计划模式）** 正是为此设计：它先以只读方式调研代码库，产出一份分步计划，**等你确认后才动代码**。这一步回答“怎么实现”：要改哪些文件、分几步、有什么风险、在哪里测。计划仍然由人审——这是你拦截“AI 想偏了”的最后一道便宜关卡（改计划比改代码便宜得多）。产出：实现计划 / 任务清单。

**阶段 3 · Implement（落地）。** 按计划小步执行。两个杠杆：其一，用 **subagent 并行**处理互相独立的任务，各自有独立上下文，互不干扰；其二，用 **TDD（测试驱动开发）**，先把第五节里的验收标准写成测试——**测试就是 spec 验收标准的代码化身**，它替你自动盯着实现别跑偏。每一步都可验证、可回滚。

**阶段 4 · Review（对标验收）。** spec 是 review 的标尺。 `/code-review` 帮你扫 diff 找 bug，但更重要的动作是**拿着 spec 的验收清单逐条核对**：每一条 acceptance，实现到了吗？测试覆盖了吗？一旦发现实现偏离了意图，要么修代码，要么——如果是当初 spec 想错了——回写 spec。

把五个阶段对到 Claude Code 的具体动作上：

| 阶段 | 目标 | 在 Claude Code 里怎么做 | 产出物 |
| --- | --- | --- | --- |
| 0 Brainstorm | 对齐问题定义 | 让它先提澄清问题，别急着写代码 | 一致的问题陈述 |
| 1 Spec | 锁定契约 | AI 起草 + 人逐条定稿，存进 `specs/` | `spec.md` |
| 2 Plan | 映射实现 | 切到 plan mode：只读调研 → 出计划 → 人确认 | 实现计划 / 任务清单 |
| 3 Implement | 小步落地 | 按计划执行，subagent 并行独立任务，TDD 先写测试 | 代码 + 测试 |
| 4 Review | 对标验收 | `/code-review` 看 diff + 对照 spec 验收逐条核 | review 结论、回写 spec |

下面是一份**最小可用的 spec 模板**（注意它六要素齐全，关键验收用了 EARS）：

```
1. # Spec：用户活动导出

3. ## 1. 背景与目标（Why）
4. 管理员需要排查异常登录，但当前只能逐条翻日志，无法按时间范围批量导出。
5. 目标：让管理员自助导出指定用户近 N 天的活动记录，减少找工程师的次数。

7. ## 2. 范围
8. 做：
9. -按 user_id +时间范围（≤90天）导出活动为 CSV
10. -仅管理员角色可用

12. 不做（Out of scope）：
13. -导出 PDF /Excel
14. -跨租户导出
15. -实时流式导出（本期一次性生成）

17. ## 3. 约束
18. -复用现有 RBAC，不新增权限模型
19. -单次导出≤50万行；超过则报错并提示收窄范围
20. -90天上限受合规要求约束，不可放宽
21. -必须写入现有审计日志（谁导出了谁的数据要可追溯）

23. ## 4. 既有决策
24. -本期同步生成，不引入新的后台任务队列
25. （2026-05已否决异步方案，因运维成本，后续再评估）

27. ## 5. 行为与验收
28. - WHEN 管理员提交合法 user_id 与≤90天范围，THE SYSTEM SHALL
29. 返回 CSV 下载，含字段：timestamp, action, ip, user_agent
30. - WHEN 范围>90天，THE SYSTEM SHALL 拒绝并返回明确错误信息
31. - WHEN 结果>50万行，THE SYSTEM SHALL 拒绝并提示收窄范围
32. - WHEN 非管理员调用，THE SYSTEM SHALL 返回403
33. -每次成功导出，SHALL 写入一条审计日志

35. ## 6. 任务拆解
36. 1.复用权限校验中间件+403路径
37. 2.查询层：user_id +时间范围，加结果行数预估
38. 3. CSV 序列化+下载响应
39. 4.审计日志埋点
40. 5.错误路径（超范围/超行数）
41. 6.测试：覆盖第5节每一条验收
```

这份 spec 不到一屏，但任何一个没参加讨论的工程师或 agent 拿到它，都能不来问你就把事做对——这就是 spec 该达到的标准。

## 六、用 AI 写 spec：三种分工与四条禁区

让 AI 参与写 spec，有三种分工模式，按场景选：

1. **AI 起草、人定稿。** 适合你已经想清楚、只想要快。你口述意图，AI 整理成结构化 spec，你逐条改定。
2. **人起草骨架、AI 补全。** 适合复杂功能。你定六要素的骨架和关键决策，让 AI 补全你容易漏的边界条件、失败路径、验收用例。
3. **对话式共建（推荐）** 你给意图，AI 反问澄清，逐轮收敛。这是最不容易出错的方式，因为它把“澄清”做在了写之前。

贯穿三种模式的核心技巧只有一句：**让 AI 提问，而不是让 AI 假设。** AI 的默认行为是“先把空缺脑补满、再交一份看起来很完整的东西”。你要反着来，逼它先暴露不确定性。前面那句“先列出你需要我澄清的 5-8 个问题”就是干这个的。

另一个关键动作是**喂背景**。AI 写不好 spec，多半不是因为笨，而是因为不知道你的项目约束和历史决策。把代码库、约束、既有决策喂给它——这正是 `CLAUDE.md` / `AGENTS.md` 这类**项目级 agent 上下文文件**的用途：把恒定的事实和红线固化下来，让每次会话都自带这份背景，而不必你每次重复。

然后是**四条禁区**，越界就等着返工：

1. **不让 AI 替你做产品决策。** 取舍、优先级、为谁牺牲谁，是人的责任。AI 可以列选项、分析利弊，但拍板的是你。
2. **不让 AI 凭空编验收标准。** 它会写得很漂亮，但可能脱离真实业务规则。验收标准必须由懂业务的人确认。
3. **不让 AI 假设约束。** 合规、性能、兼容这些它不知道的红线，你不写明，它就会乐观默认成“没限制”。
4. **警惕“假装完整”。** AI 极擅长生成**结构完整、内容空洞**的 spec——六节都在，看着像那么回事，一推敲全是占位和废话。

验收 spec 本身，用一个简单的提问：**“把这份 spec 给一个没参加讨论的人或 agent，他能不能不来问我、就把事情做对？”** 如果答案是“不能”，缺的那部分就是你还没写清楚的地方。

## 七、用 AI review spec：把它当对手，而不是啦啦队

写完 spec，最有价值的一步是让 AI 审一遍——但默认状态下，AI 是个啦啦队，会顺着你夸。**你必须显式地把它切换成对手。**

review 的标准维度清单（让 AI 逐项过）：

* **可测试性**：每条验收能否写成一个明确的测试？哪条是模糊的？
* **范围歧义**：哪些“做 / 不做”的边界不清，容易被实现者自行脑补？
* **隐藏假设**：有没有依赖未写明的前提（数据已存在、某服务可用、某权限已有）？
* **内部矛盾**：约束之间、验收之间有没有打架？
* **缺失路径**：错误、空值、超限、并发、权限不足这些失败路径，是否都有交代？
* **Over-spec**：有没有把“怎么实现”（具体库、表结构）写进了本该只描述“做什么”的地方？

更进一步，**用多视角分离 review**，对应 Claude Code 里开多个 subagent 各司其职：

* **产品视角**：会不会优化错目标？验收标准对得上用户真实痛点吗？
* **工程视角**：可实现吗？约束之间有没有冲突？依赖现实吗？
* **QA 视角**：每一条验收，能不能落成一个测试用例？

三个独立 agent 各审一遍，比让一个 agent 连审三遍更能发现盲点——因为视角不同，盯的东西不同。

一个可复用的 review prompt 模板：

```
1. 你是一名严格的 spec 审查者。目标：找出这份 spec 在交给一个
2. 没参加讨论的工程师/ agent 实现时，会导致做错的地方。
3. 不要夸奖，只列问题。逐项检查：

5. 1.可测试性：每条验收能否写成一个明确的测试？哪条是模糊的？
6. 2.范围歧义：哪些“做/不做”边界不清，会被实现者自行脑补？
7. 3.隐藏假设：是否依赖未写明的前提（数据已存在、服务可用、权限已有）？
8. 4.内部矛盾：约束之间、验收之间有没有冲突？
9. 5.缺失路径：错误/空值/超限/并发/权限不足是否都有交代？
10. 6.Over-spec：有没有把“怎么实现”写进了本该只描述“做什么”的地方？

12. 输出：按以上6类分组，每个问题给出
13. 「位置+为什么有风险+建议补什么」。
14. 最后给一个“还需向作者澄清的问题清单”。
```

可以更狠一点：**让两个 agent 对抗**——一个负责写、改 spec，另一个专门找漏洞，来回迭代，直到找不出新的问题为止。这种对抗式审查能逼出单轮 review 漏掉的边界。

但记住最后一句：**AI review 是放大镜，不是裁判。** 它把问题放大给你看，但改不改、怎么改，是人的判断。

## 八、什么时候不该写 spec：成本、收益与反模式

把 spec 当银弹是另一种错。spec 有真实成本，不是所有事都值得为它付费。

**判断法则：如果“想清楚”的时间 < “做错重来”的时间，就别写 spec，直接做。** 一行 bug 修复、改个文案、一次你还不知道想要什么的探索性 spike——这些场景下 spec 是纯负担。规格驱动开发的收益，主要在**中大型功能、绿地项目、多人 / 多 agent 协作**时才回本；小改动上，它的仪式感会拖慢你。

常见的反模式踩坑：

1. **spec 写成小说。** 长不等于清楚。又臭又长的 spec，人和 AI 都读不动，反而稀释了关键约束。
2. **提前过度设计。** 把还没验证的需求写死，逼自己和 AI 去实现一个可能根本不需要的东西。
3. **僵尸 spec。** spec 与代码脱节、没人回写，时间一长，它比没有更误导，因为后来的人或者 AI 会相信一份已经过时的“事实”。
4. **拿 spec 走流程，而非对齐。** 为了交差而写，写完没人看。spec 的唯一目的是对齐和复用，不是仪式。
5. **把 spec 当不可改的合同。** 它该随认知更新。死守一份写错了的 spec，比没有 spec 更糟。
6. **让 AI 全权代笔且不审。** 等于把方向盘交给一个没有常识背景的实习生,它会很自信地把你带沟里。

一句话总结：**spec 是为了对齐和复用，不是为了好看，也不是为了交差。** 不服务于这两个目的的 spec，就不该写，浪费时间。

## 结语：spec 是你和 AI 之间的合同

兜回开头那句话：AI 让“怎么做”变得越来越便宜，于是“做什么、做对没有”成了唯一还稀缺的东西。spec，就是这件稀缺工作的产物。

写 spec 表面上是文档工作，本质上是**思考工作**。它逼你在写代码之前先想清楚——而这件事，AI 代替不了你。因为它不知道你的业务、你的用户、你那些由真实约束逼出来的取舍。它可以帮你起草、补错、挑刺、把问题放大给你看；但做决策的，始终是你。

所以别把 AI 当成那个完全替你写 spec 的人，把它当成那个**逼你把 spec 写清楚的人**。你负责意图，它负责实现；你持有 why，它执行 how；中间那份白纸黑字的 what，就是你和 AI 之间的合同。

合同写清楚了，剩下的，才真的可以放心交给 AI。
