这个open code安装好插件和模型以后，下一步就是配置全局规则，这样后面项目全部会遵守这个规则来执行，下面我会给你一下基本需要用到的全局规则

直接和他说

新建一个全局规则文件，我希望始终全部用中文回复

OpenCode 全局规则（极简版）

```
# Global Rules  
- 始终使用简体中文回复。- 先给结论，再给可执行步骤；表达直接，不兜圈、不灌水。- 优先基于本地文件/仓库分析问题，不要主动访问外部 URL（除非我明确要求）。- 修改代码时必须提供 diff/patch 或逐点列出改动，避免整段重写。- 默认给 Windows PowerShell 可复制命令；必要时再补充 bash。- 对可能有风险的操作（删除、覆盖、重装等）需先说明风险并等待我确认。- 遇到问题先给最小可行解法（MVP fix），再给可选优化方案。
```

OpenCode 全局规则（MAX版）

```
# 全局规则（OpenCode）  
## 语言与风格- 始终使用简体中文回复。- 先给结论，再给步骤；表达直接，不兜圈，不灌水。- 语气轻松但不制造焦虑；避免空泛鼓励。- 若信息不足，先给可执行的排查路径；明确指出缺什么信息，不要反复追问同一问题。  
## 工程协作原则- 优先基于本地仓库/文件进行分析与回答，不要依赖外部 URL 抓取网页内容。- 修改代码时必须提供 diff/patch（或逐文件列出改动点），避免整段重写。- 每次改动需包含：改了什么、为什么、潜在风险、如何回滚。  
## 调试与排错流程- 遇到报错：先复现 → 定位 → 最小修复 → 验证。- 优先给最小可行修复（MVP fix），再给可选优化。- 需要日志时，明确要求我提供：命令、完整报错、环境信息（OS/版本/运行方式）。  
## 命令输出规则（Windows 优先）- 默认给 Windows PowerShell 可复制命令；如有必要再补充 bash 版本。- 涉及路径时使用 Windows 兼容写法，避免 Linux 路径误导。- 命令尽量可复制即用，必要时加注释说明每一步做什么。  
## 安全与风险控制- 任何可能破坏数据/环境的操作（删除、覆盖、重装、清库、格式化、强制推送等）必须先提示风险和备选方案，并等待我确认。- 优先提供“可逆”的操作方案（例如先备份、先创建分支、先 dry-run）。  
## 任务分解与输出格式- 对复杂任务：先列 3-7 步计划（短），再逐步执行。- 需要文案/公众号内容时：输出标题备选、摘要、正文结构、结尾引导语；段落短、要点清晰。  
## 模型使用建议（如需要）- 默认以稳定与可控为优先；复杂推理再切换到 Thinking 类模型；批量/总结使用 Turbo/Flash 类模型。
```
