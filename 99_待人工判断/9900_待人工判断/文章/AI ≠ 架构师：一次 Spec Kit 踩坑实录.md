* 项目类型：运维类的 app
* 前端架构：打算混合 app 的开发，也就是以 web（vue3） 为主，app 原生功能为辅

## MiroThinker

2026年1月5日，MiroMind联合清华大学发布搜索智能体模型MiroThinker 1.5。

它是一款超强的搜索智能体模型，至少在同类AI模型中可以说是当下最强了。

它不是那种“秒回型 AI”，而是真的会花几分钟做研究的模型。它会搜各种信息，打开几十个网页，阅读几万字的 PDF，梳理来龙去脉，最后给你一份带引用、带数据、带逻辑的完整研报。

更令人震惊的是，凭借这种“慢思考”机制，它仅用 30B 的参数量，就在深度任务上击败了庞大的 `GPT-5-High` 模型。

你可以把它理解为：**一个会主动做文献综述的搜索智能体** 。

它的行为模式大概是：

1. 拆解问题背景
2. 主动全网搜索
3. 优先查阅顶级机构/权威来源
4. 打开、阅读、比对多篇资料
5. 交叉验证信息
6. 最后才输出结论

**而不是：**

搜8条 → 编一段（总结） → 直接给答案

Spec Kit 的核心哲学：先把"做什么"和"为什么"想透，再考虑"怎么做"。

## 1. 先搞清楚：Spec Kit 在重构里的定位

这次重构 app 的尝试，我打算深度用`Speckit`来试下。

对于我的场景，可以简单理解为：

* **旧系统**：现有 app 项目代码（技术栈：Ionic3+angular4）
* **Speckit 能帮你做的事**：

1. 把“要重构什么、为什么要重构”写成结构化的需求规格（spec.md）
2. 把“系统应该长什么样、用什么技术栈、如何拆服务/模块”写成架构设计与技术方案（plan.md）
3. 自动帮你把重构拆成一堆可执行任务（tasks.md）
4. 在这个过程中，**spec.md + plan.md + constitution.md**就相当于你的「系统架构规格说明书」，而且是**可演进、可执行**的，而不是一份死文档。

关键心智模型：**“先有规格，再有架构；先有架构，再分任务；最后才动代码。”**

## 2. 现有 app 中初始化 Speckit

### 2.1  环境准备

在你的开发环境（比如重构这套 app 的 repo 根目录）保证有：

* Python 3.11+
* Git
* `<font style="color:rgb(2, 6, 23);background-color:rgba(175, 184, 193, 0.2);">uv</font>` 包管理器（Python 的轻量工具）

> 环境部分，不做过多介绍，这里不是重点

### 2.2 在「现有 app 项目」里初始化 speckit

这里要解释一下，为啥我要在原有的旧 app 项目里面创建新的项目，因为这是重构 app，不是全新的 app，旧 app 项目有很多业务逻辑，我是需要 ai 去理解，方便我在新的 app 上执行的。

进入我的 app 项目里面，创建一个新的文件夹：`newapp`

```
cd newapp  
# 推荐：持久化安装（使用 uv）  
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git  
  
# 验证安装  
specify check  
  
# 初始化项目，在 newapp上创建，使用 Claude  
specify init . --ai claude --force
```

初始化时，会问你：

* 选择 AI 助手（Claude Code / Copilot / Cursor / Gemini CLI 等）
* 选择脚本类型（Bash / PowerShell）

初始化完成后，你的项目里会多出这些东西：

```
.specify/  
  memory/  
    constitution.md      # 项目宪法（架构/编码原则）  
  scripts/               # 执行 /speckit.* 命令的脚本  
  
specs/                   # 后面所有规格、架构、任务文档都会在这里分编号放  
CLAUDE.md 或其他 AI 配置
```

## 2.3 建立「系统级架构约束」

对「重构项目」来说，**先固化架构原则，再谈怎么重构**，否则 AI 会写回老一套。

在 Claude Code 打开当前项目，输入：

```
/speckit.constitution  
创建一份适用于本 app 重构的项目宪法，重点包括：  
- 技术栈规范（如：前端 前端框架、构建工具、包管理器、状态管理、路由管理、UI组件库、HTTP客户端、类型系统、代码规范）约束  
- 分层架构原则（如：表现层（iews, components）、业务逻辑层（services, store）、数据层（ model, types）、基础设施层（utils, http））  
- 模块边界（如： 每个子模块应具备完整的业务闭环、模块间通过标准接口通信、避免模块间的直接依赖）  
- 编码规范（命名风格、异常处理方式、日志规范， TypeScript 类型定义规范、 Vue3 组件开发规范）  
- 命名规范（如：文件命名、目录命名、变量命名）  
- 测试要求  
- 性能与安全基线（如：接口响应时间、鉴权方式、敏感信息加密）  
- 代码组织原则（如： 单一职责、 依赖注入、配置驱动）  
- 性能优化规范（如： 代码分割、懒加载、缓存策略）  
请用清晰条款形式写入 constitution.md。
```

看到上面的思考过程和下面产出的宪法文件，我死心了，我的预期太高了，或者是我步骤做反了。

我的理解状态是：

1. 它会询问我用什么技术栈，然后一步步询问我大概用什么技术，帮我去理清楚我到底要如何开始，帮我梳理清楚我的思绪。

**现实是：**

没想到它简单粗暴就直接给我查当前目录文件，因为我当前目录是空的，它又查我当前目录的上层目录，由于上层目录是旧 app，它就开始分析项目，最终产生一个不伦不类的东西，即不是旧 app 的宪法，又不完全是我想要的技术栈。

最终我得出了结论：`<font style="color:#DF2A3F;">Speckit</font>`不适合在空白项目立宪法（规矩）！！！

## 3. 总结

# 那么，到底如何输出系统架构规格说明书？

既然 Spec Kit 不行，直接问大模型也不靠谱，那正确的做法是什么？

基于这次踩坑经验，我总结出一套**人机协作的架构设计流程**：

## 第一步：深度调研（人工主导 + AI 辅助）

**不要直接问 AI"我该用什么技术"，而是先自己做深度调研。**

1. 使用专业搜索工具（如前面提到的 MiroThinker）进行技术调研
2. 去**官网**查看最新版本的发布说明和 Breaking Changes
3. 查看技术社区的实践案例和讨论（GitHub Issues、Stack Overflow、技术博客）

**AI 在这个阶段的作用：**

* 帮你快速了解某个技术领域的主流方案
* 提供你可能没考虑到的技术选项
* 总结和对比不同方案的优缺点

## 第二步：确定技术栈（人工决策）

**这是最关键的一步——必须由你自己拍板。**

根据调研结果，结合以下因素确定最终方案：

* 团队技术储备和学习成本
* 项目规模和复杂度
* 未来扩展性和维护性
* 社区活跃度和生态完善度
* **去官网确认每个依赖的最新稳定版本**

> 💡 这一步 AI 可以帮你检查方案是否有明显漏洞，但**不能替代你的决策**。

## 第三步：生成架构文档（AI 执行）

技术栈确定后，这时候可以让 AI 发挥作用了：

```
# 在 Claude Code 中使用 Plan Mode  
/speckit.spec  
基于以下技术栈生成系统架构规格说明书：  
- 前端框架：Vue 3.5.x  
- 构建工具：Vite 6.x  
- 混合开发框架：Capacitor 6.x  
...
```

**关键点：**

* 你提供具体的技术栈和版本号
* AI 帮你生成规范的架构文档
* 你负责审核和调整

## 第四步：人工审核（必须！）

**AI 生成的架构文档，必须经过人工审核。**

重点检查：

* ✅ 技术栈版本是否正确
* ✅ 模块划分是否合理
* ✅ 接口定义是否清晰
* ✅ 是否有遗漏的关键配置
* ✅ 是否符合团队编码规范

## 核心原则

**记住：AI 是执行者，不是决策者。**

| 阶段 | 主导者 | AI 的角色 |
| --- | --- | --- |
| 技术调研 | 人工 | 研究助手，提供选项 |
| 方案决策 | 人工 | ❌ 不参与决策 |
| 文档生成 | AI | 执行者，按你的要求输出 |
| 方案审核 | 人工 | ❌ 不参与审核 |

只有这样输出的"系统架构规格说明书"，才是可靠、可落地的。

# 写到最后：AI ≠ 架构师

这次踩坑之旅，让我对 AI 在软件架构领域的边界有了更清醒的认知。

## AI 的三大先天局限

**1. 知识时效性鸿沟**

AI 模型的训练数据永远有截止日期，今天的技术栈版本，明天可能就过时了。就像我测试中发现的那样，各大模型给出的技术方案往往落后好几代——这不是 AI 不够聪明，而是它根本看不到最新的技术演进。

**2. 搜索深度不足**

普通大模型的"联网搜索"本质上只是"浅层查询"。搜索几个网站 → 快速总结 → 给出答案，这种模式对简单的信息查询够用，但面对需要深度调研的架构选型时，就显得力不从心。

架构决策需要纵向深挖技术细节、横向对比多个方案、追踪社区实践案例——这些都需要"深度研究"能力，而非简单的"信息汇总"。

**3. 缺乏实战经验判断**

架构师的价值不在于知道"有哪些技术"，而在于知道"为什么选这个技术"。

团队技能储备、历史遗留问题、未来扩展预期、维护成本权衡……这些隐性因素，AI 看不到，也无法理解。

## 结语

Spec Kit 和各类大模型都很好，但它们是**工具**，不是**建筑师**。

软件架构的核心，永远是**人对业务的理解、对技术的判断、对团队的洞察**。AI 可以帮你做调研、写文档、查漏洞，但最终拍板的那个人，必须是你自己。

这次实践让我明白了一个道理：**AI 能让架构师更高效，但无法替代架构师。**
