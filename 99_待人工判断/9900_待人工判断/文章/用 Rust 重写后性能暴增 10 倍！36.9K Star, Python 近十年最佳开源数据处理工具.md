#### 正文

2008 年, 一个名叫 Wes McKinney 的开发者在一个资产管理公司工作，他当时正在为公司开发一款数据处理工具，用于满足对财务数据高性能，灵活性的需要。2009 年，Wes McKinney 离职时，说服管理人员允许他把这个代码库进行开源,  这个库就是后来的 Pandas。

Pandas 的易用性, 灵活性成功让 Pandas 成为了数据科学领域的霸主。但是仍然有一朵乌云悄悄潜伏在 Pandas 上方，那就是性能问题。

Pandas 和 Numpy 不同的是，Numpy 主要是基于 C 构建的, 而 Pandas 虽然构建在 Numpy 之上, 但主要的开发代码仍然是 Python, 而提到 Python, 你很难不忽略 GIL 锁的存在，虽然 Pandas 使用了诸多措施来尝试提高性能，比如核心模块使用 Cython 或 C 编写，引入 Pyarrow 作为新的数据存储后端。

在 Wes McKinney 一篇博客(Apache Arrow and the “10 Things I Ha\*t\* About pandas) 中, Wes McKinney 分享了在创建 Pandas 中间的一些趣事，Wes McKinney 当时主要在夜晚和周末开发 Pandas, 作为一个软件工程的新人, Wes McKinney 选择边学边写, 直到 2013 年, Pandas 开源 4 年后，Wes McKinney 才开始认真学习 C 和 C++ 相关的开发知识。因此除了 Python 本身对代码的影响，Wes McKinney 也部分承认了 Pandas 早期引入了某些技术架构缺陷, 比如 Pandas 有一个经典的实践:

**如果处理 1G 的数据集，电脑需要额外准备 5G 到 10G 的内存。**

我这里也搜集了一些资料，大概有这几个原因。

第一个是: 当 Pandas 处理字符串或者混合数据时，由于 NumPy 无法使用原生的数字类型存储，因此会直接转换成 Python 的 Object 类型，当 CPU 需要处理这部分数据时，每次都需要读取指针，并且再由 Python 解释器处理解包这个 Object 为指定的 Python 对象，效率低下。

第二个则是 BlockManager，导致在数据操作频繁的时候, 会产生比较多的内存碎片，而碎片整理操作又会消耗过多的内存。

第三个则是复制的问题，在 Pandas 2.0 如果不开启 Cow(写时复制) 的话，当有些情况时，操作数据时并不是在原有的数据上修改，Pandas 会复制一份数据出来，导致内存开销和时间浪费(这就是 `SettingWithCopyWarning` 的来源)。

当然这些问题大多数都会在不知道何时才会发布的 Pandas 3.0 中得到一定程度的解决，但是社区显然等不了这么久了。于是 2020 年，Ritchie Vink 为了解决 Pandas 的问题使用 Rust 开发了 Polars，不过和 Pandas 主要由非盈利基金会支持不同,  Polars 则拿到了数千万美元的投资，因此 Polars 的推进速度很快。

当然更快的，还是 Polars 的性能，和 Pandas 不同，Polars 的核心处理逻辑全部使用 Rust 开发，同时 Polars 支持使用流式处理查询，因此可以在个人电脑上轻松处理数百G大小的数据集，在性能上也有明显优势。

在 PDS-H benchmark 最新的测试结果中，Polars 相较于 Pandas 有接近 37 倍性能的提升。

注: 该数据集的大小为 10GB 的 CSV 文件，运行在 96C192G 配置的测试服务器中。

当数据集大小为 100G 时，Pandas 已经退出了比赛，**不过我们实测小数据集下性能大概是几倍的差距, 不会有这么大差别。**

我也在本地写了一个基础的 benchmark 用于比较 polars 和 pandas 的性能，使用 4 列  1000w 行数据 和 2000万行数据两种情况, 分别运行 filter，groupby, join 和 sort 操作。

虽然没有达到 PDS-H benchmark 宣传的 30 多倍的提升, 不过推测可能是我的测试数据集太小的原因。不过也可以看到，随着数据集的变大，polars 性能的优势越来越明显。

但是这并不意味着 pandas 就是那个不值得被选择的那一个, 相较于 polars，pandas 更简单的语法，更丰富的资料和教程也是一个明显的优势，如果你不需要处理那么大的数据集的话，那么 pandas 仍然是一个很好的选择，如果你需要处理大规模数据集的话，那么 polars 则会是那个 python 数据分析的新选择。
