## 1引言：AI 能画图，不只是写代码

在 FPGA 开发中，硬件架构框图一直扮演着"设计输入"的角色 —— 先画好框图，再照着写 RTL代码。好的框图能直接映射出高质量的硬件代码，这也是为什么有经验的硬件工程师常说："把图画对了，代码就对了一半。"

然而现实是什么呢？

手绘架构图耗时耗力，常常在代码变更后未被及时更新

随着设计迭代，框图和代码之间的一致性难以保证，最终沦为误导性的历史文档。

当 AI 开始介入硬件设计流程，框图的角色出现了两种演化路径：

**路径 A** 人画框图，AI 写代码。保留设计者的控制力，但更新框图仍需手动操作，耗时耗力却未必准确。

**路径 B** AI 同时生成代码和框图。框图不再是设计输入，而是变成可视化验证输出。每次代码变更后，AI 自动重新生成框图，保证图和代码的一致性。设计者通过审查框图来理解和验证 AI 的实现结果，有利于及时发现性能瓶颈，指导 AI 进行下一轮迭代。

本文分享路径 B 的实践经验：如何让 AI 在 FPGA 开发中自动生成四类硬件架构图。

## 2四类硬件架构图

AI 分析算法的数据流结构后，能自动生成四类互补的架构图。每类图回答不同层次的设计问题。

### 系统拓扑图 模块边界与数据流

系统拓扑图展示模块划分、数据流方向和控制信号路由。实线表示样本数据流，虚线表示异步控制信号。

AI生成的系统拓扑图：模块流水线、数据流（实线）和控制信号（虚线）

这类图最直观地回答"系统架构对不对"这个问题。模块边界的划定、控制信号的扇出方式、数据流的拓扑结构 —— 这些决定了90%的最终性能。一张好的拓扑图能清晰反映出过度分解、缺失的并行路径或不合理的控制信号路由等关键设计问题。

### DATAFLOW 分解图 拆开才能并行

当一个模块只需要处理数据流中的一小部分样本时（计算稀疏度 > 2），数据处理放进主循环顺序执行会成为性能瓶颈。解决方案是将数据处理部分移出主循环进行单独设计，将数据流拆分为 EXTRACT / COMPUTE / APPLY 三个阶段，各阶段形成流水线并行工作。

三阶段EXTRACT/COMPUTE/APPLY流水线分解，标注FIFO深度和存储类型

DATAFLOW 分解图清楚地展示了每个阶段的职责边界：提取阶段负责偏移量感知的样本选择，计算阶段处理密集运算，应用阶段将结果写回数据流。FIFO 深度和存储类型的标注让资源预算一目了然。

### FSM 状态转移图 异步控制的安全网

当模块需要等待异步到达的控制信号时，状态机是唯一安全的实现方式。它确保模块在控制信号到达前持续处理数据而不阻塞流水线。

异步控制信号处理的状态机：非阻塞轮询、循环缓冲区、数据就绪后转入提取

状态转移图揭示了控制流的完整生命周期：等待状态如何轮询控制输入，缓冲区如何在等待期间积累数据，各状态之间的转移条件是什么。这类图对于诊断死锁和反压问题特别有效 —— 如果状态机缺少某个超时退出路径，框图上一眼就能看出来。

### RTL 微结构图 寄存器级的电路性能瓶颈诊断

第四类是最精细的 RTL 微结构图。它深入到模块内部，展示 FPGA 原语（DSP、BRAM、移位寄存器）、位宽标注和流水线级数。这是诊断逻辑电路性能瓶颈最直接的工具。

AI生成的RTL微结构图：三阶段DATAFLOW、FPGA原语、位宽标注、跨阶段旁路连接

拿一次实际项目来说，微结构图显示某个模块使用了全展开的移位寄存器存储历史样本，导致布线拥塞。切换到循环 BRAM 后，布线压力消失，时序余量从负转正。这种直观的电路结构图呈现，相对于阅读 HLS C++代码更加直接有效。

> 框图让物理层面的问题在逻辑设计阶段就变得可见。

## 3两种渲染方案：Excalidraw JSON vs AI 生成 SVG

AI 生成架构图有两条技术路径，各有适用场景。以下用同一个 FIR 滤波模块的微结构图展示两种方案的视觉差异。

### Excalidraw JSON Diagram-as-Code

Excalidraw 是一个开源的白板绘图工具，支持 JSON 格式描述图形元素。AI 只需用 Write 工具输出一段 JSON 文件，就能生成完整的架构图 —— 不需要任何可执行文件或图形库。

一个最小的矩形元素示例：

> {  
>   "type": "rectangle",  
>   "x": 100, "y": 100,  
>   "width": 200, "height": 60,  
>   "strokeColor": "#1e1e1e",  
>   "backgroundColor": "#a5d8ff",  
>   "roundness": { "type": 3 },  
>   "boundElements": [  
>     { "type": "text", "id": "label\_1" }  
>   ]  
> }

**Excalidraw 的优势**  
纯 JSON 输出，不依赖渲染工具链。生成后可在 Excalidraw Web App 中交互编辑，修改 JSON 即可更新框图，迭代速度很快。

**Excalidraw 的局限**  
CLI 渲染不会自动居中容器内的文本 —— 需要显式计算文本位置。箭头路由是近似的，复杂交叉场景下可能出现重叠。

Excalidraw JSON渲染的微结构图 -- 手绘风格，可交互编辑

### AI 生成像素级精确的 SVG 图

对于需要像素级控制的微结构图，AI 直接生成 SVG 标记是更好的选择。AI 输出完整的 SVG 标记语言，利用贝塞尔曲线精确控制箭头路径，用 CSS 样式定义 FPGA 组件的视觉层次。

**SVG 的优势**  
箭头路由完全可控（贝塞尔曲线，无近似），CSS 样式实现 FPGA 组件的一致性视觉语言，渲染结果所见即所得。

**SVG 的局限**  
标记更冗长（一张微结构图通常150-300 行 SVG），生成后不可交互编辑。

AI生成SVG渲染的同一模块 -- 像素级精确的箭头路由和CSS样式

### 选型决策表

| 场景 | 推荐方案 | 原因 |
| --- | --- | --- |
| 系统拓扑图 | Excalidraw JSON | 元素少，布局规则，交互编辑便利 |
| DATAFLOW分解图 | Excalidraw JSON | 流水线结构，标准布局模式 |
| FSM状态转移图 | Excalidraw JSON | 状态数有限，自环箭头用JSON足够 |
| RTL微结构图 | AI生成SVG | 复杂箭头路由，需要像素级精确 |
| 跨stage旁路连接 | AI生成SVG | 贝塞尔绕行避免遮挡 |
| 高扇出控制信号 | AI生成SVG | 箭头密集交叉，需精确间距控制 |

## 4工作流概览

从算法到 FPGA 实现，整个流程分为四个阶段，每个阶段都伴随着对应框图的自动生成。

**分解阶段**  
AI 分析算法的数据流依赖，划定模块边界和拓扑，自动输出系统拓扑图。这是最关键的阶段：模块边界和数据流拓扑决定了 90%的最终性能。

**生成阶段**  
将优化后的算法描述转换为 HLS C++代码，同步生成模块架构图、DATAFLOW 分解图（如有）和状态转移图（如有）。框图与代码保持一对一映射。

**优化阶段**  
将浮点实现转为定点，迭代优化资源使用。每轮优化后重新生成框图，更新位宽标注和资源消耗。框图成为迭代反馈的载体 —— AI 通过检查自己生成的框图来发现瓶颈。

**集成阶段**  
连接所有模块为完整系统，输出系统集成框图（标注 FIFO 深度和数据通路）。自动化验证把关每个阶段 —— 仿真未通过，就无法进入下一阶段。

核心理念是框图即迭代反馈。AI 生成框图，检查框图，发现瓶颈，重新生成代码，重新生成框图。这个闭环让框图从一次性的设计输入变成了持续更新的验证输出。

## 5关键经验总结

**框图从设计输入变为验证输出**  
在 AI 驱动的流程中，框图的核心价值是帮助设计者理解和验证生成的代码，而非指导编码。每次代码变更后自动重新生成，消除"过时框图"的顽疾。

**四类图回答不同层次的问题**  
系统拓扑图回答"边界对不对"，DATAFLOW 图回答"并行度够不够"，FSM 图回答"控制流安全不安全"，微结构图回答"瓶颈在哪里"。

**两条渲染路径互补**  
Excalidraw JSON 适合结构化布局（拓扑、流水线、状态机），AI 生成 SVG 适合需要像素级精确的微结构图。选型取决于箭头路由复杂度。

**微结构图的独特价值**  
它能暴露代码审查难以发现的物理层面反模式 —— 布线拥塞、存储瓶颈、关键路径过长 —— 让这些问题在逻辑设计阶段就变得可见。

**框图驱动迭代**  
AI 检查自身生成的框图来发现设计缺陷。这种"自我审查"机制是纯代码流程无法提供的。
