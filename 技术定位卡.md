# 技术定位卡模板

本文件定义三类输出：

- 二级类目 `AGENTS.md`：某个问题域的认知入口和排重入口。
- `AGENTS.md`：具体技术的纵向知识地图。
- `核心知识点/*.md`：从有价值文章中提取出的核心知识点。

分类和边界判断统一引用 `目录划分.md`。

用户认知水平和内容取舍统一引用 `用户画像.md`。

## 排重准则

问题指纹：

```text
技术本体 + 所属模块 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 技术本体相同 | 继续比较模块和机制，不直接判重 |
| 解决问题相同 | 如果没有新增边界、反例、证据或实践动作，则合并 |
| 只多了案例 | 案例不改变准则时，只追加相关原文 |
| 多了新机制 | 可以新建核心知识点 |
| 多了观点冲突 | 必须记录为认知校准点 |

## 实践门槛

只有阅读投入建议为 `实践` 时必须填写。其他文章可以写“不适用”。

| 门槛 | 判断 | 证据 |
|---|---|---|
| 可运行 | 是 / 否 | 最小例子、命令、配置、SQL、代码路径 |
| 可验证 | 是 / 否 | 输入、输出、指标、对照组、验收标准 |
| 可排障 | 是 / 否 | 失败模式、日志信号、定位路径 |
| 可迁移 | 是 / 否 | 能否迁移到用户已有场景 |
| 结论 | 实践 / 降为精读 | 如果任一关键门槛不满足，不能判为实践 |

## 跨域判断

如果一个技术可被多个领域使用，必须明确“本体”和“场景”的区别。

| 问题 | 判断 |
|---|---|
| 它本体属于哪里 | 按 `目录划分.md` 的归类优先级判断 |
| 这篇文章为什么可能跨域 | 例如 Airflow 被用于训练流水线、Flink 被用于实时特征、Redis 被用于缓存或状态 |
| 当前文章主问题是否改变分类 | 按 `目录划分.md` 的跨域示例和边界规则判断 |
| 应避免的误归类 | 不因标题出现模型、Agent、数据、BI 等词就直接归类 |

## 全量路由报告模板

全量路由是进入正式沉淀前的控制面，不是最终知识内容。

| 模块 | 必须回答 |
|---|---|
| 样本/全量范围 | 本次处理了多少篇、覆盖哪些目录、是否排除重复 |
| 分类分布 | 原目录分布、最终类目分布、未定数量 |
| 阅读投入分布 | 跳过、略读、精读、实践数量和判定风险 |
| 冲突点分布 | 原目录冲突、图片缺失、标题降权、低置信分类、排重冲突 |
| 技术栈认知变化 | 哪些技术 level 需要上调、下调或保持 draft |
| 正式沉淀候选 | 候选文章、所属技术、为什么值得写入 |
| 不进入正式沉淀 | 被跳过或合并的原因 |
| 下一步 | 先补目录、先补技术 index、先补图、还是先写核心知识点 |
