# 智能数据分析助手

## 技术定位

| 项 | 内容 |
|---|---|
| 技术名 | 智能数据分析助手 |
| 一级类目 | 数据分析与 BI |
| 二级类目 | 语义层与智能问数 |
| 三级节点状态 | 新增已创建 |
| 技术本体 | 用对话、代码执行、数据连接、报告生成和可视化能力，把业务问题转成可检查的数据分析过程 |
| 全局架构位置 | 位于业务问题和底层数据查询之间，连接语义层、分析工作区、执行沙箱和报告输出 |
| 主要使用者 | 分析师、业务人员、数据产品、数据平台工程 |
| 主要产出 | 分析结论、查询过程、可视化图表、报告、可复盘的分析工作区 |

## 节点边界

| 判断项 | 归入本节点 | 不归入本节点 |
|---|---|---|
| 主问题 | AI 如何完成或辅助数据分析、问数、可视化和报告生成 | 模型本身能力评测、通用 Agent 框架、纯 AI 编程工具体验 |
| 技术焦点 | 数据连接、语义约束、执行校验、分析链路、结果解释 | 只讲 prompt、只讲模型榜单、只讲通用代码生成 |
| 产出 | 可追责的数据分析过程或业务分析结论 | 无数据口径、无校验链路的工具宣传 |

## 核心模块

| 模块 | 职责 | 重点问题 |
|---|---|---|
| 问题澄清 | 把业务问题转成可执行的指标、维度、时间窗口和口径 | 是否暴露不确定性，是否要求用户补充约束 |
| 数据连接 | 连接表、文件、指标平台或 BI 数据集 | 是否有权限、口径、血缘和审计边界 |
| 执行工作区 | 运行 SQL、Python、表格计算或可视化变换 | 是否可复盘，是否隔离高风险执行 |
| 结果解释 | 生成图表、报告和业务结论 | 是否区分事实、推断和建议 |
| 质量控制 | 校验数据、代码、SQL、图表和结论 | 是否有基线、反例、人工复核和失败信号 |

## 横向对标

| 对标对象 | 对标点 | 优势 | 劣势 | 使用判断 |
|---|---|---|---|---|
| Text-to-SQL | 都把自然语言转成数据查询 | 本节点覆盖更完整的分析工作流 | 如果只讨论 SQL 生成与 Schema Linking，应回到 Text-to-SQL | 先看文章主问题是查询生成还是分析闭环 |
| 传统 BI | 都服务业务自助分析 | 对话式入口更灵活，可生成解释和报告 | 口径治理、权限和稳定性通常弱于成熟 BI | 稳定监控仍优先 BI，探索分析可用助手 |
| AI 编程工具 | 都能执行代码和生成报告 | 本节点关注数据分析结果和可追责链路 | 通用 coding agent 容易缺少业务语义层 | 如果文章主角是 Claude Code、Qwen Code 或 Skill 机制，优先归 Agent 与 AI 工程 |
