# 050101_Text-to-SQL知识地图

## 模块定位

| 项 | 内容 |
|---|---|
| 所属目录 | 05_数据分析与BI / 01_语义层与智能问数 / Text-to-SQL |
| 解决的问题 | 先定义 `Text-to-SQL` 的应用流程，再把文章吸收到流程节点 |
| 典型输入 | 本地文章、已有 AGENTS、已有核心知识点、后续新增资料 |
| 典型产出 | 流程节点知识、判断准则、风险边界、待补证据 |
| 用户阅读目标 | 读到一篇文章时，先知道它优化哪个环节，再决定是否吸收 |

## 宏观流程

| 节点 | 节点问题 | 关键知识点 | 文章吸收规则 |
|---|---|---|---|
| T00 问题理解 | 自然语言问题如何识别意图、指标、维度和时间条件 | Text-to-SQL 入口先做业务语义理解，不直接拼 SQL | 业务问题、问数意图文章进入 |
| T01 语义层与口径 | 指标口径、同义词、权限和可问范围如何约束 | 没有语义层的 Text-to-SQL 容易生成能跑但口径错的 SQL | 语义层、指标平台文章进入 |
| T02 Schema Linking | 问题中的实体如何链接到库表、字段、Join 路径和分区 | Schema Linking 是降低幻觉和错表错字段的关键环节 | Schema Linking、元数据、表字段文章进入 |
| T03 SQL 生成 | 如何选择模板、Few-shot、Agent、约束解码或模型生成 SQL | 生成策略必须受语义层、权限和方言约束 | Prompt、Agent、生成策略文章进入 |
| T04 执行校验 | SQL 如何做语法、权限、结果、空值、口径和安全校验 | 问数系统必须能验证结果并暴露不确定性 | 校验、执行、审计文章进入 |
| T05 解释与追责 | 如何解释 SQL、引用表字段、记录版本并支持人工修正 | 可追责比一次回答更重要 | 解释、日志、反馈闭环文章进入 |

## 文章进入时的对比规则

| 判断项 | 处理方式 |
|---|---|
| 补充 | 文章提供新的模块边界、失败模式、实现步骤、验证指标或工程权衡时，写入对应流程节点 |
| 冲突 | 文章观点和当前节点准则不一致时，先记录冲突原因，不直接覆盖已有沉淀 |
| 不同层次 | 入门教程、源码机制、生产实践、产品发布分属不同层次，必须标注层次后再吸收 |
| 降权 | 只有标题党、工具清单、无证据收益、泛泛最佳实践时，不进入核心知识点 |

## 使用方式

- 先读本文件的宏观流程，再读具体主题知识点。
- 新文章不要单独写摘要，先归入流程节点，再判断补充、冲突、不同层次或降权。
- 如果某个节点已有多篇文章，后续应拆成更细的 `050101_核心知识点/<主题>.md`，而不是继续扩展文章列表。
