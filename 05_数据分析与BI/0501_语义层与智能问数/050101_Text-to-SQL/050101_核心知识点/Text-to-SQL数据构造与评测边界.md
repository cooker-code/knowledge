## 核心问题

Text-to-SQL 的评测和训练不能只看模型榜单分数。高质量问数系统需要可执行、可校验、覆盖真实业务结构的数据，并能把歧义、错误标注、大小写、数据类型、空结果和复杂 SQL 表达都纳入训练与验收。

## 判断准则

| 环节 | 准则 |
|---|---|
| 数据构造 | 问题、SQL、Schema、样例值和业务口径要能互相对齐 |
| 数据清洗 | 大小写、空结果、字段缺失、类型错误和错误标注要先修复 |
| 数据多样性 | 覆盖 Join、聚合、嵌套、窗口函数、常识约束和业务同义词 |
| 歧义澄清 | 遇到“最大”“最新”“活跃”等模糊词，先向用户澄清，不直接生成 SQL |
| 示例选择 | Few-shot 示例要按结构和语义相似性选择，不是随机塞样例 |
| 评测指标 | 执行准确率、测试套件准确率、SQL 有效性和业务口径正确性要分开看 |
| 线上验收 | 需要沙箱执行、错误分类、人工复核样例和失败案例回灌 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 模型越强，Text-to-SQL 越自然变好 | 数据质量、Schema 表达、可执行验证和歧义澄清同样决定上限 |
| Benchmark 分数高就能进业务 | 真实业务有口径、权限、脏数据、表关系和历史兼容问题 |
| 合成数据越多越好 | 合成数据必须经过执行验证和分布检查，否则会放大错误模式 |
| Prompt 能解决歧义 | 歧义需要交互澄清或约束选择，不能靠模型猜 |

## 待验证缺口

- 需要用一个真实业务指标问题构建小型评测集，比较“只塞 DDL”“加样例值”“加语义层”“加歧义澄清”的差异。
- 需要记录失败 SQL 的错误类型：错表、错列、错 Join、错聚合、错时间窗口、权限越界、结果空但未解释。
