# A/B测试进阶：当随机化与观察单位不一致，你该怎么办？

## 引言

A/B测试，作为因果推断的黄金标准，通过随机化处理，让我们能自信地判断某个改变（如新功能、新广告）是否真的带来了预期的效果（如收入提升、用户满意度增加）。然而，在实际操作中，我们常常会遇到一个棘手的问题：**处理分配的单位和我们观察效果的单位不一致**。

举个例子，我们可能按“地区”来推送一个新功能，但观察的却是每个“用户”的反应；或者我们按“品牌”来调整价格，但衡量的是每个“商品”的销量。当这种情况发生时，来自同一个组（如同一地区、同一品牌）的观察数据就不再是相互独立的了。这种内在的相关性，如果我们不加以处理，会严重影响我们对实验结果的判断，尤其是标准误差、置信区间和p值都可能出现偏差。

本文将深入探讨这个问题，并介绍一种强大的解决方案：**聚类稳健标准误差（Cluster-Robust Standard Errors）**，帮助你更准确地进行A/B测试的统计推断。

## 核心问题：客户、订单与被低估的标准误差

想象一个电商平台的场景：为了提升销售额，你打算在用户结账时，展示一个“相关商品”的轮播图。为了验证这个想法，你决定进行A/B测试。

理论上，你可以为“每一笔订单”随机决定是否显示轮播图。但这样做会给用户带来混乱的体验（同一个用户有时看到轮播图，有时又看不到），很可能损害业务。因此，一个更合理的做法是在“客户”层面进行随机化：一部分客户（处理组）在他们所有的订单中都会看到轮播图，而另一部分客户（对照组）则永远看不到。

* • **随机化单位**：客户 (Customer)
* • **观察单位**：订单 (Order)

问题来了：虽然处理是针对“客户”的，但我们衡量效果的数据（如收入）是记录在“订单”上的。这就导致了随机化单位和观察单位的不一致。来自同一个客户的所有订单，都属于同一个处理组，它们之间不再是独立的。

### 一次错误的显著性判断

我们用模拟数据来看看这个问题有多严重。假设我们有100位客户的3000个订单数据。

当我们使用传统的线性回归（OLS）来分析轮播图对收入的影响时，结果可能会显示：轮播图平均使每笔订单收入减少了1.63欧元，并且p值为0.005，在1%的水平上统计显著。

Linear regression summary table

看到这个结果，你可能会立即决定下线这个功能。但请等一下！这个结论可靠吗？

由于我们是在客户层面进行的随机化，同一个客户的订单是相关的。而传统的OLS回归假设所有观察值（订单）都是独立的。这个错误的假设，导致计算出的**标准误差被严重低估**了。

### 如何发现问题？

在现实世界中，我们无法像在模拟中一样，通过多次重新随机化来观察真实的标准误差。但我们有两种方法可以检测并修正这个问题：

1. 1. **自举法 (Bootstrapping)**
   自举法的核心思想是通过重复抽样来模拟“另一个平行宇宙”中的实验结果。关键在于，我们的抽样单位必须和随机化单位保持一致。在这个案例中，我们应该**对“客户”进行有放回的抽样**，而不是对“订单”。

   具体操作是：

   通过这种方式，我们能发现，真实的标准差远比OLS报告的要大，从而避免错误的显著性判断。

* • 从客户列表中随机抽取与原始数量相同的客户（允许重复）。
* • 将被抽中客户的所有订单数据打包，形成一个新的“自举样本”。
* • 在这个新样本上重新计算处理效应。
* • 重复上千次后，计算这上千个处理效应估计值的标准差，这个标准差就是更接近真实情况的“自举标准误差”。

2. 2. **聚合数据 (Aggregating Data)**
   另一个更直接的方法是将数据聚合到随机化单位的层面。

   这种方法的缺点是会损失信息，并且由于样本量从订单数减少到客户数，统计功效可能会降低。

* • **如何操作**：不分析每一笔订单，而是计算**每个客户**的总收入或平均订单收入。
* • **新的数据集**：每一行代表一个客户，包含客户ID、是否在处理组以及该客户的总/平均收入。
* • **分析**：在这个新的、以客户为单位的数据集上，你可以直接使用标准的t检验或OLS回归。因为现在分析单位和随机化单位一致了，客户之间是独立的，标准统计方法重新变得有效。

这两种方法都能帮助我们识别并初步应对单位不一致的问题，但它们都各有局限。在下一部分，我们将详细介绍一种更通用、更强大的解决方案——聚类稳健标准误差，并深入其背后的数学原理。
