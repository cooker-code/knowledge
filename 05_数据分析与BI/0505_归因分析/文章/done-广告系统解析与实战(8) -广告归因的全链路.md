# 前言

在前述章节 [广告系统解析与实战(7) -广告归因的常见类型](https://mp.weixin.qq.com/s?__biz=MzI1NDQ5MDY4Mw==&mid=2247484498&idx=1&sn=7addccc3b34c97b941e530b9a45446c2&scene=21#wechat_redirect)  中，我们介绍了3种归因模型，分别是：

* **单触点归因模型**

  ：只把 `转化` 归功到`单个触点`。常见的有：末次点击归因模型 (Last-Click Attribution)、首次注册归因(First-Register Attribution)等。
* **多触点归因模型**

  ：把 `转化` 归功到`多个触点`，认为转化链路上的多个触点都有功劳。常见的有：线性归因模型 (Linear Attribution)、时间衰减归因模型 (Time Decay Attribution)、位置归因模型 (Position-Based / U-Shaped Attribution) 等。
* **高级归因模型**

  ：把 `转化` 通过智能算法归功到`多个触点`，合理分配触点的贡献。常见的为：数据驱动归因模型 (Data-Driven Attribution, DDA)。

在`用户`、`媒体平台`、`广告主` 三者组成的广告系统中，这些归因模型是 **广告主** 将 **转化效果** 归功到 **多个媒体平台给他下发的触点数据** 的方式。

举个例子：用户1 在A、B、C三个平台依次产生了广告点击行为，且用户1最后发生了转化行为。于是，广告主知道用户1发生了转化（平台可能不知道），各个平台又将用户的点击行为下发给广告主，此时广告主就需要按照 **上述的归因模型**，将转化功劳进行分配。那假设按照 广告主 **按 last-click 的归因模型**，广告主只会将 用户1的转化 回传给平台C，并认为平台A和B是没有功劳的，在平台A和B的广告费用都白花了。

在上面的例子中可以看到，归因模型决定了“功劳”的分配，但这背后引出了一系列更深层次的技术问题：

* 平台A、B、C是如何将“用户的点击行为下发给广告主”的？这个“下发的数据”究竟是什么？
* 广告主又是如何精确地知道，这次转化应该“回传给平台C”，而不是A或B？他凭借什么信物来和平台C进行沟通？
* 平台C收到回传数据后，如何确认这确实是源于自己平台的一次点击？
* 最终，这些归因数据又将如何影响未来的广告投放？

本章将深入这些“幕后”环节，从一次广告曝光开始，完整地拆解归因链路上的每一个技术步骤，揭示数据是如何在平台与广告主之间流动、匹配并最终创造价值的。

# **1. 流量采集与唯一ID的诞生**

一切始于用户的行为。当“用户1”打开App或网页时，App或网页内会预留广告位。

1. **广告请求与投放**

   ：用户在使用App或网页浏览内容时，就同时会向广告平台（例如平台A、B、C）发起广告请求。平台收到请求后，会进行众多广告的排序和竞价，竞价胜出的广告将被下发到用户1的设备上进行展示。
2. **生成唯一ID并记录日志**

   ：在广告曝光或被点击的瞬间，平台会为这次行为生成一个**全局唯一ID**，并记录到自己的日志系统中。这是未来一切归因的起点和“凭证”。

* **点击ID (`click_id`)**

  ：当用户1点击广告时生成，这是最重要、最精准的归因凭证。平台A、B、C会各自为用户1的点击行为生成**不同的`click_id`**。

这条日志详细记录了：“在什么时间，哪个广告活动，通过哪个媒体，触达了哪个用户（或设备）”。

# **2. 点击下发：click\_id的传递**

当用户1点击了平台C的广告后，平台C需要将“凭证”传递给广告主。

1. **URL动态拼接**

   ：平台C不会让用户1直接跳转到广告主的落地页。它会进行一次重定向（Redirect），在这个过程中，将上一步生成的`click_id`作为参数，动态地拼接到广告主设置的落地页URL后面。

* **示例**

  ：广告主设置的原始落地页是 `https://www.advertiser.com/product`。
* 用户1在平台C点击后，实际跳转的URL会变成：`https://www.advertiser.com/product?click_id=platform_C_UNIQUE_ID_12345`

2. **广告主接收并存储`click_id`**

   ：广告主的网站或App必须有相应的技术能力来捕获并存储这个`click_id`。通常会将其与当前用户会话（Session）或用户身份绑定，并存储在浏览器的Cookie、Local Storage或App的本地存储空间中。

如果用户1之前也点击了平台A和B的广告，广告主的系统可能会记录下多个平台的`click_id`，但通常会以后来的为准（这取决于广告主自身的技术逻辑，如果是last-click的归因逻辑，那就是最后一个click\_id为准）。

# **3. 广告主自归因（可选）：转化与click\_id的配对**

用户1在广告主的网站或App内继续浏览，并最终完成了购买（转化）。

1. **用户转化**

   ：用户1完成了购买行为。
2. **关联转化与`click_id`**

   ：在用户1完成购买的那一刻，广告主的服务器会查找之前为该用户存储的`click_id`。由于平台C是最后一次点击，其`click_id` (`platform_C_UNIQUE_ID_12345`) 覆盖了之前的ID或被记录为最新的ID。广告主在自己的数据库中创建一条记录，将**这次购买行为**与这个 **特定的`click_id`** 关联起来。

* **记录示例**

  ：“用户1，在2025年8月18日完成了购买，该用户对应的`click_id`是`platform_C_UNIQUE_ID_12345`。”

此时，广告主根据自己设定的“末次点击”归因模型，做出了内部判断：这次转化归功于`click_id`为`platform_C_UNIQUE_ID_12345`的这次点击。

**值得注意的是：并不是所有的广告主都会进行自归因**，有可能有的广告主没有这个技术实力，选择将所有转化一股脑回传给所有平台，这个时候需要后续的“平台归因”对数据进行清洗处理。

# **4. 转化回传：将功劳告知平台**

现在，广告主需要将这个归因结果报告给平台C。

1. **服务器到服务器（Server-to-Server, S2S）API**

   ：这是最可靠的方式。广告主的服务器直接调用平台C提供的API接口，发送一个包含转化信息的数据包。

* **核心内容**

  ：这个数据包里**必须包含**之前获取的`click_id=platform_C_UNIQUE_ID_12345`，以及转化类型（如`purchase`）、转化价值、转化时间等。

2. **基于像素或SDK（Pixel/JS SDK）**

   ：另一种方式是在“支付成功”页嵌入平台C的追踪代码。当用户1到达该页面时，代码自动执行，从Cookie中读取`click_id`并直接发送给平台C。

因为广告主遵循末次点击模型，所以他**只会向平台C回传这个转化事件**，而不会通知平台A和B。

# **5. 平台归因：最终确认与拼接**

平台C接收到广告主回传的、带有`click_id`的转化数据后，开始进行最终的归因确认。

1. **数据匹配（Matching）**

   ：平台C在其庞大的点击日志系统中，通过`platform_C_UNIQUE_ID_12345`这个关键“钥匙”，精确地找到了最初用户1的那条点击记录。
2. **信息拼接**

   ：一旦匹配成功，平台C就能将转化信息与点击信息完整地拼接起来，形成一条完整的归因链路，清晰地知道是哪个广告活动、哪个素材带来了这次购买。

# **6. 报表展示与OCPX应用：数据的最终价值**

归因完成后，数据就从原始日志变成了有商业价值的洞察，并被应用于两个核心下游场景：

1. **报表展示（Reporting）**

   ：广告主可以在平台C的后台看到：“您的XX广告活动，花费XX元，带来了1次购买转化，转化成本XX元，ROAS为XX”。而在平台A和B的后台，则只看到了点击和花费，没有转化。
2. **OCPX策略应用**

   ：这是归因数据最重要的应用，也是**后续我们将谈到的模型训练样本来源**。平台C的OCPX模型（如oCPA）接收到了这次正向反馈。它学习到“带来这次成功转化的点击，具有什么样的用户特征、媒体环境、出价时机”。未来，当遇到类似特征的流量时，系统会更愿意为之出价，从而帮助广告主找到更多像用户1这样的高价值客户，实现“在成本可控的范围内，最大化转化量”的目标。

# **总结**

本篇文章主要详细讲述了归因的全链路，总结一下：

* **归因链路的上游**

  ：包括“流量采集”、“点击下发”、“广告主自归因”（可选、不一定有）、“转化上报”。
* **归因链路的下游**

  ：包括“平台归因”、“报表展示及OCPX应用”（平台归因好的数据会进行后续的模型样本训练）

而在整个归因过程中，归因匹配的核心是**使用一个能在“广告平台”和“广告主”之间共同识别的唯一标识符**。

* **最核心的匹配字段：`click_id`（点击ID）**： 这是 **点击归因（Click-Through Attribution, CTA）** 的黄金标准。它由平台为单次点击生成，唯一且准确。整个匹配过程就是平台用广告主回传的`click_id`，去自己的点击日志库里查找对应记录。
* **其他次要的匹配字段：** 在没有click\_id的情况下（如浏览归因 View-Through Attribution，VTA），则会依赖其他次要标识符进行概率性匹配，例如设备ID（IDFA/GAID）、Cookie（Web端，效力减弱）或IP+User Agent的组合指纹。

最后，除了本文主要介绍的“点击归因”，有的广告主可能还会使用“激活归因”、“注册归因”、“播放归因”等等，其原理基本相同，只是传递的信物有差异而已，这里就不再赘述。
