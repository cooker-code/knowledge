# 指标体系
## 知识点入口

- 本模块先看宏观流程，再看文章：[知识地图](050200_核心知识点/知识地图.md)。
- 新文章必须先归入流程节点，再判断是补充、冲突、不同层次还是降权。
- `文章/` 只保留原文锚点，长期知识必须沉淀到 `050200_核心知识点/`。


## 类目定位

| 项 | 内容 |
|---|---|
| 一级类目 | 数据分析与 BI |
| 二级类目 | 指标体系 |
| 核心问题 | 指标如何被定义、分层、归责，并最终连接到业务动作 |
| 不解决什么 | 不把所有报表展示、数据建模或经营复盘都塞进指标体系；这些只在影响指标口径时引用 |
| 用户当前认知假设 | L3：概念基础已知，重点需要沉淀口径冲突、业务动作、指标平台边界和可复用设计准则 |

## 知识点组织

| 知识点类型 | 应沉淀什么 | 来源吸收方式 |
|---|---|---|
| 北极星指标 | 选择条件、拆解路径、反例 | 多篇文章按业务阶段和适用边界合并 |
| 指标分层 | 结果指标、过程指标、诊断指标、行动指标 | 案例只在形成拆解准则时进入知识点 |
| 口径治理 | 指标定义、统计周期、过滤条件、责任人 | 冲突口径要写明采用规则和降权原因 |
| 指标平台 | 指标生产、复用、权限、血缘、服务化 | 与数据工程平台文章交叉时，只吸收 BI 决策相关内容 |

## 排重准则

问题指纹：

```text
指标对象 + 口径/分层/动作/平台模块 + 解决的业务判断问题 + 适用边界 + 认知增量
```

| 判断项 | 排重规则 |
|---|---|
| 都讲指标体系建设 | 按是否补充动作闭环、口径治理、分层方法或平台边界判断 |
| 只是指标清单 | 不进入核心知识点 |
| 只是某公司案例 | 案例不改变准则时只作为例子 |
| 与经营分析重叠 | 指标定义放这里，业务复盘和行动闭环可双向引用 |
