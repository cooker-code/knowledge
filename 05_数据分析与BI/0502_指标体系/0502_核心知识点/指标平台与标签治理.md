## 核心问题

指标平台和标签体系要把指标定义、加工、服务、权限、血缘、变更和效果评估变成可治理资产，而不是做一个更漂亮的取数入口。

## 认知校准点

| 校准点 | 处理 |
|---|---|
| 平台不是 BI 包装 | 只提供取数或看板不等于指标平台 |
| 指标和标签都需要生命周期 | 创建、审批、变更、下线、回滚、效果评估缺一不可 |
| 数据虚拟化或大模型能力只是增强项 | 不能替代口径治理、权限和血缘 |
| 标签必须评估业务效果 | 只做画像展示不说明触达、转化或成本改善时降权 |

## 判断准则

| 模块 | 应检查什么 |
|---|---|
| 指标定义 | 是否有业务口径、技术口径、负责人、版本和适用场景 |
| 指标加工 | 是否能追踪上游表、计算任务、依赖和质量规则 |
| 指标服务 | 是否支持 API、BI、问数、订阅和权限控制 |
| 标签治理 | 是否定义来源、更新频率、覆盖率、稳定性和效果指标 |
| 变更管理 | 是否能评估影响面并通知下游 |
| 平台价值 | 是否把传统数仓资产升级成指标、语义层、服务接口和业务消费闭环 |

## 待验证缺口

- 指标平台与语义层、Text-to-SQL 的边界还需要真实系统案例。
- 标签效果评估需要沉淀覆盖率、准确率、稳定性、转化提升和成本收益的最小指标集。
