## 核心问题

指标体系解决的不是“有哪些指标”，而是业务目标如何被拆成可观测、可归责、可行动、可复盘的指标结构。

## 认知校准点

| 校准点 | 处理 |
|---|---|
| 指标多不等于体系强 | 没有目标、责任人和动作的指标只是报表字段 |
| 指标正确不等于可用 | 缺少时间窗口、分子分母、统计粒度、归属规则时无法跨团队复用 |
| 北极星指标不能孤立使用 | 必须拆出过程指标和护栏指标，否则会诱导单点优化 |
| 指标体系和 SQL 写法不是一层问题 | SQL 题解已迁入 SQL 书写目录，指标目录只保留口径与业务动作 |

## 判断准则

| 场景 | 准则 |
|---|---|
| 设计新指标 | 先写业务问题、动作对象、分子分母、时间窗口、统计粒度和责任归属 |
| 做指标树 | 从目标到策略到动作再到指标，不从现有表字段反推指标 |
| 选北极星指标 | 要能代表长期价值、可被团队影响、能拆到过程杠杆，并设置护栏 |
| 治理口径冲突 | 先列出使用场景和影响范围，再统一定义、版本、血缘和回归样例 |
| 衡量数仓改造 | 改造收益也要指标化：覆盖数据质量、查询效率、复用率、口径一致性和下游影响 |

## 适用边界

| 不适用场景 | 原因 |
|---|---|
| 临时探索分析 | 可以先宽松定义，但不能沉淀为正式指标 |
| 纯技术 SQL 题 | 主问题是表达模式，不是指标治理 |
| 只讲 KPI 口号 | 缺少口径、动作和复盘证据 |
