# 数据分析与 BI
## 知识点入口

- 本模块先看宏观流程，再看文章：[流程化知识点总览](knowledge/05_数据分析与BI/核心知识点/流程化知识点总览.md)。
- 新文章必须先归入流程节点，再判断是补充、冲突、不同层次还是降权。
- `文章/` 只保留原文锚点，长期知识必须沉淀到 `核心知识点/`。


## 目录目的

本目录不是文章归档区，而是“读文章时更快理解文章”的认知架构。

核心目标：

1. 一篇文章可以贡献多个知识点，不同知识点可以引用同一篇文章的不同内容。
2. 一个知识点可以吸收多篇文章，需要判断新内容是补充、冲突，还是更高/更低层次的理解。
3. `核心知识点/` 中必须写清具体内容，不能只放文章链接；要说明文章如何开发、修正或补充这个知识点。
4. 正式知识库不维护文章来源，文章只作为来源锚点存在于知识点页中；原文内容已经抽离后，不在本目录保存文章副本。

## 层级结构

```text
05_数据分析与BI/
  AGENTS.md
  二级问题域/
    AGENTS.md
    核心知识点/
      具体知识点.md
    具体技术或方法/
      AGENTS.md
      核心知识点/
        具体知识点.md
```

| 层级 | 作用 | 不应该做什么 |
|---|---|---|
| 一级目录 | 说明数据分析与 BI 的整体认知结构 | 不保存文章清单 |
| 二级目录 | 说明一个问题域的边界、知识点类型和排重准则 | 不直接堆文章摘要 |
| 具体技术或方法 | 维护某个技术/方法的纵向知识地图 | 不替代核心知识点正文 |
| `核心知识点/` | 沉淀可复用知识点，可持续吸收多篇文章 | 不只引用文章，不全文搬运 |

## 文章到知识点的关系

| 关系 | 处理方式 |
|---|---|
| 一篇文章对应多个知识点 | 拆开吸收，每个知识点只引用与自身相关的段落或图 |
| 多篇文章补充同一知识点 | 合并到同一知识点页，在“来源贡献”里标明各自贡献 |
| 多篇文章观点冲突 | 不新增平行摘要，先写入“冲突与判断”，说明采用、降权或待验证 |
| 文章只是案例 | 如果不改变准则，只作为来源锚点或例子，不单独建文件 |
| 文章只是资讯/营销 | 不进入正式知识点，必要时只在后续追查中留一句 |

## 核心知识点页要求

每个 `核心知识点/*.md` 至少回答：

| 模块 | 必须写清 |
|---|---|
| 知识点本体 | 这个知识点解决什么判断问题 |
| 来源贡献 | 哪些文章贡献了哪些内容，分别是补充、冲突还是层次升级 |
| 认知校准 | 它如何修正、确认或挑战已有理解 |
| 可复用结论 | 下次读类似文章时可直接复用的判断规则 |
| 适用边界 | 什么场景不能套用，缺什么证据时要降权 |
| 追查锚点 | 原文路径、链接、关键段落、关键图，不复制原文全文 |

## 排重准则

新文章进入本目录时，先构造问题指纹：

```text
问题域 + 知识点本体 + 核心机制 + 解决问题 + 适用边界 + 对用户的认知增量
```

| 判断 | 动作 |
|---|---|
| 只重复已有结论 | 不新建文件 |
| 补充了边界、反例、证据或实践动作 | 合并到已有知识点 |
| 与已有结论冲突 | 写入已有知识点的冲突判断 |
| 讨论的是另一层问题 | 新建更准确的知识点，而不是按文章标题建文件 |
