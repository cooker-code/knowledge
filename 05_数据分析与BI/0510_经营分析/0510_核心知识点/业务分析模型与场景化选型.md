## 核心问题

业务分析模型不是“模型清单”，而是把业务问题翻译成可观察变量、可比较结构、可执行动作和可复盘结果。选模型前先问场景，不按文章热度套模板。

## 判断准则

| 场景 | 可用模型 | 关键边界 |
|---|---|---|
| 增长漏斗 | AARRR、用户路径、转化漏斗 | 每层必须有可干预动作和责任人 |
| 用户价值分层 | RFM、LTV、分群标签 | 分层要对应触达策略、成本和验证指标 |
| 产品/品类组合 | ABC、波士顿矩阵、单品分配 | 组合决策要看库存、毛利、周转和供给约束 |
| 周期性变化 | 周期分析、同期群、节假日/季节性拆解 | 先确认周期稳定性和业务机制，不能把一次波动当规律 |
| 财务经营 | 杜邦、ROI、利润拆解 | 财务指标要穿透到业务驱动因子 |
| 战略环境 | PEST、SWOT、7S | 适合框架化讨论，不直接给出数据结论 |
| 复盘改进 | PDCA、5W2H、根因分析 | 必须落到动作、证据和下轮验证 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 模型越多越专业 | 模型只是一种提问方式，不能替代数据、口径和业务机制 |
| 套 AARRR/RFM 就能给增长方案 | 必须接入真实用户行为、成本收益和实验验证 |
| 咨询框架能直接指导数据分析 | 咨询框架适合梳理问题，数据分析还要补指标、样本和证据 |
| 单篇案例能直接迁移 | 先确认行业、阶段、约束和数据可得性是否相似 |

## 待验证缺口

- 需要整理“经营问题 -> 推荐模型 -> 必备数据 -> 输出动作”的选型表。
- 需要把模型输出和指标体系、归因分析、实验分析打通，避免停在框架解释。
