一、项目背景

• 业务场景：以 Facebook、TikTok等渠道投放短剧产品，需实时评估每批新增用户的投放效果与回本周期，指导后续投放决策与预算分配。

• 现状问题：传统以 Day-1、Day-7 ROI 衡量，无法刻画用户中长期价值；单看 ARPU 或单日留存也难准确判断回本时点。

二、项目目标

1. 构建一套可落地、易埋点的 LTV 计算模型；

2. 实现对每批（cohort）新增用户的「日 LTV 累计」与「回本天数」自动化监控；

3. 在 BI 仪表／Excel 模板上直观看出 LTV 曲线与 CPI 比较结果，为投放优化提供数据支撑。

三、术语与指标定义

  

| 指标 | 定义 |
| --- | --- |
| Cohort Size | 某天投放新增用户总数（固定不变） |
| 收入(day) | 该 Cohort 在第 N 天产生的总付费 |
| ARPU(day) | 收入(day) ÷ Cohort Size |
| 累计 LTV(D) | ∑\_{day=0..D} ARPU(day)，表示每位新增用户在 0~D 天内的平均价值 |
| CPI | 获取一位新增用户的平均成本 |
| 回本天数 | 最小 D 使得 累计 LTV(D) ≥ CPI |

四、LTV 计算模型设计

4.1 Cohort 划分

• 按投放日期为单位，每天生成一个 Cohort；

• Cohort Size 固定，后续日收入、日 ARPU 均以该基数计算，确保同一批用户价值可比。

4.2 日 ARPU 计算

```
ARPU(day N)  
= 当日收入_N（仅限该 Cohort) ÷ Cohort Size
```

• 日收入由财务/分成系统对接；

• 如需拆分国别、渠道、素材等，可对 Cohort 进一步细分。

4.3 累计 LTV 计算

```
累计 LTV(D)  
= ∑_{N=0..D} ARPU(day N)
```

• Day0：当日激活即付收入；

• Day1、Day2…：后续留存用户付费贡献，自然隐含留存影响；

• D 取值可到 Day180 或更长，视产品付费周期。

4.4 回本判断

• 回本天数：最小 D 满足 累计 LTV(D) ≥ CPI；

• 在图表中绘制两条折线：

• LTV 曲线（随天数上升）

• CPI 水平线

• 两线交点即回本天。

五、数据采集与处理

| 数据项 | 来源 | 频率 | 备注 |
| --- | --- | --- | --- |
| 当日新增用户数 | 投放平台（FB/Google Ads） | 日报 | 需去重并剔除僵尸、作弊用户 |
| 当日收入 | 后台订单数据 | 日报 | 统一美金 |
| CPI | 投放平台 API 或 BI 汇总 | 日报 | 稳定渠道可取平均值 |

• ETL 流程：每日凌晨自动拉取，将数据写入数据仓库；

• 数据清洗：剔除归因超时、退款、异常大额等干扰项；

• 数据存储：按照 Cohort+渠道+国家+素材 5 维度建表。

六、技术实现与可视化

1. Excel 模板

• Sheet1：原始数据填报（新增用户、日收入、CPI）；

• Sheet2：自动计算 ARPU、累计 LTV、回本天数；

• Sheet3：LTV vs. CPI 折线图，标注交点。

2. BI 仪表

• 以 Superset/Looker 为例，构建 Dashboard：

• 过滤器：按日期、渠道、国家、素材；

• 报表：每日 ARPU、累计 LTV、回本天数分布；

• 可导出 CSV/Excel。

3. 自动告警

• 当某 Cohort 在预设天数（如 30 天）仍未回本时，触发邮件/Slack 通知，提醒投放优化。

八、风险与应对

• 数据漏计/滞后：定期核对渠道报表与财务数据；

• 归因不准：设置合理归因窗口，剔除超时、作弊；

• CPI 波动大：可采用滑动平均或 7 日滚动 CPI 取值。
