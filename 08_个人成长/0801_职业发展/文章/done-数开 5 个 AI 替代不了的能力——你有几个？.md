这一篇就回答这个问题——**5 个 AI 短期内替代不了的能力**。

---

#

# 能力 1｜业务判断力——这数到底对不对

数开的核心信用，不是SQL写得漂亮，是"你给的数据，能不能上汇报"。

举个真实场景：

> AI 跑出昨天的 GMV 是 1.35 亿。你扫了一眼，第一反应是 "**比上周三高了 40%，不正常**"。一查，发现订单状态新增了一种 `pending_promotion`，被错误算进已支付。

这种"5 秒内对数字起疑"的本能，是你的核心资产。

AI 跑出来的数它不会怀疑。Agent 也不会替你扛——你是那个签字的人。

> **自查**：你看到一个数字，**能不能快速判断"靠不靠谱"**？

---

#

# 能力 2｜数据架构决策——这表该不该建

Agent 能写 SQL，但不能决定：

这张表该建在 DWD 还是 DWS？

这个字段叫 user\_status 还是 account\_state？

这条新需求要不要新建宽表，还是复用现有？

这个指标的口径在哪一层落地？

这些都是判断题，不是选择题。需要：

业务积累（你知道这个字段未来会被多少地方用）

工程经验（你知道宽表会带来什么维护代价）

治理意识（你知道命名不严，3 年后会是什么烂摊子）

Agent 没有"团队过去 3 年的踩坑记忆"。你有。

> **自查**：新需求来时，你会**先想"这事进哪一层"**，还是直接打开 IDE 写 SQL？

#

# 能力 3｜故障复盘 + 治理迭代——出事故能不能"长记性"

这是数开里最稀缺的一种能力。

| 段位 | 出事故后做什么 |
| --- | --- |
| 普通工程师 | 修代码 → 完事 |
| 高级工程师 | 修代码 → **写复盘 → 加监控 → 改下线流程** |
| 顶级工程师 | 修代码 → 复盘 → **抽象出一类事故的治理原则 → 推动团队改进** |

AI 能做前两步——它能改代码、能写复盘文档。它做不了第三步——把"一类事故"上升为"治理体系"。

这一步需要的是跨团队推动力 + 抽象能力 + 长期视野。

> **自查**：你团队过去半年的事故，**有没有变成可执行的规则 / 监控 /****SOP**？还是每次都"先这么修着，下次注意"？

---

#

# 能力 4｜数据×AI 工程能力——让数据"被 AI 用起来"

这是个全新岗位，市场严重缺人。

具体做的事（上篇也有提到）：

LLM Ready 元数据——让 Agent 能读懂每张表的业务用途

指标语义层——让 Agent 写 SQL 时不会乱用口径

字段级血缘——让异常追溯从 3 小时压到 5 分钟

可召回性治理——让 RAG / AI 搜索能找到该找到的数据

这些工作 5 年前根本不存在，今天它们正在变成新的核心岗位。

会做的人，简历上 1 行就能压过别人 5 行的"做了 N 张报表"。

> **自查**：你公司的数仓，**Agent 能接进来用吗**？还是 Agent 看到字段名只能瞎猜？

---

#

# 能力 5｜跨团队协作与推动力——让人达成一致

最被低估的一种能力。

数据问题里真正难的从来不是技术，是协作：

财务说 GMV 是 A，运营说是 B——你怎么让他们达成一致

业务方天天来要 ad-hoc 数据——你怎么把他们引到自助平台

研发改了字段语义没通知数据——你怎么把变更流程立起来

这些事 AI 完全帮不上——它不能开会、不能拍桌子、不能在群里 @all "这事我们必须定下来"。

但没有这种推动力，再好的技术方案都落不下去。

> **自查**：近期，**有没有靠你的协调，让 2 个以上的团队达成了一个数据共识**？

# 写在最后

数开的好时代刚开始——但不是SQL写得越快越值钱。

AI 时代要保留的，是判断、设计、治理、整合、推动。听起来都很"软"，但每一个都比"打字快"硬 10 倍。

写SQL会被工具吃掉，这 5 件事不会。
