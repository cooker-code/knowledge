# 写在前面

朋友前几天面某厂数开岗，二面被问了一题：

> "你做过的数仓里，哪一层最适合喂给 RAG 检索？为什么？"

他懵了。SQL 八股、维度建模、Spark 调优都准备到位，唯独这种题——以前数开面试没人问。

回来跟我吐槽时，他说了一句话我觉得挺真实：

> "感觉这两年数开岗位静悄悄换了一茬题，都不知道如何应对。"

这篇就把"换了哪些题、为什么换、怎么应对、最后真要往哪发力"梳理清楚。

# 一、数开面试发生了什么

把过去 2-3 年和现在的数开面试题摆一起，最大的变化不是"加了几个 AI 八股"，而是整套岗位定位变了。

以前：

数开是"做数据给人看"的工种。终点是 BI 报表、是分析看板、是数据驱动决策。

1. SQL / 维度建模 / 雪花 vs 星型

2. Hive / Spark / Flink 调优

3. 数仓分层 ODS / DWD / DWS / ADS

4. 调度、ETL、数据质量

5. 数据治理（元数据、血缘、字典）

这套能力栈，几乎没怎么变过。

现在：

数开变成"做数据给 AI 用"的工种。终点是 Agent 能稳定调用、RAG 能召回准确、模型能持续被反馈数据改进。

上面那套基础全部还要，但叠了一层新的：

1. Embedding / 向量检索 / ANN 索引

2. RAG / 混合召回 / 图谱

3. 数据契约 / Agent 数据反馈 / SQL 沙箱

4. AI 上下文治理 / hallucination 防控

目标变了：从"BI 给人看"，变成"语料给 Agent 用"。

题型不是替换，是叠加——只会老题已经过不了一轮，老题 + 新题都答得出，才有 offer。

# 二、8 道风格示意题（按现在的考察方向）

下面几道是梳理的真实考察点。

老题型示例

1. 数据倾斜的排查思路和你用过的解法

2. ODS → DWD 你怎么洗，错数怎么追溯

3. 维度建模 SCD 如何处理

......

新题型示例

1. 你做的数仓里，哪一层最适合喂给 RAG？为什么？（考的是"语料视角"）

2. 给 Agent 用 SQL 沙箱，你怎么防 SQL 注入和资源失控？（考的是 AI 安全边界）

3. RAG 召回不行，从数据侧能做哪三件事？不许说"换 embedding 模型"。（考的是 chunk 策略、清洗、索引设计）

4. AI 调用日志里怎么挖 bad case 反哺训练数据？（考的是反馈闭环）

5. 离线训练数据 vs 在线检索数据，怎么保证一致性？（考的是数据契约 + 跨链路一致性）

......

关键观察：

老题考"做没做过"

新题考"能不能从数据的角度想清楚 AI 在哪儿崩、自己能动哪一刀"

老题答得出 = 不会被刷；新题答得出 = 拿到 offer。

# 三、应对策略：不是再学一门 AI

很多人看到面试题变化，第一反应是"我得去学 LLM 八股"——刷 transformer 架构、背 prompt engineering、读 OpenAI 文档。

这是性价比最低的路线。原因是：

1. 八股题准备出来的答案 AI 一秒给得出，没区分度

2. 没动手做过的 AI 知识，深度问到第二层就崩

3. 时间花进去，但你的数据经验没用上

正确姿势：用 AI 的视角，重新理解你做过的事

数据世界里你做过的事，几乎全部有 AI 视角下的对应物。同一件事，换一套术语：

| 数据视角（你已经会） | AI 视角（同一件事的新说法） |
| --- | --- |
| 数仓建模 | 语料建模 / chunk 切片策略 |
| 数据质量 | 检索召回质量 / chunk 质量 |
| 数据治理 | AI 上下文治理 / hallucination 防控 |
| SLA / 链路稳定性 | AI 响应一致性 / cache 策略 |
| ETL / 调度 | 索引重建 / Embedding 更新 |

把你过去做过的项目，问一遍：

1. DWD 事实表，如果切成 chunk 喂 RAG，怎么切才有用？

2. 我做过的数据质量监控，能不能直接搬来做"检索召回质量"的监控？

3. 我搞过的元数据血缘，能不能告诉我"哪一条 hallucination 是上游哪张表带来的"？

这套训练做完，你不是"会一点 AI 的数开"，是"做 AI 数据的数开"——后者贵很多。

三个具体动作

复盘旧项目：把每个做过的数仓项目，写一段"如果今天重做、加上 AI 用途，会怎么改"。

找一个端到端能跑的实战：不是 toy demo，是真有数据、有指标、有错误处理的完整链路。

拿真实业务场景做 case study：模拟一次"业务方要做 Agent，我怎么从数据这一侧给方案"。

# 四、面试官最爱挖的三个硬核能力

老题问的是"知不知道"，新题问的是"做没做过"。下面三个能力，是新题问到第二层、第三层时真正分高低的：

① 数据 → Embedding → 向量库 全栈打通

不是"接个 OpenAI Embedding API"，是会做：

1. chunk 策略调优（按句、段、语义切；带不带 overlap）

2. 向量库 ANN 索引选型（HNSW vs IVF vs PQ，召回 vs 内存的权衡）

3. 写入瓶颈解决（百万级数据批量写入，怎么不打爆）

4. 增量更新（embedding 模型升级了，怎么重建不影响线上）

这条链路上每一步都对应数开的老本行：批量写、性能调优、增量更新——所以是你的优势区，不是劣势区。

② 数仓 + 图谱 + 全文 混合召回

SQL 解决不了关系问题（"和 A 公司有关联的所有产品"）；

向量解决不了精确匹配（"产品代码 = 502115"）；

全文解决不了语义相似（"我要稳健些的"）。

生产 RAG 上了量之后，三套混着用。能讲清楚这三套各自的边界、什么时候用哪个、怎么 rerank 融合，就是数开里非常稀缺的人。

③ 数据反馈闭环

AI 用了数据 → 生成调用日志 → 从日志挖 bad case → 反哺训练 / 检索数据集——这是个完整的闭环，也是数开和ML工程师真正分界线：ML 工程师懂模型，数开懂闭环里的数据流。

这一条只要面试时能讲出来你设计过的闭环（哪怕只是 demo 级），通常会进到 HM 面。

# 五、对应的实战项目

篇一：金融数仓 + 数据治理（ODS/DWD/ADS、主数据、数据契约、SLA/SLO）

篇二：实时离线 Pipeline + 知识图谱（Flink CDC 双流、EntityManager 同步、Dgraph）

篇三：搜索引擎 + 语义检索（ES + BGE 混合召回、Rerank，Recall 50% → 92%）

篇四：Data Agent + 反馈闭环（SQL 沙箱、Function Call、bad case 挖掘、训练集回流）

这条链路覆盖了上面讲的"三个硬核能力"——面试不慌。

# 写在最后

数开不会消失，会进化。

**不是"被 AI 替代"，是"成为 AI 的数据工程师"。**

过去 10 年，数开的目标是"让人能看到数据"；下一个 5 年，目标是"让 AI 能用好数据"。能完成这个迁移的人，在现在的招聘市场是稀缺的。
