## 核心问题

数据岗位的长期竞争力不再是“会写 SQL / 会用 Spark / 会接需求”，而是能否把数据能力变成可验证的业务、治理、架构和 AI 数据底座价值。

## 认知校准点

| 校准点 | 处理 |
|---|---|
| “懂业务”不能滑成“替业务负责” | 数据团队负责证据质量、口径一致、方法、风险提示和效果度量；业务团队必须负责目标选择、资源下注、策略取舍和最终结果。 |
| “少写代码”不是少产出 | 代码执行层价值下降后，精力要前移到模型设计、口径治理、因果判断、数据资产和组织协同。 |
| “数据 + AI”不是再背一套 LLM 八股 | 正确迁移是把已有数仓经验映射为语料建模、上下文治理、语义层、日志反馈、RAG/Agent 可用数据。 |
| “职级路线图”只能作参考 | 薪资、职级和年限数字证据弱，不能替代代表作、影响范围和组织位置。 |

## 能力分层

| 层级 | 典型工作 | 风险 | 升级方向 |
|---|---|---|---|
| 执行层 | 取数、写 SQL、报表、简单 ETL | AI、低代码和自助分析最容易压缩这一层 | 提升口径判断、质量审查、复杂查询诊断和需求澄清。 |
| 建模治理层 | 数据分层、指标口径、数据质量、血缘、成本治理 | 容易被业务临时需求拖成宽表维护 | 建立治理规则、Owner 机制、影响评估和复盘闭环。 |
| 业务伙伴层 | 将业务问题转成数据假设、实验、指标和策略评估 | 容易越界替业务做决策 | 做“编辑”而非“作者”：逼业务写清问题，自己给证据和边界。 |
| 平台架构层 | 元数据平台、语义层、数据产品、实时/湖仓/AI 数据底座 | 技术深但业务采用弱 | 产品化、平台化、可观测、可运营，证明被使用和节省成本。 |
| AI 数据工程层 | 让 Agent/RAG/模型稳定使用企业数据 | 容易被包装成概念项目 | 做 LLM-ready 元数据、语义口径、字段血缘、调用日志 bad case 闭环。 |

## 判断准则

| 问题 | 判断准则 |
|---|---|
| 这个需求是否值得做 | 先问它支持哪个业务决策、Owner 是谁、结果不好谁复盘和调整动作。 |
| 这个模型或指标应由谁负责 | 数据团队给可计算口径、历史影响和风险；业务 Owner 签最终口径和策略责任。 |
| 一个数仓工程师是否高阶 | 不看工具清单，看能否解释业务问题、建模权衡、质量治理、成本收益和跨团队推动。 |
| AI 是否替代该岗位 | 看被替代的是任务、流程还是责任；如果责任包含判断、签字、推动和治理，短期不能简单替代。 |
| 是否形成代表作 | 必须能写成“我做了 X，影响了 Y 指标/成本/风险/组织机制，并留下 Z 可复用资产”。 |

## 行动检查表

| 动作 | 检查问题 |
|---|---|
| 复盘最近一个数据需求 | 它最后支持了什么决策？有没有业务 Owner 签字？ |
| 梳理个人能力 | 当前更多是执行、治理、业务伙伴、平台架构还是 AI 数据工程？ |
| 打磨代表作 | 是否有可量化结果、复用资产、协作对象和后续治理规则？ |
| 设计 AI 转型 | 是否把表结构、口径、血缘、权限、bad case 日志做成 Agent 可用上下文？ |
| 控制越界 | 是否把“建议页”改成“决策请求页”，把取舍交回业务 Owner？ |

## 待验证缺口

- 用真实招聘 JD 验证“数据 x AI 工程”是否已成为稳定岗位要求。
- 用用户自己的项目复盘验证哪类代表作最能提升组织影响力。
- 后续可把“数据团队权责边界”迁移到数据管理/管理沟通知识点中做交叉引用。
