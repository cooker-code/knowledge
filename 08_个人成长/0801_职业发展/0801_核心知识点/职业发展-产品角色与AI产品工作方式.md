## 核心问题

产品角色的稀缺性不在于写更多需求文档，而在于端到端所有权、复杂问题拆解、客户同理心、质量打磨、资源交换，以及在 AI 产品中管理概率系统的能力。

## 产品负责人能力

| 能力 | 判断标准 | 风险 |
|---|---|---|
| 端到端所有权 | 对产品结果、客户体验、业务指标和发布采用负责 | 把完成开发误当成交付完成。 |
| 细节深潜 | 能进入系统、流程、客户、监管、运营细节 | 只停留在路线图和会议层。 |
| 复杂问题抽象 | 能从单个国家/客户/场景抽象出可扩展流程 | 每次定制，无法规模化。 |
| 质量标准 | 功能可精简，但体验、可靠性和关键细节不能降级 | 用 MVP 当借口发布粗糙产品。 |
| 资源交换 | 带着信息、备选路径和风险请求决策 | 只试图说服领导接受自己的方案。 |

## AI PM 与传统 PM 的差异

| 维度 | 传统产品 | AI 产品 |
|---|---|---|
| 需求表达 | 规则、流程、页面、交互 | 问题类型、输入输出、效果目标、能力边界。 |
| 数据需求 | 业务表、字段、索引 | 训练/评估/标注数据、特征、埋点、知识库和反馈数据。 |
| 功能描述 | 页面和规则 | Prompt、模型调用流程、置信度、检索、降级。 |
| 性能指标 | 响应时间、并发、可用性 | 准确率、召回率、F1、幻觉率、转人工率、满意度、成本。 |
| 异常处理 | 网络、权限、数据异常 | 低置信、幻觉、对抗输入、模型退化、知识库无匹配。 |
| 迭代方式 | 改代码和交互 | 数据闭环、bad case、评估集、模型/Prompt/检索策略迭代。 |

## AI 产品 PRD 检查表

| 模块 | 必填问题 |
|---|---|
| 问题定义 | 这是分类、预测、生成、推荐、检测还是理解问题？为什么需要 AI？ |
| 输入输出 | 输入字段、上下文、约束、输出结构和置信度怎么定义？ |
| 数据 | 数据来源、质量、标注、样本量、隐私合规、冷启动策略是什么？ |
| 评估 | 离线指标、在线 AB 指标、业务指标和人工验收怎么组合？ |
| 异常 | 低置信、幻觉、超时、越权、攻击样本如何处理？ |
| 降级 | 何时转人工、何时规则兜底、何时拒绝回答？ |
| 监控 | 线上效果、成本、延迟、bad case 和用户反馈如何进入迭代？ |
| 成本 | API、GPU、标注、人工、维护成本是否有量级估算？ |

## 交付准则

1. “功能上线”不是终点，用户知道、运营采用、客服能解释、指标可追踪，才算进入交付闭环。
2. MVP 可以限制范围，但不能牺牲核心体验和信任。
3. AI 产品上线前必须有失败路径，不允许只展示理想样例。
4. 产品人不一定亲自训练模型，但必须能问出模型边界、数据边界和评估边界。
5. 任何“AI PM 能力”如果脱离真实业务场景和指标，都只是概念包装。
