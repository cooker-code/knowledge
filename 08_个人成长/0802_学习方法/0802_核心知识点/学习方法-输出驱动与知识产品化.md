## 核心问题

学习如果只停留在输入和收藏，会形成“看上去在进步”的幻觉。输出驱动的关键，是把经验抽象成可复用规则，再把规则变成他人能理解、执行、反馈的产物。

## 判断准则

| 阶段 | 做法 | 产物 |
|---|---|---|
| 选择主题 | 选自己能持续讨论、实践和改进的领域 | 主题树 / 长期问题清单 |
| 形成观点 | 不只复述知识，明确自己反对什么、补充什么 | 观点卡 / 判断准则 |
| 抽象算法 | 把直觉拆成变量、阈值、触发动作 | 表格 / 检查表 / 流程 |
| 写成内容 | 用注意力、观点、优势、游戏化组织表达 | 文章 / 脚本 / 演示 |
| 做成工具 | 让别人输入场景，你输出可执行建议 | 模板 / Prompt / 小交互 |

## 认知偏差

| 常见错误认知 | 正确理解 |
|---|---|
| 写作是表达能力 | 写作首先是检验理解和压缩判断的能力 |
| 收藏等于积累 | 收藏只是库存，能否生成判断、动作和反馈才是资产 |
| Prompt 是问答技巧 | 好 Prompt 是把个人判断标准钉死，让 AI 能照流程执行 |
| 内容要追求漂亮 | 对知识沉淀来说，先把变量、阈值、动作写清楚，比修辞更重要 |

## 可复用检查表

| 检查项 | 问题 |
|---|---|
| 是否有具体对象 | 这篇输出服务哪类人、哪类场景、哪类问题 |
| 是否有新观点 | 相比常识，它提供了什么反常识、边界或纠偏 |
| 是否有判断算法 | 读者能否据此判断该不该做、下一步做什么、何时停止 |
| 是否能被执行 | 是否包含变量、阈值、步骤或模板 |
| 是否有反馈口 | 输出后如何知道它有没有帮助他人或改善结果 |

## 待验证缺口

- 需要把用户已有知识库文章蒸馏流程也看作输出产品，验证“知识点页是否比文章摘要更可复用”。
- APAG 框架偏内容增长场景，技术知识写作需要降权“注意力技巧”，优先保留结构、判断和证据。
