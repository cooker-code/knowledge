“在现代生活中，简单的做法一直难以实现，因为它有违某些努力寻求复杂化以证明其工作合理性的人所秉持的精神。” -- 尼古拉斯《反脆弱》

自从知道奥卡姆剃刀原理之后，我开始重新审视身边的设计，做了很多简化的设计。得益于这些简化的工作，让我有机会从复杂中抽身，深吸一口气，闭上眼睛，感受简洁带来的清爽怡人。

这篇文章，我会带领大家一起认识什么是奥卡姆剃刀，以及感受如何使用剃刀思维，化繁为简，进行简化设计。**本文大部分内容来自于我的新书《程序员的底层思维》，如果你看完有收获，也请支持这本书，谢谢：）**

# **1. 奥卡姆剃刀**

奥卡姆剃刀原理，是指如无必要，勿增实体（Entities should not be multiplied unnecessarily），即“简单有效原理”。奥卡姆在《箴言书注》2卷15题说：“切勿浪费较多东西去做，用较少的东西，同样可以做好的事情。”

比如，托勒密的地心说模型是一个本轮均轮模型。人们可以按照这个模型，定量计算行星的运动，据此推测行星所在的位置。到了中世纪后期随着观察仪器的不断改进，人们能够更加精确地测量出行星的位置和运动，观测到行星实际位置与这个模型的计算结果存在偏差，一开始还能勉强应付，后来**小本轮增加到八十多个**，却仍然不能精确地计算出行星的准确位置。

1543年，波兰天文学家哥白尼在临终时发表了一部具有历史意义的著作—《天体运行论》。明确了以太阳为中心的宇宙模型，哥白尼的计算不仅结构严谨，而且计算简单，与已经加到八十余个本轮的地心说相比，哥白尼的计算与实际观测资料能更好地吻合。

我们的软件设计也是一样，软件研发作为一个纯思维逻辑游戏。每一个功能都有很多种实现的可能性，那么，达成同样的功能，我们为什么不选择一种简单的实现呢？在华为最新的软件文化的八大原则中，特意把**简单设计**作为其中的一项，其用意也就在于不希望大家把事情搞的太复杂。

# **2. 干掉流程引擎**

我曾经一度困惑，流程引擎作为一个DSL（Domain Specific Language：领域专用语言），为何会有如此的魅力？被这么多人奉为圭臬。我分析了一下，原因可能有两个：

1. 首先，我这里的业务比较复杂，所以需要一个工具来帮我治理复杂度，而市面上最常用的解决方案莫过于流程引擎了。
2. 其次，作为业务技术人员，我需要一个东西来体现自身的技术价值，出于技术沉淀的需要，需要一些看起来比较有“技术感”的东西。

然而，对于大部分业务场景来说，流程引擎并不能化解这里的复杂度，反而会添加新的复杂度。使用了流程引擎的代码，因为流程编排的逻辑在配置文件中，导致阅读代码的体验非常差。必须要来回切换，才能看清整个代码逻辑的全貌。另外，因为每一次用户请求都会拉起一个新的流程实例（instance），其性能也非常不好。此外，你还要熟悉、学习流程引擎，这些都是额外的认知负荷。

综上，我的建议是，**除非你的应用有极强的流程可视化和编排的诉求，比如对灵活性要求极高的SaaS低代码（low-code）平台，否则非常不推荐使用流程引擎等工具**。如果不使用流程引擎也能完成同样的业务功能，而且更简单，那么就果断使用奥卡姆剃刀剔除之吧。

# **3. 极简状态机**

相比较流程引擎而言，状态机就要轻量很多了。即便如此，也架不住大家“渴望做事”的热情，就拿现在开源社区被使用最多的两个状态机引擎Spring State Machine和Squirrel State Machine来说。这两个状态机引擎是github上排名前两名的状态机引擎，**它们的优点是功能很完备，缺点也是功能很完备。**

在日常业务中，经常会碰到一些有很多状态流转的情况，使用状态机来表达会优雅很多。然而在使用这些开源状态机的时候，有两个问题我难以忍受，一个是其功能太多，我只需要状态流转，你给我那么的API（诸如parallel，fork，join等）我根本用不上；另一个是性能问题，如下图所示，每一个用户请求都会创建一个新的状态机实例，好费电啊。

鉴于此，我决定自己设计一个极简的状态机引擎。能满足我两个诉求即可：

1. 要能满足状态流转的DSL表达。
2. 状态机需要是singleton的，它只负责状态流转的表达，不需要维护状态。

在这个设计目标的指引下，我用了1196行代码（其中372行是注释）实现了一个极简状态机，目前是COLA开源中的一个组件。用过的朋友，都给予了很高的赞誉，说这就是他们想要的状态机引擎。更多内容可以参看另一篇文章。

# **4. COLA的迭代**

COLA从1.0发展到4.0，其自身就是一个典型的化繁为简的案例。

在COLA 1.0中，为了遵循CQRS（Command Query Responsibility Seperation：命令查询分离）模式，采用了命令模式来处理用户请求。设计的初衷是想通过框架，一方面强制约束Command和Query的处理方式；另一方面是把服务门面（Service Facade）里面的逻辑，强制拆分到CommandExecutor中去，防止服务代码膨胀过快。除此之外，里面还有拦截器（Interceptor）功能，扩展点（Extension Point）功能。

然而，随着我在项目中的实践，我发现很多设计并不是很有必要，比如Interceptor本质是为了AOP，而基于Spring注解的AOP似乎更加优雅。再比如扩展点虽然有用，但是也没有必要和架构捆绑在一起。

鉴于此，COLA的迭代演进之路，就是一路的做减法，作为架构，最后只保留了整洁架构的核心思想。正如Bob大叔在《Clean Architecture》中所说：

*Although these architectures all vary somewhat in their details, they are very similar. They all have the same objective, which is the separation of concerns. They all achieve this separation by dividing the software into layers. Each has at least one layer for business rules, and another layer for user and system interfaces.*

不管应用架构如何设计，其最终形态，也不过就是下图所示的两个圈圈，核心做一件事，即分离关注点，将技术面和业务面分开。当然，COLA也不例外。

# **5. 配置系统简化**

最近一次动用奥卡姆剃刀，是在一个配置系统设计中。这个项目在我进去之前，已经进行了半年，但是因为设计太复杂，所以进展一直比较缓慢。进行一番深入了解之后，我发现设计“想要”的东西太多了。比如，他们想对配置进行逻辑分组，所以在配置文件之外，又增加了一个ConfigSet的概念；为了想支持多人并行配置，又设计了一个类似于git的Master/Branch模型。再叠加本身业务部署环境的复杂性，最终把一个维度的复杂性，变成了三个维度的复杂性。

我仔细分析以后，发现ConfigSet的概念，对于业务功能基本没有帮助；Master/Branch看似很高级，但真正落到业务场景中，也基本不会有“并发”修改配置的诉求。在我的建议下，项目组最终决定去除掉这两个多余的维度。去除之后，整个配置模型才重新回归到一个简洁的状态。

**把一个事情搞复杂是一件简单的事，但要把一个复杂的事变简单，这是一件复杂的事。**愿大家都能用一颗朴素之心，行有为之事。更多有用的思维能力尽在《程序员的底层思维》。
